AWS Elastic Beanstalk

Deploy and scale web applications without managing the underlying infrastructure — upload your code, Beanstalk handles EC2, load balancing, scaling, and health monitoring.

🏗️ What is Elastic Beanstalk?

A PaaS that orchestrates EC2, Auto Scaling, ALB, RDS, CloudWatch, and more — so you focus on code, not infrastructure.

One-liner Elastic Beanstalk = upload your application code (zip/war/jar/Docker) → Beanstalk provisions and wires together EC2, Auto Scaling Group, Load Balancer, Security Groups, and CloudWatch alarms automatically. You keep full access to the underlying resources.
📦 Application

Logical container for your app. Holds Application Versions (each a deployable artifact) and Environments (where versions run).

🌍 Environment

A running collection of AWS resources (EC2, ASG, ALB, RDS optional) executing one application version. You can have multiple environments per app (dev, staging, prod).

📋 Environment Configuration

Defines the resources and settings for an environment — instance type, scaling triggers, environment variables, platform version. Stored as a saved configuration you can reuse.

🏷️ Platform

Pre-configured stack with OS + runtime: .NET on Windows/Linux, Java, Node.js, Python, PHP, Ruby, Go, Docker. AWS maintains and patches the platform.

📄 Application Version

A specific, labeled, deployable build of your code stored in S3. You can deploy any version to any environment — easy rollback to a previous version.

🔧 Underlying Resources

Beanstalk creates real EC2 instances, an ASG, an ALB/CLB, security groups, and an S3 bucket — you can access and customise all of them directly if needed.

🌍 Environment Tiers

Beanstalk supports two environment tiers depending on your workload pattern.

🌐 Web Server Environment

For request/response applications. Provisions: ALB (or Classic LB) → Auto Scaling Group → EC2 instances. Handles HTTP/HTTPS traffic directly.

Use for: REST APIs, web apps, MVC sites.

ALB Public-facing

⚙️ Worker Environment

For background processing. Provisions: SQS queue (auto-created) → Auto Scaling Group → EC2 instances running an SQS daemon that pulls and processes messages.

Use for: Async jobs, batch processing, queue consumers.

SQS-backed No public endpoint

Single Instance vs Load Balanced

1️⃣ Single Instance

One EC2 instance, no ALB, no Auto Scaling. Cheapest option. Elastic IP attached for a stable address.

Use for: Dev/test environments, low-traffic apps, cost-sensitive demos.

⚖️ Load Balanced + Auto Scaling

ALB distributes traffic across multiple EC2 instances in an Auto Scaling Group. Scales based on CPU, network, or request count triggers.

Use for: Production workloads needing HA and elastic scale.

🔄 Deployment Policies

Choose how new application versions roll out to running instances — trade speed vs risk.

⚡ All at Once
Fastest · Brief downtime
Deploys to all instances simultaneously. Causes a brief outage during deploy. Fastest option. Good for dev environments where downtime doesn't matter.
🔄 Rolling
Batch-based · Reduced capacity
Deploys in batches (e.g. 25% of fleet at a time). Old version serves traffic while new version deploys to other batches. Reduces total capacity during deploy — no extra instances launched.
➕ Rolling with Additional Batch
Batch-based · Full capacity maintained
Launches one extra batch of new instances first, then performs rolling deployment on the rest. Maintains full capacity throughout — costs slightly more during deploy.
🔵🟢 Immutable
Safest · Full new ASG
Launches an entirely new Auto Scaling Group with the new version. Once healthy, traffic shifts, old ASG terminated. Zero-downtime, easiest automatic rollback — just delete the new ASG.
🔁 Traffic Splitting
Canary · Gradual rollout
Launches new version in a temporary ASG, routes a % of traffic to it (canary). If healthy after evaluation period, shifts 100% and terminates old version. Built-in canary testing.
📥 Blue/Green (Manual)
Separate environment · Swap CNAME
Not a built-in policy — create a second environment, deploy + test, then swap environment URLs (CNAME swap). Full isolation between old and new during testing.

💡 Blue/Green Deployment Steps 1. Create a new environment (Green) with the new version. 2. Test thoroughly using its own URL. 3. Use eb swap or console "Swap Environment URLs" to switch CNAMEs — production traffic now hits Green instantly. 4. Keep Blue running briefly for instant rollback (swap back) if issues arise.

⚙️ Configuration with .ebextensions

Customise the environment beyond the console — install packages, run commands, set config, modify resources.

.ebextensions folder Place YAML/JSON config files in a .ebextensions/ folder at the root of your deployment package. Beanstalk applies them automatically during deployment — before your app starts.
YAML .ebextensions/01-environment.config — env vars + instance settings 
option_settings:
  # Environment variables for your app
  - namespace: aws:elasticbeanstalk:application:environment
    option_name: ASPNETCORE_ENVIRONMENT
    value: Production

  - namespace: aws:elasticbeanstalk:application:environment
    option_name: ConnectionStrings__DefaultConnection
    value: "Server=mydb.xyz.rds.amazonaws.com;Database=orders;"

  # Instance type and Auto Scaling
  - namespace: aws:autoscaling:launchconfiguration
    option_name: InstanceType
    value: t3.medium

  - namespace: aws:autoscaling:asg
    option_name: MinSize
    value: '2'

  - namespace: aws:autoscaling:asg
    option_name: MaxSize
    value: '10'

  # Health check path
  - namespace: aws:elasticbeanstalk:application
    option_name: Application Healthcheck URL
    value: /health
YAML .ebextensions/02-commands.config — run shell commands + install packages 
packages:
  yum:
    git: []
    jq: []

commands:
  01_create_log_dir:
    command: "mkdir -p /var/log/myapp"
    test: "[ ! -d /var/log/myapp ]"  # only run if dir doesn't exist

container_commands:
  01_migrate_db:
    command: "dotnet ef database update"
    leader_only: true   # run once, only on the "leader" instance

files:
  "/etc/cron.d/cleanup-job":
    mode: "000644"
    owner: root
    group: root
    content: |
      0 2 * * * root /usr/bin/cleanup-old-files.sh

⚠️ RDS Coupling Warning If you create an RDS instance through the Beanstalk environment wizard, it's tied to the environment's lifecycle — terminating the environment can delete the database! For production, always create RDS separately and connect via environment variables/connection string.

🩺 Health Monitoring & Auto Scaling

Beanstalk continuously monitors instance and application health, and scales based on configurable triggers.

🟢 Basic Health Reporting

Reports instance-level health only — based on EC2 status checks and Auto Scaling Group activity. No application-level insight.

Free Limited visibility

⭐ Enhanced Health Reporting

Reports detailed application-level metrics: request latency, HTTP status code counts (2xx/3xx/4xx/5xx), CPU utilisation, and a calculated health colour per instance.

Recommended No extra cost

Health Colour States

ColourMeaningTypical Cause
🟢 GreenHealthy — passing all checksNormal operation
🟡 YellowDegraded — elevated error rate or latencyApp slow, partial failures, high 4xx rate
🔴 RedSevere — failing health checksApp crashed, high 5xx rate, instance unresponsive
⚫ GreyNo data / pendingInstance launching, deployment in progress

Auto Scaling Triggers

📊 Metric-Based Scaling

Scale out/in based on CloudWatch metrics: Average CPU Utilization, Network In/Out, Request Count Per Target (ALB).

Set upper/lower thresholds, breach duration, and scaling step size (e.g. +1 instance at a time).

⏰ Scheduled Scaling

Combine with custom CloudWatch alarms or use predictable patterns — e.g. scale up before known traffic spikes (marketing campaigns, sales events).

Configure via aws:autoscaling:scheduledaction namespace in .ebextensions.

YAML Auto Scaling trigger configuration 
option_settings:
  - namespace: aws:autoscaling:trigger
    option_name: MeasureName
    value: CPUUtilization

  - namespace: aws:autoscaling:trigger
    option_name: Unit
    value: Percent

  - namespace: aws:autoscaling:trigger
    option_name: UpperThreshold
    value: '70'

  - namespace: aws:autoscaling:trigger
    option_name: LowerThreshold
    value: '30'

  - namespace: aws:autoscaling:trigger
    option_name: BreachDuration
    value: '5'   # minutes

🧩 C# / .NET on Elastic Beanstalk

Deploying ASP.NET Core applications — from Visual Studio, CLI, or CI/CD.

Two Platform Options for .NET .NET Core on Linux — modern, cheaper (no Windows licensing), recommended for new apps. .NET on Windows Server — needed for legacy .NET Framework or Windows-specific dependencies (IIS features, COM interop).
Bash Publish .NET app for Beanstalk deployment 
# Publish self-contained or framework-dependent build
dotnet publish -c Release -o ./publish

# Zip the published output (Beanstalk deploys zip files)
cd publish
zip -r ../deploy.zip .
cd ..

# Deploy via EB CLI
eb deploy
Program.cs Health check endpoint + read config from Beanstalk env vars 
var builder = WebApplication.CreateBuilder(args);

// Beanstalk injects env vars set in .ebextensions or console
var connString = builder.Configuration["ConnectionStrings:DefaultConnection"]
    ?? Environment.GetEnvironmentVariable("ConnectionStrings__DefaultConnection");

builder.Services.AddDbContext<AppDbContext>(opts =>
    opts.UseSqlServer(connString));

builder.Services.AddHealthChecks()
    .AddDbContextCheck<AppDbContext>();

var app = builder.Build();

// Beanstalk health check hits this — must return 200 for "Healthy"
app.MapHealthChecks("/health");

// IMPORTANT: Beanstalk's load balancer / nginx proxy forwards on port 80/443 -> app on 5000
// Ensure Kestrel binds to the port Beanstalk expects (default via ASPNETCORE_URLS)

app.Run();

{

    "aspNetCoreWeb": [
      {
        "name": "order-api",

💡 .NET-Specific Tips Set ASPNETCORE_ENVIRONMENT via environment properties (never hardcode). Use IConfiguration to read Beanstalk env vars — double underscore __ maps to nested config sections (e.g. ConnectionStrings__DefaultConnection → ConnectionStrings:DefaultConnection). For Linux platform, ensure your csproj targets a supported .NET version matching the platform branch.

⌨️ EB CLI Commands

The official Elastic Beanstalk command-line tool — install with pip install awsebcli.

CLI Initial setup 
# Install EB CLI
pip install awsebcli --upgrade --user

# Initialise — creates .elasticbeanstalk/config.yml, asks platform + region
eb init my-order-app --platform "docker" --region us-east-1

# Create environment (provisions ALB, ASG, EC2)
eb create production-env \
  --instance-type t3.medium \
  --min-instances 2 \
  --max-instances 10 \
  --envvars ASPNETCORE_ENVIRONMENT=Production
CLI Day-to-day deployment operations 
# Deploy current directory to the active environment
eb deploy

# Deploy with a specific label
eb deploy --label v1.2.3

# Check environment status and health
eb status
eb health

# Tail logs
eb logs
eb logs --stream     # live tail

# Open the app in your browser
eb open

# SSH into an instance
eb ssh

# List all environments
eb list

# Switch active environment context
eb use staging-env
CLI Blue/Green deployment with environment swap 
# 1. Create a new "green" environment with the new version
eb create production-green --cname myapp-green

# 2. Test thoroughly using the green environment's own URL
eb open production-green

# 3. Swap CNAMEs — production traffic instantly moves to green
eb swap production-blue --destination_name production-green

# 4. If issues arise, swap back instantly
eb swap production-green --destination_name production-blue

# 5. Once confident, terminate the old environment
eb terminate production-blue
CLI Configuration & cleanup 
# View / edit environment config interactively
eb config

# Set environment variables
eb setenv LOG_LEVEL=info MAX_RETRIES=5

# Scale manually
eb scale 4   # set to 4 instances

# Terminate an environment (deletes all resources)
eb terminate staging-env

💡 Best Practices

Production-grade Elastic Beanstalk setup for reliability, security, and maintainability.

✅ Decouple RDS from the Environment

Never create RDS through the Beanstalk environment wizard for production. Create it separately, connect via environment variables. Prevents accidental data loss when terminating environments.

✅ Use Immutable or Blue/Green for Production

Avoid "All at Once" in production — it causes downtime. Immutable deployments give you automatic rollback safety; Blue/Green gives you full pre-production testing on real infrastructure.

✅ Commit .ebextensions to Git

Treat .ebextensions/ as infrastructure code. Version control every config change. Use numbered prefixes (01-, 02-) to control execution order explicitly.

✅ Enable Enhanced Health Reporting

Always use Enhanced over Basic — no extra cost, far more visibility (latency, status codes, per-instance health). Essential for diagnosing production issues quickly.

✅ Set Proper Health Check Path

Don't rely on the default "/" health check if your app needs warm-up or has expensive root route logic. Use a dedicated lightweight /health endpoint that checks DB connectivity.

✅ Use Managed Platform Updates

Enable "Managed Updates" to auto-apply minor platform patches during a defined maintenance window. Keeps OS and runtime patched without manual intervention — reduces security risk.

✅ Store Secrets in Secrets Manager / SSM

Don't put database passwords directly as plain environment variables in .ebextensions (visible in console/CLI). Reference Secrets Manager or SSM Parameter Store and fetch at app startup instead.

✅ Use Separate Environments per Stage

Create distinct dev/staging/production environments — not just config differences in one environment. Gives full isolation and the ability to test deployment policies safely before prod.
