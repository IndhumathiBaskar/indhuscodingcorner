🧭 How to Approach Any System Design Problem

👶 Start Here — Every Example Follows the Same Recipe

System design can feel overwhelming because the problems are open-ended. But almost every problem — from a URL shortener to Netflix — can be approached with the same 5-step framework. Once you know the framework, you just apply it to whatever system you're asked about. Every example in this guide follows these exact steps.

The 5-Step Framework

1

Clarify Requirements

Split into Functional (what the system does — "users can shorten a URL") and Non-Functional (qualities — "must be highly available, low latency"). Ask: how many users? Read-heavy or write-heavy? Do we need real-time updates?

2

Estimate Scale (Back-of-Envelope)

Rough numbers for: requests per second, storage needed per year, bandwidth. These numbers drive every design decision — a system for 100 users looks completely different from one for 100 million.

3

High-Level Design

Draw the main boxes: client, API/load balancer, application servers, database, cache. Show how a request flows through the system for the main use cases.

4

Deep Dive into Components

Pick the 1-2 most interesting/critical parts and go deeper — e.g., "how exactly do we generate a unique short code?" or "how does the database schema look?"

5

Identify Bottlenecks & Trade-offs

What breaks first as scale grows? What did we sacrifice (consistency, cost, complexity) for what we gained (speed, availability)? There's no perfect design — only good tradeoffs for the stated requirements.

✅ Functional Requirements

🔗 URL Shortener (like bit.ly / TinyURL)

Let's say you're building this. Someone pastes a long ugly link, you give them back something short like short.ly/aZ9kP2, and when anyone opens that short link, they land on the original page. That's the whole product. Let's build it the dumbest way possible first, and see what happens.

Version 1 — The Simplest Thing That Could Work

🏗️ The Naive Build

One server. One database table called urls with two columns: short_code and long_url. When someone creates a link, you generate a random 6-character code, save the row. When someone visits /aZ9kP2, you run SELECT long_url FROM urls WHERE short_code = 'aZ9kP2', and redirect them there. Done. You launch it.

🤔 Wait — does this actually work?

Yes! Genuinely. For a small project, a side tool, a few hundred users — this is completely fine. Don't let anyone tell you "you need Redis and microservices to build a URL shortener". You don't. The interesting part is what happens if it gets popular.

Version 1 Goes Viral — Now What?

📈 Six Months Later

Someone famous starts using your shortener in their social media bio. Suddenly your links are everywhere. You check your server dashboard one morning and your database CPU is sitting at 95%. Pages are loading slowly. You're getting emails saying "your links are broken / slow".

🤔 But it's just one tiny SELECT query — why would that be slow?

Because of HOW MANY TIMES it's happening. One query is nothing. But if your shortener now gets 4,000 redirect requests every single second, that's 4,000 identical-shaped database queries per second, all day, every day. Your database isn't doing anything complicated — it's just being asked the SAME question over and over, millions of times an hour, and every single one of those questions involves a disk lookup.

Let's actually put numbers on this, because the numbers are what tell you whether you have a problem at all — and they're what every system design conversation starts with.

Step 1 — What Does "Done" Even Mean? (Requirements)

FUNCTIONAL — what it DOES
  • User submits a long URL and gets back a short one
  • Visiting the short URL redirects to the original
  • (Optional) custom aliases, expiry dates, click counts
NON-FUNCTIONAL — how WELL it needs to do it
  • Redirects must basically never fail — that's the entire product
  • Redirects should feel instant (~50ms or less)
  • WAY more people click links than create them — this is the key fact that shapes everything
🤔 Why split requirements into these two boxes — isn't this just... a list of features?

Because they lead to completely different decisions. "Users can create short URLs" (functional) is satisfied by literally Version 1 above — a database table. But "redirects must feel instant even under heavy load" (non-functional) is the requirement that's currently BREAKING. The functional requirements tell you WHAT to build. The non-functional ones tell you WHERE it'll hurt.

💡 The "Aha" Moment

Look at that ratio: 4,000 reads per second vs only 40 writes per second — that's 100 reads for every 1 write. Your database isn't struggling because it's storing too much data (600GB total is tiny — any database handles that easily). It's struggling because it's getting hammered with the same type of read, over and over, far more often than anything is ever written.

🤔 Okay so... just get a bigger/faster database server?

You could — that's "vertical scaling" from Part 1, and it buys you time. But think about WHAT'S actually happening: the same handful of popular short links (that famous person's bio link) are being looked up thousands of times per second, and the database keeps re-finding the EXACT SAME ANSWER each time. A bigger database is still doing repeated, wasteful work. What if, instead, the most popular answers were just... sitting somewhere super fast, ready to hand out instantly, without bothering the database at all?

💡 That Idea Has a Name

That's a cache. You already met this concept in Part 1 — it's just memory that sits in front of your database and holds onto recently/frequently-asked-for answers. Once you add one, here's what changes: when someone visits /aZ9kP2, your app first asks the cache "do you know this one?" If yes (a cache hit) — instant answer, database never touched. If no (a cache miss) — ask the database, then ALSO save that answer in the cache so next time it's a hit.

Step 3 — The Design, Now That We Know Why Every Box Is There

🔗 URL Shortener — High-Level Architecture

Step 4 — A Different Question: How Do We Even MAKE "aZ9kP2"?

We've sorted out the read problem. But let's rewind to the write side for a second — when someone submits a URL, where does that random-looking 6-character code actually come from? This is a smaller problem, but it's a classic "wait, how do two different approaches even compare?" moment.
🤔 Could we just... number them? URL #1, #2, #3...?

You could, and that's exactly Approach 1 below — take the database's auto-increment number (1, 2, 3...) and convert it into a short string using more characters than just 0-9. That's literally all "Base62" means: instead of only digits 0-9 (base 10), use a-z, A-Z, AND 0-9 (62 symbols total), so the same number takes fewer characters to write down. It's the same trick as why hexadecimal colour codes are shorter than writing the number in decimal.

🤔 So what's the downside of just numbering them sequentially?

Two things. First — it's guessable. If short.ly/3 exists, someone can guess short.ly/4, /5, /6... and scrape through everyone's links. Second, and more subtly: if you ever run MULTIPLE servers handling URL creation at the same time, they all need to agree on "what's the next number?" — that shared counter becomes its own bottleneck and a single point of failure. The fix people usually reach for instead is Approach 2: just generate something random, and check the database "does this already exist? No? Great, use it."

🔢 Approach 1 — Base62 of an Auto-Increment Number

  • Database gives each new URL the next number: 1, 2, 3...
  • Convert that number to Base62 (a-z, A-Z, 0-9)
  • Number 125 → "21" — much shorter to write
  • 7 Base62 characters covers ~3.5 trillion codes

🎲 Approach 2 — Just Pick Something Random

  • Generate a random 7-character string
  • Check the database: does this code already exist?
  • If yes (extremely rare), try again
  • No shared counter needed — works fine across many servers
C# — Base62 encoding example
public static class Base62
{
    private const string Chars = "0123456789abcdefghijklmnopqrstuvwxyzABCDEFGHIJKLMNOPQRSTUVWXYZ";

    public static string Encode(long id)
    {
        if (id == 0) return "0";
        var sb = new StringBuilder();
        while (id > 0)
        {
            sb.Insert(0, Chars[(int)(id % 62)]);
            id /= 62;
        }
        return sb.ToString();
    }
}

// Usage: Base62.Encode(125) → "21"
// Database auto-increment ID 1,000,000,000 → Base62 → "15ftgG" (only 6 chars!)
💡 Which One Would You Actually Use?

For most real systems, Approach 2 (random + check) wins — mainly because of that "multiple servers" problem. Notice this is a SMALL decision compared to the caching one — it affects how URLs are CREATED (40/sec, remember?), not how they're READ (4,000/sec). That's worth remembering: not every decision in a system carries equal weight. Spend your thinking time on the high-traffic paths.

Step 4b — Where Does All This Actually Live? (Database Schema)

Just to make it concrete — here's literally what the database table looks like. Nothing fancy: one row per short URL, and the short_code is the primary key (which is also, conveniently, exactly what we look up on every redirect).

Step 6 — What Would Go Wrong NEXT?

We fixed the "4,000 reads/sec hammering the database" problem with a cache. Good systems design isn't "now it's perfect" — it's "now THIS is fixed, what breaks next?" Here's what an experienced engineer would immediately start worrying about:

⚠️ What Breaks First as Scale Grows

  • Cache too small? If you can't fit all the "popular" links in cache memory, you start getting cache misses again — you'd increase cache size or be smarter about what gets evicted
  • One MASSIVELY viral link — if literally everyone is hitting one single cache entry, even the cache server itself can get overwhelmed; the fix is having multiple copies of the cache (replication)
  • 301 vs 302 redirects: a 301 ("permanent redirect") lets the USER'S BROWSER remember it — meaning next time they don't even ask your server at all. Even less load! But then you can't count clicks anymore, since the browser skips you entirely

🔄 Decisions With No "Correct" Answer

  • SQL vs NoSQL: this data is just key→value (code→URL), so either works fine. NoSQL options tend to scale out to more servers more easily if you ever need that
  • 301 vs 302 again: it's a genuine trade — speed/cost (301) vs analytics (302). Pick based on what the business actually cares about
  • Custom aliases: a "nice to have" feature that adds a small cost — every write now needs to check "is this alias already taken?"
🤔 So... is this "done" now?

For the scale we estimated (4,000 reads/sec), yes — a cache in front of a database comfortably handles this. The honest answer in system design is always "done for THIS scale, with THESE trade-offs". If the numbers were 100x bigger, we'd be having a different conversation (probably about sharding the database, from Part 1). That's the whole game: match the solution to the actual numbers, not to what sounds impressive.

🚦 Rate Limiter Service

Imagine you run a public API — maybe a weather API, maybe something you built for other developers to use. It's free to use, and it's great... until one developer accidentally writes a loop with no delay, and suddenly their app is hitting your API 10,000 times per second. Your servers fall over, and now EVERYONE'S requests are failing — not just theirs.
🤔 Can't we just tell that one developer to stop?

Sure, after the fact — but by then the damage is done. What you actually want is something that says, automatically, in real-time: "you've made 100 requests in the last minute, that's your limit, come back in a bit" — BEFORE it ever reaches your actual servers. That's a rate limiter. It's less of a "system" on its own and more of a bouncer standing in front of your API.

Step 1 — What Are We Actually Asking This Bouncer To Do?

FUNCTIONAL
  • For every request, check: has this client made too many requests recently?
  • If over the limit → reject with 429 Too Many Requests
  • Different limits for different people (free users vs paying users)
NON-FUNCTIONAL
  • This check happens on EVERY single request — it must be nearly instant
  • If you run multiple servers, they all need to agree on "how many requests has this client made?"
🤔 That second non-functional point — why is it tricky?

Picture this: you have 5 servers behind a load balancer. A client makes 100 requests. The load balancer spreads them roughly evenly — 20 requests to each server. If each server keeps its OWN little counter in its own memory, each one thinks "this client has only made 20 requests, that's fine!" — and lets them all through. The client just made 100 requests despite a limit of, say, 80. The counter needs to be SHARED across all servers.

Step 2 — How Often Does This Check Even Happen?

50,000
Requests / second (peak)
1M
Active API keys
~1ms
Budget for this check
🤔 So... could we just store each client's count in our normal SQL database, shared by all servers?

Technically yes — that solves the "shared" problem. But think about the cost: 50,000 times PER SECOND, every server needs to do "read the current count, add 1, write it back, compare to the limit". A regular database, designed for durability (safely writing to disk), is just not built to be hammered at that rate for something this trivial. We need something that lives in memory, is shared across servers, and can do "add 1 and tell me the new total" as ONE instant operation.

💡 Enter Redis

This is the textbook use case for Redis — an in-memory data store that every server can connect to, and which has a built-in INCREMENT command that's "atomic" (meaning: even if two servers call it at the EXACT same microsecond, Redis handles them one at a time correctly — no double-counting). Each client gets a tiny counter in Redis. Every request increments it. If the result is over the limit, reject.

Step 3 — Putting It in Front of the API

🚦 Rate Limiter — Sits Between Client and API
🤔 What's that "TTL = 60s" thing in the Redis box?

TTL = "Time To Live". We're setting each counter to automatically delete itself after 60 seconds. That's how the limit "resets" — without us writing any cleanup code. Client makes their first request → counter created, set to expire in 60s → counter increments with each request → after 60 seconds with no NEW counter, Redis just throws it away → next request starts a fresh counter at 1. This gives us a rolling "100 requests per minute" limit almost for free.

Step 4 — Let's See the Actual Code

This is exactly the conversation above, turned into C#. Notice it's genuinely short — the "increment, check, set expiry once" logic is maybe 10 lines. The complexity in rate limiting is less about the code and more about the THINKING we just did.

C# — Distributed rate limiter using Redis
public class RedisRateLimiter
{
    private readonly IConnectionMultiplexer _redis;
    private const int LimitPerWindow = 100;
    private static readonly TimeSpan Window = TimeSpan.FromMinutes(1);

    public RedisRateLimiter(IConnectionMultiplexer redis) => _redis = redis;

    public async Task<RateLimitResult> CheckAsync(string clientId)
    {
        var db  = _redis.GetDatabase();
        var key = $"ratelimit:{clientId}";

        // Atomically increment the counter — this is the "shared" part
        long count = await db.StringIncrementAsync(key);

        // First request in this window — set it to expire in 60s
        if (count == 1)
            await db.KeyExpireAsync(key, Window);

        var ttl = await db.KeyTimeToLiveAsync(key) ?? Window;

        return new RateLimitResult
        {
            Allowed   = count <= LimitPerWindow,
            Limit     = LimitPerWindow,
            Remaining = Math.Max(0, LimitPerWindow - count),
            ResetIn   = ttl
        };
    }
}

public class RateLimitResult
{
    public bool     Allowed   { get; set; }
    public long     Limit     { get; set; }
    public long     Remaining { get; set; }
    public TimeSpan ResetIn   { get; set; }
}

// Middleware usage — this runs before every request reaches the real API:
var result = await rateLimiter.CheckAsync(apiKey);
if (!result.Allowed)
{
    context.Response.StatusCode = 429;
    context.Response.Headers["X-RateLimit-Limit"]     = result.Limit.ToString();
    context.Response.Headers["X-RateLimit-Remaining"] = "0";
    context.Response.Headers["Retry-After"]           = result.ResetIn.TotalSeconds.ToString("0");
    return;
}
await next();  // continue to the backend

Step 6 — One Important "What If"

Here's a question that trips a lot of people up: what happens to your ENTIRE API if Redis — this thing we just made critical to every single request — goes down for a minute?

🅰️ "Fail Closed" — Reject Everything

If we can't check the limit, assume the worst and block all requests. Your rate limiter just took down your whole API because of a tiny outage in a side component. Safer for abuse, terrible for uptime.

🅱️ "Fail Open" — Allow Everything

If we can't check the limit, just let requests through (and alert someone). For the brief duration of the Redis blip, you're temporarily unprotected — but your core API stays up.

💡 What Most Real Systems Choose

Fail open, with alerting. The reasoning: a rate limiter is a PROTECTION mechanism — it shouldn't become a bigger outage risk than the thing it's protecting against. A short window of "unprotected" is usually far less damaging than your entire API returning errors to every legitimate user.

⚠️ Other Things That Break First

  • One client's traffic overwhelms a single Redis key — even though Redis is fast, if ONE client is making 10,000 req/sec, that single counter key gets very "hot"
  • Clock differences between servers can cause small inconsistencies in exactly when a window resets — usually not worth worrying about for most apps

🔄 Recap of the Big Decision

  • The entire design exists because of ONE constraint: many servers need to agree on ONE shared count, fast
  • Redis + atomic increment + TTL solves exactly that, in about 10 lines of code

📋 Pastebin (Text/Code Sharing Service)

You're a developer and you want to share a 200-line error log with a colleague on Slack. Pasting it directly makes the chat unreadable. So instead, you paste it into a tool, hit submit, and get back a link: paste.ly/x7Bq2. You send THAT link instead. Your colleague clicks it, sees the nicely formatted text. That's a pastebin.
🤔 Isn't this basically the URL shortener again?

Good instinct — it RHYMES with the URL shortener (short code → "thing"), but there's one big difference. The URL shortener stores a tiny string (a URL, maybe 100 characters) and just redirects. A pastebin needs to store the actual CONTENT — which could be 50KB of log output, or a 2MB code dump. That difference in size changes where we should store it.

Step 1 — What's the Job Here?

FUNCTIONAL
  • User submits text, gets back a shareable link
  • Anyone with the link can view it
  • Optional: the paste auto-deletes after a set time (1 day, 1 week, etc.)
NON-FUNCTIONAL
  • Content can be sizeable — KBs up to a few MB
  • Once created, a paste shouldn't just vanish (unless it was meant to expire)

Step 2 — Let's Just Try the Obvious Thing First

1M
New pastes / day
10KB
Average paste size
~10GB
New data / day
🏗️ The Naive Build

One table: id, content, created_at, expires_at. The content column is a big text field. Just like the URL shortener — generate an ID, save the row, done.

🤔 So what's actually wrong with that? 10GB/day doesn't sound huge.

It's not WRONG exactly — it would work. But here's the thing: databases are optimized for small, structured rows that you search, filter, and join. They're not great at being a dumping ground for big blobs of text. Every time the database backs itself up, replicates to another server, or runs a query that happens to scan this table, it's now dragging around megabytes of raw text content that it doesn't need to "understand" at all — it's just dead weight. There's a much better home for "just store these bytes and give them back later, unchanged".

💡 Split It Into Two Different Jobs

Think about what we ACTUALLY need to look up vs what we just need to STORE. We need to look up: "does paste x7Bq2 exist? Is it expired? What language is it?" — small, structured facts. That's a database's job. We need to store: the actual 50KB of text — we never search inside it, never filter by it, just fetch the whole thing and hand it back. That's a job for object storage (think: a giant filing cabinet of files, identified by name) — the same kind of storage that holds images and videos on the internet.

Step 3 — Two Storage Systems, One Request

📋 Pastebin — Metadata in DB, Content in Object Storage
🤔 Isn't two lookups slower than one?

Marginally, yes — but both are fast (a tiny database row lookup, then a direct file fetch by exact name — no searching involved). What we get in return: the database stays small and fast (it's only ever storing tiny rows of metadata, even if you have BILLIONS of pastes), and object storage is purpose-built to serve files of any size cheaply and reliably. We're using each tool for what it's actually good at.

Step 4 — What About Pastes That Expire?

Someone creates a paste and picks "delete after 1 day". A day passes. What ACTUALLY makes it disappear?
🤔 Does something need to actively go delete it?

There are two options. Option A: write a background job that runs every hour, asks the database "find me everything where expires_at is in the past", and deletes those rows AND their files. Option B (much less work): most object storage services have a built-in "lifecycle rule" — you literally just configure "delete files in this folder after N days" and the CLOUD does it for you, no code required. The schema below supports either approach.

SQL — Metadata table + expiration cleanup
CREATE TABLE pastes (
    id            VARCHAR(10)  PRIMARY KEY,        -- "x7Bq2"
    storage_path  VARCHAR(255) NOT NULL,        -- "pastes/x7Bq2.txt"
    language      VARCHAR(30)  NULL,            -- "csharp", "python", null = plain
    size_bytes    INT          NOT NULL,
    created_at    TIMESTAMP    NOT NULL DEFAULT NOW(),
    expires_at    TIMESTAMP    NULL,             -- NULL = never expires
    view_count    BIGINT       NOT NULL DEFAULT 0
);

-- Index for the cleanup job to find expired pastes efficiently
CREATE INDEX idx_expires_at ON pastes(expires_at)
    WHERE expires_at IS NOT NULL;

-- Option A: a background job runs periodically (e.g., every hour):
DELETE FROM pastes WHERE expires_at IS NOT NULL AND expires_at < NOW();
-- (plus delete the matching file from object storage)
-- Option B: configure the object storage bucket to auto-delete old files
-- — much less code, but you'd still tidy up the small metadata rows separately
C# — Create and retrieve a paste (the two-step dance from the diagram)
public class PasteService
{
    private readonly IObjectStorage _storage;   // abstracts S3 / Blob / GCS
    private readonly IPasteRepository _repo;

    public async Task<string> CreatePasteAsync(string content, string? language, TimeSpan? ttl)
    {
        var id   = GenerateShortId();              // same idea as the URL shortener
        var path = $"pastes/{id}.txt";

        // 1. Put the actual text in object storage
        await _storage.UploadAsync(path, content);

        // 2. Put the FACTS about it in the database
        await _repo.AddAsync(new PasteMetadata
        {
            Id          = id,
            StoragePath = path,
            Language    = language,
            SizeBytes   = Encoding.UTF8.GetByteCount(content),
            ExpiresAt   = ttl.HasValue ? DateTime.UtcNow.Add(ttl.Value) : (DateTime?)null
        });

        return id;
    }

    public async Task<string?> GetPasteContentAsync(string id)
    {
        var meta = await _repo.GetByIdAsync(id);

        // Check it exists and hasn't expired — cheap, since it's just one small row
        if (meta == null || (meta.ExpiresAt.HasValue && meta.ExpiresAt < DateTime.UtcNow))
            return null;

        await _repo.IncrementViewCountAsync(id);
        return await _storage.DownloadAsync(meta.StoragePath);  // fetch the actual text
    }
}

Step 6 — What's Next to Worry About?

⚠️ What Breaks First

  • Someone pastes a 50MB file — without a size limit, one user could dominate your storage costs or slow down the page trying to render it
  • A paste goes viral (e.g., shared on a forum) — suddenly thousands of people fetch the SAME object storage file; a CDN in front absorbs this, same trick as the URL shortener's cache
  • Abuse — people using your free pastebin to host spam or malware links

🔄 A Genuinely Interesting Trade-off

  • What if MOST pastes are tiny (a one-line error message)? Then the "two storage systems" approach adds an extra network hop for the common case. Some real systems handle this with a hybrid: pastes under ~1KB get stored directly IN the database row (cheap, one lookup); bigger ones go to object storage. Neither approach is "wrong" — it depends on your actual usage pattern

🅿️ Parking Lot Management System

Here's one that surprises a lot of people: designing a parking lot system is a popular system design question — and its difficulty has nothing to do with scale. There are maybe 2,000 parking spots and a few hundred cars coming and going per day. That's almost nothing. The tricky part is something completely different.
🤔 If scale isn't the problem, what IS the problem?

Picture this: it's 8:58am on a Monday morning, rush hour. Ten cars are approaching the ten different entry gates of the car park, all at the same time. Each gate's computer asks the system "is there a free car spot?" All ten get back "yes, spot 42 is free." All ten try to assign spot 42. Now ten cars think they have the same parking spot. That's not a scale problem — it's a correctness under concurrency problem. Two things trying to grab the same resource at the same moment.

Step 1 — What Does This System Need to Do?

FUNCTIONAL
  • Car arrives → find a free spot (right size for the vehicle) → print a ticket → mark spot as taken
  • Car leaves → scan ticket → calculate how long they stayed → charge them → free the spot
  • Display boards at the entrance show "Floor 2: 14 spaces available"
  • Support different vehicle sizes — motorcycle, car, large/van (they don't fit in each other's spots)
NON-FUNCTIONAL
  • Never assign the same spot to two vehicles — this is THE requirement that shapes everything
  • Traffic is tiny by web standards — a few hundred cars per day, not millions of requests per second

0.06 events per second means one event every 17 seconds on average. We are NOT building for massive traffic — we're building for correctness. No need for caching layers, CDNs, horizontal scaling, sharding, or any of the big-scale tricks. We need one reliable database that handles concurrent requests correctly. That's it.

🅿️ Parking Lot — Entities and How They Connect

Step 4 — Solving the "Two Cars, Same Spot" Problem

Back to that Monday morning rush. Ten gates simultaneously asking "give me a free car spot." Here's why the naive approach fails and why the fix is actually just one SQL keyword.
🤔 Why does "SELECT then UPDATE" fail?

Between the moment you SELECT a free spot and the moment you UPDATE it to "occupied", another gate's request can slip in and SELECT the SAME spot. They're reading the same "free" status at nearly the same instant. This tiny gap between read and write is where the bug lives. The fix: do it in one atomic operation so there's no gap for anyone to slip through.

SQL — Why two steps fails, and how one step fixes it
-- ❌ THE BUG: Two separate steps
-- Gate 1 runs this: SELECT spot_id FROM spots WHERE type='Car' AND is_occupied=false LIMIT 1
-- Gate 2 runs this AT THE SAME TIME: gets the SAME spot_id (both see it as free)
-- Gate 1 runs: UPDATE spots SET is_occupied=true WHERE spot_id = 42
-- Gate 2 runs: UPDATE spots SET is_occupied=true WHERE spot_id = 42
-- Result: both gates think they own spot 42. Two cars, one spot. Chaos.

-- ✅ THE FIX: One atomic operation — no gap to slip through
-- FOR UPDATE SKIP LOCKED means: "lock this row for me, and if it's
-- already locked by someone else, just skip it and find the next one"
-- The database engine handles this correctly even with many concurrent requests
UPDATE spots
SET is_occupied = true, occupied_by_ticket = @ticketId
WHERE spot_id = (
    SELECT spot_id FROM spots
    WHERE type = @vehicleType AND is_occupied = false
    LIMIT 1
    FOR UPDATE SKIP LOCKED
)
RETURNING spot_id;
-- If no rows returned: the lot is full for this vehicle type
C# — Entry and exit, built on the atomic SQL above
public class ParkingService
{
    public async Task<Ticket> VehicleEntryAsync(string licensePlate, VehicleType type)
    {
        // This calls the atomic SQL above — either gets a spot or lot is full
        var spotId = await _repo.AssignFreeSpotAsync(type)
            ?? throw new ParkingFullException(type);

        var ticket = new Ticket
        {
            TicketId     = Guid.NewGuid(),
            LicensePlate = licensePlate,
            SpotId       = spotId,
            EntryTime    = DateTime.UtcNow
        };
        await _repo.SaveTicketAsync(ticket);
        return ticket;
    }

    public async Task<decimal> VehicleExitAsync(Guid ticketId)
    {
        var ticket = await _repo.GetTicketAsync(ticketId)
            ?? throw new TicketNotFoundException();

        ticket.ExitTime = DateTime.UtcNow;
        var duration = ticket.ExitTime.Value - ticket.EntryTime;
        ticket.Fee = CalculateFee(duration);

        // Free the spot + save the completed ticket as ONE transaction
        await _repo.FreeSpotAndCloseTicketAsync(ticket);
        return ticket.Fee;
    }

    private decimal CalculateFee(TimeSpan duration)
        => (decimal)Math.Ceiling(duration.TotalHours) * 2.50m;  // £2.50/hour, rounded up
}
🤔 Why PostgreSQL specifically? Can't we use any database?

FOR UPDATE SKIP LOCKED is a feature that exists in PostgreSQL and MySQL 8+, but NOT in all databases. This is a rare case where the choice of specific database technology matters — we're relying on a specific concurrency feature. Most SQL databases support some form of row locking, but the "skip locked" part (which lets other gates find the NEXT available spot rather than waiting in a queue) is what makes this work cleanly for parallel entry gates.

Step 6 — Edge Cases Worth Thinking About

⚠️ What Happens If the Database Goes Down?

A gate with no database connection can either: block all entry (safe but frustrating), or let cars in and reconcile later (risky but keeps moving). Physical car parks usually prefer keeping the barrier UP — people can always pay when they exit. Software systems rarely have this "physical world" fallback, which is what makes this problem interesting.

🤔 "What if someone loses their ticket?"

Store the license plate at entry. Gate cameras can also read plates. On exit, look up by plate if they have no ticket. Charge them from the entry time recorded in the database. This "lost ticket" flow is great OOP design practice — it's a different code path that reuses most of the same logic.

💡 The Big Lesson From This Example

Not every system design problem is about handling millions of users. Sometimes the interesting challenge is correctness when things happen at the same time — which can be a problem even with just 10 simultaneous gates. Recognising WHICH type of problem you have (scale vs correctness) is itself an important system design skill.

💬 Chat Application (1-to-1 Messaging)

Every app we've built so far works the same way: the USER asks, the SERVER answers. You click, it responds. But a chat app has a completely different problem: when Alice sends Bob a message, Bob's screen needs to update WITHOUT Bob doing anything. Bob didn't click anything. Bob didn't ask for anything. The server needs to push information to Bob's browser/phone uninvited. How does that work?
🤔 Can't Bob's phone just... ask "any new messages?" every second?

Yes! That's called polling, and it's the simple, naive approach. Bob's phone sends "any new messages?" to the server every second. If there are, great. If not, the server says "nope" and the phone asks again in a second. It works, and some apps actually do this. The problems: every user sends one request per second 24/7 — even at 3am when nobody is chatting. That's a lot of "nope" responses. Also, there's up to a 1-second delay before a message appears. For a chat app that should feel instant, that's noticeable.

💡 The Better Idea — Keep the Connection Open

What if instead of Bob repeatedly calling the server, the server just holds a persistent connection open to Bob's phone? When Alice's message arrives, the server immediately pushes it down that already-open pipe. No polling. No delay. No wasted "nope" requests. This is called a WebSocket — a special kind of connection that stays open and lets the server talk first.

Step 1 — What Does a Chat App Actually Need?

FUNCTIONAL
  • Alice sends a message → Bob sees it appear in real-time
  • When Bob opens a conversation, he sees the history of past messages
  • Show whether Bob is online or offline
NON-FUNCTIONAL
  • Messages should arrive in under ~1 second
  • If Bob is offline when Alice sends a message, he should get it when he comes back online
  • At scale: millions of users with persistent open connections simultaneously

🤔 2 million open connections — doesn't that mean 2 million servers?

No — a single server can hold tens of thousands of WebSocket connections simultaneously (they're lightweight when no messages are actively flowing). But 2 million connections needs maybe 50-200 servers, each holding some slice of the connections. This creates a new problem: when Alice sends a message, her connection might be on Server A, but Bob's connection might be on Server B. Server A needs to tell Server B "hey, forward this to Bob." The servers need to coordinate.

Step 3 — How Messages Get from Alice to Bob

💬 Chat App — WebSocket Connections and Message Routing
🤔 What's that "Presence Store" box — why do we need ANOTHER Redis?

When Alice's message arrives at Server 1, that server needs to know: "which of my 50 other servers is currently holding Bob's WebSocket connection?" That's the Presence Store — a shared Redis map where each server registers its users: "Bob is connected to me, Server 2." When Alice's message needs to reach Bob, the Message Service does a fast Redis lookup: "where is Bob?" → "Server 2" → forward the message there. Without this, Server 1 would have no way to find Bob.

Step 4 — Storing Messages So History Loads Fast

SQL — Messages table, built for "load last 50 messages"
CREATE TABLE messages (
    message_id      BIGINT      PRIMARY KEY,   -- Snowflake ID: encodes timestamp + server ID
    conversation_id BIGINT      NOT NULL,      -- links all messages between two users
    sender_id       BIGINT      NOT NULL,
    content         TEXT        NOT NULL,
    sent_at         TIMESTAMP   NOT NULL DEFAULT NOW(),
    status          VARCHAR(10) NOT NULL DEFAULT 'sent'
);

-- The one query that matters: "give me the last 50 messages in conversation X"
-- Without this index, a table with billions of rows would be painfully slow
CREATE INDEX idx_conversation_time
    ON messages(conversation_id, sent_at DESC);

-- Why "Snowflake ID" instead of auto-increment?
-- An auto-increment counter needs a central coordinator (single point of failure).
-- A Snowflake ID packs timestamp + server ID + sequence into one number —
-- any server can generate unique, roughly-time-ordered IDs independently.
-- This is how Twitter, Discord, and many others handle message IDs at scale.

Step 6 — What Gets Hard Next

⚠️ When One Server Restarts

If WS Server 2 restarts, all connections it was holding drop simultaneously. Every affected user's app reconnects at the same moment — a "reconnection storm" that briefly spikes load on other servers. Good clients use exponential backoff (retry after 1s, then 2s, then 4s...) rather than all hammering at once.

🔄 Group Chats Multiply Everything

A group with 500 members means one message needs to be delivered to up to 500 WebSocket connections, potentially spread across many servers. The fan-out here mirrors the Twitter problem — at large scale you need to think carefully about whether to push to all 500 immediately or batch it.

🔔 Notification System

Black Friday. It's 9:00am. Your e-commerce platform just launched its biggest sale. The marketing team clicks "Send" on a notification: "BLACK FRIDAY — 70% OFF EVERYTHING". That one click needs to send emails, SMS messages, and push notifications to 5 million users. Right now. All at once. Your app server got one click. What happens next?
🤔 Just... send them all from the web server that received the click?

Let's think through what that looks like. Your web server receives the click and starts calling your email provider's API — 5 million times. Each email API call takes maybe 100ms. That's 5 million × 100ms = 500,000 seconds. One server, sending serially, would finish in about 5.7 DAYS. By which point Black Friday is long over. Even with 100 parallel connections: still 83 hours. And while the server is doing all this, it's completely unresponsive to other requests. The whole site is effectively down.

💡 The Insight — Uncouple "Decide to Send" from "Actually Sending"

The marketing team's click should take milliseconds — just enough to say "yes, got it, will send." The actual sending of 5 million notifications should happen in the BACKGROUND, spread across many workers, over the next few minutes. A message queue is the bridge: the click drops 5 million "send this notification" jobs into the queue instantly, and a fleet of worker processes drain the queue steadily. The click returns immediately. Workers do the heavy lifting asynchronously.

Step 1 — What Does This System Need to Do?

FUNCTIONAL
  • Other parts of the app say "notify user X about Y" — notification system handles the rest
  • Sends via email, SMS, push notification, or in-app — depending on what's relevant
  • Respects user preferences ("I don't want marketing SMS")
  • Uses templates so "Your order shipped!" looks the same everywhere
NON-FUNCTIONAL
  • The system that REQUESTS the notification should never have to wait for it to be DELIVERED
  • Must handle massive bursts — 5 million notifications from one campaign click
  • Never send the same notification twice, even if the requesting system retries
  • If the email provider is temporarily down, retry — don't just drop the notification

🤔 That's nearly a 100x difference between average and peak — how do you even plan for that?

You DON'T build infrastructure for the peak, because that would mean paying for 100x your normal capacity just to handle occasional bursts. Instead, you build a queue that absorbs the spike. Think of it like a theme park queue: when a show ends and 500 people all want hot dogs at once, the hot dog stand doesn't magically get 500x faster — people just wait in a queue. The stand works at a steady pace. The queue handles the burst. Your notification workers work at whatever steady rate they can handle, and during a spike the queue temporarily grows rather than anything breaking.

Step 3 — The Queue-Based Fan-Out Architecture

🔔 Notification System — Queue Absorbs the Burst, Workers Process Steadily
🤔 What's a "Dead Letter Queue"?

Imagine you try to send an email, it fails. You retry. Fails again. You retry 3 more times. Still failing. At some point you have to stop retrying — but you also don't want to just silently throw away the notification. A Dead Letter Queue (DLQ) is where failed messages go after they've exhausted their retries. An engineer can then look at the DLQ, understand WHY they failed (wrong email address? provider outage?), and decide whether to replay them or discard them. It's a safety net — nothing silently disappears.

Step 4 — The "Send It Twice" Problem (Idempotency)

OrderService sends "notify user about shipment." The notification API receives it, enqueues the job, starts to respond... network blip. OrderService didn't get the response, so it assumes it failed and tries again. Now the notification is in the queue TWICE. User gets two "Your order shipped!" emails. Annoying and looks broken.
C# — Stopping duplicates with an idempotency key
public record NotificationRequest(
    string IdempotencyKey,   // caller provides: "order-shipped-{orderId}" — unique per event
    int    UserId,
    string TemplateName,     // "OrderShipped"
    Dictionary<string, string> TemplateData  // { "OrderId": "42", "TrackingUrl": "..." }
);

public class NotificationApi
{
    public async Task RequestNotificationAsync(NotificationRequest req)
    {
        // 1. Has this EXACT notification already been accepted before?
        //    Check using the caller-provided idempotency key in Redis (fast lookup)
        var alreadySeen = await _idempotencyStore.ExistsAsync(req.IdempotencyKey);
        if (alreadySeen)
            return;   // silently ignore the duplicate — already handled

        // Mark as seen (expires after 7 days — after that, retrying is fine)
        await _idempotencyStore.SetAsync(req.IdempotencyKey, TimeSpan.FromDays(7));

        // 2. Check user preferences — what channels do they allow for this type?
        var prefs    = await _preferencesRepo.GetAsync(req.UserId);
        var channels = prefs.GetEnabledChannels(req.TemplateName);

        // 3. Enqueue one job per enabled channel — each worker handles its own channel
        foreach (var channel in channels)
            await _queue.PublishAsync(new ChannelNotification(req.UserId, channel, req.TemplateName, req.TemplateData));
    }
}

// Throwing an exception causes the queue to redeliver (up to N retries),
// then the message moves to the Dead Letter Queue
🤔 Why use templates instead of just passing in the message text directly?

If OrderService hardcodes "Your order has shipped!" in its code, and marketing wants to change the wording, a developer has to update OrderService and redeploy. With templates, the wording lives in a separate template store. Marketing updates the template. OrderService just says "use the OrderShipped template with these values", and automatically gets the new wording. Decoupled — each team owns their piece.

Step 6 — What Gets Complicated

⚠️ Your Email Provider Has Its Own Limits

Even if your queue can drain 50,000/sec, your email provider (SES, SendGrid) has its own rate limit and daily sending quota. You can't just hammer them. Workers need to respect the provider's limits — this is often called "respecting backpressure from downstream."

🔄 One Queue or One Per Channel?

If email and SMS share one queue and the SMS provider goes down, a growing SMS backlog blocks email jobs too. Separate queues per channel (one for email, one for SMS, one for push) means a problem with one channel doesn't affect others.
