🚀 What Changes When You Go from Beginner to Advanced?

In Part 2, every "breaking point" had a simple fix: add a cache, add a queue, add an index. That's because the numbers were big enough to be interesting but small enough that a single well-chosen tool solved the whole problem. The six systems in this part operate one or two orders of magnitude higher — and at that altitude, some of those "nice to have" tricks from Part 2 stop being optional. Let's see exactly where that line gets crossed.
👶 Start Here — Same Framework, Bigger Numbers

Good news: the 5-step framework from Part 2 (Requirements → Estimation → High-Level Design → Deep Dive → Trade-offs) still applies exactly the same way. What changes is the scale — and at massive scale, problems that were "nice to have" solutions in Part 2 become absolutely mandatory. A cache was an optimisation for the URL shortener; for Netflix, without a CDN the system simply could not function at all.

The Big Shifts at Massive Scale

📈 Beginner Scale (Part 2)

  • Single database often sufficient
  • One cache layer (Redis) covers most needs
  • A handful of services, simple architecture
  • Storage measured in GB-TB
  • "What if this breaks?" is mostly theoretical

🌍 Advanced Scale (This Part)

  • Sharding and multi-region databases are mandatory
  • Multiple specialised caching layers, CDN as core infrastructure
  • Dozens to hundreds of microservices
  • Storage measured in PB-EB (petabytes to exabytes)
  • Failure is CONSTANT — designing for failure is the default mindset

New Concepts That Appear at This Scale

🌎

Multi-Region

Data and services replicated across geographic regions for latency and disaster recovery.

📊

Data Pipelines

Streaming/batch pipelines process huge volumes of events for analytics, ML, recommendations.

🧮

Consistent Hashing

Distributes data/load across many nodes while minimising redistribution when nodes are added/removed.

🗺️

Geospatial Indexing

Efficient "find things near this location" queries at huge scale (geohashing, quadtrees).

🎬

Pre-computation

Expensive results computed ahead of time and cached/stored, rather than calculated per-request.

🔥

Hotspot Mitigation

Strategies for when one piece of data (viral post, celebrity) gets disproportionate traffic.

🎬 Netflix — Video Streaming Platform

Picture the absolute simplest version of "stream a video over the internet": you put an MP4 file on a server, and when someone wants to watch it, the server sends them the bytes. This actually works — that's literally how a lot of small video sites function. So why does Netflix need thousands of engineers and one of the most sophisticated content delivery setups on Earth to do basically the same thing, just bigger?
🤔 Couldn't you just... add more powerful servers as more people watch?

Let's actually run the numbers, because this is where it gets interesting. Picture a hit show drops a new season on a Friday night. Within the hour, 35 million people hit play — at the same time, in living rooms scattered across six continents. Even if you built the most powerful single data center conceivable, you'd hit a wall that has NOTHING to do with server power: the speed of light. A request from Australia to a server in Virginia takes time just to travel the distance — physics, not computing power. No amount of "bigger server" fixes a problem that's fundamentally about geography.

💡 The Real Insight — Move the Video, Not the Viewer

If the problem is distance, the fix is obvious once you see it: put COPIES of the video physically close to wherever people are watching. A viewer in Sydney should stream from a server in Sydney, not Virginia. This is a CDN (Content Delivery Network) — and at Netflix's scale, it's not a performance optimization bolted on later, it's the entire foundation the system is built around. Without it, the product simply doesn't work, full stop.

Step 1 — What's Actually Required Here?

FUNCTIONAL
  • Users browse a catalogue and press play on something
  • Video quality automatically adapts to however fast (or slow) the viewer's internet is
  • Pause on your phone, resume on your TV, from the exact same spot
  • Studios upload new content, which gets prepared for streaming behind the scenes
NON-FUNCTIONAL
  • Pressing play should feel instant — under 2 seconds to start, and no annoying buffering mid-watch
  • Must survive MASSIVE simultaneous spikes (a hit show's release night)
  • Storing and moving video is the single biggest cost in the whole system — efficiency really matters here

🤔 "Petabytes per day" — is that actually a database/server problem, or something else?

It's genuinely a different KIND of problem than anything in Part 2. A database problem is about organizing and looking up data quickly. This is purely about MOVING an enormous volume of bytes across the physical internet to millions of separate locations, continuously, in real-time, without interruption. There's no clever query or index that fixes "the data has to physically travel thousands of miles." The only lever you actually have is: reduce the distance.

🎬 Netflix — Upload Pipeline + Playback Path

Step 4 — "Why Not Just Resize the Video on the Fly?"

Here's the next question worth slowing down on. The viewer's connection speed varies — sometimes great wifi, sometimes a shaky phone connection on a train. Netflix needs to serve different quality levels to different people, even the SAME person at different moments. Obvious first idea: when someone requests the video, check their current speed and resize/compress it for them in that instant.
🤔 Why doesn't Netflix just transcode the video in real-time when you press play?

Because transcoding (converting video to a different resolution/bitrate) is computationally expensive — it can take LONGER than the video itself to process, even on powerful hardware. Now multiply that by 35 million simultaneous viewers, each potentially needing a different quality at different moments as their connection fluctuates. Doing this live, per-viewer, would require a genuinely absurd amount of compute, running constantly, just to reproduce work that's identical for millions of people watching the same movie.

💡 Do the Expensive Work Once, Reuse It Forever

So instead: when a video is uploaded, it gets converted into EVERY quality level it'll ever need — 240p, 480p, 720p, 1080p, 4K — just once, ahead of time. This is pre-computation, the same principle as caching but applied to processing instead of just storage. Now serving any viewer at any quality is just: hand them the already-prepared file. No live computation, ever, for playback.

🤔 Okay but how does the video player actually SWITCH quality mid-stream, smoothly, without restarting?

This is the other half of the trick: each quality version isn't stored as one giant file — it's chopped into small chunks, typically 6-10 seconds each. The player downloads chunk after chunk, and before requesting the NEXT chunk, it checks "how fast did the last one download?" If your connection just got slow, it requests the next chunk from the LOWER quality version instead. Because every quality level is split at the exact same chunk boundaries, switching is invisible — you just get a slightly blurrier or sharper next 8 seconds, never a stutter or restart.

Conceptual — HLS manifest (simplified)
#EXTM3U
# Master playlist — lists available quality levels

#EXT-X-STREAM-INF:BANDWIDTH=2800000,RESOLUTION=1280x720

Step 4b — "Because You Watched..." — Where Do Those Come From?

🤔 Surely recommendations have to be calculated live based on what I just watched?

You'd think so, but the same "do it once, reuse it" logic applies here too. Computing a personalized ranking for 230 million people, on-demand, every single time someone opens the app, would be enormously expensive computation happening constantly. Instead, a background pipeline runs periodically (think: overnight) churning through everyone's viewing history through ML models, and writes out the RESULT — "here's the row order for User X's homepage" — to a fast lookup store. When you open the app, it's just reading a pre-written answer, not calculating one live. The trade-off: your recommendations are slightly stale (a day old, typically), which is a price almost nobody notices paying.

Step 6 — What's Genuinely Hard Here

⚠️ What Breaks First

  • A brand new release — there's no "popularity built up over time" to rely on; the CDN needs the content pushed to edge locations BEFORE release, predicted in advance, not reactively cached after the first few requests
  • CDN cache misses are expensive — every time a viewer's request can't be served from a nearby edge and has to go back to origin storage, that's real bandwidth cost; hit ratio is one of the most closely watched numbers in this whole system

🔄 A Trade-off Worth Sitting With

  • Pre-transcoding into ~5+ quality levels means storing roughly 5x the data of just the original file. That's a real cost. But the alternative — transcoding on-demand per viewer — would cost vastly more in compute, repeated millions of times for the exact same outcome. Paying more for storage to save drastically more on compute is the calculation being made here

🐦 Twitter / X — News Feed System

You post a tweet. It needs to show up in the home timeline of everyone who follows you, almost instantly. For most people that's no big deal — a few hundred followers, easy. Now picture a celebrity account with 100 million followers posting something. That ONE tweet needs to somehow reach 100 million different people's feeds. Let's think through how that actually could work.
🤔 Why not just, when someone opens the app, look up who they follow and grab their recent tweets right then?

That's actually a completely reasonable approach, and it's called "fan-out on READ" — do the work of assembling the timeline at the moment someone asks for it. The problem appears when you check the numbers: this system gets 500,000 timeline reads per second, but only 6,000 tweet writes per second. That's a 80:1 ratio. If every single one of those 500,000 reads-per-second has to go fetch-and-merge tweets from potentially hundreds of people someone follows, your read path is doing an enormous amount of repeated work, constantly, for the most frequent operation in the entire system.

🤔 Okay, so flip it — do the work when someone TWEETS instead, so reading is instant?

That's "fan-out on WRITE" — when you post a tweet, immediately go push it into the pre-built timeline of every single follower. Now reading your timeline really is just one fast lookup. This works great... until that celebrity with 100 million followers posts something. Now ONE tweet write needs to trigger 100 million separate timeline updates. That single tweet could take minutes to fully propagate, and it'd hammer your infrastructure every time anyone famous posts anything.

💡 Neither Extreme Works Alone — So Don't Pick Just One

Here's the move: use fan-out-on-WRITE for everyone with a normal-sized following (fast reads, and the write cost is small — pushing to 1,000 followers is nothing). But for celebrities — anyone above some follower threshold — skip the fan-out entirely. Just store their tweet once. Then at READ time, when assembling someone's timeline, separately fetch "recent tweets from any celebrities I follow" and merge them in. You're doing fan-out-on-read, but ONLY for the rare celebrity case, which is a tiny fraction of all tweets.

Step 1 — What's Actually Required Here?

FUNCTIONAL
  • Post a tweet, follow/unfollow people
  • Your home timeline shows tweets from everyone you follow, newest first
  • Like, retweet, reply
NON-FUNCTIONAL
  • Opening the app and seeing your feed needs to feel instant
  • Reads vastly outnumber writes — by a wide margin
  • A handful of accounts have wildly more followers than everyone else (this unevenness is the whole challenge)

Step 4 — What's Actually Sitting in That "Timeline Cache"?

🤔 When you fan-out a tweet to 1,000 followers, are you copying the WHOLE tweet 1,000 times?

No — and this matters a lot at scale. Each person's timeline only stores the tweet ID, not the full tweet content. Think about why: if a tweet goes viral and ends up in a million people's timelines, you don't want a million copies of the same text sitting around. Instead, every timeline is just a short list of IDs, and the actual tweet content lives in ONE place, fetched once and reused for everyone who needs to display it.

Conceptual — Timeline cache structure (Redis)
// Each user's timeline is just a sorted list of tweet IDs, NOT full tweet content
// Redis sorted set: score = timestamp, member = tweet ID

ZADD timeline:user_42  1718000001  tweet_998877
ZADD timeline:user_42  1718000045  tweet_998901
ZADD timeline:user_42  1718000102  tweet_999023

// Reading: get the most recent 50 tweet IDs
ZREVRANGE timeline:user_42 0 49

// Then batch-fetch the actual tweet content for those IDs separately
// MGET tweet:998877 tweet:998901 tweet:999023 ...

// Why bother splitting these into two lookups?
// If a tweet goes viral, it might be referenced in a MILLION timelines.
// Storing the ID once and fetching content once means that viral tweet's
// text exists in ONE place — not duplicated a million times over.
C# — The fan-out decision in code
public class TweetFanOutWorker
{
    private const int CelebrityThreshold = 1_000_000;  // followers

    public async Task ProcessNewTweetAsync(Tweet tweet)
    {
        // 1. Always store the tweet content itself, once
        await _tweetStore.SaveAsync(tweet);

        var followerCount = await _userService.GetFollowerCountAsync(tweet.AuthorId);

        if (followerCount < CelebrityThreshold)
        {
            // 2a. Normal user — fan out to all followers' timelines right now
            var followerIds = await _userService.GetFollowerIdsAsync(tweet.AuthorId);
            foreach (var followerId in followerIds)
            {
                await _timelineCache.AddToTimelineAsync(followerId, tweet.Id, tweet.CreatedAt);
            }
        }
        // 2b. Celebrity — do nothing extra. The tweet sits in _tweetStore,
        //     and gets pulled in later at READ time by anyone following them.
    }
}

public class TimelineReadService
{
    public async Task<List<Tweet>> GetHomeTimelineAsync(int userId)
    {
        // 1. Pre-built timeline — already assembled from fan-out-on-write
        var baseTimelineIds = await _timelineCache.GetRecentTweetIdsAsync(userId, count: 50);

        // 2. The exception case — celebrities this user follows, fetched live
        var celebrityIds = await _userService.GetFollowedCelebrityIdsAsync(userId);
        var celebrityTweets = await _tweetStore.GetRecentTweetsByAuthorsAsync(celebrityIds, count: 50);

        // 3. Merge both lists by time and return the top 50 — invisible to the user
        var allTweets = (await _tweetStore.GetByIdsAsync(baseTimelineIds))
            .Concat(celebrityTweets)
            .OrderByDescending(t => t.CreatedAt)
            .Take(50)
            .ToList();

        return allTweets;
    }
}

Step 6 — Where the Threshold Itself Gets Tricky

⚠️ What Breaks First

  • The threshold is a blunt instrument — an account with 900,000 followers (just under your 1-million cutoff) still triggers 900,000 timeline writes per tweet. There's no perfectly clean line
  • A "normal" tweet can suddenly go viral if a celebrity retweets it — now a tweet that was fanned-out-on-write to 1,000 people needs to ALSO reach that celebrity's 100 million followers through the read path

🔄 A Trade-off Worth Noticing

  • Because fan-out happens asynchronously through workers, there's a small delay — maybe a few seconds — before a tweet appears in every follower's timeline. For a social feed, nobody notices or cares. The same approach would be unacceptable for, say, a banking transaction, where "eventually correct" isn't good enough

🚗 Uber — Real-Time Ride Matching

You open the app and tap "request ride." Somewhere out there, among 5 million active drivers worldwide, the system needs to find the few who happen to be close enough to you to actually pick you up — and do it in under a second. Let's think about how you'd even begin to search for "things near this point" when there are millions of them, all constantly moving.
🤔 Couldn't you just store everyone's latitude/longitude in a database and query "find drivers within 2km"?

You could write that query, sure — but think about what the database actually has to DO to answer it. Without special handling, finding "everything within 2km" means calculating the distance from the rider to EVERY driver in the database, one by one, then keeping the ones close enough. With 5 million drivers and 1.25 million location updates arriving every second, asking the database to do millions of distance calculations per ride request would be brutally slow. We need a way to instantly narrow down to "roughly nearby" before doing any precise math at all.

💡 Turn "Nearby" Into a String Comparison

This is the idea behind geohashing: take a (latitude, longitude) and convert it into a short string code, by repeatedly slicing the map into smaller and smaller grid squares. Here's the trick that makes it useful — two points that are physically close to each other will USUALLY produce geohash strings that share the same starting characters. So "find things near me" becomes "find entries whose geohash starts with the same few characters as mine" — a fast string lookup instead of a math problem run millions of times.

Step 1 — What's Actually Required Here?

FUNCTIONAL
  • Drivers continuously report their current location
  • Rider requests a ride → system finds the nearest available driver
  • Both sides see each other's live position during the trip
NON-FUNCTIONAL
  • "Find nearby drivers" must complete in well under 100ms
  • Location data is constantly changing — old positions become useless within seconds
  • Massive continuous write volume from every driver's app

🤔 So is geohashing the WHOLE solution, or just part of it?

Just part — but the key part. We still need somewhere fast to actually STORE all these constantly-updating positions. A normal disk-backed database, even with geohashing, would struggle to absorb 1.25 million writes per second sustainably. That points to an in-memory store. And since a driver's location 5 minutes ago is essentially useless for matching (they've moved), we don't even need to keep history around — just the current snapshot, refreshed constantly.

🚗 Uber — Location Updates & Ride Matching

Step 4 — How a Geohash Actually Narrows Things Down

🤔 Can you walk through an actual example of how this string-matching trick works?

Sure. A geohash gets longer (more precise) the more characters it has — each extra character shrinks the area it represents. So you pick a precision that roughly matches "the radius I care about" (say, ~1.2km), compute the rider's geohash at that precision, and then look up: which drivers currently have a geohash starting with those same characters? That's your candidate pool — maybe a few dozen drivers instead of 5 million. ONLY THEN do you bother calculating exact distances on that small group, to find the truly closest one.

Conceptual — Geohash precision levels
Geohash "gcpv"     → covers a region roughly 39km x 19km (4 characters)
Geohash "gcpvj"    → covers a region roughly 4.9km x 4.9km (5 characters)
Geohash "gcpvj0"   → covers a region roughly 1.2km x 0.6km (6 characters)

// To find drivers "near" a rider at geohash "gcpvj0":
// 1. Compute the rider's geohash at the precision matching ~1.2km cells
// 2. Also check the 8 neighbouring cells (a nearby driver might be just
//    across a cell boundary — checking only the exact cell would miss them)
// 3. Look up all drivers whose current geohash matches any of these 9 cells
// 4. ONLY NOW calculate precise distance, on this small candidate set

// "Search millions of moving points" becomes "look up ~9 dictionary keys,
// then do a handful of distance calculations" — a massive difference
C# — Driver location update + nearby search
public class LocationService
{
    private readonly IGeoIndex _geoIndex;   // Redis with geo commands, or similar

    // Called every ~4 seconds by each driver's app
    public async Task UpdateDriverLocationAsync(int driverId, double lat, double lng)
    {
        // Stores the position, indexed for radius queries — geohashing happens internally
        await _geoIndex.AddOrUpdateAsync("drivers:available", driverId, lat, lng);

        // Short TTL — if a driver stops sending updates (app closed, went offline),
        // they automatically vanish from the index after ~30 seconds. No cleanup job needed.
        await _geoIndex.ExpireDriverAsync(driverId, TimeSpan.FromSeconds(30));
    }

    public async Task<List<NearbyDriver>> FindNearbyDriversAsync(
        double riderLat, double riderLng, double radiusKm = 3)
    {
        // This single call does the cell-narrowing + distance ranking from the walkthrough above
        var nearby = await _geoIndex.SearchRadiusAsync(
            key: "drivers:available",
            lat: riderLat, lng: riderLng,
            radiusKm: radiusKm,
            maxResults: 10);

        return nearby;  // already sorted by distance, ready for ranking
    }
}
🤔 Do I have to implement geohashing myself?

No — Redis has built-in commands (GEOADD, GEOSEARCH) that handle the geohashing internally; you just feed in coordinates and ask for "nearby", and it returns a sorted list. This is true on every major cloud's managed Redis offering. For routing and ETA calculation specifically (actual road networks, traffic), all three clouds also offer dedicated mapping APIs rather than you building that from scratch.

Step 6 — Where the Cracks Start to Show

⚠️ What Breaks First

  • Dense city centers at rush hour — thousands of drivers can land in the SAME geohash cell, meaning the "narrow it down" step doesn't narrow much; some systems use finer-grained cells specifically in high-density areas
  • Sudden demand spikes — a concert ending, severe weather — concentrate ride requests in one geo area, spiking matching service load very locally rather than globally

💬 WhatsApp — Messaging at Billions-of-Users Scale

Part 2's Chat App had a server that stores and routes messages — perfectly normal, the server can see "Alice said hi to Bob" because it needs to know what to forward. But WhatsApp makes a much stronger promise: not even WhatsApp itself can read your messages. Let's think about what that actually requires changing, mechanically.
🤔 If the server can't read messages, how does it know WHERE to send them?

Good distinction to make: the server can still see the METADATA — who's sending to whom, and when — it just can't see the CONTENT. The message text itself gets encrypted on the sender's device, before it ever leaves their phone, using a key that only the recipient's device possesses. The server receives a blob of encrypted bytes it has no ability to decrypt, and simply forwards that blob to the right destination based on the (unencrypted) "from" and "to" fields. It's like a sealed envelope — the postal service can read the address, never the letter inside.

🤔 What does WhatsApp lose by doing this? Surely there's a cost?

A real one. Plenty of features that seem simple become impossible server-side. Want to search all your old messages? In Part 2's chat app, the server could run that search directly on its database. Here, the server has nothing readable to search — so search has to happen entirely on YOUR device, against messages already decrypted and stored locally. Content moderation, spam detection, anything that requires reading message text — none of it can happen on the server. The privacy guarantee is real, but it's not free.

Step 1 — What's Actually Required Here?

FUNCTIONAL
  • Everything from Part 2's chat app, but now encrypted end-to-end
  • Send media — photos, voice notes, videos
  • Same account, active simultaneously on phone, web, and desktop
NON-FUNCTIONAL
  • 2+ billion users — Part 2's chat challenge, but orders of magnitude bigger
  • The server must be able to route messages it cannot decrypt
  • Messages need to stay correctly synced across every device a user has active

🤔 Why does having 3 devices per user matter for how encryption works?

Because encryption is done per-DEVICE, not per-person. If you send a message to someone with a phone, a laptop, and a browser tab all logged in, your device has to encrypt that SAME message three separate times — once with each of their device's individual key. So 1.2 million messages/second at the message-pipeline layer becomes roughly 3.6 million encrypted payloads/second once you account for multi-device fan-out. That's a real multiplier, and it compounds further in group chats.

Step 3 — Encrypted Routing + a Separate Path for Media

💬 WhatsApp — Encrypted Routing + Separate Media Path
🤔 Why does media get a totally separate path instead of just going through the same pipeline as text?

Think about what the real-time message pipeline (the WebSocket-based system from Part 2) is optimized for: tiny, frequent payloads, delivered instantly — text messages, read receipts, "is typing" indicators. Now imagine a 50MB video trying to squeeze through that same pipe. It would clog the path that millions of other tiny text messages are also trying to use at that exact moment. So instead: the photo gets encrypted and uploaded to object storage SEPARATELY, and the only thing that travels through the fast real-time pipeline is a tiny message containing a link and a decryption key. The recipient's device fetches the actual media on its own, at its own pace, without blocking anyone else's text messages.

Step 4 — Following One Message Across 3 Devices

Conceptual — Multi-device message fan-out
// User A sends "Hello" to User B, who has 3 devices registered:

// 1. User A's device encrypts the message SEPARATELY for each of
//    User B's device keys — this is how E2E encryption handles
//    multiple devices: the sender does the work multiple times

EncryptedMessage[] payloads = {
    Encrypt("Hello", recipientDeviceKey: "B-phone"),
    Encrypt("Hello", recipientDeviceKey: "B-desktop"),
    Encrypt("Hello", recipientDeviceKey: "B-web")
};

// 2. Server receives 3 encrypted payloads + routing info.
//    To the server, these are 3 unrelated blobs of bytes going
//    to 3 device IDs — it has no idea they all say "Hello"

// 3. Server delivers each payload to its device when that device
//    is online (or holds it in a per-device queue if offline)

// 4. Each device decrypts its OWN copy independently.
//    All 3 now display "Hello" — synced in outcome, but each
//    device did its own separate decryption

// This is why linking a new device requires a "pairing" step:
// the new device needs ITS OWN key registered before anyone's
// future messages can be encrypted for it.

🤔 Which cloud product actually does the encryption?

None of them — and that's worth sitting with for a second. Unlike almost everything else in this entire guide, the encryption itself isn't a cloud service you configure; it's code running on the CLIENT apps (phone, desktop, browser), using cryptographic libraries like implementations of the Signal Protocol. The cloud's job here is narrower than usual: reliably move encrypted bytes around and store them — which is why the table above looks a lot like Part 2's chat app, just operating at a much larger scale.

Step 6 — What Gets Hard at This Scale

⚠️ What Breaks First

  • Offline devices accumulate backlogs — a phone that's been off for a week comes back to potentially thousands of queued encrypted messages, all needing delivery and local decryption at once
  • Group chats multiply the encryption work further — a 256-person group means each message gets encrypted once per member, per device. One message can mean 700+ separate encrypted payloads

🔄 A Trade-off Worth Sitting With

  • "Search my messages" and "server-side backup" are SERVER features that simply can't exist when the server can't read content. Backups have to be encrypted separately, with a key only the user controls — meaning if a user loses that key, there's genuinely no way for anyone, including WhatsApp, to recover their backup. That's the actual cost of the privacy guarantee, not a minor footnote

📷 Instagram — Photo/Video Sharing & Feed

You upload a photo. It needs to look crisp on a smartwatch, a phone, and a giant desktop monitor — three wildly different screen sizes, all from the one file you just uploaded. And that same photo might get viewed by your 200 followers a total of 50,000 times over the next week. Let's think about when the "right size for this screen" work should actually happen.
🤔 Why not just store the original and resize it on-the-fly whenever someone views it?

You could — and for a small personal site, that's totally reasonable. But think about the read pattern here: ONE upload, potentially tens of thousands of views. If you resize on every view, you're redoing the EXACT SAME resizing work, over and over, for the exact same output, every single time anyone looks at that photo. That's the same wasteful pattern we saw with the URL shortener's database being asked the same question repeatedly — except now it's CPU-intensive image processing instead of a database lookup.

💡 Same Principle as Netflix — Do the Work Once, Serve It Forever

Generate the handful of sizes you'll actually need — thumbnail, feed-size, full-size — ONCE, right when the photo is uploaded. Every view afterward is just: hand over the file that's already sitting there, no computation required. This is the exact same trade (a bit more storage, for a massive reduction in repeated compute) that Netflix makes with pre-transcoding video into multiple resolutions.

Step 1 — What's Actually Required Here?

  • Followers see it in their feed
  • Like, comment, view profiles
NON-FUNCTIONAL
  • Images must load fast regardless of device or connection
  • Massively read-heavy — one upload, potentially millions of views
  • Processing shouldn't make the person wait after they hit "share"

🤔 If you're generating 5 sizes upfront, doesn't that mean 5x the storage for every single photo?

Yes, roughly — and that's a real cost being deliberately accepted. The alternative (resizing on every view) would cost vastly more in repeated compute across 100 billion feed views per day, for a result that's IDENTICAL every single time you compute it. Paying a bit more in storage to eliminate massive repeated computation is the same calculation Netflix makes, just applied to photos instead of video.

Step 3 — Two Separate Concerns: Uploading vs Feeding

📷 Instagram — Upload Processing Pipeline + Feed Delivery
🤔 Why does "Feed Service" only return URLs instead of the actual images?

This is the same separation-of-concerns idea from Twitter's timeline cache — keep the thing that's queried constantly (the feed list) lightweight, and let a completely different, purpose-built system (the CDN) handle the heavy lifting of actually moving image bytes. If the Feed Service had to fetch and stream image data itself, it would be doing two very different jobs at once, and doing neither particularly well.

Step 4 — Making "Posting" Feel Instant Despite the Processing Work

You hit "share." Generating 5 different image sizes takes a moment — maybe 2-3 seconds of actual processing work. But the app shows your post as live almost immediately. How?
🤔 Does the upload request just wait those 2-3 seconds before responding?

Only for the part that genuinely needs to happen synchronously — uploading the original file. Everything else (resizing into 5 versions, triggering the feed fan-out) gets handed off to a queue and a background worker, the exact pattern from Part 2's Notification System. The user gets back "posted!" the moment the original is safely stored, while the actual image processing happens in the background over the next couple of seconds. The post briefly shows as "processing" before its various sizes are ready.

C# — Upload returns fast, processing happens in background
public class UploadController : ControllerBase
{
    [HttpPost("upload")]
    public async Task<IActionResult> Upload(IFormFile photo, string caption)
    {
        var postId = Guid.NewGuid();
        var originalPath = $"originals/{postId}.jpg";

        // 1. Upload original to object storage — this IS synchronous, user waits
        await _storage.UploadAsync(originalPath, photo);

        // 2. Save post metadata with status "processing"
        await _postRepo.CreateAsync(new Post
        {
            Id = postId, Caption = caption, Status = "processing",
            OriginalPath = originalPath
        });

        // 3. Hand off the heavy lifting — return to the user IMMEDIATELY
        await _queue.PublishAsync(new ImageProcessingJob(postId, originalPath));

        return Ok(new { postId, status = "processing" });
        // "Posted!" appears instantly — the various sizes finish a moment later
    }
}

// Background worker — does the actual resizing
public class ImageProcessingWorker
{
    private static readonly (string Name, int Width)[] Sizes =
    {
        ("thumbnail", 150),
        ("feed",      640),
        ("full",      1080)
    };

    public async Task ProcessAsync(ImageProcessingJob job)
    {
        var original = await _storage.DownloadAsync(job.OriginalPath);

        foreach (var (name, width) in Sizes)
        {
            var resized = ResizeImage(original, width);
            await _storage.UploadAsync($"posts/{job.PostId}/{name}.jpg", resized);
        }

        // Mark post as ready — now visible in feeds
        await _postRepo.UpdateStatusAsync(job.PostId, "published");

        // Trigger feed fan-out (same pattern as Twitter)
        await _fanOutService.PublishPostToFollowersAsync(job.PostId);
    }
}

Step 6 — What's Worth Worrying About

⚠️ What Breaks First

  • Processing queue backlog during major events — if everyone posts at once (a big live event), the queue grows and posts appear with a noticeable delay before all sizes are ready
  • Storage costs accumulate — 5x the original storage adds up fast; rarely-viewed old photos are good candidates to move to cheaper "cold" storage tiers

🔄 A Trade-off Worth Noticing

  • Generating a FIXED set of sizes (thumbnail/feed/full) is simple and very cacheable, but inflexible — if a new device with an unusual screen size becomes popular, you either accept slightly-wrong-sized images or add a 6th pre-generated size. On-demand resizing handles any size perfectly, but reintroduces the per-request cost we were trying to avoid in the first place

📁 Google Drive — File Storage & Sync

You're uploading a 2GB video file on a slow hotel wifi connection. At 73% progress, the connection drops. With most simple upload implementations, you'd have to start completely over — re-send all 2GB from scratch. That's clearly not how Google Drive behaves in practice. Let's figure out what it would take to make "resume where you left off" actually work.
🤔 Why can't you just resume sending the remaining bytes from wherever it stopped?

You actually could, in principle — but think about what the SERVER needs to know to make that work reliably: exactly which bytes it already received, and exactly where to pick back up, even across a connection drop, even if the user closes the app and comes back tomorrow. Tracking "byte 1,847,293,012 onward" as one continuous stream is fragile. What if, instead, you broke the file into clean, fixed-size pieces from the start — say, 4MB chunks — and tracked progress per CHUNK instead of per byte?

💡 Files Are Chunks, Not One Giant Blob

Split every file into fixed-size chunks before uploading. Now "resume" just means: ask the server "which chunk numbers do you already have?", and only send the missing ones. This same idea pays off in two OTHER places too, which is why it's worth doing even ignoring the resume problem: if you edit a tiny part of a huge file, only the changed CHUNKS need to re-sync, not the whole file — and if two different people upload the exact same content, their chunks will be byte-for-byte identical, which opens the door to something pretty elegant.

Step 1 — What's Actually Required Here?

FUNCTIONAL
  • Upload/download files, organize into folders
  • Files sync automatically across all your devices
  • Share files/folders with specific permissions
NON-FUNCTIONAL
  • Large uploads must be resumable — a dropped connection shouldn't mean starting over
  • Avoid storing identical content twice, at the scale of a billion users this adds up enormously
  • Handle the same file being edited on two different devices while both were offline

🤔 30% storage savings sounds significant — where does that actually come from?

Picture a company where 500 employees all received the same 50MB onboarding PDF as an email attachment, and many of them saved a copy to their Drive. Without deduplication, that's 500 separate copies of an identical file — 25GB for content that's really just one 50MB file. If each chunk is identified by a hash of its CONTENT rather than a random ID, then identical content — even uploaded by completely different people who've never interacted — naturally produces identical chunk hashes, and gets stored exactly once.

Step 3 — Watching Deduplication Happen

📁 Google Drive — Chunked Upload with Deduplication
🤔 So how does "resume after a dropped connection" actually use this same mechanism?

Exactly the same lookup. When your 2GB upload drops at 73%, your client already knows which chunks it computed hashes for. On reconnect, it just asks the server again: "which of these hashes do you already have?" The chunks from before the drop come back as "already have it" — skip them. Only the chunks that never made it through get re-sent. You're not tracking byte offsets at all; you're just re-running the SAME deduplication check that handles the "two users, same file" case.

Step 4 — The Harder Problem: Editing While Offline

⚠️ The Scenario

You edit "report.docx" on your laptop while offline on a flight. Later that day, you also edit the SAME file on your phone — also offline at the time. Both devices eventually reconnect. Both now have a DIFFERENT version of the same file. The server has to decide what "correct" means here.

🤔 Why not just keep whichever edit reconnects LAST, since that's presumably the most "final"?

Because "reconnected last" has nothing to do with which edit the user actually cares about more. If your phone happens to sync a few minutes after your laptop, your phone's edit would silently overwrite your laptop's work — work you might have spent an hour on. Silently losing someone's edits because of network timing, rather than their actual intent, is exactly the kind of bug that erodes trust in a product whose entire job is "don't lose my files."

Conceptual — Version-based conflict detection
// Every file has a version number, incremented on each save
// File "report.docx" — last synced version: 5

// Laptop (offline edit, based on version 5):
//   Tries to save as version 6

// Phone (offline edit, ALSO based on version 5):
//   Tries to save as version 6 too

// Laptop reconnects first:
//   Server: "Current version is 5, your edit is based on 5 → fine, now version 6"

// Phone reconnects second:
//   Server: "Current version is 6, but your edit is based on 5 → CONFLICT"
//   The server can tell something's wrong because the phone's edit is
//   based on a version that's no longer the latest — that mismatch IS
//   the detection mechanism, no guessing required.

//   What happens next is a genuine design choice:
//   1. Keep both — save phone's edit as "report (conflicted copy).docx"
//      Simple, never loses data, but pushes the merge decision to the user
//   2. Operational Transform / CRDTs — used by Google Docs for real-time
//      collaborative editing: merge non-conflicting changes automatically
//   3. Last-write-wins — simplest, but SILENTLY discards the phone's edit

// Google Docs (built for live collaboration) uses approach 2.
// Google Drive (arbitrary files like .docx, .pdf) typically uses approach 1 —
// because automatically merging an unknown binary file format isn't possible.

🔐 The Other Thing Checked on Every Single File Access

Every file download, every folder listing, asks: does this user actually have permission? With direct shares, shared folders, and permissions inherited from parent folders, this check gets genuinely complex to evaluate quickly. The practical fix is aggressive caching — permission results for a given user+file pair get cached briefly, since permissions change far less often than files get accessed, but the cache has to invalidate correctly the moment a permission actually changes.

Step 6 — What's Genuinely Tricky at This Scale

⚠️ What Breaks First

  • Deleting a file isn't actually deleting bytes — if another file shares one of its chunks (because the content was identical), you can only safely delete a chunk once NOTHING references it anymore. That requires careful reference counting, not a simple delete
  • Huge folders — a folder with millions of files needs paginated listing, not "load everything at once"

🔄 A Trade-off Worth Noticing

  • Smaller chunks give finer-grained deduplication and sync (only re-upload exactly what changed), but more chunks means more hashes to track and more metadata overhead per file. 4-8MB is the typical sweet spot — small enough to dedupe well, large enough that the metadata doesn't dominate
