🔭 What is AWS X-Ray?

AWS X-Ray is a distributed tracing service that lets you understand how your application and its underlying services are performing. Instead of guessing where a slow request spent its time, X-Ray gives you a visual, end-to-end trace showing every service hop, its latency, and any errors — from the first API call to the final database query.

💡 X-Ray in One Sentence

X-Ray = a tracing system that stitches together every service a request touches into a single visual timeline, so you can find the slow or broken piece in seconds instead of hours.

The Problem X-Ray Solves

😰 Without X-Ray

  • A request takes 4 seconds — which service caused it?
  • You grep through 6 different CloudWatch log groups
  • Correlating logs across services requires matching timestamps manually
  • You don't know if the bottleneck is your Lambda, DynamoDB, or a downstream API
  • 5xx errors from API Gateway — which backend service threw them?

😌 With X-Ray

  • Open the trace — one timeline shows all 6 services
  • The DynamoDB subsegment is highlighted in orange: 3.8s
  • Click it — see the exact table, operation, and response code
  • Trace ID propagated automatically across Lambda, ECS, API Gateway
  • Service Map shows which node is red (errors) and which is orange (slow)

🏗️ X-Ray Architecture — How a Trace is Built

X-Ray works by having each service instrument its code with the X-Ray SDK. The SDK sends segment data to a local X-Ray Daemon (over UDP), which buffers and forwards it to the X-Ray API. AWS stitches segments from all services into a single trace using the Trace ID header.

🔭 X-Ray — End-to-End Tracing Architecture

How a Trace ID Propagates

👤 Client sends HTTP request
  • No Trace ID yet — first entry point generates one
  • Format: Root=1-{timestamp}-{96bit_hex};Parent={64bit_hex};Sampled=1
↓
API Gateway injects X-Amzn-Trace-Id header
🌐 API Gateway — Root Segment
  • Creates root trace segment with Trace ID
  • Passes X-Amzn-Trace-Id header to backend Lambda
  • Records latency, status code, method, resource path
↓
Lambda SDK reads Trace ID from context — creates child segment
⚡ Lambda — Child Segment + Subsegments
  • X-Ray SDK auto-wraps DynamoDB / HTTP calls as subsegments
  • Your code adds custom subsegments for business logic
  • Segment sent to local daemon; daemon forwards to X-Ray API
↓
X-Ray API stitches all segments by Trace ID
📊 X-Ray Console — Full Trace Timeline
  • Waterfall view: all segments on a single time axis
  • Click any segment to see annotations, metadata, errors
  • Total duration = time from first to last segment

📦 Segments, Subsegments & Annotations

The fundamental building block of X-Ray is the segment. Every service in a trace contributes one segment. Within a segment, individual operations (DB calls, HTTP requests, custom code) are subsegments. You enrich them with annotations (indexed, searchable) and metadata (rich data, not searchable).

Annotations — Searchable Key-Value Pairs

🏷️

What annotations are for

Annotations let you filter traces in the console using expressions like annotation.userId = "u_9abc" or annotation.orderStatus = "failed". You add up to 50 annotations per trace. Values must be strings, numbers, or booleans.

📋

What metadata is for

Metadata is for richer, non-searchable data — e.g. the full request body, an object graph, or debug context. There's no size limit on metadata values. Use metadata for things you'd want to inspect in a trace but don't need to filter by.

🎚️ Sampling Rules — Controlling Trace Volume

Tracing every single request would be expensive and unnecessary. Sampling rules let you control what percentage of requests get traced. X-Ray evaluates rules in priority order — the first match wins.

💡 Default Sampling Rule

AWS default: trace 1 request per second per service + 5% of additional requests. This gives you a baseline signal without high cost. You can create custom rules with higher rates for critical paths.

Sampling Rule Priority Order

1

Critical Path Rule (Priority 50)

Match: POST /checkout · Rate: 100% · Reservoir: 50/s — all checkout requests traced regardless of volume.

2

Exclude Health Checks (Priority 80)

Match: GET /health, GET /ping · Rate: 0% · Reservoir: 0 — health check noise eliminated from traces.

3

Error Monitoring Rule (Priority 90)

Match: * (all paths) · Rate: 100% for error responses — every error always traced regardless of default rate.

4

Default Rule (Priority 10000 — always last)

Match: * · Rate: 5% · Reservoir: 1/s — catch-all for everything not matched by higher-priority rules.

⚠️ Sampling is Per-Service

Each service in a trace makes its own sampling decision based on the incoming Sampled flag. If API Gateway samples a request (Sampled=1), it passes this flag downstream — Lambda will also trace it. If Sampled=0, the downstream service skips tracing too. You can override with Sampled=? to let the downstream decide independently.

X-Ray Analytics — Beyond the Service Map

🔍 Trace Search & Filter

  • Filter by duration: responsetime > 2
  • Filter by annotation: annotation.userId = "u123"
  • Filter by status: fault = true or error = true
  • Combine: service("order-svc") AND responsetime > 1
  • URL filter: http.url CONTAINS "/orders"

📊 X-Ray Groups

  • Save a filter expression as a named Group
  • Groups have their own sampling rules and Service Map
  • Example: Group checkout-errors = fault = true AND http.url CONTAINS "/checkout"
  • Set CloudWatch alarms on group metrics
  • Enable Insights on a group for auto-anomaly detection

X-Ray Insights

💡 What are Insights?

X-Ray Insights automatically detects anomalies in your traced services — e.g. "error rate on payment-svc increased from 0.5% to 8% at 14:32". It creates an Insight event, notifies you via SNS, and shows the root-cause service in the Service Map. Enable Insights per X-Ray Group.

Encryption

🔒 Trace Data Encryption

  • All trace data encrypted at rest by default (AWS-managed key)
  • Customer-managed KMS CMK supported for compliance
  • Configurable per region in X-Ray Console → Encryption config
  • Changing key is applied to new traces; existing traces remain with old key

🔗 In-Transit Encryption

  • SDK → Daemon: UDP (local loopback only — no network exposure)
  • Daemon → X-Ray API: HTTPS/TLS — always encrypted
  • Console → X-Ray API: HTTPS via AWS console — always encrypted
  • VPC endpoint available for private subnet daemons

💡 Sensitive Data in Traces

Never put PII (passwords, SSNs, payment data) in annotations or metadata — traces are stored for 30 days and accessible to anyone with xray:BatchGetTraces. Use opaque IDs (e.g. userId = "u_abc123") rather than email addresses or names. If you accidentally trace sensitive data, you cannot delete individual traces — the entire retention window applies.

Pattern 2 — Custom Subsegments + Annotations + Metadata

C# · Lambda handler · Custom tracing 
using Amazon.XRay.Recorder.Core;
using Amazon.Lambda.Core;

public class OrderHandler
{
    private readonly IAmazonDynamoDB _dynamo;
    private readonly IAmazonSQS _sqs;

    public async Task<APIGatewayProxyResponse> HandleAsync(
        APIGatewayProxyRequest request, ILambdaContext context)
    {
        var order = JsonSerializer.Deserialize<Order>(request.Body)!;

        // ── Add annotations (INDEXED — searchable in console) ──
        AWSXRayRecorder.Instance.AddAnnotation("userId", order.UserId);
        AWSXRayRecorder.Instance.AddAnnotation("orderId", order.OrderId);
        AWSXRayRecorder.Instance.AddAnnotation("orderTotal", order.Total);

        // ── Add metadata (NOT indexed — rich debug data) ──
        AWSXRayRecorder.Instance.AddMetadata("requestPayload", request.Body);
        AWSXRayRecorder.Instance.AddMetadata("lambdaRequestId", context.AwsRequestId);

        // ── Custom subsegment around validation logic ──
        Order? validated = null;
        AWSXRayRecorder.Instance.BeginSubsegment("ValidateOrder");
        try
        {
            validated = ValidateOrder(order);
            AWSXRayRecorder.Instance.AddAnnotation("validationPassed", true);
        }
        catch (ValidationException ex)
        {
            // Mark subsegment as error (4xx — client caused it)
            AWSXRayRecorder.Instance.AddException(ex);
            AWSXRayRecorder.Instance.MarkError();
            throw;
        }
        finally
        {
            AWSXRayRecorder.Instance.EndSubsegment();
        }

        // ── DynamoDB call — auto-traced because of RegisterXRayForAllServices ──
        await _dynamo.PutItemAsync(new PutItemRequest
        {
            TableName = "Orders",
            Item      = ToDynamoItem(validated!)
        });

        // ── Custom async subsegment using lambda helper ──
        await AWSXRayRecorder.Instance.BeginSubsegmentAsync(
            "PublishOrderEvent",
            async () =>
            {
                await _sqs.SendMessageAsync(new SendMessageRequest
                {
                    QueueUrl    = Environment.GetEnvironmentVariable("ORDER_QUEUE_URL"),
                    MessageBody = JsonSerializer.Serialize(validated)
                });
            });

        return new APIGatewayProxyResponse { StatusCode = 201 };
    }
}

Pattern 3 — ASP.NET Core Middleware + HttpClient Instrumentation

C# · Program.cs · ASP.NET Core + instrumented HttpClient 
using Amazon.XRay.Recorder.Core;
using Amazon.XRay.Recorder.Handlers.AspNetCore;
using Amazon.XRay.Recorder.Handlers.Http;

var builder = WebApplication.CreateBuilder(args);

// Register X-Ray services
builder.Services.AddDefaultAWSOptions(builder.Configuration.GetAWSOptions());
AWSSDKHandler.RegisterXRayForAllServices();   // patch all AWS SDK clients

// Register an HttpClient instrumented for X-Ray outgoing calls
builder.Services.AddHttpClient("payment-api", client =>
{
    client.BaseAddress = new Uri("https://payment.internal.svc");
})
.AddHttpMessageHandler(() => new TracingMessageHandler());
// TracingMessageHandler adds X-Amzn-Trace-Id to outgoing HTTP requests
// so the downstream service can continue the trace

var app = builder.Build();

// ── X-Ray middleware: MUST be first in pipeline ──
// Creates a segment per HTTP request, reads X-Amzn-Trace-Id header
app.UseXRay("order-service");

app.UseRouting();
app.MapControllers();
app.Run();


// ── In your controller — annotations available via current segment ──
[ApiController]
[Route("[controller]")]
public class OrdersController : ControllerBase
{
    [HttpPost]
    public async Task<IActionResult> CreateOrder([FromBody] OrderRequest req)
    {
        // Segment already created by UseXRay middleware
        AWSXRayRecorder.Instance.AddAnnotation("userId", req.UserId);
        AWSXRayRecorder.Instance.AddAnnotation("sku", req.Sku);

        try
        {
            // ... process order ...
            return Created($"/orders/{req.OrderId}", req);
        }
        catch (Exception ex)
        {
            AWSXRayRecorder.Instance.AddException(ex);
            AWSXRayRecorder.Instance.MarkFault();  // 5xx — server fault
            throw;
        }
    }
}

✅ Best Practices & Common Mistakes

Instrumentation Best Practices

  • Call AWSSDKHandler.RegisterXRayForAllServices() at startup — auto-instruments all DynamoDB, SQS, S3, SNS calls with zero per-call code
  • Use UseXRay("service-name") middleware as the FIRST middleware in ASP.NET Core pipelines — ensures trace context is set before any handler runs
  • Always use TracingMessageHandler on HttpClients calling other services — propagates Trace ID so downstream services join the same trace
  • Wrap custom subsegments in try/finally — EndSubsegment() must always be called even on exception, or the trace will be incomplete
  • Prefer the async BeginSubsegmentAsync helper for async code blocks — avoids accidentally ending the wrong subsegment in concurrent execution
  • Keep annotation keys consistent and lowercase across services — enables reliable cross-service filtering in the console

Annotations & Metadata

  • Add 3–5 key annotations per trace: userId, orderId, region, tenantId, featureFlag — enough to filter without bloating every trace
  • Never put PII in annotations or metadata — traces are retained 30 days and searchable by anyone with read access
  • Use metadata for full payloads, exception context, and config state — it's for debugging, not filtering
  • Annotation values must be string, number, or boolean — serialize complex objects to JSON string if needed
  • Max 50 annotations per trace — prioritise identifiers and outcome flags

Sampling Strategy

  • Create a rule to trace 100% of error responses — you always want full traces for every failure
  • Exclude health check endpoints (GET /health, GET /ping) — they dominate counts and add zero diagnostic value
  • Trace 100% of critical paths (checkout, payment, auth) — the cost is low and the insight is high
  • Use reservoir + fixed rate together — reservoir guarantees some coverage at low traffic; fixed rate scales proportionally at high traffic
  • Review sampling statistics monthly — adjust rules if you're tracing too many health checks or missing spikes on critical paths

Common Mistakes to Avoid

⚠️ Top X-Ray Mistakes

❌ Not calling EndSubsegment() in a finally block — if an exception escapes, the subsegment is never closed, orphaned segments clog the trace and missing timings make it unreadable
❌ Forgetting TracingMessageHandler on HttpClient — outbound HTTP calls to other services won't carry the Trace ID; downstream segments appear as a disconnected new trace, breaking the service map
❌ Storing PII in annotations/metadata — userId = "alice@example.com" is a GDPR/compliance issue; use opaque internal IDs instead
❌ Not enabling X-Ray on API Gateway — Lambda Active Tracing alone doesn't show the API Gateway latency; the trace starts mid-chain and you miss total request time
❌ Using default sampling for everything — 5% means you miss 95% of errors at low traffic (e.g. 5 req/min → only 1 request traced every 4 minutes); always add a 100% fault sampling rule
❌ Not using X-Ray Groups — without groups, your service map shows everything; a noisy service drowns out signals from the service that's actually failing
❌ Ignoring IteratorAge equivalent for X-Ray — segment send failures — check CloudWatch for xray:PutTraceSegments throttling if traces disappear; the daemon silently drops segments when the API is throttled
