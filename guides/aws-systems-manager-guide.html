🏗️ What is AWS Systems Manager?

A collection of capabilities that gives you visibility and control over your AWS infrastructure — no bastion hosts needed.

One-liner SSM = unified control plane for EC2, on-prem, and ECS/EKS nodes — run commands, manage config, patch OS, store secrets, automate ops, and open secure shell sessions without opening inbound ports.

🔑 Parameter Store

Hierarchical, versioned storage for config values and secrets — free tier available, integrates natively with IAM and KMS.

📁 Parameter Types

String — plain text value. Free.

StringList — comma-separated list. Free.

SecureString — KMS-encrypted at rest. Free (default KMS) or pay for CMK.

📊 Standard vs Advanced

Standard — Up to 10,000 params, max 4 KB value, free, no parameter policies.

Advanced — 100,000+ params, max 8 KB, parameter policies (TTL, expiry notification), paid.

Hierarchy Structure

Use path-based hierarchy for organisation Parameters are organised like a file system: /app/env/name. You can fetch an entire path at once with GetParametersByPath and apply IAM policies at the path level.

🔄 Versioning

Every parameter update creates a new version automatically. Retrieve a specific version with GetParameter + version suffix. Rollback by referencing an older version in code.

🔔 Parameter Policies (Advanced)

Expiration — auto-delete parameter after date.

ExpirationNotification — SNS alert N days before expiry.

NoChangeNotification — alert if param unchanged for N days.

🤖 Run Command

Execute scripts and commands across fleets of managed instances — no SSH, no bastion host.

How it works SSM Agent on each instance polls the SSM service endpoint over HTTPS. When a Run Command is sent, the agent receives the command, executes it locally, and sends output back to SSM (and optionally S3 / CloudWatch Logs).
🎯 Target Selection

Target instances by Instance ID, Tag (e.g. Env=Prod), Resource Group, or All managed instances.

📄 SSM Documents

Commands are defined in SSM Documents (JSON/YAML). AWS provides 100+ built-in docs. You can create custom documents.

📋 Output Options

Last 2,500 chars in console. Full output to S3 bucket or CloudWatch Logs. SNS notification on completion/failure.

Common Built-in Documents

User / EventBridge → SendCommand API
  ├── Target: Tag: Env=Production
  ├── Document: AWS-RunShellScript
  └── Command: "systemctl restart myapp"

SSM Service → SSM Agent on each matched instance
  ├── Agent polls endpoint (HTTPS 443 outbound)
  ├── Executes command locally
  └── Reports: Success / Failed / TimedOut

Output → S3 + CloudWatch Logs + SNS notification

📋 Session Manager

Secure, auditable shell access to EC2 and on-prem instances — zero open ports, zero bastion hosts.

Why Session Manager over SSH? No inbound port 22 needed. No key pair management. Full session audit log to CloudTrail + S3 + CloudWatch Logs. Access controlled entirely via IAM policies. Works for EC2, on-prem, and ECS Exec.
🔐 Access via IAM

Grant or deny session access with IAM policies. Can restrict to specific instances, specific users, or specific tags — no OS-level user management needed.

No SSH keys No port 22

📝 Audit Logging

Every session is logged. Session metadata (start/end, user, instance) → CloudTrail. Full session transcript → S3 or CloudWatch Logs.

Compliance ready

🚇 Port Forwarding

Forward local port to a remote port on the instance or beyond (RDS, internal services). No VPN or bastion needed — all through SSM tunnel.

aws ssm start-session \
  --target i-0abc1234 \
  --document-name \
    AWS-StartPortForwardingSession \
  --parameters \
    '{"portNumber":["3306"],
      "localPortNumber":["13306"]}'
🐳 ECS Exec

Drop into a running ECS task container using Session Manager under the hood. No need to expose container ports or exec via SSH.

aws ecs execute-command \
  --cluster prod-cluster \
  --task abc123 \
  --container app \
  --interactive \
  --command "/bin/sh"

IAM Policy for Session Manager

JSON Minimum IAM policy to allow SSM Session Manager 
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "ssm:StartSession",
        "ssm:TerminateSession",
        "ssm:ResumeSession",
        "ssm:DescribeSessions",
        "ssm:GetConnectionStatus"
      ],
      "Resource": [
        "arn:aws:ec2:*:*:instance/*",
        "arn:aws:ssm:*:*:session/${aws:username}/*"
      ],
      "Condition": {
        "StringLike": {
          "ssm:resourceTag/Environment": ["Production"]
        }
      }
    }
  ]
}

🔧 Patch Manager

Automate OS and application patching across your fleet with baselines, maintenance windows, and compliance dashboards.

📋 Patch Baseline

Defines which patches are approved. Default baselines exist for Amazon Linux, Windows, Ubuntu, RHEL, SUSE. Custom baselines override defaults.

Rules: Auto-approve after N days, severity filter, classification filter.

🪟 Maintenance Window

Defines when patching runs — cron expression, duration, cutoff. Targets instances by tag. Prevents patching during business hours.

Also used for Run Command and Automation tasks.

📊 Patch Compliance

After patching, SSM reports each instance's patch state: Compliant or Non-Compliant. View in SSM console or query via AWS Config.

Patch Lifecycle

  • 1
    Define Patch BaselineSet approval rules: auto-approve Critical + Important patches after 7 days. Reject patches with known issues.
  • 2
    Create Maintenance WindowSchedule: every Sunday 02:00 UTC. Duration: 4 hours. Cutoff: 1 hour before end (stop new tasks).
  • 3
    Register TargetsAdd instances to the maintenance window by tag (e.g. PatchGroup=WebServers). Associate the patch baseline.
  • 4
    Run AWS-RunPatchBaselineSSM runs this document on all targets. Scans, downloads, installs approved patches, reboots if needed.
  • 5
    Check ComplianceView patch compliance in SSM Patch Manager dashboard or query with DescribeInstancePatchStates API.
⚠️ Patch Groups Tag instances with Patch Group (exact key name) to assign them to a specific patch baseline. One instance → one patch group → one baseline. Use this to patch dev/staging/prod at different times.

📜 Automation (Runbooks)

Orchestrate multi-step operational workflows using SSM Documents — from AMI creation to DR failover.

Automation vs Run Command Run Command = run a script on instances. Automation = orchestrate a multi-step workflow across AWS services (EC2, Lambda, SNS, Step Functions, wait for approval, branch on conditions). Automation runbooks are SSM Documents of type Automation.
⚙️ Automation Actions

aws:runInstances — launch EC2 instance

aws:createImage — create AMI snapshot

aws:executeScript — run Python/Node inline

aws:invokeLambdaFunction — call Lambda

aws:approve — pause for human approval

aws:waitForAwsResourceProperty — poll until ready

aws:branch — conditional branching

🚀 Common Use Cases

🖼️ Golden AMI pipeline — launch, configure, test, create AMI, deregister old

🔄 Instance remediation — detect non-compliant → fix → verify

💾 Backup automation — snapshot RDS/EBS on schedule

🆙 Blue/Green deploy — launch new, test, swap, terminate old

🚨 Incident response — isolate instance, capture forensics

Sample Automation Document (YAML)

YAML SSM Automation — Stop, snapshot, restart EC2 
schemaVersion: "0.3"
description: "Stop EC2, create snapshot, restart"
parameters:
  InstanceId:
    type: String
    description: "EC2 instance to snapshot"
mainSteps:
  - name: stopInstance
    action: aws:changeInstanceState
    inputs:
      InstanceIds: ["{{ InstanceId }}"]
      DesiredState: stopped

  - name: waitForStop
    action: aws:waitForAwsResourceProperty
    inputs:
      Service: ec2
      Api: DescribeInstanceStatus
      InstanceIds: ["{{ InstanceId }}"]
      PropertySelector: $.InstanceStatuses[0].InstanceState.Name
      DesiredValues: ["stopped"]

  - name: createAmi
    action: aws:createImage
    inputs:
      InstanceId: "{{ InstanceId }}"
      ImageName: "Backup-{{ InstanceId }}-{{ global:DATE }}"
      NoReboot: true

  - name: restartInstance
    action: aws:changeInstanceState
    inputs:
      InstanceIds: ["{{ InstanceId }}"]
      DesiredState: running

Execution Modes

⚡ Simple Execution

Run once on a single target. Manual trigger from console or CLI. Good for one-off operational tasks.

🎯 Rate Control

Run on a fleet with concurrency and error thresholds. E.g. patch 10% of fleet at a time, stop if >5% fail.

✅ Change Manager

Approval workflow before automation runs. Integrates with ServiceNow / Jira. Full change record for compliance.

🧩 C# / .NET SDK Examples

Using AWSSDK.SimpleSystemsManagement NuGet package.

NuGet dotnet add package AWSSDK.SimpleSystemsManagement
C# Get a parameter from Parameter Store 
using Amazon.SimpleSystemsManagement;
using Amazon.SimpleSystemsManagement.Model;

var ssm = new AmazonSimpleSystemsManagementClient();

// Get a single parameter (WithDecryption=true for SecureString)
var response = await ssm.GetParameterAsync(new GetParameterRequest
{
    Name            = "/OrderService/Production/DatabaseConnectionString",
    WithDecryption  = true   // decrypt SecureString with KMS
});

var connStr = response.Parameter.Value;
Console.WriteLine($"Version: {response.Parameter.Version}");
Console.WriteLine($"LastModified: {response.Parameter.LastModifiedDate}");
C# Get all parameters under a path hierarchy 
// Fetch entire /OrderService/Production/ path at once
var allParams = new Dictionary<string, string>();
string? nextToken = null;

do
{
    var page = await ssm.GetParametersByPathAsync(new GetParametersByPathRequest
    {
        Path           = "/OrderService/Production/",
        Recursive      = true,
        WithDecryption = true,
        NextToken      = nextToken
    });

    foreach (var p in page.Parameters)
        allParams[p.Name] = p.Value;

    nextToken = page.NextToken;
}
while (nextToken is not null);

// Access like a dictionary
var maxRetries = int.Parse(allParams["/OrderService/Production/MaxRetries"]);
C# Put / update a parameter 
await ssm.PutParameterAsync(new PutParameterRequest
{
    Name        = "/OrderService/Production/MaxRetries",
    Value       = "5",
    Type        = ParameterType.String,
    Overwrite   = true,   // update existing
    Description = "Max retry count for order API calls",
    Tags        =
    [
        new Tag { Key = "App",         Value = "OrderService" },
        new Tag { Key = "Environment", Value = "Production"  }
    ]
});
C# Send Run Command to tagged instances 
var cmd = await ssm.SendCommandAsync(new SendCommandRequest
{
    DocumentName    = "AWS-RunShellScript",
    Targets         =
    [
        new Target
        {
            Key    = "tag:Environment",
            Values = ["Production"]
        }
    ],
    Parameters      = new Dictionary<string, List<string>>
    {
        ["commands"] = ["systemctl restart orderservice", "systemctl status orderservice"]
    },
    OutputS3BucketName = "my-ssm-output-bucket",
    OutputS3KeyPrefix  = "run-command-output/",
    CloudWatchOutputConfig = new CloudWatchOutputConfig
    {
        CloudWatchOutputEnabled = true,
        CloudWatchLogGroupName  = "/ssm/run-command"
    }
});

Console.WriteLine($"CommandId: {cmd.Command.CommandId}");
C# Start SSM Automation execution 
var exec = await ssm.StartAutomationExecutionAsync(new StartAutomationExecutionRequest
{
    DocumentName    = "AWS-CreateImage",
    Parameters      = new Dictionary<string, List<string>>
    {
        ["InstanceId"] = ["i-0abc123456"],
        ["NoReboot"]   = ["true"]
    }
});

// Poll for completion
while (true)
{
    var status = await ssm.GetAutomationExecutionAsync(
        new GetAutomationExecutionRequest
            { AutomationExecutionId = exec.AutomationExecutionId });

    var state = status.AutomationExecution.AutomationExecutionStatus;
    Console.WriteLine($"Status: {state}");

    if (state is "Success" or "Failed" or "Cancelled") break;
    await Task.Delay(5000);
}

💡 Best Practices

Production-grade SSM setup for security, operations, and cost efficiency.

✅ Use Parameter Hierarchy

Always structure as /App/Env/Key. Fetch entire env config in one GetParametersByPath call. Apply IAM path-level policies for environment isolation.

✅ Replace SSH with Session Manager

Remove port 22 from all EC2 security groups. Use Session Manager for all interactive access. Gain full audit trail without bastion host cost.

✅ Tag Instances for Targeting

Tag EC2 with Environment, App, PatchGroup. Use tags in Run Command and Patch Manager targets — avoid hardcoding instance IDs.

✅ Least Privilege for SSM

Attach AmazonSSMManagedInstanceCore policy to EC2 instance role — minimum needed for SSM agent. Don't use AmazonSSMFullAccess on instances.

✅ SecureString for Secrets

Never store passwords or API keys as plain String. Use SecureString with a customer-managed KMS key (CMK) for sensitive values.

✅ Cache Parameters in App

Don't call SSM on every request — cache parameters in memory at startup. Refresh periodically or on config change event. SSM has API rate limits.

✅ Patch in Waves

Create separate maintenance windows for Dev (weekly), Staging (weekly+1 day), Production (bi-weekly). Test patches in lower envs before prod.

✅ Monitor with Inventory

Enable Inventory collection on all managed nodes. Track installed software, config changes, and OS versions. Integrate with AWS Config for drift detection.

🤖 Run Command

Inventory: collect software/config metadata. State Manager: enforce desired config on schedule — prevent config drift across your fleet.
