📬 What is Amazon SQS?

SQS = Simple Queue Service. A fully managed message queue that decouples producers (who send work) from consumers (who process work). The queue stores messages durably — up to 14 days — until a consumer is ready to process them. If the consumer crashes mid-process, the message reappears and gets retried automatically.

💡 Real-life analogy

SQS is like a post office's sorting room. Letters (messages) arrive and sit in sorted trays (queues). Postal workers (consumers) pick up one letter at a time, sign for it (visibility timeout starts), deliver it, and mark it done (delete it). If a worker goes home sick mid-delivery (consumer crash), the unsigned letter reappears in the tray after a timeout and another worker picks it up. Letters that repeatedly fail delivery go to a separate "problem tray" (Dead-Letter Queue).

Why SQS? The Problems It Solves

😓 Without SQS — tight coupling

  • Order API calls Inventory Service directly (synchronous)
  • Inventory down → order fails immediately
  • Traffic spike → Inventory overwhelmed
  • Slow processing → Order API times out
  • No retry — failed = gone
  • Both services must be up simultaneously

✅ With SQS — loose coupling

  • Order API sends to queue → responds instantly to user
  • Inventory down → messages wait safely in queue
  • Traffic spike → queue absorbs the burst
  • Slow processing → queue buffers, no timeouts
  • Consumer crash → message reappears, retried
  • Services operate completely independently

🔄 Message Lifecycle — Birth to Deletion

Every message in SQS goes through a precise lifecycle. Understanding each state is crucial for debugging and building reliable systems.

The Complete Message Journey

📤 Step 1 — Producer Sends Message
  • Producer calls SendMessage API
  • Message body: up to 256 KB (JSON, XML, text)
  • Optional: MessageAttributes, DelaySeconds, DeduplicationId (FIFO)
  • SQS returns: MessageId — unique identifier
  • Producer's job is done — fire and forget
↓
📬 Step 2 — Message Sits in Queue (VISIBLE)
  • Message stored durably across multiple AZs
  • Status: VISIBLE — available to be received
  • Waits here until a consumer polls the queue
  • If DelaySeconds set → DELAYED first, then VISIBLE
  • Retention clock starts ticking (default 4 days)
↓
🔒 Step 3 — Consumer Receives Message (IN-FLIGHT)
  • Consumer calls ReceiveMessage API
  • SQS returns message + ReceiptHandle (unlock key)
  • Status: IN-FLIGHT — invisible to other consumers
  • Visibility timeout clock starts ticking
  • Only THIS consumer can see and delete it
  • Message NOT yet removed from queue!
↓
✅ Success Path
  • Consumer processes message
  • Calls DeleteMessage with ReceiptHandle
  • Message PERMANENTLY removed from queue
  • Job complete — zero message loss risk
⚠️ Failure Path
  • Consumer crashes OR timeout expires
  • Message becomes VISIBLE again
  • Another consumer receives it
  • After maxReceiveCount → goes to DLQ

Message Delay — Control When It Becomes Visible

⏰ Queue-Level Delay
  • Set on the queue (0–900 seconds)
  • ALL messages delayed on arrival
  • Good for: batch jobs that should wait
  • Good for: scheduled retry systems
⏰ Per-Message Delay
  • Set on individual SendMessage call
  • Overrides queue-level delay
  • Good for: "process this in 5 minutes"
  • Only works on Standard queues

💡 "At-least-once" — what this really means

SQS Standard guarantees at-least-once delivery — a message may occasionally be delivered more than once due to distributed system internals. Design your consumers to be idempotent: processing the same message twice should produce the same result. Track processed MessageIds in DynamoDB to detect and safely ignore duplicates.

⏳ Visibility Timeout — The Safety Mechanism

Visibility timeout is SQS's crash-recovery mechanism. When a consumer receives a message, it becomes invisible to others for the timeout duration. If the consumer doesn't delete it in time — the message reappears automatically, ready for another consumer.

💡 Analogy

Imagine picking up a parcel from a collection desk. You sign a collection slip (ReceiveMessage) — the parcel is marked "collected" (invisible) for 30 minutes. If you return and confirm delivery (DeleteMessage) → done. If you don't return in 30 minutes (crash / timeout) → the desk marks it uncollected again and gives it to the next person asking.

Three Visibility Timeout Scenarios

✅ Scenario 1 — Happy Path
  • VisibilityTimeout: 30 seconds
  • Consumer receives message at t=0
  • Processes in 8 seconds
  • Calls DeleteMessage at t=8 → message gone forever
  • No risk of duplicate processing
↓
⚠️ Scenario 2 — Processing Takes Longer Than Timeout
  • VisibilityTimeout: 30 seconds
  • Processing takes 45 seconds (slow DB query)
  • At t=30, timeout expires — message REAPPEARS
  • Second consumer picks it up → DUPLICATE PROCESSING!
  • Fix: extend timeout at t=25 using ChangeMessageVisibility
↓
💥 Scenario 3 — Consumer Crashes
  • Consumer receives message, crashes at t=10
  • No DeleteMessage called
  • At t=30, timeout expires → message REAPPEARS ✅
  • Another healthy consumer picks it up and processes it
  • This is the built-in crash recovery — zero message loss

Choosing the Right Visibility Timeout

⚙️ Visibility Timeout = Maximum Processing Time × 1.5 (safety buffer)

If your Lambda function timeout is 15 seconds → set visibility timeout to ~25 seconds
If your ECS task processes in ~2 minutes → set visibility timeout to ~3 minutes

Too short: messages reappear before processing finishes → duplicates
Too long: crashed consumer's message is invisible for a long time → slower recovery

Best practice for Lambda: set SQS visibility timeout to 6× the Lambda function timeout. This gives headroom for Lambda retries and batch processing.

💀 Dead-Letter Queues — When Messages Fail Repeatedly

Some messages will always fail — malformed data, a bug in the consumer, a downstream service that's permanently down. Without a DLQ these messages loop forever, blocking real messages. A DLQ catches them after N failures so you can inspect and fix them.

💡 Analogy

A DLQ is like a "return to sender" pile at the post office. If a letter has been attempted 3 times and keeps bouncing back (address invalid, nobody home repeatedly) — instead of attempting a 4th time, it goes to a special "undeliverable" tray where a supervisor can review it and decide what to do.

Message Journey to the DLQ

💀 maxReceiveCount = 3 — after 3 failures, message moves to DLQ
Attempt 1 → Consumer receives MSG-42 → tries to parse JSON → malformed data → throws exception → no DeleteMessage called t=0s
Reappears ↻ Visibility timeout expires (30s) → MSG-42 becomes visible again → ReceiveCount: 1 → 2 t=30s
Attempt 2 → Different consumer receives → same failure → exception → timeout again t=35s
Reappears ↻ Timeout expires again → ReceiveCount: 2 → 3 t=65s
Attempt 3 → Third consumer → same result → ReceiveCount reaches maxReceiveCount (3) t=70s
→ DLQ → SQS moves MSG-42 to Dead-Letter Queue automatically. Original queue stops seeing it. CloudWatch alarm fires. t=100s

What to Do When Messages Reach the DLQ

🔔 Step 1 — Alert fires (CloudWatch alarm)
  • Alarm: DLQ ApproximateNumberOfMessagesVisible > 0
  • SNS alert → Slack / email → on-call engineer notified
  • Never ignore DLQ messages — each is a real failure
↓
🔍 Step 2 — Investigate failed messages
  • View DLQ messages in SQS console (or receive via SDK)
  • Check message body: is the data malformed?
  • Check CloudWatch Logs for consumer error at time of failure
  • Identify root cause: bug in consumer? Bad data? Downstream outage?
↓
🔧 Step 3 — Fix and Redrive
  • Fix the consumer bug → deploy fix
  • Use SQS Dead-Letter Queue Redrive (console button)
  • Messages replayed back to source queue automatically
  • Consumer processes them successfully this time
  • Zero message loss — all recovered

✅ DLQ configuration golden rule

Set maxReceiveCount based on whether failures are transient (retry more) or permanent (retry less). For transient failures (network hiccup, temporary outage): set to 5–10. For permanent failures (bad data format): set to 3. Always pair every source queue with a DLQ — not optional in production.

🔀 Standard vs FIFO Queues

SQS offers two fundamentally different queue types. Standard is the default — unlimited throughput but no order guarantee. FIFO guarantees strict order and exactly-once processing, but at a lower throughput cap.

Side-by-Side Comparison

🔶 Standard Queue (Default)

  • Throughput: virtually unlimited
  • Order: best-effort — may be out of order
  • Delivery: at-least-once — duplicates possible
  • Use: most workloads, high-volume processing
  • Cost: lower
  • Good for: email notifications, log processing, batch jobs, analytics

🔷 FIFO Queue (name ends in .fifo)

  • Throughput: 300 msg/sec (3000 with batching)
  • Order: strict FIFO per MessageGroupId
  • Delivery: exactly-once within 5-minute dedup window
  • Use: financial transactions, inventory, sequential workflows
  • Cost: ~10% higher per million requests
  • Good for: payments, stock trading, order state machines

When Order Matters — Bank Transfer Example

❌ Standard — wrong order possible
  • Sent: Deposit(+500) → Withdraw(-200) → Deposit(+100)
  • Received: Withdraw(-200) → Deposit(+500) → Deposit(+100)
  • Balance goes: $0 → -$200 (overdraft!) → $300 → $400
  • Overdraft fee incorrectly charged
✅ FIFO — exact order guaranteed
  • Sent: Deposit(+500) → Withdraw(-200) → Deposit(+100)
  • Received in exact same order
  • Balance: $0 → $500 → $300 → $400
  • Correct — no overdraft

FIFO Queue Concepts

🗂️ MessageGroupId — Parallelism Within FIFO

FIFO doesn't mean ALL messages are strictly ordered globally — only within a group. Each unique MessageGroupId is its own ordered lane. Messages in different groups process in parallel.

Example: use orderId as the group. All events for order-123 are ordered (placed → paid → shipped). Orders 123, 456, 789 process simultaneously.

🔁 MessageDeduplicationId

Within a 5-minute window, any two messages with the same DeduplicationId → second is silently discarded.

Use for: safe retries when you're unsure if the first publish succeeded. Set it to a hash of the message content, or use content-based deduplication (SHA-256 of body auto-generated by SQS).

🎯 Polling, Consumers & Scaling

Consumers don't receive messages passively — they poll SQS for messages. How you poll and how many consumers you run dramatically affects cost, latency, and throughput.

Short Polling vs Long Polling

⚡ Short Polling (Default)

  • WaitTimeSeconds = 0
  • SQS checks a subset of servers immediately
  • Returns instantly — even if queue is empty
  • Empty response = empty poll = wasted API call
  • 1000 polls/hour on empty queue = $0.40 wasted
  • Higher CPU on consumer from constant looping
  • Avoid unless you need sub-second latency

✅ Long Polling (Recommended)

  • WaitTimeSeconds = 1–20 (set to 20)
  • SQS holds the connection open — waits for message
  • Returns as soon as a message arrives (or after 20s)
  • Dramatically fewer empty responses
  • Up to 95% reduction in empty poll costs
  • Lower CPU — consumer sleeps between messages
  • Always use long polling in production

Scaling Strategies

⚡ Lambda (Serverless — easiest)

  • Lambda Event Source Mapping polls SQS automatically
  • Scales from 0 to 1000 concurrent workers instantly
  • Processes up to 10 messages per invocation (batch)
  • Failed batch: entire batch returned to queue (or partial)
  • Zero infrastructure management — just write the handler
  • Ideal for: variable, spiky workloads

🐳 ECS / EC2 (Controlled scaling)

  • Containers poll SQS in a loop
  • Scale on CloudWatch metric: ApproximateNumberOfMessages
  • Target tracking: keep queue depth per consumer low
  • More control: long-running jobs, stateful processing
  • Better for: heavy jobs (>15 min Lambda limit)
  • Ideal for: ML inference, video encoding, large file processing

Lambda Event Source Mapping — How It Works

📬 SQS Queue has messages
  • Lambda service polls on your behalf (long polling)
  • You never write polling code — Lambda handles it
↓ Batch of up to 10 messages
λ Lambda invoked with a batch
  • event.Records = array of up to 10 SQS messages
  • Process all records — return success or throw
  • Success: Lambda service deletes all messages automatically
  • Exception: entire batch returns to queue (at-least-once)
↓
💡 Partial Batch Response (ReportBatchItemFailures)
  • Return only the failed messageIds in batchItemFailures
  • Successfully processed messages are deleted
  • Only failed messages return to queue
  • Avoids reprocessing already-successful messages
  • Always use this with Lambda + SQS in production

SQS Producer Service

SqsProducerService.cs 
using Amazon.SQS;
using Amazon.SQS.Model;
using System.Text.Json;

public class SqsProducerService
{
    private readonly IAmazonSQS _sqs;

    public SqsProducerService(IAmazonSQS sqs) => _sqs = sqs;

    // ── Send single message ────────────────────────
    public async Task<string> SendAsync<T>(
        string queueUrl, T payload,
        int delaySeconds = 0)
    {
        var response = await _sqs.SendMessageAsync(
            new SendMessageRequest
        {
            QueueUrl     = queueUrl,
            MessageBody  = JsonSerializer.Serialize(payload),
            DelaySeconds = delaySeconds
        });
        return response.MessageId;
    }

    // ── Send batch (up to 10 — cheaper!) ──────────
    public async Task SendBatchAsync<T>(
        string queueUrl, IEnumerable<T> payloads)
    {
        var entries = payloads
            .Select((payload, i) =>
                new SendMessageBatchRequestEntry
                {
                    Id          = Guid.NewGuid().ToString(),
                    MessageBody = JsonSerializer.Serialize(payload)
                })
            .ToList();

        // SQS allows max 10 per batch call
        foreach (var chunk in entries.Chunk(10))
        {
            var result = await _sqs.SendMessageBatchAsync(
                new SendMessageBatchRequest
            {
                QueueUrl = queueUrl,
                Entries  = chunk.ToList()
            });

            if (result.Failed.Any())
                throw new Exception(
                    $"{result.Failed.Count} messages failed to send");
        }
    }

    // ── Send to FIFO queue ─────────────────────────
    public async Task<string> SendFifoAsync<T>(
        string  queueUrl,
        T       payload,
        string  messageGroupId,
        string? deduplicationId = null)
    {
        var request = new SendMessageRequest
        {
            QueueUrl       = queueUrl,
            MessageBody    = JsonSerializer.Serialize(payload),
            MessageGroupId = messageGroupId,
        };

        // If null, SQS uses content-based deduplication
        if (deduplicationId != null)
            request.MessageDeduplicationId = deduplicationId;

        var response = await _sqs.SendMessageAsync(request);
        return response.MessageId;
    }
}

SQS Consumer Service (Long-Polling Loop)

SqsConsumerService.cs — for ECS / EC2 consumers 
public class SqsConsumerService : BackgroundService
{
    private readonly IAmazonSQS _sqs;
    private readonly string _queueUrl;
    private readonly ILogger<SqsConsumerService> _log;

    protected override async Task ExecuteAsync(
        CancellationToken stoppingToken)
    {
        while (!stoppingToken.IsCancellationRequested)
        {
            var response = await _sqs.ReceiveMessageAsync(
                new ReceiveMessageRequest
            {
                QueueUrl            = _queueUrl,
                MaxNumberOfMessages = 10,        // batch of 10
                WaitTimeSeconds     = 20,        // LONG POLLING!
                VisibilityTimeout   = 60,        // 60s to process
                AttributeNames      = ["ApproximateReceiveCount"]
            }, stoppingToken);

            foreach (var msg in response.Messages)
            {
                try
                {
                    await ProcessMessageAsync(msg, stoppingToken);

                    // SUCCESS: Delete message from queue
                    await _sqs.DeleteMessageAsync(
                        _queueUrl, msg.ReceiptHandle, stoppingToken);
                }
                catch (Exception ex)
                {
                    _log.LogError(ex,
                        "Failed to process message {Id}. Attempt {Count}.",
                        msg.MessageId,
                        msg.Attributes.GetValueOrDefault(
                            "ApproximateReceiveCount"));

                    // Don't delete — let visibility timeout expire
                    // Message will reappear for retry
                }
            }
        }
    }

    private async Task ProcessMessageAsync(
        Message msg, CancellationToken ct)
    {
        var order = JsonSerializer
            .Deserialize<OrderPlacedEvent>(msg.Body);

        // Do your business logic here...
        await Task.CompletedTask;
    }
}

Lambda Handler with Partial Batch Failure

OrderProcessorLambda.cs — production pattern 
using Amazon.Lambda.SQSEvents;
using Amazon.Lambda.Core;

public class OrderProcessorLambda
{
    // Returns SQSBatchResponse → enables partial batch failure
    public async Task<SQSBatchResponse> HandleAsync(
        SQSEvent sqsEvent, ILambdaContext ctx)
    {
        var failures = new List<SQSBatchResponse.BatchItemFailure>();

        foreach (var record in sqsEvent.Records)
        {
            try
            {
                ctx.Logger.LogInformation(
                    $"Processing: {record.MessageId}");

                var order = JsonSerializer
                    .Deserialize<OrderPlacedEvent>(record.Body);

                await ProcessOrderAsync(order!);

                // Success — do NOT add to failures list
                // Lambda will auto-delete this message
            }
            catch (Exception ex)
            {
                ctx.Logger.LogError(ex,
                    $"Failed message: {record.MessageId}");

                // Add to failures → this message goes back to queue
                // Other messages in this batch are still deleted!
                failures.Add(new() {
                    ItemIdentifier = record.MessageId });
            }
        }

        return new SQSBatchResponse
        {
            BatchItemFailures = failures
        };
    }

    private async Task ProcessOrderAsync(OrderPlacedEvent order)
    {
        // Reserve inventory, send confirmation, etc.
        await Task.CompletedTask;
    }
}

    string OrderId, string UserId, decimal Amount);

✅ Best Practices & Common Mistakes

Queue Configuration

  • Always set WaitTimeSeconds to 20 — long polling reduces cost by up to 95%
  • Set visibility timeout to 6× Lambda timeout, or 1.5× max consumer processing time
  • Always create a DLQ for every source queue — no exceptions in production
  • Set DLQ retention to 14 days — give yourself time to investigate failures
  • Enable SSE encryption on all queues containing business or sensitive data
  • For FIFO: choose content-based deduplication unless you need custom deduplication IDs

Producer Best Practices

  • Use batch send (up to 10 messages) — 10× fewer API calls, 10× cheaper
  • Include a correlationId and timestamp in every message body
  • Keep message body under 256 KB — for larger, store in S3 and send the URL
  • Design for idempotency — consumers may process the same message twice
  • Use per-message delay for scheduled processing instead of sleeping in your app
  • Validate message structure before sending — don't push bad data to the queue

Consumer Best Practices

  • For Lambda: always return SQSBatchResponse with partial failure support
  • For ECS: run as BackgroundService with IHostedService for clean shutdown
  • Never delete the message before processing is complete — delete only on success
  • Use ChangeMessageVisibility to extend timeout for long-running jobs
  • Log the ApproximateReceiveCount — track how many times a message has been retried
  • Set CloudWatch alarm on DLQ depth AND message age — both are critical

Common Mistakes to Avoid

⚠️ Top SQS Mistakes

❌ Short polling (WaitTimeSeconds = 0) — costs money on empty polls, wastes CPU
❌ Deleting message before processing — crash after delete = message lost forever
❌ Visibility timeout shorter than processing time — duplicate processing
❌ No DLQ configured — failed messages loop forever, blocking healthy messages
❌ Not using batch send/receive — paying 10× more API costs unnecessarily
❌ Lambda without partial batch failure — one bad message retries entire batch
❌ Not logging ReceiveCount — blind to which messages are repeatedly failing
❌ No CloudWatch alarm on message age — silent consumer death goes undetected for hours
