🏗️ What is AWS Copilot?

A CLI tool that makes deploying containerised apps on ECS and Fargate as simple as copilot deploy.

One-liner AWS Copilot = opinionated CLI that abstracts ECS, Fargate, ALB, VPC, IAM, CloudWatch, and CloudFormation into simple commands. Write a manifest, run copilot deploy — Copilot handles the rest.
🏢 Application

Top-level grouping for all services that belong to the same product (e.g. my-ecommerce-app). Owns the shared VPC, service discovery namespace, and pipeline.

🌍 Environment

A deployment target: test, staging, production. Each gets its own VPC, ECS cluster, and ALB provisioned automatically.

📦 Service / Job

A containerised workload — Load Balanced Web Service, Backend Service, Worker Service, Scheduled Job, or Request-Driven (App Runner).

What Copilot creates under the hood VPC + subnets + NAT gateway · ECS Cluster · Task Definition · ECS Service · ALB + Target Group · Security Groups · IAM Roles · CloudWatch Log Groups · Service Discovery · CloudFormation stacks — all from a single manifest file.

📦 Service & Job Types

Copilot offers 6 workload types — each maps to a different ECS / App Runner pattern.

🌐
Load Balanced Web Service
type: Load Balanced Web Service
Internet-facing service behind an ALB or NLB. HTTPS, path-based routing, health checks. Most common type for APIs and web apps.
⚙️
Backend Service
type: Backend Service
Private service — no public endpoint. Reachable by other services via Service Connect / service discovery. For internal APIs, data processors.
📨
Worker Service
type: Worker Service
Consumes messages from SQS queues. Copilot auto-creates the queue and dead-letter queue. Scale based on queue depth.
⏰
Scheduled Job
type: Scheduled Job
Runs on a cron or rate schedule via EventBridge. Container runs to completion and exits. For batch processing, reports, cleanup tasks.
🚀
Request-Driven Web Service
type: Request-Driven Web Service
Backed by AWS App Runner — scales to zero, pay per request. No ALB. Best for low-traffic services or variable workloads.
🏃
Static Site
type: Static Site
Host a static website on S3 + CloudFront. Copilot provisions the bucket, CloudFront distribution, and optionally a custom domain.
Service Connect vs Service Discovery Copilot supports both. Service Connect (recommended) uses ECS Service Connect — built-in service mesh with metrics and retries. Service Discovery uses Route 53 private hosted zones. Backend Services are reachable at http://<svc-name>:<port> from other services in the same environment.

📄 Manifest Files

The manifest is the single source of truth for your service — Copilot generates CloudFormation from it.

Manifest = Your Service Contract Stored at copilot/<svc-name>/manifest.yml. Commit it to git. Copilot translates it into CloudFormation stacks. Override per-environment using the environments: block.
YAML Load Balanced Web Service manifest — full example 
# copilot/order-api/manifest.yml
name: order-api
type: Load Balanced Web Service

# Container image
image:
  build: Dockerfile          # or use location: ECR URI
  port: 8080

# CPU and memory (Fargate)
cpu: 512                    # 0.5 vCPU
memory: 1024               # 1 GB

# HTTP routing
http:
  path: '/'
  healthcheck:
    path: '/health'
    healthy_threshold: 3
    interval: 10

# Auto-scaling
count:
  range: 1-10
  cooldown:
    in: 30s
    out: 60s
  cpu_percentage: 70         # scale at 70% CPU
  requests: 1000             # or by ALB request count

# Environment variables
variables:
  LOG_LEVEL: info
  NODE_ENV: production

# Secrets from SSM Parameter Store or Secrets Manager
secrets:
  DB_PASSWORD: /order-api/db-password       # SSM SecureString
  JWT_SECRET:
    from: arn:aws:secretsmanager:us-east-1:123:secret:jwt-xyz

# IAM permissions for the task role
taskdef_overrides:
  - path: "ContainerDefinitions[0].Environment[-]"
    value:
      Name: REGION
      Value: !Ref AWS::Region

# Environment-specific overrides
environments:
  production:
    cpu: 1024               # 1 vCPU in prod
    memory: 2048
    count:
      range: 3-20
      cpu_percentage: 60
  test:
    count: 1                # fixed count in test
    cpu: 256
    memory: 512
YAML Worker Service manifest — SQS consumer 
name: order-processor
type: Worker Service

image:
  build: Dockerfile
cpu: 256
memory: 512

# SQS subscription — Copilot creates queue + DLQ automatically
subscribe:
  topics:
    - name: orderPlaced
      service: order-api      # SNS topic published by order-api
  queue:
    timeout: 30s
    dead_letter:
      tries: 3               # move to DLQ after 3 failures

# Scale by queue depth
count:
  range: 0-5
  queue_delay:
    acceptable_backlog: 100
    cooldown:
      in: 60s

Key Manifest Fields

image.buildPath to Dockerfile — Copilot builds and pushes to ECR automatically
image.locationUse existing ECR URI instead of building — great for pre-built images
cpu / memoryFargate task size — valid combos: 256/512, 512/1024, 1024/2048, 2048/4096, 4096/8192…
count.rangeMin–Max for Auto Scaling. Use with cpu_percentage, memory_percentage, or requests
secretsInject SSM or Secrets Manager values as env vars at container startup — never in image
environmentsOverride any top-level key per environment — great for prod/test size differences
network.vpcImport an existing VPC instead of letting Copilot create one
storageMount EFS volumes or create S3 buckets / DynamoDB tables managed by Copilot

🌍 Environments

Each environment is an isolated deployment target with its own VPC, ECS cluster, and ALB.

One command — full environment copilot env init --name production --profile prod-aws-profile provisions a complete VPC, public/private subnets, NAT gateways, ECS cluster, ALB, security groups, and CloudWatch log groups — all via CloudFormation. Takes about 5 minutes.
🏗️ What Each Environment Gets

🌐 VPC — /16 CIDR with public + private subnets across 2 AZs

🔀 NAT Gateway — for private subnet outbound traffic

⚖️ ALB — internet-facing or internal, shared across services

🐳 ECS Cluster — Fargate capacity provider

🔒 Security Groups — ALB → service, service → service

📊 CloudWatch — log groups per service

🔧 Environment Customisation

Import VPC — use existing VPC instead of Copilot-created

Import certs — attach ACM certificate to ALB for HTTPS

Internal ALB — private environments (no internet-facing LB)

Multi-account — deploy to separate AWS accounts per env

Custom CIDR — specify VPC CIDR range

Environment Manifest (env.yml)

YAML copilot/environments/production/manifest.yml 
name: production
type: Environment

network:
  vpc:
    id: vpc-0abc123          # import existing VPC
    public_subnets:
      ids: [subnet-aaa, subnet-bbb]
    private_subnets:
      ids: [subnet-ccc, subnet-ddd]

http:
  public:
    certificates:
      - arn:aws:acm:us-east-1:123:certificate/abc   # HTTPS cert

observability:
  container_insights: true   # ECS Container Insights

🔁 CI/CD Pipelines

Copilot can generate a CodePipeline that builds, tests, and deploys across environments automatically.

One command pipeline copilot pipeline init generates a pipeline.yml and a buildspec.yml. Push to git → CodePipeline triggers → CodeBuild builds image → deploys to test → approval → deploys to production.
YAML copilot/pipeline.yml — multi-env pipeline 
name: order-api-pipeline
version: 1

source:
  provider: GitHub
  properties:
    branch: main
    repository: https://github.com/indhu/order-api
    connection_arn: arn:aws:codestar-connections:...

build:
  buildspec: copilot/buildspec.yml

stages:
  - name: test
    test_commands:
      - dotnet test --logger trx

  - name: staging
    requires_approval: false

  - name: production
    requires_approval: true   # manual approval gate

Pipeline Stages

  • 1
    SourceGit push to main branch triggers CodePipeline via CodeStar Connection (GitHub / GitLab / Bitbucket).
  • 2
    BuildCodeBuild runs buildspec.yml — builds Docker image, pushes to ECR, runs unit tests, packages Copilot artifacts.
  • 3
    Deploy to TestCopilot deploys new image to the test environment. Runs integration test commands if defined.
  • 4
    Deploy to StagingAutomatic promotion to staging. Runs E2E test commands.
  • 5
    Manual ApprovalPipeline pauses — a team member reviews and approves in CodePipeline console or via SNS notification.
  • 6
    Deploy to ProductionCopilot deploys to production with ECS rolling update (or blue/green if configured).

🔑 Secrets & Storage

Inject secrets safely and attach persistent storage — EFS, S3, DynamoDB — managed by Copilot.

🔒 Secrets Injection

Reference SSM Parameter Store or Secrets Manager values in the manifest secrets: block. Copilot grants the task role permission to read them and injects values as environment variables at container startup.

Never in image Auto IAM grants

📦 Managed Storage

Declare S3 buckets, DynamoDB tables, or RDS databases in the storage: block. Copilot creates the resource, grants the task role access, and injects the ARN/name as env vars.

Auto IAM grants Auto env vars

CLI Add a secret to SSM and reference in manifest 
# 1. Store secret in SSM via Copilot
copilot secret init \
  --name DB_PASSWORD \
  --values test=supersecret123,production=ultrasecret456

# Copilot stores as /copilot/{app}/{env}/secrets/DB_PASSWORD

# 2. Reference in manifest.yml
# secrets:
#   DB_PASSWORD: /copilot/${COPILOT_APPLICATION_NAME}/${COPILOT_ENVIRONMENT_NAME}/secrets/DB_PASSWORD

# 3. Deploy — secret injected automatically
copilot svc deploy
YAML Storage — S3 bucket + DynamoDB table in manifest 
name: order-api
type: Load Balanced Web Service

storage:
  volumes:
    # EFS volume for shared file storage
    myEFS:
      path: /var/data
      read_only: false
      efs:
        id: fs-abc123

# In copilot/order-api/addons/ — CloudFormation addon for DynamoDB
# Copilot auto-grants task role access and injects ARN as env var
YAML copilot/order-api/addons/orders-table.yml — DynamoDB addon 
Parameters:
  App:
    Type: String
  Env:
    Type: String
  Name:
    Type: String

Resources:
  OrdersTable:
    Type: AWS::DynamoDB::Table
    Properties:
      TableName: !Sub '${App}-${Env}-${Name}-orders'
      BillingMode: PAY_PER_REQUEST
      AttributeDefinitions:
        - AttributeName: orderId
          AttributeType: S
      KeySchema:
        - AttributeName: orderId
          KeyType: HASH

Outputs:
  OrdersTableName:
    Value: !Ref OrdersTable
    Export:
      Name: !Sub '${App}-${Env}-${Name}-OrdersTableName'

⌨️ Essential CLI Commands

The Copilot CLI commands you'll use every day — from init to debugging.

CLI Initial setup — app, environment, service 
# Install Copilot (macOS)
brew install aws/tap/copilot-cli

# Install (Linux)
curl -Lo copilot https://github.com/aws/copilot-cli/releases/latest/download/copilot-linux
chmod +x copilot && sudo mv copilot /usr/local/bin/copilot

# 1. Initialise application
copilot app init my-ecommerce-app

# 2. Create environment (provisions VPC, ECS cluster, ALB)
copilot env init \
  --name production \
  --profile prod-aws-profile \
  --app my-ecommerce-app

# Deploy the environment infrastructure
copilot env deploy --name production

# 3. Initialise a service (walks you through type selection)
copilot svc init \
  --name order-api \
  --svc-type "Load Balanced Web Service" \
  --dockerfile ./Dockerfile

# 4. Deploy service to environment
copilot svc deploy \
  --name order-api \
  --env production
CLI Day-to-day operations 
# View service status
copilot svc status --name order-api --env production

# Tail live logs (like CloudWatch Logs Insights)
copilot svc logs \
  --name order-api \
  --env production \
  --follow             # live tail

# Open interactive shell in a running container
copilot svc exec \
  --name order-api \
  --env production \
  --command "/bin/sh"

# Show service info (endpoint, task count, CPU/mem)
copilot svc show --name order-api

# List all services in app
copilot svc ls

# List all environments
copilot env ls

# Pause / resume a service
copilot svc pause --name order-api --env test
copilot svc resume --name order-api --env test
CLI Pipeline commands 
# Initialise a pipeline (generates pipeline.yml + buildspec.yml)
copilot pipeline init

# Deploy the pipeline (creates CodePipeline)
copilot pipeline deploy

# View pipeline status
copilot pipeline status

# Show pipeline info
copilot pipeline show
CLI Secrets, jobs, and cleanup 
# Add secret (stores in SSM per environment)
copilot secret init \
  --name API_KEY \
  --values test=key123,production=key456

# Run a one-off task (like kubectl run)
copilot task run \
  --dockerfile ./Dockerfile \
  --env production \
  --command "dotnet ef database update"

# Trigger scheduled job manually
copilot job run --name report-generator --env production

# View job logs
copilot job logs --name report-generator --env production

# Delete a service
copilot svc delete --name order-api

# Delete entire app (all envs, services, pipelines)
copilot app delete

💡 Best Practices

Production-grade Copilot setup for reliability, security, and team workflows.

✅ Commit Manifests to Git

Treat the copilot/ directory as infrastructure code. Commit all manifest files and addons. Use pull requests for config changes — full audit trail via git history.

✅ Use Environment Overrides

Set smaller CPU/memory in test and larger in production via the manifest environments: block. Keeps costs low in non-prod without separate manifests.

✅ Never Hardcode Secrets

Use copilot secret init + manifest secrets: block. Secrets stored in SSM SecureString, injected at container start — never baked into Docker images.

✅ Use Addons for Resources

Don't manually create DynamoDB, S3, RDS resources. Use Copilot addons (CloudFormation in copilot/<svc>/addons/) — Copilot auto-grants IAM and injects ARNs.

✅ Health Check First

Always define a proper http.healthcheck path in manifests. Without it, ECS may route traffic to unhealthy tasks during deployments. Use a dedicated /health endpoint.

✅ Enable Container Insights

Set observability.container_insights: true in environment manifest. Get CPU, memory, network metrics per task and per service in CloudWatch — essential for production tuning.

✅ Separate AWS Accounts per Env

Use different AWS accounts for test and production. Copilot supports multi-account via AWS profiles. Stronger blast radius isolation than separate VPCs in the same account.

✅ Use copilot task run for Migrations

Run DB migrations with copilot task run --command "dotnet ef database update" before deploying the service. Runs in the same VPC with the same secrets injected.
