🗂️ What is Amazon EFS?

Amazon Elastic File System (EFS) is a fully managed, serverless, shared network file system for EC2, ECS, EKS, Lambda, and on-premises servers. It uses the NFSv4.1/4.0 protocol and scales automatically from kilobytes to petabytes with no provisioning required — you just mount it and use it.

💡 EFS in One Sentence

EFS = a shared, elastic, fully-managed NFS drive that thousands of servers can mount simultaneously, automatically grows and shrinks as you add or delete files, and costs you only for what you actually store.

Key EFS Concepts

🗂️

File System

The EFS resource itself. Has a unique ID and DNS name. Can be mounted by thousands of clients simultaneously.

🔌

Mount Target

A network endpoint (elastic network interface) in a VPC subnet. Clients connect to the mount target in their AZ.

🚪

Access Point

Application-specific entry points that enforce a root directory and POSIX identity. Simplify access for containerised apps.

💾

Storage Classes

Standard, Standard-IA, One Zone, One Zone-IA. Lifecycle policy automatically moves infrequently accessed files to IA.

⚡

Throughput Modes

Elastic (recommended), Provisioned, or Bursting. Controls how much throughput your file system can deliver.

🔑

Encryption

AES-256 at rest (KMS) and in transit (TLS). Both can be enforced independently.

📊

Performance Modes

General Purpose (default, low latency) or Max I/O (higher throughput, higher latency). Choose at creation.

🔄

Replication

Replicate EFS file systems to another AWS region for disaster recovery with RPO in minutes.

🌐

On-Premises Access

Mount EFS from on-premises servers via AWS Direct Connect or VPN using the EFS mount helper.

EFS Key Facts

📏

Scale

Scales automatically from 0 to petabytes. No pre-provisioning, no capacity planning, no resize operations.

💰

Pricing

Pay only for data stored — no minimum fee. Standard: $0.30/GB-month. IA tier: $0.025/GB-month (92% cheaper).

🌍

Availability

Regional EFS spans all AZs in a region (Standard/Standard-IA). One Zone EFS is limited to a single AZ (cheaper).

🐧

Linux only

EFS uses NFS — Linux and macOS only. For Windows shared storage use FSx for Windows File Server instead.

🔢

Connections

Thousands of concurrent NFS clients. No connection limit per file system — scales horizontally automatically.

⏱️

Durability

Regional EFS: 99.999999999% (11 nines) durability. Data stored across multiple AZs automatically.

Common EFS Use Cases

✅ Great Fits for EFS

  • CMS and web content shared across multiple web servers (WordPress, Drupal)
  • Container persistent storage shared across ECS tasks or EKS pods
  • Home directories for hundreds of Linux users
  • Machine learning training data shared across multiple GPU instances
  • CI/CD pipeline artifacts shared across build agents
  • Lambda functions that need shared file storage between invocations

❌ Poor Fits for EFS

  • Relational databases — use EBS (block storage is faster for DB workloads)
  • Windows file shares — use FSx for Windows File Server (SMB protocol)
  • Object/blob storage — use S3 (cheaper and purpose-built for objects)
  • High-frequency trading or sub-millisecond latency — use EBS io2
  • Single-instance workloads where sharing is not needed — EBS is cheaper

Intelligent-Tiering — Automatic Cost Optimisation

📁 New file written — lands in Standard tier
  • All new files land in Standard (or One Zone if One Zone file system)
  • Immediately available for low-latency access
  • EFS tracks last-accessed timestamp for each file
↓ Lifecycle policy monitors access patterns
📦 File not accessed for N days → moved to IA tier
  • Configurable threshold: 7, 14, 30, 60, or 90 days of inactivity
  • Move is transparent — file stays at same path, same permissions
  • Storage cost drops from $0.30 → $0.025 per GB-month (92% savings)
  • Retrieval fee applies: $0.01/GB when an IA file is accessed
↓ File accessed again
🔄 Transition back to Standard (optional)
  • Enable "Transition to Primary Storage on first access" in lifecycle policy
  • File moves back to Standard tier after being read once
  • Useful for workloads with unpredictable access spikes on archived files
  • If disabled: file stays in IA but is served with slightly higher latency

⚠️ One Zone — Not for Production Critical Data

One Zone EFS stores data in a single Availability Zone. If that AZ experiences an outage, your file system is unavailable. Use One Zone for dev/test, build caches, and non-critical workloads only. For production applications, always use Regional (Standard) EFS which replicates across all AZs in the region automatically.

Bursting Throughput — Credit Model

📊 How Burst Credits Work

  • Baseline throughput = 50 KB/s per GB of storage (minimum 100 MB/s guaranteed)
  • Burst throughput = 100 MB/s (for file systems under 1 TB) or more for larger systems
  • When using less than baseline, you earn burst credits at 0.1 MB/s per GB stored
  • Credits accumulate up to 2.1 TB per TB of storage (capped)
  • When you burst above baseline, you spend credits at (burst rate − baseline rate)
  • If credits are exhausted, throughput is throttled to baseline until credits recover

⚠️ Bursting Throughput Trap for Small File Systems

A 10 GB EFS file system on Bursting mode has a baseline of only 500 KB/s. It can burst to 100 MB/s using earned credits, but a small file system earns credits slowly and depletes them quickly. If your application has continuous I/O on a small EFS, use Elastic throughput mode instead — it scales to 3 GB/s on demand with no credit model.

🏗️ EFS Architecture

EFS is a regional service. A Regional EFS file system is accessible from all AZs in a region through per-AZ mount targets. Understanding how mount targets, access points, and VPC networking work together is key to deploying EFS correctly.

🗂️ EFS Regional Architecture — Multi-AZ Access

Mount Targets — Rules & Best Practices

🔌 Mount Target Facts

  • One mount target per AZ (per VPC)
  • Each mount target gets a private IP in the subnet
  • Mount targets have their own Security Group — control which clients can connect on port 2049 (NFS)
  • DNS: fs-xxxx.efs.region.amazonaws.com resolves to the mount target IP in your AZ automatically
  • Create a mount target in every AZ where your clients run — otherwise clients cross AZs (latency + cost)

🚪 EFS Access Points

  • Application-specific entry points into the file system
  • Enforce a root directory — app only sees its subtree, not the full file system
  • Enforce POSIX user and group identity — all writes appear as a specific UID/GID regardless of the connecting client
  • Simplify container storage — each ECS task or EKS pod gets its own access point
  • Required when using EFS with Lambda

Mounting EFS on Linux

1

Install the EFS mount helper

The Amazon EFS mount helper simplifies mounting and enables TLS encryption in transit. On Amazon Linux 2023: sudo yum install -y amazon-efs-utils. On Ubuntu: install via the GitHub package.

2

Mount via DNS name with TLS

Run: sudo mount -t efs -o tls fs-0abc1234:/ /mnt/efs. The -o tls flag encrypts all data in transit. The mount helper handles TLS negotiation automatically using stunnel under the hood.

3

Persist mount in /etc/fstab

Add to /etc/fstab: fs-0abc1234:/ /mnt/efs efs _netdev,tls 0 0. The _netdev option ensures the mount waits for the network before mounting at boot — critical for EFS which requires network connectivity.

4

Mount via Access Point (for containers)

Run: sudo mount -t efs -o tls,accesspoint=fsap-0abc1234 fs-0abc1234:/ /mnt/app. The access point enforces the root directory and POSIX identity automatically, isolating the container from other parts of the file system.

EFS with ECS / EKS / Lambda

🐳 ECS Task with EFS Volume

  • Define EFS volume in the task definition with the file system ID and access point
  • Mount the volume in the container definition at a specific path
  • ECS mounts EFS before starting the container — persistent across task restarts
  • Use access points to isolate data per task family
  • Works with both EC2 and Fargate launch types

λ Lambda with EFS

  • Lambda must be in a VPC to access EFS
  • Configure an EFS access point in the Lambda function settings
  • EFS appears at /mnt/efs (or your chosen path) inside the function
  • Enables sharing large ML models or data between Lambda invocations
  • Access point required — Lambda does not mount raw EFS directly

🔐 EFS Security

EFS security operates at four layers: network (VPC, Security Groups), file system policy (resource-based IAM), access point (POSIX identity enforcement), and encryption (at rest and in transit).

Network Security

1

Mount Target Security Group

Every mount target has a Security Group. Add an inbound rule: NFS port 2049 from the Security Group of your EC2 instances, ECS tasks, or Lambda functions. Never open port 2049 to 0.0.0.0/0 — use Security Group references.

2

Client Security Group

Your EC2/ECS/Lambda Security Group needs an outbound rule on port 2049 to the mount target Security Group (or use the default allow-all outbound). Add the mount target SG as the source — not an IP range.

3

VPC-only access

EFS is only accessible from within a VPC (or via Direct Connect/VPN). There is no public endpoint for EFS — it is inherently private. Lambda functions must be VPC-attached to access EFS.

EFS File System Policy (IAM Resource Policy)

📋 What EFS File System Policies Control

  • Resource-based IAM policy attached directly to the EFS file system
  • Controls which IAM principals (roles, users, accounts) can mount and access the file system
  • Can enforce TLS-only mounting — deny any client not using aws:SecureTransport
  • Can deny root access to the file system from all clients
  • Can restrict access to specific access points only
  • Applied in addition to Security Group rules — both must allow access
EFS File System Policy — Enforce TLS and restrict to IAM role 
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "EnforceTLS",
      "Effect": "Deny",
      "Principal": { "AWS": "*" },
      "Action": "*",
      "Condition": {
        "Bool": { "aws:SecureTransport": "false" }
      }
    },
    {
      "Sid": "AllowAppRole",
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::123456789012:role/my-app-role"
      },
      "Action": [
        "elasticfilesystem:ClientMount",
        "elasticfilesystem:ClientWrite"
      ]
    }
  ]
}

Encryption

🔑 Encryption at Rest

  • AES-256 using AWS KMS — enable at file system creation
  • Cannot enable encryption on an existing unencrypted EFS — must create new and migrate data
  • AWS-managed key (free) or Customer Managed Key (CMK)
  • All files, directories, and metadata are encrypted
  • No performance impact on encryption at rest

🔒 Encryption in Transit (TLS)

  • Enabled via the EFS mount helper with -o tls mount option
  • Uses stunnel to establish a TLS 1.2+ tunnel between client and mount target
  • Enforce via file system policy: deny requests where aws:SecureTransport = false
  • Slight CPU overhead (~1–3%) for TLS encryption on the client
  • Always enable for production workloads regardless of VPC isolation

💡 Enable Both Encryption Types

Enable both encryption at rest AND in transit for all production EFS file systems. In-transit TLS protects against network interception within the VPC. At-rest encryption protects the underlying storage layer. Both are free (KMS API call costs are negligible) and required by most compliance frameworks.

🔄 EFS Lifecycle Management & Backup

EFS lifecycle management automatically moves files between storage classes based on access patterns, while AWS Backup and EFS replication protect against data loss.

Lifecycle Policy Configuration

Policy SettingOptionsRecommendation
Transition to IA7, 14, 30, 60, 90 days of no access30 days for most workloads — saves ~80% on cold files
Transition to Archive90, 180, 270, 365 days of no access180+ days for data you almost never access
Transition back from IAOn first access / NeverEnable "on first access" for workloads with random access spikes

💡 Lifecycle Cost Savings Example

A 10 TB file system where 80% of files are infrequently accessed: without lifecycle policy costs $3,000/month (all in Standard). With 30-day lifecycle policy: 2 TB in Standard ($600) + 8 TB in IA ($200) = $800/month — 73% savings. Retrieval fees on the IA tier add back a small amount, but the savings are substantial for cold-heavy file systems.

EFS Backup

🔄 AWS Backup Integration

  • Enable in the EFS console or via AWS Backup service
  • Schedules daily (or more frequent) backups automatically
  • Backup stored in a separate Backup vault — not affected by EFS deletion
  • Can restore individual files or the entire file system
  • Cross-region and cross-account copy supported
  • Cost: $0.05/GB-month for backup storage

🌍 EFS Replication

  • Replicate to another AWS region for disaster recovery
  • RPO (Recovery Point Objective) in minutes
  • One-to-one: one source EFS → one destination EFS
  • Destination is read-only until you fail over
  • Failover: stop replication → destination becomes standalone read-write
  • Cost: $0.10–$0.50/GB-month for replicated data (depends on destination region)

EFS Archive Storage Class

📦 EFS Archive — Lowest Cost Long-Term Storage

  • Price: $0.008/GB-month — 97% cheaper than EFS Standard
  • Retrieval: $0.03/GB retrieval fee + first byte latency of milliseconds to minutes
  • Use for: compliance archives, old project data, infrequently accessed logs
  • Configure via lifecycle policy: transition files to Archive after 90–365 days
  • Files in Archive still appear at their original path — access is transparent but slower

⌨️ C# SDK — EFS Code Examples

All examples use AWSSDK.ElasticFileSystem NuGet package. Install: dotnet add package AWSSDK.ElasticFileSystem

1. Create an EFS File System

C# — Create EFS File System with Lifecycle Policy 
using Amazon.ElasticFileSystem;
using Amazon.ElasticFileSystem.Model;

public class EfsService(IAmazonElasticFileSystem efs)
{
    // Create a production-ready EFS file system
    public async Task<string> CreateFileSystemAsync(string name)
    {
        var resp = await efs.CreateFileSystemAsync(new CreateFileSystemRequest
        {
            CreationToken    = Guid.NewGuid().ToString(),  // Idempotency token
            PerformanceMode  = PerformanceMode.GeneralPurpose,
            ThroughputMode   = ThroughputMode.Elastic,
            Encrypted        = true,
            Tags             = [
                new Tag { Key = "Name",        Value = name          },
                new Tag { Key = "Environment", Value = "production"  },
                new Tag { Key = "ManagedBy",   Value = "dotnet-sdk"  }
            ]
        });

        var fsId = resp.FileSystemId;
        Console.WriteLine($"Created EFS: {fsId} (state: {resp.LifeCycleState})");

        // Wait until available before adding mount targets
        await WaitUntilAvailableAsync(fsId);

        // Apply lifecycle policy: move to IA after 30 days, archive after 180
        await efs.PutLifecycleConfigurationAsync(new()
        {
            FileSystemId = fsId,
            LifecyclePolicies =
            [
                new LifecyclePolicy
                    { TransitionToIA = TransitionToIARules.AFTER_30_DAYS },
                new LifecyclePolicy
                    { TransitionToArchive = TransitionToArchiveRules.AFTER_180_DAYS },
                new LifecyclePolicy
                    { TransitionToPrimaryStorageClass = TransitionToPrimaryStorageClassRules.AFTER_1_ACCESS }
            ]
        });
        Console.WriteLine($"Lifecycle policy applied to {fsId}");
        return fsId;
    }

    // Poll until file system state is "available"
    private async Task WaitUntilAvailableAsync(string fsId, int timeoutSecs = 120)
    {
        var deadline = DateTime.UtcNow.AddSeconds(timeoutSecs);
        while (DateTime.UtcNow < deadline)
        {
            var resp = await efs.DescribeFileSystemsAsync(new()
                { FileSystemId = fsId });
            if (resp.FileSystems[0].LifeCycleState == LifeCycleState.Available) return;
            await Task.Delay(4_000);
        }
        throw new TimeoutException($"EFS {fsId} did not become available in time");
    }
}

2. Create Mount Targets and Access Points

C# — Mount Targets + Access Points 
// Create mount targets — one per subnet (AZ)
public async Task CreateMountTargetsAsync(
    string   fsId,
    string[] subnetIds,        // one per AZ e.g. ["subnet-az1", "subnet-az2", "subnet-az3"]
    string[] securityGroupIds) // e.g. ["sg-efs-mount-sg"]
{
    foreach (var subnetId in subnetIds)
    {
        var mt = await efs.CreateMountTargetAsync(new CreateMountTargetRequest
        {
            FileSystemId   = fsId,
            SubnetId       = subnetId,
            SecurityGroups = [.. securityGroupIds]
        });
        Console.WriteLine($"Mount target {mt.MountTargetId} in {mt.SubnetId}: {mt.LifeCycleState}");
    }
}

// Create an access point for a specific application / container
public async Task<string> CreateAccessPointAsync(
    string fsId,
    string rootPath,  // e.g. "/app/myservice"
    int    uid = 1001,
    int    gid = 1001)
{
    var resp = await efs.CreateAccessPointAsync(new CreateAccessPointRequest
    {
        FileSystemId = fsId,
        PosixUser    = new PosixUser { Uid = uid, Gid = gid },
        RootDirectory = new RootDirectory
        {
            Path             = rootPath,
            CreationInfo     = new CreationInfo
            {
                OwnerUid    = uid,
                OwnerGid    = gid,
                Permissions = "755"  // Creates directory if it doesn't exist
            }
        },
        Tags = [new Tag { Key = "Name", Value = $"ap-{rootPath.Replace('/', '-')}" }]
    });

    Console.WriteLine($"Access point: {resp.AccessPointId} → {rootPath} (UID:{uid})");
    return resp.AccessPointId;
}

3. Describe File Systems and Storage Usage

C# — List EFS File Systems and Storage Metrics 
// List all EFS file systems with storage breakdown
public async Task PrintFileSystemsAsync()
{
    var resp = await efs.DescribeFileSystemsAsync(new DescribeFileSystemsRequest());

    foreach (var fs in resp.FileSystems)
    {
        var name    = fs.Tags.FirstOrDefault(t => t.Key == "Name")?.Value ?? "(no name)";
        var sizeGb  = fs.SizeInBytes.Value / (1024.0 * 1024 * 1024);
        var sizeIaGb = fs.SizeInBytes.ValueInIA / (1024.0 * 1024 * 1024);
        var sizeArcGb = fs.SizeInBytes.ValueInArchive / (1024.0 * 1024 * 1024);

        Console.WriteLine($"""
            ── {name} ({fs.FileSystemId}) ──
               State:         {fs.LifeCycleState}
               Throughput:    {fs.ThroughputMode}
               Performance:   {fs.PerformanceMode}
               Total size:    {sizeGb:F2} GB
               Standard:      {sizeGb - sizeIaGb - sizeArcGb:F2} GB  @ $0.30/GB
               Infrequent:    {sizeIaGb:F2} GB  @ $0.025/GB
               Archive:       {sizeArcGb:F2} GB  @ $0.008/GB
               Encrypted:     {fs.Encrypted}
               AZ:            {fs.AvailabilityZoneName ?? "Regional"}
            """);
    }
}

// List access points for a file system
public async Task PrintAccessPointsAsync(string fsId)
{
    var resp = await efs.DescribeAccessPointsAsync(new()
        { FileSystemId = fsId });

    foreach (var ap in resp.AccessPoints)
        Console.WriteLine(
            $"  {ap.AccessPointId}  root={ap.RootDirectory.Path}  uid={ap.PosixUser?.Uid}  state={ap.LifeCycleState}");
}

✅ Best Practices & Common Mistakes

Architecture Best Practices

  • Create one mount target per AZ — clients always connect to their local AZ mount target to avoid cross-AZ data transfer charges ($0.01/GB) and latency
  • Use Access Points for every application or container workload — enforces root directory isolation and POSIX identity, prevents one service from accessing another's data
  • Use Regional EFS for production — One Zone is 47% cheaper but an AZ failure makes your data unavailable. Never use One Zone for critical production data
  • Use Elastic throughput mode — eliminates burst credit management, scales instantly, no capacity planning. Only use Provisioned if you have a very consistent, well-characterised throughput requirement
  • Mount with the EFS mount helper and TLS — the amazon-efs-utils package handles TLS, automatic reconnect, and DNS resolution to the correct AZ mount target

Cost Optimisation

  • Always enable Intelligent-Tiering lifecycle policy (30-day IA transition) — for most file systems, 60–80% of data is cold and moves to IA, reducing costs by 70%+ at no effort
  • Add Archive lifecycle tier (180+ days) for compliance data, old projects, or logs that must be retained but rarely accessed — 97% cheaper than Standard
  • Use One Zone for dev/test and build caches — 47% cheaper, AZ failure risk acceptable for non-production
  • Monitor BurstCreditBalance CloudWatch metric — if consistently hitting zero on Bursting mode, switch to Elastic throughput (better performance and pricing for active file systems)
  • Review PercentIOLimit metric — if consistently above 80% on General Purpose, consider switching to Max I/O performance mode at the next recreation

Security Best Practices

  • Enable encryption at rest at file system creation — cannot be added later without migration
  • Always mount with -o tls and enforce via file system policy (deny non-SecureTransport) — protects data in transit within the VPC
  • Use specific Security Group references on port 2049, never open to 0.0.0.0/0
  • Apply a file system policy to restrict access to specific IAM roles — prevents unauthorised mounts from other instances in the VPC
  • Enable AWS Backup for all production EFS file systems — protects against accidental file deletion which EFS itself does not prevent

Common Mistakes to Avoid

⚠️ Top EFS Mistakes

❌ Not creating mount targets in all AZs — if your EC2 instance is in AZ-2 and there's no mount target in AZ-2, it connects cross-AZ. This costs $0.01/GB in data transfer and adds latency. Create one mount target per AZ at setup.
❌ Using EFS for a single-instance database — NFS latency (1–3ms) is much higher than EBS (0.1–1ms). Never use EFS as the storage backend for MySQL, PostgreSQL, or any relational database. Use EBS instead.
❌ Not enabling lifecycle policy — without it, all files stay in Standard tier ($0.30/GB) indefinitely. Enable 30-day IA transition and save 70%+ on storage cost for any file system with cold data.
❌ Using Bursting mode for a small active file system — a 10 GB EFS on Bursting mode has only 500 KB/s baseline. It exhausts burst credits quickly and throttles. Switch to Elastic throughput for active small file systems.
❌ Not using Access Points for containerised workloads — without access points, all ECS tasks share the same file system root and can overwrite each other's data. Always use access points to isolate per-service directories.
❌ Forgetting to add _netdev to /etc/fstab — without _netdev, the OS tries to mount EFS before the network is up at boot time and the instance gets stuck. Always use _netdev,tls,_netdev in fstab entries for EFS.
❌ Not backing up with AWS Backup — EFS stores data but has no built-in protection against accidental rm -rf. AWS Backup provides file-level recovery. Enable it for every production file system.
