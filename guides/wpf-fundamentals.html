🖥️ What is WPF?

👶 Complete Beginner? Start Here

WPF stands for Windows Presentation Foundation. It is Microsoft's framework for building desktop applications that run on Windows. Think of it as the tools and rules for building a Windows app with buttons, text boxes, lists, menus — all the visual things a user sees and interacts with.

💡 Real World Analogy

Think of WPF like building a house. XAML is the architectural blueprint (what goes where, what it looks like). C# is the electrical wiring and plumbing (the logic that makes things actually work). You need both — the blueprint describes the rooms, and the wiring makes the lights turn on.

Why Was WPF Created?

Before WPF, Windows apps were built with WinForms. WinForms was simple but had big limitations — every pixel had to be drawn manually, resizing was painful, and animations were nearly impossible. WPF was introduced in 2006 with .NET 3.0 to fix all of this.

🎨

GPU Rendering

WPF uses DirectX under the hood — your graphics card renders the UI, not the CPU. Everything is smooth and scalable.

📐

Vector-Based

WPF draws everything as vectors. Resize your window to any size — text and shapes stay crisp and sharp.

🔗

Data Binding

Connect UI controls directly to your data. Change the data, the screen updates automatically — no manual refresh code.

🎭

Styles & Templates

Change how any control looks completely. A button doesn't have to look like a button — it can look like anything.

✨

Animations

Smooth animations built in — fade in, slide, rotate, resize. All declarable in XAML without complex code.

📝

XAML

Describe your UI in a readable XML-based language. Designers and developers can work on the same file.

💡 Should You Learn WPF in 2024+?

Yes — WPF is still widely used in enterprise environments (banking, healthcare, manufacturing). The XAML and MVVM patterns you learn in WPF transfer directly to MAUI, WinUI 3, and even Xamarin. It is an excellent foundation for any Windows or .NET UI development.

How a WPF App is Structured

🏗️ WPF Application Structure

The Two Files Every Window Has

📄 MainWindow.xaml — The Blueprint

  • Written in XAML (a form of XML)
  • Describes what the UI looks like and where things are placed
  • Designers can edit this without touching C# logic
  • Visual Studio shows a live preview of this file
  • Converted to C# objects automatically at compile time

⚙️ MainWindow.xaml.cs — The Logic

  • Written in C# — this is the "code-behind"
  • Contains event handlers (what happens when button is clicked)
  • Uses InitializeComponent() to load the XAML
  • Can access any named XAML element directly
  • In MVVM pattern, this file stays nearly empty

📝 XAML Basics

👶 Beginner Explanation

XAML (pronounced "zammel") is like HTML but for Windows apps. Just like HTML uses tags like <div> and <button>, XAML uses tags like <Grid> and <Button>. Each tag creates a visual element on the screen.

XAML Syntax Rules

XAML — Basic Syntax Breakdown 
<!-- 1. Every element is an XML tag -->
<Button Content="Click Me" Width="100" Height="30"/>

<!-- 2. Attributes set properties on the control -->
<TextBox
    Width="200"
    Height="30"
    Text="Type here..."
    FontSize="14"
    Foreground="DarkBlue"/>

<!-- 3. Controls can contain other controls (children) -->
<StackPanel>
    <Label Content="Name:"/>
    <TextBox Width="200"/>
    <Button  Content="Save"/>
</StackPanel>

<!-- 4. Property Element Syntax — for complex values -->
<Button>
    <Button.Background>
        <LinearGradientBrush>
            <GradientStop Color="Blue"  Offset="0"/>
            <GradientStop Color="Purple"Offset="1"/>
        </LinearGradientBrush>
    </Button.Background>
    <Button.Content>Fancy Button</Button.Content>
</Button>

The x: Namespace — Special XAML Features

🔑 What x: Prefix Means

The x: prefix gives you access to special XAML features. The most important ones you'll use every day:

  • x:Name="myButton" — gives the control a name so you can access it in C# code
  • x:Key="myStyle" — gives a resource a key so you can reference it elsewhere
  • x:Type="local:MyClass" — references a C# type
  • x:Static — references a static field or property
  • x:Null — sets a property to null
XAML — x:Name to access controls in C# 
<!-- XAML: give controls names -->
<StackPanel>
    <TextBox  x:Name="txtName"    Width="200"/>
    <TextBox  x:Name="txtEmail"   Width="200"/>
    <Button   x:Name="btnSave"    Content="Save"
              Click="btnSave_Click"/>
    <TextBlockx:Name="txtResult"  Text=""/>
</StackPanel>
C# Code-Behind — access named controls 
private void btnSave_Click(object sender, RoutedEventArgs e)
{
    // Access controls by their x:Name directly
    string name  = txtName.Text;
    string email = txtEmail.Text;

    if (string.IsNullOrWhiteSpace(name))
    {
        txtResult.Text = "Please enter a name!";
        return;
    }
    txtResult.Text = $"Saved: {name} ({email})";
}

Markup Extensions — the Curly Brace Syntax

👶 What are Curly Braces in XAML?

When you see {...} inside an attribute value in XAML, it is called a Markup Extension. It means "don't treat this as a plain string — go compute/find a value". The most common ones are {Binding}, {StaticResource}, and {DynamicResource}.

XAML — Common Markup Extensions 
<!-- StaticResource: look up a resource by key (resolved once at load) -->
<Button Background="{StaticResource PrimaryBrush}"
        Style="{StaticResource PrimaryButtonStyle}"/>

<!-- DynamicResource: same but re-evaluated if resource changes at runtime -->
<TextBlock Foreground="{DynamicResource ThemeForeground}"/>

<!-- Binding: connect to C# data (covered in depth in Tab 5) -->
<TextBox Text="{Binding UserName}"/>
<Label   Content="{Binding FullName, Mode=OneWay}"/>

<!-- x:Static: reference a static C# value -->
<TextBlock Text="{x:Static local:AppConstants.AppTitle}"/>

<!-- x:Null: explicitly set a property to null -->
<Button Background="{x:Null}"/>

<!-- RelativeSource: bind to a property of a related element -->
<TextBlock Text="{Binding Title,
    RelativeSource={RelativeSource AncestorType=Window}}"/>

XAML Namespaces

XAML — Understanding Namespace Declarations 
<Window
    <!-- Default namespace = WPF controls (Button, Grid, etc.) -->
    xmlns="http://schemas.microsoft.com/winfx/2006/xaml/presentation"

    <!-- x: namespace = XAML language features (x:Name, x:Key etc.) -->
    xmlns:x="http://schemas.microsoft.com/winfx/2006/xaml"

    <!-- d: namespace = design-time only (Visual Studio designer) -->
    xmlns:d="http://schemas.microsoft.com/expression/blend/2008"

    <!-- mc: namespace = markup compatibility -->
    xmlns:mc="http://schemas.openxmlformats.org/markup-compatibility/2006"

    <!-- local: namespace = your own C# classes in the project -->
    xmlns:local="clr-namespace:MyApp"

    x:Class="MyApp.MainWindow"
    Title="My Application"
    Width="800" Height="450">

    <Grid>
        <!-- Content goes here -->
    </Grid>
</Window>

💡 You Don't Need to Type Namespaces Manually

Visual Studio auto-generates these namespace declarations when you create a new WPF project. You only need to understand what they mean, not memorise them. The one you'll add yourself most often is xmlns:local to reference your own classes, and custom control libraries like xmlns:mahApps.

🧩 WPF Controls

👶 What is a Control?

A control is anything the user can see or interact with — a button, a text box, a checkbox, a dropdown list. WPF comes with dozens of built-in controls. You place them in XAML and they appear on screen.

💡 Visibility vs Collapsed

Visibility="Hidden" hides the control but it still takes up space in the layout. Visibility="Collapsed" hides it AND removes it from the layout — other controls move to fill the gap. Use Collapsed for showing/hiding sections of a form.

Grid — The Most Important Panel

Grid is the most used panel in WPF. It works like an HTML table — you define rows and columns, then place controls in specific cells using Grid.Row and Grid.Column attached properties.

XAML — Grid layout (login form example) 
<Grid Margin="20">
    <!-- Define 3 rows: Auto = fits content, * = fills remaining space -->
    <Grid.RowDefinitions>
        <RowDefinition Height="Auto"/>  <!-- Row 0: fits label height -->
        <RowDefinition Height="Auto"/>  <!-- Row 1: fits textbox height -->
        <RowDefinition Height="Auto"/>  <!-- Row 2: fits label height -->
        <RowDefinition Height="Auto"/>  <!-- Row 3: fits textbox height -->
        <RowDefinition Height="*"/>     <!-- Row 4: fills remaining space -->
        <RowDefinition Height="Auto"/>  <!-- Row 5: fits button height -->
    </Grid.RowDefinitions>

    <!-- Define 2 columns: Auto for labels, * for inputs -->
    <Grid.ColumnDefinitions>
        <ColumnDefinition Width="Auto"/>   <!-- Column 0: label width -->
        <ColumnDefinition Width="*"/>      <!-- Column 1: fills rest -->
    </Grid.ColumnDefinitions>

    <!-- Row 0, Column 0 -->
    <Label Grid.Row="0" Grid.Column="0"
           Content="Username:" Margin="0,0,10,5"/>

    <!-- Row 0, Column 1 -->
    <TextBox Grid.Row="0" Grid.Column="1"
             x:Name="txtUsername" Margin="0,0,0,5"/>

    <Label   Grid.Row="2" Grid.Column="0" Content="Password:"/>
    <PasswordBoxGrid.Row="2" Grid.Column="1" x:Name="txtPassword"/>

    <!-- Span across both columns using ColumnSpan -->
    <Button Grid.Row="5" Grid.Column="0"
            Grid.ColumnSpan="2"
            Content="Login"
            HorizontalAlignment="Right"
            Width="80"/>
</Grid>

🔗 Data Binding

👶 What is Data Binding?

Data binding is a connection between a UI control and a piece of data. When the data changes, the UI updates automatically. When the user types in a TextBox, the underlying data updates automatically. You don't write any code to sync them — WPF does it for you. This is one of the most powerful features of WPF.

💡 Real World Analogy

Think of a thermometer on a wall. The thermometer display (UI) is bound to the actual temperature (data). When the temperature changes, the display changes automatically — you don't manually update the number. Data binding in WPF works the same way.

Binding Modes — How Data Flows

ModeData FlowWhen UI changesWhen to Use
OneWayData → UI onlyDoes NOT update dataRead-only display (labels, text blocks)
TwoWayData ↔ UI both waysUpdates data tooForms, editable fields
OneWayToSourceUI → Data onlyUpdates data onlyRare — when data comes from UI
OneTimeData → UI once at loadNever updates againStatic content, initial values

Step 1 — Create a C# class to bind to

C# — A simple data class (Model) 
// This is the data that our UI will display and edit
public class Person
{
    public string FirstName { get; set; } = "";
    public string LastName  { get; set; } = "";
    public int    Age       { get; set; }
    public string FullName  => $"{FirstName} {LastName}";
}

Step 2 — Set DataContext in code-behind

C# — Setting DataContext (the binding source) 
public partial class MainWindow : Window
{
    private Person _person = new Person
    {
        FirstName = "Alice",
        LastName  = "Smith",
        Age       = 30
    };

    public MainWindow()
    {
        InitializeComponent();
        // DataContext = the object all bindings in this window look at
        DataContext = _person;
    }
}

Step 3 — Bind controls to properties in XAML

XAML — Binding to Person properties 
<StackPanel Margin="20">
    <!-- OneWay: just display the full name -->
    <TextBlock Text="{Binding FullName}"
               FontSize="18" FontWeight="Bold"/>

    <!-- TwoWay: user can edit, changes go back to Person object -->
    <TextBox   Text="{Binding FirstName, Mode=TwoWay,
               UpdateSourceTrigger=PropertyChanged}"
               Width="200"/>

    <TextBox   Text="{Binding LastName, Mode=TwoWay,
               UpdateSourceTrigger=PropertyChanged}"
               Width="200"/>

    <!-- Bind to Age as a number -->
    <Slider    Value="{Binding Age, Mode=TwoWay}"
               Minimum="0" Maximum="120"/>
    <TextBlock Text="{Binding Age}"/>
</StackPanel>

INotifyPropertyChanged — Making Live Updates Work

⚠️ The Missing Piece — Why UI Doesn't Update

If you run the above example, the UI won't update when the data changes in code. That's because WPF doesn't know the data changed. To make WPF listen for changes, your class must implement INotifyPropertyChanged and fire PropertyChanged whenever a property value changes.

C# — Person with INotifyPropertyChanged (live binding) 
using System.ComponentModel;
using System.Runtime.CompilerServices;

public class Person : INotifyPropertyChanged
{
    private string _firstName = "";
    private string _lastName  = "";
    private int    _age;

    public string FirstName
    {
        get => _firstName;
        set
        {
            if (_firstName == value) return;
            _firstName = value;
            OnPropertyChanged();           // Tell WPF this changed
            OnPropertyChanged(nameof(FullName)); // FullName also changed
        }
    }

    public string LastName
    {
        get => _lastName;
        set { if (_lastName != value) { _lastName = value; OnPropertyChanged(); OnPropertyChanged(nameof(FullName)); } }
    }

    public int Age
    {
        get => _age;
        set { if (_age != value) { _age = value; OnPropertyChanged(); } }
    }

    public string FullName => $"{FirstName} {LastName}";

    // INotifyPropertyChanged implementation
    public event PropertyChangedEventHandler? PropertyChanged;

    protected void OnPropertyChanged([CallerMemberName] string? name = null)
        => PropertyChanged?.Invoke(this, new PropertyChangedEventArgs(name));
}

Binding to Collections — ObservableCollection

C# — ObservableCollection for live list binding 
using System.Collections.ObjectModel;

public partial class MainWindow : Window
{
    // ObservableCollection notifies WPF when items are added/removed
    public ObservableCollection<Person> People { get; } = new();

    public MainWindow()
    {
        InitializeComponent();
        DataContext = this;  // Bind to the Window itself

        People.Add(new Person { FirstName = "Alice", Age = 30 });
        People.Add(new Person { FirstName = "Bob",   Age = 25 });
    }

    private void AddPerson_Click(object sender, RoutedEventArgs e)
    {
        // ListBox updates automatically — no code needed to refresh UI
        People.Add(new Person { FirstName = "New Person", Age = 20 });
    }
}

Value Converters — Transform Data for Display

C# — BooleanToVisibilityConverter example 
// Converts true → Visible, false → Collapsed
public class BoolToVisibilityConverter : IValueConverter
{
    public object Convert(object value, Type targetType,
                          object parameter, CultureInfo culture)
    {
        return value is true ? Visibility.Visible : Visibility.Collapsed;
    }

    public object ConvertBack(object value, Type targetType,
                              object parameter, CultureInfo culture)
    {
        return value is Visibility.Visible;
    }
}
XAML — Using the converter 
<Window.Resources>
    <local:BoolToVisibilityConverter x:Key="BoolToVis"/>
</Window.Resources>

<!-- Panel visible only when IsLoggedIn = true -->
<StackPanel Visibility="{Binding IsLoggedIn,
             Converter={StaticResource BoolToVis}}">
    <TextBlock Text="Welcome back!"/>
</StackPanel>

<!-- Note: WPF has a built-in BooleanToVisibilityConverter too -->
<Window.Resources>
    <BooleanToVisibilityConverter x:Key="BoolToVis"/>
</Window.Resources>

🎨 Styles & Templates

👶 Styles = CSS for WPF

If you know CSS for web development, WPF Styles work the same way. A Style is a collection of property settings you define once and apply to many controls. Instead of setting FontSize="14" Foreground="Navy" Padding="8" on every button, you define a Style once and just say Style="{StaticResource PrimaryButton}".

Basic Style

XAML — Defining and using a Style 
<Window.Resources>

    <!-- Named style — apply it manually with Style="{StaticResource...}" -->
    <Style x:Key="PrimaryButtonStyle" TargetType="Button">
        <Setter Property="Background"  Value="#5B3FD4"/>
        <Setter Property="Foreground"  Value="White"/>
        <Setter Property="FontSize"    Value="14"/>
        <Setter Property="FontWeight"  Value="SemiBold"/>
        <Setter Property="Padding"     Value="16,8"/>
        <Setter Property="BorderThickness" Value="0"/>
        <Setter Property="Cursor"     Value="Hand"/>
    </Style>

    <!-- Implicit style — auto applied to ALL TextBoxes (no x:Key) -->
    <Style TargetType="TextBox">
        <Setter Property="FontSize"        Value="13"/>
        <Setter Property="Padding"         Value="6,4"/>
        <Setter Property="BorderBrush"     Value="#AAAACC"/>
        <Setter Property="BorderThickness" Value="1"/>
    </Style>

</Window.Resources>

<!-- Use the named style -->
<Button Content="Save" Style="{StaticResource PrimaryButtonStyle}"/>
<Button Content="Cancel" Style="{StaticResource PrimaryButtonStyle}"/>

<!-- TextBox style applied automatically -->
<TextBox Width="200"/>

Style Triggers — React to State Changes

XAML — Style with Triggers (hover effect, disabled state) 
<Style x:Key="HoverButtonStyle" TargetType="Button">
    <Setter Property="Background"  Value="#5B3FD4"/>
    <Setter Property="Foreground"  Value="White"/>
    <Setter Property="Opacity"    Value="1"/>

    <Style.Triggers>
        <!-- Mouse hover: darken background -->
        <Trigger Property="IsMouseOver" Value="True">
            <Setter Property="Background" Value="#3a20a0"/>
        </Trigger>

        <!-- Pressed: slightly smaller scale (via render transform) -->
        <Trigger Property="IsPressed" Value="True">
            <Setter Property="Background" Value="#2a1070"/>
        </Trigger>

        <!-- Disabled: grey out -->
        <Trigger Property="IsEnabled" Value="False">
            <Setter Property="Background" Value="#AAAAAA"/>
            <Setter Property="Opacity"    Value="0.6"/>
        </Trigger>

        <!-- DataTrigger: trigger based on a binding value -->
        <DataTrigger Binding="{Binding HasErrors}" Value="True">
            <Setter Property="Background" Value="#C00000"/>
        </DataTrigger>
    </Style.Triggers>
</Style>

⚡ Events & Commands

👶 Events vs Commands

An Event is something that happens — a button click, a key press, mouse movement. You write C# code to react to events. A Command is a more advanced pattern — instead of wiring events directly to code-behind, you bind a button to a command object on your ViewModel. Commands are used heavily in MVVM (covered in Part 2).

Routed Events — WPF's Special Event System

📡 What Makes WPF Events Different

WPF uses Routed Events which travel through the visual tree. There are three types:

  • Bubbling — starts at source, travels UP to root. Example: Click on a button inside a Grid — the Grid also receives the Click event.
  • Tunneling — starts at root, travels DOWN to source. Preview events (PreviewMouseDown, PreviewKeyDown) are tunneling — parent sees it first.
  • Direct — only fires on the source element. Like regular .NET events.
XAML — Common event wiring 
<StackPanel>
    <!-- Button events -->
    <Button Content="Save"
            Click="SaveButton_Click"/>

    <!-- TextBox events -->
    <TextBox TextChanged="TextBox_TextChanged"
             GotFocus="TextBox_GotFocus"
             LostFocus="TextBox_LostFocus"
             KeyDown="TextBox_KeyDown"/>

    <!-- Mouse events -->
    <Border Width="100" Height="100"
            Background="LightBlue"
            MouseEnter="Border_MouseEnter"
            MouseLeave="Border_MouseLeave"
            MouseLeftButtonDown="Border_Click"/>

    <!-- Window-level keyboard: PreviewKeyDown intercepts before child gets it -->
    <Grid PreviewKeyDown="Grid_PreviewKeyDown"/>
</StackPanel>
C# — Event handler patterns 
// Button click — simplest event
private void SaveButton_Click(object sender, RoutedEventArgs e)
{
    // sender = the Button that was clicked
    var button = (Button)sender;
    button.IsEnabled = false;      // disable after click
    SaveData();
    button.IsEnabled = true;
}

// TextChanged — fires on every keystroke
private void TextBox_TextChanged(object sender, TextChangedEventArgs e)
{
    var tb = (TextBox)sender;
    // Validate as user types
    bool valid = tb.Text.Length >= 3;
    tb.Background = valid ? Brushes.White : Brushes.LightPink;
}

// KeyDown — respond to specific keys
private void TextBox_KeyDown(object sender, KeyEventArgs e)
{
    if (e.Key == Key.Enter)
    {
        SaveButton_Click(this, new RoutedEventArgs());
        e.Handled = true;  // stops event bubbling further
    }
    if (e.Key == Key.Escape)
        Close();
}

// MouseEnter/Leave — hover effects in code
private void Border_MouseEnter(object sender, MouseEventArgs e)
    => ((Border)sender).Background = Brushes.SkyBlue;

private void Border_MouseLeave(object sender, MouseEventArgs e)
    => ((Border)sender).Background = Brushes.LightBlue;

ICommand — Commands for MVVM

C# — RelayCommand implementation (reusable) 
// A reusable RelayCommand class — put this in your project
public class RelayCommand : ICommand
{
    private readonly Action<object?>    _execute;
    private readonly Func<object?, bool>? _canExecute;

    public RelayCommand(Action<object?> execute,
                        Func<object?, bool>? canExecute = null)
    {
        _execute    = execute;
        _canExecute = canExecute;
    }

    // WPF calls this to know if button should be enabled/disabled
    public bool CanExecute(object? parameter)
        => _canExecute?.Invoke(parameter) ?? true;

    // WPF calls this when button is clicked
    public void Execute(object? parameter)
        => _execute(parameter);

    public event EventHandler? CanExecuteChanged
    {
        add    => CommandManager.RequerySuggested += value;
        remove => CommandManager.RequerySuggested -= value;
    }
}
C# + XAML — Using RelayCommand in a ViewModel 
// ViewModel with a SaveCommand
public class FormViewModel : INotifyPropertyChanged
{
    private string _name = "";
    public string Name
    {
        get => _name;
        set { _name = value; OnPropertyChanged(); }
    }

    // Command — button binds to this instead of Click event
    public ICommand SaveCommand { get; }

    public FormViewModel()
    {
        SaveCommand = new RelayCommand(
            execute:    _ => Save(),
            canExecute: _ => !string.IsNullOrWhiteSpace(Name) // disabled when empty
        );
    }

    private void Save() => MessageBox.Show($"Saved: {Name}");
    public event PropertyChangedEventHandler? PropertyChanged;
    protected void OnPropertyChanged([CallerMemberName] string? n = null)
        => PropertyChanged?.Invoke(this, new PropertyChangedEventArgs(n));
}
XAML — Bind Button to Command 
<!-- Button auto-disabled when CanExecute returns false -->
<StackPanel>
    <TextBox  Text="{Binding Name, UpdateSourceTrigger=PropertyChanged}"
              Width="200"/>
    <Button   Content="Save"
              Command="{Binding SaveCommand}"/>
</StackPanel>

📁 Resources & Resource Dictionaries

👶 What are Resources?

Resources are reusable objects you define once and use anywhere in your app. Colours, brushes, styles, templates, strings — anything can be a resource. Instead of writing Background="#5B3FD4" on 20 buttons, you define one resource called PrimaryColor and reference it everywhere. When you want to change the colour, you change it in one place.

Where to Define Resources

🪟 Window.Resources
  • Accessible to all controls in this window only
  • Define inside <Window.Resources>
  • Good for window-specific styles
📦 Control.Resources
  • Accessible only inside that control and its children
  • e.g. <Grid.Resources>
  • Good for localised styles
↑ More specific = higher priority
🌍 App.xaml Application.Resources — Global (recommended)
  • Accessible from EVERY window and control in the entire app
  • Best place for colours, global styles, converters
  • Define inside <Application.Resources> in App.xaml
XAML — App.xaml with global resources 
<Application x:Class="MyApp.App"
             xmlns="http://schemas.microsoft.com/winfx/2006/xaml/presentation"
             xmlns:x="http://schemas.microsoft.com/winfx/2006/xaml"
             StartupUri="MainWindow.xaml">
    <Application.Resources>

        <!-- Colour resources -->
        <Color  x:Key="PrimaryColor">#5B3FD4</Color>
        <Color  x:Key="AccentColor">#2A7ABF</Color>

        <!-- SolidColorBrush resources (use these for Background/Foreground) -->
        <SolidColorBrush x:Key="PrimaryBrush"
                         Color="{StaticResource PrimaryColor}"/>
        <SolidColorBrush x:Key="AccentBrush"
                         Color="{StaticResource AccentColor}"/>
        <SolidColorBrush x:Key="DangerBrush"  Color="#C00000"/>

        <!-- String resources -->
        <sys:String x:Key="AppTitle">My Awesome App</sys:String>

        <!-- Global font size -->
        <sys:Double x:Key="BaseFontSize">13</sys:Double>

        <!-- Converters -->
        <BooleanToVisibilityConverter x:Key="BoolToVis"/>

        <!-- Merge in a separate ResourceDictionary file -->
        <ResourceDictionary.MergedDictionaries>
            <ResourceDictionary Source="Themes/ButtonStyles.xaml"/>
            <ResourceDictionary Source="Themes/TextStyles.xaml"/>
        </ResourceDictionary.MergedDictionaries>

    </Application.Resources>
</Application>

C# — MainWindow.xaml.cs (complete code-behind) 
using System.Collections.ObjectModel;
using System.Windows;

namespace ContactManager;

public partial class MainWindow : Window
{
    private readonly ObservableCollection<Contact> _contacts = new();

    public MainWindow()
    {
        InitializeComponent();

        // Seed some sample data
        _contacts.Add(new Contact { Name="Alice Smith", Email="alice@example.com", Phone="07700000001" });
        _contacts.Add(new Contact { Name="Bob Jones",  Email="bob@example.com",   Phone="07700000002" });

        // Set the DataContext of the ListView to the collection
        lvContacts.DataContext = _contacts;
    }

    private void AddContact_Click(object sender, RoutedEventArgs e)
    {
        // Validate
        if (string.IsNullOrWhiteSpace(txtName.Text))
        {
            txtError.Text = "Name is required.";
            txtName.Focus();
            return;
        }

        // Add the new contact — ListView updates automatically
        _contacts.Add(new Contact
        {
            Name  = txtName.Text.Trim(),
            Email = txtEmail.Text.Trim(),
            Phone = txtPhone.Text.Trim()
        });

        // Clear the form
        txtName.Text  = "";
        txtEmail.Text = "";
        txtPhone.Text = "";
        txtError.Text = "";
        txtName.Focus();
    }

    private void DeleteContact_Click(object sender, RoutedEventArgs e)
    {
        if (lvContacts.SelectedItem is Contact selected)
        {
            var result = MessageBox.Show(
                $"Delete {selected.Name}?",
                "Confirm Delete",
                MessageBoxButton.YesNo,
                MessageBoxImage.Warning);

            if (result == MessageBoxResult.Yes)
                _contacts.Remove(selected);
        }
        else
        {
            MessageBox.Show("Please select a contact to delete.");
        }
    }
}
