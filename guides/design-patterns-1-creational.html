🏗️ What Are Design Patterns?

👶 Start Here — What is a Design Pattern?

A design pattern is a proven solution to a common programming problem. Think of it like a recipe — you don't invent pasta from scratch every time, you follow a recipe that experts refined over years. Design patterns are recipes that experienced developers discovered work well for specific types of problems. They give us a shared vocabulary and save us from reinventing the wheel.

The Three Categories

🏗️

Creational (This Guide)

How objects are created. Control object creation so your code doesn't depend on specific classes.

🔩

Structural (Part 2)

How objects are composed together. Organise classes and objects into larger structures.

💬

Behavioural (Part 3)

How objects communicate. Define how objects interact and distribute responsibility.

The 5 Creational Patterns — At a Glance

1️⃣
Singleton
Ensure a class has only ONE instance ever created, and provide a global point of access to it.
"Only one instance should ever exist"
🏭
Factory Method
Define an interface for creating an object, but let subclasses decide which class to instantiate.
"Let subclasses decide what to create"
🏪
Abstract Factory
Create families of related objects without specifying their concrete classes.
"Create related objects that belong together"
🔨
Builder
Construct complex objects step by step. Separate construction from representation.
"Build a complex object step by step"
📋
Prototype
Create new objects by copying (cloning) an existing object.
"Create by copying an existing object"

Why Creational Patterns?

❌ Without Creational Patterns

  • Code is full of new ConcreteClass() everywhere
  • Changing a class means hunting through every file that uses new
  • Testing is hard — can't swap real objects for fakes
  • Object creation logic mixed with business logic
  • Tight coupling between caller and the created class

✅ With Creational Patterns

  • Creation logic is centralised in one place
  • Caller doesn't know or care what concrete class it gets
  • Easy to swap implementations — great for testing
  • Clean separation of creation from use
  • Flexible, extensible object creation

How Patterns Connect to SOLID

🔗 SOLID Connections

Design patterns are SOLID principles in action:

  • Factory Method → Open/Closed (add new types by adding new classes)
  • Abstract Factory → Dependency Inversion (depend on abstraction, not concrete factory)
  • Builder → Single Responsibility (separate construction from representation)
  • Singleton → Used carefully with Dependency Inversion (inject the singleton)
  • Prototype → Open/Closed (add new cloneable types without changing client)

1️⃣ Singleton Pattern

👶 In Plain English

The Singleton pattern ensures that a class creates exactly one instance of itself — ever. No matter how many times you ask for it, you always get back the same object. It also provides a single global point of access to that instance.

💡 Real World Analogy

A country has one government. No matter who you ask — a citizen, a tourist, a business — they all refer to the same government. You can't create a second government for the same country. The government is a Singleton. Similarly, your application's logger or configuration object should exist only once.

The Problem Without Singleton

PROBLEM C# — Multiple instances cause inconsistency
// ❌ Without Singleton — multiple Logger instances created accidentally
public class Logger
{
    private List<string> _logs = new();

    public void Log(string message) => _logs.Add(message);
    public List<string> GetLogs() => _logs;
}

// In OrderService:
var logger1 = new Logger();   // creates instance A
logger1.Log("Order placed");

// In PaymentService:
var logger2 = new Logger();   // creates instance B — different object!
logger2.Log("Payment processed");

// logger1.GetLogs() = ["Order placed"]
// logger2.GetLogs() = ["Payment processed"]
// ❌ Logs are split across two separate instances — can't see the full picture

Classic Singleton Implementation

C# — Thread-safe Singleton (recommended)
public sealed class Logger
{
    // Step 1: Private static instance — holds the one and only instance
    private static readonly Lazy<Logger> _instance =
        new Lazy<Logger>(() => new Logger());

    // Step 2: Private constructor — prevents anyone calling new Logger()
    private Logger()
    {
        _logs = new List<string>();
    }

    // Step 3: Public static property — the only way to get the instance
    public static Logger Instance => _instance.Value;

    // The actual logger functionality
    private readonly List<string> _logs;
    private readonly object _lock = new();

    public void Log(string message)
    {
        lock (_lock)
        {
            var entry = $"[{DateTime.Now:HH:mm:ss}] {message}";
            _logs.Add(entry);
            Console.WriteLine(entry);
        }
    }

    public IReadOnlyList<string> GetLogs() => _logs.AsReadOnly();
}

// Usage — always the SAME instance
Logger.Instance.Log("Order placed");     // from OrderService
Logger.Instance.Log("Payment processed"); // from PaymentService

// Both calls go to the SAME list
var allLogs = Logger.Instance.GetLogs();
// ["[10:01:01] Order placed", "[10:01:02] Payment processed"] ✅

// Proof it's the same instance:
bool same = object.ReferenceEquals(Logger.Instance, Logger.Instance);
// same = true ✅

Why Lazy<T>? — Thread Safety Explained

🔒 The Threading Problem

Without thread safety, if two threads both check "does the instance exist?" simultaneously and both see "no", they both create an instance — breaking the Singleton guarantee. Lazy<T> handles this automatically with .NET's built-in thread-safe initialisation. It creates the instance only once, lazily (on first access), and is guaranteed thread-safe.

Singleton with Dependency Injection (Modern C#)

C# — Singleton via .NET DI (preferred in modern apps)
// In modern ASP.NET / .NET apps — use the DI container instead of static Singleton
// This avoids the testability problems of the static pattern

// Define an interface (for testability)
public interface IAppLogger
{
    void Log(string message);
    IReadOnlyList<string> GetLogs();
}

public class AppLogger : IAppLogger
{
    private readonly List<string> _logs = new();

    public void Log(string message) => _logs.Add($"[{DateTime.Now:T}] {message}");
    public IReadOnlyList<string> GetLogs() => _logs.AsReadOnly();
}

// Register as Singleton in Program.cs / Startup.cs
builder.Services.AddSingleton<IAppLogger, AppLogger>();
// .NET DI ensures only ONE AppLogger is ever created — same as Singleton pattern
// BUT: now you can inject a fake IAppLogger in tests ✅

// Any class that needs it — inject via constructor
public class OrderService
{
    private readonly IAppLogger _logger;
    public OrderService(IAppLogger logger) => _logger = logger;

    public void PlaceOrder(Order order)
    {
        _logger.Log($"Order {order.Id} placed");
    }
}

When to Use / When NOT to Use

✅ Use Singleton When

  • Logging — one central log store
  • Configuration — app settings loaded once
  • Cache — shared in-memory cache
  • Connection pool — shared database connections
  • Device drivers — one interface to hardware

❌ Avoid Singleton When

  • The object holds mutable state that differs per user/request
  • You need multiple configurations (multi-tenant apps)
  • It makes unit testing harder (prefer DI instead)
  • The "singleton" just ends up being a disguised global variable

⚠️ Singleton is Considered an Anti-Pattern by Some

Classic static Singleton (with Instance property) introduces hidden global state and makes testing difficult. In modern C# apps, prefer registering your class as a singleton via the DI container (AddSingleton). You get the same "one instance" behaviour with full testability and no static coupling.

SOLID: D — Use DI to inject the singleton instead of static access S — Singleton handles one concern only

🏭 Factory Method Pattern

👶 In Plain English

Instead of using new to create objects directly, you call a factory method that creates the object for you. The caller doesn't need to know which specific class gets created — it just asks for "a notification" and the factory decides whether to create an EmailNotification, SmsNotification, or PushNotification based on the situation.

💡 Real World Analogy

When you go to a staffing agency and say "I need a developer", you don't create the developer yourself. The agency (factory) decides whether to send you a junior, senior, or contractor based on your requirements. You just asked for "a developer" — the factory handled the specific type. Your business doesn't care about the hiring process, only the result.

The Problem Without Factory Method

PROBLEM C# — Hardcoded new() everywhere is inflexible
// ❌ Without Factory Method
// Adding a new notification type means editing OrderService
// OrderService knows too much about concrete notification classes

public class OrderService
{
    public void PlaceOrder(Order order, string notifyVia)
    {
        // ❌ OrderService must know all notification types
        if (notifyVia == "email")
        {
            var notif = new EmailNotification(); // ❌ hardcoded new
            notif.Send($"Order {order.Id} placed");
        }
        else if (notifyVia == "sms")
        {
            var notif = new SmsNotification();   // ❌ hardcoded new
            notif.Send($"Order {order.Id} placed");
        }
        // ❌ Adding "push" notification? Edit OrderService!
    }
}

Factory Method Solution

C# — Factory Method pattern
// ✅ FACTORY METHOD PATTERN

// Step 1: Define what all notifications must do (abstraction)
public interface INotification
{
    void Send(string message);
}

// Step 2: Concrete implementations
public class EmailNotification : INotification
{
    public void Send(string message)
        => Console.WriteLine($"📧 EMAIL: {message}");
}

public class SmsNotification : INotification
{
    public void Send(string message)
        => Console.WriteLine($"📱 SMS: {message}");
}

public class PushNotification : INotification
{
    public void Send(string message)
        => Console.WriteLine($"🔔 PUSH: {message}");
}

// Step 3: The Factory — ONE place that handles creation
public class NotificationFactory
{
    public static INotification Create(string type)
        => type.ToLower() switch
        {
            "email" => new EmailNotification(),
            "sms"   => new SmsNotification(),
            "push"  => new PushNotification(),
            _       => throw new ArgumentException($"Unknown type: {type}")
        };
}

// Step 4: OrderService now depends ONLY on INotification
public class OrderService
{
    public void PlaceOrder(Order order, string notifyVia)
    {
        // ✅ Doesn't know or care what concrete class it gets
        INotification notification = NotificationFactory.Create(notifyVia);
        notification.Send($"Order {order.Id} confirmed!");
    }
}

// Usage:
var service = new OrderService();
service.PlaceOrder(order, "email");  // 📧 EMAIL: Order 42 confirmed!
service.PlaceOrder(order, "sms");    // 📱 SMS: Order 42 confirmed!

// ✅ Adding "WhatsApp" notification?
// 1. Create WhatsAppNotification : INotification
// 2. Add "whatsapp" case to factory
// 3. OrderService — ZERO changes needed

Real-World Example — Payment Processor Factory

C# — Payment processor factory (production pattern)
public interface IPaymentProcessor
{
    Task<bool> ChargeAsync(decimal amount, string currency);
    string ProviderName { get; }
}

public class StripeProcessor : IPaymentProcessor
{
    public string ProviderName => "Stripe";
    public async Task<bool> ChargeAsync(decimal amount, string currency)
    {
        // Stripe API call...
        await Task.Delay(100);
        Console.WriteLine($"Stripe charged {amount} {currency}");
        return true;
    }
}

public class PayPalProcessor : IPaymentProcessor
{
    public string ProviderName => "PayPal";
    public async Task<bool> ChargeAsync(decimal amount, string currency)
    {
        await Task.Delay(150);
        Console.WriteLine($"PayPal charged {amount} {currency}");
        return true;
    }
}

// Factory with DI-friendly interface
public interface IPaymentProcessorFactory
{
    IPaymentProcessor Create(string provider);
}

public class PaymentProcessorFactory : IPaymentProcessorFactory
{
    public IPaymentProcessor Create(string provider)
        => provider.ToUpper() switch
        {
            "STRIPE" => new StripeProcessor(),
            "PAYPAL" => new PayPalProcessor(),
            _        => throw new NotSupportedException($"Provider '{provider}' not supported")
        };
}

// Checkout service — only knows about the interface
public class CheckoutService
{
    private readonly IPaymentProcessorFactory _factory;
    public CheckoutService(IPaymentProcessorFactory factory) => _factory = factory;

    public async Task<bool> ProcessPaymentAsync(string provider, decimal amount)
    {
        var processor = _factory.Create(provider);
        Console.WriteLine($"Using {processor.ProviderName}");
        return await processor.ChargeAsync(amount, "GBP");
    }
}

When to Use / When NOT to Use

  • When you don't know at compile time which class to instantiate
  • When you want to centralise all object creation in one place
  • When adding new types should not require changing existing code (OCP)
  • When writing tests — inject a factory that returns mock objects
SOLID: O — Add new types without modifying factory callers D — Callers depend on INotification, not EmailNotification S — Factory handles creation; service handles business logic

🏪 Abstract Factory Pattern

👶 In Plain English

Abstract Factory is like a factory of factories. It creates families of related objects that work together, without knowing their specific classes. Imagine you need a Dark Theme for your app — you need a dark button, dark textbox, and dark checkbox. The DarkThemeFactory creates all of them as a matching family. Switch to Light Theme? Use LightThemeFactory — same structure, different objects.

💡 Real World Analogy

Think of IKEA furniture collections. The "KALLAX" collection has a matching shelf, drawer unit, and door insert — they all fit together stylistically. The "BILLY" collection has its own matching bookcase, extension, and door. You pick a collection (Abstract Factory) and get a family of items that work together. You don't mix KALLAX with BILLY — they're separate families.

Factory Method vs Abstract Factory

🏭 Factory Method

  • Creates one product type
  • Subclasses decide which concrete class
  • Example: NotificationFactory creates one Notification

🏪 Abstract Factory

  • Creates families of related products
  • Entire factory is swapped for a different family
  • Example: ThemeFactory creates Button + TextBox + CheckBox (all matching)

UI Theme Example — Abstract Factory

C# — UI Theme Abstract Factory
// ── Step 1: Abstract Products — what each UI control must support ──
public interface IButton   { void Render(); }
public interface ITextBox  { void Render(); }
public interface ICheckBox { void Render(); }

// ── Step 2: Concrete Products — Dark Theme family ──────────────────
public class DarkButton   : IButton   { public void Render() => Console.WriteLine("[Dark Button]"); }
public class DarkTextBox  : ITextBox  { public void Render() => Console.WriteLine("[Dark TextBox]"); }
public class DarkCheckBox : ICheckBox { public void Render() => Console.WriteLine("[Dark CheckBox]"); }

// ── Concrete Products — Light Theme family ─────────────────────────
public class LightButton   : IButton   { public void Render() => Console.WriteLine("[Light Button]"); }
public class LightTextBox  : ITextBox  { public void Render() => Console.WriteLine("[Light TextBox]"); }
public class LightCheckBox : ICheckBox { public void Render() => Console.WriteLine("[Light CheckBox]"); }

// ── Step 3: Abstract Factory — defines the factory contract ────────
public interface IUIFactory
{
    IButton   CreateButton();
    ITextBox  CreateTextBox();
    ICheckBox CreateCheckBox();
}

// ── Step 4: Concrete Factories — each creates a matching family ────
public class DarkThemeFactory : IUIFactory
{
    public IButton   CreateButton()   => new DarkButton();
    public ITextBox  CreateTextBox()  => new DarkTextBox();
    public ICheckBox CreateCheckBox() => new DarkCheckBox();
}

public class LightThemeFactory : IUIFactory
{
    public IButton   CreateButton()   => new LightButton();
    public ITextBox  CreateTextBox()  => new LightTextBox();
    public ICheckBox CreateCheckBox() => new LightCheckBox();
}

// ── Step 5: Application uses the factory — doesn't know Dark or Light
public class LoginForm
{
    private readonly IButton   _submitBtn;
    private readonly ITextBox  _usernameBox;
    private readonly ICheckBox _rememberMe;

    // ✅ LoginForm only knows IUIFactory — not Dark or Light
    public LoginForm(IUIFactory factory)
    {
        _submitBtn   = factory.CreateButton();
        _usernameBox = factory.CreateTextBox();
        _rememberMe  = factory.CreateCheckBox();
    }

    public void Render()
    {
        _usernameBox.Render();
        _rememberMe.Render();
        _submitBtn.Render();
    }
}

// Switch entire theme by changing ONE line:
IUIFactory factory = new DarkThemeFactory();  // or LightThemeFactory
var form = new LoginForm(factory);
form.Render();
// [Dark TextBox]
// [Dark CheckBox]
// [Dark Button]

Real-World Example — Database Provider Factory

C# — Database provider abstract factory
// Create SQL Server vs SQLite families for production vs testing
public interface IDbConnection { void Open(); }
public interface IDbCommand    { void Execute(string sql); }

// SQL Server family
public class SqlConnection  : IDbConnection { public void Open()                  => Console.WriteLine("SQL Server connected"); }
public class SqlCommand     : IDbCommand    { public void Execute(string sql) => Console.WriteLine($"SQL: {sql}"); }

// SQLite family (for unit tests)
public class SqliteConnection: IDbConnection { public void Open()                  => Console.WriteLine("SQLite connected"); }
public class SqliteCommand   : IDbCommand    { public void Execute(string sql) => Console.WriteLine($"SQLite: {sql}"); }

public interface IDbFactory
{
    IDbConnection CreateConnection();
    IDbCommand    CreateCommand();
}

public class SqlServerFactory : IDbFactory
{
    public IDbConnection CreateConnection() => new SqlConnection();
    public IDbCommand    CreateCommand()    => new SqlCommand();
}

public class SqliteFactory : IDbFactory
{
    public IDbConnection CreateConnection() => new SqliteConnection();
    public IDbCommand    CreateCommand()    => new SqliteCommand();
}

// DataAccess doesn't care which DB — works with either factory
public class DataAccess
{
    private readonly IDbFactory _factory;
    public DataAccess(IDbFactory factory) => _factory = factory;

    public void RunQuery(string sql)
    {
        var conn = _factory.CreateConnection();
        var cmd  = _factory.CreateCommand();
        conn.Open();
        cmd.Execute(sql);
    }
}

// Production: SQL Server · Tests: SQLite
var prod = new DataAccess(new SqlServerFactory());
var test = new DataAccess(new SqliteFactory());
SOLID: O — Add new theme/provider by adding new factory class D — App depends on IUIFactory, not DarkThemeFactory L — Any IUIFactory can substitute for another

🔨 Builder Pattern

👶 In Plain English

The Builder pattern lets you construct complex objects step by step. Instead of a constructor with 10 parameters (most of which are optional), you chain method calls to set only what you need. The object is only created when you call Build() at the end. It is especially useful when an object has many optional settings.

💡 Real World Analogy

Ordering a custom burger. You don't say "give me burger #4" (constructor with 4th preset). You say: bread = brioche, protein = double beef, add cheese, add lettuce, no tomato, sauce = bbq. You build it step by step, choosing only what you want. The kitchen (Builder) assembles it and hands you the finished burger (Build()). The result is exactly configured to your needs.

The Problem — Constructor Hell

PROBLEM C# — Too many constructor parameters
// ❌ CONSTRUCTOR HELL — what do all these parameters mean?
var email = new Email(
    "noreply@company.com",
    "alice@example.com",
    null,
    null,
    "Your order is confirmed",
    "<h1>Order #42</h1>",
    true,
    false,
    null,
    3);
// What is the 7th parameter? What does false mean? What is 3?
// Impossible to read without looking at the constructor signature

// Also problematic — too many optional parameters:
public Email(string from, string to,
             string? cc = null, string? bcc = null,
             string subject = "", string body = "",
             bool isHtml = false, bool highPriority = false,
             List<string>? attachments = null, int retryCount = 1) { }

Builder Pattern Solution

C# — Email Builder with fluent interface
// ── Step 1: The object being built ────────────────────────────────
public class Email
{
    public string       From        { get; init; } = "";
    public string       To          { get; init; } = "";
    public string?      Cc          { get; init; }
    public string       Subject     { get; init; } = "";
    public string       Body        { get; init; } = "";
    public bool         IsHtml      { get; init; }
    public bool         HighPriority{ get; init; }
    public List<string> Attachments { get; init; } = new();
}

// ── Step 2: The Builder ────────────────────────────────────────────
public class EmailBuilder
{
    private string       _from        = "";
    private string       _to          = "";
    private string?      _cc;
    private string       _subject     = "";
    private string       _body        = "";
    private bool         _isHtml;
    private bool         _highPriority;
    private readonly List<string> _attachments = new();

    // Each method sets one property and returns THIS — enabling chaining
    public EmailBuilder From(string from)        { _from = from; return this; }
    public EmailBuilder To(string to)            { _to = to; return this; }
    public EmailBuilder Cc(string cc)            { _cc = cc; return this; }
    public EmailBuilder WithSubject(string s)    { _subject = s; return this; }
    public EmailBuilder WithBody(string b)       { _body = b; return this; }
    public EmailBuilder AsHtml()                 { _isHtml = true; return this; }
    public EmailBuilder HighPriority()           { _highPriority = true; return this; }
    public EmailBuilder Attach(string path)      { _attachments.Add(path); return this; }

    // Validate and build the final object
    public Email Build()
    {
        if (string.IsNullOrWhiteSpace(_from)) throw new InvalidOperationException("From is required");
        if (string.IsNullOrWhiteSpace(_to))   throw new InvalidOperationException("To is required");

        return new Email
        {
            From         = _from,
            To           = _to,
            Cc           = _cc,
            Subject      = _subject,
            Body         = _body,
            IsHtml       = _isHtml,
            HighPriority = _highPriority,
            Attachments  = _attachments
        };
    }
}

// ✅ Fluent usage — reads like English, crystal clear
var email = new EmailBuilder()
    .From("noreply@company.com")
    .To("alice@example.com")
    .WithSubject("Your order is confirmed")
    .WithBody("<h1>Order #42 confirmed!</h1>")
    .AsHtml()
    .HighPriority()
    .Attach("invoice.pdf")
    .Build();

Real-World Example — SQL Query Builder

C# — Query Builder (like EF Core's fluent API)
public class QueryBuilder
{
    private string  _table   = "";
    private readonly List<string> _selects    = new();
    private readonly List<string> _conditions = new();
    private string? _orderBy;
    private int?    _limit;

    public QueryBuilder From(string table)  { _table = table; return this; }
    public QueryBuilder Select(params string[] cols) { _selects.AddRange(cols); return this; }
    public QueryBuilder Where(string cond)  { _conditions.Add(cond); return this; }
    public QueryBuilder OrderBy(string col) { _orderBy = col; return this; }
    public QueryBuilder Limit(int n)        { _limit = n; return this; }

    public string Build()
    {
        var cols = _selects.Count > 0 ? string.Join(", ", _selects) : "*";
        var sql  = $"SELECT {cols} FROM {_table}";
        if (_conditions.Count > 0) sql += $" WHERE {string.Join(" AND ", _conditions)}";
        if (_orderBy != null)        sql += $" ORDER BY {_orderBy}";
        if (_limit   != null)        sql += $" LIMIT {_limit}";
        return sql;
    }
}

// Usage — reads like SQL, builds it step by step
var query = new QueryBuilder()
    .From("Orders")
    .Select("Id", "CustomerName", "Total")
    .Where("Total > 100")
    .Where("Status = 'Active'")
    .OrderBy("Total DESC")
    .Limit(50)
    .Build();
// "SELECT Id, CustomerName, Total FROM Orders WHERE Total > 100 AND Status = 'Active' ORDER BY Total DESC LIMIT 50"

When to Use Builder

  • Object has more than 4 constructor parameters — especially optional ones
  • Object construction involves validation that should happen at the end
  • You want to create different representations of the same object (HTML email vs plain text email)
  • Configuration objects — HttpClient settings, database connection options
  • Test data builders — create test objects with only the fields you care about
SOLID: S — Builder handles construction; object handles behaviour O — Add new build steps without modifying existing ones

📋 Prototype Pattern

👶 In Plain English

The Prototype pattern creates new objects by copying (cloning) an existing object instead of creating a brand new one from scratch. When creating a new object is expensive (lots of setup, database calls, calculations), or when you want a new object that starts with the same state as an existing one, cloning is the answer.

💡 Real World Analogy

In biology, a cell divides by copying itself — the new cell starts with the same DNA as the original. In a game, when you spawn 100 enemy soldiers they all start with the same configuration — health, weapons, speed — copied from a prototype soldier template. You don't reconfigure each one from scratch; you clone the template and then adjust individual differences.

Shallow Clone vs Deep Clone

🔍 Shallow Clone (MemberwiseClone)

  • Copies all value types (int, bool, struct) by value
  • Copies reference types by reference — both objects share the same nested objects
  • Fast but dangerous if nested objects are mutable
  • Use when all fields are value types or immutable

📋 Deep Clone

  • Copies everything recursively — all nested objects are also new
  • Changing the clone does NOT affect the original
  • Slower but safe for mutable nested objects
  • Use when the object has mutable reference-type fields
C# — Shallow vs Deep clone (ICloneable)
// ── Shallow Clone — fast, but shared references ───────────────────
public class Address
{
    public string Street { get; set; } = "";
    public string City   { get; set; } = "";
}

public class Person : ICloneable
{
    public string  Name    { get; set; } = "";
    public int     Age     { get; set; }
    public Address Address { get; set; } = new();

    // Shallow clone — Address is SHARED between original and clone
    public object Clone() => MemberwiseClone();
}

var original = new Person { Name = "Alice", Age = 30, Address = new Address { City = "London" } };
var shallow  = (Person)original.Clone();

shallow.Name       = "Bob";       // ✅ doesn't affect original
shallow.Address.City = "Paris";  // ❌ ALSO changes original.Address.City!
// original.Address.City = "Paris" — both share the same Address object
C# — Deep clone (safe, independent copy)
public class Person
{
    public string  Name    { get; set; } = "";
    public int     Age     { get; set; }
    public Address Address { get; set; } = new();

    // ✅ Deep clone — creates a NEW Address object too
    public Person DeepClone() => new Person
    {
        Name    = Name,
        Age     = Age,
        Address = new Address { Street = Address.Street, City = Address.City }
    };
}

var original = new Person { Name = "Alice", Age = 30, Address = new Address { City = "London" } };
var deep     = original.DeepClone();

deep.Name        = "Bob";    // ✅ doesn't affect original
deep.Address.City = "Paris"; // ✅ doesn't affect original.Address.City
// original.Name = "Alice", original.Address.City = "London" ✅

Real-World Example — Game Character Template

C# — Game enemy spawning with Prototype
public class Enemy
{
    public string         Name        { get; set; } = "";
    public int            Health      { get; set; }
    public int            Damage      { get; set; }
    public float          Speed       { get; set; }
    public List<string>  Abilities   { get; set; } = new();

    // Deep clone — new enemy starts with same config as template
    public Enemy Clone() => new Enemy
    {
        Name      = Name,
        Health    = Health,
        Damage    = Damage,
        Speed     = Speed,
        Abilities = new List<string>(Abilities)  // ✅ new list, same contents
    };
}

// Create prototype templates once
var goblinTemplate = new Enemy
{
    Name      = "Goblin",
    Health    = 50,
    Damage    = 10,
    Speed     = 1.5f,
    Abilities = { "Sneak", "Steal" }
};

var trollTemplate = new Enemy
{
    Name      = "Troll",
    Health    = 200,
    Damage    = 40,
    Speed     = 0.8f,
    Abilities = { "Smash", "Regenerate" }
};

// Spawn 100 goblins — each is a fast clone of the template
var enemies = Enumerable.Range(0, 100)
    .Select(_ =>
    {
        var e = goblinTemplate.Clone();
        e.Health += Random.Shared.Next(-10, 10);  // slight variation per enemy
        return e;
    })
    .ToList();

// Each enemy is independent — modifying one doesn't affect others or the template

Modern C# Alternative — Record with with

C# — Records make cloning trivial
// C# records have built-in non-destructive mutation with "with"
// Perfect replacement for Prototype in many modern scenarios

public record EnemyConfig(
    string Name,
    int    Health,
    int    Damage,
    float  Speed
);

var goblinProto = new EnemyConfig("Goblin", 50, 10, 1.5f);

// "with" creates a new record with some properties changed — the rest are copied
var bossGoblin = goblinProto with { Name = "Boss Goblin", Health = 200, Damage = 30 };
var fastGoblin = goblinProto with { Speed = 3.0f };

// goblinProto unchanged ✅
// bossGoblin: Name=Boss Goblin, Health=200, Damage=30, Speed=1.5f
// fastGoblin: Name=Goblin, Health=50, Damage=10, Speed=3.0f

When to Use Prototype

  • Creating a new object from scratch is expensive (complex setup, DB calls, file loading)
  • You need many similar objects that differ in only a few properties
  • You want to save and restore object states (combined with Memento pattern)
  • Game development — spawning enemies, bullets, particles from templates
  • Document editors — duplicating slides, pages, or templates
