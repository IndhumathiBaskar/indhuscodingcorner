Virtual servers in the cloud · instance types · storage · networking · security · C# SDK examples · beginner to expert

🖥️ What is Amazon EC2?

Amazon Elastic Compute Cloud (EC2) provides resizable virtual servers (called instances) in the cloud. You choose the OS, CPU, RAM, storage, and network capacity — and have root access to configure it exactly as you need. EC2 is the backbone of most AWS architectures.

💡 EC2 in One Sentence

EC2 = rent a virtual machine of any size on demand, pay by the second, and scale from 1 to thousands of instances in minutes — without owning any physical hardware.

When to Use EC2

✅ Use EC2 When You Need…

  • Full OS-level control (custom kernel, drivers, software)
  • Long-running processes (web servers, databases, APIs)
  • Custom networking and firewall rules
  • GPU compute for ML training or rendering
  • Lift-and-shift of existing on-premises applications
  • Stateful applications requiring persistent local disk

⚠️ Consider Alternatives When…

  • Short event-driven functions → use Lambda
  • Containerized microservices → use ECS / EKS
  • Simple static websites → use S3 + CloudFront
  • Managed database → use RDS / Aurora
  • No ops overhead desired → use App Runner / Fargate
  • Fully managed ML inference → use SageMaker

⚙️ EC2 Instance Types & Families

EC2 instances are grouped into families optimized for different workloads. The name format is family + generation + size — e.g. m7g.xlarge = general-purpose, 7th gen, Graviton3, xlarge (4 vCPU / 16 GB).

💡 Reading Instance Names

m7g.2xlarge → m = general purpose family · 7 = 7th generation · g = Graviton (ARM) processor · 2xlarge = 8 vCPU / 32 GB RAM. Suffixes: n = high network, d = local NVMe disk, a = AMD EPYC, i = Intel.

T-Series Burstable Instances — How CPU Credits Work

🟢 Accruing Credits (CPU < baseline)
  • CPU usage below baseline → earn CPU credits over time
  • t3.micro baseline = 10% of 1 vCPU
  • Max credit balance: 24 hours worth of credits
↕
🟠 Spending Credits (CPU > baseline)
  • CPU spikes above baseline → spend accumulated credits
  • When credits exhausted → CPU throttled to baseline %
  • T3 Unlimited mode: bursts without throttle, pay extra per vCPU-hour

⚠️ T2/T3 Unlimited Billing Surprise

T3 instances default to Unlimited mode. If your workload runs at high CPU continuously, you can receive unexpected charges for surplus CPU credits. Monitor CPUSurplusCreditBalance in CloudWatch. For consistently high CPU, use an m-series instead.

💡 Always Use gp3 (Not gp2)

gp3 is both cheaper ($0.08 vs $0.10/GB) AND lets you independently configure IOPS and throughput at no extra cost. There is no reason to use gp2 for new volumes. Migrate existing gp2 volumes to gp3 via the console with zero downtime.

EBS Snapshots — Key Facts

📸 How Snapshots Work

  • Incremental — only changed blocks stored after first snapshot
  • Stored in S3 (managed by AWS, not visible in your S3 console)
  • Can create new EBS volume from snapshot in any AZ or Region
  • Copy snapshots cross-region for DR or AMI sharing
  • Snapshots can be automated via AWS Data Lifecycle Manager

💰 Snapshot Pricing

  • $0.05/GB-month (only changed data after first full)
  • Fast Snapshot Restore: $0.75/DSU-hour per AZ enabled
  • Recycle Bin: retain deleted snapshots from $0.0075/GB-month
  • Cross-region copy incurs destination storage + data transfer cost
  • Delete old snapshots — orphaned snapshots accumulate cost silently

Every EC2 instance lives inside a VPC (Virtual Private Cloud). Networking controls how traffic reaches your instance from the internet, other AWS services, or your on-premises network.

Key Networking Concepts

🔌 Elastic Network Interface (ENI)

  • Virtual NIC attached to an instance
  • Has a private IP, optional public IP, MAC address
  • Can be detached and moved to another instance (failover pattern)
  • Multiple ENIs per instance → multi-homed networking

🌐 Elastic IP (EIP)

  • Static public IPv4 address allocated to your account
  • Free while associated with a running instance
  • $0.005/hr when not associated (idle EIP) — release unused ones
  • Can remap to another instance in seconds for failover

🔀 Enhanced Networking (ENA)

  • SR-IOV driver for higher PPS and lower jitter
  • Enabled by default on modern instances (C5, M5, R5+)
  • EFA (Elastic Fabric Adapter) for HPC MPI workloads — up to 400 Gbps
  • No extra cost — just requires the ENA driver in your AMI

🏠 VPC Placement Groups

  • Cluster: same rack → lowest latency, highest throughput
  • Spread: different racks → fault isolation, max 7/AZ
  • Partition: groups on different hardware → HDFS, Cassandra
  • No additional charge for placement groups

💡 Savings Plans vs Reserved Instances

Savings Plans are generally preferred over Reserved Instances today. They give similar discounts but are more flexible — Compute Savings Plans apply across any instance family, region, OS, and even Lambda/Fargate. You commit to a $/hour spend, not a specific instance type.

Spot Instance Best Practices

✅ Good Spot Workloads
  • ML training jobs with checkpoint/resume logic
  • CI/CD build pipelines — job restarts are acceptable
  • Video transcoding, image processing (stateless)
  • Big data analytics (Spark/EMR with task node Spot)
  • Rendering farms — frames are independently restartable
vs
❌ Bad Spot Workloads
  • Production web servers (interruption = outage)
  • Stateful databases without multi-AZ failover
  • Long transactions that can't be checkpointed
  • Jobs with hard deadlines (Spot unavailability is unpredictable)

Hidden EC2 Costs

⚠️ Costs That Surprise People

💸 Data transfer OUT to internet — first 100 GB/month free, then $0.09/GB. High-traffic sites can accumulate significant transfer costs.
💸 Idle Elastic IPs — $0.005/hr (~$3.60/month) per EIP not attached to a running instance. Delete unused EIPs.
💸 EBS volumes for stopped instances — you still pay for EBS storage even when instance is stopped. Delete if not needed.
💸 Snapshots — accumulate over time. Old snapshots of terminated instances are often forgotten and still billed.
💸 NAT Gateway — $0.045/hr + $0.045/GB. Private subnets with high internet traffic pay heavily for NAT GW.
💸 T3 Unlimited burst charges — CPU-intensive workloads on burstable instances can spike cost unexpectedly.

Cost Optimization Strategy

  • Right-size first — CloudWatch CPU metrics often show instances 10–15% utilized. Downsize before applying discounts.
  • Use Graviton (ARM) instances — 20% cheaper and often faster than equivalent x86 for modern workloads
  • Commit with Compute Savings Plans — 1-year no-upfront gets ~40% off with full flexibility
  • Use Spot for batch and dev/test — combine with Auto Scaling groups using mixed instances policy
  • Enable AWS Cost Anomaly Detection — get alerted when spending deviates from expected patterns
  • Schedule dev/test environments to stop nights/weekends — saves ~65% on non-production instances
  • Use AWS Trusted Advisor — identifies idle instances, underutilized EBS, and unused Elastic IPs automatically

🔐 EC2 Security

EC2 security is a shared responsibility: AWS secures the physical infrastructure; you secure the OS, applications, network rules, and access management.

IAM Instance Profiles — Secure AWS API Access

1

Create an IAM Role for EC2

Go to IAM → Roles → Create Role → Select "EC2" as trusted entity. Attach only the policies your app needs (principle of least privilege).

2

Attach the Role to Your Instance

At launch (or modify existing instance → Actions → Security → Modify IAM Role). No access keys stored on disk — credentials are rotated automatically by the instance metadata service.

3

Access AWS Services from Code

AWS SDKs automatically pick up credentials from the instance metadata endpoint 169.254.169.254/latest/meta-data/iam/security-credentials/. No hardcoded keys needed.

❌ Never Store AWS Access Keys on EC2

Never run aws configure on an EC2 instance and store long-term IAM access keys. If the instance is compromised, those keys are stolen. Always use IAM Instance Profiles (roles) — they auto-rotate credentials every hour via IMDSv2.

IMDSv2 — Instance Metadata Service v2

🛡️ Why IMDSv2 Matters

IMDSv1 allowed any process on the instance (including SSRF attacks) to query metadata and steal IAM credentials. IMDSv2 requires a session token (PUT request first), blocking SSRF attacks.

  • Enforce IMDSv2 at launch: set --metadata-options HttpTokens=required
  • Or enforce via AWS Config rule: ec2-imdsv2-check
  • Modern SDKs use IMDSv2 automatically

Encryption

🔑 EBS Encryption

  • AES-256, keys managed by AWS KMS
  • Enable per volume, or set account-level default encryption
  • Encrypted snapshots and AMIs — copies remain encrypted
  • No performance impact on modern instances (hardware acceleration)
  • Existing unencrypted volumes: snapshot → copy (encrypt) → restore

🔒 In-Transit Encryption

  • Use TLS/HTTPS for all application traffic
  • Enable encryption in transit for EFS mounts
  • Use AWS Certificate Manager (ACM) with ALB — free TLS certs
  • Enable TLS termination at the load balancer, not the instance
  • Use VPC endpoints for AWS services — traffic stays on AWS backbone

SSH / Remote Access Best Practices

  • Use EC2 Instance Connect or AWS Systems Manager Session Manager — no SSH port (22) needed on Security Group
  • If SSH must be open: restrict Security Group to your IP only, not 0.0.0.0/0
  • Use ED25519 key pairs (more secure than RSA-2048)
  • Store key pairs in AWS Secrets Manager, not in version control
  • Rotate key pairs periodically — remove old public keys from ~/.ssh/authorized_keys
  • For Windows RDP: never open port 3389 to the internet — use SSM Session Manager port forwarding instead

1. List Running Instances

C# — List EC2 Instances 
using Amazon.EC2;
using Amazon.EC2.Model;

var ec2 = new AmazonEC2Client();  // uses IAM role or env credentials

var request = new DescribeInstancesRequest
{
    Filters =
    [
        new Filter { Name = "instance-state-name", Values = ["running"] }
    ]
};

DescribeInstancesResponse response;
do
{
    response = await ec2.DescribeInstancesAsync(request);

    foreach (var reservation in response.Reservations)
    foreach (var inst in reservation.Instances)
    {
        var name = inst.Tags.FirstOrDefault(t => t.Key == "Name")?.Value ?? "(no name)";
        Console.WriteLine($"{inst.InstanceId}  {inst.InstanceType,-14} {inst.PrivateIpAddress,-16} {name}");
    }

    request.NextToken = response.NextToken;
} while (!string.IsNullOrEmpty(response.NextToken));  // paginate

2. Start / Stop / Reboot Instances

C# — Lifecycle Management 
public class Ec2LifecycleService(IAmazonEC2 ec2)
{
    // Stop instances (state → stopping → stopped). EBS data preserved.
    public async Task StopAsync(params string[] instanceIds)
    {
        var resp = await ec2.StopInstancesAsync(new() { InstanceIds = [..instanceIds] });
        foreach (var s in resp.StoppingInstances)
            Console.WriteLine($"Stopping {s.InstanceId}: {s.PreviousState.Name} → {s.CurrentState.Name}");
    }

    // Start stopped instances
    public async Task StartAsync(params string[] instanceIds)
    {
        var resp = await ec2.StartInstancesAsync(new() { InstanceIds = [..instanceIds] });
        foreach (var s in resp.StartingInstances)
            Console.WriteLine($"Starting {s.InstanceId}: {s.PreviousState.Name} → {s.CurrentState.Name}");
    }

    // Reboot — does NOT change the instance's public IP
    public async Task RebootAsync(params string[] instanceIds)
        => await ec2.RebootInstancesAsync(new() { InstanceIds = [..instanceIds] });

    // Wait until all instances reach target state
    public async Task WaitUntilAsync(string[] ids, string targetState, TimeSpan? timeout = null)
    {
        var deadline = DateTime.UtcNow + (timeout ?? TimeSpan.FromMinutes(5));
        while (DateTime.UtcNow < deadline)
        {
            var resp = await ec2.DescribeInstanceStatusAsync(new()
            {
                InstanceIds      = [..ids],
                IncludeAllInstances = true
            });
            var states = resp.InstanceStatuses.Select(s => s.InstanceState.Name.ToString()).ToHashSet();
            if (states.All(s => s == targetState)) return;
            await Task.Delay(3_000);
        }
        throw new TimeoutException($"Instances did not reach '{targetState}' within timeout");
    }
}

4. Create EBS Snapshot & Manage Volumes

C# — EBS Snapshots 
// Create a snapshot of an EBS volume
public async Task<string> CreateSnapshotAsync(string volumeId, string description)
{
    var snap = await ec2.CreateSnapshotAsync(new()
    {
        VolumeId    = volumeId,
        Description = description,
        TagSpecifications =
        [
            new TagSpecification
            {
                ResourceType = ResourceType.Snapshot,
                Tags = [new { Key = "CreatedBy", Value = "automated-backup" }]
            }
        ]
    });

    Console.WriteLine($"Snapshot {snap.SnapshotId} state: {snap.State}");
    // State will be "pending" — wait for "completed" before using
    return snap.SnapshotId;
}

// List all snapshots owned by this account, sorted by start time
public async Task<List<Snapshot>> ListSnapshotsAsync()
{
    var resp = await ec2.DescribeSnapshotsAsync(new()
    {
        OwnerIds = ["self"],
        Filters  = [new Filter { Name = "status", Values = ["completed"] }]
    });
    return [.. resp.Snapshots.OrderByDescending(s => s.StartTime)];
}

// Describe volume status — check for degraded or impaired volumes
public async Task CheckVolumeHealthAsync(string volumeId)
{
    var resp = await ec2.DescribeVolumeStatusAsync(new()
    {
        VolumeIds = [volumeId]
    });
    foreach (var vs in resp.VolumeStatuses)
    {
        Console.WriteLine($"Volume {vs.VolumeId}: IO={vs.VolumeStatus.Status}");
        if (vs.VolumeStatus.Status != "ok")
            Console.WriteLine($"⚠️ Volume health issue: {vs.VolumeStatus.Details[0].Status}");
    }
}

✅ Best Practices & Common Mistakes

Compute Best Practices

  • Right-size instances using CloudWatch CPU and memory metrics — most instances run at <20% average CPU
  • Prefer Graviton (ARM) instances (m7g, c7g, r7g) — ~20% cheaper, often faster for modern runtimes
  • Use gp3 EBS volumes by default — cheaper than gp2 and independently configurable IOPS/throughput
  • Enforce IMDSv2 (HttpTokens=required) on all instances — blocks SSRF credential theft attacks
  • Always attach an IAM Instance Profile — never store AWS access keys on the instance filesystem
  • Enable detailed CloudWatch monitoring ($0.01/instance/hr) for 1-minute granularity on production instances

Networking Best Practices

  • Use private subnets for application and database instances — only load balancers and bastion hosts need public subnets
  • Scope Security Group rules to the minimum required IP range and port — avoid 0.0.0.0/0 on ports like 22 and 3389
  • Use VPC endpoints for AWS services (S3, DynamoDB, SSM) — traffic stays on the AWS backbone and avoids NAT Gateway costs
  • Use SSM Session Manager instead of SSH — eliminates need to manage SSH keys or open port 22
  • Set up a NAT Gateway per AZ for high-availability outbound internet access from private subnets

Scaling & Availability

  • Use Auto Scaling Groups (ASG) with Application Load Balancer for all stateless workloads — not a fixed fleet
  • Deploy across at least 2 Availability Zones — single-AZ deployments have no fault tolerance
  • Use Launch Templates (not Launch Configurations) — they support versioning and newer features
  • Set ASG scaling policies based on CPU and custom CloudWatch metrics — use Target Tracking for simplicity
  • Use EC2 Instance Health Checks on your ASG — replaces unhealthy instances automatically

Common Mistakes to Avoid

⚠️ Top EC2 Mistakes

❌ Opening Security Group port 22 to 0.0.0.0/0 — your instance will receive automated brute-force attacks within minutes. Use SSM Session Manager instead, or restrict SSH to your IP only.
❌ Storing AWS access keys on the instance — use IAM Instance Profiles (roles). Access keys on disk are a critical security risk if the instance is compromised.
❌ Using a single AZ — AZ outages happen. Deploy across 2+ AZs with an Auto Scaling Group and Load Balancer.
❌ Forgetting to snapshot before major changes — take an EBS snapshot before OS upgrades, software installs, or config changes. Snapshots are incremental and cheap.
❌ Not tagging resources — without tags, you can't identify what instances are for, who owns them, or filter billing reports. Tag everything at launch.
❌ Using gp2 instead of gp3 — gp2 is more expensive and less flexible. All new volumes should be gp3. Migrate existing gp2 volumes in the console with zero downtime.
❌ Ignoring Trusted Advisor — AWS Trusted Advisor identifies idle instances, underutilized EBS, unused Elastic IPs, and security risks. Check it monthly.
❌ Not enabling termination protection on production instances — one accidental API call or console click can terminate a production instance. Enable termination protection.
