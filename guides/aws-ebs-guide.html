💽 What is Amazon EBS?

Amazon Elastic Block Store (EBS) provides persistent block storage volumes for use with EC2 instances. Think of it as a high-performance, network-attached hard drive. EBS volumes persist independently from the life of an instance — you can stop an instance, the data remains, and you can reattach the volume to another instance.

💡 EBS in One Sentence

EBS = a durable, high-performance network-attached disk you can plug into any EC2 instance in the same AZ, resize on the fly, snapshot to S3, and encrypt with one checkbox — without ever touching physical hardware.

Key EBS Concepts

💽

Volume

The EBS disk itself. Attached to one EC2 instance at a time (except io2 Multi-Attach). Lives in one AZ.

📸

Snapshot

Point-in-time backup of a volume stored in S3. Incremental — only changed blocks stored after the first snapshot.

🔑

Encryption

AES-256 at rest using AWS KMS. Transparent — no app code changes. Enable at volume creation.

⚡

IOPS

Input/Output Operations Per Second — the key metric for databases and transactional workloads.

🚀

Throughput

MB/s — the key metric for sequential workloads like data warehousing, log processing, video streaming.

🔄

Elastic Volumes

Modify volume type, size, IOPS, and throughput on a live attached volume — zero downtime.

🗓️

DLM

Data Lifecycle Manager — automate snapshot creation, retention, and cross-region copy policies.

🔗

Multi-Attach

io2 only: attach one volume to up to 16 EC2 instances simultaneously in the same AZ.

🌍

Fast Snapshot Restore

Pre-warm snapshots so volumes restored from them have full performance immediately — no warm-up delay.

EBS Key Facts

📍

AZ-bound

EBS volumes are locked to one AZ. To move to another AZ: snapshot → restore in target AZ.

⏱️

Billing

Per GB-month provisioned (not used). Plus per IOPS-month for io1/io2. gp3 IOPS and throughput are free up to baseline.

📏

Max size

64 TB per volume. No limit on number of volumes. Single instance can have hundreds of EBS volumes attached.

🔌

Network-attached

EBS traffic uses the EC2 instance's network bandwidth. High-performance instances have dedicated EBS bandwidth.

🛡️

Durability

99.8–99.9% annual durability for gp2/gp3. 99.999% for io2 Block Express. Replicated within the AZ.

💰

Free tier

30 GB of EBS (gp2 or gp3) per month free for 12 months under the AWS Free Tier.

📊 EBS Volume Types

EBS has five volume types across two storage technologies: SSD (optimised for IOPS and random I/O) and HDD (optimised for throughput and sequential I/O). Choosing the right type is the single biggest lever for both performance and cost.

💡 Quick Decision Rule

Database or boot volume → gp3 (default). High IOPS critical database (Oracle, SAP HANA) → io2 Block Express. Big data sequential reads → st1. Cold archival data rarely accessed → sc1. Never use gp2 for new volumes — gp3 is cheaper and better.

gp3 vs gp2 — Why Always Use gp3

✅ gp3 Advantages

  • Cheaper: $0.08/GB vs $0.10/GB — 20% less expensive
  • Always 3,000 IOPS baseline regardless of volume size
  • Independently configure IOPS (up to 16,000) and throughput (up to 1,000 MB/s) without paying more per GB
  • A 1 GB gp3 volume gets 3,000 IOPS — a 1 GB gp2 gets only 3 IOPS
  • Zero downtime migration: modify gp2 → gp3 in-place

⚠️ gp2 Problems

  • IOPS tied to size: 3 IOPS per GB, so small volumes are IOPS-starved
  • Burst bucket model: small volumes deplete burst credits and throttle
  • Max throughput only 250 MB/s vs gp3's 1,000 MB/s
  • More expensive per GB with no benefits over gp3
  • Still the default in some older CloudFormation/CDK templates — always override

The Two Bottlenecks — Volume vs Instance

⚠️ Critical: EBS Performance Has TWO Limits

Your actual EBS throughput is limited by whichever is lower: the volume limit OR the EC2 instance's EBS bandwidth limit. Many people provision expensive io2 volumes and are surprised performance doesn't improve — because the instance was the bottleneck, not the volume. Always check both limits.

EBS-Optimized Instances

💡 EBS-Optimized — Always Enabled on Modern Instances

EBS-optimized instances provide dedicated network bandwidth for EBS traffic, separate from regular network traffic. Without it, EBS I/O competes with application network traffic causing unpredictable latency. All current-generation instances (C5, M5, R5, m7i, r7g, etc.) are EBS-optimized by default at no extra charge. On older instances (C3, M3), EBS optimization was optional and cost extra.

gp3 IOPS Configuration — Cost Calculation

💰 gp3 Pricing Breakdown Example (100 GB volume)

  • Storage: 100 GB × $0.08 = $8.00/month
  • IOPS free baseline: 3,000 IOPS included at no charge
  • Extra IOPS (e.g. 10,000 total): (10,000 − 3,000) × $0.005 = $35.00/month
  • Extra throughput (e.g. 500 MB/s total): (500 − 125) × $0.04 = $15.00/month
  • Total: $8 + $35 + $15 = $58.00/month for 100 GB, 10K IOPS, 500 MB/s
  • vs io2 for same IOPS: 100 GB × $0.125 + 10,000 × $0.065 = $662/month — 11× more expensive!

📸 EBS Snapshots

EBS snapshots are incremental point-in-time backups stored in S3. The first snapshot copies the entire volume; subsequent snapshots only store changed blocks. Snapshots are the foundation of EBS backup, disaster recovery, and volume migration across AZs or regions.

How Incremental Snapshots Work

📸 Snapshot 1 (Full — e.g. Monday)
  • Copies entire volume data to S3 — e.g. 50 GB used → 50 GB stored
  • Takes longest to complete but is the baseline for all future snapshots
  • Cost: 50 GB × $0.05 = $2.50/month while retained
↓
📸 Snapshot 2 (Incremental — Tuesday)
  • Only stores blocks that changed since Snapshot 1 — e.g. 5 GB changed → 5 GB stored
  • Much faster and cheaper than a full backup
  • Still represents the full point-in-time state — restores to a complete volume
↓
📸 Snapshot 3 (Incremental — Wednesday)
  • Only stores blocks changed since Snapshot 2
  • Deleting Snapshot 1 does NOT break Snapshot 2 or 3 — AWS moves required blocks automatically
  • You can safely delete any snapshot; the chain remains intact

Snapshot Features

🌍 Cross-Region Copy

  • Copy snapshots to any AWS region for disaster recovery
  • Can encrypt or change KMS key during copy
  • Creates a fully independent snapshot in the destination region
  • Cost: destination storage ($0.05/GB-month) + data transfer fees
  • Use DLM to automate cross-region copy on a schedule

🔗 Cross-Account Sharing

  • Share snapshots with specific AWS account IDs
  • Or make public (not recommended for sensitive data)
  • Recipient copies the snapshot to their account before use
  • Encrypted snapshots can only be shared if you also share the KMS key
  • Used for AMI distribution across teams/accounts

⚡ Fast Snapshot Restore (FSR)

  • Pre-warms snapshot data so restored volumes have full performance immediately
  • Without FSR: restored volumes start throttled, ramp up over hours as blocks are fetched from S3
  • Enable per snapshot per AZ — cost: $0.75/DSU-hour per AZ
  • Use for: RDS, high-performance databases, AMI-backed Auto Scaling groups
  • Turn off when not needed — FSR charges are per hour regardless of usage

🗑️ Recycle Bin

  • Retains accidentally deleted snapshots for a configurable period (1 day to 1 year)
  • Recover deleted snapshots before they're permanently gone
  • Cost: $0.0075/GB-month for retained snapshots in Recycle Bin
  • Create Recycle Bin rules in the EC2 → Recycle Bin console section
  • Best practice: enable for all production snapshot types

Data Lifecycle Manager (DLM) — Automated Snapshots

1

Create a DLM Policy

Go to EC2 → Elastic Block Store → Lifecycle Manager → Create snapshot lifecycle policy. DLM automates snapshot creation, retention, and cross-region copy on a schedule you define.

2

Target volumes by tag

DLM matches volumes using tags (e.g. Backup=daily). Any volume with that tag gets snapshot automatically. Tag new volumes at launch to enroll them automatically.

3

Set schedule and retention

Example: daily at 02:00 UTC, retain 7 snapshots. DLM creates the snapshot and deletes the oldest one automatically. No Lambda functions or cron jobs needed.

4

Add cross-region copy rule (optional)

Within the same DLM policy, add a cross-region copy action. DLM copies each new snapshot to the target region with its own retention count. One policy handles create + copy + cleanup.

⚠️ Snapshot Cost Accumulates Silently

Orphaned snapshots from terminated instances are never automatically deleted. They silently accumulate at $0.05/GB-month. A 1 TB volume with 30 daily snapshots costs $50/month just in snapshots. Use DLM retention policies, and periodically audit with describe-snapshots --owner-ids self to find and delete unused ones.

🔐 EBS Encryption

EBS encryption provides AES-256 encryption of data at rest, in transit between the instance and the volume, and in snapshots — all transparently with no application changes required.

What Gets Encrypted

🔐 EBS Encryption Scope

Encryption Key Types

🔑 AWS-Managed Key (aws/ebs)

  • Free — no KMS key cost
  • AWS manages rotation automatically (every 3 years)
  • Cannot share encrypted snapshots across accounts using this key
  • Cannot use for cross-account snapshot sharing
  • Good for: single-account environments, simple compliance requirements

🔑 Customer Managed Key (CMK)

  • $1/month per key + $0.03 per 10,000 API calls
  • You control key rotation, deletion, access policy
  • Can share encrypted snapshots across accounts by sharing the KMS key
  • Required for some compliance frameworks (FedRAMP, PCI-DSS strict)
  • Key deletion has a 7–30 day waiting period — can't be undone

Enable Account-Level Default Encryption

1

Enable in EC2 Settings (per region)

EC2 Console → Settings → EBS encryption → Enable. Select default KMS key (aws/ebs or your CMK). From this point, every new EBS volume created in this region is automatically encrypted — even if not explicitly requested at launch.

2

Enforce via AWS Config rule

Use the encrypted-volumes AWS Config managed rule to detect any unencrypted EBS volumes and trigger alerts or auto-remediation. Use AWS Organizations SCPs to prevent creating unencrypted volumes across all accounts.

3

Encrypt existing unencrypted volumes

You cannot encrypt an existing volume in-place. Process: create snapshot → copy snapshot with encryption enabled → restore new encrypted volume from snapshot → swap volumes on the instance (stop instance, detach old, attach new, start). Use AWS Backup or a script to automate this at scale.

💡 No Performance Impact

EBS encryption uses hardware-accelerated AES-256 on the Nitro hypervisor. There is no measurable performance difference between encrypted and unencrypted volumes on all current-generation instances. There is no reason not to encrypt everything. Enable account-level default encryption in every region today.

🔄 EBS Volume Lifecycle & Elastic Volumes

EBS volumes have a lifecycle from creation to deletion, and uniquely among block storage, can be modified while live and attached — changing type, size, IOPS, and throughput without stopping your instance.

Volume States

🟡 Creating
  • Volume being provisioned
  • Not yet attachable
🟢 Available
  • Ready to attach
  • Not attached to any instance
🟢 In-Use
  • Attached to an EC2 instance
  • Being read/written to
↓
🟠 Modifying
  • Elastic Volume change in progress
  • Still fully usable during change
🔴 Deleting
  • Being permanently deleted
  • Data is gone — cannot recover

  • After a type change (e.g. gp2→gp3)
  • Performance fully available during this

Elastic Volumes — Live Modification

💡 Change Volumes Without Downtime

Elastic Volumes let you modify a volume while it is attached and in use. The change takes effect immediately and the volume stays online throughout. The only caveat: after a modification, you must wait 6 hours before modifying the same volume again.

How to Extend a Filesystem After Volume Resize

1

Resize the EBS volume (console or API)

Increase volume size via EC2 console or modify-volume API. The volume enters "modifying" state and returns to "in-use" quickly. The OS still sees the old size until you extend the partition.

2

Extend the partition (if needed)

Run sudo growpart /dev/nvme0n1 1 to extend partition 1 to fill the new disk size. Check with lsblk first to confirm the device and partition names.

3

Extend the filesystem

For ext4: sudo resize2fs /dev/nvme0n1p1. For XFS (Amazon Linux 2023 default): sudo xfs_growfs /. For LVM: pvresize → lvextend → filesystem resize. Verify with df -h.

Multi-Attach (io2 only)

🔗 What Multi-Attach Does

  • Attach one io2 volume to up to 16 EC2 instances in the same AZ simultaneously
  • All instances have read and write access to the same volume
  • Use case: clustered databases that manage their own locking (Oracle RAC, clustered filesystems)
  • Requires a cluster-aware filesystem (e.g. GFS2, OCFS2) — NOT ext4 or XFS

⚠️ Multi-Attach Limitations

  • io2 volumes only — not gp3 or any other type
  • All instances and the volume must be in the same AZ
  • You are responsible for I/O consistency — no built-in locking
  • Standard filesystems (ext4, XFS) will corrupt data if used with Multi-Attach
  • Cannot be used as a boot volume

1. Create and Attach a Volume

C# — Create gp3 Volume and Attach to EC2 
using Amazon.EC2;
using Amazon.EC2.Model;

public class EbsVolumeService(IAmazonEC2 ec2)
{
    // Create a gp3 volume in a specified AZ
    public async Task<string> CreateVolumeAsync(
        string availabilityZone,
        int    sizeGb       = 100,
        int    iops         = 3000,
        int    throughputMb = 125)
    {
        var resp = await ec2.CreateVolumeAsync(new CreateVolumeRequest
        {
            AvailabilityZone = availabilityZone,
            VolumeType       = VolumeType.Gp3,
            Size             = sizeGb,
            Iops             = iops,
            Throughput       = throughputMb,
            Encrypted        = true,          // Always encrypt
            TagSpecifications =
            [
                new TagSpecification
                {
                    ResourceType = ResourceType.Volume,
                    Tags =
                    [
                        new { Key = "Name",       Value = "app-data-prod" },
                        new { Key = "Backup",     Value = "daily"        },
                        new { Key = "ManagedBy", Value = "dotnet-sdk"   }
                    ]
                }
            ]
        });

        var volumeId = resp.Volume.VolumeId;
        Console.WriteLine($"Created: {volumeId} ({resp.Volume.State})");

        // Wait until available before attaching
        await WaitForStateAsync(volumeId, "available");
        return volumeId;
    }

    // Attach volume to an EC2 instance
    public async Task AttachVolumeAsync(
        string volumeId,
        string instanceId,
        string device = "/dev/sdf")
    {
        var resp = await ec2.AttachVolumeAsync(new AttachVolumeRequest
        {
            VolumeId   = volumeId,
            InstanceId = instanceId,
            Device     = device
        });
        Console.WriteLine($"Attaching {volumeId} to {instanceId} as {device}: {resp.Attachment.State}");
    }

    // Poll until volume reaches desired state
    private async Task WaitForStateAsync(string volumeId, string targetState, int timeoutSecs = 120)
    {
        var deadline = DateTime.UtcNow.AddSeconds(timeoutSecs);
        while (DateTime.UtcNow < deadline)
        {
            var resp = await ec2.DescribeVolumesAsync(new()
                { VolumeIds = [volumeId] });
            var state = resp.Volumes[0].State.ToString();
            if (state == targetState) return;
            Console.WriteLine($"  {volumeId}: {state}...");
            await Task.Delay(5_000);
        }
        throw new TimeoutException($"{volumeId} did not reach '{targetState}'");
    }
}

2. Modify Volume (gp2 → gp3 migration)

C# — Elastic Volume Modification 
// Upgrade all gp2 volumes in the account to gp3 (saves ~20% cost)
public async Task MigrateAllGp2ToGp3Async()
{
    string? nextToken = null;
    var migrated = 0;

    do
    {
        var resp = await ec2.DescribeVolumesAsync(new DescribeVolumesRequest
        {
            Filters   = [new Filter { Name = "volume-type", Values = ["gp2"] }],
            NextToken = nextToken,
            MaxResults = 100
        });

        foreach (var vol in resp.Volumes)
        {
            try
            {
                await ec2.ModifyVolumeAsync(new ModifyVolumeRequest
                {
                    VolumeId   = vol.VolumeId,
                    VolumeType = VolumeType.Gp3,
                    Iops       = 3000,   // Match or improve on gp2 burst IOPS
                    Throughput = 125    // gp3 baseline (gp2 max was 250 only at 83+ GB)
                });
                Console.WriteLine($"✅ Migrated {vol.VolumeId} ({vol.Size} GB) gp2 → gp3");
                migrated++;
            }
            catch (AmazonEC2Exception ex) when (ex.ErrorCode == "VolumeModificationRateExceeded")
            {
                Console.WriteLine($"⏳ {vol.VolumeId}: throttled, skip (modified recently)");
            }
        }
        nextToken = resp.NextToken;
    } while (!string.IsNullOrEmpty(nextToken));

    Console.WriteLine($"\nTotal migrated: {migrated} volumes");
}

3. Snapshot Management

C# — Snapshot Create, List, Copy, Delete 
public class SnapshotService(IAmazonEC2 ec2)
{
    // Create a snapshot
    public async Task<string> CreateSnapshotAsync(string volumeId, string description)
    {
        var snap = await ec2.CreateSnapshotAsync(new()
        {
            VolumeId    = volumeId,
            Description = description,
            TagSpecifications =
            [
                new TagSpecification
                {
                    ResourceType = ResourceType.Snapshot,
                    Tags = [new { Key = "AutoBackup", Value = "true" }]
                }
            ]
        });
        Console.WriteLine($"Snapshot {snap.SnapshotId}: {snap.State}");
        return snap.SnapshotId;
    }

    // List all completed snapshots owned by this account, newest first
    public async Task<List<Snapshot>> ListSnapshotsAsync(string? volumeId = null)
    {
        var filters = new List<Filter> { new { Name = "status", Values = ["completed"] } };
        if (volumeId != null)
            filters.Add(new { Name = "volume-id", Values = [volumeId] });

        var resp = await ec2.DescribeSnapshotsAsync(new()
        {
            OwnerIds = ["self"],
            Filters  = filters
        });
        return [.. resp.Snapshots.OrderByDescending(s => s.StartTime)];
    }

    // Copy snapshot to another region for DR
    public async Task<string> CopySnapshotToRegionAsync(
        string snapshotId,
        string sourceRegion,
        string destRegion)
    {
        var destClient = new AmazonEC2Client(Amazon.RegionEndpoint.GetBySystemName(destRegion));
        var resp = await destClient.CopySnapshotAsync(new()
        {
            SourceRegion     = sourceRegion,
            SourceSnapshotId = snapshotId,
            Description      = $"DR copy from {sourceRegion}",
            Encrypted        = true  // Encrypt in destination too
        });
        Console.WriteLine($"Copy started in {destRegion}: {resp.SnapshotId}");
        return resp.SnapshotId;
    }

    // Delete snapshots older than N days (cost cleanup)
    public async Task<int> DeleteOldSnapshotsAsync(string volumeId, int keepDays = 30)
    {
        var snaps   = await ListSnapshotsAsync(volumeId);
        var cutoff  = DateTime.UtcNow.AddDays(-keepDays);
        var old     = snaps.Where(s => s.StartTime < cutoff).ToList();
        foreach (var snap in old)
        {
            await ec2.DeleteSnapshotAsync(new() { SnapshotId = snap.SnapshotId });
            Console.WriteLine($"Deleted {snap.SnapshotId} from {snap.StartTime:yyyy-MM-dd}");
        }
        return old.Count;
    }
}

✅ Best Practices & Common Mistakes

Volume Best Practices

  • Always use gp3 for new volumes — 20% cheaper than gp2 with a free 3,000 IOPS baseline regardless of size
  • Enable account-level EBS default encryption in every region — zero performance cost, one-time setup
  • Right-size volumes — you pay per GB provisioned, not used. Start small and use Elastic Volumes to grow on demand
  • Separate OS and data onto different volumes — makes snapshots, resizes, and instance migrations cleaner
  • Use gp3 for databases under 16,000 IOPS — io2 is 11× more expensive per IOPS; only use it when gp3 is insufficient
  • Tag all volumes at creation with Name, Environment, and a Backup tag for DLM targeting

Snapshot Best Practices

  • Use Data Lifecycle Manager (DLM) for automated snapshots — never rely on manual or cron-based snapshots for production
  • Enable Recycle Bin for snapshots with at least 7-day retention — prevents accidental permanent deletion
  • For databases, use application-consistent snapshots — quiesce writes first, or use AWS Backup with VSS (Windows) or freeze/thaw hooks
  • Audit orphaned snapshots monthly with describe-snapshots and delete anything from terminated instances
  • Archive snapshots older than 90 days to snapshot archive tier — 75% cheaper, with 24–72h restore if needed
  • Copy critical snapshots to a second region via DLM cross-region copy rules for DR coverage

Performance Best Practices

  • Check EC2 instance EBS bandwidth limit before provisioning io2 — the instance is often the bottleneck, not the volume
  • Use CloudWatch metrics — BurstBalance (gp2 legacy), VolumeReadOps, VolumeWriteOps, VolumeQueueLength to spot bottlenecks
  • VolumeQueueLength above 1 per provisioned IOPS indicates the volume is saturated — increase IOPS or upgrade instance
  • Place latency-sensitive workloads on io2 with provisioned IOPS — gp3 has occasional I/O latency spikes under heavy load
  • Use RAID 0 across multiple gp3 volumes to exceed single-volume IOPS/throughput limits on older instance types

Common Mistakes to Avoid

⚠️ Top EBS Mistakes

❌ Using gp2 for new volumes — gp2 is more expensive and worse in every dimension vs gp3. Run the migrate-all-gp2-to-gp3 script immediately.
❌ Not encrypting volumes — there is zero performance cost on current-generation Nitro instances. Enable account-level default encryption now — it takes 30 seconds and protects all future volumes permanently.
❌ Forgetting to extend the filesystem after resize — resizing the EBS volume does not automatically expand the filesystem inside the OS. You must run growpart and resize2fs / xfs_growfs after the volume modification completes.
❌ Orphaned snapshots from deleted instances — AWS never auto-deletes snapshots when you terminate an instance. They accumulate silently. A 1 TB volume snapshotted daily for a year costs over $600 in snapshot storage alone if never cleaned up.
❌ Provisioning io2 when gp3 is sufficient — io2 costs 11× more per IOPS than gp3. Measure your actual IOPS requirements with CloudWatch before committing to io2. Most workloads do fine on gp3 at 16,000 IOPS.
❌ Not checking instance EBS bandwidth before provisioning high IOPS volumes — a t3.medium maxes out at ~2.9 Gbps of EBS bandwidth (~160 MB/s). An io2 volume with 64,000 IOPS is wasted on it. Always check both limits.
❌ Snapshots as the only backup strategy — snapshots are AZ and region-level. Use DLM cross-region copy or AWS Backup for true disaster recovery coverage.

EBS vs Instance Store — When to Use Each

💽 Use EBS When…

  • Data must survive instance stop/start/replacement
  • You need snapshots or point-in-time backups
  • Running a database, application server, or any stateful service
  • You need to resize storage without re-creating the instance
  • Cross-AZ detach/re-attach is required for DR

⚡ Use Instance Store When…

  • You need maximum IOPS with microsecond latency (NVMe)
  • Data is truly ephemeral — caches, temp files, scratch space
  • Running a distributed system that replicates data across nodes (Kafka, Cassandra, HDFS)
  • You have fault-tolerant batch jobs that can restart from scratch
  • Cost is critical and data can be rebuilt from S3 or a peer node
