🚪 What is Amazon API Gateway?

API Gateway is a fully managed AWS service that acts as the front door to your backend services. It receives HTTP requests from clients, applies rules (auth, throttling, transformation), and forwards them to Lambda, ECS, EC2, or any HTTP endpoint.

💡 Real-life analogy

API Gateway is like a hotel receptionist. Every guest (client) talks to the receptionist (API Gateway) — never directly to the kitchen (Lambda), the housekeeping team (ECS), or the manager (backend). The receptionist checks ID (auth), enforces rules (throttling), and routes requests to the right department.

  • ➡️ Forwards to backend integration

Happy Path — Successful Request

📱 Client — GET /v1/orders/123
  • Header: Authorization: Bearer eyJhbGc...
  • Header: x-api-key: abc123def456
  • URL: https://abc123.execute-api.ap-south-1.amazonaws.com/prod/orders/123
↓
🔐 Stage 1 — Authentication
  • Cognito Authorizer: validates JWT signature + expiry
  • Claims extracted: userId=usr_456, role=customer
  • If invalid → 401 Unauthorized returned immediately
  • Claims passed to backend as headers
↓
🚦 Stage 2 — Throttle Check
  • Account limit: 10,000 req/sec
  • Stage limit: 1,000 req/sec
  • Route limit: GET /orders/* = 500 req/sec
  • Client within limits ✅ → continue
↓
📦 Stage 3 — Cache Lookup
  • Cache key: GET /orders/123 + userId header
  • Cache miss — not in cache yet
  • Continue to backend (if hit → skip to response)
↓
λ Stage 4 — Lambda Invocation
  • API GW invokes: GetOrderFunction
  • Passes: pathParameters.id=123, requestContext.authorizer.userId=usr_456
  • Lambda fetches order from DynamoDB
  • Returns: statusCode 200 + order JSON body
↓
🔄 Stage 5 — Response Transform + Cache Store
  • Adds CORS headers: Access-Control-Allow-Origin: *
  • Stores response in cache for 300 seconds
  • Sets response headers from mapping template
↓
✅ Client receives 200 OK + order JSON
  • Total time: ~80ms (Lambda cold start: first call may be ~300ms)
  • Subsequent identical requests: ~5ms (from cache)

Error Paths — What Goes Wrong and When

🔴 401 Unauthorized
  • Missing or invalid JWT
  • Expired token
  • Wrong API Key
  • Blocked at auth stage — backend never called
🔴 403 Forbidden
  • Resource policy denies access
  • WAF block rule matched
  • IP not in allow list
🔴 429 Too Many Requests
  • Client exceeded rate limit
  • Burst limit exceeded
  • API GW adds Retry-After header
🔴 502/504 Gateway Error
  • 502: Lambda threw exception
  • 504: Backend timeout (29s max)
  • Backend returned malformed response

WebSocket Flow — Real-Time Connections

📱 Client opens WebSocket connection
  • wss://abc123.execute-api.ap-south-1.amazonaws.com/prod
  • API GW accepts the persistent connection
  • Assigns unique connectionId: abc-def-ghi
↓
🚪 API GW manages connections
  • $connect route → Lambda: store connectionId in DynamoDB
  • $disconnect route → Lambda: remove connectionId
  • $default route → Lambda: handle all messages
  • Custom routes: e.g. sendMessage → Lambda
↓
λ Backend pushes to client
  • Lambda calls: POST /@connections/{connectionId}
  • API GW delivers message to that specific client
  • Works even when client hasn't sent a message
  • Use case: chat, live notifications, dashboards

📦 API Types — REST vs HTTP vs WebSocket

API Gateway offers three distinct API types. Choosing the right one affects cost, features, and performance significantly.

🟠 REST API

  • Full feature set
  • Request/response transform
  • Usage plans + API keys
  • Response caching
  • Private VPC endpoints
  • More expensive
  • Use: full enterprise APIs

🟢 HTTP API

  • Simpler, faster, cheaper (~70%)
  • JWT auth built-in (Cognito/OIDC)
  • No response caching
  • No request transforms
  • Lambda + HTTP backends
  • Use: most modern APIs ✅

🔵 WebSocket API

  • Persistent connections
  • Bidirectional messaging
  • Backend pushes to clients
  • connectionId per client
  • Use: chat, live updates

✅ Which should you choose?

Start with HTTP API — it handles 90% of use cases, costs 70% less, and is simpler to configure. Upgrade to REST API only if you specifically need: response caching, request transforms, usage plans, WAF, or private VPC endpoints.

🔐 Authentication & Security

API Gateway supports multiple authentication mechanisms. You can use one or combine them for layered security.

Authentication Options — Decision Flow

📨 Incoming Request
  • Who is calling? Internal service, end user, or partner?
↓
🔵 JWT / Cognito
  • End users (mobile/web)
  • Bearer token in header
  • HTTP API native support
  • Validates signature + expiry
🟠 API Key
  • Partner / B2B APIs
  • x-api-key header
  • Usage plans + quotas
  • REST API only
λ Lambda Authorizer
  • Custom auth logic
  • Any token format
  • Result cached (TTL)
  • Returns IAM policy

Lambda Authorizer — Deep Dive

📨 Request with custom token
  • Authorization: Custom abc123xyz (not a standard JWT)
  • API GW extracts the token
  • Calls the Authorizer Lambda
↓
λ Authorizer Lambda — your custom logic
  • Validates token against your own database
  • Checks user's subscription status, IP, rate limits
  • Returns IAM policy: Allow or Deny + context
  • Result cached in API GW for TTL seconds (e.g. 300s)
↓
🚪 API GW uses the returned policy
  • Allow → forward request to backend
  • Deny → return 403 Forbidden
  • Context from authorizer passed to Lambda as event
  • Cached result used for next request with same token (fast!)

CORS — Cross-Origin Resource Sharing

🌐 When to configure CORS

Required when a browser app on app.mycompany.com calls api.mycompany.com. The browser blocks cross-origin requests unless the API returns the right CORS headers.

HTTP API: Enable CORS in console → set allowed origins, methods, headers — done.
REST API: Enable CORS per resource + add OPTIONS method + return correct headers from Lambda.

🚦 Throttling, Quotas & Usage Plans

API Gateway has multiple layers of throttling to protect your backend from overload. Requests that exceed limits are rejected with HTTP 429 — backend is never hit.

Three Layers of Throttling

🔴 Layer 1 — AWS Account Limit
  • Default: 10,000 req/sec per region (hard limit)
  • Burst: 5,000 req (token bucket algorithm)
  • Shared across ALL your APIs in the region
  • Request a limit increase via AWS Support if needed
↓
🟠 Layer 2 — Stage Limit
  • Per stage (prod, staging, dev)
  • Set per-stage default: e.g. prod = 2,000 req/sec
  • Override per route: GET /orders = 500 req/sec
  • More restrictive routes protect specific expensive endpoints
↓
🔵 Layer 3 — Usage Plan (per API Key)
  • Per customer / per API key throttling
  • Rate: max 100 req/sec per key
  • Quota: max 10,000 requests per month per key
  • Exceeded quota → 429 for that key, others unaffected
↓
✅ Backend receives request (all limits passed)
  • Lambda / ECS only invoked for allowed requests
  • Protects backend from traffic spikes

Token Bucket Algorithm — Burst Explained

🪣 How burst limit works

Think of a bucket that holds tokens. New tokens fill in at 10,000/sec (rate limit). Each request consumes one token. You can burst up to 5,000 requests instantly (bucket capacity) — then you're limited to the refill rate (10,000/sec).

Example: If your API gets 0 requests for 1 second, it builds up 10,000 tokens. Next second you can fire 10,000 + 5,000 burst = handles the spike. After burst is exhausted, back to steady 10,000/sec.

Usage Plans — Per Customer Quotas

🆓 Free Tier Plan

  • Rate: 10 req/sec
  • Burst: 20 requests
  • Quota: 1,000 req/month
  • Assigned API Key: free-tier-key

💎 Pro Plan

  • Rate: 500 req/sec
  • Burst: 1,000 requests
  • Quota: 1,000,000 req/month
  • Assigned API Key: pro-tier-key

Response Caching (REST API only)

📨 GET /products/456 (first request)
  • Cache miss → forward to Lambda
  • Lambda responds with product data
  • API GW stores response in cache for 300 seconds
↓ Within 300 seconds…
⚡ GET /products/456 (second request)
  • Cache HIT → return cached response instantly

🚀 Stages & Deployment

A stage is a named snapshot of your API deployment (dev, staging, prod). Each stage has its own URL, settings, throttling, and variables.

Stages Architecture

🚪 API Gateway — Single API definition
  • Routes, integrations, auth all defined once
  • Deployed to multiple named stages
↓ Deploy to stages
🟢 dev stage
  • /dev/orders
  • Lambda: dev alias
  • No throttle
  • Verbose logging
🟠 staging stage
  • /staging/orders
  • Lambda: staging alias
  • Light throttle
  • Info logging
🔵 prod stage
  • /prod/orders
  • Lambda: prod alias
  • Full throttle
  • Error logging only

Stage Variables — Config Without Code Change

📋 What are Stage Variables?

Key-value pairs set per stage. Your Lambda integration uses them as placeholders. Change the variable → different behaviour, no code change.

Example: Integration URI uses ${stageVariables.lambdaAlias}
dev stage: lambdaAlias = dev → invokes Lambda:dev
prod stage: lambdaAlias = prod → invokes Lambda:prod

Also use for: database endpoints, feature flags, log levels.

Canary Deployments

🟠 Prod Stage with Canary enabled
  • Canary: 10% of traffic → new deployment (v2)
  • Baseline: 90% of traffic → current deployment (v1)
  • Monitor metrics and errors on canary
  • Promote canary → 100% when healthy
  • Roll back → delete canary if issues found

Lambda Handler for HTTP API (Proxy Integration)

OrdersFunction.cs 
using Amazon.Lambda.APIGatewayEvents;
using Amazon.Lambda.Core;
using System.Text.Json;

[assembly: LambdaSerializer(
    typeof(Amazon.Lambda.Serialization.SystemTextJson.DefaultLambdaJsonSerializer))]

public class OrdersFunction
{
    // HTTP API → APIGatewayHttpApiV2ProxyRequest
    public async Task<APIGatewayHttpApiV2ProxyResponse>
        HandleAsync(APIGatewayHttpApiV2ProxyRequest request,
                    ILambdaContext context)
    {
        var method = request.RequestContext.Http.Method;
        var path   = request.RequestContext.Http.Path;

        context.Logger.LogInformation(
            $"Received {method} {path}");

        // Extract path parameters
        request.PathParameters.TryGetValue(
            "orderId", out var orderId);

        // Extract claims from JWT (passed by authorizer)
        request.RequestContext.Authorizer.Jwt.Claims
            .TryGetValue("sub", out var userId);

        // Extract query params
        request.QueryStringParameters?.TryGetValue(
            "status", out var status);

        // Your business logic here...
        var order = new {
            Id = orderId, UserId = userId,
            Status = status ?? "all" };

        return new APIGatewayHttpApiV2ProxyResponse
        {
            StatusCode = 200,
            Body       = JsonSerializer.Serialize(order),
            Headers    = new Dictionary<string, string>
            {
                ["Content-Type"] = "application/json"
            }
        };
    }
}

REST API Lambda Handler

REST API uses APIGatewayProxyRequest (V1) 
public async Task<APIGatewayProxyResponse>
    HandleAsync(APIGatewayProxyRequest request,
                ILambdaContext context)
{
    // Path param from {orderId} route
    var orderId = request.PathParameters["orderId"];

    // Claims from Lambda Authorizer context
    request.RequestContext.Authorizer
        .TryGetValue("userId", out var userId);

    // Query string
    request.QueryStringParameters?
        .TryGetValue("page", out var page);

    // Body (for POST/PUT)
    var body = request.IsBase64Encoded
        ? Encoding.UTF8.GetString(
            Convert.FromBase64String(request.Body))
        : request.Body;

    return new APIGatewayProxyResponse
    {
        StatusCode = 200,
        Body       = JsonSerializer.Serialize(new { orderId, userId }),
        Headers    = new Dictionary<string, string>
        {
            ["Content-Type"] = "application/json",
            ["X-Request-Id"] = context.AwsRequestId
        }
    };
}

Lambda Authorizer Function

CustomAuthorizerFunction.cs 
public APIGatewayCustomAuthorizerResponse
    Authorize(APIGatewayCustomAuthorizerRequest request,
              ILambdaContext context)
{
    var token = request.AuthorizationToken;
    // e.g. "Bearer my-custom-token-abc"

    var isValid = ValidateMyCustomToken(token);
    var effect  = isValid ? "Allow" : "Deny";

    return new APIGatewayCustomAuthorizerResponse
    {
        PrincipalID = "user-123",
        PolicyDocument = new APIGatewayCustomAuthorizerPolicy
        {
            Version = "2012-10-17",
            Statement = [new APIGatewayCustomAuthorizerPolicy
                .IAMPolicyStatement
            {
                Action   = ["execute-api:Invoke"],
                Effect   = effect,
                Resource = [request.MethodArn]
            }]
        },
        // Passed to your Lambda as event.requestContext.authorizer
        Context = new APIGatewayCustomAuthorizerContextOutput
        {
            ["userId"]    = "usr_456",
            ["userRole"]  = "admin",
            ["tenantId"]  = "tenant_789"
        }
    };
}

Manage API via SDK (ApiGatewayV2 Management)

ApiGatewayManagementService.cs 
using Amazon.ApiGatewayV2;
using Amazon.ApiGatewayV2.Model;

public class ApiGatewayManagement
{
    private readonly IAmazonApiGatewayV2 _gw;

    public ApiGatewayManagement(IAmazonApiGatewayV2 gw)
        => _gw = gw;

    // Create a new HTTP API programmatically
    public async Task<string> CreateApiAsync(string name)
    {
        var api = await _gw.CreateApiAsync(new CreateApiRequest
        {
            Name         = name,
            ProtocolType = ProtocolType.HTTP,
            CorsConfiguration = new Cors
            {
                AllowOrigins  = ["https://app.mycompany.com"],
                AllowMethods  = ["GET", "POST", "PUT", "DELETE"],
                AllowHeaders  = ["Authorization", "Content-Type"],
                MaxAge        = 86400
            }
        });
        return api.ApiId;
    }

    // Push message to WebSocket client
    public async Task SendToConnectionAsync(
        string connectionId, string message, string endpoint)
    {
        var mgmt = new AmazonApiGatewayManagementApiClient(
            new AmazonApiGatewayManagementApiConfig
            {
                ServiceURL = endpoint
                // e.g. https://abc123.execute-api.region.amazonaws.com/prod
            });

        await mgmt.PostToConnectionAsync(
            new PostToConnectionRequest
        {
            ConnectionId = connectionId,
            Data = new MemoryStream(
                Encoding.UTF8.GetBytes(message))
        });
    }
}

✅ Best Practices & Common Mistakes

API Design

  • Use HTTP API over REST API unless you need caching, transforms, or WAF
  • Always version your API: /v1/, /v2/ — never break existing clients
  • Use path parameters for resource IDs: /orders/{orderId}
  • Use query parameters for filtering: /orders?status=pending&page=2
  • Use $default stage for simple single-stage APIs — cleaner URLs
  • Export OpenAPI spec from console — use as documentation for your team

Security

  • Always protect routes with JWT authorizer or Lambda Authorizer
  • Use specific CORS origins — never wildcard * with authenticated APIs
  • Enable AWS WAF on REST APIs to block common web attacks
  • Use resource policies to restrict access by IP or VPC
  • Rotate API keys regularly for partner integrations
  • Never put secrets in stage variables — use Secrets Manager
  • Set authorizer cache TTL to reduce Lambda invocations

Performance & Cost

  • Enable response caching on REST API for frequently read data
  • Set cache key carefully — include auth headers if responses differ per user
  • Enable Lambda Provisioned Concurrency for latency-sensitive routes
  • Use HTTP API — 70% cheaper than REST API for most cases
  • Set reasonable throttle limits — protects Lambda from concurrency spikes
  • Monitor 4XX/5XX rates in CloudWatch — set alarms on them

Operations

  • Always enable CloudWatch access logging on production stages
  • Use X-Ray tracing to debug end-to-end latency (API GW + Lambda)
  • Use canary deployments for zero-risk API updates
  • Set integration timeout below 29s — API GW max is 29s hard limit
  • Use stage variables to point to Lambda aliases (dev/staging/prod)
  • Tag all APIs and stages with App, Environment, Team for cost allocation

Common Mistakes to Avoid

⚠️ Top API Gateway Mistakes

❌ No auth on any route — public API with no protection at all
❌ CORS wildcard * with JWT auth — security misconfiguration
❌ Lambda timeout > 29s — API GW cuts it off at 29s regardless
❌ Full request/response logging in prod — logs passwords, tokens, PII
❌ No throttling configured — one bad actor can exhaust Lambda concurrency
❌ REST API when HTTP API would do — paying 3.5x more unnecessarily
❌ One Lambda for all routes — hard to scale, debug, and deploy separately
❌ No versioning — breaking change in v1 breaks all existing clients
