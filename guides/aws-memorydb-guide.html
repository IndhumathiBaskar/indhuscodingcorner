🧠 In-Memory Database

AWS MemoryDB

A Redis-compatible, durable, in-memory database — microsecond reads, millisecond writes, Multi-AZ transactions, and zero data loss. Not just a cache — a primary database.

⚡ Microsecond Reads 🔴 Redis Compatible 💾 Durable 🔁 Multi-AZ 📋 Transactions 📦 Cluster Mode 🔒 TLS + KMS 📸 Snapshots

🏗️ What is AWS MemoryDB?

A fully managed, Redis-compatible, durable in-memory database service — built for ultra-fast, persistent primary database workloads.

One-liner MemoryDB = Redis speed (microsecond reads) + database durability (Multi-AZ transaction log, zero data loss) + fully managed. Use it as a primary database, not just a cache.
⚡ Ultra-Fast Performance

Microsecond read latency. Single-digit millisecond write latency. All data served from memory — no disk I/O on the hot path. Handles millions of operations per second per shard.

💾 True Durability

Writes are persisted to a Multi-AZ transaction log before acknowledging success. Zero data loss on node failure. Data survives complete cluster restarts.

🔴 Redis Compatible

Full Redis API compatibility — all commands, data structures, Lua scripting, transactions (MULTI/EXEC), pub/sub, Sorted Sets, Streams. Drop-in replacement using StackExchange.Redis.

🔁 Multi-AZ HA

Automatic failover across AZs. Primary node failure → replica promoted in seconds. Transaction log replicated synchronously across AZs before write acknowledged.

📦 Cluster Mode

Up to 500 shards. Each shard: 1 primary + up to 5 replicas. Horizontal scaling for both reads and writes. Online resharding — add/remove shards without downtime.

📸 Snapshots & Restore

Daily automatic snapshots retained up to 35 days. Manual snapshots retained indefinitely. Restore to new cluster from any snapshot. Export snapshot to S3.

MemoryDB vs ElastiCache — The Key Difference ElastiCache Redis = cache in front of a database (data loss acceptable, eventual consistency OK). MemoryDB = the primary database itself (zero data loss, strong consistency, durability guaranteed).

🆚 MemoryDB vs ElastiCache Redis

They both run Redis — but serve very different purposes. Choosing the wrong one is expensive.

🏛️ Architecture Deep Dive

How MemoryDB achieves both Redis speed and database-grade durability simultaneously.

The Secret: Multi-AZ Transaction Log Every write to MemoryDB is written to a distributed, Multi-AZ transaction log synchronously before the write is acknowledged to the client. The primary node serves reads from memory at microsecond speed, but writes aren't confirmed until the transaction log is persisted across AZs. This gives you Redis speed for reads + database durability for writes.
📝 Write Path (Durable)

Step 1: Client sends write command to primary node

Step 2: Primary writes to in-memory store

Step 3: Write is simultaneously persisted to Multi-AZ transaction log (synchronous)

Step 4: ACK returned to client only after log persisted

Step 5: Async replication propagates to replica nodes

Latency: Single-digit milliseconds (due to sync log write)

⚡ Read Path (Ultra-Fast)

Step 1: Client sends read command

Step 2: Primary (or replica for read scale) serves from in-memory data

Step 3: Returns result — no disk I/O

Latency: Microseconds (sub-millisecond)

Reads from replicas may lag slightly behind primary (async propagation). Use primary endpoint for strong consistency reads.

Node Recovery Flow

SCENARIO: Primary node in Shard 1 fails

1. MemoryDB detects primary node failure (~10 seconds)
2. Replica with most up-to-date transaction log is elected
3. Replica replays any missing transactions from Multi-AZ log
4. Replica promoted to primary (~60 seconds total)
5. DNS endpoint updated → clients reconnect automatically
6. New replica provisioned to replace failed node

RESULT: Zero data loss — all committed writes are in transaction log
DOWNTIME: ~60 seconds for failover (automatic, no manual steps)
📸 Snapshots

Automatic: Daily snapshot, retained 0–35 days (configurable).

Manual: Trigger anytime, retained indefinitely.

Export: Export snapshot to S3 as RDB file. Import into self-managed Redis.

Restore: Create a new cluster from any snapshot point.

🌍 Multi-AZ Topology

Each shard spreads its primary and replicas across multiple AZs automatically. Transaction log is replicated across AZs — one AZ failure cannot cause data loss.

Replicas in different AZs serve read traffic, reducing primary load while maintaining geographic resilience.

🗂️ Redis Data Structures in MemoryDB

Full Redis API — all data structures work exactly as in self-managed Redis, now with durability.

🔤 STRING

Plain key-value. Text, numbers, binary (up to 512 MB). Atomic INCR/DECR. SETEX for TTL. GET/SET/MGET/MSET.

Use: Counters, tokens, cached HTML, config values, feature flags.

🗂️ HASH

Map of field → value pairs. Memory-efficient for structured objects. HGET/HSET/HMGET/HGETALL/HINCRBY.

Use: User profiles, product catalogs, session data, config objects.

📋 LIST

Ordered linked list. O(1) push/pop from head or tail. LPUSH/RPUSH/LPOP/RPOP/LRANGE/LLEN. Max 2^32-1 elements.

Use: Activity feeds, queues, recent items, timelines.

🔵 SET

Unordered unique values. O(1) add/remove/check. SADD/SMEMBERS/SISMEMBER/SINTER/SUNION/SDIFF.

Use: Tags, unique visitor tracking, friend lists, deduplication.

🏆 SORTED SET (ZSet)

Set with float score per member. Always ordered by score. ZADD/ZRANGE/ZRANGEBYSCORE/ZRANK/ZINCRBY.

Use: Leaderboards, priority queues, rate limiting, time-based queries.

📡 STREAM

Append-only log with consumer groups. XADD/XREAD/XGROUP/XACK. Fan-out, replay, exactly-once with ACKs. Now durable in MemoryDB!

Use: Event sourcing, IoT data, durable message bus.

📍 GEO

Store lat/long coordinates. Query by radius. GEOADD/GEODIST/GEOSEARCH/GEOPOS. Internally a Sorted Set.

Use: Nearby drivers, store locator, delivery tracking, geo-fencing.

🔢 BITMAP

Bit-level operations. SETBIT/GETBIT/BITCOUNT/BITOP. Extremely space-efficient. 1 bit per user for 100M users = 12.5 MB.

Use: Daily active users, feature flags, attendance tracking.

📊 HyperLogLog

Probabilistic cardinality estimation. PFADD/PFCOUNT/PFMERGE. ~0.81% error. Always uses <12 KB regardless of input size.

Use: Unique visitor count, distinct query count at massive scale.

💡 Transactions — MULTI/EXEC in MemoryDB MemoryDB supports Redis transactions via MULTI/EXEC/DISCARD/WATCH. Commands queued inside MULTI execute atomically — no other client can interleave. Now durable: if the primary fails mid-transaction, the transaction log ensures committed transactions survive.
Redis Commands MULTI/EXEC transaction — atomically update balance + log 
# Watch key for optimistic locking
WATCH user:123:balance

MULTI
  DECRBY user:123:balance 500
  INCR   user:123:txcount
  LPUSH  user:123:txlog "debit:500:2024-01-15"
EXEC

# If balance was modified between WATCH and EXEC → returns nil (retry)
# All 3 commands execute atomically if EXEC succeeds

📦 Cluster Mode & Scaling

MemoryDB always runs in cluster mode — distribute data across shards for horizontal scale.

MemoryDB always uses cluster mode Unlike ElastiCache which has both cluster-disabled and cluster-enabled options, MemoryDB always runs as a cluster. Minimum is a single shard (1 primary + optional replicas). Scale to 500 shards online without downtime.
🔢 Hash Slot Distribution

Redis Cluster uses 16,384 hash slots. Key's slot = CRC16(key) % 16384. Slots distributed evenly across shards.

3 shards example:
Shard 1: slots 0–5460
Shard 2: slots 5461–10922
Shard 3: slots 10923–16383

Use hash tags {user:123} to colocate related keys on same shard.

📈 Online Scaling

Scale out: Add shards → slot migration happens online. No downtime, no client changes.

Scale in: Remove shards → slots migrated off before removal.

Scale up: Change node type (e.g. r7g.large → r7g.xlarge). Rolling update — no downtime.

Read scale: Add replicas per shard (up to 5). Reader endpoint load-balances.

C# Durable primary database — user profile store 
// MemoryDB as PRIMARY database — no separate DB needed
public class UserProfileStore
{
    private readonly IDatabase _db;

    public UserProfileStore(IDatabase db) => _db = db;

    public async Task SaveProfileAsync(UserProfile profile)
    {
        var key = $"user:{profile.Id}";
        // Stored durably in MemoryDB — survives node failure
        await _db.HashSetAsync(key,
        [
            new HashEntry("name",      profile.Name),
            new HashEntry("email",     profile.Email),
            new HashEntry("tier",      profile.Tier),
            new HashEntry("createdAt", profile.CreatedAt.ToString("O"))
        ]);
    }

    public async Task<UserProfile?> GetProfileAsync(string userId)
    {
        var key = $"user:{userId}";
        var fields = await _db.HashGetAllAsync(key);
        if (fields.Length == 0) return null;

        return new UserProfile
        {
            Id        = userId,
            Name      = fields.FirstOrDefault(f => f.Name == "name").Value!,
            Email     = fields.FirstOrDefault(f => f.Name == "email").Value!,
            Tier      = fields.FirstOrDefault(f => f.Name == "tier").Value!,
        };
    }
}
C# Durable leaderboard with Sorted Set 
// Leaderboard as primary data — scores survive restarts
public class LeaderboardService
{
    private readonly IDatabase _db;
    private const string Key = "leaderboard:global";

    public LeaderboardService(IDatabase db) => _db = db;

    // Upsert score — atomic, durable
    public async Task<double> AddScoreAsync(string playerId, double score)
        => await _db.SortedSetAddAsync(Key, playerId, score)
             ? score
             : (double)await _db.SortedSetScoreAsync(Key, playerId);

    // Top N players (highest score first)
    public async Task<LeaderboardEntry[]> GetTopAsync(int n = 10)
    {
        var entries = await _db.SortedSetRangeByRankWithScoresAsync(
            Key, 0, n - 1, Order.Descending);

        return entries.Select((e, i) => new LeaderboardEntry
        {
            Rank     = i + 1,
            PlayerId = e.Element!,
            Score    = e.Score
        }).ToArray();
    }

    // Player's current rank (1-indexed)
    public async Task<long> GetRankAsync(string playerId)
    {
        var rank = await _db.SortedSetRankAsync(Key, playerId, Order.Descending);
        return (rank ?? -1) + 1;
    }
}

C# MULTI/EXEC transaction — atomic balance transfer 
// Atomic, durable balance transfer using Redis transactions
public async Task<bool> TransferAsync(string from, string to, decimal amount)
{
    var fromKey = $"wallet:{from}";
    var toKey   = $"wallet:{to}";

    // WATCH for optimistic locking
    var tran = db.CreateTransaction();
    tran.AddCondition(Condition.HashEqual(fromKey, "balance",
        (await db.HashGetAsync(fromKey, "balance")).ToString()));

    // Queue commands — execute atomically
    _ = tran.HashDecrementAsync(fromKey, "balance", (double)amount);
    _ = tran.HashIncrementAsync(toKey,   "balance", (double)amount);
    _ = tran.ListLeftPushAsync($"txlog:{from}",
            $"transfer:{amount}:{to}:{DateTime.UtcNow:O}");

    // Execute — returns false if condition failed (retry)
    return await tran.ExecuteAsync();
}

🔒 Security

MemoryDB is designed for primary database workloads — security is non-negotiable.

🔐 TLS In-Transit Encryption

TLS is mandatory for MemoryDB — cannot be disabled. All client-to-node and node-to-node communication is encrypted. Minimum TLS 1.2.

Clients must connect with ssl=true in connection string. AWS manages TLS certificates automatically.

Always on Cannot disable

🔑 Encryption at Rest (KMS)

All data encrypted at rest using AWS KMS. Choose between AWS-managed key (default) or customer-managed CMK for full key control and audit trail in CloudTrail.

Encryption covers: in-memory data, transaction log, snapshots, replicas.

CMK recommended for compliance workloads

👤 ACL (Access Control Lists)

Redis 6+ ACL support. Create named users with specific permissions — restrict which commands and keyspaces each user can access.

Default user: default — disable or restrict for production.

Assign ACL users to clusters. Authenticate with AUTH username password.

🌐 VPC Isolation

MemoryDB runs inside your VPC. No public endpoint — accessible only from within the VPC or via VPC peering/Transit Gateway.

Restrict Security Group inbound to port 6379 from your app's SG only. Never expose to 0.0.0.0/0.

No public internet exposure

ACL User Configuration Example

AWS CLI Create ACL user for order service — restricted permissions 
# Create ACL user with specific command and key permissions
aws memorydb create-user \
  --user-name order-service \
  --authentication-mode Type=password,Passwords=SecurePass123! \
  --access-string "on ~order:* ~leaderboard:* +GET +SET +HGET +HSET +ZADD +ZRANGE"

# Create ACL (group of users)
aws memorydb create-acl \
  --acl-name prod-acl \
  --user-names order-service admin-user

# Attach ACL to cluster
aws memorydb update-cluster \
  --cluster-name my-memorydb-cluster \
  --acl-name prod-acl

# ACL access string syntax:
# on        = user is enabled
# ~order:*  = can only access keys starting with "order:"
# +GET      = allow GET command
# -DEBUG    = deny DEBUG command
# allcommands = allow all commands
# allkeys   = allow access to all keys

💡 Best Practices

Production-grade MemoryDB setup for durability, performance, and cost efficiency.

✅ Use MemoryDB Only When You Need Durability

Don't replace ElastiCache with MemoryDB everywhere. If your use case tolerates data loss (caching, rate limiting, ephemeral sessions) → stay with ElastiCache. MemoryDB's premium pays for the durability guarantee.

✅ Key Naming Convention

Use colon-separated namespaces: user:{id}:profile, order:{id}:state, leaderboard:weekly. Consistent naming enables ACL key-pattern restrictions per service and easier debugging.

✅ Always Enable Replicas in Production

At least 1 replica per shard (ideally 2 for read-heavy workloads). Replicas in different AZs. Multi-AZ failover only works if replicas exist. Without replicas, failover requires creating a new node from the transaction log.

✅ Use Singleton ConnectionMultiplexer

Create one ConnectionMultiplexer at app startup and reuse it. Never create a new connection per request — very expensive with TLS handshake overhead. Register as singleton in DI container.

✅ Set TTLs Appropriately

Even as a primary database, set TTLs on transient data (sessions, tokens, temp state). MemoryDB charges per GB stored — expired keys that don't get TTLs accumulate and cost money.

✅ Use Hash Tags for Transactions

If you use MULTI/EXEC across multiple keys, ensure all keys share the same hash tag: order:{orderId}:state and order:{orderId}:log — both land on the same shard, enabling atomic transactions.

✅ Monitor Key Metrics

DatabaseMemoryUsagePercentage (alert at 80%) · EngineCPUUtilization · ReplicationLag · CurrConnections · Evictions (should be 0 in MemoryDB as primary DB).

✅ Test Failover Regularly

Use the FailoverShard API or AWS Fault Injection Service (FIS) to test failover. Verify your app reconnects automatically. Measure actual failover time vs your RTO requirement.

⚠️ Evictions in MemoryDB = Data Loss Unlike ElastiCache where evictions are expected behaviour for a cache, evictions in MemoryDB mean you are losing primary database records. Configure maxmemory-policy noeviction (default) and alarm on any eviction. Scale before you run out of memory.
