🔐 What is AWS Certificate Manager?

ACM provisions, manages, and auto-renews SSL/TLS certificates for your AWS resources — completely free. No more buying certificates, manually installing them, or panicking when they expire at 2am.

💡 Real-life analogy

ACM is like a government passport office — but for your website's identity. When you visit a website with 🔒, your browser asked "who is this website?" and a trusted authority (like ACM via Amazon Trust Services) issued a passport (certificate) proving the site is genuine. ACM is the passport office — it issues, renews, and revokes these digital identity documents for free.

🤝 How TLS Works — The Handshake Explained

Every HTTPS connection starts with a TLS handshake. This happens in milliseconds before any data is exchanged. Understanding this shows exactly what ACM certificates do.

TLS 1.3 Handshake — Step by Step

🤝 TLS 1.3 Handshake — Browser connects to myapp.com
1
CLIENT
ClientHello
Browser says: "I want TLS 1.3, here are cipher suites I support, here is my random nonce"
TLS_AES_256_GCM_SHA384, TLS_CHACHA20_POLY1305_SHA256...
2
SERVER
ServerHello + Certificate
Server picks cipher, sends its ACM certificate (the digital passport), and its public key
Certificate: myapp.com | Issued by: Amazon RSA 2048 M02
3
CLIENT
Certificate Validation
Browser verifies: Is the certificate for myapp.com? Is it expired? Is the CA trusted? Is it revoked?
Chain: myapp.com ← Amazon RSA 2048 M02 ← Amazon Root CA 1 ✓
4
BOTH
Key Exchange (Diffie-Hellman)
Both sides derive the same session key without ever transmitting it. Magic of asymmetric cryptography.
Session key derived — shared secret established
5
SERVER
Finished (server)
Server sends Finished message encrypted with the new session key. Proves it has the private key matching the certificate.
HMAC of full handshake — encrypted
✓
BOTH
Handshake Complete — Encrypted Channel Open
Total time: ~1ms (TLS 1.3 is 1-RTT). All subsequent data is encrypted with AES-256. The 🔒 appears in the browser.
GET / HTTP/2 → encrypted → 200 OK → encrypted

Where ACM Fits in the Connection

🌐 Browser — User visits https://myapp.com
  • TCP connection to ALB on port 443
  • Initiates TLS handshake (ClientHello)
↓
⚖️ ALB — TLS Termination (ACM cert lives here)
  • ALB presents the ACM certificate to the browser
  • Holds the private key — AWS manages it securely
  • Completes TLS handshake with the browser
  • Decrypts the HTTPS request → plain HTTP internally
↓ Plain HTTP (within VPC — safe)
🐳 Your App (ECS / EC2) — receives plain HTTP
  • Processes the request — no TLS handling needed
  • Returns response to ALB
  • ALB re-encrypts for browser if using end-to-end TLS

💡 Why is HTTPS important beyond just privacy?

Without HTTPS: browsers show "Not Secure" warnings, Google penalizes your SEO ranking, many browser APIs (geolocation, camera, service workers) refuse to work, HTTP/2 is unavailable (slower), and users lose trust. ACM makes HTTPS free — there's no reason NOT to use it.

⛓️ Certificate Trust Chain — How Browsers Trust ACM

No single certificate works in isolation. Trust is established through a chain of certificates — each one vouches for the next, all the way up to a Root CA that browsers inherently trust.

💡 Trust chain analogy

Imagine a chain of endorsements: a stranger signs a document (your cert), but your bank knows that stranger (intermediate CA), and your government knows the bank (root CA), and you trust the government. You trust the stranger's signature because you trust the whole chain.

ACM's Certificate Trust Hierarchy

🏛️
Amazon Root CA 1 ROOT CA
Self-signed · RSA 2048 · Valid until 2038 · Pre-installed in ALL browsers and operating systems · This is the anchor of trust
Thumbprint: 8D:A7:F9:65:EC:5E:FC...
🏢
Amazon RSA 2048 M02 INTERMEDIATE CA
Signed by Amazon Root CA 1 · Issues end-entity certificates · AWS rotates these periodically for security · Your browser validates this chain
Issued by: Amazon Root CA 1 · Issues to: customer certs
🌱
*.myapp.com YOUR CERT (LEAF)
Signed by Amazon RSA 2048 M02 · Valid 13 months · Contains your domain names · This is what ACM issues to you
Subject: CN=*.myapp.com · SAN: myapp.com, *.myapp.com

How Browser Validates the Chain

🌐 Browser receives *.myapp.com certificate
  • Step 1: Is the domain in the certificate? *.myapp.com matches myapp.com ✓
  • Step 2: Is it expired? Valid until Apr 2026 — current date within range ✓
↓
🔗 Verify Intermediate CA signature
  • Browser gets "Amazon RSA 2048 M02" from cert's Issuer field
  • Fetches intermediate cert (or gets it from TLS handshake)
  • Verifies *.myapp.com was signed by Amazon RSA 2048 M02 ✓
↓
🏛️ Verify Root CA signature
  • Intermediate signed by Amazon Root CA 1
  • Browser checks its built-in trusted CA list
  • "Amazon Root CA 1" IS in the trusted list ✓
  • Chain is complete — trust established!
↓
✅ Certificate trusted — 🔒 shown in browser
  • Full chain validates — connection proceeds securely
  • Any break in the chain → browser shows security warning

What makes a certificate invalid?

🔴 Certificate Errors

  • Domain mismatch: cert for api.myapp.com, visiting www.myapp.com
  • Expired: past the validity end date
  • Self-signed: root CA not in browser's trust store
  • Revoked: CA revoked the cert (breach)
  • Chain incomplete: intermediate CA not provided

✅ Why ACM never has these

  • Domain validated before issuance — always matches
  • Auto-renews 60 days early — never expires
  • Amazon Root CA is pre-trusted by all browsers
  • AWS handles revocation and CRL/OCSP
  • Chain always provided correctly in handshake

✅ Domain Validation — Proving You Own the Domain

Before issuing a certificate, ACM must verify you actually control the domain. Two methods: DNS validation (recommended) and Email validation.

DNS Validation — Recommended ✅

📋 You request certificate for myapp.com
  • ACM generates a unique CNAME record for you
  • Name: _a3b4c5d6e7.myapp.com
  • Value: _f8g9h0i1j2.acm-validations.aws
  • This proves you control the domain's DNS
↓
🌍 You add the CNAME to Route 53
  • If domain is in Route 53: one-click "Create record in Route 53" button
  • If external registrar: manually add CNAME to your DNS
  • This is a one-time setup per domain
↓
🔍 ACM continuously checks the CNAME
  • ACM queries: _a3b4c5d6e7.myapp.com → does it resolve?
  • DNS propagation: 1–30 minutes typically
  • Once ACM sees the CNAME → validation passes ✓
  • Certificate issued — usually within 5 minutes of DNS propagation
↓
✅ Certificate ISSUED — auto-renews forever
  • As long as CNAME stays in DNS, auto-renewal works every year
  • Never touch this CNAME record again
  • Works even if your website is down — DNS only!

Email Validation — When DNS Access is Limited

📋 You request certificate for myapp.com
  • ACM sends validation emails to standard addresses
  • admin@myapp.com, administrator@myapp.com
  • hostmaster@myapp.com, postmaster@myapp.com, webmaster@myapp.com
  • Also sends to WHOIS registrant email
↓
📧 You receive the email — click approve
  • Email contains a unique approval link
  • Click the link → validation complete
  • Must be done within 72 hours
  • Certificate issued within minutes of approval
↓
⚠️ Renewal requires manual click every year
  • Unlike DNS validation — email approval needed again at renewal
  • If you miss the email → certificate expires!
  • This is why DNS validation is strongly preferred

📜 Certificate Types & Real-World Uses

Certificate Coverage Types

🎯 Single Domain

  • Covers: exactly one domain
  • Example: api.myapp.com
  • Does NOT cover: www.api.myapp.com
  • Does NOT cover: myapp.com (apex)
  • Use for: single specific subdomain

⭐ Wildcard Certificate

  • Covers: *.myapp.com = all direct subdomains
  • Covers: api.myapp.com, www.myapp.com, admin.myapp.com
  • Does NOT cover: myapp.com itself (add separately)
  • Does NOT cover: sub.api.myapp.com (only one level)
  • Use for: multiple subdomains on same domain

📋 Multi-Domain (SAN)

  • Subject Alternative Names
  • One cert covers multiple different domains
  • Example: myapp.com, api.myapp.com, myother.com
  • Up to 10 names per ACM certificate
  • Use for: multiple products under one cert

🔒 Imported Certificate

  • Your own cert from any CA (DigiCert, Comodo etc.)
  • Import .pem files into ACM
  • ACM will NOT auto-renew these
  • You manage renewal yourself
  • Use for: EV certificates, code signing

💻 C# SDK — Manage ACM Programmatically

ACM Service — Full Operations

AcmService.cs 
using Amazon.CertificateManager;
using Amazon.CertificateManager.Model;

public class AcmService
{
    private readonly IAmazonCertificateManager _acm;

    public AcmService(IAmazonCertificateManager acm)
        => _acm = acm;

    // ── Request certificate ────────────────────────
    public async Task<string> RequestCertificateAsync(
        string primaryDomain,
        List<string> additionalNames)
    {
        var response = await _acm.RequestCertificateAsync(
            new RequestCertificateRequest
        {
            DomainName              = primaryDomain,
            // e.g. "myapp.com"
            SubjectAlternativeNames = additionalNames,
            // e.g. ["*.myapp.com", "api.myapp.com"]
            ValidationMethod        = ValidationMethod.DNS,
            KeyAlgorithm            = KeyAlgorithm.RSA_2048,
            Tags = [
                new Tag { Key="Environment", Value="production" },
                new Tag { Key="ManagedBy",  Value="terraform"   }
            ]
        });
        return response.CertificateArn;
    }

    // ── Get DNS validation records ─────────────────
    public async Task<List<ResourceRecord>>
        GetValidationRecordsAsync(string certArn)
    {
        var detail = await _acm.DescribeCertificateAsync(
            new DescribeCertificateRequest
            { CertificateArn = certArn });

        return detail.Certificate.DomainValidationOptions
            .Where(d => d.ResourceRecord != null)
            .Select(d => d.ResourceRecord!)
            .ToList();
        // Returns: Name + Type (CNAME) + Value
        // Add these to your DNS to validate
    }

    // ── Poll until issued ──────────────────────────
    public async Task<string> WaitForIssuanceAsync(
        string certArn,
        TimeSpan? timeout = null)
    {
        var deadline = DateTime.UtcNow +
            (timeout ?? TimeSpan.FromMinutes(30));

        while (DateTime.UtcNow < deadline)
        {
            var detail = await _acm.DescribeCertificateAsync(
                new DescribeCertificateRequest
                { CertificateArn = certArn });

            var status = detail.Certificate.Status;

            if (status == CertificateStatus.ISSUED)
                return certArn;

            if (status == CertificateStatus.FAILED)
                throw new Exception(
                    $"Certificate issuance failed: {detail.Certificate.FailureReason}");

            await Task.Delay(TimeSpan.FromSeconds(15));
        }
        throw new TimeoutException("Certificate not issued within timeout");
    }

    // ── List all certificates ──────────────────────
    public async Task<List<CertificateSummary>>
        ListCertificatesAsync(
            CertificateStatus? filterStatus = null)
    {
        var request = new ListCertificatesRequest();
        if (filterStatus.HasValue)
            request.CertificateStatuses = [filterStatus.Value.Value];

        var response = await _acm.ListCertificatesAsync(request);
        return response.CertificateSummaryList;
    }

    // ── Import existing certificate ────────────────
    public async Task<string> ImportCertificateAsync(
        string certPem,        // certificate body
        string privateKeyPem,  // private key
        string chainPem)       // certificate chain
    {
        var response = await _acm.ImportCertificateAsync(
            new ImportCertificateRequest
        {
            Certificate      = new MemoryStream(
                Encoding.UTF8.GetBytes(certPem)),
            PrivateKey       = new MemoryStream(
                Encoding.UTF8.GetBytes(privateKeyPem)),
            CertificateChain = new MemoryStream(
                Encoding.UTF8.GetBytes(chainPem))
        });
        return response.CertificateArn;
    }

    // ── Check certs expiring soon ──────────────────
    public async Task<List<CertificateSummary>>
        GetExpiringCertificatesAsync(int withinDays = 30)
    {
        var certs = await _acm.ListCertificatesAsync(
            new ListCertificatesRequest());

        var expiring = new List<CertificateSummary>();
        foreach (var cert in certs.CertificateSummaryList)
        {
            var detail = await _acm.DescribeCertificateAsync(
                new DescribeCertificateRequest
                { CertificateArn = cert.CertificateArn });

            var expiry = detail.Certificate.NotAfter;
            if (expiry.HasValue &&
                expiry.Value <= DateTime.UtcNow
                    .AddDays(withinDays))
            {
                expiring.Add(cert);
            }
        }
        return expiring;
    }
}

✅ Best Practices & Common Mistakes

Certificate Requests

  • Always request *.myapp.com AND myapp.com together as one certificate — wildcard doesn't cover apex
  • Always use DNS validation — email validation breaks auto-renewal
  • For CloudFront: request the certificate in us-east-1 ONLY — regardless of your region
  • For ALB/API GW: request in the same region as the resource
  • Use ECDSA P-256 for performance-critical APIs — smaller keys, faster handshakes
  • Tag every certificate: App, Environment, Owner, ExpiryYear

DNS Validation Records

  • Never delete the CNAME validation records after issuance — auto-renewal needs them permanently
  • Use Route 53 for DNS when possible — one-click validation setup
  • If using external DNS (Cloudflare): add the full CNAME name including your domain
  • One CNAME record validates both myapp.com AND *.myapp.com — they share the same record
  • Verify validation with: dig _abc.myapp.com CNAME — should return the ACM value

TLS Configuration

  • Always add HTTP→HTTPS redirect listener on port 80 of your ALB
  • Use TLS policy: ELBSecurityPolicy-TLS13-1-2-2021-06 — supports TLS 1.2 and 1.3
  • Never allow TLS 1.0 or 1.1 — deprecated and insecure
  • Enable HSTS header in your app — tells browsers to always use HTTPS
  • Test your TLS configuration with SSL Labs (ssllabs.com/ssltest) — aim for A+ rating

Monitoring & Alerts

  • Set up EventBridge rule for ACM expiry events — your safety net if auto-renewal fails
  • Check certificate status monthly — look for any in FAILED or INACTIVE state
  • Audit "in use by" field — remove certs not attached to any resource (unnecessary cost risk)
  • Use the C# GetExpiringCertificates function in a scheduled Lambda for proactive monitoring

Common Mistakes to Avoid

⚠️ Top ACM Mistakes

❌ Wildcard only (*.myapp.com) — forgetting the apex (myapp.com) — browsers reject myapp.com with no cert match
❌ Deleting validation CNAME records — auto-renewal silently fails, cert expires a year later
❌ CloudFront cert in wrong region — it MUST be us-east-1, no exceptions
❌ Using email validation — requires manual approval every year, easy to miss
❌ A record pointing to ALB IP instead of alias — combined with cert expiry = total outage
❌ No HTTP→HTTPS redirect — users on http:// see no lock, mixed content warnings
❌ Allowing TLS 1.0/1.1 in security policy — vulnerable to POODLE, BEAST attacks
❌ No expiry alerting — auto-renewal can fail silently if DNS changes
