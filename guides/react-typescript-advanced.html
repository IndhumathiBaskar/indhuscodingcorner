🐻 Zustand — Global State Management

A minimal state management library — no Provider wrapping, no boilerplate, just a hook.

Why Zustand over Context for global state? Context re-renders every consumer when any value changes. Zustand lets components subscribe to only the specific slice of state they need — so a cart count update doesn't re-render a component that only reads the user's name.
Terminal Install Zustand 
npm install zustand
TSX store/useCartStore.ts — a fully typed Zustand store 
import { create } from 'zustand';

type CartItem = { id: number; name: string; price: number; quantity: number };

// 1. Define the shape of state AND the actions together
type CartState = {
  items: CartItem[];
  addItem: (item: Omit<CartItem, 'quantity'>) => void;
  removeItem: (id: number) => void;
  clearCart: () => void;
  totalPrice: () => number;     // derived value, computed on demand
};

// 2. create()(...) — the extra () is required for TypeScript inference
export const useCartStore = create<CartState>()((set, get) => ({
  items: [],

  addItem: (item) => set((state) => {
    const existing = state.items.find(i => i.id === item.id);
    if (existing) {
      return {
        items: state.items.map(i =>
          i.id === item.id ? { ...i, quantity: i.quantity + 1 } : i
        ),
      };
    }
    return { items: [...state.items, { ...item, quantity: 1 }] };
  }),

  removeItem: (id) => set((state) => ({
    items: state.items.filter(i => i.id !== id),
  })),

  clearCart: () => set({ items: [] }),

  totalPrice: () => get().items.reduce((sum, i) => sum + i.price * i.quantity, 0),
}));
TSX Using the store — no Provider needed anywhere 
import { useCartStore } from '../../store/useCartStore';

function CartBadge() {
  // Subscribes ONLY to items.length — only re-renders when count changes
  const itemCount = useCartStore((state) => state.items.length);
  return <span>🛒 {itemCount}</span>;
}

function ProductCard({ product }: { product: { id: number; name: string; price: number } }) {
  // Only needs the action, not the items array — no unnecessary re-renders
  const addItem = useCartStore((state) => state.addItem);

  return (
    <button onClick={() => addItem(product)}>
      Add to Cart
    </button>
  );
}

function CartSummary() {
  const items = useCartStore((state) => state.items);
  const total = useCartStore((state) => state.totalPrice());

  return (
    <div>
      {items.map(i => <p key={i.id}>{i.name} × {i.quantity}</p>)}
      <strong>Total: ${total.toFixed(2)}</strong>
    </div>
  );
}

Persisting State (e.g. cart survives page refresh)

TSX store/useAuthStore.ts — using the persist middleware 
import { create } from 'zustand';
import { persist } from 'zustand/middleware';

type User = { id: number; name: string };

type AuthState = {
  user: User | null;
  token: string | null;
  login: (user: User, token: string) => void;
  logout: () => void;
};

// persist() wraps the store and automatically syncs to localStorage
export const useAuthStore = create<AuthState>()(
  persist(
    (set) => ({
      user: null,
      token: null,
      login: (user, token) => set({ user, token }),
      logout: () => set({ user: null, token: null }),
    }),
    { name: 'auth-storage' }   // localStorage key
  )
);

// User stays logged in even after a page refresh — no extra code needed!

🌐 Axios — Typed HTTP Client

A more capable alternative to the browser's built-in fetch — with interceptors, automatic JSON parsing, and clean TypeScript generics.

🆚 Axios vs fetch

fetch: built-in, but you must manually call .json(), manually check response.ok, no built-in interceptors or timeout.

Axios: auto-parses JSON, throws on non-2xx by default, built-in interceptors, request cancellation, upload progress.

🔌 What's an Interceptor?

A function that runs on every request or response, automatically. Use it to attach an auth token to every outgoing request, or to handle 401 errors globally in one place.

Terminal Install Axios 
npm install axios
TSX api/axiosClient.ts — one shared, configured instance 
import axios from 'axios';
import { useAuthStore } from '../store/useAuthStore';

// One configured instance, reused everywhere — never call axios.get() directly
export const axiosClient = axios.create({
  baseURL: 'https://api.myapp.com',
  timeout: 10000,
  headers: { 'Content-Type': 'application/json' },
});

// REQUEST interceptor — attach the auth token to every outgoing request
axiosClient.interceptors.request.use((config) => {
  const token = useAuthStore.getState().token;
  if (token) {
    config.headers.Authorization = `Bearer ${token}`;
  }
  return config;
});

// RESPONSE interceptor — handle expired sessions globally, in one place
axiosClient.interceptors.response.use(
  (response) => response,
  (error) => {
    if (error.response?.status === 401) {
      useAuthStore.getState().logout();
      window.location.href = '/login';
    }
    return Promise.reject(error);
  }
);
TSX api/userApi.ts — typed API functions, one per endpoint 
import { axiosClient } from './axiosClient';

export type User = { id: number; name: string; email: string };
export type CreateUserPayload = { name: string; email: string };

// Each function is typed: what it sends AND what it returns
export const userApi = {
  getAll: async (): Promise<User[]> => {
    const res = await axiosClient.get<User[]>('/users');
    return res.data;   // axios auto-parses JSON — no .json() call needed
  },

  getById: async (id: number): Promise<User> => {
    const res = await axiosClient.get<User>(`/users/${id}`);
    return res.data;
  },

  create: async (payload: CreateUserPayload): Promise<User> => {
    const res = await axiosClient.post<User>('/users', payload);
    return res.data;
  },

  update: async (id: number, payload: Partial<CreateUserPayload>): Promise<User> => {
    const res = await axiosClient.patch<User>(`/users/${id}`, payload);
    return res.data;
  },

  delete: async (id: number): Promise<void> => {
    await axiosClient.delete(`/users/${id}`);
  },
};
TSX Using the typed API in a component 
import { useEffect, useState } from 'react';
import { userApi } from '../../api/userApi';
import type { User } from '../../api/userApi';

function UserList() {
  const [users, setUsers] = useState<User[]>([]);

  useEffect(() => {
    userApi.getAll().then(setUsers).catch(console.error);
  }, []);

  const handleDelete = async (id: number) => {
    await userApi.delete(id);
    setUsers(users.filter(u => u.id !== id));   // optimistic UI update
  };

  return (
    <ul>
      {users.map(u => (
        <li key={u.id}>
          {u.name} <button onClick={() => handleDelete(u.id)}>Delete</button>
        </li>
      ))}
    </ul>
  );
}
⚠️ Always handle errors Axios throws an error for any non-2xx response — wrap calls in try/catch or attach .catch(). Check error.response?.status for the HTTP status code, and error.response?.data for the API's error body.

🔗 Combining Zustand + Axios

The real production pattern — async actions inside a Zustand store, calling typed Axios functions.

The Pattern Instead of fetching data inside component useEffects scattered everywhere, put the fetch logic inside the store itself as an async action. Components just call store.fetchUsers() and read store.users / store.loading.
TSX store/useUserStore.ts — store with built-in async data fetching 
import { create } from 'zustand';
import { userApi } from '../api/userApi';
import type { User } from '../api/userApi';

type UserState = {
  users: User[];
  loading: boolean;
  error: string | null;
  fetchUsers: () => Promise<void>;
  addUser: (name: string, email: string) => Promise<void>;
  removeUser: (id: number) => Promise<void>;
};

export const useUserStore = create<UserState>()((set, get) => ({
  users: [],
  loading: false,
  error: null,

  fetchUsers: async () => {
    set({ loading: true, error: null });
    try {
      const users = await userApi.getAll();
      set({ users, loading: false });
    } catch (err) {
      set({ error: 'Failed to load users', loading: false });
    }
  },

  addUser: async (name, email) => {
    const newUser = await userApi.create({ name, email });
    set({ users: [...get().users, newUser] });   // append without re-fetching
  },

  removeUser: async (id) => {
    await userApi.delete(id);
    set({ users: get().users.filter(u => u.id !== id) });
  },
}));
TSX pages/Users/Users.tsx — the component becomes very thin 
import { useEffect } from 'react';
import { useUserStore } from '../../store/useUserStore';

function Users() {
  const { users, loading, error, fetchUsers, removeUser } = useUserStore();

  useEffect(() => {
    fetchUsers();   // trigger once on mount — logic lives in the store, not here
  }, [fetchUsers]);

  if (loading) return <p>Loading...</p>;
  if (error) return <p>{error}</p>;

  return (
    <ul>
      {users.map(u => (
        <li key={u.id}>
          {u.name} <button onClick={() => removeUser(u.id)}>✕</button>
        </li>
      ))}
    </ul>
  );
}
💡 Why this is better The data-fetching logic lives in exactly one place (the store) instead of being duplicated across every component that needs the user list. Any component anywhere can call useUserStore.getState().fetchUsers() or read the same cached data — no prop drilling, no duplicate API calls.

🧪 Jest + React Testing Library

Jest runs your tests. React Testing Library renders components and queries them the way a real user would.

RTL's Guiding Principle "The more your tests resemble the way your software is used, the more confidence they can give you." Query elements by what a user sees (text, labels, roles) — not by implementation details like CSS class names or component internals.

TSX components/Button/Button.test.tsx — first component test 
import { render, screen } from '@testing-library/react';
import userEvent from '@testing-library/user-event';
import { describe, it, expect, vi } from 'vitest';  // or 'jest' if using Jest directly
import Button from './Button';

describe('Button', () => {
  it('renders the label text', () => {
    render(<Button label="Save" onClick={() => {}} />);

    // Query by role + accessible name — exactly how a screen reader user finds it
    const button = screen.getByRole('button', { name: 'Save' });
    expect(button).toBeInTheDocument();
  });

  it('calls onClick when clicked', async () => {
    const handleClick = vi.fn();   // jest.fn() if using Jest directly
    render(<Button label="Save" onClick={handleClick} />);

    const user = userEvent.setup();
    await user.click(screen.getByRole('button', { name: 'Save' }));

    expect(handleClick).toHaveBeenCalledTimes(1);
  });

  it('is disabled when the disabled prop is true', () => {
    render(<Button label="Save" onClick={() => {}} disabled />);
    expect(screen.getByRole('button', { name: 'Save' })).toBeDisabled();
  });
});

TSX store/useCartStore.test.ts — testing a Zustand store directly 
import { useCartStore } from './useCartStore';
import { beforeEach, describe, it, expect } from 'vitest';

describe('useCartStore', () => {
  // Reset the store before every test — stores are singletons, state leaks between tests otherwise
  beforeEach(() => {
    useCartStore.setState({ items: [] });
  });

  it('adds a new item to the cart', () => {
    useCartStore.getState().addItem({ id: 1, name: 'Keyboard', price: 49.99 });

    const items = useCartStore.getState().items;
    expect(items).toHaveLength(1);
    expect(items[0].quantity).toBe(1);
  });

  it('increments quantity when adding the same item twice', () => {
    const { addItem } = useCartStore.getState();
    addItem({ id: 1, name: 'Keyboard', price: 49.99 });
    addItem({ id: 1, name: 'Keyboard', price: 49.99 });

    const items = useCartStore.getState().items;
    expect(items).toHaveLength(1);          // still one entry
    expect(items[0].quantity).toBe(2);  // quantity incremented
  });

  it('calculates total price correctly', () => {
    const { addItem, totalPrice } = useCartStore.getState();
    addItem({ id: 1, name: 'Keyboard', price: 50 });
    addItem({ id: 2, name: 'Mouse', price: 25 });

    expect(totalPrice()).toBe(75);
  });
});
TSX api/userApi.test.ts — mocking Axios with vitest 
import { vi, describe, it, expect, beforeEach } from 'vitest';
import { axiosClient } from './axiosClient';
import { userApi } from './userApi';

// Mock the entire axios client module
vi.mock('./axiosClient');

describe('userApi', () => {
  beforeEach(() => {
    vi.clearAllMocks();
  });

  it('getAll returns the list of users', async () => {
    const mockUsers = [{ id: 1, name: 'Indhu', email: 'indhu@example.com' }];

    // Make the mocked axios.get resolve with our fake data
    vi.mocked(axiosClient.get).mockResolvedValueOnce({ data: mockUsers });

    expect(axiosClient.get).toHaveBeenCalledWith('/users');
    expect(result).toEqual(mockUsers);

  it('create sends a POST with the correct payload', async () => {

TSX hooks/useDebounce.test.ts — testing a custom hook with renderHook 
import { renderHook, act } from '@testing-library/react';
import { vi, describe, it, expect } from 'vitest';
import { useDebounce } from './useDebounce';

describe('useDebounce', () => {
  it('returns the debounced value after the delay', () => {
    vi.useFakeTimers();   // control time manually instead of waiting in real-time

    const { result, rerender } = renderHook(
      ({ value }) => useDebounce(value, 500),
      { initialProps: { value: 'a' } }
    );

    expect(result.current).toBe('a');

    rerender({ value: 'ab' });
    expect(result.current).toBe('a');   // hasn't updated yet — delay not elapsed

    act(() => { vi.advanceTimersByTime(500); });
    expect(result.current).toBe('ab');  // now updated

    vi.useRealTimers();
  });
});

💡 Best Practices

Production wisdom for the full stack — Zustand, Axios, and testing.

✅ One Store per Domain

Don't put everything in one giant Zustand store. useAuthStore, useCartStore, useUiStore — separate concerns mean smaller, more testable, more focused files.

✅ Never Call axios.get() Directly in Components

Always go through a typed API function (userApi.getAll()). This centralises the URL, the type, and any transformation logic in one place — and makes mocking in tests trivial.

✅ Put Async Logic in the Store, Not the Component

Components should call store.fetchUsers() and read store.loading — not contain their own useEffect + useState fetching logic duplicated everywhere.

✅ Reset Store State Between Tests

Zustand stores are singletons — state persists across test files unless you reset it. Use beforeEach(() => store.setState(initialState)) to avoid test pollution.

✅ Query by Role and Text, Not Test IDs

Prefer getByRole/getByText over data-testid wherever possible — it tests your app the way real users and screen readers experience it.

✅ Mock at the Network Boundary

Mock the Axios client (or use MSW — Mock Service Worker) rather than mocking your own API functions. This tests more of your real code path while still avoiding real network calls.

✅ Use Interceptors for Cross-Cutting Concerns

Auth token attachment, global 401 handling, request logging — these belong in Axios interceptors, written once, rather than repeated in every API call.

✅ Don't Test Implementation Details

Avoid asserting on internal state shape or component internals. Test what the user sees and can do — button text, displayed values, whether something is disabled.
