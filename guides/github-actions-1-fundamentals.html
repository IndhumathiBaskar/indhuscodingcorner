⚙️ What Is a GitHub Actions Workflow?

You push code to GitHub. You want tests to run automatically, a Docker image to build, and — if everything passes on the main branch — the app to deploy to AWS. Doing this by hand every time is slow and error-prone. GitHub Actions watches your repository for events (a push, a pull request, a schedule) and runs a YAML-defined set of instructions in response — automatically, every single time.
🤔 Is this the same as Jenkins or other CI tools?

Same goal (automate build/test/deploy), different home. Jenkins is self-hosted and tool-agnostic — you install it anywhere. GitHub Actions lives INSIDE GitHub itself: your workflow files sit in your repo, trigger off GitHub-native events (PRs, issues, releases) and the entire ecosystem of 20,000+ pre-built "Actions" is one line away. If your code is already on GitHub, Actions removes an entire layer of external tooling.

The Core Hierarchy — Just Four Levels

⚡ Workflow → Job → Step → Action
🤔 What exactly is an "Action" vs a "step"?

A step is one instruction inside a job — it either runs a shell command (run:) or calls a reusable package called an Action (uses:). An Action is pre-built, versioned code published to the GitHub Marketplace — e.g. actions/checkout@v4 clones your repo, aws-actions/configure-aws-credentials@v4 sets up AWS auth. You never have to write that logic yourself — just reference the Action by name and version.

Key Concepts

📄 Workflow

One YAML file = one workflow. Lives in .github/workflows/. Triggered by events. Contains one or more jobs.

💼 Job

A set of steps that run on the same runner (VM). Jobs run in parallel by default unless you declare dependencies.

👟 Step

One instruction inside a job — either run: a shell command or uses: a pre-built Action.

🧩 Action

Reusable, versioned packaged code from the Marketplace (or your own). Referenced as owner/repo@version.

🖥️ Runner

The VM that actually executes your job — GitHub-hosted (Ubuntu/Windows/macOS) or your own self-hosted machine.

⚡ Event

What triggers the workflow — a push, pull request, schedule, manual click, or another workflow finishing.

What the YAML Looks Like

YAML — The basics at a glance
# .github/workflows/ci.yml
name: CI

# WHEN does this run?
on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

# WHAT runs?
jobs:
  test:                              # job ID — any name you choose
    runs-on: ubuntu-latest          # which VM to run on
    steps:
      - name: Checkout code
        uses: actions/checkout@v4     # pre-built Action

      - name: Set up Node.js
        uses: actions/setup-node@v4
        with:
          node-version: "20"

      - name: Install & test
        run: |                       # shell command(s) instead of an Action
          npm ci
          npm test

# Address pattern for referencing things:
# ${{ secrets.NAME }}      — a secret
# ${{ github.ref }}        — workflow context info
# ${{ steps.STEP_ID.outputs.X }} — a previous step's output

💡 Jobs Run in Parallel by Default

Unless you add needs: [job_name], every job in a workflow starts at the same time on separate runners. This is GitHub Actions' equivalent of Terraform's implicit dependency graph — except here, YOU must declare it explicitly with needs:.

📁 Folder Structure & Triggers

Unlike GitLab (one .gitlab-ci.yml file) or Terraform (flexible folder layout), GitHub Actions has ONE fixed rule: every workflow file must live in .github/workflows/ at your repo root. Beyond that, you decide how many files, what each one is responsible for, and what triggers each one.

Standard Folder Layout

my-app/ ├── .github/ │ └── workflows/ # THE ONLY required location — fixed by GitHub │ ├── ci.yml # runs tests on every push/PR │ ├── deploy-staging.yml # deploys to staging on merge to develop │ ├── deploy-prod.yml # deploys to prod on merge to main (often with approval) │ ├── pr-checks.yml # linting/security scans on every PR │ └── nightly.yml # scheduled cron job — e.g. dependency audit ├── src/ ├── Dockerfile └── package.json
🤔 Why split into multiple files instead of one big workflow?

Each YAML file is an independently triggered, independently visible workflow in the GitHub UI ("Actions" tab). Splitting by purpose (CI vs deploy-staging vs deploy-prod) means you can re-run just the failed one, see clear pass/fail history per concern, and avoid one giant file becoming unreadable. This is the opposite of GitLab's single-file philosophy — and is GitHub's biggest structural difference.

Trigger Types — the on: Keyword

📤 push

Runs when commits are pushed. Filter by branches:, tags:, or paths: to avoid running on every tiny change.

🔀 pull_request

Runs on PR open/update. Use pull_request_target carefully — it runs with base branch permissions, a common security trap.

👆 workflow_dispatch

Manual trigger — adds a "Run workflow" button in the GitHub UI. Can accept typed inputs: like a form.

⏰ schedule

Cron-based trigger. cron: '0 2 * * *' = every day at 2am UTC. Good for nightly builds, dependency scans.

🏷️ release

Runs when a GitHub Release is published — common trigger for production deployment workflows.

🔗 workflow_call

Makes this workflow reusable — callable FROM another workflow. The foundation of "reusable workflows" (Part 3).

Triggers in Practice

YAML — Combining multiple triggers with filters
on:
  # Run on push, but only to main/develop, and only if relevant files changed
  push:
    branches: [main, develop]
    paths:
      - 'src/**'
      - 'package.json'
    paths-ignore:
      - '**.md'           # skip docs-only changes

  # Run on every PR targeting main
  pull_request:
    branches: [main]
    types: [opened, synchronize, reopened]

  # Manual trigger with typed inputs — shows up as a form in the GitHub UI
  workflow_dispatch:
    inputs:
      environment:
        description: 'Target environment'
        required: true
        type: choice
        options: [staging, production]

  # Scheduled — nightly dependency audit at 2am UTC
  schedule:
    - cron: '0 2 * * *'

⚠️ schedule: Only Runs on the Default Branch

Scheduled workflows only trigger from whatever YAML exists on your repo's default branch (usually main) — editing the cron schedule in a feature branch PR has no effect until merged.

💼 Jobs & Steps — The Execution Model

A workflow can have many jobs. By default they all start simultaneously on separate runners — fast, but sometimes wrong (you don't want "deploy" starting before "test" finishes). needs: is how you build a dependency graph, GitHub's answer to Terraform's automatic dependency detection — except here it's always explicit.
🤔 Why would I want multiple jobs instead of one job with more steps?

Separate jobs run on SEPARATE runners — meaning they can run in parallel (faster overall pipeline) and each gets a clean environment. Steps inside ONE job always run sequentially on the SAME runner and share filesystem state. Use multiple jobs when tasks are independent (lint + unit-test can run at once) or need different runners (build on Linux, but a separate signing job needs Windows).

Job Dependency Graph

⚡ needs: Builds a DAG

The Pattern in YAML

YAML — Multiple jobs with needs: and outputs
jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm run lint

  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm test

  build:
    needs: [lint, test]      # waits for BOTH to succeed
    runs-on: ubuntu-latest
    outputs:
      image-tag: ${{ steps.meta.outputs.tag }}  # pass data to the next job
    steps:
      - uses: actions/checkout@v4
      - id: meta                       # step ID — referenced above
        run: echo "tag=$(git rev-parse --short HEAD)" >> $GITHUB_OUTPUT
      - run: docker build -t myapp:${{ steps.meta.outputs.tag }} .

  deploy:
    needs: build                   # single dependency, no brackets needed
    runs-on: ubuntu-latest
    steps:
      - run: echo "Deploying image ${{ needs.build.outputs.image-tag }}"
💡 Aha Moment

Notice needs.build.outputs.image-tag in the deploy job — that's how data flows BETWEEN jobs (which run on separate, isolated VMs). Within ONE job, steps share a filesystem so you don't need this; across jobs, outputs: + $GITHUB_OUTPUT is the only bridge.

🔐 Secrets & Variables

Your deploy job needs an AWS access key. You absolutely cannot put that in the YAML file — it's committed to git, visible to anyone with repo access, forever in history. GitHub gives you two separate stores: Secrets (encrypted, masked in logs) and Variables (plain text, for non-sensitive config) — both settable at repository, environment, or organization level.
🤔 Secrets vs Variables — when do I use which?

If a value leaking would be a security incident (API keys, passwords, tokens) → Secret. GitHub automatically masks secret values in workflow logs (replaced with ***) even if you accidentally echo them. If it's just configuration that's fine to be public (an AWS region name, a Node version, a feature flag) → Variable. Using a Secret for everything makes debugging harder since you can never see the actual value in logs.

Where Secrets/Variables Live

📦 Repository level

Settings → Secrets and variables. Available to all workflows in that repo. Good default for most projects.

🌍 Environment level

Tied to a named Environment (e.g. "production"). Can require manual approval before a job using it runs — critical for prod deploys.

🏢 Organization level

Shared across multiple repos in your GitHub org. Can restrict which repos may access a given org secret.

Using Secrets in a Workflow

YAML — AWS credentials via OIDC (no long-lived keys!) + secrets
jobs:
  deploy:
    runs-on: ubuntu-latest
    permissions:
      id-token: write          # required for OIDC — no AWS access keys stored at all!
      contents: read
    environment: production     # ties this job to env-scoped secrets + approval gates
    steps:
      - uses: actions/checkout@v4

      # Modern AWS auth — assumes an IAM role, no static keys to leak
      - name: Configure AWS credentials
        uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: ${{ secrets.AWS_DEPLOY_ROLE_ARN }}
          aws-region: ${{ vars.AWS_REGION }}      # vars. = a Variable, not secret

      - name: Deploy to ECS
        run: |
          aws ecs update-service \
            --cluster ${{ vars.ECS_CLUSTER }} \
            --service ${{ vars.ECS_SERVICE }} \
            --force-new-deployment

      # Example: a database connection string — genuinely sensitive
      - name: Run migrations
        env:
          DB_CONNECTION: ${{ secrets.PROD_DB_CONNECTION_STRING }}
        run: dotnet ef database update

⚠️ Secrets Are NOT Available to Forked-Repo PRs

By design, workflows triggered by a pull request from a FORK don't get access to repository secrets — this prevents a malicious PR from exfiltrating your AWS keys. This is a common "why isn't my secret working" trap for open-source projects with external contributors.

OIDC vs Long-Lived Access Keys

🔑 Old way — Access Keys

Store AWS_ACCESS_KEY_ID + AWS_SECRET_ACCESS_KEY as secrets. Works, but the keys are long-lived — if leaked, valid until manually rotated.

🛡️ Modern way — OIDC

GitHub issues a short-lived OIDC token; AWS IAM trusts GitHub's identity provider and grants temporary credentials via sts:AssumeRoleWithWebIdentity. No secret AWS keys stored anywhere, ever.

🖥️ Runners & Environments

Every job needs somewhere to actually execute — that's the runner. GitHub gives you fully managed VMs for free (within limits), or you can run your own. Environments are a separate concept — a named deployment target (staging, production) that can require manual approval, restrict which branches can deploy to it, and scope secrets.

Environments — Approval Gates for Production

⚡ Environment Protection Rules
YAML — A job gated by a protected environment
jobs:
  deploy-production:
    runs-on: ubuntu-latest
    environment:
      name: production
      url: https://myapp.com     # shows as a clickable link in the run summary
    steps:
      - run: echo "Deploying to production..."
    # This job will PAUSE and wait for a manual approval click
    # in the GitHub UI if "production" has required reviewers configured

💡 Self-Hosted Runners for AWS-Native Pipelines

For teams already running on AWS, a common pattern is a self-hosted runner on an EC2 instance (or ECS/Fargate via auto-scaling runner controllers) inside your private VPC — lets the runner reach private RDS instances or internal services directly, something GitHub-hosted runners can't do without a VPN/tunnel.

📦 Artifacts & Caching

Two related but different problems. Artifacts: your build job produces a compiled binary or test report — how does the deploy job (on a totally separate, fresh VM) get that file? Caching: re-downloading every npm/NuGet package on every single run is slow — how do you skip that when nothing changed?
🤔 Aren't artifacts and cache basically the same thing?

No — different purpose, different lifecycle. Artifacts are the actual OUTPUT of your pipeline (a built .zip, test results, a Docker image tarball) — meant to be downloaded, inspected, or passed to another job/workflow, kept for a defined retention period (default 90 days). Cache is purely a speed optimisation for INPUTS (dependencies) — it's a best-effort, can be evicted anytime, and you should never rely on it for correctness.

Passing Build Output Between Jobs

YAML — upload-artifact in build, download-artifact in deploy
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: dotnet publish -c Release -o ./publish

      - name: Upload build artifact
        uses: actions/upload-artifact@v4
        with:
          name: app-build
          path: ./publish
          retention-days: 7

  deploy:
    needs: build
    runs-on: ubuntu-latest     # a FRESH VM — has none of build's files
    steps:
      - name: Download build artifact
        uses: actions/download-artifact@v4
        with:
          name: app-build
          path: ./publish
      - run: ls ./publish    # the files from build are now here

Speeding Up Builds with Cache

YAML — caching npm and NuGet dependencies
steps:
  - uses: actions/checkout@v4

  # Manual cache — key changes when package-lock.json changes, invalidating old cache
  - name: Cache node_modules
    uses: actions/cache@v4
    with:
      path: ~/.npm
      key: ${{ runner.os }}-npm-${{ hashFiles('package-lock.json') }}
      restore-keys: |
        ${{ runner.os }}-npm-

  - run: npm ci

  # Many setup-* actions have caching BUILT IN — simpler, recommended when available
  - uses: actions/setup-dotnet@v4
    with:
      dotnet-version: "8.0"
      cache: true            # handles NuGet caching automatically
💡 Aha Moment

Notice the cache key includes hashFiles('package-lock.json') — if your lockfile changes, the hash changes, the key changes, and you automatically get a FRESH cache instead of stale dependencies. This hash-based invalidation pattern shows up everywhere in CI caching, across every platform.

🔢 Matrix Builds & Conditionals

You need to test your app against Node 18, 20, AND 22 — on Ubuntu AND Windows. Writing 6 separate jobs would be insane. strategy.matrix generates every combination automatically from a short list. Combined with if: conditionals, you get fine control over exactly when each job or step actually runs.

Matrix Builds — One Job, Many Combinations

YAML — testing across multiple Node versions and OSes
jobs:
  test:
    runs-on: ${{ matrix.os }}
    strategy:
      matrix:
        os: [ubuntu-latest, windows-latest]
        node-version: ["18", "20", "22"]
      fail-fast: false            # don't cancel other combos if one fails
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node-version }}
      - run: npm test
    # Generates 2 × 3 = 6 jobs automatically:
    # (ubuntu,18) (ubuntu,20) (ubuntu,22) (windows,18) (windows,20) (windows,22)

Conditionals — the if: Keyword

YAML — controlling exactly when jobs/steps run
jobs:
  deploy:
    # Only deploy if this is a push directly to main (not a PR, not another branch)
    if: github.ref == 'refs/heads/main' && github.event_name == 'push'
    runs-on: ubuntu-latest
    steps:
      - run: echo "Deploying..."

      # Step-level conditional — only notify Slack if a PREVIOUS step failed
      - name: Notify failure
        if: failure()
        run: curl -X POST $SLACK_WEBHOOK -d "Deploy failed!"

      # Always run cleanup, even if earlier steps failed
      - name: Cleanup
        if: always()
        run: echo "Cleaning up temp resources"

Common Status Check Functions

✅ success()

True if all previous steps succeeded. This is the implicit default if you write no if: at all.

❌ failure()

True if any previous step in the job failed. Useful for sending failure notifications.

♾️ always()

Runs regardless of success/failure — great for cleanup steps that must execute no matter what.

🚫 cancelled()

True if the workflow run was manually cancelled mid-execution.

⚠️ Matrix Jobs Multiply Your Minutes Usage

A 2×3 matrix = 6 full job runs, each consuming runner minutes against your usage quota. Be deliberate about matrix size — testing every permutation of every dependency version quickly becomes expensive on private repos.

📖 Keyword Reference Cheat Sheet

Every keyword used across this guide, in one place — so you can come back here instead of re-reading whole tabs once you already understand the concepts.

Top-Level Workflow Keywords

KeywordPurpose
name:Display name of the workflow, shown in the Actions tab
on:What event(s) trigger this workflow (push, pull_request, schedule, workflow_dispatch...)
permissions:Sets the GITHUB_TOKEN's permission scope for this workflow/job (e.g. id-token: write for OIDC)
env:Environment variables available to all jobs/steps below this level
jobs:Container for all jobs in the workflow
concurrency:Prevents overlapping runs — e.g. cancel an in-progress deploy if a newer one starts

Job-Level Keywords

KeywordPurpose
runs-on:Which runner/VM this job executes on
needs:List of job IDs that must complete successfully first — builds the dependency graph
if:Conditional expression — job only runs if this evaluates true
environment:Ties the job to a named Environment (approval gates, scoped secrets)
strategy.matrix:Generates multiple job runs from combinations of variables
outputs:Values this job exposes for other jobs to read via needs.JOB.outputs.X
steps:The ordered list of instructions this job executes
timeout-minutes:Kills the job if it runs longer than this — prevents stuck jobs burning minutes forever

Step-Level Keywords

KeywordPurpose
name:Display label for this step in logs/UI (optional but recommended)
id:Reference name so other steps can read this step's outputs
uses:Calls a pre-built Action — format owner/repo@version
run:Executes shell command(s) directly instead of an Action
with:Input parameters passed to an Action used via uses:
env:Environment variables scoped to just this step
if:Conditional — this individual step only runs if true
continue-on-error:Job keeps running even if this step fails (doesn't fail the whole job)
