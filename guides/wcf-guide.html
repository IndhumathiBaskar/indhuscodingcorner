🔧 What is WCF? (Simple Explanation)

WCF = Windows Communication Foundation
Framework to build .NET services that communicate across networks. Enable different apps to talk to each other!

🔗 Real Life Analogy

Without WCF:
Each application is isolated. Cannot communicate ❌
Desktop app, web app, mobile app all separate

With WCF:
Create a Service that all apps can use ✅
- Desktop app calls WCF service
- Web app calls same WCF service
- Mobile app calls same WCF service
- All use same data and logic

WCF = Universal communication bridge!

WCF Communication Flow

1️⃣

Client Calls Service

Client sends request to WCF service

2️⃣

Find Endpoint

WCF finds matching endpoint (Address + Binding + Contract)

3️⃣

Message Flows Through Binding

Message serialized, encrypted, transformed through binding layers

4️⃣

Call Service Method

WCF calls matching method on service implementation

5️⃣

Method Executes

Service method runs, processes data, returns result

6️⃣

Response Flows Back

Message serialized and sent back through binding

7️⃣

Client Receives Response

Client gets result and continues execution

🎯 WCF Core Concepts (The ABC's)

WCF is built on 3 fundamental concepts: Address, Binding, Contract (ABC)

A = Address (Where?)

📍 Address - Where is the service?

What: URI that clients use to find the service
Format: protocol://host:port/path
Examples:
- http://localhost:8080/MyService
- tcp://server:9000/Service
- net.pipe://localhost/NamedPipeService

Parts:
protocol = How to connect (http, tcp, named pipes)
host = Where service runs (localhost, server name)
port = Communication port number
path = Service identifier

B = Binding (How?)

🔗 Binding - How to communicate?

What: How messages are sent (protocol, encoding, security)
Main Bindings:
- BasicHttpBinding: Simple HTTP/SOAP
- WSHttpBinding: Secure HTTP/SOAP
- NetTcpBinding: Fast binary over TCP
- NetNamedPipeBinding: Local machine communication
- WebHttpBinding: REST/JSON

Binding Configures:
- Transport (HTTP, TCP, Named Pipes)
- Encoding (Text, Binary, MTOM)
- Security (None, Transport, Message)
- Message format (SOAP, JSON, XML)

C = Contract (What?)

📋 Contract - What methods are available?

What: Interface defining service methods
Types:
1. Service Contract: Methods client can call
2. Data Contract: Data structures exchanged
3. Message Contract: Custom message format
4. Fault Contract: Errors that can occur

Example:
[ServiceContract]
public interface ICustomerService
{
  [OperationContract]
  Customer GetCustomer(int id);
}

📋 Service Contract (Define Your Service Methods)

ServiceContract tells clients what methods they can call

Simple Service Contract Example

using System.ServiceModel;

// Step 1: Define Service Contract (what methods available)
[ServiceContract]
public interface ICalculatorService
{
    // Step 2: Mark methods as operations
    [OperationContract]
    int Add(int a, int b);
    
    [OperationContract]
    int Subtract(int a, int b);
    
    [OperationContract]
    int Multiply(int a, int b);
}

// Step 3: Implement the service
public class CalculatorService : ICalculatorService
{
    public int Add(int a, int b)
    {
        return a + b;
    }
    
    public int Subtract(int a, int b)
    {
        return a - b;
    }
    
    public int Multiply(int a, int b)
    {
        return a * b;
    }
}

What Do These Attributes Do?

Attribute What It Does Example
[ServiceContract] Mark interface as service contract [ServiceContract] public interface IMyService
[OperationContract] Mark method as callable operation [OperationContract] int Add(int a, int b);
[DataContract] Mark class as exchangeable data [DataContract] public class Customer
[DataMember] Mark property as data member [DataMember] public string Name;
[FaultContract] Define possible errors [FaultContract(typeof(Exception))]

📦 Data Contract (Define Your Data)

DataContract defines data structures exchanged between client and service

Complete Data Contract Example

using System.ServiceModel;
using System;

// Define data that will be exchanged
[DataContract]
public class Customer
{
    [DataMember]
    public int Id { get; set; }
    
    [DataMember]
    public string Name { get; set; }
    
    [DataMember]
    public string Email { get; set; }
    
    [DataMember]
    public decimal Balance { get; set; }
}

// Service that returns Customer data
[ServiceContract]
public interface ICustomerService
{
    [OperationContract]
    Customer GetCustomer(int id);
    
    [OperationContract]
    void UpdateCustomer(Customer customer);
}

// Implement the service
public class CustomerService : ICustomerService
{
    public Customer GetCustomer(int id)
    {
        // Get from database
        return new Customer
        {
            Id = id,
            Name = "John Smith",
            Email = "[email protected]",
            Balance = 1000m
        };
    }
    
    public void UpdateCustomer(Customer customer)
    {
        // Save to database
        System.Console.WriteLine("Customer updated: " + customer.Name);
    }
}

🔗 Bindings (Different Ways to Communicate)

Bindings determine HOW your service communicates

Main Bindings & When to Use

Binding Name Protocol Best For Security Performance
BasicHttpBinding HTTP/SOAP Web services, legacy systems Low Medium
WSHttpBinding HTTP/SOAP Secure web services High Medium
NetTcpBinding TCP/Binary Intranet apps, speed matters High Very High
NetNamedPipeBinding Named Pipes Local machine only High Highest
WebHttpBinding HTTP/REST/JSON Web APIs, mobile apps Medium Medium
NetMsmqBinding MSMQ/Queue Async, reliable messaging High Medium

Binding Configuration Example

// In App.config or Web.config
<configuration>
  <system.serviceModel>
    <services>
      <service name="CalculatorService">
        
        <!-- Endpoint 1: HTTP SOAP -->
        <endpoint address="http://localhost:8080/Calculator"
                  binding="basicHttpBinding"
                  contract="ICalculatorService" />
        
        <!-- Endpoint 2: TCP Binary -->
        <endpoint address="net.tcp://localhost:9000/Calculator"
                  binding="netTcpBinding"
                  contract="ICalculatorService" />
        
        <!-- Endpoint 3: REST JSON -->
        <endpoint address="http://localhost:8080/CalculatorRest"
                  binding="webHttpBinding"
                  contract="ICalculatorService" />
        
      </service>
    </services>
  </system.serviceModel>
</configuration>

🏠 Hosting WCF Service (Where It Runs)

Hosting Options

🖥️ Console App (Self-Hosted)

Easy: Simple to setup
Control: Full control
Use: Dev/testing
Deployment: Run exe file

🌐 IIS (Internet Information Server)

What: Host in IIS web server
Advantage: Professional hosting
Management: IIS handles lifecycle
Use: Production
Deployment: Copy to IIS folder

⚙️ Windows Service

What: Run as Windows service
Advantage: Auto-start on reboot
Management: Services.msc controls it
Use: Production backend
Deployment: Install via SC command

🔵 Windows Process Activation Service

What: Managed by Windows
Advantage: Automatic scaling
Management: Auto-start on demand
Use: Enterprise
Deployment: Via IIS

🌍 Complete WCF Example (Order Service)

Step 1: Define Contracts

using System.ServiceModel;

// Step 1: Data Contract - What data
[DataContract]
public class Order
{
    [DataMember]
    public int OrderId { get; set; }
    
    [DataMember]
    public string CustomerName { get; set; }
    
    [DataMember]
    public decimal Total { get; set; }
}

// Step 2: Service Contract - What methods
[ServiceContract]
public interface IOrderService
{
    [OperationContract]
    Order GetOrder(int orderId);
    
    [OperationContract]
    bool CreateOrder(Order order);
    
    [OperationContract]
    bool CancelOrder(int orderId);
}

Step 2: Implement Service

// Step 3: Implement the service
public class OrderService : IOrderService
{
    // Sample data (in real app, use database)
    private static List<Order> orders = new List<Order>();
    
    public Order GetOrder(int orderId)
    {
        var order = orders.FirstOrDefault(o => o.OrderId == orderId);
        if (order == null)
            throw new FaultException("Order not found");
        return order;
    }
    
    public bool CreateOrder(Order order)
    {
        try
        {
            order.OrderId = orders.Count + 1;
            orders.Add(order);
            System.Console.WriteLine("Order created: " + order.OrderId);
            return true;
        }
        catch (Exception ex)
        {
            System.Console.WriteLine("Error: " + ex.Message);
            return false;
        }
    }
    
    public bool CancelOrder(int orderId)
    {
        var order = orders.FirstOrDefault(o => o.OrderId == orderId);
        if (order == null)
            return false;
        
        orders.Remove(order);
        System.Console.WriteLine("Order cancelled: " + orderId);
        return true;
    }
}

Step 3: Host the Service

// Step 4: Self-host in console app
class Program
{
    static void Main()
    {
        // Create service host
        var serviceHost = new ServiceHost(
            typeof(OrderService),
            new Uri("http://localhost:8080/OrderService")
        );
        
        // Add endpoint with HTTP binding
        serviceHost.AddServiceEndpoint(
            typeof(IOrderService),
            new BasicHttpBinding(),
            "http://localhost:8080/OrderService"
        );
        
        // Start the service
        serviceHost.Open();
        System.Console.WriteLine("Order Service is running...");
        System.Console.WriteLine("Address: http://localhost:8080/OrderService");
        System.Console.WriteLine("Press Enter to stop...");
        System.Console.ReadLine();
        
        serviceHost.Close();
    }
}

Step 4: Call from Client

// Client calls the service
class Program
{
    static void Main()
    {
        // Create proxy to service
        var proxy = new ChannelFactory<IOrderService>(
            new BasicHttpBinding(),
            new EndpointAddress("http://localhost:8080/OrderService")
        ).CreateChannel();
        
        // Create order
        var newOrder = new Order 
        { 
            CustomerName = "John Smith", 
            Total = 99.99m 
        };
        bool created = proxy.CreateOrder(newOrder);
        System.Console.WriteLine("Order created: " + created);
        
        // Get order
        var order = proxy.GetOrder(1);
        System.Console.WriteLine("Order: " + order.OrderId + " - " + order.Total);
    }
}

🔐 WCF Security (Protecting Your Service)

Security Modes

Security Mode Encryption Authentication Best For
None ❌ No ❌ No Development/Testing
Transport ✅ Yes (SSL/TLS) ✅ Basic Auth HTTP (HTTPS)
Message ✅ Yes (Encrypted) ✅ Token-based Complex scenarios
TransportWithMessageCredential ✅ Both ✅ Both High security

Security Configuration Example

// In Web.config - Enable HTTPS Security
<configuration>
  <system.serviceModel>
    <bindings>
      <basicHttpBinding>
        <binding name="SecureBinding">
          <security mode="Transport">
            <transport clientCredentialType="Basic" />
          </security>
        </binding>
      </basicHttpBinding>
    </bindings>
    
    <services>
      <service name="MyService">
        <endpoint address="https://localhost/MyService"
                  binding="basicHttpBinding"
                  bindingConfiguration="SecureBinding"
                  contract="IMyService" />
      </service>
    </services>
  </system.serviceModel>
</configuration>

✅ WCF Best Practices

Design Practices

  • Keep contracts simple and focused
  • Use versioning for service evolution
  • Avoid circular references in data contracts
  • Use appropriate binding for your scenario
  • Always implement proper error handling (FaultContract)
  • Use OneWay operation for fire-and-forget calls
  • Implement proper resource cleanup (IDisposable)
  • Use asynchronous operations for long-running tasks

    • Security Practices

  • Never use security mode "None" in production
  • Always use HTTPS for internet services
  • Implement proper authentication (Windows, certificates, tokens)
  • Use authorization to control access
  • Enable auditing for sensitive operations
  • Validate all input data
  • Use secure credentials management
  • Implement throttling to prevent abuse

    • Performance Practices

  • Use NetTcpBinding for intranet apps (faster than HTTP)
  • Implement caching where appropriate
  • Use streaming for large files
  • Configure appropriate timeouts
  • Monitor service performance regularly
  • Use DataContract (serialization) efficiently
  • Implement connection pooling
  • Use asynchronous calls to avoid blocking

    • ⚠️ Common WCF Mistakes

    ❌ Mistake 1: Missing [DataContract] or [DataMember]

    Client won't receive data! Always mark classes and properties that need to be serialized.

    ❌ Mistake 2: Circular References in Data Contracts

    Customer → Order → Customer causes serialization errors. Flatten your data structures.

    ❌ Mistake 3: Using Security Mode "None" in Production

    Anyone can intercept your data! Always use Transport or Message security.

    ❌ Mistake 4: Not Implementing FaultContract

    Clients don't know how to handle errors. Define expected exceptions with FaultContract.

    ❌ Mistake 5: Wrong Binding Choice

    Using BasicHttpBinding for internal app? Use NetTcpBinding for better performance!

    ❌ Mistake 6: No Timeout Configuration

    Default timeout is 1 minute. Long operations fail! Configure appropriate timeouts.

    ❌ Mistake 7: Resource Leaks (Not Disposing)

    Service consumes resources. Implement IDisposable and dispose connections properly.

    ⚖️ WCF vs Web API (Which to Use?)

    Aspect WCF Web API
    Framework .NET Framework ASP.NET Core (newer)
    Protocols HTTP, TCP, Named Pipes, MSMQ HTTP/HTTPS only
    Best For Enterprise, SOAP, multiple transports REST APIs, web/mobile apps
    Learning Curve Steep (many options) Easy (simplified)
    Performance High (binary TCP) Very High (optimized)
    Modern? Legacy Current standard
    Use Today? Legacy systems only ✅ Yes! (recommended)

    Recommendation

    💡 When to Use WCF

    - Maintaining legacy systems
    - Need multiple transport protocols (TCP, Named Pipes)
    - Enterprise SOAP services
    - Need complex security scenarios

    💡 When to Use Web API

    - Building new services (recommended!)
    - REST APIs
    - Web and mobile applications
    - Cloud services (Azure)
    - You should prefer this!
