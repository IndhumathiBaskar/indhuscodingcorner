☸️ What is Amazon EKS?

EKS = Elastic Kubernetes Service. A fully managed Kubernetes control plane that AWS runs, patches, and scales for you. You bring your workloads as Pods; EKS handles the Kubernetes API server, etcd, scheduler, and controller manager — the complex brains of a cluster — so you only manage your worker nodes and applications.

💡 Real-life analogy — The Airport

EKS is like a major airport. The Control Plane is Air Traffic Control — it knows where every plane (Pod) is, decides which runway (Node) each lands on, and reroutes instantly when a runway closes. Nodes (EC2 workers) are the runways and terminals — the actual physical capacity where planes land and operate. Pods are the planes — your containerized applications. Services are the boarding gates — stable entry points that passengers (traffic) reach regardless of which terminal their plane is at. AWS is the airport authority — maintaining the control tower, runways, and security so airlines (your team) focus on their flights, not infrastructure.

What AWS Manages vs What You Manage

✅ AWS manages (Control Plane)

  • Kubernetes API server — highly available across 3 AZs
  • etcd — the distributed key-value store for cluster state
  • Kubernetes scheduler — decides which node runs each pod
  • Controller manager — handles replication, endpoints, etc.
  • Control plane patching and upgrades (managed rollout)
  • Control plane scaling — handles thousands of nodes
  • Certificate rotation for cluster communication

🛠️ You manage (Data Plane)

  • Worker nodes — EC2 instances running your Pods
  • Node group upgrades (you trigger, EKS executes)
  • Pod deployment manifests (Deployments, Services, etc.)
  • Namespaces and workload isolation
  • Add-ons: CoreDNS, kube-proxy, VPC CNI, metrics-server
  • Application-level monitoring and logging
  • Horizontal and vertical pod scaling rules

How kubectl Reaches the Cluster

💻 Developer runs: kubectl get pods
  • kubeconfig references EKS cluster endpoint
  • AWS credential chain invoked (IAM role/profile)
  • aws eks get-token generates a bearer token from IAM
↓ HTTPS request + IAM bearer token
⚙️ EKS API Server authenticates request
  • Validates IAM identity via AWS STS token
  • Looks up IAM principal in aws-auth ConfigMap
  • Maps IAM role → Kubernetes user/group
↓ RBAC authorization check
✅ Response returned to kubectl
  • Pod list, deployment status, logs — whatever was requested
  • If RBAC denies → 403 Forbidden returned immediately

💡 Two endpoint modes — know which you need

Public endpoint (default): API server is accessible from the internet. Secured by IAM + RBAC. Fine for most teams. Private endpoint only: API server only reachable from within your VPC — requires VPN or AWS VPN/Direct Connect for developer access. Use private-only for high-security environments like banking or healthcare. You can also enable both — internet access plus in-VPC access.

Managed Node Group — How Upgrades Work

📋 You trigger: Update node group version
  • Console / CLI / Terraform — select new K8s version
  • AWS provisions new nodes with the updated AMI
↓ rolling update begins
⚙️ EKS cordon + drain old nodes
  • Marks old nodes as unschedulable (cordon)
  • Safely evicts existing pods (drain) — respects PodDisruptionBudgets
  • New nodes register with control plane, pods rescheduled
↓ old nodes terminated
✅ Upgrade complete — zero downtime
  • All nodes on new version — cluster fully healthy
  • PodDisruptionBudget ensures minimum replicas always running

Spot Instances — Save 70–90% on Node Costs

✅ Good workloads for Spot nodes

  • Batch processing jobs (ML training, ETL pipelines)
  • CI/CD build runners — interruption-tolerant by design
  • Stateless microservices with multiple replicas
  • Dev and staging environments where interruption is OK
  • Data processing with checkpointing built in

❌ Don't use Spot for

  • Stateful workloads (databases, caches) without persistent volumes
  • Single-replica production services — one interruption = outage
  • Long-running sessions requiring connection stability
  • Anything without graceful shutdown handling (SIGTERM)
  • Node where PodDisruptionBudget would block drain

💡 Mix On-Demand + Spot with node selectors

Create two node groups: one On-Demand (labeled node-type=stable) for critical services, one Spot (labeled node-type=spot) for batch/dev work. Use nodeSelector or nodeAffinity in your Pod specs to pin workloads to the right group. This gives you cost savings without risking production availability.

🌐 Networking — Services, Ingress & Load Balancers

VPC CNI — Why Every Pod Gets a VPC IP

⚠️ Standard Kubernetes (overlay network)
  • Pods get IPs from a separate overlay network (10.244.0.0/16)
  • Traffic must NAT to reach VPC resources
  • Complex routing — hard to debug with VPC flow logs
  • Security groups cannot be applied per-pod
↓ EKS uses VPC CNI instead
✅ EKS VPC CNI — pods are first-class VPC citizens
  • Every Pod gets a real VPC IP from your subnet's CIDR
  • No NAT — direct routing between pods, RDS, ElastiCache, etc.
  • VPC Flow Logs capture pod-level traffic by IP
  • Security groups can be applied per-pod (Security Groups for Pods)

⚠️ Plan your IP address space carefully

With VPC CNI, each running Pod consumes a VPC IP address. A node running 30 pods uses 30 IPs from your subnet. A cluster with 100 nodes × 30 pods = 3,000 IPs. A /24 subnet only has 256 IPs. Always use at least /22 subnets for pod subnets, or use prefix delegation (assigns a /28 block per ENI, up to 16 IPs per slot). Plan this before creating the cluster — you cannot change the VPC CIDR later.

Network Policies — Zero-Trust Between Pods

❌ Without Network Policies (default)

  • Any pod can reach any other pod in the cluster
  • A compromised frontend pod could talk to your database pod
  • No traffic isolation between namespaces
  • Blast radius of a breach is the entire cluster

✅ With Network Policies (recommended)

  • Whitelist-only model — deny all by default, allow explicitly
  • Frontend pods can only reach backend pods on port 8080
  • Backend pods can only reach RDS on port 5432
  • Namespaces are isolated — dev cannot reach production

📈 Scaling — Pods, Nodes & Karpenter

EKS scaling works at two levels: Pod-level (HPA/VPA — more replicas of your app) and Node-level (Cluster Autoscaler or Karpenter — more EC2 instances). Both levels must work together: if you scale pods but have no free node capacity, pods stay pending.

HPA — Scaling Pods on CPU

YAML — HPA targeting 60% CPU, 2–10 replicas
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: my-api-hpa
  namespace: production
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: my-api
  minReplicas: 2
  maxReplicas: 10
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 60  # scale up when avg CPU hits 60%

Scaling Chain — From Request Spike to New Node

📈 Traffic spike — CPU rises to 85%
  • HPA polls metrics-server every 15 seconds
  • CPU > 60% target → HPA calculates needed replicas
  • HPA scales Deployment from 2 → 5 replicas
↓ pods scheduled — no free capacity
⏳ 3 new pods stuck in Pending
  • Scheduler cannot fit pods on existing nodes
  • Karpenter / Cluster Autoscaler detects pending pods
  • Provisions new EC2 node (Karpenter ~30s, CA ~2–3min)
↓ node joins cluster
✅ Pods scheduled — cluster healthy
  • New node registers with control plane
  • Pending pods scheduled onto the new node
  • CPU drops below 60% — HPA holds at 5 replicas
  • After traffic drops → HPA scales down, CA removes idle node

💡 Karpenter vs Cluster Autoscaler — prefer Karpenter for new clusters

Cluster Autoscaler scales existing Node Groups — it can only use instance types pre-configured in that group. Karpenter (AWS-developed) provisions nodes directly — it reads your pod requirements and picks the optimal instance type, size, and spot/on-demand mix in real time. Karpenter typically provisions nodes 5–10× faster than Cluster Autoscaler and achieves better bin-packing (fewer idle nodes).

🔐 IAM & IRSA — Secure AWS Access from Pods

Pods sometimes need to call AWS APIs — read from S3, write to DynamoDB, publish to SQS. The wrong approach is embedding credentials. The right approach is IRSA (IAM Roles for Service Accounts) — pods get temporary AWS credentials via their Kubernetes Service Account, scoped to exactly the IAM role they need.

The Problem IRSA Solves

❌ Old approach — Node-level IAM role

  • All pods on a node share the same EC2 instance role
  • A compromised payment pod can access the same S3 buckets as the logging pod
  • Cannot scope permissions per-service
  • If one pod is breached — all AWS access is exposed
  • No audit trail of which pod called which API

✅ IRSA — per-pod IAM role binding

  • Each Kubernetes Service Account maps to one IAM Role
  • Temporary credentials injected as env vars into pod
  • Payment pod → only S3 access for receipts
  • Logging pod → only CloudWatch Logs write access
  • Credentials auto-rotated every 15 minutes

How IRSA Works — Step by Step

⚙️ Setup: Create OIDC provider + IAM Role
  • EKS cluster has an OIDC issuer URL (auto-created)
  • Register this as an Identity Provider in IAM
  • Create IAM role with trust policy scoped to the service account
  • Annotate the Kubernetes Service Account with the role ARN
↓ pod starts
🔑 Pod token injected at runtime
  • EKS mutating webhook injects a projected volume with a service account token
  • Token is short-lived (15 min TTL) and audience-restricted
  • AWS SDK automatically finds the token via env var paths
↓ pod calls AWS API
✅ STS AssumeRoleWithWebIdentity
  • SDK exchanges the Kubernetes token for STS temporary credentials
  • Credentials scoped to exactly the IAM Role you annotated
  • Auto-refreshed — your application code needs zero changes

IRSA Setup — Commands

bash — IRSA setup with eksctl
# 1. Create OIDC provider for the cluster (one-time per cluster)
eksctl utils associate-iam-oidc-provider \
  --cluster my-cluster \
  --region ap-south-1 \
  --approve

# 2. Create service account + IAM role in one step
eksctl create iamserviceaccount \
  --name my-api-sa \
  --namespace production \
  --cluster my-cluster \
  --region ap-south-1 \
  --attach-policy-arn arn:aws:iam::aws:policy/AmazonS3ReadOnlyAccess \
  --approve

# 3. Annotate service account (if creating manually)
kubectl annotate serviceaccount my-api-sa \
  -n production \
  eks.amazonaws.com/role-arn=arn:aws:iam::123456789012:role/my-api-role

aws-auth ConfigMap — Granting kubectl Access

Before IRSA, there's a separate question: who can run kubectl against the cluster? The aws-auth ConfigMap maps IAM roles/users to Kubernetes RBAC groups.

YAML — aws-auth ConfigMap (kube-system namespace)
apiVersion: v1
kind: ConfigMap
metadata:
  name: aws-auth
  namespace: kube-system
data:
  mapRoles: |
    - rolearn: arn:aws:iam::123456789012:role/EKSNodeGroupRole
      username: system:node:{{EC2PrivateDNSName}}
      groups:
        - system:bootstrappers
        - system:nodes
    - rolearn: arn:aws:iam::123456789012:role/DevOpsTeamRole
      username: devops
      groups:
        - system:masters   # cluster-admin — use sparingly!

⚠️ EKS Access Entries — replace aws-auth with the newer approach

The aws-auth ConfigMap is error-prone — a YAML typo can lock everyone out of the cluster, including you. AWS released EKS Access Entries (2024) as the newer approach: manage cluster access via IAM-native API calls (aws eks create-access-entry), not kubectl edits to a ConfigMap. Use Access Entries for all new clusters.

💻 C# SDK — Working with EKS Programmatically

The AWS SDK for .NET lets you create and manage EKS clusters, node groups, Fargate profiles, and add-ons from C#. For deploying workloads, use the KubernetesClient NuGet package. Install: AWSSDK.EKS and KubernetesClient.

Setup — EKS Client & Describe Cluster

C# — describe cluster and list node groups
using Amazon.EKS;
using Amazon.EKS.Model;

public class EksService
{
    private readonly IAmazonEKS _eks;

    public EksService(IAmazonEKS eks) => _eks = eks;

    // ── Describe a cluster ─────────────────────────────────────
    public async Task<Cluster> DescribeClusterAsync(string clusterName)
    {
        var resp = await _eks.DescribeClusterAsync(
            new DescribeClusterRequest { Name = clusterName });

        var cluster = resp.Cluster;
        Console.WriteLine($"Cluster: {cluster.Name}");
        Console.WriteLine($"Status: {cluster.Status}");
        Console.WriteLine($"K8s version: {cluster.Version}");
        Console.WriteLine($"Endpoint: {cluster.Endpoint}");

        return cluster;
    }

    // ── List all node groups ────────────────────────────────────
    public async Task<List<string>> ListNodeGroupsAsync(string clusterName)
    {
        var resp = await _eks.ListNodegroupsAsync(
            new ListNodegroupsRequest { ClusterName = clusterName });
        return resp.Nodegroups;
    }

    // ── Describe a node group (instance types, status, scaling)──
    public async Task<Nodegroup> DescribeNodeGroupAsync(
        string clusterName, string nodeGroupName)
    {
        var resp = await _eks.DescribeNodegroupAsync(
            new DescribeNodegroupRequest
        {
            ClusterName   = clusterName,
            NodegroupName = nodeGroupName
        });
        return resp.Nodegroup;
    }
}

            }
        });
        return resp.Nodegroup;
    }

Deploy a Workload via KubernetesClient

C# — deploy a Deployment + Service using KubernetesClient
using k8s;
using k8s.Models;

public class K8sDeploymentService
{
    private readonly IKubernetes _k8s;

    public K8sDeploymentService(IKubernetes k8s) => _k8s = k8s;

    // ── Create or update a Deployment ─────────────────────────
    public async Task DeployAsync(string ns, string name,
        string image, int replicas)
    {
        var deployment = new V1Deployment
        {
            Metadata = new V1ObjectMeta
            {
                Name              = name,
                NamespaceProperty = ns,
                Labels            = new Dictionary<string, string>
                                    { ["app"] = name }
            },
            Spec = new V1DeploymentSpec
            {
                Replicas = replicas,
                Selector = new V1LabelSelector
                {
                    MatchLabels = new Dictionary<string, string>
                                  { ["app"] = name }
                },
                Template = new V1PodTemplateSpec
                {
                    Metadata = new V1ObjectMeta
                    {
                        Labels = new Dictionary<string, string>
                                 { ["app"] = name }
                    },
                    Spec = new V1PodSpec
                    {
                        Containers = [new V1Container
                        {
                            Name  = name,
                            Image = image,
                            Ports = [new V1ContainerPort
                                     { ContainerPort = 8080 }],
                            Resources = new V1ResourceRequirements
                            {
                                Requests = new Dictionary<string,
                                    ResourceQuantity>
                                {
                                    ["cpu"]    = new("250m"),
                                    ["memory"] = new("256Mi")
                                },
                                Limits = new Dictionary<string,
                                    ResourceQuantity>
                                {
                                    ["cpu"]    = new("500m"),
                                    ["memory"] = new("512Mi")
                                }
                            }
                        }]
                    }
                }
            }
        };

        await _k8s.CreateNamespacedDeploymentAsync(deployment, ns);
    }
}

✅ Best Practices & Common Mistakes

Cluster & Node Configuration

  • Always spread node groups across at least 2 AZs — one AZ outage should not take down your cluster
  • Use Managed Node Groups over self-managed — AWS handles draining, replacing, and patching
  • Set Minimum: 2 nodes — ensures at least one node per AZ for HA even when Cluster Autoscaler scales down
  • Use Graviton (m7g, c7g) instances — ~20% better price/performance for most workloads vs x86
  • Never use t-class (t3, t4g) for production nodes — CPU credits exhaust under sustained load causing throttling
  • Keep control plane and node group Kubernetes versions within 1 minor version of each other
  • Enable control plane logging (API, Audit, Authenticator) from day one — essential for debugging auth issues

Security

  • Use IRSA for every pod that needs AWS API access — never rely on the node-level EC2 instance role
  • Use EKS Access Entries (not aws-auth ConfigMap) for managing kubectl access on new clusters
  • Enable secrets encryption with KMS at cluster creation — it cannot be enabled after
  • Apply NetworkPolicies in all namespaces — default-deny-all + explicit allows per service
  • Run non-root containers — add runAsNonRoot: true and readOnlyRootFilesystem: true to all pod specs
  • Enable private API endpoint — restrict public endpoint to office/VPN IP ranges if possible
  • Scan all container images in ECR before deploying — block on CRITICAL CVEs in your CI/CD pipeline

Reliability & Cost

  • Always set CPU and memory requests and limits — without them HPA cannot scale and pods may be evicted unexpectedly
  • Set a PodDisruptionBudget for every critical service — prevents drain from removing too many pods at once
  • Use Karpenter over Cluster Autoscaler for new clusters — faster node provisioning, better instance diversity
  • Use Spot instances for stateless, fault-tolerant workloads — 70–90% node cost savings
  • Set minReplicas: 2 in HPA for all production services — a single replica is a single point of failure
  • Use Fargate for batch/ephemeral workloads to avoid idle node costs
  • Tag all EKS resources — node groups, load balancers, volumes — with environment/team/cost-center for billing visibility

Common Mistakes to Avoid

⚠️ Top EKS Mistakes

❌ No resource requests on pods — HPA cannot scale, scheduler over-packs nodes, pods get OOMKilled unexpectedly
❌ Using aws-auth ConfigMap for access on new clusters — one YAML typo locks everyone out including the cluster creator; use EKS Access Entries instead
❌ Single-AZ node group — one AZ outage kills all capacity; always span at least 2 AZs
❌ Node-level IAM role for S3/DynamoDB access — all pods on the node get access; a breach exposes all AWS resources; use IRSA
❌ Not setting PodDisruptionBudgets — node drain during upgrade can take ALL replicas of a service offline simultaneously
❌ Using t3/t4g instances for production — CPU credit exhaustion causes sudden throttling exactly when you have traffic spikes
❌ Forgetting subnet IP space — VPC CNI gives every pod a VPC IP; a /24 subnet runs out fast at scale; use /22 minimum
❌ Skipping metrics-server — HPA silently fails without it; pods never scale even under 100% CPU load
