🤔 Why Do We Need Async Programming?

👶 Start Here — The Problem Async Solves

Imagine you are at a coffee shop. You order a coffee. The barista could stare at the machine doing nothing while it brews (synchronous). Or the barista could take the next customer's order while your coffee brews (asynchronous). Async programming lets your program keep working while waiting for slow operations — database queries, API calls, file reads — instead of freezing and doing nothing.

Synchronous vs Asynchronous — The Core Difference

⏱️ Sync vs Async — Timeline Comparison

The Three Slow Operations That Need Async

🌐

Network / API calls

HTTP requests, REST APIs, web services. Can take 10ms to 10 seconds. Thread should not wait.

🗄️

Database queries

SQL queries, ORMs, NoSQL. Can take milliseconds to minutes. Classic async candidate.

💾

File / disk I/O

Reading files, writing logs, accessing network drives. Disk speed is much slower than CPU.

What Happens Without Async

❌ Synchronous Problems
  • UI apps freeze — window becomes unresponsive while waiting
  • Web servers run out of threads — under load, new requests queue up
  • Thread is 100% blocked — wasting memory (1MB per thread stack)
  • Scalability ceiling hit quickly — maybe 100 concurrent users
✅ Async Benefits
  • UI stays responsive — progress bars, cancel buttons work
  • Web server handles thousands of concurrent requests on few threads
  • Thread returns to pool during I/O — serves other requests
  • Massively better scalability — same hardware serves more users

Async Is NOT About Speed — It's About Efficiency

💡 Critical Misconception to Avoid

Async does NOT make a single database query run faster. If a query takes 2 seconds, it still takes 2 seconds. What async does is free the thread during those 2 seconds to do other work. On a web server handling 1000 simultaneous users, this is the difference between needing 1000 threads (crashing) and needing 10 threads (fine). Async is about resource efficiency and scalability, not raw speed.

📦 Task and Task<T>

👶 What is a Task?

A Task is an object that represents an ongoing operation. Think of it like a receipt you get when you order food — the food isn't ready yet, but you have a receipt that represents "your food will be ready eventually". You can check on it, wait for it, or cancel it. In C#, when a method does async work it returns a Task instead of making you wait.

Task vs Task<T> — The Difference

📋 Task — operation with no return value

  • Like void but for async operations
  • Use when the operation just does something (save, send email, delete)
  • Example: Task SaveAsync()
  • Completes, fails, or gets cancelled — no result value

📦 Task<T> — operation that returns a value

  • Like a T return type but for async operations
  • Use when you need a result (query returns data, API returns JSON)
  • Example: Task<User> GetUserAsync(int id)
  • When awaited, you get back the T value

Task States

⏳ Created / WaitingToRun
  • Task has been created but not started yet
🔄 Running / WaitingForActivation
  • Operation is in progress, I/O is happening
↓ completes in one of three ways
✅ RanToCompletion
  • Finished successfully
  • Result available (if Task<T>)
❌ Faulted
  • An exception was thrown
  • Exception stored in Task.Exception
🚫 Cancelled
  • CancellationToken was cancelled
  • OperationCanceledException thrown

Creating Tasks

C# — Ways to create Tasks
// ── Method 1: async method (most common) ─────────────────────────
// Just mark the method async — it automatically returns a Task
public async Task SaveDataAsync()
{
    await Task.Delay(1000);  // simulates slow work
}

public async Task<string> GetNameAsync()
{
    await Task.Delay(500);
    return "Alice";  // Task<string> automatically wraps this
}

// ── Method 2: Task.Run — offload CPU work to background thread ─────
public Task<int> HeavyComputeAsync()
    => Task.Run(() =>
    {
        // Runs on a ThreadPool thread — doesn't block caller
        int sum = 0;
        for (int i = 0; i < 1_000_000; i++) sum += i;
        return sum;
    });

// ── Method 3: Task.FromResult — return a completed Task immediately ─
// Use when you have the result already (cache hit, test stub)
public Task<string> GetCachedNameAsync()
    => Task.FromResult("Bob");  // instantly completed Task

// ── Method 4: Task.CompletedTask — return a completed void Task ────
public Task DoNothingAsync()
    => Task.CompletedTask;  // use in interface implementations

// ── Method 5: TaskCompletionSource — manual control ───────────────
// Use when wrapping non-async APIs (events, callbacks)
public Task<string> WaitForUserInputAsync()
{
    var tcs = new TaskCompletionSource<string>();
    // Somewhere else: tcs.SetResult("user typed this");
    // Or on error:    tcs.SetException(new Exception("..."));
    return tcs.Task;
}

Task Utility Methods

C# — Essential Task methods
// ── Task.Delay — async version of Thread.Sleep ────────────────────
await Task.Delay(2000);                        // wait 2 seconds, don't block thread
await Task.Delay(TimeSpan.FromSeconds(2));   // same, more readable

// ── Task.WhenAll — wait for MULTIPLE tasks to ALL finish ──────────
var task1 = GetUserAsync(1);
var task2 = GetUserAsync(2);
var task3 = GetUserAsync(3);

// All 3 run concurrently — total time = slowest single task
User[] users = await Task.WhenAll(task1, task2, task3);

// ── Task.WhenAny — wait until FIRST task finishes ─────────────────
var winner = await Task.WhenAny(task1, task2, task3);
Console.WriteLine($"First to finish: {winner.Result.Name}");

// ── Timeout pattern using WhenAny ────────────────────────────────
var operationTask = LongRunningOperationAsync();
var timeoutTask   = Task.Delay(TimeSpan.FromSeconds(5));

var completed = await Task.WhenAny(operationTask, timeoutTask);

if (completed == timeoutTask)
    throw new TimeoutException("Operation took too long!");

var result = await operationTask;  // safe to await — already completed

// ── Task.Run — move CPU work to background thread ─────────────────
var result2 = await Task.Run(() =>
{
    // Heavy CPU work — won't block UI thread
    return ProcessLargeDataSet(data);
});

Checking Task Status in Code

C# — Inspecting Task state
var task = GetDataAsync();

// Check if done (don't block — just check)
if (task.IsCompleted)         Console.WriteLine("Done (success, fault, or cancel)");
if (task.IsCompletedSuccessfully) Console.WriteLine("Finished successfully");
if (task.IsFaulted)           Console.WriteLine($"Error: {task.Exception}");
if (task.IsCanceled)          Console.WriteLine("Was cancelled");

// Get result synchronously (WARNING: can deadlock — see Tab 8)
// Only safe if you KNOW the task is already completed
if (task.IsCompletedSuccessfully)
{
    var result = task.Result;  // safe only when already done
}

⚡ async and await Keywords

👶 The Two Keywords That Do Everything

async marks a method as asynchronous. await says "pause here and wait for this task to complete, but don't block the thread — let it go do other work and come back when the task is done". They always work together — you can only use await inside an async method.

The Simplest Possible Example

C# — Your first async method
// BEFORE: synchronous — BLOCKS the thread for 2 seconds
public void SlowMethod()
{
    Thread.Sleep(2000);           // ❌ blocks thread completely
    Console.WriteLine("Done");
}

// AFTER: async — frees the thread during the 2 second wait
public async Task SlowMethodAsync()    // ← async keyword here
{
    await Task.Delay(2000);          // ← await keyword here, thread is FREE
    Console.WriteLine("Done");        // ← resumes here after 2 seconds
}

// With a return value:
public async Task<string> GetMessageAsync()
{
    await Task.Delay(500);        // simulate network call
    return "Hello from async!";  // C# wraps this in Task<string> automatically
}

// Calling it:
string message = await GetMessageAsync();  // await unwraps Task<string> to string
Console.WriteLine(message);  // "Hello from async!"

What await Actually Does — Step by Step

1

You call an async method

The method starts running normally, just like a synchronous method, until it hits an await.

2

await hits a Task that isn't completed yet

The C# compiler registers a "continuation" — a callback saying "when this task finishes, come back to line X". Then the method returns a Task to the caller.

3

The thread is released

The thread goes back to the thread pool. In a web server, it picks up another incoming request. In a WPF app, it goes back to processing UI events. Nothing is blocked.

4

I/O completes in the background

The operating system or network stack signals completion. The .NET runtime picks a thread from the pool to continue your method.

5

Method resumes from where it left off

Execution continues on the line after the await. All local variables are still there exactly as you left them. It feels like the method never paused.

Async Method Signatures

C# — All valid async method signatures
// ── Returns Task (no value, like async void but correct) ──────────
public async Task SaveAsync() { await DoSomethingAsync(); }

// ── Returns Task<T> (returns a value) ────────────────────────────
public async Task<string>  GetNameAsync()   { await Task.Delay(1); return "Alice"; }
public async Task<int>     GetCountAsync()  { await Task.Delay(1); return 42; }
public async Task<List<User>> GetUsersAsync() { await Task.Delay(1); return new(); }

// ── Returns ValueTask (performance optimisation — see Tab 7) ──────
public async ValueTask<int> GetCachedCountAsync()
{
    if (_cache != null) return _cache.Value;  // no allocation if cached
    return await FetchFromDbAsync();
}

// ── async void — ONLY for event handlers (exceptions are unhandled!) ─
private async void Button_Click(object sender, EventArgs e)
{
    await LoadDataAsync();  // acceptable here — event handler
}
// ⚠️ Never use async void except for event handlers

Exception Handling in Async

C# — try/catch works naturally with async/await
// ✅ Exceptions from async methods are caught normally with try/catch
public async Task LoadDataAsync()
{
    try
    {
        var data = await _repository.GetDataAsync();  // might throw
        var result = await _service.ProcessAsync(data); // might throw
        Console.WriteLine(result);
    }
    catch (HttpRequestException ex)
    {
        Console.WriteLine($"Network error: {ex.Message}");
    }
    catch (TimeoutException ex)
    {
        Console.WriteLine($"Timed out: {ex.Message}");
    }
    catch (Exception ex)
    {
        Console.WriteLine($"Unexpected error: {ex.Message}");
    }
    finally
    {
        // finally always runs — async or not
        Console.WriteLine("Cleanup done");
    }
}

// Handling Task.WhenAll exceptions — multiple exceptions can occur
public async Task RunMultipleAsync()
{
    var task1 = MightFailAsync();
    var task2 = MightFailAsync();

    try
    {
        await Task.WhenAll(task1, task2);
    }
    catch
    {
        // Check each task individually for its exception
        if (task1.IsFaulted) Console.WriteLine($"Task1 failed: {task1.Exception?.InnerException?.Message}");
        if (task2.IsFaulted) Console.WriteLine($"Task2 failed: {task2.Exception?.InnerException?.Message}");
    }
}

ConfigureAwait — What It Does

C# — ConfigureAwait explained
// By default, await tries to resume on the ORIGINAL context
// (UI thread for WPF/WinForms, request context for ASP.NET)

// In a WPF button click — resumes on UI thread (needed to update UI)
private async void Button_Click(object sender, EventArgs e)
{
    var data = await LoadDataAsync();     // resumes on UI thread
    lblResult.Text = data;                  // safe to update UI
}

// ConfigureAwait(false) — don't need to resume on original context
// Use in library code and services — slightly more efficient
public async Task<string> LibraryMethodAsync()
{
    var data = await FetchFromNetworkAsync()
        .ConfigureAwait(false);             // resumes on any thread pool thread
    return data.Trim();                     // no UI update here — fine
}

// Rule of thumb:
// - Application code (WPF, controller): use default (no ConfigureAwait)
// - Library/service code:               use ConfigureAwait(false)

🔀 Async Patterns — Sequential vs Concurrent

👶 The Biggest Performance Decision in Async

When you have multiple async operations to run, you have a choice: run them one after another (sequential) or all at the same time (concurrent). Sequential is simpler but slower. Concurrent is faster but slightly more complex. Knowing when to use each is crucial.

Sequential Async — One After Another

C# — Sequential: each waits for the previous to finish
// Sequential — total time = sum of all individual times
// GetUser = 1s, GetOrders = 1s, GetAddress = 1s → TOTAL = 3 seconds

public async Task LoadProfileSequentialAsync(int userId)
{
    var user    = await GetUserAsync(userId);       // wait 1s → then continue
    var orders  = await GetOrdersAsync(userId);     // wait 1s → then continue
    var address = await GetAddressAsync(userId);    // wait 1s → then continue

    Console.WriteLine($"{user.Name}, {orders.Count} orders, {address.City}");
    // Total: ~3 seconds
}

// USE sequential when:
// - Each step DEPENDS on the result of the previous step
// - Example: get user → use user.Id to get orders
public async Task LoadDependentDataAsync(string username)
{
    var user   = await GetUserByNameAsync(username);     // need user first
    var orders = await GetOrdersAsync(user.Id);           // need user.Id
    var items  = await GetItemsAsync(orders[0].OrderId);  // need order.Id
    // Sequential here is CORRECT — each step depends on the previous
}

Concurrent Async — All at the Same Time

FASTER C# — Concurrent: all run simultaneously
// Concurrent — total time = slowest individual task
// GetUser = 1s, GetOrders = 1s, GetAddress = 1s → TOTAL = 1 second !

public async Task LoadProfileConcurrentAsync(int userId)
{
    // Start ALL tasks first — none is awaited yet
    var userTask    = GetUserAsync(userId);    // starts immediately
    var ordersTask  = GetOrdersAsync(userId);  // starts immediately
    var addressTask = GetAddressAsync(userId); // starts immediately

    // Now wait for ALL to complete
    await Task.WhenAll(userTask, ordersTask, addressTask);

    // Access results — all complete by now
    var user    = userTask.Result;
    var orders  = ordersTask.Result;
    var address = addressTask.Result;

    Console.WriteLine($"{user.Name}, {orders.Count} orders, {address.City}");
    // Total: ~1 second (3× faster!)
}

// Even cleaner — await and get results in one step:
public async Task LoadConcurrentCleanAsync(int userId)
{
    (User user, List<Order> orders, Address address) =
        await Task.WhenAll(
            GetUserAsync(userId),
            GetOrdersAsync(userId),
            GetAddressAsync(userId)
        );  // Note: WhenAll with mixed types returns Task[] — use tuples for typed
}

// USE concurrent when:
// - Operations are INDEPENDENT of each other
// - Example: get user + get orders + get address in parallel

The Key Question — Sequential or Concurrent?

Use Sequential when…

  • Step B needs the result of Step A
  • Operations must happen in a specific order
  • You are looping through items and each iteration is independent but you want controlled throughput
  • Processing a pipeline where each stage feeds the next

Use Concurrent (WhenAll) when…

  • Operations are fully independent of each other
  • Fetching data from multiple sources at once
  • Making multiple API calls simultaneously
  • Loading related but independent data for a dashboard

Async Loops — Looping Over Collections

C# — Different ways to async loop
// ── Pattern 1: Sequential loop (process one at a time) ────────────
var users = new List<int> { 1, 2, 3, 4, 5 };

foreach (var userId in users)
{
    var user = await GetUserAsync(userId);  // one at a time
    Console.WriteLine(user.Name);
}
// Total time: sum of all individual calls

// ── Pattern 2: Concurrent loop (all at same time) ─────────────────
var tasks = users.Select(id => GetUserAsync(id));
var results = await Task.WhenAll(tasks);
// Total time: slowest single call — but ALL run at once
// ⚠️ Be careful: too many concurrent requests can overwhelm a DB or API

// ── Pattern 3: Throttled concurrent (controlled parallelism) ──────
// Use SemaphoreSlim to limit to N concurrent requests at a time
var semaphore = new SemaphoreSlim(3);  // max 3 at a time

var throttledTasks = users.Select(async id =>
{
    await semaphore.WaitAsync();          // wait for a slot
    try
    {
        return await GetUserAsync(id);   // do the work
    }
    finally
    {
        semaphore.Release();              // always release the slot
    }
});
var throttledResults = await Task.WhenAll(throttledTasks);
// Max 3 concurrent requests — doesn't overwhelm the server

🚫 Cancellation with CancellationToken

👶 Why Do We Need Cancellation?

Imagine a user clicks "Search", then immediately clicks "Cancel". Without cancellation, your app would keep querying the database and making API calls even though nobody wants the result anymore — wasting server resources. CancellationToken is the C# way to politely say "stop what you're doing". It is a signal that gets passed through your entire call chain.

💡 Real World Analogy

A CancellationToken is like a manager walking over to say "that project is cancelled — stop working on it". The CancellationTokenSource is the manager who has the authority to cancel. You pass the token through your entire chain so every piece of work can check if it should stop.

The Three Pieces

🎛️

CancellationTokenSource

The controller. Call .Cancel() on it to signal cancellation. Only the caller creates this.

🎫

CancellationToken

The signal. Passed into methods. Readonly — can only check the signal, not cancel it.

⏰

Timeout + Token

CancellationTokenSource can auto-cancel after a timeout — combines timeout and manual cancel.

Complete Cancellation Example

C# — Full cancellation pattern
// ── Step 1: Method accepts a CancellationToken ────────────────────
// Best practice: always include CancellationToken as last parameter
public async Task<List<Product>> SearchProductsAsync(
    string            query,
    CancellationToken cancellationToken = default)  // default = no cancellation
{
    // Pass token to every async call inside
    var results = await _dbContext.Products
        .Where(p => p.Name.Contains(query))
        .ToListAsync(cancellationToken);         // ← pass token here

    // Manually check for cancellation between steps
    cancellationToken.ThrowIfCancellationRequested();

    var enriched = await _enrichmentService
        .EnrichAsync(results, cancellationToken);  // ← pass token here too

    return enriched;
}

// ── Step 2: Caller creates the source and triggers cancellation ───
public async Task DemoAsync()
{
    // Create source — this controls cancellation
    using var cts = new CancellationTokenSource();

    // Example: auto-cancel after 5 seconds (timeout)
    cts.CancelAfter(TimeSpan.FromSeconds(5));

    try
    {
        var results = await SearchProductsAsync("laptop", cts.Token);
        Console.WriteLine($"Found {results.Count} products");
    }
    catch (OperationCanceledException)
    {
        // This is thrown when cancelled — always catch it
        Console.WriteLine("Search was cancelled or timed out");
    }
}

// ── Manual cancel (e.g., user clicks Cancel button) ──────────────
CancellationTokenSource? _searchCts;

private async void SearchButton_Click(object sender, EventArgs e)
{
    _searchCts?.Cancel();   // cancel previous search if running
    _searchCts = new CancellationTokenSource();

    try
    {
        var results = await SearchProductsAsync(searchBox.Text, _searchCts.Token);
        listBox.ItemsSource = results;
    }
    catch (OperationCanceledException) { }  // ignore — expected
}

private void CancelButton_Click(object sender, EventArgs e)
{
    _searchCts?.Cancel();  // signal all running searches to stop
}

Checking Cancellation Inside Your Code

C# — Three ways to check cancellation
public async Task ProcessItemsAsync(List<Item> items, CancellationToken ct)
{
    foreach (var item in items)
    {
        // Method 1: ThrowIfCancellationRequested — throws OperationCanceledException
        ct.ThrowIfCancellationRequested();  // clean and idiomatic ✅

        // Method 2: IsCancellationRequested — check without throwing
        if (ct.IsCancellationRequested)
        {
            Console.WriteLine("Stopping gracefully...");
            break;  // or return, or clean up and then throw
        }

        await ProcessOneItemAsync(item, ct);  // pass token down
    }
}

// Combine with Task.Delay for cancellable waits
public async Task PollAsync(CancellationToken ct)
{
    while (!ct.IsCancellationRequested)
    {
        await DoWorkAsync(ct);
        await Task.Delay(TimeSpan.FromSeconds(5), ct);  // cancellable delay
    }
}

CancellationToken Best Practices

  • Every public async method should accept a CancellationToken as its last parameter
  • Always pass the token down to every async call inside your method
  • Always catch OperationCanceledException at the top-level call site
  • Use CancelAfter() to set timeouts — better than writing separate timeout logic
  • Use default as the parameter default — callers who don't care can omit it
  • Dispose CancellationTokenSource with using — it holds unmanaged resources

⚙️ Parallel Programming & PLINQ

👶 Async vs Parallel — Know the Difference

Async is for waiting on I/O (database, network, file) without blocking a thread. Parallel is for CPU-heavy work where you want multiple CPU cores to literally process data at the same time. They solve different problems. Don't use Parallel.ForEach on database calls — use await. Don't use async/await for heavy computation — use Parallel.

Parallel.ForEach — Multiple CPU Cores

C# — Parallel.ForEach for CPU-bound work
// ✅ USE Parallel.ForEach for CPU-intensive operations
var images = GetAllImages();  // list of 1000 images to process

// Process all images using all available CPU cores simultaneously
Parallel.ForEach(images, image =>
{
    // Each image processed on a different thread/core
    ResizeImage(image);       // CPU work — benefits from parallelism
    ApplyWatermark(image);
    SaveToDisk(image);
});
Console.WriteLine("All images processed");

// With options — control degree of parallelism
var options = new ParallelOptions
{
    MaxDegreeOfParallelism = Environment.ProcessorCount,  // use all cores
    CancellationToken      = cancellationToken
};

Parallel.ForEach(images, options, image =>
{
    ProcessImage(image);
});

// Parallel.For — numeric ranges
Parallel.For(0, 1000, i =>
{
    results[i] = HeavyCalculation(i);
});

PLINQ — Parallel LINQ

C# — PLINQ parallel queries
var numbers = Enumerable.Range(1, 1_000_000).ToList();

// Regular LINQ — sequential, one core
var sequential = numbers
    .Where(n => IsPrime(n))   // CPU work per item
    .ToList();

// PLINQ — parallel, all cores (just add .AsParallel())
var parallel = numbers
    .AsParallel()                          // ← magic switch to PLINQ
    .Where(n => IsPrime(n))               // runs on multiple cores
    .ToList();

// PLINQ with options
var controlled = numbers
    .AsParallel()
    .WithDegreeOfParallelism(4)             // max 4 threads
    .WithCancellation(cancellationToken)    // cancellable
    .Where(n => IsPrime(n))
    .OrderBy(n => n)                        // preserve order if needed
    .ToList();

// Parallel aggregation
var sum = numbers
    .AsParallel()
    .Where(n => n % 2 == 0)
    .Sum();

When to Use What

ScenarioUse ThisWhy
Querying a databaseasync/awaitI/O bound — don't waste threads waiting
Calling a REST APIasync/awaitI/O bound — network latency
Reading a large fileasync/awaitI/O bound — disk speed
Resizing 1000 imagesParallel.ForEachCPU bound — use all cores
Computing prime numbersPLINQCPU bound — parallel computation
Processing large data in-memoryPLINQCPU bound — data transformation
CPU work + I/OTask.Run + awaitMove CPU work to background, await I/O

⚠️ Don't Mix These Up

Using Parallel.ForEach with async lambdas does NOT work as expected — the parallel loop completes before the async work is done. Use Task.WhenAll for concurrent async operations, and Parallel.ForEach only for synchronous CPU-bound work.

🔬 Advanced Async Topics

ValueTask — When to Use It

👶 ValueTask in Simple Terms

Every time you return a Task, .NET allocates a new object on the heap — even if the result is immediately available (like a cache hit). ValueTask avoids that allocation when the result is ready synchronously. Use it in high-performance scenarios where you call the method millions of times and often have a cached result.

C# — ValueTask for hot paths
// Use Task<T> by default — simple and correct
public async Task<int> GetValueAsync(int key)
{
    return await _db.GetIntAsync(key);
}

// Use ValueTask<T> when often returns synchronously (cache scenarios)
private Dictionary<int, int> _cache = new();

public ValueTask<int> GetCachedValueAsync(int key)
{
    // Cache hit: returns synchronously with NO heap allocation
    if (_cache.TryGetValue(key, out var cached))
        return new ValueTask<int>(cached);  // struct, no allocation

    // Cache miss: falls back to async (allocates Task normally)
    return new ValueTask<int>(FetchFromDbAsync(key));
}

// Rules for ValueTask:
// ✅ Only await once
// ✅ Don't store and await later
// ✅ Convert to Task with .AsTask() if you need to store/re-await
// ❌ Don't use unless profiling shows Task allocation is a bottleneck

IAsyncEnumerable — Async Streams

👶 What is IAsyncEnumerable?

Normally when you query a database you wait for ALL results to arrive, then process them. With IAsyncEnumerable you can process results as they arrive — one by one, asynchronously. Perfect for large datasets, live feeds, or streaming APIs where you don't want to wait for everything before starting work.

C# — IAsyncEnumerable streaming
// ── Producer: yields items one at a time asynchronously ────────── public async IAsyncEnumerable<LogEntry> ReadLogsAsync( [EnumeratorCancellation] CancellationToken ct = default) { await using var reader = await OpenLogReaderAsync(); while (await reader.ReadAsync(ct)) // read one line at a time from DB { yield return reader.GetEntry(); // yield back to caller immediately // Caller processes this entry while we fetch the next one } } // ── Consumer: process each item as it arrives ───────────────────── public async Task ProcessLogsAsync() { // await foreach — the async version of foreach await foreach (var entry in ReadLogsAsync()) { // Processes each log entry as it comes — no waiting for all await ProcessEntryAsync(entry); } } // With cancellation token await foreach (var entry in ReadLogsAsync().WithCancellation(ct)) { await ProcessEntryAsync(entry); } // Benefits vs returning List<T>: // ✅ Start processing before all data is loaded // ✅ Lower memory: don't hold entire dataset in memory // ✅ Better perceived performance: first result appears faster // ✅ Works great with large files, large DB result sets, live streams

SemaphoreSlim — Async Locking

C# — SemaphoreSlim for async mutual exclusion
// ❌ lock{} doesn't work with async — never do this
private readonly object _syncRoot = new();
public async Task BadAsync()
{
    lock (_syncRoot)                  // ❌ can't await inside lock
    {
        await DoWorkAsync();          // compiler error
    }
}

// ✅ SemaphoreSlim — the async-safe equivalent of lock
private readonly SemaphoreSlim _lock = new SemaphoreSlim(1, 1);  // 1 = mutex

public async Task SafeAsync()
{
    await _lock.WaitAsync();          // async wait for lock
    try
    {
        await DoWorkAsync();          // ✅ only one caller at a time
    }
    finally
    {
        _lock.Release();              // always release
    }
}

// SemaphoreSlim(N) — allow N concurrent callers (not just 1)
// e.g. limit to 5 concurrent DB connections:
private readonly SemaphoreSlim _dbThrottle = new SemaphoreSlim(5, 5);

Async Disposal — IAsyncDisposable

C# — IAsyncDisposable and await using
// Resources that do async cleanup implement IAsyncDisposable
public class DatabaseConnection : IAsyncDisposable
{
    public async ValueTask DisposeAsync()
    {
        await FlushPendingWritesAsync();  // async cleanup
        await _connection.CloseAsync();   // async close
    }
}

// Use "await using" instead of "using" for IAsyncDisposable
public async Task UseConnectionAsync()
{
    await using var conn = new DatabaseConnection();
    await conn.QueryAsync("SELECT * FROM Users");
    // conn.DisposeAsync() called automatically here — awaited properly
}

// HttpClient, SqlConnection, Stream — all support async disposal
await using var stream = File.OpenRead("data.bin");
await using var reader = new StreamReader(stream);

⚠️ Common Mistakes & Pitfalls

Mistake 1 — async void (except event handlers)

BAD C# — async void loses exceptions
// ❌ async void — exceptions are UNHANDLED and crash the app
public async void LoadDataAsync()  // ❌ never do this (except events)
{
    var data = await _repo.GetAsync();
    // If exception thrown here — it cannot be caught by caller
    // App will crash with unhandled exception!
}

// Caller can't catch it:
try
{
    LoadDataAsync();  // ❌ can't await — returns void
}
catch (Exception)  // ❌ never catches anything from async void
{ }
GOOD C# — return Task so callers can await and catch
// ✅ Return Task — callers can await and handle exceptions
public async Task LoadDataAsync()
{
    var data = await _repo.GetAsync();
}

try
{
    await LoadDataAsync();  // ✅ exceptions are catchable
}
catch (Exception ex)
{
    Console.WriteLine(ex.Message);  // ✅ this works
}

Mistake 2 — Deadlock with .Result and .Wait()

🔴 The Most Dangerous Async Mistake

Calling .Result or .Wait() on a Task in a synchronisation context (WPF, WinForms, ASP.NET classic) causes a deadlock. The thread blocks waiting for the Task, but the Task needs that same thread to complete. Both wait for each other forever.

BAD C# — Deadlock pattern
// ❌ DEADLOCK — In WPF button click handler:
private void Button_Click(object sender, EventArgs e)
{
    // .Result blocks the UI thread waiting for the task
    // The task tries to resume on the UI thread after await
    // But the UI thread is BLOCKED waiting for .Result
    // DEADLOCK — both wait for each other forever
    var data = GetDataAsync().Result;   // ❌ DEADLOCK
    var more = GetMoreAsync().GetAwaiter().GetResult();  // ❌ ALSO DEADLOCK
}
GOOD C# — async all the way down
// ✅ Make the caller async too — async all the way up the call chain private async void Button_Click(object sender, EventArgs e) { var data = await GetDataAsync(); // ✅ no blocking, no deadlock lblResult.Text = data; } // The golden rule: never call .Result or .Wait() on a Task // unless you are 100% sure there is no SynchronizationContext // (console apps are usually safe; WPF/WinForms/ASP.NET classic are not)

Mistake 3 — Accidentally Sequential with await in a loop

BAD C# — Accidentally sequential
// ❌ These look concurrent but are actually SEQUENTIAL // The await means "wait before starting the next one" var user1 = await GetUserAsync(1); // wait 1s, THEN start next var user2 = await GetUserAsync(2); // wait 1s, THEN start next var user3 = await GetUserAsync(3); // wait 1s // Total: 3 seconds
GOOD C# — Truly concurrent
// ✅ Start all tasks FIRST, then await all at once var t1 = GetUserAsync(1); // start immediately var t2 = GetUserAsync(2); // start immediately var t3 = GetUserAsync(3); // start immediately await Task.WhenAll(t1, t2, t3); // wait for all — total: 1 second var user1 = t1.Result; var user2 = t2.Result; var user3 = t3.Result;

Mistake 4 — Ignoring the returned Task (fire and forget wrong)

BAD C# — Silently swallowing exceptions
// ❌ Not awaiting = exceptions are silently swallowed SaveToDbAsync(); // ❌ exception thrown inside? You'll never know SendEmailAsync(); // ❌ fails silently // ✅ If you genuinely want fire-and-forget, handle exceptions explicitly _ = SaveToDbAsync() .ContinueWith(t => { if (t.IsFaulted) _logger.LogError(t.Exception, "Save failed"); }, TaskContinuationOptions.OnlyOnFaulted); // Or use Task.Run and log inside: _ = Task.Run(async () => { try { await SaveToDbAsync(); } catch (Exception ex) { _logger.LogError(ex, "Background save failed"); } });

Mistake 5 — Task.Run in ASP.NET

BAD C# — Unnecessary Task.Run in ASP.NET
// ❌ WRONG in ASP.NET — wrapping async I/O in Task.Run // This wastes two threads instead of zero public async Task<IActionResult> GetUsers() { var users = await Task.Run(() => // ❌ unnecessary thread allocation _repo.GetUsersAsync()); // GetUsersAsync is already async! return Ok(users); }
GOOD C# — Just await the async method directly
// ✅ CORRECT — just await the async method directly public async Task<IActionResult> GetUsers() { var users = await _repo.GetUsersAsync(); // ✅ direct await return Ok(users); } // Task.Run is for CPU-bound work in desktop apps to keep UI responsive // Not for wrapping async I/O in web apps

The Async Mistakes Cheat Sheet

  • Never use async void except for event handlers
  • Never call .Result or .Wait() in UI or ASP.NET contexts
  • Never await inside a loop when the operations are independent — use WhenAll
  • Never ignore the returned Task from an async method without handling its exceptions
  • Never use Task.Run to wrap already-async I/O methods in ASP.NET
  • Never use lock with async code — use SemaphoreSlim instead
  • Never assume that ConfigureAwait(false) prevents deadlocks — it helps in libraries but is not a fix

🏭 Real-World Async Patterns

Pattern 1 — Async Repository (Database)

C# — Full async repository with EF Core
public interface IUserRepository
{
    Task<User?>          GetByIdAsync(int id, CancellationToken ct = default);
    Task<List<User>>   GetAllAsync(CancellationToken ct = default);
    Task                  AddAsync(User user, CancellationToken ct = default);
    Task                  UpdateAsync(User user, CancellationToken ct = default);
    Task                  DeleteAsync(int id, CancellationToken ct = default);
}

public class UserRepository : IUserRepository
{
    private readonly AppDbContext _ctx;
    public UserRepository(AppDbContext ctx) => _ctx = ctx;

    public async Task<User?> GetByIdAsync(int id, CancellationToken ct = default)
        => await _ctx.Users.FindAsync(new object[] { id }, ct);

    public async Task<List<User>> GetAllAsync(CancellationToken ct = default)
        => await _ctx.Users.AsNoTracking().ToListAsync(ct);

    public async Task AddAsync(User user, CancellationToken ct = default)
    {
        await _ctx.Users.AddAsync(user, ct);
        await _ctx.SaveChangesAsync(ct);
    }

    public async Task UpdateAsync(User user, CancellationToken ct = default)
    {
        _ctx.Users.Update(user);
        await _ctx.SaveChangesAsync(ct);
    }

    public async Task DeleteAsync(int id, CancellationToken ct = default)
    {
        var user = await GetByIdAsync(id, ct);
        if (user != null)
        {
            _ctx.Users.Remove(user);
            await _ctx.SaveChangesAsync(ct);
        }
    }
}

Pattern 2 — Async HTTP Client

C# — Async REST API calls with HttpClient
public class WeatherService
{
    private readonly HttpClient  _http;
    private readonly ILogger     _logger;

    public WeatherService(HttpClient http, ILogger<WeatherService> logger)
    {
        _http   = http;
        _logger = logger;
    }

    public async Task<WeatherData?> GetWeatherAsync(
        string            city,
        CancellationToken ct = default)
    {
        try
        {
            // .GetFromJsonAsync deserialises JSON automatically
            var weather = await _http
                .GetFromJsonAsync<WeatherData>(
                    $"weather?city={city}", ct);

            return weather;
        }
        catch (HttpRequestException ex)
        {
            _logger.LogError(ex, "Failed to get weather for {City}", city);
            return null;
        }
        catch (TaskCanceledException) when (!ct.IsCancellationRequested)
        {
            // TaskCanceledException when NOT cancelled = timeout
            _logger.LogWarning("Weather request timed out for {City}", city);
            return null;
        }
    }

    public async Task<Dictionary<string, WeatherData?>> GetMultipleCitiesAsync(
        string[]          cities,
        CancellationToken ct = default)
    {
        // Fetch all cities concurrently
        var tasks = cities.Select(city =>
            GetWeatherAsync(city, ct)
                .ContinueWith(t => (city, result: t.Result), ct));

        var results = await Task.WhenAll(tasks);
        return results.ToDictionary(r => r.city, r => r.result);
    }
}

Pattern 3 — Background Service (.NET Worker)

C# — Async background worker / hosted service
// Runs continuously in the background — processes a queue, polls API, etc.
public class EmailQueueWorker : BackgroundService
{
    private readonly IEmailQueue   _queue;
    private readonly IEmailSender  _sender;
    private readonly ILogger        _logger;

    public EmailQueueWorker(IEmailQueue queue, IEmailSender sender,
                            ILogger<EmailQueueWorker> logger)
    { _queue = queue; _sender = sender; _logger = logger; }

    // This method runs for the lifetime of the application
    protected override async Task ExecuteAsync(CancellationToken stoppingToken)
    {
        _logger.LogInformation("Email worker started");

        // Keep running until the app shuts down (stoppingToken cancelled)
        while (!stoppingToken.IsCancellationRequested)
        {
            try
            {
                var email = await _queue.DequeueAsync(stoppingToken);

                if (email != null)
                    await _sender.SendAsync(email, stoppingToken);
                else
                    await Task.Delay(TimeSpan.FromSeconds(1), stoppingToken);
            }
            catch (OperationCanceledException)
            {
                break;  // graceful shutdown
            }
            catch (Exception ex)
            {
                // Log error but keep running — don't crash the worker
                _logger.LogError(ex, "Failed to process email");
                await Task.Delay(TimeSpan.FromSeconds(5), stoppingToken);
            }
        }

        _logger.LogInformation("Email worker stopped");
    }
}

Async Best Practices Summary

  • Async all the way — if one method is async, make all its callers async too. Don't mix sync and async.
  • Every async method name ends in Async by convention — GetUserAsync, SaveAsync
  • Always pass and respect CancellationToken through your entire call chain
  • Use Task.WhenAll for independent concurrent operations — never sequential await for independent work
  • Use ConfigureAwait(false) in library and service code to avoid context capture overhead
  • Use SemaphoreSlim for async mutual exclusion — never lock with async
  • Always wrap async operations in try/catch — exceptions in Tasks are not automatically surfaced
  • Don't use Task.Run in ASP.NET to wrap already-async methods — wasteful
  • Use IAsyncEnumerable for large data streams — don't buffer everything into a List
  • Use ValueTask only when profiling shows Task allocation is a bottleneck — not by default
