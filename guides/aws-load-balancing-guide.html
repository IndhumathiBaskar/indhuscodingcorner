⚖️ What is Load Balancing?

A load balancer sits in front of your servers and distributes incoming traffic across multiple targets — so no single server gets overwhelmed.

💡 Real-life analogy

Think of a supermarket with 5 checkout lanes. A load balancer is the staff member at the entrance who directs each customer to the least busy lane — everyone gets served faster, no lane is overloaded.

Without Load Balancer vs With Load Balancer

❌ Without Load Balancer
  • All 1000 users hit Server 1
  • Server 1 crashes under load
  • Server 2, 3, 4 sit idle
  • Users get 503 errors
  • Single point of failure
→
✅ With Load Balancer
  • 1000 users hit the Load Balancer
  • LB spreads: 250 each to Servers 1–4
  • No server overwhelmed
  • Fast responses for all users
  • If one fails, LB routes away

How AWS Load Balancing Works

👤 Users / Clients
  • Send requests to a single DNS name
  • e.g. myapp.com → resolves to Load Balancer IP
  • Never talk to your servers directly
↓
⚖️ AWS Load Balancer
  • Receives all incoming traffic
  • Checks health of each target
  • Applies routing rules
  • Selects target using chosen algorithm
  • Forwards request to healthy target
↓
🟢 Target Group (Your Servers)
  • EC2 instances, ECS tasks, Lambda, IPs
  • Each target registers with the LB
  • Only healthy targets receive traffic
  • Response goes back through LB to user

What AWS Load Balancing gives you

🔀 Traffic Distribution

Spread load evenly across healthy targets

💓 Health Monitoring

Auto-removes unhealthy targets from rotation

🔐 SSL/TLS Termination

Handle HTTPS certificates centrally

📈 Auto-Scaling Integration

New instances register automatically

🌍 Multi-AZ

Spans availability zones for high availability

📊 CloudWatch Metrics

Request count, latency, errors built-in

📦 Types of AWS Load Balancers

AWS offers 3 load balancer types. Each is designed for a specific use case. Choosing the right one matters for performance and cost.

ALB

Application

Layer 7 (HTTP/HTTPS). Understands web traffic. Routes based on URL path, headers, host.

  • Best for web apps & APIs
  • WebSocket support
  • gRPC support

NLB

Network

Layer 4 (TCP/UDP). Extremely fast. Static IP addresses. Ultra-low latency.

  • Best for real-time apps
  • IoT, gaming, finance
  • Millions of req/sec

GWLB

Gateway

Layer 3 (IP). Routes traffic through 3rd-party security appliances.

  • Best for firewalls & IDS
  • Network inspection
  • Security appliances

How to Choose — Decision Flow

❓ What type of traffic?
  • HTTP / HTTPS web traffic?
  • TCP / UDP raw network traffic?
  • Need security appliance inspection?
↓
ALB ✅
  • Web apps, REST APIs
  • Microservices routing
  • WebSocket, gRPC
  • Path/host-based routing
NLB ✅
  • Real-time, high throughput
  • IoT, gaming, VoIP
  • Need static IP
  • Ultra-low latency
GWLB ✅
  • Firewall inspection
  • Intrusion detection
  • DDoS protection
  • Network monitoring

🌐 ALB — Application Load Balancer Deep Dive

ALB is the most commonly used load balancer. It operates at Layer 7 and understands HTTP — meaning it can make smart routing decisions based on the content of requests.

ALB Request Journey

👤 Client Request
  • GET https://myapp.com/api/orders
  • DNS resolves to ALB endpoint
  • TCP connection established to ALB
↓
⚖️ ALB — Listener (Port 443)
  • SSL/TLS terminated here (HTTPS → HTTP internally)
  • Certificate managed by ACM (AWS Certificate Manager)
  • Evaluates listener rules top to bottom
↓
📋 Routing Rules Evaluated
  • Rule 1: Path /api/* → Target Group: api-services
  • Rule 2: Path /admin/* → Target Group: admin-service
  • Rule 3: Host api.myapp.com → Target Group: api-v2
  • Default: → Target Group: web-frontend
↓
🟢 Matched Target Group: api-services
  • Round-robin: selects next healthy target
  • Forwards request to: EC2 instance / ECS task / Lambda
  • Target processes request and returns response
  • ALB returns response to client

Key ALB Features

🔀 Content-Based Routing

  • Path: /api/* → API servers
  • Host: admin.myapp.com → Admin servers
  • Header: X-Version: v2 → v2 servers
  • Query: ?env=staging → staging servers

🔐 SSL/TLS Termination

  • HTTPS handled at ALB level
  • Certificate from AWS ACM (free)
  • Backends use plain HTTP internally
  • Supports SNI — multiple certs per LB

🧠 Sticky Sessions

  • Same user → same target every time
  • Uses a cookie (AWSALB)
  • Duration: 1 second to 7 days
  • Useful for stateful apps

🔁 Redirects & Fixed Responses

  • HTTP → HTTPS redirect (built-in)
  • 301/302 redirects without a server
  • Return fixed JSON/HTML without backend
  • Maintenance page with no backend

✅ Use ALB when

You're building a web app, REST API, microservices, or anything HTTP-based. It's the right choice for 90% of web workloads. ECS + ALB is the most common production pattern.

⚡ NLB — Network Load Balancer Deep Dive

NLB operates at Layer 4 (TCP/UDP). It doesn't look at HTTP headers — it just forwards raw TCP/UDP packets at extreme speed. Designed for ultra-low latency and millions of requests per second.

NLB Request Journey

👤 Client
  • TCP connection to 52.10.20.30:443 (static IP!)
  • NLB has fixed Elastic IP — doesn't change
  • Useful for whitelisting in firewalls
↓
⚡ NLB — Layer 4 Forwarding
  • Does NOT inspect HTTP content
  • Reads only: IP address + Port + Protocol
  • Hash-based routing (IP + port + protocol)
  • Same client always hits same target (natural stickiness)
  • Forwards in microseconds — no overhead
↓
🟢 Target (EC2 / ECS / IP)
  • Receives original client IP (no X-Forwarded-For needed)
  • Direct TCP connection — lowest possible latency
  • Can handle millions of simultaneous connections

NLB vs ALB — When to use which

ScenarioUse ALBUse NLB
REST API or web app✅—
Need path-based routing✅—
Real-time game server—✅
IoT device connections—✅
Need static IP for firewall—✅
Financial trading (sub-ms latency)—✅
WebSocket for chat app✅—
gRPC microservices✅—
UDP streaming / VoIP—✅

⚠️ NLB does NOT support

Path-based routing · Header-based routing · SSL termination (unless TLS listener) · HTTP redirects · Fixed responses · WAF integration. If you need any of these — use ALB instead.

🎯 Target Groups — Everything Explained

A Target Group is a collection of targets (servers) that your load balancer routes traffic to. One LB can have many target groups — each serving a different purpose.

Target Group Architecture

⚖️ ALB — single entry point
  • One DNS name for all traffic
  • Multiple listeners (port 80, 443)
  • Routes to different target groups based on rules
↓
TG: api-service
  • Rule: /api/*
  • 3 × ECS tasks
  • Port 5000
TG: frontend
  • Rule: /* (default)
  • 2 × EC2 instances
  • Port 80
TG: admin
  • Rule: /admin/*
  • 1 × Lambda function
  • N/A

Target Types

🖥️ Instance (EC2)

Register EC2 by instance ID. Traffic goes to the instance's primary IP. Auto-scaling groups register automatically.

🐳 IP Address

Register any private IP (ECS Fargate tasks, on-premises servers). Required for Fargate since tasks don't have instance IDs.

λ Lambda

ALB can invoke Lambda directly. Great for serverless APIs. ALB converts HTTP to Lambda event format automatically.

⚖️ ALB (nested)

NLB can forward to an ALB. Useful when you need static IP (NLB) + smart routing (ALB) together.

Load Balancing Algorithms

🔄 Round Robin (Default — ALB)

Requests distributed equally in order: Target 1 → Target 2 → Target 3 → Target 1...
Best when all targets have equal capacity.

🔗 Least Outstanding Requests (ALB)

Send to target with fewest in-flight requests.
Best when targets have varying response times.

#️⃣ Flow Hash (NLB)

Hash of IP + port + protocol → same client always hits same target.
Provides natural stickiness for TCP connections.

⚖️ Weighted (ALB)

Target A gets 90% traffic, Target B gets 10%.
Perfect for canary deployments — test new version with small traffic.

💓 Health Checks — How LB Monitors Targets

The load balancer continuously checks if each target is healthy. Unhealthy targets are automatically removed from rotation — users never hit a broken server.

Health Check Cycle

⚖️ ALB — Health Check Scheduler
  • Every 30 seconds (configurable): 5–300s
  • Sends GET /health to each target
  • Waits for response within timeout (5s default)
↓
✅ Healthy Response
  • Returns HTTP 200
  • Within 5 seconds
  • Stays in rotation
  • Counter resets to 0
❌ Unhealthy Response
  • Timeout or non-200 status
  • Fail count: 1 of 3
  • After 3 consecutive fails
  • Marked UNHEALTHY
↓
🔄 Unhealthy Target Handling
  • Immediately removed from rotation
  • New requests only go to healthy targets
  • Health checks continue every 30s
  • After 5 consecutive passes → HEALTHY → back in rotation

Health Check Configuration

⚙️ Key Settings

  • Protocol: HTTP / HTTPS / TCP
  • Path: /health (or /ping, /status)
  • Port: traffic port or override
  • Interval: 5–300 seconds
  • Timeout: 2–120 seconds
  • Healthy threshold: 2–10 successes
  • Unhealthy threshold: 2–10 failures
  • Success codes: 200, 200-299, 301

💡 Best Health Check Endpoint

  • Return 200 only when fully ready
  • Check database connectivity
  • Check cache connectivity
  • Check dependent services
  • Return 503 during startup
  • Return 503 during graceful shutdown
  • Respond fast — under 1 second

C# Health Check Endpoint

HealthController.cs 
using Microsoft.AspNetCore.Mvc;

[ApiController]
[Route("health")]
public class HealthController : ControllerBase
{
    private readonly IDbConnection _db;

    public HealthController(IDbConnection db) => _db = db;

    [HttpGet]
    public async IActionResult GetHealth()
    {
        try
        {
            // Check DB is reachable
            await _db.QueryAsync("SELECT 1");

            return Ok(new {
                status  = "healthy",
                time    = DateTime.UtcNow,
                version = "1.0.0"
            });
        }
        catch
        {
            // Return 503 — LB removes this target
            return StatusCode(503, new { status = "unhealthy" });
        }
    }
}

📋 Routing Rules — Smart Traffic Distribution

ALB evaluates listener rules in priority order (1 = highest priority). The first rule that matches decides where the request goes.

How Rules Are Evaluated

📨 Incoming Request
  • POST https://myapp.com/api/v2/orders
  • Header: X-Platform: mobile
  • Query: ?env=staging
↓
🔍 Rule Priority 1 — Check first
  • Condition: Path = /api/v2/* AND Header X-Platform = mobile
  • Match? YES ✅
  • Action: Forward → Target Group: mobile-api-v2
  • Rule evaluation stops here
↓
🟢 Forwarded to: mobile-api-v2
  • Optimized mobile API service
  • Request processed and response returned

Rule Condition Types

🗂️ Path Pattern

  • /api/* → API service
  • /admin/* → Admin service
  • /static/* → S3 / CDN
  • *.jpg → Image service

🌐 Host Header

  • api.myapp.com → API servers
  • admin.myapp.com → Admin servers
  • *.myapp.com → wildcard match

📋 HTTP Header

  • X-Version: v2 → v2 servers
  • Accept-Language: fr → French service
  • Authorization: * → Auth required

🔍 Query String

  • ?env=staging → staging TG
  • ?version=2 → v2 TG
  • ?canary=true → canary TG

Rule Actions

➡️ Forward

Send to a target group. Can be weighted for canary: 90% to TG-v1, 10% to TG-v2.

↪️ Redirect

HTTP → HTTPS redirect. 301 or 302. Change host, path, port, or protocol.

📝 Fixed Response

Return a static response (200, 404, 503) without hitting any backend. Useful for maintenance pages.

🔐 Authenticate (OIDC/Cognito)

ALB can handle authentication via Cognito or OIDC — your app doesn't need auth code.

💻 C# SDK — Create and Manage Load Balancers

Install NuGet

Install-Package AWSSDK.ElasticLoadBalancingV2
Install-Package AWSSDK.Core

Create ALB + Target Group + Listener

LoadBalancerService.cs 
using Amazon.ElasticLoadBalancingV2;
using Amazon.ElasticLoadBalancingV2.Model;

public class LoadBalancerService
{
    private readonly IAmazonElasticLoadBalancingV2 _elb;

    public LoadBalancerService(IAmazonElasticLoadBalancingV2 elb)
        => _elb = elb;

    // Step 1: Create ALB
    public async Task<string> CreateAlbAsync()
    {
        var response = await _elb.CreateLoadBalancerAsync(
            new CreateLoadBalancerRequest
        {
            Name    = "my-alb",
            Subnets = ["subnet-abc1", "subnet-abc2"], // 2 AZs!
            SecurityGroups = ["sg-abc123"],
            Scheme  = LoadBalancerSchemeEnum.InternetFacing,
            Type    = LoadBalancerTypeEnum.Application,
            IpAddressType = IpAddressTypeEnum.Ipv4
        });
        return response.LoadBalancers[0].LoadBalancerArn;
    }

    // Step 2: Create Target Group
    public async Task<string> CreateTargetGroupAsync(string vpcId)
    {
        var response = await _elb.CreateTargetGroupAsync(
            new CreateTargetGroupRequest
        {
            Name             = "api-targets",
            Protocol         = ProtocolEnum.HTTP,
            Port             = 80,
            VpcId            = vpcId,
            TargetType       = TargetTypeEnum.Ip, // for ECS Fargate
            // Health check settings
            HealthCheckPath              = "/health",
            HealthCheckIntervalSeconds   = 30,
            HealthCheckTimeoutSeconds    = 5,
            HealthyThresholdCount        = 2,
            UnhealthyThresholdCount      = 3,
            Matcher = new Matcher { HttpCode = "200" }
        });
        return response.TargetGroups[0].TargetGroupArn;
    }

    // Step 3: Create HTTPS Listener with redirect + routing rules
    public async Task CreateListenerAsync(
        string albArn, string targetGroupArn, string certArn)
    {
        await _elb.CreateListenerAsync(new CreateListenerRequest
        {
            LoadBalancerArn = albArn,
            Protocol        = ProtocolEnum.HTTPS,
            Port            = 443,
            Certificates    = [new Certificate
                { CertificateArn = certArn }],
            DefaultActions  = [new Action
            {
                Type           = ActionTypeEnum.Forward,
                TargetGroupArn = targetGroupArn
            }]
        });
    }

    // Register ECS task as target
    public async Task RegisterTargetAsync(
        string targetGroupArn, string ip, int port)
    {
        await _elb.RegisterTargetsAsync(new RegisterTargetsRequest
        {
            TargetGroupArn = targetGroupArn,
            Targets = [new TargetDescription
            {
                Id   = ip,   // ECS task private IP
                Port = port
            }]
        });
    }
}

✅ Best Practices & Checklist

Architecture

  • Always deploy ALB/NLB across at least 2 Availability Zones
  • Put the Load Balancer in public subnets, targets in private subnets
  • Use separate security groups for LB and targets
  • Enable deletion protection on production load balancers
  • Use ACM (AWS Certificate Manager) for free SSL certificates
  • Enable HTTP → HTTPS redirect on port 80 listener

Health Checks

  • Create a dedicated /health endpoint — never use / (root)
  • Health endpoint must check real dependencies (DB, cache)
  • Return 503 during app startup — not 200
  • Return 503 during graceful shutdown to drain connections
  • Set unhealthy threshold to 3 (not 2) to avoid flapping
  • Set health check interval to 30s — 10s is usually too aggressive

Performance & Cost

  • Enable ALB access logs to S3 — helps debug production issues
  • Use connection draining (deregistration delay) — default 300s is fine
  • Enable cross-zone load balancing for even distribution
  • Use weighted routing for canary deployments — not blue-green
  • Monitor RequestCount, TargetResponseTime, HTTPCode_ELB_5XX in CloudWatch

Security

  • Attach AWS WAF to ALB — protect against common web attacks
  • Use security group on ALB: allow 80/443 from internet only
  • Use security group on targets: allow traffic from ALB SG only
  • Enable sticky sessions only when truly needed — stateless is better
  • Use HTTPS between ALB and targets for end-to-end encryption

Common Mistakes to Avoid

⚠️ Top mistakes

❌ Single AZ deployment → whole app down if AZ fails
❌ Targets in public subnets → unnecessary security risk
❌ Using / as health check path → always 200, hides real issues
❌ Too low health check interval → false positives, instability
❌ Same security group for LB and targets → no isolation
❌ Forgetting connection draining → requests dropped during deploys
❌ No ALB access logs → blind in production when something breaks
