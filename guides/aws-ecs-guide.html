🚀 What is Amazon ECS?

ECS = Elastic Container Service. A fully managed AWS service to run, scale, and orchestrate Docker containers — without managing the underlying servers yourself.

💡 Simple Analogy

ECS is like a concert conductor. Your Docker containers are the musicians. ECS tells each one what to play, when to start, how many to add during a busy show, and replaces anyone who stops.

🏗️ ECS Architecture — Complete Flow

From your Docker image all the way to users receiving a response:

📦 Step 1 — Docker Image in ECR
  • App packaged as Docker image and pushed to Amazon ECR
  • Tagged with version: myapi:v1.0 (~50–500 MB)
  • Contains app code + runtime + all dependencies
↓
📋 Step 2 — Task Definition (Blueprint)
  • JSON config: CPU 0.25 vCPU, Memory 512 MB, Port 80
  • Environment variables, IAM roles, CloudWatch logging
  • Health check endpoint: GET /health
↓
⚙️ Step 3 — ECS Service (Manager)
  • Desired count: 5 tasks always running
  • Launch type: Fargate (serverless)
  • Connected to ALB for load balancing
↓
🟢 Step 4 — Tasks Running (Live Containers)
  • Task 1: Running at 10.0.1.10
  • Task 2: Running at 10.0.1.11
  • Tasks 3–5: Running on separate private IPs
↓
⚖️ Step 5 — ALB Health Checks & Traffic
  • ALB pings each task: GET /health
  • All healthy → distributes traffic across all 5
  • Users get response in ~100ms
↓
📈 Step 6 — Auto-Scaling
  • CPU rises to 85% → target is 70% → add 2 tasks
  • New tasks pass health checks → added to ALB
  • CPU drops to 72% → removes 2 tasks → back to 5
↓
🔄 Step 7 — Rolling Deployment (Zero Downtime)
  • New image myapi:v2 → ECS creates 2 v2 tasks
  • v2 passes health checks → removes 2 v1 tasks
  • Repeats: 5→7→5→3→5→1→5 — no downtime!
↓
👤 Users Access Your App
  • http://myapp.com → ALB → 5 healthy tasks
  • Fast, reliable, scalable, always available ✅

🧩 5 Core Components — Everything Explained

1

Docker Image (from ECR)

What: Blueprint for your container
Contains: App code + runtime + all dependencies
Stored in: Amazon ECR (Elastic Container Registry)
Size: Typically 50–500 MB
Pulled when: ECS starts a new task

2

Task Definition (JSON Blueprint)

What: JSON config describing how to run your container
Defines: Image URI, CPU, Memory, ports, env vars, secrets, logging, IAM roles, health check
Versioned: myapp:1 → myapp:2 → myapp:3 (immutable)
Can include: Multiple containers (sidecar pattern)

3

Task (Running Container Instance)

Lifecycle: PROVISIONING → PENDING → ACTIVATING → RUNNING → STOPPING → STOPPED
Gets: Unique private IP + hostname
Logs: Sent to CloudWatch automatically
If it fails: ECS service replaces it automatically

4

Service (Task Manager)

What: Keeps your desired number of tasks always running
Auto-Scaling: Target Tracking, Step Scaling, Scheduled
Load Balancer: ALB, NLB, or CLB integration
Deployments: Rolling, Blue-Green, Canary
Service Discovery: DNS-based via AWS Cloud Map

5

Cluster (Infrastructure Group)

What: Logical group of tasks and services
Launch types: Fargate (serverless) or EC2 (you manage)
Networking: Lives inside your VPC — subnets + security groups
Best practice: Separate cluster per environment (dev/staging/prod)

🎯 Launch Types — Fargate vs EC2

✅ TL;DR — Use Fargate for 90% of apps

Unless you need GPUs, very large memory, or specific compliance requirements — Fargate is the right choice. AWS manages everything, you just define what to run.

FeatureFargate ✅ RecommendedEC2 (Advanced)
Server ManagementZERO — AWS handles allYou manage OS, patches
InfrastructureInvisible — AWS providesYou provide EC2 instances
ScalingAutomatic — set desired countManual or complex auto-scaling
PricingPer task (vCPU-hour + GB-hour)Per EC2 instance (24/7 cost)
Idle CostZERO — pay only when runningEC2 costs even when idle
Startup Time~30 secondsDepends on EC2 readiness
Max Memory30 GB per taskEC2 instance dependent
GPU Support❌ Not supported✅ Supported
Best ForAPIs, microservices, web appsGPU workloads, large memory

Fargate Pricing Example

💰 Cost Calculation — 5 tasks running 24/7

Task: 0.25 vCPU + 512 MB RAM per task, 30 days

vCPU: 0.25 × $0.04048/hr × 720 hrs = $7.29 per task
Memory: 0.5 GB × $0.004445/GB/hr × 720 hrs = $1.60 per task
Per task: ~$8.89/month → 5 tasks = ~$44.45/month

⚙️ Console Setup — Step by Step

1

Push Docker Image to ECR

AWS Console → ECR → Create repository → name: myapi
Click repo → View push commands → run the 4 commands shown
docker build -t myapi .
docker push [account].dkr.ecr.[region].amazonaws.com/myapi:latest

2

Create Task Definition

ECS → Task Definitions → Create new → Fargate
Family name: myapi | CPU: 0.25 vCPU | Memory: 0.5 GB
Add container → image URI from ECR → port 80
Add env vars → CloudWatch logging → Create

3

Create Cluster

ECS → Clusters → Create cluster
Name: production-cluster
Infrastructure: AWS Fargate (serverless) ← choose this
Click Create — takes ~30 seconds

4

Create Service

Cluster → Services → Create
Task definition: myapi:1 | Desired tasks: 2
Launch type: Fargate | Platform: LATEST
VPC → subnets → security group (allow port 80)
Load balancer: ALB → Create service

5

Verify Deployment

ECS → Clusters → production-cluster → Services → myapi
Tasks tab → wait for Status: RUNNING (~1-2 min)
Copy ALB DNS name → paste in browser → your app is live! 🎉

💻 C# SDK — Complete Implementation

ECS Service Class

EcsService.cs 
using Amazon.ECS;
using Amazon.ECS.Model;

public class EcsService
{
    private readonly IAmazonECS _ecs;

    public EcsService(IAmazonECS ecs) => _ecs = ecs;

    // Run a one-off task (e.g. background job)
    public async Task<string> RunTaskAsync(
        string cluster, string taskDefinition)
    {
        var response = await _ecs.RunTaskAsync(new RunTaskRequest
        {
            Cluster        = cluster,
            TaskDefinition = taskDefinition,
            LaunchType     = LaunchType.FARGATE,
            Count          = 1,
            NetworkConfiguration = new NetworkConfiguration
            {
                AwsvpcConfiguration = new AwsVpcConfiguration
                {
                    Subnets        = ["subnet-abc123"],
                    SecurityGroups = ["sg-abc123"],
                    AssignPublicIp = AssignPublicIp.ENABLED
                }
            }
        });
        return response.Tasks[0].TaskArn;
    }

    // List all running tasks in a cluster
    public async Task<List<string>> ListRunningTasksAsync(string cluster)
    {
        var response = await _ecs.ListTasksAsync(new ListTasksRequest
        {
            Cluster       = cluster,
            DesiredStatus = "RUNNING"
        });
        return response.TaskArns;
    }

    // Deploy new version — rolling update
    public async Task DeployNewVersionAsync(
        string cluster, string service, string newTaskDef)
    {
        await _ecs.UpdateServiceAsync(new UpdateServiceRequest
        {
            Cluster            = cluster,
            Service            = service,
            TaskDefinition     = newTaskDef,
            ForceNewDeployment = true
        });
    }

    // Scale service up or down
    public async Task ScaleServiceAsync(
        string cluster, string service, int desiredCount)
    {
        await _ecs.UpdateServiceAsync(new UpdateServiceRequest
        {
            Cluster      = cluster,
            Service      = service,
            DesiredCount = desiredCount
        });
    }
}

Register in DI (Program.cs)

Program.cs 
builder.Services.AddAWSService<IAmazonECS>();
builder.Services.AddScoped<EcsService>();

📈 Auto-Scaling — Scale Automatically

1

Target Tracking (Most Common)

Set a target metric — ECS keeps it there automatically
Example: Keep average CPU at 70%
CPU > 70% → ECS adds tasks | CPU < 70% → ECS removes tasks
Also works with: Memory utilization, ALB requests per target

2

Step Scaling (Rules-Based)

Define rules for specific thresholds:
CPU 70–80% → add 1 task
CPU 80–90% → add 2 tasks
CPU 90%+ → add 4 tasks
Good for predictable, non-linear traffic patterns

3

Scheduled Scaling (Time-Based)

Pre-scale based on known traffic times:
9am Mon–Fri → scale to 10 tasks
6pm Mon–Fri → scale to 3 tasks
Great for business-hours traffic patterns

C# Auto-Scaling Setup

using Amazon.ApplicationAutoScaling;
using Amazon.ApplicationAutoScaling.Model;

// Step 1: Register scalable target
await autoScaling.RegisterScalableTargetAsync(new
    RegisterScalableTargetRequest
{
    ServiceNamespace  = ServiceNamespace.Ecs,
    ResourceId        = "service/production-cluster/myapi",
    ScalableDimension = ScalableDimension.EcsServiceDesiredCount,
    MinCapacity       = 2,
    MaxCapacity       = 20
});

// Step 2: Target tracking — keep CPU at 70%
await autoScaling.PutScalingPolicyAsync(new PutScalingPolicyRequest
{
    PolicyName        = "cpu-70-percent",
    PolicyType        = PolicyType.TargetTrackingScaling,
    ResourceId        = "service/production-cluster/myapi",
    ScalableDimension = ScalableDimension.EcsServiceDesiredCount,
    ServiceNamespace  = ServiceNamespace.Ecs,
    TargetTrackingScalingPolicyConfiguration = new
        TargetTrackingScalingPolicyConfiguration
    {
        TargetValue = 70.0,
        PredefinedMetricSpecification = new PredefinedMetricSpecification
        {
            PredefinedMetricType =
                MetricType.ECSServiceAverageCPUUtilization
        },
        ScaleInCooldown  = 300,
        ScaleOutCooldown = 60
    }
});

🔄 Deployment Strategies

Rolling Update (Default)

How it works

ECS replaces tasks gradually — old and new run simultaneously during deployment

Example with 5 tasks:
5 v1 → 7 (5v1+2v2) → 5 (3v1+2v2) → 5 (1v1+4v2) → 5 v2

✅ Zero downtime  |  ✅ Simple  |  ✅ Default choice
⚠️ Both versions run simultaneously during deployment

Blue-Green Deployment

How it works

Two complete environments — Blue (current) and Green (new):

1. Deploy v2 to Green (same size as Blue)
2. Health check Green fully
3. Switch ALB from Blue → Green (instant!)
4. Keep Blue running 10 min as rollback safety
5. Terminate Blue after confirmation

✅ Instant rollback  |  ✅ Clean cutover  |  ⚠️ 2x cost during transition

Canary Deployment

How it works

Route a small % of real traffic to v2, monitor, gradually increase:

5% v2 → monitor errors → 20% v2 → monitor → 100% v2

✅ Safest — catch issues with real traffic
✅ Easy rollback at any step  |  ⚠️ More complex to configure

Rollback in C#

// Roll back to previous task definition version
await _ecs.UpdateServiceAsync(new UpdateServiceRequest
{
    Cluster            = "production-cluster",
    Service            = "myapi",
    TaskDefinition     = "myapi:5", // previous stable version
    ForceNewDeployment = true
});

✅ Best Practices & Checklist

Infrastructure

  • Use Fargate unless you need GPUs or very large memory
  • Always define health checks in task definition (/health endpoint)
  • Put ALB in public subnet, tasks in private subnet
  • Use separate clusters per environment (dev/staging/prod)
  • Enable Container Insights for CloudWatch metrics

Task Definitions

  • Never hardcode secrets — use AWS Secrets Manager or SSM
  • Set CPU and memory limits — never leave unlimited
  • Always configure CloudWatch logging for every container
  • Pin your Docker image version — never use :latest in production
  • Separate execution role (ECR pull) and task role (app permissions)

Scaling & Performance

  • Set minimum 2 tasks for high availability across AZs
  • Start with target tracking at 70% CPU — tune from real data
  • Scale-in cooldown longer than scale-out (300s vs 60s)
  • Use Spot Fargate for non-critical workloads — save up to 70%

Deployments

  • Enable deployment circuit breaker — auto rollback on failure
  • Tag all images with git commit SHA for full traceability
  • Test in staging with production-like task sizes first
  • Never deploy on Friday afternoon 😄

⚠️ Top Mistakes to Avoid

❌ Using :latest image tag in production (no rollback traceability)
❌ Not setting health checks → tasks start but fail silently
❌ Too little memory → tasks crash with OOMKilled error
❌ Forgetting IAM task role → app can't access S3, DynamoDB
❌ Not enabling deployment circuit breaker → bad deploy loops forever
