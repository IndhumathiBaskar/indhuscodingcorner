🎟️ What is AWS STS?

STS = Security Token Service. The AWS service that issues short-lived, temporary security credentials. Instead of using long-lived IAM access keys (risky), STS gives you credentials that expire automatically — reducing your security blast radius.

💡 Real-life analogy

STS is like a visitor badge system at a secure office. Instead of giving someone a permanent employee key card, you issue a time-limited visitor badge that works for 1 hour, gives access only to specific floors, and automatically expires. If it's stolen — it's useless after expiry.

Permanent Keys vs Temporary Credentials

❌ Permanent Access Keys
  • Never expire — valid until manually deleted
  • If leaked → attacker has unlimited access
  • Hard to rotate without breaking apps
  • Stored in config files — risky
  • Same key used everywhere — hard to audit
→
✅ STS Temporary Credentials
  • Auto-expire in 15 min to 12 hours
  • If leaked → useless after expiry
  • Automatically rotated by SDK
  • Never stored — fetched fresh each time
  • Per-session identity in CloudTrail

Where STS Fits in the AWS Ecosystem

👤 Who calls STS?
  • IAM Users (assume a role for elevated access)
  • AWS services (EC2, ECS, Lambda — auto credentials)
  • External identity providers (Google, Cognito, SAML)
  • Other AWS accounts (cross-account access)
  • CI/CD pipelines (GitHub Actions, CodeBuild)
↓
🎟️ STS — Issues Temporary Credentials
  • Validates caller identity
  • Checks trust policy of the target role
  • Returns: AccessKeyId + SecretKey + SessionToken + Expiry
  • Credentials valid for configured duration
↓
✅ Caller Uses Temp Credentials
  • Makes AWS API calls as the assumed role
  • All actions logged under role + session name
  • Credentials auto-refreshed by SDK before expiry
  • When expired — must call STS again

⚙️ How STS Works — Deep Dive

STS Request Lifecycle

📤 Step 1 — Caller requests credentials
  • Caller: IAM user, role, or external identity
  • API: AssumeRole / AssumeRoleWithWebIdentity / GetFederationToken
  • Provides: Role ARN, session name, optional session policy
  • Optionally: External ID, MFA token, duration
↓
🔍 Step 2 — STS validates the request
  • Is the caller authenticated? (IAM check)
  • Does the role's trust policy allow this caller?
  • Is the caller allowed to call sts:AssumeRole?
  • Are all conditions met (MFA, External ID, IP)?
↓
🎟️ Step 3 — STS issues credentials
  • AccessKeyId: ASIA... (starts with ASIA for temp)
  • SecretAccessKey: random 40-char string
  • SessionToken: long token (must be sent with every API call)
  • Expiration: UTC timestamp when credentials expire
↓
✅ Step 4 — Caller uses credentials
  • Signs all API requests with AccessKey + SecretKey + SessionToken
  • AWS validates all 3 on every call
  • Session token missing? → Denied immediately
  • Credentials expired? → Denied, must get new ones

Temporary Credentials vs Permanent Keys — Structure

🔑 Permanent IAM Access Key

  • AccessKeyId: AKIA... (starts with AKIA)
  • SecretAccessKey: 40 chars
  • No session token needed
  • Never expires unless manually deleted

🎟️ STS Temporary Credentials

  • AccessKeyId: ASIA... (starts with ASIA)
  • SecretAccessKey: 40 chars
  • SessionToken: required on every call!
  • Expires: ISO 8601 timestamp

💡 How to tell if credentials are temporary

Check the AccessKeyId prefix: AKIA = permanent IAM key. ASIA = temporary STS credentials. If you see ASIA, a SessionToken is always required alongside it — calls without it will be rejected.

🎭 AssumeRole — The Core STS Operation

AssumeRole is the most important STS API. It lets an IAM identity temporarily take on the permissions of a different IAM role. Used by EC2, ECS, Lambda, developers, and CI/CD pipelines.

AssumeRole — Complete Flow

👤 Caller — IAM User or Role
  • Example: IAM user "indhu-admin" (Account A)
  • Has permission: sts:AssumeRole on target role
  • Calls: sts:AssumeRole with role ARN + session name
↓
🔍 STS — Validates Trust Relationship
  • Opens target role's trust policy
  • Is "indhu-admin" in the Principal list? YES → continue
  • Are all Conditions satisfied? (MFA, IP, ExternalId)
  • Does caller have sts:AssumeRole in their IAM policy?
↓
🎟️ STS — Returns Temporary Credentials
  • AccessKeyId, SecretAccessKey, SessionToken
  • Expiry: Now + requested duration (max 12 hours)
  • Session name used in CloudTrail: indhu-admin/my-session
↓
✅ Caller Now Acts as the Assumed Role
  • Has exactly the permissions of the assumed role
  • Original permissions temporarily suspended
  • All actions logged as: assumed-role/target-role/session-name
  • When credentials expire → automatically back to original identity

Trust Policy — The Other Side of AssumeRole

The role being assumed must have a trust policy that explicitly allows the caller. Think of it as the role saying "I trust these callers to wear my permissions."

Trust policy — allow specific IAM user to assume this role 
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Principal": {
      "AWS": "arn:aws:iam::111122223333:user/indhu-admin"
    },
    "Action": "sts:AssumeRole",
    "Condition": {
      "Bool": {
        "aws:MultiFactorAuthPresent": "true"
      }
    }
  }]
}
Trust policy — allow EC2 service to assume this role 
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Principal": {
      "Service": "ec2.amazonaws.com"
    },
    "Action": "sts:AssumeRole"
  }]
}

Session Policy — Narrowing Permissions Further

🔬 What is a Session Policy?

When calling AssumeRole, you can optionally pass a session policy — an inline JSON policy that further restricts the assumed role's permissions for that specific session only.

Result: effective permissions = assumed role permissions ∩ session policy (intersection — whichever is more restrictive).

Use case: A deployment pipeline assumes a broad DevOps role, but adds a session policy limiting it to only the specific service being deployed.

🔁 Cross-Account Access — STS in Action

One of the most powerful STS patterns. Account A (your app) assumes a role in Account B (another AWS account) to access its resources — without sharing credentials between accounts.

Cross-Account Access Flow

🏢 Account A — Your App (111122223333)
  • EC2/ECS task running your application
  • Has its own IAM role: my-app-role
  • Needs to read from S3 in Account B
  • Calls: sts:AssumeRole on Account B's role ARN
↓
🎟️ STS — Cross-Account Validation
  • Account B's role trust policy: Principal = arn:aws:iam::111122223333:role/my-app-role
  • Does Account A's role have sts:AssumeRole IAM permission? Yes
  • All conditions met → issue credentials
↓
🎟️ Temp Credentials for Account B's Role
  • AccessKeyId, SecretAccessKey, SessionToken
  • These credentials belong to Account B
  • Permission: only what Account B's role allows
↓
✅ App reads from Account B's S3
  • Uses temp credentials to call s3:GetObject in Account B
  • CloudTrail in Account B logs: assumed-role/crossaccount-role/my-session
  • Credentials expire → app requests fresh credentials

Setup Required on Both Sides

🏢 Account A — The Caller

  • IAM policy on caller role must allow:
  • sts:AssumeRole on Account B's role ARN
  • No changes needed in Account A's S3/other resources

🏢 Account B — The Target

  • Create IAM role with appropriate permissions (e.g. S3 read)
  • Trust policy: allow Account A's role ARN as Principal
  • Optional: add ExternalId condition for extra security

External ID — The Confused Deputy Problem

🤔 What is the Confused Deputy Problem?

Imagine you tell a partner company "assume our cross-account role using your account ID." If a bad actor knows your partner's account ID, they could trick the partner into assuming your role on their behalf.

Solution: External ID — a secret token you share privately with the trusted party. Only they know it. The trust policy requires it. Without the exact External ID → assumption denied.

Use when: Granting cross-account access to 3rd-party partners or SaaS vendors.

Trust policy with External ID for 3rd party access 
{
  "Statement": [{
    "Effect": "Allow",
    "Principal": {
      "AWS": "arn:aws:iam::PARTNER-ACCOUNT:root"
    },
    "Action": "sts:AssumeRole",
    "Condition": {
      "StringEquals": {
        "sts:ExternalId": "your-secret-token-only-partner-knows"
      }
    }
  }]
}

Common Cross-Account Scenarios

🔧 Dev/Staging → Prod Access

  • Dev account assumes read-only role in Prod
  • Developers can view prod logs/metrics
  • Cannot modify prod resources
  • Full audit trail in prod CloudTrail

🔎 Audit Access

  • Security audit account assumes role in all accounts
  • Read-only access for compliance review
  • External ID required for auditor access
  • Session policy limits to audit-only actions

🚀 CI/CD Deployment

  • Build account assumes deploy role in each env
  • Separate deploy roles per environment
  • Session policy limits to deployment actions only
  • Each deployment gets unique session name

🤝 SaaS Integration

  • SaaS vendor assumes role in your account
  • External ID prevents confused deputy
  • Minimal permissions — only what SaaS needs
  • Easy to revoke — just update trust policy

🌐 Federation & OIDC — External Identity → AWS

Federation lets users authenticated by an external identity provider (Google, Cognito, corporate AD, GitHub) get temporary AWS credentials — without having IAM users.

Web Identity Federation Flow (OIDC)

👤 User logs in with Google / Cognito
  • User authenticates with their Google account
  • Google returns an OIDC ID token (JWT)
  • Token contains: sub (user ID), email, aud (client ID)
↓
📤 App calls STS with the OIDC token
  • API: AssumeRoleWithWebIdentity
  • Provides: Role ARN + WebIdentityToken (the JWT) + session name
  • STS validates the JWT with Google's public key
↓
🔍 STS validates and issues credentials
  • Validates JWT signature against Google's JWKS endpoint
  • Checks role trust policy allows web identity federation
  • Optional: check aud, sub conditions in trust policy
  • Issues temporary AWS credentials
↓
✅ User has temporary AWS access
  • Can call AWS APIs (S3, DynamoDB) with temp credentials
  • Permissions limited to what the assumed role allows
  • CloudTrail logs: assumed-role/web-user-role/google-user-id

GitHub Actions → AWS (OIDC, No Stored Keys)

🤖 GitHub Actions workflow starts
  • GitHub automatically generates an OIDC token for the workflow
  • Token contains: repo, branch, job, workflow info
  • No AWS keys stored in GitHub Secrets!
↓
🎟️ AWS Action calls STS AssumeRoleWithWebIdentity
  • OIDC provider: token.actions.githubusercontent.com
  • STS validates token and checks trust policy conditions
  • Trust policy can require: specific repo, branch, or environment
  • Returns temp credentials valid for 1 hour
↓
✅ GitHub workflow deploys to AWS
  • Deploys ECS, pushes to ECR, updates Lambda — all with temp creds
  • Credentials expire when job ends — zero cleanup needed
  • Each run gets fresh credentials — no shared secret

Trust Policy for GitHub Actions OIDC

Trust policy — allow only specific GitHub repo/branch 
{
  "Statement": [{
    "Effect": "Allow",
    "Principal": {
      "Federated": "arn:aws:iam::ACCOUNT:oidc-provider/token.actions.githubusercontent.com"
    },
    "Action": "sts:AssumeRoleWithWebIdentity",
    "Condition": {
      "StringEquals": {
        "token.actions.githubusercontent.com:aud": "sts.amazonaws.com"
      },
      "StringLike": {
        "token.actions.githubusercontent.com:sub":
          "repo:IndhumathiBaskar/indhuscodingcorner:ref:refs/heads/main"
      }
    }
  }]
}

SAML Federation — Corporate SSO → AWS Console

1

User logs into corporate portal (Active Directory / Okta)

Employee authenticates with their company username and password via the corporate identity provider (IdP).

2

IdP generates a SAML assertion

The SAML assertion (XML) is signed by the IdP. Contains user attributes like groups, email, and which AWS role to assume.

3

STS AssumeRoleWithSAML called

The SAML assertion is submitted to STS. STS validates the assertion signature and issues temporary credentials for the AWS role mapped to that user's SAML group.

4

Employee gets AWS Console or API access

Redirected to AWS Console as the assumed role. No AWS password needed — their corporate password is the only one they manage.

🎟️ Temporary Credentials — Everything You Need to Know

Anatomy of STS Temporary Credentials

📦 What STS Returns
  • AccessKeyId: ASIA + 16 random chars (always starts ASIA)
  • SecretAccessKey: 40-char random string
  • SessionToken: long Base64 string (100s of chars)
  • Expiration: 2025-04-04T08:30:00Z (UTC timestamp)
↓
📋 How the SDK uses them
  • Signs every API request with HMAC-SHA256 (SigV4)
  • All three values used in signing — missing one = denied
  • Checks Expiration before each request
  • Auto-refreshes when < 5 min remaining
↓
✅ AWS validates on every call
  • Validates HMAC signature
  • Checks SessionToken is still valid
  • Checks Expiration not passed
  • Checks IAM/role permissions

GetSessionToken — MFA-Protected API Access

🔐 When to use GetSessionToken

When an IAM user needs to use the API or CLI with MFA enforcement. Some sensitive operations (like disabling MFA or deleting resources) can be restricted to only work when the caller used MFA.

Flow: IAM user calls GetSessionToken, provides MFA code → gets temporary credentials that include aws:MultiFactorAuthPresent: true in the session → can now perform MFA-protected operations.

Revoking Temporary Credentials

⚠️ You cannot directly cancel STS credentials

Once issued, STS credentials cannot be revoked individually — they must expire naturally. However, you CAN revoke all active sessions for a role by adding a deny policy with a condition on the issue time. This effectively invalidates all credentials issued before a specific moment.

Emergency revoke all sessions — add to role policy 
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Deny",
    "Action": "*",
    "Resource": "*",
    "Condition": {
      "DateLessThan": {
        "aws:TokenIssueTime": "2025-04-04T10:00:00Z"
      }
    }
  }]
}
// All credentials issued before 10:00 UTC are now denied

STS Service — All Key Operations

StsService.cs 
using Amazon.SecurityToken;
using Amazon.SecurityToken.Model;

public class StsService
{
    private readonly IAmazonSecurityTokenService _sts;

    public StsService(IAmazonSecurityTokenService sts)
        => _sts = sts;

    // ── AssumeRole ────────────────────────────────
    public async Task<Credentials> AssumeRoleAsync(
        string roleArn,
        string sessionName,
        int durationSeconds = 3600)
    {
        var response = await _sts.AssumeRoleAsync(
            new AssumeRoleRequest
        {
            RoleArn         = roleArn,
            RoleSessionName = sessionName,
            DurationSeconds = durationSeconds
            // Optional: Policy = sessionPolicyJson
            // Optional: ExternalId = "secret-token"
        });
        return response.Credentials;
    }

    // ── AssumeRole with ExternalId (3rd party) ────
    public async Task<Credentials> AssumeRoleWithExternalIdAsync(
        string roleArn,
        string sessionName,
        string externalId)
    {
        var response = await _sts.AssumeRoleAsync(
            new AssumeRoleRequest
        {
            RoleArn         = roleArn,
            RoleSessionName = sessionName,
            ExternalId      = externalId,
            DurationSeconds = 3600
        });
        return response.Credentials;
    }

    // ── AssumeRole with Session Policy ───────────
    public async Task<Credentials> AssumeRoleWithSessionPolicyAsync(
        string roleArn,
        string sessionName,
        string sessionPolicyJson)
    {
        var response = await _sts.AssumeRoleAsync(
            new AssumeRoleRequest
        {
            RoleArn         = roleArn,
            RoleSessionName = sessionName,
            Policy          = sessionPolicyJson, // narrows permissions
            DurationSeconds = 3600
        });
        return response.Credentials;
    }

    // ── GetCallerIdentity (Who am I?) ─────────────
    public async Task<GetCallerIdentityResponse>
        GetCallerIdentityAsync()
    {
        return await _sts.GetCallerIdentityAsync(
            new GetCallerIdentityRequest());
    }

    // ── GetSessionToken (MFA-protected access) ────
    public async Task<Credentials> GetSessionTokenAsync(
        string mfaSerialNumber,
        string mfaTokenCode)
    {
        var response = await _sts.GetSessionTokenAsync(
            new GetSessionTokenRequest
        {
            DurationSeconds = 43200, // 12 hours
            SerialNumber    = mfaSerialNumber,
            TokenCode       = mfaTokenCode
        });
        return response.Credentials;
    }
}

Use Assumed Role Credentials with Another AWS Service

Using assumed credentials to access S3 in another account 
using Amazon;
using Amazon.Runtime;
using Amazon.S3;

public async Task<List<string>> ListFilesInOtherAccountAsync()
{
    // Step 1: Assume the cross-account role
    var creds = await _sts.AssumeRoleAsync(
        "arn:aws:iam::444455556666:role/cross-account-s3-role",
        "my-app-session");

    // Step 2: Build credentials object from STS response
    var sessionCreds = new SessionAWSCredentials(
        creds.AccessKeyId,
        creds.SecretAccessKey,
        creds.SessionToken);

    // Step 3: Create S3 client using assumed role's credentials
    var s3 = new AmazonS3Client(sessionCreds,
        RegionEndpoint.APSouth1);

    // Step 4: Now access Account B's S3
    var response = await s3.ListObjectsV2Async(
        new Amazon.S3.Model.ListObjectsV2Request
    {
        BucketName = "account-b-data-bucket"
    });

    return response.S3Objects
        .Select(o => o.Key)
        .ToList();
}

✅ Best Practices & Common Mistakes

General STS Usage

  • Always prefer temporary STS credentials over permanent IAM access keys
  • Use IAM Roles on EC2/ECS/Lambda — SDK fetches STS credentials automatically
  • Set the shortest practical duration — don't default to max 12 hours
  • Use meaningful session names — they appear in CloudTrail for traceability
  • Use AssumeRoleAWSCredentials in SDK for automatic credential refresh
  • Always call GetCallerIdentity in debug/startup to verify current identity

Cross-Account Access

  • Always use External ID when granting access to 3rd-party partners/vendors
  • Scope trust policy to specific role/user ARNs — not entire account root
  • Use session policies to narrow permissions for specific operations
  • Create separate cross-account roles per purpose — not one role for all
  • Enable CloudTrail in the target account to log cross-account activity
  • Require MFA in trust policy for sensitive cross-account roles

Federation & OIDC

  • Use GitHub Actions OIDC — never store AWS keys in GitHub Secrets
  • Restrict OIDC trust policy to specific repo, branch, or environment
  • Use Cognito Identity Pools for mobile/web apps — not direct OIDC
  • Validate OIDC token audience (aud) claim in trust policy conditions
  • Set short duration for web identity sessions (1 hour max)
  • Rotate SAML IdP signing certificates before they expire

Security & Monitoring

  • Enable CloudTrail in ALL regions to capture STS API calls
  • Alert on unusual AssumeRole patterns — new principals, new regions
  • Use aws:SourceIp condition to restrict assumptions from known IPs
  • Use aws:TokenIssueTime deny policy to emergency-revoke all sessions
  • Regularly review trust policies — remove principals that no longer need access
  • Use AWS GuardDuty to detect credential misuse and anomalous activity

Common Mistakes to Avoid

⚠️ Top STS Mistakes

❌ Storing STS temp credentials in files/DB — they expire, use SDK auto-refresh
❌ Not including SessionToken in API calls — all 3 values are always required
❌ Trust policy with Principal: * — anyone in the world can assume your role!
❌ No External ID for 3rd-party access — vulnerable to confused deputy attack
❌ Hardcoding role ARN with account ID in code — use config/environment variables
❌ Maximum duration for all roles — use shortest duration that works
❌ Not logging session names meaningfully — "test123" in CloudTrail is useless
❌ Using GetFederationToken instead of AssumeRole — AssumeRole is preferred
