🏗️ What is AWS VPC?

VPC = Virtual Private Cloud. Your own private isolated network inside AWS. You control everything — IP ranges, subnets, routing, and security.

💡 Real-life analogy

VPC is your private neighborhood inside AWS city. You own the houses (EC2), the streets (subnets), the gates (security groups), and the fence around the whole thing. Only people YOU invite can enter.

Without VPC vs With VPC

❌ Without VPC
  • Servers in AWS shared network
  • Anyone with access can reach them
  • No IP control — AWS assigns
  • No isolation between customers
  • Security relies only on instance firewalls
→
✅ With VPC
  • Your own private network in AWS
  • You define IP ranges (10.0.0.0/16)
  • Full isolation from other customers
  • Multiple security layers
  • Connect to on-premises via VPN

VPC Architecture Overview

🌍 Internet
  • Public users, external services
  • Cannot directly reach private resources
↓
🚪 Internet Gateway (IGW)
  • The only door between internet and your VPC
  • Attached to VPC — one per VPC
  • Only public subnets route traffic through it
↓
🟢 Public Subnet (10.0.1.0/24)
  • ALB, NAT Gateway, Bastion host live here
  • Route table: 0.0.0.0/0 → IGW
  • Instances can have public IPs
↓
🔐 Private Subnet (10.0.11.0/24)
  • App servers, ECS tasks live here
  • Route table: 0.0.0.0/0 → NAT Gateway
  • No public IP — not directly reachable
↓
🗄️ DB Subnet (10.0.21.0/24)
  • RDS, ElastiCache — most restricted
  • No route to internet at all
  • Only accessible from app subnet

🧩 VPC Components — The Building Blocks

A VPC is made of 8 key components working together to create your private network.

1

VPC (Virtual Private Cloud)

The container for your entire private network.
CIDR block: 10.0.0.0/16 = 65,536 IP addresses
Rule: Cannot be changed after creation — plan carefully!
Scope: Spans all AZs in a region

2

Subnet

A division of the VPC into smaller networks. Each subnet lives in ONE availability zone.
10.0.1.0/24 = 256 IPs (251 usable — AWS reserves 5)
Types: Public (internet-facing) or Private (internal only)

3

Internet Gateway (IGW)

The only door between your VPC and the public internet.
Attach: One IGW per VPC
Used by: Public subnets only (via route table)
Cost: Free — you pay for data transfer only

4

Route Table

Rules that tell traffic where to go. Every subnet must be associated with a route table.
Public route: 0.0.0.0/0 → igw-xxxx
Private route: 0.0.0.0/0 → nat-xxxx
Local route (always): 10.0.0.0/16 → local

5

Security Group

Virtual firewall for your EC2 instances and ECS tasks.
Stateful: Return traffic automatically allowed
Default: Deny all inbound, allow all outbound
Rules: Allow only — cannot deny specific IPs

6

Network ACL (NACL)

Firewall at the subnet level — optional extra layer.
Stateless: Must define both inbound AND outbound
Default: Allow all (unlike security groups)
Use for: Blocking specific IPs, DDoS protection

7

NAT Gateway

Lets private instances reach the internet (for updates, APIs) without being reachable from internet.
Placed in: Public subnet
Cost: ~$0.045/hour + data processed

8

VPC Endpoints

Connect to AWS services (S3, DynamoDB, SQS) without going through the internet — faster and cheaper.
Gateway endpoint: S3, DynamoDB (free!)
Interface endpoint: Most other AWS services

📍 Subnets — Dividing Your VPC

Subnets divide your VPC into smaller isolated networks. Each subnet lives in exactly one Availability Zone.

Public vs Private Subnet Flow

🌐 Public Subnet
  • Route to Internet Gateway
  • Instances can get public IPs
  • Direct internet access in/out
  • For: ALB, NAT GW, Bastion
  • Example: 10.0.1.0/24
🔐 Private Subnet
  • Route to NAT Gateway only
  • No public IP assigned
  • Can reach internet via NAT
  • For: App servers, ECS tasks
  • Example: 10.0.11.0/24
🗄️ DB Subnet
  • No route to internet at all
  • Most isolated
  • Only reachable from app subnet
  • For: RDS, ElastiCache, Redis
  • Example: 10.0.21.0/24

Production Multi-AZ Subnet Design

🏗️ VPC: 10.0.0.0/16 — spanning 2 Availability Zones

Public — AZ-1a
  • 10.0.1.0/24 (256 IPs)
  • ALB, NAT GW here
Public — AZ-1b
  • 10.0.2.0/24 (256 IPs)
  • ALB, NAT GW here
↓
App — AZ-1a
  • 10.0.11.0/24
  • ECS tasks, EC2
App — AZ-1b
  • 10.0.12.0/24
  • ECS tasks, EC2
↓
DB — AZ-1a
  • 10.0.21.0/24
  • RDS primary
DB — AZ-1b
  • 10.0.22.0/24
  • RDS standby

💡 CIDR Planning Tip

Start with /16 for the VPC (65k IPs) and /24 for subnets (256 IPs). Leave big gaps between subnets — you can't change the VPC CIDR later! Leave room between 10.0.1.x and 10.0.11.x for future subnets.

🛡️ Security Groups & Network ACLs

AWS VPC has two layers of firewalls. Understanding both is critical for production security.

Two Layers of Protection

🌍 Internet Traffic
  • Incoming request to your app
↓
🚪 Layer 1: Network ACL (Subnet boundary)
  • Checked when traffic enters/leaves subnet
  • Stateless — must define both directions
  • Can ALLOW and DENY
  • Rules evaluated by number (lowest first)
↓
🛡️ Layer 2: Security Group (Instance boundary)
  • Checked when traffic reaches EC2/ECS
  • Stateful — return traffic auto allowed
  • Allow rules only — cannot deny specific IPs
  • Multiple SGs per instance allowed
↓
✅ Your Application Receives Request
  • Only traffic that passed BOTH layers gets through

Security Groups — Deep Dive

🌐 ALB Security Group

  • Inbound: 80 from 0.0.0.0/0
  • Inbound: 443 from 0.0.0.0/0
  • Outbound: All to App SG

⚙️ App Server Security Group

  • Inbound: 5000 from ALB SG only
  • Inbound: 22 from Bastion SG only
  • Outbound: 443 to 0.0.0.0/0 (APIs)
  • Outbound: 5432 to DB SG

🗄️ Database Security Group

  • Inbound: 5432 from App SG ONLY
  • Nothing else inbound — ever!
  • Outbound: None needed

💡 Security Group Chaining

Instead of IP ranges, reference other security groups as source. ALB SG → App SG → DB SG. Much safer than IP-based rules — automatically updates when instances scale.

Security Groups vs Network ACLs

FeatureSecurity GroupNetwork ACL
LevelInstance / ENISubnet boundary
DefaultDeny all inboundAllow all
Stateful✅ Yes — return auto-allowed❌ No — define both ways
Rule typesAllow onlyAllow AND Deny
Rule evaluationAll rules evaluatedLowest number first
Use forMost security rulesBlock IPs, DDoS protection
Multiple per instance✅ Yes (up to 5)One per subnet

⚠️ Never do these

❌ Open port 3306/5432 to 0.0.0.0/0 — database exposed to internet!
❌ Open port 22 (SSH) to 0.0.0.0/0 — brute force attacks guaranteed
❌ Allow all traffic (0.0.0.0/0 all ports) — defeats the whole point

🌐 NAT Gateway — Private Internet Access

NAT (Network Address Translation) Gateway lets private instances reach the internet for outbound traffic — without exposing them to inbound connections from the internet.

How NAT Gateway Works

🔐 Private Instance (10.0.11.50)
  • Needs to call external API or download updates
  • Sends packet to: 8.8.8.8 (Google DNS)
  • Source IP: 10.0.11.50 (private — unreachable!)
↓
🔄 NAT Gateway (in Public Subnet)
  • Intercepts the outbound request
  • Replaces source IP: 10.0.11.50 → 52.1.1.100 (Elastic IP)
  • Records the translation in NAT table
  • Sends packet to internet with public IP
↓
🌍 Internet (8.8.8.8)
  • Sees request from 52.1.1.100 (NAT's public IP)
  • Has no idea 10.0.11.50 exists
  • Responds to 52.1.1.100
↓
🔄 NAT Gateway — Response Return
  • Receives response at 52.1.1.100
  • Looks up NAT table: this belongs to 10.0.11.50
  • Translates back: 52.1.1.100 → 10.0.11.50
  • Forwards to private instance
↓
✅ Private Instance Gets Response
  • Downloaded updates, called external API — success!
  • Internet never knew the private IP existed
  • Internet CANNOT initiate a connection to private instance

NAT Gateway vs NAT Instance

FeatureNAT Gateway ✅NAT Instance (old)
ManagementAWS fully manages itYou manage EC2 instance
AvailabilityHighly available in AZSingle point of failure
PerformanceUp to 45 GbpsLimited by EC2 size
PatchingAWS handlesYou patch the OS
Cost$0.045/hr + $0.045/GBEC2 + EIP cost
Use in production✅ Always❌ Avoid

💰 NAT Gateway Cost Tip

For dev/test environments, you can share one NAT Gateway across all private subnets in the same AZ to save cost. In production, use one NAT Gateway per AZ for high availability. Also use VPC Endpoints for S3/DynamoDB — free and faster than NAT!

💻 C# SDK — Create & Manage VPC Resources

Install NuGet

Install-Package AWSSDK.EC2
Install-Package AWSSDK.Core

Create VPC + Subnet + IGW

VpcService.cs 
using Amazon.EC2;
using Amazon.EC2.Model;

public class VpcService
{
    private readonly IAmazonEC2 _ec2;

    public VpcService(IAmazonEC2 ec2) => _ec2 = ec2;

    // Step 1: Create VPC
    public async Task<string> CreateVpcAsync()
    {
        var vpc = await _ec2.CreateVpcAsync(new CreateVpcRequest
        {
            CidrBlock = "10.0.0.0/16"
        });

        // Tag it with a name
        await _ec2.CreateTagsAsync(new CreateTagsRequest
        {
            Resources = [vpc.Vpc.VpcId],
            Tags = [new Tag { Key = "Name", Value = "production-vpc" }]
        });

        return vpc.Vpc.VpcId;
    }

    // Step 2: Create Subnet
    public async Task<string> CreateSubnetAsync(
        string vpcId, string cidr, string az, string name)
    {
        var subnet = await _ec2.CreateSubnetAsync(new CreateSubnetRequest
        {
            VpcId            = vpcId,
            CidrBlock        = cidr,
            AvailabilityZone = az
        });

        await _ec2.CreateTagsAsync(new CreateTagsRequest
        {
            Resources = [subnet.Subnet.SubnetId],
            Tags = [new Tag { Key = "Name", Value = name }]
        });

        return subnet.Subnet.SubnetId;
    }

    // Step 3: Create and attach Internet Gateway
    public async Task<string> CreateAndAttachIgwAsync(string vpcId)
    {
        var igw = await _ec2.CreateInternetGatewayAsync(
            new CreateInternetGatewayRequest());

        await _ec2.AttachInternetGatewayAsync(new AttachInternetGatewayRequest
        {
            InternetGatewayId = igw.InternetGateway.InternetGatewayId,
            VpcId             = vpcId
        });

        return igw.InternetGateway.InternetGatewayId;
    }
}

Create Security Group with Rules

// Create security group
var sg = await _ec2.CreateSecurityGroupAsync(new CreateSecurityGroupRequest
{
    GroupName   = "app-sg",
    Description = "App server security group",
    VpcId       = vpcId
});

// Add inbound rule: allow port 5000 from ALB SG
await _ec2.AuthorizeSecurityGroupIngressAsync(new
    AuthorizeSecurityGroupIngressRequest
{
    GroupId = sg.GroupId,
    IpPermissions = [new IpPermission
    {
        IpProtocol = "tcp",
        FromPort   = 5000,
        ToPort     = 5000,
        // Reference ALB SG instead of IP — best practice!
        UserIdGroupPairs = [new UserIdGroupPair
        {
            GroupId = albSecurityGroupId
        }]
    }]
});

Describe VPC Resources

// List all subnets in a VPC
var subnets = await _ec2.DescribeSubnetsAsync(new DescribeSubnetsRequest
{
    Filters = [new Filter
    {
        Name   = "vpc-id",

foreach (var subnet in subnets.Subnets)

✅ Best Practices & Common Mistakes

Network Design

  • Plan CIDR blocks before creating — you cannot change VPC CIDR later
  • Use 10.0.0.0/16 for VPC and /24 for subnets — standard and easy
  • Always use at least 2 Availability Zones for high availability
  • Separate subnets for each tier: web, app, database
  • Leave IP gaps between subnets — room for future growth
  • Use descriptive names: public-us-east-1a, app-us-east-1a
  • Enable VPC Flow Logs from day one — invaluable for debugging
  • Use VPC Endpoints for S3 and DynamoDB — free and faster than NAT

Security

  • Default deny — allow only ports that are actually needed
  • Use security group references (not IP ranges) where possible
  • Databases ALWAYS in private subnets — no exceptions
  • Port 22 (SSH) should NEVER be open to 0.0.0.0/0
  • Use Bastion host or AWS Systems Manager Session Manager for SSH access
  • Regularly audit security group rules — remove unused ones
  • Enable GuardDuty for threat detection using VPC flow logs

Cost Optimization

  • Share one NAT Gateway across AZ for dev/test (one per AZ for prod)
  • Use VPC Endpoints for S3/DynamoDB — eliminates NAT data cost
  • Monitor NAT Gateway bandwidth — unexpected spikes cost money
  • Remove unused Elastic IPs — they cost money even when unattached
  • Use VPC Peering instead of internet for cross-VPC traffic
  • Clean up unused subnets, route tables, and ENIs

Common Mistakes to Avoid

⚠️ Top VPC Mistakes

❌ Database in public subnet — database must always be private
❌ DB port open to 0.0.0.0/0 — only app SG should reach the DB
❌ Single AZ deployment — one AZ failure brings down everything
❌ Overlapping CIDR ranges — breaks VPC peering and VPN routing
❌ No NAT for private instances — they can't download updates
❌ Not enabling Flow Logs — blind when diagnosing network issues
❌ Using default VPC in production — poor security defaults, no isolation

Quick Start Checklist — Creating Your First VPC

  • Create VPC with 10.0.0.0/16 CIDR
  • Create 6 subnets (2 public, 2 app private, 2 DB private) across 2 AZs
  • Create Internet Gateway and attach to VPC
  • Create public route table → add route 0.0.0.0/0 → IGW
  • Create NAT Gateway in public subnet with Elastic IP
  • Create private route table → add route 0.0.0.0/0 → NAT GW
  • Associate subnets with correct route tables
  • Create security groups (ALB SG, App SG, DB SG) with proper rules
  • Enable auto-assign public IP on public subnets
  • Enable VPC Flow Logs to CloudWatch
  • Test connectivity end-to-end
