Chapter 01

What is MQTT?

A lightweight, publish/subscribe protocol designed for constrained devices and low-bandwidth, high-latency networks.

📡
Publish / Subscribe
Senders (publishers) don't send directly to receivers. They publish to a topic. Receivers subscribe to topics they care about. Fully decoupled.
🪶
Ultra Lightweight
Minimum 2-byte fixed header. Designed for devices with 256KB RAM, cellular modems, and satellite links with 300ms+ latency.
🔌
Always Connected
Persistent TCP connections with keep-alive heartbeats. Clients stay connected; the broker routes messages instantly.
🏗️
Broker-Centric
A central broker (like Mosquitto or HiveMQ) receives all published messages and routes them to matching subscribers.
⚡
Real-Time
Sub-millisecond delivery on local networks. Used for live sensor data, remote control commands, and telemetry streams.
🌐
IoT Native
Powers billions of devices — smart home sensors, industrial PLCs, wearables, fleet tracking, and more.
Protocol History
v3.1 1999 Original spec by IBM (Andy Stanford-Clark) & Arcom for satellite pipeline monitoring. Low bandwidth focus.
v3.1.1 2014 OASIS standard. Cleaned up specs, improved interoperability. Most widely deployed version today.
v5.0 2019 Major upgrade: message expiry, reason codes, user properties, topic aliases, shared subscriptions, request/response pattern.
💡
Analogy: MQTT is like a radio station. The station (broker) broadcasts on channels (topics). Anyone with a radio (subscriber) can tune in. Listeners don't need to know who's broadcasting — just what channel to tune to.

Chapter 03

Topics & Wildcards

Topics are UTF-8 strings that act as routing keys. They define what a message is about and where it should go.

Topic Anatomy

home / livingroom / sensor / temperature
Level 1
home
Namespace / domain
Level 2
livingroom
Location / group
Level 3
sensor
Device type
Level 4
temperature
Metric / action

Wildcards

+
Single-Level Wildcard
Matches exactly ONE topic level.
home/+/temperature Matches any room's temperature home/+/+ Any room, any sensor type fleet/truck123/+ All metrics for truck123
#
Multi-Level Wildcard
Matches everything from that level down. Must be the LAST character.
home/# Everything under home/ fleet/truck123/# All data for truck123 # ⚠️ ALL topics (use with care!)
⚠️
Best Practice: Topics starting with $ are reserved for system use (e.g., $SYS/broker/clients/connected). Wildcards like # do NOT match $SYS topics. Design a clear naming hierarchy before deploying to production.

Topic Naming Best Practices

✅ Good
factory/line1/motor/rpm
home/bedroom/light/state
user/u123/alerts
❌ Avoid
/leading/slash
topic with spaces
ALLCAPS/TOPICS
💡 Tips
  • Lowercase always
  • No leading/trailing slashes
  • Keep depth ≤ 6 levels
  • Use IDs not names for devices
Chapter 04

Quality of Service (QoS)

QoS defines the guarantee of message delivery between publisher and broker, and broker and subscriber.

0
At Most Once — "Fire and Forget"
Message sent once. No acknowledgment, no retry. May be lost if connection drops. Fastest & lowest overhead.
→ Publisher sends PUBLISH
✗ No ACK expected
✗ No retry on failure
Possible Loss Fastest
Use when: Sensor readings sent every second (missing one is fine)
1
At Least Once — "Acknowledged"
Message delivered at least once. Broker sends PUBACK. Publisher retries until ACK received. May cause duplicates.
→ Publisher sends PUBLISH
← Broker sends PUBACK
↻ Retry if no PUBACK received
Possible Duplicate Balanced
Use when: Alerts, notifications — must arrive, duplicates OK
2
Exactly Once — "4-Way Handshake"
Message delivered exactly once. Uses a 4-step handshake to ensure no loss and no duplicates. Highest overhead.
→ PUBLISH (packetId)
← PUBREC
→ PUBREL
← PUBCOMP
Guaranteed Once Highest Overhead
Use when: Financial transactions, billing events, critical commands

Property QoS 0 QoS 1 QoS 2
Message LossPossibleNoneNone
DuplicatesNonePossibleNone
Network Overhead1 packet2 packets4 packets
LatencyLowestMediumHighest
Stored by brokerNoUntil PUBACKFull handshake
Chapter 05

Key Features

The powerful mechanisms that make MQTT reliable in unreliable environments.

📌 Retained Messages

The broker stores the last retained message for a topic. New subscribers immediately receive it without waiting for the next publish.

📡
Sensor publishes
retain=true
🗄️
Broker stores
last message
⏱️
Time passes...
new client connects
📱
New subscriber
joins topic
⚡
Instantly receives
last known value!
✅
Perfect for: Device status (ON/OFF), last sensor reading, configuration values, firmware version. Clear a retained message by publishing an empty payload to the same topic with retain=true.

☠️ Last Will & Testament (LWT)

When a client connects, it registers a will message. If the client disconnects unexpectedly (crash, network loss), the broker automatically publishes the will on its behalf.

🔌
Client connects
with LWT registered
💚
Normal operation
heartbeats OK
💥
Client crashes
no DISCONNECT sent
⏱️
Broker detects
keepalive timeout
📢
Broker publishes
LWT message!

💓 Keep Alive / PING

How It Works
Client sends PINGREQ packets at regular intervals (keep-alive seconds). Broker replies with PINGRESP. If broker gets no packet for 1.5× the keep-alive interval, it assumes the client is disconnected and fires the LWT.
Typical Values
Mobile devices30–60s
IoT sensors60–300s
Industrial devices120–600s
0 = disabledNo timeout

💾 Persistent Sessions (Clean Session)

cleanSession = true
Every connect starts fresh. No subscriptions remembered. No queued messages. Simple, no state overhead.
Best for: Dashboards, clients that always re-subscribe on connect.
cleanSession = false
Broker remembers subscriptions and queues missed QoS 1/2 messages. On reconnect, queued messages delivered automatically.
Best for: IoT devices that go offline periodically but must receive all messages.
Chapter 06

C# Code Examples

Using MQTTnet — the most popular .NET/C# MQTT library. Install via NuGet: dotnet add package MQTTnet

1. Basic Client — Connect & Subscribe

C# — MQTTnet v4 
using MQTTnet;
using MQTTnet.Client;

var factory = new MqttFactory();
var client = factory.CreateMqttClient();

// 1. Build connection options
var options = new MqttClientOptionsBuilder()
    .WithTcpServer("broker.hivemq.com", 1883)  // host, port
    .WithClientId("my-sensor-01")
    .WithKeepAlivePeriod(TimeSpan.FromSeconds(60))
    .WithCleanSession(true)
    .Build();

// 2. Handle incoming messages
client.ApplicationMessageReceivedAsync += e =>
{
    string topic   = e.ApplicationMessage.Topic;
    string payload = e.ApplicationMessage.ConvertPayloadToString();
    Console.WriteLine($"📩 [{topic}] → {payload}");
    return Task.CompletedTask;
};

// 3. Connect
await client.ConnectAsync(options);
Console.WriteLine("✅ Connected to broker!");

// 4. Subscribe to a topic
var subOptions = new MqttClientSubscribeOptionsBuilder()
    .WithTopicFilter(f => f
        .WithTopic("home/sensor/temperature")
        .WithQualityOfServiceLevel(MqttQualityOfServiceLevel.AtLeastOnce))
    .Build();

await client.SubscribeAsync(subOptions);
Console.WriteLine("👂 Subscribed to home/sensor/temperature");

// Wait for messages...
await Task.Delay(TimeSpan.FromMinutes(5));

// 5. Clean disconnect
await client.DisconnectAsync();

2. Publishing a Message

C# — Publish with QoS & Retain 
// Build the message
var message = new MqttApplicationMessageBuilder()
    .WithTopic("home/sensor/temperature")
    .WithPayload("{ \"temp\": 22.5, \"unit\": \"C\" }")
    .WithQualityOfServiceLevel(MqttQualityOfServiceLevel.AtLeastOnce)  // QoS 1
    .WithRetainFlag(true)  // retain last value
    .Build();

await client.PublishAsync(message);
Console.WriteLine("📤 Published temperature reading");

3. Connecting with Last Will & Testament

C# — Last Will Setup 
var willMessage = new MqttApplicationMessageBuilder()
    .WithTopic("devices/sensor-01/status")
    .WithPayload("offline")
    .WithQualityOfServiceLevel(MqttQualityOfServiceLevel.AtLeastOnce)
    .WithRetainFlag(true)
    .Build();

var options = new MqttClientOptionsBuilder()
    .WithTcpServer("localhost")
    .WithClientId("sensor-01")
    .WithWillMessage(willMessage)  // 👈 register LWT here
    .Build();

await client.ConnectAsync(options);

// Publish "online" status on connect (retained)
await client.PublishAsync(new MqttApplicationMessageBuilder()
    .WithTopic("devices/sensor-01/status")
    .WithPayload("online")
    .WithRetainFlag(true)
    .Build());

    .WithCredentials("myuser", "mypassword")

        {
            // Validate server certificate (production: use proper CA validation)
            return true;
        }))

await client.ConnectAsync(options);
Console.WriteLine("🔐 Secure connection established");

5. Auto-Reconnect with Retry

C# — Managed Client with Auto-Reconnect 
var factory = new MqttFactory();
var managedClient = factory.CreateManagedMqttClient();

var options = new ManagedMqttClientOptionsBuilder()
    .WithAutoReconnectDelay(TimeSpan.FromSeconds(5))
    .WithClientOptions(new MqttClientOptionsBuilder()
        .WithTcpServer("localhost")
        .WithClientId("resilient-client")
        .Build())
    .Build();

// Events fire automatically on reconnect
managedClient.ConnectedAsync += e =>
{
    Console.WriteLine("✅ Reconnected!");
    return Task.CompletedTask;
};

managedClient.DisconnectedAsync += e =>
{
    Console.WriteLine($"❌ Disconnected: {e.Reason}. Retrying...");
    return Task.CompletedTask;
};

await managedClient.StartAsync(options);

// Managed client queues messages when offline and sends on reconnect
await managedClient.EnqueueAsync("sensors/temp", "21.3");

6. Embedded MQTT Broker in C#

C# — Run Broker Inside Your App 
// Great for local IoT gateways, testing, edge devices
var factory = new MqttFactory();
var server = factory.CreateMqttServer();

var serverOptions = new MqttServerOptionsBuilder()
    .WithDefaultEndpoint()              // port 1883
    .WithDefaultEndpointPort(1883)
    .Build();

// Handle client connections
server.ClientConnectedAsync += e =>
{
    Console.WriteLine($"Client connected: {e.ClientId}");
    return Task.CompletedTask;
};

// Intercept messages (logging, routing, validation)
server.InterceptingPublishAsync += e =>
{
    Console.WriteLine($"[{e.ClientId}] → {e.ApplicationMessage.Topic}");
    e.ProcessPublish = true; // allow the message
    return Task.CompletedTask;
};

await server.StartAsync(serverOptions);
Console.WriteLine("🚀 MQTT Broker running on port 1883");
await Task.Delay(Timeout.Infinite); // run forever

7. MQTT 5.0 Features in C#

C# — MQTT 5.0: User Properties, Message Expiry, Response Topic 
// Connect with MQTT v5
var options = new MqttClientOptionsBuilder()
    .WithTcpServer("localhost")
    .WithProtocolVersion(MqttProtocolVersion.V500)  // 👈 MQTT 5.0
    .Build();

// Publish with MQTT 5 features
var message = new MqttApplicationMessageBuilder()
    .WithTopic("commands/start")
    .WithPayload("{ \"action\": \"start\", \"speed\": 50 }")
    .WithMessageExpiryInterval(30)          // expires in 30 seconds
    .WithResponseTopic("commands/response")  // for request/response pattern
    .WithCorrelationData(Guid.NewGuid().ToByteArray())
    .WithUserProperty("source", "control-panel")
    .WithUserProperty("version", "2.1")
    .Build();

await client.PublishAsync(message);

Security

MQTT was designed for constrained networks. Security must be added intentionally — in layers.

🛡️
Layer 4 — Authorization
ACLs: control which clients can pub/sub to which topics
🔑
Layer 3 — Authentication
Username/password, X.509 client certificates, JWT tokens
🔐
Layer 2 — Encryption (TLS/SSL)
All traffic encrypted via TLS 1.2/1.3 on port 8883
🌐
Layer 1 — Network
VPN, private subnets, firewall rules, VPC
Ports at a Glance
1883MQTT (plaintext) — dev only
8883MQTT over TLS — production
9001MQTT over WebSockets
8084MQTT over Secure WebSockets
ACL Example (Mosquitto)
# /etc/mosquitto/acl.conf
user sensor-01
topic write home/sensor/#

user dashboard
topic read home/#

user admin
topic readwrite #
🚨
Never expose port 1883 to the public internet without authentication. Use TLS + strong passwords or client certificates for any production deployment. Default Mosquitto config allows anonymous access — always disable this first.
Chapter 09

Advanced Patterns

Design patterns for real-world MQTT architectures.

🔄 Request / Response (MQTT 5.0)
Publisher sets responseTopic and correlationData in message. Receiver publishes reply to that topic. Enables RPC-style communication over MQTT without a full REST layer.
📊 Bridge Pattern
Connect two separate MQTT brokers (e.g., edge broker → cloud broker). Messages on matching topics are forwarded automatically. Used for OT/IT integration, on-premise to cloud sync, reducing cloud bandwidth.
👥 Shared Subscriptions (MQTT 5.0)
Multiple consumer instances share a single subscription: $share/group/topic. Load balanced across consumers. Great for processing pipelines without all consumers seeing all messages.
🔌 Device Shadow / Digital Twin
Two topics per device: devices/{id}/state (current actual state) and devices/{id}/desired (desired state). Backend publishes to desired; device acts on delta and confirms via state. Used by AWS IoT, Azure IoT Hub.
🏭 MQTT + Time-Series Database
A bridge service subscribes to all sensor topics and writes to InfluxDB/TimescaleDB. Decouples ingestion from storage. MQTT handles real-time delivery; the database handles history & analytics.

Topic Design Pattern: Device Hierarchy

Topic Schema — Industrial IoT 
# Telemetry (device → cloud)
factory/{factoryId}/line/{lineId}/machine/{machineId}/telemetry
factory/plant-01/line/L3/machine/cnc-07/telemetry

# Commands (cloud → device)
factory/{factoryId}/line/{lineId}/machine/{machineId}/command

# Device status (retained)
factory/{factoryId}/line/{lineId}/machine/{machineId}/status

# Alarms (subscribed by ops team)
factory/{factoryId}/alarms/{severity}   # severity = critical/warning/info

# Subscribe to all alarms at any factory:
factory/+/alarms/critical

# Subscribe to everything for one machine:
factory/plant-01/line/L3/machine/cnc-07/#

C# — React to motion and control lights:

client.ApplicationMessageReceivedAsync += async e =>
{
    var topic = e.ApplicationMessage.Topic;
    var payload = e.ApplicationMessage
                    .ConvertPayloadToString();

    if (topic.EndsWith("/motion") && payload == "ON")
    {
        // Turn on the light in same room
        var room = topic.Split('/')[1];
        await client.PublishAsync(
            $"home/{room}/light/set", "ON");
    }
};

Fleet GPS Tracking
Vehicles publish location every 10 seconds to the cloud, even over spotty cellular
C# — Vehicle Tracker with QoS 1 + Persistent Session 
// Vehicle client — must not lose location updates
var options = new MqttClientOptionsBuilder()
    .WithTcpServer("fleet.mycompany.com", 8883)
    .WithClientId($"truck-{TruckId}")
    .WithCleanSession(false)        // persistent: don't lose msgs
    .WithCredentials("fleet-user", "secret")
    .WithWillMessage(new MqttApplicationMessageBuilder()
        .WithTopic($"fleet/truck-{TruckId}/status")
        .WithPayload("offline")
        .WithRetainFlag(true)
        .Build())
    .Build();

await client.ConnectAsync(options);

// Publish GPS every 10 seconds
while (true)
{
    var gps = new { lat = GetLat(), lng = GetLng(), ts = DateTimeOffset.UtcNow };
    await client.PublishAsync(new MqttApplicationMessageBuilder()
        .WithTopic($"fleet/truck-{TruckId}/gps")
        .WithPayload(JsonSerializer.Serialize(gps))
        .WithQualityOfServiceLevel(MqttQualityOfServiceLevel.AtLeastOnce)
        .Build());
    await Task.Delay(10_000);
}

MQTT vs REST vs WebSocket

Choosing the right protocol for the right job.

Feature MQTT REST / HTTP WebSocket gRPC
Pattern Pub/Sub Request/Response Bidirectional RPC/Streaming
Overhead 2 bytes min ~700 bytes headers Medium Low (binary)
IoT Suitability Excellent Poor OK OK
Many-to-Many Native Complex Manual Manual
Offline Tolerance Built-in (QoS/Session) None None None
Low Bandwidth Excellent Wasteful Good Good
Firewall Friendly Needs port 1883/8883 Port 80/443 Port 80/443 Port 443
Ecosystem / Tooling IoT-focused Massive Medium Backend services
💡
Rule of thumb: Use MQTT when devices need to push data frequently, when bandwidth is limited, or when you have many devices that may go offline. Use REST when humans (browsers, mobile apps) are making infrequent requests and need simple integration. Use WebSockets when a web browser needs real-time data — often combined: device → MQTT → backend → WebSocket → browser.
Chapter 12

Production Checklist

Before going live with MQTT in production — make sure you've covered these.

🔒 Security
  • Disable anonymous authentication
  • Use TLS/SSL (port 8883) for all connections
  • Unique clientId per device
  • Strong passwords or X.509 certificates
  • Topic ACLs: least-privilege per client
  • Rate limiting on connections & publishes
📐 Architecture
  • Document topic naming conventions
  • Use retained messages for device state
  • Implement LWT for all critical devices
  • Choose QoS level deliberately per use case
  • Use persistent sessions for offline devices
  • Plan for broker clustering/HA if needed
📦 Client Code
  • Use managed client for auto-reconnect
  • Handle disconnection events gracefully
  • Include error handling on publish/subscribe
  • Don't use # wildcard in production subs
  • Publish "online" retained on connect
  • Keep payload schema documented (JSON schema)
📊 Operations
  • Monitor broker via $SYS topics
  • Set up alerts on client disconnects
  • Log all messages for audit trail
  • Test with simulated network failures
  • Load test: simulate peak device count
  • Document broker version and config
