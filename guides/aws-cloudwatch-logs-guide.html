📋 What is AWS CloudWatch Logs?

AWS CloudWatch Logs is a fully managed log storage, search, and monitoring service. Every AWS service — Lambda, ECS, EC2, API Gateway, RDS, VPC Flow Logs — can stream their logs here. You store them, query them, set alarms on patterns, and route them to other services like S3 or OpenSearch.

💡 CloudWatch Logs in One Sentence

CloudWatch Logs = AWS's centralised logging system — ingest logs from any source, store them durably, search with Logs Insights SQL-like queries, alert on patterns, and export to S3 or OpenSearch.

⚠️ Ingestion cost is the big one

CloudWatch Logs ingestion (~$0.50/GB) adds up fast with verbose logging. Set log levels correctly (INFO in production, DEBUG only in dev), use retention policies, and consider the Infrequent Access log class for logs you rarely query.

🧱 Core Concepts — Log Groups, Streams & Events

🏗️ CloudWatch Logs — Hierarchy

The Three Layers Explained

📁 Log Group

  • Container for log streams from the same source
  • Named by convention: /aws/lambda/FnName
  • Retention policy set at this level
  • KMS encryption key set at this level
  • IAM access controlled per group
  • Metric filters defined per group

📄 Log Stream

  • Sequence of events from one source instance
  • Lambda: one stream per container instance
  • EC2: one stream per instance (via agent)
  • ECS: one stream per task
  • Auto-created by AWS services
  • Events within a stream are ordered by time

📝 Log Event

  • A single log entry — timestamp + message
  • Max size: 256 KB per event
  • Timestamp in milliseconds (epoch)
  • Message: any string (JSON recommended)
  • Events are immutable once ingested
  • Queryable by Logs Insights

Log Classes — Standard vs Infrequent Access

⚡ Standard Log Class

  • Full feature set — all query, filter, subscription features
  • Ingestion: ~$0.50/GB
  • Use for: active production logs, frequently queried
  • Live Tail available

💰 Infrequent Access Log Class

  • 50% cheaper ingestion (~$0.25/GB)
  • No metric filters, no subscriptions, no Live Tail
  • Logs Insights queries still work (at higher per-GB rate)
  • Use for: audit logs, compliance archives, rarely queried logs

Structured Logging — JSON is Best

💡 Always log structured JSON

When your log messages are JSON, Logs Insights can query individual fields. Instead of "User 123 placed order 456", log {"userId":"123","orderId":"456","action":"place_order","duration_ms":45}. This makes your logs searchable by field — filter by userId, aggregate durations, count by action.

🔍 CloudWatch Logs Insights — Query Your Logs

Logs Insights is a fast, SQL-like query engine for your logs. Query across multiple log groups simultaneously, aggregate data, filter fields, and visualise results.

Query Syntax — Key Commands

Essential Query Patterns

logs insights — find all errors in last hour 
fields @timestamp, @message, @logStream
| filter @message like /(?i)error/
| sort @timestamp desc
| limit 100
logs insights — lambda cold starts count over time 
fields @timestamp, @message
| filter @message like /Init Duration/
| stats count(*) as coldStarts by bin(5m)
| sort @timestamp asc
logs insights — lambda p50 p95 p99 duration 
filter @type = "REPORT"
| parse @message "Duration: * ms" as duration
| stats
    avg(duration)                    as avg_ms,
    pct(duration, 50)                as p50_ms,
    pct(duration, 95)                as p95_ms,
    pct(duration, 99)                as p99_ms,
    max(duration)                    as max_ms
by bin(1h)
logs insights — query structured json logs 
fields @timestamp, userId, orderId, action, duration_ms
| filter action = "place_order"
| filter duration_ms > 1000
| stats
    count(*) as slow_orders,
    avg(duration_ms) as avg_duration,
    max(duration_ms) as max_duration
by userId
| sort slow_orders desc
| limit 20
logs insights — extract fields from unstructured log 
fields @timestamp, @message
| parse @message '[*] * * * "*"' as level, method, path, status, userAgent
| filter level = "ERROR"
| stats count(*) as errors by path
| sort errors desc
logs insights — top 10 slowest api endpoints 
fields @timestamp, path, duration_ms, statusCode
| filter ispresent(duration_ms)
| stats
    count(*) as requests,
    avg(duration_ms) as avg_ms,
    pct(duration_ms, 95) as p95_ms
by path
| sort p95_ms desc
| limit 10
logs insights — error rate by service over time 
fields @timestamp, service, level
| filter level in ["ERROR", "FATAL"]
| stats count(*) as errors by service, bin(10m)
| sort @timestamp asc
logs insights — vpc flow logs — rejected traffic 
fields @timestamp, srcAddr, dstAddr, dstPort, action
| filter action = "REJECT"
| stats count(*) as rejectCount by srcAddr, dstPort
| sort rejectCount desc
| limit 20

📊 Metric Filters & Alarms

Metric Filters let you turn log patterns into CloudWatch Metrics — and from there, set alarms. This is how you get paged when your Lambda starts throwing errors.

🔔 Metric Filter → Metric → Alarm → Notification Flow
📋 Log Group receives a log event
  • e.g. {"level":"ERROR","message":"DB connection timeout"}
↓
🔍 Metric Filter scans event
  • Pattern: { $.level = "ERROR" }
  • Match? → Increment custom metric by 1 (or by extracted value)
↓
📊 CloudWatch Custom Metric updated
  • Namespace: MyApp/Errors   Metric: ErrorCount
  • Data point added to the time series
↓
🔔 CloudWatch Alarm evaluates metric
  • Condition: ErrorCount > 5 in 5 minutes
  • State transitions: OK → ALARM → OK
↓
📣 Action triggered
  • SNS topic → Email / PagerDuty / Slack via Lambda
  • Or: Auto Scaling action, EC2 action, SSM document

Creating a Metric Filter — CLI

cli — create metric filter for error count 
aws logs put-metric-filter \
  --log-group-name /aws/lambda/MyFunction \
  --filter-name ErrorCount \
  --filter-pattern '{ $.level = "ERROR" }' \
  --metric-transformations \
    metricName=ErrorCount,\
    metricNamespace=MyApp/Lambda,\
    metricValue=1,\
    defaultValue=0,\
    unit=Count

Metric Filter Patterns — Reference

filter patterns — common examples 
# Simple string match (any message containing ERROR)
ERROR

# Case-insensitive string match
?ERROR ?error

# JSON field match (structured logs)
{ $.level = "ERROR" }

# JSON field not equal
{ $.statusCode != 200 }

# JSON field numeric comparison
{ $.duration_ms > 1000 }

# Multiple conditions (AND)
{ $.level = "ERROR" && $.service = "payment-svc" }

# Multiple conditions (OR)
{ $.statusCode = 500 || $.statusCode = 503 }

# Extract a numeric value for the metric (e.g. record actual duration)
# metricValue = $.duration_ms
[timestamp, requestId, level="ERROR", ...]

Creating a CloudWatch Alarm — CLI

cli — create alarm on the metric filter above 
aws cloudwatch put-metric-alarm \
  --alarm-name "Lambda-ErrorRate-High" \
  --alarm-description "Alert when Lambda errors exceed 5 in 5 minutes" \
  --namespace MyApp/Lambda \
  --metric-name ErrorCount \
  --dimensions Name=FunctionName,Value=MyFunction \
  --statistic Sum \
  --period 300 \
  --evaluation-periods 1 \
  --threshold 5 \
  --comparison-operator GreaterThanOrEqualToThreshold \
  --treat-missing-data notBreaching \
  --alarm-actions arn:aws:sns:eu-west-1:123456789012:ops-alerts \
  --ok-actions    arn:aws:sns:eu-west-1:123456789012:ops-alerts

🔄 Subscription Filters — Real-time Log Streaming

Subscription filters let you stream log events in real time to another service — Lambda, Kinesis Data Streams, or Kinesis Firehose. This is how you build real-time log pipelines.

🔄 Subscription Filter Destinations
⚡ → Lambda
  • Real-time transform & route log events
  • Parse, enrich, and forward to Slack/PagerDuty
  • Max 2 subscriptions per log group
  • Events arrive base64-encoded, gzip-compressed
  • Best for: custom alerts, log enrichment
🚀 → Kinesis Data Streams
  • High-throughput real-time streaming
  • Fan out to multiple consumers
  • Buffer and process in batches
  • Best for: high-volume log pipelines
  • From there: OpenSearch, S3, analytics
🔥 → Kinesis Firehose
  • Buffer & deliver to S3 / OpenSearch / Splunk
  • Built-in retry and backup to S3
  • Transform via Lambda before delivery
  • Best for: log archiving to S3 or OpenSearch

Create a Subscription Filter — Stream to Lambda

cli — create subscription filter (log group → lambda) 
aws logs put-subscription-filter \
  --log-group-name /aws/lambda/MyFunction \
  --filter-name ForwardErrorsToSlack \
  --filter-pattern '{ $.level = "ERROR" }' \
  --destination-arn arn:aws:lambda:eu-west-1:123456789012:function:LogAlerter

# Lambda receives base64-gzip encoded payload
# Structure after decode:
# {
#   "messageType": "DATA_MESSAGE",
#   "owner": "123456789012",
#   "logGroup": "/aws/lambda/MyFunction",
#   "logStream": "2025/03/01/[$LATEST]abc",
#   "subscriptionFilters": ["ForwardErrorsToSlack"],
#   "logEvents": [
#     { "id": "xxx", "timestamp": 1234567890, "message": "..." }
#   ]
# }

Lambda Handler — Decode Subscription Payload (C#)

c# — lambda handler decoding cloudwatch logs subscription 
using System.IO.Compression;
using System.Text.Json;
using Amazon.Lambda.Core;

public class LogAlerterFunction
{
    public async Task HandleAsync(CloudWatchLogsEvent cwEvent, ILambdaContext ctx)
    {
        // Decode: base64 → gzip decompress → JSON parse
        var compressed = Convert.FromBase64String(cwEvent.AwsLogs.Data);
        using var gzip   = new GZipStream(
            new MemoryStream(compressed), CompressionMode.Decompress);
        using var reader = new StreamReader(gzip);
        var json = await reader.ReadToEndAsync();

        var payload = JsonSerializer.Deserialize<CWLogsData>(json);

        foreach (var evt in payload!.LogEvents)
        {
            ctx.Logger.LogLine($"[{payload.LogGroup}] {evt.Message}");
            // Forward to Slack, PagerDuty, SNS, etc.
            await ForwardToSlackAsync(payload.LogGroup, evt.Message);
        }
    }
}

public record CWLogsData(
    string LogGroup,
    string LogStream,
    List<LogEvent> LogEvents);

public record LogEvent(string Id, long Timestamp, string Message);

Stream to Kinesis Firehose → S3

cli — subscription filter to kinesis firehose 
aws logs put-subscription-filter \
  --log-group-name /aws/lambda/MyFunction \
  --filter-name ArchiveAllLogs \
  --filter-pattern "" \
  --destination-arn arn:aws:firehose:eu-west-1:123456789012:deliverystream/LogArchive

# Empty filter-pattern "" means ALL events are forwarded
# Kinesis Firehose buffers and delivers to S3 in Parquet/JSON/text
# From S3 you can run Athena queries on archived logs

⚠️ Max 2 subscription filters per log group

Each log group supports a maximum of 2 subscription filters. If you need to send logs to more than 2 destinations, use a Kinesis Data Stream as the first destination, then fan out from there to multiple consumers.

🗑️ Retention Policies & Cost Optimisation

🚨 Default retention = Never Expire — costs add up!

By default, CloudWatch log groups have no retention policy — logs are stored forever. This is the #1 source of unexpected CloudWatch costs. Always set a retention policy on every log group.

Available Retention Periods

cli — set retention policy on a log group 
# Retention options (days):
# 1, 3, 5, 7, 14, 30, 60, 90, 120, 150, 180, 365,
# 400, 545, 731, 1096, 1827, 2192, 2557, 2922, 3288, 3653

# Set 30-day retention on a Lambda log group
aws logs put-retention-policy \
  --log-group-name /aws/lambda/MyFunction \
  --retention-in-days 30

# Remove retention (back to Never Expire)
aws logs delete-retention-policy \
  --log-group-name /aws/lambda/MyFunction

Set Retention on ALL Log Groups at Once (Bash)

bash — set 90-day retention on all log groups without a policy 
#!/bin/bash
RETENTION_DAYS=90

for GROUP in $(aws logs describe-log-groups \
  --query 'logGroups[?retentionInDays==`null`].logGroupName' \
  --output text); do
  echo "Setting ${RETENTION_DAYS}d retention on: $GROUP"
  aws logs put-retention-policy \
    --log-group-name "$GROUP" \
    --retention-in-days $RETENTION_DAYS
done

echo "Done!"

Cost Optimisation Checklist

  • Set retention policies on every log group — never leave at "Never Expire"
  • Use Infrequent Access log class for audit/compliance logs you rarely query
  • Set log level to INFO (not DEBUG) in production — reduces ingestion volume
  • Use structured JSON logging so Logs Insights queries scan less data
  • Export historical logs to S3 and use Athena for cheaper long-term queries
  • Use --filter-pattern in subscriptions to forward only needed events (not all)
  • Review and delete unused log groups regularly
  • Use Lambda Powertools for .NET — structured logging built-in, avoids verbose output

1. Write Logs via SDK (PutLogEvents)

c# — put log events to cloudwatch 
using Amazon.CloudWatchLogs;
using Amazon.CloudWatchLogs.Model;

public class CloudWatchLogger
{
    private readonly IAmazonCloudWatchLogs _client;
    private readonly string _logGroupName  = "/myapp/application";
    private readonly string _logStreamName = "api-server-1";
    private string? _sequenceToken;

    public CloudWatchLogger()
    {
        _client = new AmazonCloudWatchLogsClient(
            Amazon.RegionEndpoint.EUWest1);
    }

    public async Task EnsureLogGroupAsync()
    {
        try
        {
            await _client.CreateLogGroupAsync(
                new CreateLogGroupRequest(_logGroupName));
            await _client.PutRetentionPolicyAsync(
                new PutRetentionPolicyRequest(_logGroupName, 30));
        }
        catch (ResourceAlreadyExistsException) { }

        try
        {
            await _client.CreateLogStreamAsync(
                new CreateLogStreamRequest(_logGroupName, _logStreamName));
        }
        catch (ResourceAlreadyExistsException) { }
    }

    public async Task LogAsync(string level, string message, object? extra = null)
    {
        var payload = JsonSerializer.Serialize(new {
            timestamp = DateTime.UtcNow,
            level,
            message,
            extra
        });

        var request = new PutLogEventsRequest
        {
            LogGroupName  = _logGroupName,
            LogStreamName = _logStreamName,
            SequenceToken = _sequenceToken,
            LogEvents = new List<InputLogEvent>
            {
                new() {
                    Timestamp = DateTime.UtcNow,
                    Message   = payload
                }
            }
        };

        var response = await _client.PutLogEventsAsync(request);
        _sequenceToken = response.NextSequenceToken;
    }
}

2. Query Logs with Logs Insights from C#

c# — run logs insights query programmatically 
public async Task<List<Dictionary<string,string>>> QueryLogsAsync(
    string logGroupName,
    string queryString,
    DateTime startTime,
    DateTime endTime)
{
    using var client = new AmazonCloudWatchLogsClient();

    // Start the query
    var startResp = await client.StartQueryAsync(new StartQueryRequest
    {
        LogGroupName = logGroupName,
        QueryString  = queryString,
        StartTime    = ToEpochSeconds(startTime),
        EndTime      = ToEpochSeconds(endTime),
        Limit        = 100
    });

    // Poll until complete
    GetQueryResultsResponse results;
    do
    {
        await Task.Delay(500);
        results = await client.GetQueryResultsAsync(
            new GetQueryResultsRequest { QueryId = startResp.QueryId });
    }
    while (results.Status == QueryStatus.Running
        || results.Status == QueryStatus.Scheduled);

    // Map results to dictionaries
    return results.Results
        .Select(row => row.ToDictionary(f => f.Field, f => f.Value))
        .ToList();
}

// Usage:
var rows = await QueryLogsAsync(
    "/aws/lambda/MyFunction",
    @"fields @timestamp, @message
      | filter @message like /ERROR/
      | sort @timestamp desc
      | limit 20",
    DateTime.UtcNow.AddHours(-1),
    DateTime.UtcNow
);

# Lambda duration stats (p50/p95/p99)
filter @type = "REPORT"
| parse @message "Duration: * ms" as dur
| stats avg(dur) as avg, pct(dur,95) as p95, pct(dur,99) as p99, max(dur) as max by bin(1h)

# Lambda cold starts
filter @message like /Init Duration/
| stats count(*) as coldStarts by bin(5m)

# Top 10 slowest endpoints
fields path, duration_ms
| filter ispresent(duration_ms)
| stats avg(duration_ms) as avg, pct(duration_ms,95) as p95 by path
| sort p95 desc | limit 10

# Error count by service
fields service, level
| filter level = "ERROR"
| stats count(*) as errors by service
| sort errors desc

Common Mistakes

  • No retention policy set — logs stored forever, costs spiral
  • Logging at DEBUG level in production — massive ingestion cost
  • Unstructured string logs — can't query fields in Logs Insights
  • Running Logs Insights on large time windows without filters — expensive scan
  • Using subscription filter with empty pattern on high-volume groups — Lambda throttling
  • Forgetting to give Lambda/ECS role logs:PutLogEvents permission
  • Using Export Task for continuous archiving — only one runs at a time, use Firehose instead

Production Checklist

  • Retention policy set on every log group (30 days for app logs, 365 for audit)
  • Structured JSON logging enabled — every log field queryable
  • Metric filter for ERROR count → alarm → SNS notification
  • Lambda timeout metric filter and alarm configured
  • Log level set to INFO (not DEBUG) in production
  • Subscription filter → Firehose → S3 for long-term compliance logs
  • IAM policies restricting CloudWatch log access by team/environment
  • Infrequent Access class set for rarely-queried log groups
