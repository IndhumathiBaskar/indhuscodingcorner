🚀 What is AWS CodeDeploy?

AWS CodeDeploy is a fully managed deployment service that automates software deployments to EC2 instances, on-premises servers, Lambda functions, and ECS services. It handles the coordination of rolling updates, blue/green traffic switching, lifecycle hooks, health checks, and automatic rollback — without you writing any orchestration code.

💡 The Core Mental Model

CodeDeploy is the last mile of your CI/CD pipeline. CodeBuild compiles and packages your app. CodeDeploy takes that package and safely delivers it to your compute targets — tracking health at each step and rolling back automatically if something goes wrong.

🎯 Deployment Types & Configurations

⚡ In-Place (Rolling)
Deploys new version on existing instances one-by-one (or in batches). Instance is taken out of LB, updated, re-added.
EC2 · On-Premises
🔵🟢 Blue/Green (EC2)
New instances provisioned with new version. Traffic switched after health check. Old instances kept as fallback, then terminated.
EC2 · Auto Scaling
🐦 Lambda Canary
Shifts N% of Lambda traffic to new version for T minutes, then shifts 100% if no alarms fire. Auto-rollback if alarm triggers.
Lambda Functions
📊 Lambda Linear
Gradually shifts traffic +N% every T minutes until 100%. Slow, safe rollout ideal for critical Lambda functions.
Lambda Functions
🐳 ECS Blue/Green
New ECS task definition deployed alongside old. ALB/NLB listener rules shifted. Old tasks drained then stopped.

⚡ AllAtOnce
Deploy to all targets simultaneously. Fastest but highest risk — downtime if deploy fails on all instances.

📄 appspec.yml — Deployment Blueprint

The appspec.yml file is the heart of every CodeDeploy deployment. It lives in the root of your revision bundle and tells CodeDeploy what to copy, where to put it, and which scripts to run at each lifecycle hook.

🧬 appspec.yml Top-Level Sections
version
Always 0.0 — the only valid value.
Required
os
Target OS — linux or windows. EC2/on-prem only.
EC2 only
files
Source → destination file mappings. Copy your compiled .NET app to /opt/watson/.
EC2 only
permissions
Set file ownership and permissions on deployed files.
Optional
hooks
Scripts to run at lifecycle events (BeforeInstall, AfterInstall, ValidateService, etc.).
Optional
resources
Lambda: specify function name + aliases. ECS: task definition + container + port.
Lambda/ECS

🔗 Lifecycle Hooks — Full Reference

📋 All Lifecycle Hooks — EC2 / On-Premises
ApplicationStop
Gracefully stop the currently running application. Run before downloading the new revision. Skipped on first deployment.
EC2
DownloadBundle
Agent downloads the revision from S3/GitHub. Not scriptable — agent handles this automatically.
Agent only
BeforeInstall
Run before files are copied. Ideal for: create backup of config files, set up directories, decrypt files.
EC2

Agent copies files per the files section in appspec.yml. Not scriptable.
Agent only
AfterInstall
Runs after files are copied. Ideal for: change file permissions, configure the application, run database migrations.
EC2
ApplicationStart
Start the application services. Run your systemctl start or service start commands here.
EC2
ValidateService
Health check the new deployment. HTTP probe, smoke test, or any validation. Failure causes rollback.
EC2
BeforeBlockTraffic
Runs before instance is removed from LB. Allows graceful request draining.
EC2 + LB

Agent removes instance from load balancer. Not scriptable.
Agent only
AfterBlockTraffic
Runs after instance removed from LB. Good for flushing caches or draining queues.
EC2 + LB

Agent only
BeforeAllowTraffic

Lambda · ECS
AfterAllowTraffic

Hook Scripts — Bash Examples for .NET

bash · scripts/stop_application.sh 
#!/bin/bash
# Stop the .NET application gracefully

# Check if service exists before stopping
if systemctl is-active --quiet watson-api; then
    echo "Stopping watson-api service..."
    systemctl stop watson-api
    
    # Wait up to 30s for graceful shutdown
    for i in {1..30}; do
        if ! systemctl is-active --quiet watson-api; then
            echo "Service stopped successfully"
            exit 0
        fi
        sleep 1
    done
    echo "Service did not stop cleanly, forcing..."
    systemctl kill watson-api
else
    echo "Service not running — skipping stop"
fi
exit 0
bash · scripts/validate_service.sh 
#!/bin/bash
# Validate .NET API is healthy after deployment

APP_URL="http://localhost:8080"
MAX_RETRIES=12
SLEEP_INTERVAL=5

echo "Waiting for watson-api to become healthy..."

for i in $(seq 1 $MAX_RETRIES); do
    HTTP_STATUS=$(curl -s -o /dev/null -w "%{http_code}" \
        --max-time 5 \
        "${APP_URL}/health")

    if [ "$HTTP_STATUS" = "200" ]; then
        echo "✅ Health check passed (HTTP 200) on attempt $i"
        exit 0
    fi

    echo "Attempt $i/$MAX_RETRIES: got $HTTP_STATUS — retrying in ${SLEEP_INTERVAL}s"
    sleep $SLEEP_INTERVAL
done

echo "❌ Health check FAILED after $MAX_RETRIES attempts — triggering rollback"
exit 1

⚡ Lambda Deployments — Traffic Shifting

Lambda Alias Traffic Shifting — How It Works

📌 The Alias Pattern

  • Lambda functions publish numbered versions (v1, v2, v3...)
  • An alias (e.g., live) points to a version
  • CodeDeploy shifts traffic between two versions via alias weights
  • Your application always calls the alias — never a version ARN directly
  • SAM's AutoPublishAlias handles version publishing automatically

🔄 Traffic Shift Flow

  • Before deploy: alias live → 100% v5
  • Deploy starts: alias live → 90% v5 + 10% v6
  • If canary OK: alias live → 100% v6
  • If alarm fires: alias live → 100% v5 (rollback)
  • Hook Lambdas validate before & after the shift

Pre/Post Traffic Hook Lambda — C#

C# · PreTrafficHook Lambda Function using Amazon.CodeDeploy; using Amazon.CodeDeploy.Model; using Amazon.Lambda.Core; public class PreTrafficHookFunction { private readonly IAmazonCodeDeploy _codeDeploy; private readonly HttpClient _http; public PreTrafficHookFunction() { _codeDeploy = new AmazonCodeDeployClient(); _http = new HttpClient(); } public async Task HandleAsync( IDictionary<string, string> input, ILambdaContext context) { var hookStatus = "Succeeded"; var deploymentId = input["DeploymentId"]; var hookId = input["LifecycleEventHookExecutionId"]; try { context.Logger.LogLine($"Running pre-traffic validation for {deploymentId}"); // Test the NEW version of the Lambda directly var newVersionArn = System.Environment.GetEnvironmentVariable( "NEW_VERSION_ARN")!; var response = await _http.GetAsync( $"https://api.watson.internal/health?target={newVersionArn}"); if (!response.IsSuccessStatusCode) { context.Logger.LogLine($"❌ Pre-traffic check FAILED: {response.StatusCode}"); hookStatus = "Failed"; } else { context.Logger.LogLine("✅ Pre-traffic validation passed"); } } catch (Exception ex) { context.Logger.LogLine($"❌ Exception in pre-traffic hook: {ex.Message}"); hookStatus = "Failed"; } finally { // MUST call this — otherwise deployment hangs until timeout await _codeDeploy.PutLifecycleEventHookExecutionStatusAsync( new PutLifecycleEventHookExecutionStatusRequest { DeploymentId = deploymentId, LifecycleEventHookExecutionId = hookId, Status = hookStatus == "Succeeded" ? LifecycleEventStatus.Succeeded : LifecycleEventStatus.Failed }); } } }

🐳 ECS Blue/Green Deployments

ECS Blue/Green with CodeDeploy creates a second set of tasks (the "green" environment), validates them, then shifts ALB/NLB traffic. The blue environment is kept for a configurable period before being terminated.

ECS Blue/Green Architecture

🔵 Blue Environment (Current)

  • Old task definition (revision 41)
  • Currently receiving 100% of production traffic
  • Still running during green deployment
  • Kept alive for rollback window (default 1 hour)
  • Terminated after termination wait time

🟢 Green Environment (New)

  • New task definition (revision 42)
  • Spun up in parallel, zero initial traffic
  • Gets test listener traffic for validation
  • Traffic shifted via ALB weighted target groups
  • Becomes "blue" after successful full shift

Best Practices & Common Mistakes

  • Always add at least one CloudWatch alarm to AlarmConfiguration — error rate, latency p99, and throttle count are the essentials
  • Set ValidateService hook with a real HTTP health check — this is your last line of defence before the deployment is marked successful
  • Use OneAtATime for production EC2 in-place deploys — slower but zero-downtime with an ALB in front
  • Tag EC2 instances with Environment: prod and use that tag in the deployment group — avoids targeting wrong instances
  • Use Canary10Percent5Minutes for Lambda prod deploys — catches regressions in under 5 minutes with minimal blast radius
  • Never use AllAtOnce on production EC2 — the entire fleet is taken offline simultaneously if deployment fails
  • Keep hook scripts idempotent — they may be re-run if a deployment is retried after a partial failure
  • Set reasonable timeouts on hooks — the default is 3600s (1 hour); a hung script costs a full hour before rollback triggers

⚠️ Top CodeDeploy Mistakes

❌ No ValidateService hook — CodeDeploy declares success even if the app is broken, since it only checks script exit codes
❌ AllAtOnce in production — if deploy fails, 100% of instances are down simultaneously
❌ Hook scripts not idempotent — failing midway and retrying may cause double-stop, double-start side effects
❌ No CloudWatch alarms — deployment succeeds even when error rate spikes; auto-rollback never triggers
❌ Forgetting PutLifecycleEventHookExecutionStatus in Lambda hooks — deployment hangs for the full timeout (1 hour default)
❌ Agent not installed on new EC2 instances — new Auto Scaling instances must have the agent pre-installed via Launch Template User Data
❌ EC2 instance role missing CodeDeploy policy — agent can't download revision from S3, deployment fails immediately
