🌍 Context API — Avoiding Prop Drilling

Share data across many components without manually passing props through every level in between.

The problem Context solves If App has the current user, and a deeply nested Avatar component needs it, passing it as a prop through every component in between ("prop drilling") is tedious and brittle. Context lets Avatar read it directly.
TSX context/AuthContext.tsx — typed context with a Provider 
import { createContext, useContext, useState } from 'react';
import type { ReactNode } from 'react';

type User = { id: number; name: string };

type AuthContextType = {
  user: User | null;
  login: (user: User) => void;
  logout: () => void;
};

// undefined default forces consumers to check they're inside a Provider
const AuthContext = createContext<AuthContextType | undefined>(undefined);

export function AuthProvider({ children }: { children: ReactNode }) {
  const [user, setUser] = useState<User | null>(null);

  const login = (newUser: User) => setUser(newUser);
  const logout = () => setUser(null);

  return (
    <AuthContext.Provider value={{ user, login, logout }}>
      {children}
    </AuthContext.Provider>
  );
}

// Custom hook — the ONLY way components should access this context
export function useAuth() {
  const context = useContext(AuthContext);
  if (!context) {
    throw new Error('useAuth must be used inside an AuthProvider');
  }
  return context;
}
TSX Wrap the app once — App.tsx 
import { AuthProvider } from './context/AuthContext';

function App() {
  return (
    <AuthProvider>
      <Header />
      <MainContent />   {/* Avatar is nested deep inside here */}
    </AuthProvider>
  );
}
TSX Any nested component can now read it directly — no props needed 
import { useAuth } from '../../context/AuthContext';

function Avatar() {
  const { user, logout } = useAuth();   // no props passed in — reads context directly

  if (!user) return null;

  return (
    <div>
      <span>{user.name}</span>
      <button onClick={logout}>Log out</button>
    </div>
  );
}
⚠️ Don't put everything in one giant Context Any component reading a Context re-renders when any value in it changes. Split unrelated state into separate Contexts (AuthContext, ThemeContext) so a theme change doesn't re-render every component reading auth state.

🪝 Custom Hooks — Reusable Logic

Extract repeated stateful logic into your own hook — the single biggest lever for clean React code.

The Rule If you find yourself writing the same useState + useEffect pattern in two or more components, that's your signal to extract it into a custom hook. A custom hook is just a function whose name starts with use and that calls other hooks inside it.
TSX hooks/useFetch.ts — a generic, reusable data-fetching hook 
import { useState, useEffect } from 'react';

type FetchState<T> = {
  data: T | null;
  loading: boolean;
  error: string | null;
};

// Generic  — works for any data shape: useFetch, useFetch
export function useFetch<T>(url: string): FetchState<T> {
  const [state, setState] = useState<FetchState<T>>({
    data: null,
    loading: true,
    error: null,
  });

  useEffect(() => {
    let isCancelled = false;
    setState({ data: null, loading: true, error: null });

    fetch(url)
      .then(res => {
        if (!res.ok) throw new Error(`HTTP ${res.status}`);
        return res.json();
      })
      .then((data: T) => {
        if (!isCancelled) setState({ data, loading: false, error: null });
      })
      .catch((err: Error) => {
        if (!isCancelled) setState({ data: null, loading: false, error: err.message });
      });

    return () => { isCancelled = true; };
  }, [url]);

  return state;
}
TSX Using useFetch — clean, typed, no duplicate logic 
import { useFetch } from '../../hooks/useFetch';

type Post = { id: number; title: string };

function PostList() {
  // TypeScript knows `data` is Post[] | null — full autocomplete
  const { data: posts, loading, error } = useFetch<Post[]>('/api/posts');

  if (loading) return <p>Loading...</p>;
  if (error) return <p>Error: {error}</p>;

  return (
    <ul>
      {posts?.map(post => <li key={post.id}>{post.title}</li>)}
    </ul>
  );
}

Another Useful Custom Hook — Debounce

TSX hooks/useDebounce.ts — delay reacting to fast-changing values (e.g. search input) 
import { useState, useEffect } from 'react';

export function useDebounce<T>(value: T, delayMs: number): T {
  const [debounced, setDebounced] = useState(value);

  useEffect(() => {
    const timer = setTimeout(() => setDebounced(value), delayMs);
    return () => clearTimeout(timer);   // cancel if value changes before delay finishes
  }, [value, delayMs]);

  return debounced;
}

🛣️ React Router — Multi-Page Navigation

React apps are usually "single page" — React Router makes different URLs render different components.

Terminal Install React Router 
npm install react-router-dom
TSX router/routes.tsx — centralised route configuration 
import { createBrowserRouter } from 'react-router-dom';
import Home from '../pages/Home/Home';
import Profile from '../pages/Profile/Profile';
import Login from '../pages/Login/Login';
import NotFound from '../pages/NotFound/NotFound';
import ProtectedRoute from '../components/ProtectedRoute/ProtectedRoute';

export const router = createBrowserRouter([
  { path: '/', element: <Home /> },
  { path: '/login', element: <Login /> },
  {
    path: '/profile/:userId',        // :userId is a dynamic route param
    element: (
      <ProtectedRoute>
        <Profile />
      </ProtectedRoute>
    ),
  },
  { path: '*', element: <NotFound /> },   // catch-all 404
]);

Reading Route Params & Navigating

TSX pages/Profile/Profile.tsx — useParams + useNavigate 
import { useParams, useNavigate, Link } from 'react-router-dom';

function Profile() {
  // Typed param extraction — matches ":userId" from the route path
  const { userId } = useParams<{ userId: string }>();
  const navigate = useNavigate();

  const goHome = () => navigate('/');   // programmatic navigation

  return (
    <div>
      <h2>Viewing profile for user: {userId}</h2>
      <Link to="/">Back to Home</Link>   {/* declarative nav */}
      <button onClick={goHome}>Or click here</button>
    </div>
  );
}
TSX components/ProtectedRoute/ProtectedRoute.tsx — guard authenticated routes 
import type { ReactNode } from 'react';
import { Navigate } from 'react-router-dom';
import { useAuth } from '../../context/AuthContext';

function ProtectedRoute({ children }: { children: ReactNode }) {
  const { user } = useAuth();

  if (!user) {
    return <Navigate to="/login" replace />;
  }

  return <>{children}</>;
}

export default ProtectedRoute;

📌 useRef — Direct DOM Access & Persistent Values

For when you need to reach the actual DOM node, or store a value that survives re-renders without triggering one.

useRef vs useState Changing a ref's value does not trigger a re-render. Use it for things React doesn't need to "watch" — DOM nodes, previous values, timer IDs, anything that's an implementation detail rather than something the UI displays.
TSX Focus an input on mount — typed DOM ref 
import { useRef, useEffect } from 'react';

function SearchBox() {
  // Typed as HTMLInputElement | null — starts null until React attaches it
  const inputRef = useRef<HTMLInputElement>(null);

  useEffect(() => {
    inputRef.current?.focus();   // ?. because it could still be null
  }, []);

  return <input ref={inputRef} type="text" placeholder="Search..." />;
}
TSX Store a value across renders WITHOUT causing a re-render 
import { useRef, useEffect, useState } from 'react';

function Stopwatch() {
  const [seconds, setSeconds] = useState(0);
  // Holds the interval ID — doesn't need to trigger re-renders when it changes
  const intervalRef = useRef<number | undefined>(undefined);

  const start = () => {
    intervalRef.current = setInterval(() => setSeconds(s => s + 1), 1000);
  };

  const stop = () => clearInterval(intervalRef.current);

  useEffect(() => stop, []);  // always clean up on unmount

  return (
    <div>
      <p>{seconds}s</p>
      <button onClick={start}>Start</button>
      <button onClick={stop}>Stop</button>
    </div>
  );
}
NeedUseTriggers Re-render?
Data shown on screenuseState✅ Yes
Direct DOM node access (focus, scroll, measure)useRef❌ No
Store a timer/interval IDuseRef❌ No
Track "previous value" of a prop/stateuseRef + useEffect❌ No

⚡ Performance Basics

useMemo, useCallback, and React.memo — preventing unnecessary work, used correctly (and not too early).

⚠️ Don't optimise prematurely These tools add complexity. Only reach for them when you've actually noticed a performance problem (e.g. via React DevTools Profiler) — not as a default habit on every component.
useMemo
Caches the result of an expensive calculation. Only recalculates when its dependencies change.
useCallback
Caches a function itself, so it keeps the same reference between renders — useful when passing callbacks to memoized children.
React.memo
Wraps a component so it skips re-rendering if its props haven't changed.
TSX useMemo — avoid recalculating an expensive filter on every render 
import { useMemo, useState } from 'react';

type Product = { id: number; name: string; price: number };

function ProductList({ products }: { products: Product[] }) {
  const [search, setSearch] = useState('');

  // Only re-filters when `products` or `search` actually change —
  // not on every render caused by unrelated state elsewhere
  const filtered = useMemo(
    () => products.filter(p => p.name.toLowerCase().includes(search.toLowerCase())),
    [products, search]
  );

  return (
    <div>
      <input value={search} onChange={(e) => setSearch(e.target.value)} />
      <ul>{filtered.map(p => <li key={p.id}>{p.name}</li>)}</ul>
    </div>
  );
}
TSX useCallback + React.memo working together 
import { useCallback, useState, memo } from 'react';

// Without memo, ExpensiveChild re-renders every time Parent re-renders,
// even if its own props never changed.
const ExpensiveChild = memo(function ExpensiveChild({ onSave }: { onSave: () => void }) {
  console.log('ExpensiveChild rendered');
  return <button onClick={onSave}>Save</button>;
});

function Parent() {
  const [count, setCount] = useState(0);

  // Without useCallback, a NEW function is created every render,
  // which would defeat memo() — the child sees a "different" prop each time
  const handleSave = useCallback(() => {
    console.log('Saving...');
  }, []);  // no dependencies — this function never needs to change

  return (
    <div>
      <p>Count: {count}</p>
      <button onClick={() => setCount(c => c + 1)}>Increment</button>
      <ExpensiveChild onSave={handleSave} />   {/* won't re-render on Increment clicks */}
    </div>
  );
}
ToolMemoizesUse When
useMemoA computed valueExpensive calculation (filtering, sorting large lists)
useCallbackA function referencePassing callbacks to memoized children, or as a dependency elsewhere
React.memoAn entire component's render outputComponent re-renders often with the same props, and rendering is costly

🏷️ Advanced TypeScript Patterns

Patterns that make components more flexible and your code more self-documenting.

TSX Generic component — works with any list item type 
// A reusable List component that works for ANY item type
type ListProps<T> = {
  items: T[];
  renderItem: (item: T) => React.ReactNode;
  keyExtractor: (item: T) => string | number;
};

function List<T>({ items, renderItem, keyExtractor }: ListProps<T>) {
  return (
    <ul>
      {items.map(item => (
        <li key={keyExtractor(item)}>{renderItem(item)}</li>
      ))}
    </ul>
  );
}

// Usage with Users:
//  u.id} renderItem={u => {u.name}} />

// Usage with Products — SAME component, different type, fully typed both times:
//  p.id} renderItem={p => {p.price}} />
TSX Discriminated unions — modelling state that can be one of several distinct shapes 
// Each variant has a "status" field that TypeScript uses to narrow the type
type RequestState<T> =
  | { status: 'idle' }
  | { status: 'loading' }
  | { status: 'success'; data: T }
  | { status: 'error'; message: string };

function renderRequest<T>(state: RequestState<T>) {
  switch (state.status) {
    case 'idle':
      return <p>Waiting to start...</p>;
    case 'loading':
      return <p>Loading...</p>;
    case 'success':
      // TypeScript KNOWS state.data exists here — only on the success branch
      return <pre>{JSON.stringify(state.data)}</pre>;
    case 'error':
      // And here, state.message exists — TypeScript narrows per-branch
      return <p>Error: {state.message}</p>;
  }
}
TSX Utility types — Partial, Pick, Omit 
type User = {
  id: number;
  name: string;
  email: string;
  role: 'admin' | 'viewer';
};

// Partial — makes every field optional. Great for "update" functions.
function updateUser(id: number, changes: Partial<User>) { /* ... */ }
updateUser(1, { name: 'New Name' });   // only "name" needed, rest optional

// Pick — select only certain fields
type UserPreview = Pick<User, 'id' | 'name'>;
// = { id: number; name: string }

// Omit — everything EXCEPT certain fields
type UserWithoutEmail = Omit<User, 'email'>;
// = { id: number; name: string; role: 'admin' | 'viewer' }
💡 Why discriminated unions beat booleans Instead of { loading: boolean; error: string | null; data: T | null } (which allows invalid states like loading=true AND data set), a discriminated union makes invalid states impossible to represent — you can only ever be in exactly one of the defined states.

💡 Best Practices

Patterns that keep an intermediate-sized React + TypeScript codebase maintainable.

✅ One Context per Concern

Separate AuthContext, ThemeContext, CartContext rather than one giant AppContext. Limits unnecessary re-renders and keeps each context easy to reason about.

✅ Always Wrap Context Access in a Hook

Never call useContext(AuthContext) directly in components. Wrap it in useAuth() so you get the undefined-check and a single place to evolve the API.

✅ Extract Hooks Once You Repeat Yourself Twice

The moment you copy a useState+useEffect pattern into a second component, stop and make it a custom hook in hooks/.

✅ Centralise Routes in One File

Keep all route definitions in router/routes.tsx. Don't scatter <Route> elements across multiple files — one source of truth for your app's URL structure.

✅ Profile Before Optimising

Use React DevTools Profiler to find actual slow renders before reaching for useMemo/useCallback/memo. Premature optimisation adds complexity for no measurable benefit.

✅ Model State with Unions, Not Booleans

Prefer { status: 'loading' | 'success' | 'error' } over multiple boolean flags. It makes impossible states impossible and the code self-documenting.

✅ Type Function Parameters, Let Return Types Infer

Always type what goes into a function. Let TypeScript infer the return type unless the function is complex enough that an explicit return type improves readability.

✅ Name Custom Hooks Starting with "use"

Required by React's rules of hooks (so the linter can verify correct hook usage) and instantly communicates intent to other developers reading the code.
