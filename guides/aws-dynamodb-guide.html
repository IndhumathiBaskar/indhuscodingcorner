⚡ What is Amazon DynamoDB?

DynamoDB is AWS's fully managed, serverless NoSQL database. No servers to provision, no OS to patch, no replication to configure. It stores data as items (rows) in tables, scales from zero to millions of requests per second automatically, and delivers single-digit millisecond performance at any scale.

💡 Real-life analogy — City Filing System

DynamoDB is like a city's filing system. The city is divided into postal districts (partitions) based on address (partition key). Within each district, files are sorted chronologically (sort key). To find all files for a resident — you go to their district and scan their sorted drawer. Cross-reference indexes (GSI) let you find files by phone number instead of address. Each district clerk (partition) operates independently — the city handles millions of requests simultaneously with no bottlenecks.

DynamoDB vs RDS — The Right Tool

⚡ Choose DynamoDB when

  • Need single-digit ms latency at any scale
  • Access patterns are known and simple
  • Traffic is unpredictable / spiky
  • No complex joins or ad-hoc queries needed
  • Serverless / event-driven architecture
  • Key-value lookups dominate workload
  • Gaming, IoT, sessions, user profiles, carts

🗄️ Choose RDS / PostgreSQL when

  • Complex reporting and analytics queries
  • Many-to-many relationships with joins
  • Ad-hoc queries that aren't predefined
  • ACID transactions across many tables
  • Strong SQL familiarity in team
  • Financial ledgers with complex constraints
  • OLAP workloads — use Redshift instead

🔑 Keys & Partitions — The Foundation

DynamoDB's performance and data model are entirely determined by how you design your keys. Get this right and everything else follows.

The Three Key Configurations

🔵 Simple Key (PK only)

  • One attribute acts as the unique identifier
  • Each item must have a unique PK value
  • Like a dictionary — key → single value
  • Good for: sessions, config, user-to-profile maps
  • Example: userId = "usr_001"
  • Cannot store multiple items per PK

🟣 Composite Key (PK + SK)

  • Two attributes together form the unique key
  • Multiple items can share PK — differentiated by SK
  • Like a file cabinet — drawer (PK) has sorted files (SK)
  • Query all items with same PK efficiently
  • Good for: 99% of production use cases
  • Example: PK=USER#001 SK=ORDER#2025-04-01

Sort Key Superpowers

🔍 Sort Key enables rich range queries

Once you've selected a partition (via PK), the sort key lets you query a subset of items efficiently:

begins_with(SK, "ORDER#2025-04") → all April 2025 orders
between(SK, "ORDER#2025-01-01", "ORDER#2025-03-31") → Q1 orders
SK = "PROFILE" → just the user profile
begins_with(SK, "ORDER#") → all orders for this user

This is why using a timestamp or date prefix in your SK is so powerful — range queries by time are free and fast.

Choosing a Good Partition Key

✅ Good Partition Keys

  • High cardinality — many distinct values
  • Uniform access — no single "hot" partition
  • userId, orderId, deviceId, sessionId
  • Composite: tenantId#userId
  • Roughly equal traffic across all PK values

❌ Bad Partition Keys

  • Low cardinality: status ("pending"/"done")
  • Date only: "2025-04-01" — all today's writes to one partition
  • Boolean: "isActive" — only 2 partitions!
  • Sequential IDs starting from 1 — all writes to partition 1 initially
  • Country code — India has 1B users, all in one partition

🔧 Operations — Reading & Writing Data

DynamoDB has a distinct set of operations. Choosing the right one dramatically impacts performance and cost. Never Scan when you can Query. Never Query when you can GetItem.

Read Operations — Performance Hierarchy

GetItem

Single item by exact key

Provide exact PK (+ SK if composite). Returns one item. O(1) — always fast regardless of table size.

⚡ 0.5 RCU (eventual) / 1 RCU (strong)
Query

Multiple items — same partition

Provide exact PK + optional SK condition. Returns all matching items sorted. Only reads within one partition. Fast.

⚡ Reads only matching items
Query GSI

Items by non-key attribute

Query on a Global Secondary Index. Same efficiency as base table Query, but on different attributes.

💰 GSI read capacity separate
Scan

⚠️ Reads ENTIRE table

Reads every single item. Filtered after read. Consumes RCU for ALL data, not just results. Avoid in production hotpaths.

🔥 Reads entire table — expensive

Write Operations

✍️ PutItem

  • Creates or completely replaces an item
  • If item exists with same key → overwritten silently
  • Use condition to prevent accidental overwrite
  • 1 WCU per KB written
  • Most common write operation

🔄 UpdateItem

  • Modifies specific attributes of an existing item
  • Other attributes unchanged
  • Atomic operations: SET, REMOVE, ADD, DELETE
  • Atomic counter: ADD views 1
  • Conditional: update only if attribute matches

🗑️ DeleteItem

  • Deletes item by exact key
  • Returns deleted item (if you want it)
  • Condition expression: only delete if condition met
  • Example: only delete if status = "cancelled"

⚡ Batch / Transact

  • BatchWriteItem: up to 25 PutItem/DeleteItem at once
  • BatchGetItem: up to 100 GetItem at once
  • TransactWriteItems: ACID across up to 100 items
  • TransactGetItems: consistent read up to 100 items
  • Transactions consume 2× the WCU/RCU

Conditional Writes — Optimistic Locking

📱 App updates product stock
  • Current stock: 5 units
  • User wants to buy 3
  • Condition: only update IF stock >= 3
↓
✅ Condition met — stock updated
  • Expression: stock >= :qty (3)
  • Atomic: stock 5 → 2
  • Another request at same time → condition fails → retry
  • No overselling — thread-safe update
❌ Condition fails — ConditionalCheckFailedException
  • Stock was already 2 (someone else bought)
  • App catches exception → tells user "out of stock"
  • No transaction needed — condition is atomic

💡 Strongly Consistent vs Eventually Consistent reads

Eventually Consistent (default, 0.5 RCU): May return slightly stale data (milliseconds old). Reads from any of the 3 AZ replicas. Use for: most reads where absolute freshness isn't critical.

Strongly Consistent (1 RCU): Always returns most recent data. Reads from the leader replica. Use for: after a write that must be immediately visible, financial balances, inventory counts. Double the cost.

📑 Global & Local Secondary Indexes

Indexes let you query your table on attributes that aren't the primary key — without expensive Scans. Think of them as separate sorted copies of your data, maintained automatically by DynamoDB.

GSI vs LSI — Side by Side

🟢 Global Secondary Index (GSI)

  • Completely different PK and SK from base table
  • Created at any time — even after table exists
  • Has its own separate throughput capacity
  • Eventually consistent reads only
  • Up to 20 GSIs per table
  • Data is projected (copied) to the GSI
  • Good for: query by any attribute
  • Example: query orders by status + date

🔵 Local Secondary Index (LSI)

  • Same PK as base table, different SK
  • Must be created at table creation — cannot add later!
  • Shares base table's throughput capacity
  • Supports strongly consistent reads
  • Up to 5 LSIs per table
  • 10 GB per partition limit (with LSI)
  • Good for: query same partition by different sort
  • Example: user's orders sorted by amount instead of date

💡 GSI Projection — what data to copy to the index

KEYS_ONLY: Only PK, SK, and GSI keys projected. Cheapest. Use if you'll fetch the full item separately after querying the GSI.
INCLUDE: Keys + specific attributes you name. Middle ground.
ALL: Entire item projected to GSI. Most expensive (double storage + write cost). Use only if you need all attributes from the index query result without a second fetch.

💰 Capacity Modes & Pricing

DynamoDB offers two capacity modes. Choosing the right one can halve your costs or prevent throttling. You can switch modes once every 24 hours.

✅ On-Demand (Pay Per Request)

  • No capacity planning required
  • Scales instantly on any traffic spike
  • Pay per read/write request unit
  • $1.25 per million WRU (Write Request Units)
  • $0.25 per million RRU (Read Request Units)
  • Best for: unpredictable, spiky, or new workloads
  • No throttling — auto-scales to any level

🔵 Provisioned (Manual Capacity)

  • You set RCU + WCU manually
  • $0.00065 per RCU/hour · $0.00327 per WCU/hour
  • Cheaper than On-Demand for predictable load
  • Can add Auto Scaling to adjust automatically
  • Throttled if you exceed provisioned limits
  • Best for: steady, predictable workloads
  • Use reserved capacity for 1-year: up to 77% savings

Read/Write Capacity Units Explained

📖 Read Capacity Unit (RCU)

  • 1 RCU = 1 strongly consistent read of up to 4 KB/sec
  • 1 RCU = 2 eventually consistent reads of up to 4 KB/sec
  • Larger items consume more RCU (rounded up to 4 KB)
  • Example: 10 KB item, strong consistent = 3 RCU (ceil(10/4))
  • Transact reads = 2× RCU cost

✍️ Write Capacity Unit (WCU)

  • 1 WCU = 1 write of up to 1 KB/sec
  • Larger items consume more WCU (rounded up to 1 KB)
  • Example: 3.5 KB item = 4 WCU
  • Transact writes = 2× WCU cost
  • Conditional writes still consume full WCU

Which Mode to Choose?

✅ On-Demand if:
  • New application — unknown traffic patterns
  • Traffic spikes without warning (e-commerce sales)
  • Very low traffic most of the time (dev/staging)
  • Can't predict read/write ratio
  • Simplicity over cost optimisation
✅ Provisioned if:
  • Stable, predictable traffic patterns
  • Cost is a priority (up to 5× cheaper at steady load)
  • Can use Auto Scaling for minor fluctuations
  • Using reserved capacity for maximum savings
  • Production apps with 30+ days of traffic data

  • Any PutItem, UpdateItem, DeleteItem operation

🏗️ Single-Table Design — The DynamoDB Way

In relational DBs, you create one table per entity type (users, orders, products). In DynamoDB, the recommended pattern is to store all entity types in a single table — using key prefixes and overloaded indexes to model relationships.

💡 Why Single-Table Design?

DynamoDB charges per table for provisioned capacity and you pay for cross-table operations separately. More importantly — you can fetch related data (user + their orders + their profile) in a single Query call instead of multiple round trips. Each round trip adds latency. Single-table = fewer API calls = lower latency.

Generic Key Naming Convention

📋 The "Overloaded GSI" Pattern

Use generic attribute names for the GSI keys (e.g. GSI1PK, GSI1SK) and populate them with different values per entity type:

For User items: GSI1PK = EMAIL#indhu@myapp.com → query by email
For Order items: GSI1PK = STATUS#pending → query all pending orders
For Product items: GSI1PK = CAT#electronics → query by category

One GSI, multiple access patterns. The GSI is "overloaded" — it serves different queries for different entity types. This is the canonical single-table design pattern advocated by AWS.

⚠️ Single-table design trade-offs

Single-table design is powerful but adds complexity. It requires upfront access pattern analysis. Adding new access patterns later may require a new GSI. The data model is harder to understand for new team members. Start with your access patterns — write them all down first, then design the table around them. Never design the table then figure out queries later.

Low-Level Client — Full Control

DynamoDbService.cs 
using Amazon.DynamoDBv2;
using Amazon.DynamoDBv2.Model;

public class DynamoDbService
{
    private readonly IAmazonDynamoDB _ddb;
    private const string Table = "AppTable";

    public DynamoDbService(IAmazonDynamoDB ddb) => _ddb = ddb;

    // ── PutItem ────────────────────────────────────
    public async Task PutOrderAsync(Order order)
    {
        await _ddb.PutItemAsync(new PutItemRequest
        {
            TableName = Table,
            Item = new()
            {
                ["PK"]       = new AttributeValue
                    { S = $"USER#{order.UserId}" },
                ["SK"]       = new AttributeValue
                    { S = $"ORDER#{order.CreatedAt:O}#{order.OrderId}" },
                ["GSI1PK"]  = new AttributeValue
                    { S = $"STATUS#{order.Status}" },
                ["GSI1SK"]  = new AttributeValue
                    { S = order.CreatedAt.ToString("O") },
                ["orderId"] = new AttributeValue
                    { S = order.OrderId },
                ["amount"]  = new AttributeValue
                    { N = order.Amount.ToString() },
                ["status"]  = new AttributeValue
                    { S = order.Status }
            },
            // Prevent overwriting existing order
            ConditionExpression = "attribute_not_exists(PK)"
        });
    }

    // ── GetItem ────────────────────────────────────
    public async Task<Dictionary<string,AttributeValue>?>
        GetUserProfileAsync(string userId)
    {
        var response = await _ddb.GetItemAsync(new GetItemRequest
        {
            TableName = Table,
            Key = new()
            {
                ["PK"] = new AttributeValue{ S = $"USER#{userId}"},
                ["SK"] = new AttributeValue{ S = "PROFILE" }
            },
            ConsistentRead = true // strong consistency
        });
        return response.Item.Count > 0 ? response.Item : null;
    }

    // ── Query — all orders for user ────────────────
    public async Task<List<Dictionary<string,AttributeValue>>>
        QueryUserOrdersAsync(string userId)
    {
        var response = await _ddb.QueryAsync(new QueryRequest
        {
            TableName              = Table,
            KeyConditionExpression =
                "PK = :pk AND begins_with(SK, :prefix)",
            ExpressionAttributeValues = new()
            {
                [":pk"]     = new AttributeValue
                    { S = $"USER#{userId}" },
                [":prefix"] = new AttributeValue
                    { S = "ORDER#" }
            },
            ScanIndexForward = false // newest first
        });
        return response.Items;
    }

    // ── UpdateItem — atomic counter ─────────────────
    public async Task IncrementViewCountAsync(
        string productId)
    {
        await _ddb.UpdateItemAsync(new UpdateItemRequest
        {
            TableName = Table,
            Key = new()
            {
                ["PK"] = new AttributeValue
                    { S = $"PRODUCT#{productId}" },
                ["SK"] = new AttributeValue
                    { S = "DETAILS" }
            },
            UpdateExpression = "ADD #views :one",
            ExpressionAttributeNames  = new(){ ["#views"]="views" },
            ExpressionAttributeValues = new()
            {
                [":one"] = new AttributeValue{ N="1" }
            }
        });
    }
}

Object Persistence Model (Higher Level)

Order.cs + DynamoDBContext — map C# objects directly 
using Amazon.DynamoDBv2.DataModel;

// ── Model with DynamoDB attributes ─────────────
[DynamoDBTable("AppTable")]
public class Order
{
    [DynamoDBHashKey("PK")]       // partition key
    public string Pk { get; set; } = "";

    [DynamoDBRangeKey("SK")]      // sort key
    public string Sk { get; set; } = "";

    [DynamoDBProperty("orderId")]
    public string OrderId { get; set; } = "";

    [DynamoDBProperty("amount")]
    public decimal Amount { get; set; }

    [DynamoDBProperty("status")]
    public string Status { get; set; } = "";

    [DynamoDBIgnore]               // not stored
    public string UserId { get; set; } = "";

    public DateTime CreatedAt { get; set; }
}

// ── Usage with DynamoDBContext ──────────────────
public class OrderRepository
{
    private readonly IDynamoDBContext _ctx;

    public OrderRepository(IDynamoDBContext ctx) => _ctx = ctx;

    public async Task SaveAsync(Order order)
        => await _ctx.SaveAsync(order);

    public async Task<Order?> GetAsync(string pk, string sk)
        => await _ctx.LoadAsync<Order>(pk, sk);

    public async Task DeleteAsync(string pk, string sk)
        => await _ctx.DeleteAsync<Order>(pk, sk);

    public AsyncSearch<Order> QueryUserOrders(string userId)
    {
        var pk = $"USER#{userId}";
        return _ctx.QueryAsync<Order>(pk,
            QueryOperator.BeginsWith, ["ORDER#"]);
    }
}

✅ Best Practices & Common Mistakes

Data Modelling

  • Start with access patterns — list every query your app needs BEFORE designing the table
  • Use composite keys (PK + SK) for almost all tables — simple key limits your flexibility
  • Prefix key values to distinguish entity types: USER#, ORDER#, PRODUCT#
  • Use ISO 8601 timestamps in sort keys to enable time-range queries
  • Prefer single-table design — fewer tables, fewer round trips, lower latency
  • Plan LSIs at table creation — you cannot add them later
  • Keep item size small — large items consume more RCU/WCU and slow queries

Performance & Cost

  • Never use Scan in production hot paths — always Query via PK or GSI
  • Use eventually consistent reads (default) unless you specifically need strong consistency
  • Use BatchGetItem and BatchWriteItem to reduce API call overhead
  • Project only needed attributes to GSIs — KEYS_ONLY or INCLUDE, not ALL
  • Enable Auto Scaling on Provisioned capacity to avoid over-provisioning
  • Use On-Demand for dev/test and unpredictable prod; Provisioned for steady prod workloads
  • Monitor ConsumedReadCapacityUnits and ConsumedWriteCapacityUnits in CloudWatch

Reliability & Safety

  • Always enable Point-in-Time Recovery (PITR) on production tables
  • Use conditional writes to prevent race conditions (attribute_not_exists, version check)
  • Enable DynamoDB Streams for event sourcing, audit logs, and cross-region replication
  • Set TTL attribute to auto-expire old items — clean up sessions, temp data for free
  • Use transactions (TransactWriteItems) for multi-item ACID operations
  • Tag tables with Environment, App, Team — enables cost allocation and access policies

Common Mistakes to Avoid

⚠️ Top DynamoDB Mistakes

❌ Using Scan in production queries — reads entire table, charges for every item
❌ Choosing status as partition key — only 3-4 distinct values → hot partition
❌ Designing table before listing access patterns — table won't support your queries
❌ Forgetting to add LSI at table creation — cannot add later, must recreate table
❌ Projecting ALL attributes to every GSI — doubles storage and write costs
❌ Not using condition expressions on writes — accidental overwrites go undetected
❌ No PITR enabled — one bad bulk delete = unrecoverable data loss
❌ Item size over 400 KB — DynamoDB rejects the write entirely
