🌍 What is Amazon Route 53?

Route 53 is AWS's highly available, globally distributed DNS (Domain Name System) service. It translates human-readable domain names (myapp.com) into IP addresses, and intelligently routes traffic using configurable policies — based on latency, geography, health, or weights.

💡 Real-life analogy

Route 53 is like a global air traffic control system. When you say "I want to go to Tokyo" (visit myapp.com), the controller (Route 53) checks the weather (health checks), finds the fastest runway (lowest latency server), and routes your flight (DNS query) to the best destination — automatically avoiding closed runways (unhealthy endpoints). The name "Route 53" comes from port 53 — the port DNS operates on.

Route 53's Three Core Functions

🌐 Domain Registration

Buy and manage domain names directly in AWS. No third-party registrar needed. Auto-renewal available.

📖 DNS Service

Host DNS records for your domain. Map myapp.com → your ALB, EC2, S3, or any IP. Globally distributed DNS servers.

❤️ Health Checking

Continuously monitor endpoints. Automatically route away from unhealthy targets. Failover in seconds.

  • DNS servers on every AWS edge location globally
  • Latency as low as ~1ms for cached responses

TTL — Time to Live (The Caching Knob)

⏱️ How TTL affects your DNS

Every DNS record has a TTL (seconds). Once a resolver caches the record, it won't ask Route 53 again until TTL expires. Choosing the right TTL is a trade-off:

Low TTL (30–60s): Changes propagate fast (failover in < 1 min). More DNS queries → higher cost + load on Route 53.
High TTL (300–3600s): Fewer queries → cheaper. But changes take longer to propagate. Failover may take minutes.

Best practice: Set TTL to 60s normally. Before a planned change (migration, IP swap), lower to 30s a day in advance. After change completes, raise back to 300s.

📋 DNS Record Types — Every One Explained

DNS records are instructions stored in Route 53. Each record type serves a specific purpose. Understanding which to use is fundamental to DNS configuration.

A Address Record Domain → IPv4
myapp.com → 52.66.100.200

Maps a domain name to an IPv4 address. The most common record. Use for EC2 instances, on-premises servers, or any IPv4 endpoint.

AAAA IPv6 Address Record Domain → IPv6
myapp.com → 2600:1f18:a0:b900::1

Maps a domain to an IPv6 address. Create alongside A record for dual-stack support. Required for modern IPv6-capable networks.

CNAME Canonical Name Name → another name
www.myapp.com → myapp.com

Points one domain name to another. Browser resolves the target further. Cannot be used at zone apex (myapp.com itself — use ALIAS instead). Use for subdomains pointing to CloudFront, etc.

ALIAS AWS Alias Record ✅ Name → AWS resource (FREE)
myapp.com → my-alb-123.ap-south-1.elb.amazonaws.com

AWS-specific extension — not a standard DNS type. Works at zone apex unlike CNAME. Queries to AWS resources (ALB, CloudFront, S3, API GW) are FREE. IP auto-updates when AWS resource IP changes. Always prefer ALIAS over CNAME for AWS resources.

MX Mail Exchange Email routing
myapp.com MX 10 → mail.myapp.com

Tells email servers where to deliver email for your domain. Priority number (lower = higher priority). Multiple MX records = redundant mail servers.

TXT Text Record Verification + SPF/DKIM
myapp.com TXT → "v=spf1 include:amazonses.com ~all"

Stores arbitrary text. Common uses: domain ownership verification (Google, ACM), SPF records for email anti-spam, DKIM signatures, DMARC policies.

NS Name Server Which DNS servers handle this domain
myapp.com NS → ns-123.awsdns-45.com

Delegates a domain (or subdomain) to specific name servers. Route 53 assigns 4 NS records per hosted zone. Register these at your domain registrar so the internet knows Route 53 handles your DNS.

💡 ALIAS vs CNAME — Rule of thumb

For AWS resources → always use ALIAS (free, works at apex, auto-updates IPs). For external services (Heroku, Netlify, etc.) → use CNAME. Never use a raw A record pointing to an AWS service IP — those IPs change!

🚦 Routing Policies — Smart Traffic Control

Route 53's routing policies decide which record to return when multiple exist for the same domain. This is what makes Route 53 more than just DNS — it's an intelligent global traffic manager.

SIMPLE

Simple

One record, one answer. No intelligence. Use for single-server setups.

WEIGHTED

Weighted

Split traffic by percentage. 90% v1, 10% v2. Perfect for canary deploys.

LATENCY

Latency

Route to lowest latency AWS region. User in India → Mumbai. User in US → Virginia.

FAILOVER

Failover

Active/standby. Primary healthy? Route there. Primary down? Auto-switch to secondary.

GEOLOCATION

Geolocation

Route by user's country/continent. India → Mumbai endpoint. EU → Frankfurt.

GEO-PROXIMITY

Geoproximity

Route by physical distance. Bias slider lets you expand/shrink a region's zone of influence.

MULTI-VALUE

Multi-Value

Return up to 8 healthy IPs. Client picks one. Poor man's load balancer at DNS level.

IP-BASED

IP-Based

Route based on client's IP CIDR range. Office IPs → dev endpoint. Other IPs → prod.

Weighted Routing — Canary Deployment in Action

🎯 Deploying v2 — gradual traffic shift

🟢 v1 (stable)
90% api-v1.myapp.com → ALB-v1
🔵 v2 (canary)
10% api-v1.myapp.com → ALB-v2

Same domain name — two records with different weights. Route 53 randomly selects proportionally. Monitor v2 errors → if good, adjust to 50/50 → then 100% v2.

  • User connects to closest, fastest endpoint

💓 Health Checks — Monitoring Your Endpoints

How Health Checks Work

📡 Route 53 Health Checkers (15+ global locations)
  • Probes from: us-east-1, eu-west-1, ap-southeast-1, ap-northeast-1...
  • Every 10 or 30 seconds (configurable)
  • Each checker independently validates your endpoint
  • Majority rules — if most checkers report fail → UNHEALTHY
↓
✅ Healthy Response
  • HTTP 200 within timeout
  • Response body check (optional)
  • Stay in DNS rotation
  • Counter: 3 consecutive passes needed
🔴 Unhealthy Response
  • Timeout, connection refused, non-2xx
  • After 3 consecutive failures → UNHEALTHY
  • Removed from DNS responses automatically
  • Failover kicks in if configured
↓ When endpoint recovers
🔄 Auto-Recovery
  • After 3 consecutive healthy checks → HEALTHY again
  • Automatically added back to DNS rotation
  • CloudWatch alarm triggered for ops team notification

Health Check Types

🌐 Endpoint Health Check

  • HTTP, HTTPS, or TCP probe
  • Specify: IP/domain, port, path (/health), protocol
  • Optional: match string in response body
  • Optional: request interval (10s or 30s)
  • Failure threshold: 1–10 consecutive failures

📊 Calculated Health Check

  • Combines results of multiple child health checks
  • Logic: AND (all must pass) or OR (any must pass)
  • Use for: "healthy only if DB + API + cache all pass"
  • Parent health check = composite view
  • Trigger failover based on combined status

📈 CloudWatch Alarm Health Check

  • Health based on a CloudWatch alarm state
  • "Healthy" = alarm is OK state
  • Use for: ECS CPU, RDS connections, Lambda errors
  • Failover when service degrades — not just crashes
  • Protects against partial failures

⚙️ Health Check for Private Resources

  • Route 53 checkers are outside your VPC
  • Private resources not directly reachable
  • Solution: CloudWatch metric → Alarm → HC
  • EC2 pushes metrics → alarm when unhealthy
  • Health check reads the alarm state

🔄 Failover — Automatic Disaster Recovery

Route 53 failover routing automatically redirects traffic from an unhealthy primary endpoint to a healthy secondary — usually within 60 seconds of detecting a failure.

Normal Operation — Primary Healthy

👤 User queries myapp.com
  • Route 53 receives the DNS query
↓
🛡️ Route 53 checks health
  • Primary record (ap-south-1 Mumbai): HEALTHY ✓
  • Returns primary endpoint IP
↓
✅ Traffic → Mumbai (Primary)
  • All users routed to Mumbai region
  • Singapore (secondary) receives zero traffic
  • Health check probing both continuously

Failover Event — Primary Goes Down

💥 Mumbai (Primary) — FAILURE
  • AZ outage / deployment failure / DDoS
  • Health check: GET /health → 503 Service Unavailable
  • Failure 1 of 3 threshold...
  • Failure 2 of 3...
  • Failure 3 of 3 → marked UNHEALTHY
↓ ~30–60 seconds
🔄 Route 53 activates failover
  • Primary record: UNHEALTHY — stop returning it
  • Secondary record (Singapore): HEALTHY ✓
  • DNS response switches to Singapore IP
  • SNS alert fires to ops team
↓
✅ Traffic → Singapore (Secondary)
  • New DNS queries return Singapore IP
  • Existing connections (cached TTL) reconnect naturally
  • Users experience brief disruption (TTL expiry time)
  • Lower TTL (30s) = faster failover perception
↓ When Mumbai recovers
🔁 Automatic Failback
  • 3 consecutive health check passes → Mumbai HEALTHY
  • Route 53 automatically restores primary routing
  • Traffic gradually shifts back as TTL expires
  • No manual intervention required

Active-Passive vs Active-Active

😴 Active-Passive (Failover Policy)

  • Primary: handles ALL traffic normally
  • Secondary: idle, receives zero traffic
  • Secondary activates ONLY on primary failure
  • Secondary can be smaller (standby capacity)
  • Use for: DR, simple HA setups
  • Route 53 routing: Failover policy

⚡ Active-Active (Weighted/Latency + HC)

  • All endpoints handle traffic simultaneously
  • Load spread across multiple regions/AZs
  • If one fails: Route 53 stops returning its IP
  • Remaining healthy endpoints absorb traffic
  • Use for: global apps, high throughput
  • Route 53 routing: Weighted or Latency + health checks

🎯 Failover in numbers

Health check interval: 10s · Failure threshold: 3 · Time to detect failure: ~30s · DNS TTL: 30–60s · Total failover time: 60–90 seconds from failure to full traffic shift. This is why low TTL matters during critical operations.

💻 C# SDK — Manage Route 53 Programmatically

Route 53 Service — Complete Operations

Route53Service.cs 
using Amazon.Route53;
using Amazon.Route53.Model;

public class Route53Service
{
    private readonly IAmazonRoute53 _r53;

    public Route53Service(IAmazonRoute53 r53) => _r53 = r53;

    // ── Create / Update a DNS record ──────────────
    public async Task<string> UpsertRecordAsync(
        string hostedZoneId,
        string name,           // e.g. "api.myapp.com."
        string ip,             // e.g. "52.66.100.200"
        int    ttl = 60)
    {
        var response = await _r53.ChangeResourceRecordSetsAsync(
            new ChangeResourceRecordSetsRequest
        {
            HostedZoneId = hostedZoneId,
            ChangeBatch  = new ChangeBatch
            {
                Comment = $"Upsert {name} at {DateTime.UtcNow:O}",
                Changes = [new Change
                {
                    Action            = ChangeAction.UPSERT,
                    ResourceRecordSet = new ResourceRecordSet
                    {
                        Name = name,
                        Type = RRType.A,
                        TTL  = ttl,
                        ResourceRecords = [new ResourceRecord
                            { Value = ip }]
                    }
                }]
            }
        });
        return response.ChangeInfo.Id;
    }

    // ── Create ALIAS record pointing to ALB ────────
    public async Task<string> CreateAliasRecordAsync(
        string hostedZoneId,
        string name,
        string albDnsName,
        string albHostedZoneId)
    {
        var response = await _r53.ChangeResourceRecordSetsAsync(
            new ChangeResourceRecordSetsRequest
        {
            HostedZoneId = hostedZoneId,
            ChangeBatch  = new ChangeBatch
            {
                Changes = [new Change
                {
                    Action            = ChangeAction.UPSERT,
                    ResourceRecordSet = new ResourceRecordSet
                    {
                        Name = name,
                        Type = RRType.A,
                        // ALIAS — no TTL needed
                        AliasTarget = new AliasTarget
                        {
                            DNSName              = albDnsName,
                            HostedZoneId         = albHostedZoneId,
                            EvaluateTargetHealth = true
                        }
                    }
                }]
            }
        });
        return response.ChangeInfo.Id;
    }

    // ── Create Weighted records (canary deploy) ────
    public async Task CreateWeightedRecordsAsync(
        string hostedZoneId,
        string name,
        string ipV1, int weightV1,
        string ipV2, int weightV2)
    {
        await _r53.ChangeResourceRecordSetsAsync(
            new ChangeResourceRecordSetsRequest
        {
            HostedZoneId = hostedZoneId,
            ChangeBatch  = new ChangeBatch
            {
                Changes = [
                    new Change
                    {
                        Action = ChangeAction.UPSERT,
                        ResourceRecordSet = new ResourceRecordSet
                        {
                            Name            = name,
                            Type            = RRType.A,
                            SetIdentifier   = "v1-stable",
                            Weight          = weightV1, // e.g. 90
                            TTL             = 60,
                            ResourceRecords = [new ResourceRecord
                                { Value = ipV1 }]
                        }
                    },
                    new Change
                    {
                        Action = ChangeAction.UPSERT,
                        ResourceRecordSet = new ResourceRecordSet
                        {
                            Name            = name,
                            Type            = RRType.A,
                            SetIdentifier   = "v2-canary",
                            Weight          = weightV2, // e.g. 10
                            TTL             = 60,
                            ResourceRecords = [new ResourceRecord
                                { Value = ipV2 }]
                        }
                    }
                ]
            }
        });
    }

    // ── Get health check status ────────────────────
    public async Task<string> GetHealthCheckStatusAsync(
        string healthCheckId)
    {
        var response = await _r53.GetHealthCheckStatusAsync(
            new GetHealthCheckStatusRequest
        {
            HealthCheckId = healthCheckId
        });

        var passing = response.HealthCheckObservations
            .Count(o => o.StatusReport.Status
                .StartsWith("Success"));
        return passing >= 6 ? "HEALTHY" : "UNHEALTHY";
    }

    // ── Wait for DNS change to propagate ──────────
    public async Task WaitForChangeAsync(string changeId)
    {
        ChangeStatus status;
        do
        {
            await Task.Delay(TimeSpan.FromSeconds(5));
            var resp = await _r53.GetChangeAsync(
                new GetChangeRequest { Id = changeId });
            status = resp.ChangeInfo.Status;
        }
        while (status == ChangeStatus.PENDING);
        // Status: PENDING → INSYNC (usually <60s)
    }
}

✅ Best Practices & Common Mistakes

DNS Records

  • Always use ALIAS records for AWS resources (ALB, CloudFront, S3) — free + auto IP updates
  • Never use raw A records pointing to AWS service IPs — they change without notice
  • Use CNAME only for external services (Heroku, Netlify) where ALIAS isn't possible
  • Add both A (IPv4) and AAAA (IPv6) records for modern dual-stack support
  • Set TTL to 60s normally — lower to 30s before planned IP changes, restore after
  • Always add trailing dot in CLI/SDK record names: myapp.com. (not myapp.com)

Routing Policies

  • Use Latency routing for global apps — users automatically reach the closest region
  • Use Weighted routing for canary deploys — start at 5% before going 100%
  • Always attach health checks to weighted/latency/failover records
  • Use Failover policy for disaster recovery — don't build this manually
  • Use Geolocation to serve region-specific content or comply with data residency laws
  • Set a default Geolocation record — handles countries not explicitly listed

Health Checks

  • Use /health endpoint that checks dependencies (DB, cache) — not just HTTP 200
  • Fast health checks (10s) for critical production routes worth the extra $0.10/month
  • Use calculated health checks for complex "healthy only when all services pass"
  • Enable SNS notifications on health check state change — never miss a failover
  • For private resources: use CloudWatch alarm health checks (not direct endpoint)
  • Test failover manually before you need it — lower TTL, fail the HC, verify switch

Operations

  • Use meaningful record IDs for weighted/latency records — aids debugging
  • Tag all health checks with App and Environment
  • Keep hosted zone per environment (myapp.com for prod, dev.myapp.com for dev)
  • Use private hosted zones for internal service discovery in VPC
  • Export hosted zone records before any migration — backup your DNS
  • Use Route 53 Resolver for hybrid cloud DNS resolution to on-premises

Common Mistakes to Avoid

⚠️ Top Route 53 Mistakes

❌ High TTL before migration — stuck with old IP for hours during cutover
❌ A record pointing to ALB IP — ALB IPs change, your site breaks silently
❌ No health checks on failover records — failover never triggers
❌ CNAME at zone apex — DNS spec violation, most resolvers reject it
❌ Not testing failover before production incident — first test under pressure
❌ Forgetting NS record update at registrar — hosted zone created but DNS still points elsewhere
❌ No default Geolocation record — unlisted countries get NXDOMAIN (site not found!)
❌ Deleting hosted zone with active records — cascades to site outage
