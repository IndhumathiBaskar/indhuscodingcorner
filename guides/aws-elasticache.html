AWS ElastiCache

Fully managed in-memory data store and cache — sub-millisecond latency at massive scale with Redis or Memcached.

🔴 Redis 🔵 Memcached ⚡ Sub-ms Latency 🔁 Replication 📦 Cluster Mode 🛡️ Failover 🔒 Encryption 📊 Pub/Sub

🏗️ What is ElastiCache?

AWS managed in-memory caching service — dramatically reduces database load and response times.

One-liner ElastiCache = fully managed Redis or Memcached clusters on AWS — microsecond to sub-millisecond reads, automatic failover, scaling, backups, and patching handled for you.
⚡ Performance

Sub-millisecond read latency. Serves millions of requests per second. Offloads hot reads from RDS/DynamoDB entirely.

🔴 Redis Engine

Rich data structures, persistence, replication, pub/sub, Lua scripting, Sorted Sets, Streams. The default choice for most use cases.

🔵 Memcached Engine

Pure simple key-value cache. Multi-threaded, horizontally scalable. No persistence or replication. Best for pure cache-only scenarios.

🔁 Replication

Redis supports up to 5 read replicas per shard. Auto-failover promotes a replica to primary within seconds if the primary fails.

📦 Cluster Mode

Redis Cluster shards data across up to 500 nodes. Each shard has its own primary + replicas. Enables horizontal scaling of writes.

🔒 Security

VPC isolation, TLS in-transit, at-rest encryption (KMS), Redis AUTH / RBAC, IAM auth for Valkey/Redis 7+.

When to use ElastiCache? Database query caching · Session storage · Leaderboards (Sorted Sets) · Rate limiting · Pub/Sub messaging · Real-time analytics · Distributed locking · Job queues

🆚 Redis vs Memcached

Choose the right engine for your workload — they serve very different purposes.

🔴 Redis
Feature-Rich Cache + Data Store
  • ✅ Rich data structures (String, Hash, List, Set, Sorted Set, Stream)
  • ✅ Persistence (RDB snapshots + AOF logs)
  • ✅ Replication + Auto-failover
  • ✅ Cluster mode (sharding)
  • ✅ Pub/Sub messaging
  • ✅ Lua scripting + Transactions (MULTI/EXEC)
  • ✅ Sorted Sets for leaderboards
  • ✅ Geospatial commands
  • ✅ Backup & restore
  • ❌ Single-threaded (per shard)
🔵 Memcached
Simple Pure Cache
  • ✅ Simple key-value only
  • ✅ Multi-threaded — uses all CPU cores
  • ✅ Horizontally scalable (add nodes)
  • ✅ Lower memory overhead per item
  • ❌ No persistence
  • ❌ No replication
  • ❌ No failover
  • ❌ No pub/sub
  • ❌ No backup/restore
  • ❌ No cluster mode (only partitioning)

🎯 Quick decision rule Need persistence, replication, rich data types, pub/sub, or failover? → Redis. Need pure high-throughput simple key-value with multi-threading and don't care about data loss? → Memcached.

In practice, 95% of AWS use cases choose Redis.

🗂️ Redis Data Structures

Redis is more than a cache — it's a data structure server. Each type solves specific problems.

STRING
Plain value — text, number, binary. Max 512 MB. Supports INCR/DECR, APPEND, SETEX (with TTL).
Cache HTML, counters, tokens
HASH
Map of field → value. Like a mini object / row. HGET, HSET, HMGET. Memory efficient for objects.
User profiles, sessions
LIST
Ordered linked list. LPUSH/RPUSH, LPOP/RPOP, LRANGE. Preserves insertion order.
Message queues, activity feeds
SET
Unordered unique values. SADD, SMEMBERS, SINTER (intersection), SUNION. Fast membership test.
Tags, unique visitors, dedup
SORTED SET (ZSet)
Set with a float score per member. ZADD, ZRANGE, ZRANGEBYSCORE. Members ordered by score automatically.
Leaderboards, rate limiting
STREAM
Append-only log of entries. XADD, XREAD, XGROUP. Consumer groups for fan-out. Like a lightweight Kafka.
Event streams, IoT data
GEO
Store lat/long, query by radius. GEOADD, GEODIST, GEORADIUS. Internally uses Sorted Sets.
Nearby stores, delivery tracking
BITMAP
Bit-level operations on strings. SETBIT, GETBIT, BITCOUNT. Extremely space efficient.
Daily active users, feature flags
HyperLogLog
Probabilistic cardinality estimation. PFADD, PFCOUNT. ~0.81% error, uses <12 KB regardless of input size.
Unique count at scale
💡 TTL on any key Every Redis key (regardless of type) can have a TTL set via EXPIRE key seconds or EXPIREAT key timestamp. Expired keys are lazily deleted plus active expiry scans run in background.

🔁 Replication & Failover

Redis replication keeps your data available even when a node fails.

🔁 Replication Group

One primary node handles all writes. Up to 5 read replicas per shard receive async replication from primary.

Replicas serve read traffic, reducing load on primary. Spread replicas across multiple AZs for HA.

🛡️ Auto-Failover (Multi-AZ)

When primary fails, ElastiCache automatically promotes the replica with the least replication lag to primary.

DNS endpoint flips to the new primary within ~1 minute. Apps using the cluster endpoint reconnect automatically.

📍 Endpoints

Primary Endpoint → always points to the write primary

Reader Endpoint → load balances across all replicas

Node Endpoint → direct connection to a specific node

💾 Persistence (Redis)

RDB Point-in-time snapshot. Compact. Faster restart. Some data loss between snapshots.

AOF Append-only log of every write. Near-zero data loss. Slower restart. Larger file.

Both can be used together in ElastiCache.

Replication Lag & Consistency

REPLICATION FLOW:

Client WRITE → Primary Node
  ├── Acknowledges write to client (async replication)
  ├── Sends command to Replica 1 (AZ-a)
  └── Sends command to Replica 2 (AZ-b)

Client READ → Reader Endpoint → Replica (round-robin)
  NOTE: May read slightly stale data (eventual consistency)

FAILOVER:
Primary fails → ElastiCache detects (~10s)
  → Elects replica with lowest replication offset
  → Promotes to Primary (~60s total)
  → DNS CNAME updated → app reconnects

📦 Cluster Mode

Shard data across multiple nodes for horizontal write scaling beyond a single node's memory.

📦 Cluster Mode Disabled

Single shard with 1 primary + up to 5 replicas. All data on one node. Simple setup. Max node size ~768 GB RAM.

Good for: most workloads that fit in one node's RAM.

Simpler Easy failover

🔀 Cluster Mode Enabled

Up to 500 shards, each with 1 primary + up to 5 replicas. Keys distributed via hash slots (0–16383).

Good for: massive datasets, very high write throughput.

More complex Unlimited scale

Hash Slot Distribution

How cluster sharding works Redis Cluster uses 16,384 hash slots. Each key is hashed via CRC16 % 16384 to determine its slot. Slots are distributed evenly across shards. E.g. 3 shards → Shard 1 gets slots 0–5460, Shard 2: 5461–10922, Shard 3: 10923–16383.

C# String operations — cache-aside pattern 
// Cache-Aside: check cache first, fallback to DB
public async Task<Product> GetProductAsync(int productId)
{
    var key = $"product:{productId}";

    // 1. Try cache
    var cached = await db.StringGetAsync(key);
    if (cached.HasValue)
        return JsonSerializer.Deserialize<Product>(cached!)!;

    // 2. Cache miss → fetch from DB
    var product = await _db.GetProductByIdAsync(productId);
    if (product is null) return null!;

    // 3. Write to cache with TTL
    await db.StringSetAsync(key,
        JsonSerializer.Serialize(product),
        TimeSpan.FromMinutes(10));  // 10-min TTL

    return product;
}

// Invalidate on update
public async Task UpdateProductAsync(Product product)
{
    await _db.UpdateProductAsync(product);
    await db.KeyDeleteAsync($"product:{product.Id}");
}

🎯 Caching Patterns

Standard patterns for integrating a cache layer — each with different consistency guarantees.

📖 Cache-Aside (Lazy Loading)

Read: Check cache → miss → read DB → write cache → return.

Write: Write DB → invalidate (delete) cache key.

Pros Only caches what's needed. Cache failure doesn't block reads.

Cons Cache miss = extra latency. Stale data possible until TTL expires.

✍️ Write-Through

Write: Write to cache AND DB simultaneously on every write.

Read: Always hits cache (always warm).

Pros Always consistent. No cold start problem.

Cons Write latency increases. Cache fills with unread data.

⏪ Write-Behind (Write-Back)

Write: Write to cache only. Async flush to DB later.

Pros Extremely fast writes. DB write batching.

Cons Risk of data loss if cache crashes before flush. Complex.

🔥 Read-Through

Cache sits in front of DB. On miss, cache itself loads from DB and populates — not the app.

Pros App code simpler — just reads from cache.

Cons First request always slow. Cache must support read-through.

💡 Best Practices

Production-grade ElastiCache setup for performance, cost, and resilience.

✅ Key Naming

Use colon-separated namespaces: user:{id}:profile, product:{id}:cache. Consistent naming enables pattern scanning and easier debugging.

✅ Always Set TTL

Never cache data without a TTL unless intentional. Without TTL, keys accumulate and eat memory. Match TTL to data staleness tolerance.

✅ Connection Pooling

Use a singleton ConnectionMultiplexer (StackExchange.Redis). Don't create a new connection per request — very expensive with ElastiCache.

✅ Multi-AZ + Replicas

Always enable Multi-AZ with at least 1 replica in a different AZ for production. Enables auto-failover without manual intervention.

✅ Monitor Key Metrics

CacheHitRate (aim >90%), Evictions (should be 0 or low), CurrConnections, ReplicationLag, FreeableMemory.

✅ VPC + Security Groups

Always place ElastiCache inside a VPC. Only allow inbound 6379 from your app's security group — never expose to public internet.

✅ Avoid Large Keys

Keep values under 1 MB. Large values slow replication and increase memory fragmentation. Split large objects or use S3 for big payloads.

✅ Handle Cache Stampede

Use probabilistic early expiration or a distributed lock to prevent all instances refilling the cache simultaneously after expiry (thundering herd).

🏁 Summary

Everything you need to know about AWS ElastiCache at a glance.

🔴 Redis vs 🔵 Memcached

Redis = rich types + persistence + replication + failover + pub/sub. Memcached = simple pure cache, multi-threaded. Redis wins 95% of the time.

🗂️ Data Structures

String, Hash, List, Set, Sorted Set, Stream, Geo, Bitmap, HyperLogLog. Each type solves a specific problem efficiently in memory.

🔁 Replication

Primary + up to 5 read replicas. Multi-AZ auto-failover promotes replica in ~60s. Reader endpoint load-balances across replicas.

📦 Cluster Mode

Up to 500 shards, 16,384 hash slots. Enables massive horizontal write scaling. Hash tags keep related keys on same shard.

🎯 Caching Patterns

Cache-Aside (most common), Write-Through (consistent), Write-Behind (fast writes), Read-Through (simple app code). Choose based on consistency needs.

🔒 Security

VPC isolation + Security Groups + TLS in-transit + KMS at-rest encryption + Redis AUTH token or IAM auth. Never expose port 6379 publicly.
