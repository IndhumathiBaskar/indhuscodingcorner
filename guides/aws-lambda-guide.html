⚡ What is AWS Lambda?

Lambda is AWS's serverless compute service. You write a function, upload it, and Lambda runs it in response to events — no servers to provision, patch, or manage. You pay only for the milliseconds your code actually runs.

💡 Real-life analogy — The Vending Machine

Lambda is like a vending machine. It sits idle, consuming almost nothing. Press a button (event arrives) → it springs to life, executes exactly one operation (dispenses your snack), and returns to idle. You pay per button press — not for the machine sitting there. Another person can press a different button simultaneously (concurrency). If 1000 people press buttons at once — 1000 operations run in parallel, automatically. No staff needed.

Why Lambda? Traditional vs Serverless

🖥️ Traditional Server (EC2)

  • Pay 24/7 even at 0% CPU
  • Manually patch OS and runtime
  • Scale: provision new instances (minutes)
  • You manage: servers, OS, runtime, app
  • Capacity planning required upfront
  • Idle nights/weekends: still paying

⚡ Lambda (Serverless)

  • Pay only for execution (per millisecond)
  • AWS patches everything below your code
  • Scale: 0 → 1,000 concurrent instantly
  • You manage: only your code
  • No capacity planning — auto-scales
  • Idle nights/weekends: $0.00

Init Code vs Handler Code — Critical Distinction

Lambda execution model — what runs ONCE vs EVERY invocation 
// ═══════════════════════════════════════════════════════
// INIT CODE — runs ONCE per container lifetime (cold start)
// Place expensive operations here!
// ═══════════════════════════════════════════════════════

public class Function
{
    // These are created ONCE and reused across invocations
    private static readonly HttpClient _http =
        new HttpClient(); // ✅ Create once, reuse

    private static readonly IConfiguration _config;
    private static readonly IDatabaseService _db;

    // Static constructor — runs once at cold start
    static Function()
    {
        _config = new ConfigurationBuilder()
            .AddEnvironmentVariables()
            .Build();

        // DB connection pooling — expensive, do it once
        _db = new DatabaseService(
            _config["DB_CONNECTION_STRING"]!);
    }

    // ═══════════════════════════════════════════════════════
    // HANDLER — runs on EVERY invocation
    // Keep this lean — reuse objects from init code above
    // ═══════════════════════════════════════════════════════
    public async Task<string> FunctionHandler(
        string input, ILambdaContext context)
    {
        context.Logger.LogInformation($"Input: {input}");

        // ✅ Reuses _http and _db from init code
        var result = await _db.QueryAsync(input);
        return result;
    }
}

💡 The /tmp directory persists between warm invocations

Files written to /tmp survive between warm invocations of the same container. Use this for caching: download a large model or config file on cold start, store it in /tmp, check if it exists on subsequent invocations. Available from 512 MB to 10 GB (configurable).

🥶 Cold Start — The Latency Tax & How to Beat It

A cold start happens when Lambda must initialise a new execution environment before running your function. It adds latency the first time and after periods of inactivity. Understanding it — and mitigating it — is one of the most important Lambda skills.

When Does a Cold Start Happen?

🥶 Cold start triggers
  • First invocation ever (new deployment)
  • No recent invocations (container expired)
  • Concurrent requests exceed warm containers
  • Deployment of new code version
  • Configuration change (env var, memory, etc.)
  • Lambda needs to scale up beyond current capacity
☀️ Warm — no cold start
  • Same container handles another request
  • Recent invocation kept it warm
  • Provisioned concurrency pre-warmed it
  • Init code already ran — skipped
  • HTTP clients, DB connections reused
  • Typical API response: single-digit milliseconds

Mitigation Strategies

🔥 Provisioned Concurrency

  • Pre-initialises N execution environments
  • Always warm — zero cold start for those N instances
  • Extra cost: charged even when idle
  • Use for: latency-sensitive APIs, user-facing endpoints
  • Schedule: reduce at night using Application Auto Scaling
  • Set to: peak expected concurrent users

⚡ Native AOT (.NET 8)

  • Compiles .NET to native binary at build time
  • No JIT at runtime → cold start < 100ms
  • Smaller deployment package
  • Trade-off: no runtime reflection, longer build
  • Use for: latency-critical functions without complex reflection
  • SDK: Amazon.Lambda.RuntimeSupport AOT mode

📦 Minimise Package Size

  • Smaller package → faster download → faster init
  • .NET: trim unused assemblies (PublishTrimmed=true)
  • Use Lambda Layers for shared dependencies
  • Avoid bundling unused NuGet packages
  • Target: keep deployment < 10 MB

🏗️ Optimise Init Code

  • Lazy-load what you don't always need
  • Cache secrets in memory after first fetch
  • Use static readonly for reusable clients
  • Pre-warm DB connection in static constructor
  • Avoid synchronous I/O in init code

Three Concurrency Types

🔶 Unreserved Concurrency (Default)

  • Shared pool across all Lambda functions in region
  • Default: 1,000 concurrent (soft limit — raise via Support)
  • One "noisy" function can exhaust capacity for others
  • Use when: low-priority, batch jobs, internal tools

🔷 Reserved Concurrency

  • Guarantees N slots for this function exclusively
  • Also caps the function — it can never exceed N
  • Protects other functions from being starved
  • Protects downstream services from being overwhelmed
  • Use for: production APIs, rate-sensitive backends

🔥 Provisioned Concurrency

  • Pre-initialises N environments — always warm
  • Zero cold start for up to N concurrent requests
  • Beyond N → falls back to unreserved (cold starts)
  • Charged: per provisioned environment-hour
  • Use for: user-facing APIs with strict latency SLAs

📊 Burst Concurrency (Regional)

  • Initial burst: 500–3,000 (varies by region)
  • After burst limit: scales by 500/minute
  • Example: 500 → 1000 after 1 min → 1500 after 2 min
  • Sudden 10,000 request spike: most throttled initially
  • Mitigate: SQS buffer → controlled Lambda fan-out

⚠️ Concurrency and downstream systems

Lambda scales to 1,000 concurrent easily. But your RDS database handles 100 connections max. Without Reserved Concurrency set to 50, a traffic spike can create 1,000 simultaneous DB connections → connection exhaustion → every Lambda invocation fails. Always set Reserved Concurrency to protect your downstream services.

📡 Triggers — What Can Invoke Lambda

Lambda can be invoked by dozens of AWS services. Triggers fall into three invocation models: synchronous, asynchronous, and event source mappings (streams/queues).

Invocation Models — The Critical Difference

🔵 Synchronous

  • Caller WAITS for Lambda to return a response
  • Errors must be handled by the caller
  • No built-in retry — caller decides
  • Example: API Gateway → Lambda → response to user
  • Use for: user-facing APIs, request/response patterns

🟢 Asynchronous

  • Caller fires and immediately returns (202 Accepted)
  • Lambda invokes in background, caller doesn't wait
  • Built-in retry: 2 retries on failure
  • Success/failure can route to SQS/SNS (Destinations)
  • Example: S3 event → Lambda processes file
  • Use for: background jobs, event-driven processing

  • Lambda retries up to 2 times on failure

  • Send to SQS DLQ
  • OR notify SNS (alert team)

💡 Function URLs — the API Gateway alternative

Lambda Function URLs give you a dedicated HTTPS endpoint without needing API Gateway. Simpler for single-function APIs, internal tools, or webhooks. They support both sync and streaming responses. IAM auth or no auth. Limitation: no API GW features (throttling, WAF, caching, custom domain without CloudFront).

💡 Why use Layers?

Layers separate dependencies from your function code. Benefits: faster deployments (only upload changed code, not 80 MB of NuGet packages), share common code across functions, separate update cycles (update dependencies independently from function), stay within the 250 MB unzipped package limit.

// 1. Request cost:
1,000,000 requests × $0.0000002 = $0.20
// First 1M requests per month: FREE

// 2. Duration cost:
// Price per GB-second = $0.0000166667
// GB-second = (memory in MB / 1024) × duration in seconds

// Example: 128 MB, 200ms, 1M invocations
GB_seconds = (128 / 1024) × 0.200 × 1_000_000  // = 25,000
Duration_cost = 25_000 × $0.0000166667          // = $0.42
Total = $0.20 + $0.42 = $0.62 // for 1M requests!

Total = $0.20 + $0.83 = $1.03
// More memory = more cost here. But faster, so check SLA.

Pattern 1 — API Gateway HTTP Function

ApiFunction.cs — sync, returns HTTP response 
[assembly: LambdaSerializer(
    typeof(DefaultLambdaJsonSerializer))]

public class ApiFunction
{
    // ── Init code: runs ONCE per container ────────
    private static readonly HttpClient _http = new();
    private static readonly string _connStr;

    static ApiFunction()
    {
        // Fetch secret once at cold start — cached in memory
        _connStr = Environment.GetEnvironmentVariable(
            "DB_SECRET_NAME")!;
    }

    // ── Handler: called on every invocation ───────
    public async Task<APIGatewayHttpApiV2ProxyResponse>
        HandleAsync(APIGatewayHttpApiV2ProxyRequest req,
                    ILambdaContext ctx)
    {
        ctx.Logger.LogInformation(
            "RequestId: {Id}", ctx.AwsRequestId);

        var id = req.PathParameters["id"];
        var userId = req.RequestContext
            .Authorizer?.Jwt?.Claims["sub"];

        // Business logic...
        var result = new { orderId = id, userId };

        return new APIGatewayHttpApiV2ProxyResponse
        {
            StatusCode = 200,
            Body       = JsonSerializer.Serialize(result),
            Headers    = new() {
                ["Content-Type"] = "application/json" }
        };
    }
}

Pattern 2 — SQS Consumer with Partial Batch Failure

SqsFunction.cs — production-grade batch handler 
using Amazon.Lambda.SQSEvents;

public class SqsFunction
{
    public async Task<SQSBatchResponse> HandleAsync(
        SQSEvent evt, ILambdaContext ctx)
    {
        var failures = new
            List<SQSBatchResponse.BatchItemFailure>();

        var tasks = evt.Records.Select(async record =>
        {
            try
            {
                var order = JsonSerializer
                    .Deserialize<OrderEvent>(record.Body);

                await ProcessAsync(order!, ctx);
            }
            catch (Exception ex)
            {
                ctx.Logger.LogError(ex,
                    "Failed: {Id}", record.MessageId);

                lock (failures)
                    failures.Add(new () {
                        ItemIdentifier = record.MessageId });
            }
        });

        await Task.WhenAll(tasks); // parallel processing

        return new SQSBatchResponse
            { BatchItemFailures = failures };
    }

    private async Task ProcessAsync(
        OrderEvent order, ILambdaContext ctx)
    {
        ctx.Logger.LogInformation(
            "Processing order {Id}", order.OrderId);
        // inventory, email, analytics...
        await Task.CompletedTask;
    }
}

Pattern 3 — S3 Trigger (Process Uploaded File)

S3Function.cs — react to file uploads 
using Amazon.Lambda.S3Events;
using Amazon.S3;

public class S3Function
{
    private static readonly IAmazonS3 _s3 =
        new AmazonS3Client();

    public async Task HandleAsync(
        S3Event evt, ILambdaContext ctx)
    {
        foreach (var record in evt.Records)
        {
            var bucket = record.S3.Bucket.Name;
            var key    = record.S3.Object.Key;
            var size   = record.S3.Object.Size;

            ctx.Logger.LogInformation(
                "Processing s3://{B}/{K} ({S} bytes)",
                bucket, key, size);

            using var stream = await _s3
                .GetObjectStreamAsync(bucket, key, null);

            // Process the file stream...
            await ResizeImageAsync(stream, bucket, key);
        }
    }

    private Task ResizeImageAsync(
        Stream s, string b, string k)
        => Task.CompletedTask; // your image processing
}

Pattern 4 — Invoke Lambda from Code

Invoke another Lambda function programmatically 
using Amazon.Lambda;
using Amazon.Lambda.Model;

public class LambdaInvokerService
{
    private static readonly IAmazonLambda _lambda =
        new AmazonLambdaClient();

    // ── Sync invoke — wait for result ─────────────
    public async Task<TResult> InvokeAsync<TPayload, TResult>(
        string functionName, TPayload payload)
    {
        var response = await _lambda.InvokeAsync(
            new InvokeRequest
        {
            FunctionName   = functionName,
            InvocationType = InvocationType.RequestResponse,
            Payload        = JsonSerializer.Serialize(payload)
        });

        return JsonSerializer.Deserialize<TResult>(
            response.Payload)!;
    }

    // ── Async invoke — fire and forget ─────────────
    public async Task InvokeFireAndForgetAsync<T>(
        string functionName, T payload)
    {
        await _lambda.InvokeAsync(new InvokeRequest
        {
            FunctionName   = functionName,
            InvocationType = InvocationType.Event, // async
            Payload        = JsonSerializer.Serialize(payload)
        });
        // Returns immediately — Lambda runs in background
    }
}

✅ Best Practices & Common Mistakes

Function Design

  • Move expensive operations (DB connections, HTTP clients, secrets) to init code — they run once
  • Keep functions small and focused — one function, one responsibility
  • Use static readonly for reusable objects — HttpClient, DB connections, config
  • Choose arm64 (Graviton) architecture — 20% cheaper, often faster
  • Use Lambda Power Tuning tool to find optimal memory setting for your workload
  • Cache secrets in memory at startup — don't call Secrets Manager on every invocation
  • Use structured logging (JSON) — makes CloudWatch Logs Insights queries fast

Cold Start Reduction

  • Use Native AOT for latency-critical .NET functions — cold start under 100ms
  • Minimise package size — use PublishTrimmed=true, remove unused packages
  • Use Lambda Layers for shared dependencies to keep function zips small
  • Enable Provisioned Concurrency for user-facing APIs with strict latency SLAs
  • Lazy-load resources you don't always need — don't load everything at init
  • Cache results in /tmp across warm invocations to avoid redundant downloads

Concurrency & Safety

  • Always set Reserved Concurrency — protect downstream services from Lambda scale-out
  • Set SQS visibility timeout to 6× Lambda timeout to prevent duplicate processing
  • Use Destinations for async Lambda — route failures to SQS DLQ or SNS alert
  • Always use SQSBatchResponse with partial batch item failures for SQS triggers
  • Set CloudWatch alarm on Lambda errors AND throttles — both are production incidents
  • Set log retention (30–90 days) — default is never expire = growing cost

Common Mistakes to Avoid

⚠️ Top Lambda Mistakes

❌ Creating new HttpClient in every invocation — socket exhaustion, port reuse issues
❌ Connecting to DB inside handler — reconnects on every call, slow + expensive
❌ No Reserved Concurrency — Lambda scale-out overwhelms RDS connection pool
❌ Timeout set to 15 minutes for all functions — slow failures, high billing for bugs
❌ Hardcoding secrets in env vars — visible in console and CloudTrail, use Secrets Manager
❌ $LATEST in production — use versioned aliases for stable deployments
❌ Not handling SQS partial batch failure — one bad message retries entire batch
❌ Ignoring cold start in latency-sensitive paths — p99 latency 10× worse than average
