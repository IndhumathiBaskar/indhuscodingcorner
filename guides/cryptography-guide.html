🔐 The Complete Cryptography Landscape

Cryptography is the science of protecting information. Every secure system — HTTPS, password storage, JWT tokens, file encryption, two-factor authentication — is built on a small set of well-understood primitives. Understanding which tool solves which problem is the most important skill.

💡 Golden Rule

Never roll your own crypto. Use vetted libraries. Choose the right algorithm. Most breaches aren't from broken math — they're wrong choices, bad key management, and missing integrity checks.

🔒 Symmetric Encryption — AES

Symmetric encryption uses the same key to encrypt and decrypt. AES (Advanced Encryption Standard) with GCM mode is the standard — giving you encryption + integrity in one pass.

GCM provides both encryption AND integrity in one pass. The 16-byte auth tag detects any tampering — no separate HMAC needed. Use Additional Authenticated Data (AAD) to bind context (user ID, version) to the ciphertext without encrypting it.

On decryption: split first 12 bytes as IV, last 16 as tag, middle as ciphertext. GCM verifies the tag automatically — throws AuthenticationTagMismatchException if tampered.

C# — AES-256-GCM with AAD (Additional Authenticated Data)

C# · AES-256-GCM · Encrypt + Decrypt + AAD 
using System.Security.Cryptography;
using System.Text;

public static class AesGcmHelper
{
    private const int KeySize   = 32;   // 256-bit key
    private const int NonceSize = 12;   // 96-bit nonce — GCM optimal
    private const int TagSize   = 16;   // 128-bit authentication tag

    public static byte[] GenerateKey()
        => RandomNumberGenerator.GetBytes(KeySize);

    // Encrypt → returns [nonce(12)] + [ciphertext] + [tag(16)]
    public static byte[] Encrypt(string plaintext, byte[] key,
        byte[]? aad = null)  // AAD = Additional Authenticated Data (not encrypted, but verified)
    {
        var plain  = Encoding.UTF8.GetBytes(plaintext);
        var nonce  = RandomNumberGenerator.GetBytes(NonceSize); // MUST be unique every call!
        var cipher = new byte[plain.Length];
        var tag    = new byte[TagSize];

        using var aes = new AesGcm(key, TagSize);
        aes.Encrypt(nonce, plain, cipher, tag, aad);
        // AAD is included in the authentication tag — any change = decryption failure

        var result = new byte[NonceSize + cipher.Length + TagSize];
        Buffer.BlockCopy(nonce,  0, result, 0,                        NonceSize);
        Buffer.BlockCopy(cipher, 0, result, NonceSize,                 cipher.Length);
        Buffer.BlockCopy(tag,    0, result, NonceSize + cipher.Length, TagSize);
        return result;
    }

    public static string Decrypt(byte[] encrypted, byte[] key,
        byte[]? aad = null)
    {
        var ctLen  = encrypted.Length - NonceSize - TagSize;
        var nonce  = encrypted[..NonceSize];
        var cipher = encrypted[NonceSize..(NonceSize + ctLen)];
        var tag    = encrypted[^TagSize..];
        var plain  = new byte[ctLen];

        using var aes = new AesGcm(key, TagSize);
        aes.Decrypt(nonce, cipher, tag, plain, aad);
        // ^ throws AuthenticationTagMismatchException if tampered or wrong AAD
        return Encoding.UTF8.GetString(plain);
    }
}

// Usage — AAD binds metadata to ciphertext without encrypting it
var key = AesGcmHelper.GenerateKey();
var aad = Encoding.UTF8.GetBytes("userId=u_9abc;version=1"); // not encrypted, but authenticated
var enc = AesGcmHelper.Encrypt("Team Watson secret", key, aad);
var dec = AesGcmHelper.Decrypt(enc, key, aad);  // must pass same AAD

⚠️ IV / Nonce Rules

Never reuse a nonce with the same key. In GCM, nonce reuse completely breaks confidentiality AND leaks the auth key. Always use RandomNumberGenerator.GetBytes(12) — never a counter, timestamp, or hardcoded value. With 12 random bytes the collision probability is negligible up to 2³² (~4 billion) encryptions per key.

🗝️ Asymmetric Encryption — RSA & ECC

Asymmetric crypto uses a mathematically linked key pair. Public key encrypts or verifies. Private key decrypts or signs. Solves the key-distribution problem but is ~100× slower than AES — so it's used to protect small payloads or keys, not bulk data.

Hybrid Encryption Pattern — The Real-World Way

📄 Any-size Plaintext
  • Could be MB or GB — RSA can't touch this directly
↓
🎲 Generate ephemeral AES-256 key
  • 32 random bytes, one-time use per message
🔒 Encrypt data with AES-GCM
  • Fast symmetric encryption for any size
↓

  • Only 32 bytes — fits RSA perfectly; only private key can unwrap
↓

  • Recipient: unwrap AES key with private key → decrypt ciphertext

C# — RSA-OAEP + ECDSA Signature

C# · RSA Hybrid Encrypt + ECDSA Sign/Verify 
using System.Security.Cryptography;

// ═══ RSA-OAEP: Hybrid Encrypt / Decrypt ═══
public static class RsaHelper
{
    public static (string pub, string priv) GenerateKeyPair(int bits = 2048)
    {
        using var rsa = RSA.Create(bits);
        return (rsa.ExportRSAPublicKeyPem(), rsa.ExportRSAPrivateKeyPem());
    }

    public static (byte[] wrappedKey, byte[] ciphertext) HybridEncrypt(
        string plaintext, string pubPem)
    {
        var aesKey  = AesGcmHelper.GenerateKey();
        var ct      = AesGcmHelper.Encrypt(plaintext, aesKey);
        using var rsa = RSA.Create(); rsa.ImportFromPem(pubPem);
        var wrapped = rsa.Encrypt(aesKey, RSAEncryptionPadding.OaepSHA256);
        return (wrapped, ct);
    }

    public static string HybridDecrypt(byte[] wrappedKey, byte[] ct, string privPem)
    {
        using var rsa = RSA.Create(); rsa.ImportFromPem(privPem);
        var aesKey = rsa.Decrypt(wrappedKey, RSAEncryptionPadding.OaepSHA256);
        return AesGcmHelper.Decrypt(ct, aesKey);
    }
}

// ═══ ECDSA: Sign / Verify (faster than RSA) ═══
public static class EcdsaHelper
{
    public static (string pub, string priv) GenerateKeyPair()
    {
        using var ec = ECDsa.Create(ECCurve.NamedCurves.nistP256);
        return (ec.ExportSubjectPublicKeyInfoPem(), ec.ExportPkcs8PrivateKeyPem());
    }

    public static byte[] Sign(byte[] data, string privPem)
    {
        using var ec = ECDsa.Create(); ec.ImportFromPem(privPem);
        return ec.SignData(data, HashAlgorithmName.SHA256);
    }

    public static bool Verify(byte[] data, byte[] sig, string pubPem)
    {
        using var ec = ECDsa.Create(); ec.ImportFromPem(pubPem);
        return ec.VerifyData(data, sig, HashAlgorithmName.SHA256);
    }
}

🧬 Hashing — SHA-2, SHA-3 & the Broken Ones

SHA-256 vs SHA-512 — When to Use Which

SHA-256 — Use When

  • General purpose integrity (checksums, fingerprints)
  • HMAC keys (256-bit output = correct HMAC size)
  • JWT HS256/RS256/ES256
  • 32-byte output is sufficient for your use case
  • 32-bit CPU or mobile — SHA-256 is faster here

SHA-512 — Use When

  • 64-byte output needed (e.g. HMAC-SHA512 for wider keys)
  • 64-bit CPU — SHA-512 is actually faster than SHA-256 on 64-bit
  • Higher security margin needed (512-bit vs 256-bit)
  • Key derivation needing wider output
  • JWT HS512 / RS512 / ES512
C# · SHA-256 · SHA-512 · SHA3 · File hash · Timing-safe compare 
using System.Security.Cryptography;
using System.Text;

public static class HashHelper
{
    public static byte[] Sha256(string s) => SHA256.HashData(Encoding.UTF8.GetBytes(s));
    public static byte[] Sha512(string s) => SHA512.HashData(Encoding.UTF8.GetBytes(s));
    public static byte[] Sha3_256(string s) => SHA3_256.HashData(Encoding.UTF8.GetBytes(s)); // .NET 8+

    // Hash a large file without loading it all into memory
    public static async Task<string> HashFileAsync(string path)
    {
        await using var stream = File.OpenRead(path);
        var hash = await SHA256.HashDataAsync(stream);
        return Convert.ToHexString(hash).ToLowerInvariant();
    }

    // ALWAYS use FixedTimeEquals — never string == — prevents timing attacks
    public static bool VerifyHash(byte[] expected, byte[] actual)
        => CryptographicOperations.FixedTimeEquals(expected, actual);

    public static string Sha256Hex(string s)
        => Convert.ToHexString(Sha256(s)).ToLowerInvariant();
}

❌ MD5 and SHA-1 — Cryptographically Broken

MD5 collision found in 1996. SHA-1 broken by Google SHAttered attack in 2017 (produced two real PDFs with identical SHA-1). Both banned by NIST. Fine for non-security checksums (deduplication, cache keys) but never for integrity, signatures, or password hashing.

🔑 Password Hashing — BCrypt, PBKDF2, Argon2

Password hashing ≠ regular hashing. SHA-256 computes billions/second on a GPU — trivially brute-forced. Password algorithms are intentionally slow and memory-hard, making attacks expensive.

💡 Why Salt is Essential

A salt is a unique random value added per password before hashing. Without salt: same password → same hash → rainbow table attacks crack thousands at once. With salt: each hash is unique even for identical passwords. BCrypt, PBKDF2, Argon2 include salt automatically — you don't manage it separately.

C# · BCrypt + PBKDF2 · Hash + Verify + Rehash 
// dotnet add package BCrypt.Net-Next
using BCrypt.Net;
using System.Security.Cryptography;

public static class PasswordHelper
{
    // BCrypt — work factor 12 ≈ 300ms. Increase yearly as hardware improves.
    public static string HashBCrypt(string password, int workFactor = 12)
        => BCrypt.HashPassword(password, workFactor);

    public static bool VerifyBCrypt(string password, string hash)
    {
        try { return BCrypt.Verify(password, hash); }
        catch (SaltParseException) { return false; }
    }

    public static bool NeedsRehash(string hash, int targetCost = 12)
        => BCrypt.PasswordNeedsRehash(hash, targetCost);

    // PBKDF2 — NIST/FIPS compliant, built-in .NET
    public static string HashPbkdf2(string password)
    {
        var salt = RandomNumberGenerator.GetBytes(16);
        var hash = Rfc2898DeriveBytes.Pbkdf2(
            password, salt, 600_000, HashAlgorithmName.SHA256, 32);
        return $"pbkdf2:600000:{Convert.ToBase64String(salt)}:{Convert.ToBase64String(hash)}";
    }

    public static bool VerifyPbkdf2(string password, string stored)
    {
        var p = stored.Split(':');
        var salt     = Convert.FromBase64String(p[2]);
        var expected = Convert.FromBase64String(p[3]);
        var actual   = Rfc2898DeriveBytes.Pbkdf2(
            password, salt, int.Parse(p[1]), HashAlgorithmName.SHA256, 32);
        return CryptographicOperations.FixedTimeEquals(expected, actual);
    }
}

// Login pattern — transparent rehash on successful login
var stored = db.GetPasswordHash(username);
if (PasswordHelper.VerifyBCrypt(inputPassword, stored))
{
    if (PasswordHelper.NeedsRehash(stored))
        db.UpdateHash(username, PasswordHelper.HashBCrypt(inputPassword));
    // grant access
}

🔏 HMAC & Digital Signatures

A plain hash has no secret — anyone can recompute it. HMAC adds a shared secret key. Digital Signatures use asymmetric keys — anyone with the public key can verify, only private key holder can sign.

🔐 HMAC — Symmetric Auth

  • Shared secret key — both parties need it
  • Proves sender knows the key
  • Very fast — hash-based
  • Use: webhook verification, JWT HS256, AWS Sig V4, API auth
  • ❌ Cannot prove which party signed (no non-repudiation)

✍️ Digital Signature — Asymmetric Auth

  • Private key signs — public key verifies
  • Anyone can verify without sharing a secret
  • Provides non-repudiation — signer cannot deny
  • Use: JWT RS256/ES256, TLS certs, code signing, legal docs
  • Slower — asymmetric math overhead
C# · HMAC-SHA256 Webhook Verify + RSA-PSS Sign/Verify 
using System.Security.Cryptography;
using System.Text;

public static class HmacHelper
{
    public static byte[] Sign(string message, byte[] key)
    {
        using var hmac = new HMACSHA256(key);
        return hmac.ComputeHash(Encoding.UTF8.GetBytes(message));
    }

    // Always constant-time — prevents timing attacks
    public static bool Verify(string message, byte[] expected, byte[] key)
        => CryptographicOperations.FixedTimeEquals(Sign(message, key), expected);

    // GitHub-style webhook: X-Hub-Signature-256: sha256=abc123...
    public static bool VerifyWebhook(string body, string sigHeader, string secret)
    {
        var mac      = Sign(body, Encoding.UTF8.GetBytes(secret));
        var expected = "sha256=" + Convert.ToHexString(mac).ToLowerInvariant();
        return CryptographicOperations.FixedTimeEquals(
            Encoding.ASCII.GetBytes(expected),
            Encoding.ASCII.GetBytes(sigHeader));
    }
}

public static class RsaSignHelper
{
    public static byte[] Sign(byte[] data, string privPem)
    {
        using var rsa = RSA.Create(); rsa.ImportFromPem(privPem);
        return rsa.SignData(data, HashAlgorithmName.SHA256, RSASignaturePadding.Pss);
    }

    public static bool Verify(byte[] data, byte[] sig, string pubPem)
    {
        using var rsa = RSA.Create(); rsa.ImportFromPem(pubPem);
        return rsa.VerifyData(data, sig, HashAlgorithmName.SHA256, RSASignaturePadding.Pss);
    }
}

🔑 Key Derivation & Key Wrapping

KDFs turn a low-entropy input (password, shared secret) into high-entropy cryptographic key material. Key Wrapping protects keys at rest using another key — the foundation of AWS KMS envelope encryption.

HKDF — Derive Multiple Keys from One Master

🌱 Input Key Material (IKM)
  • ECDH shared secret, master key, or random bytes
↓
HKDF-Extract(salt, IKM) → PRK
🧬 Pseudorandom Key (PRK) — 32 bytes
  • Uniform distribution regardless of IKM quality
↓
HKDF-Expand(PRK, info="enc/auth/iv", length)
🔒 Enc Key
  • info="encryption"
  • 32 bytes AES-256
🏷️ Auth Key
  • info="auth"
  • 32 bytes HMAC
🎫 IV Material
  • info="iv"
  • 12 bytes nonce

Envelope Encryption / Key Wrapping — AWS KMS Pattern

📄 Large Data to Encrypt
↓
🎲 Generate Data Encryption Key (DEK)
  • Random AES-256 key, used for this data only
🔑 Key Encryption Key (KEK)
  • Master key — lives in AWS KMS / HSM
  • Never leaves the secure boundary
↓
📦 Store: [AES-KW wrapped DEK] + [AES-GCM ciphertext]
  • To decrypt: send wrapped DEK to KMS → get DEK back → decrypt data
  • Rotating the KEK: re-wrap DEK without re-encrypting the entire dataset
C# · HKDF subkey derivation + Key Wrapping with AES-KeyWrap 
using System.Security.Cryptography;
using System.Text;

public static class KdfHelper
{
    // HKDF: derive separate enc + auth keys from one master
    public static (byte[] encKey, byte[] macKey) DeriveKeys(byte[] masterSecret)
    {
        var prk = HKDF.Extract(HashAlgorithmName.SHA256, masterSecret);
        var enc = HKDF.Expand(HashAlgorithmName.SHA256, prk, 32,
                      Encoding.UTF8.GetBytes("encryption"));
        var mac = HKDF.Expand(HashAlgorithmName.SHA256, prk, 32,
                      Encoding.UTF8.GetBytes("authentication"));
        return (enc, mac);
    }

    // PBKDF2: turn a user password into an AES key
    public static byte[] DeriveKeyFromPassword(string pw, byte[] salt, int len = 32)
        => Rfc2898DeriveBytes.Pbkdf2(pw, salt, 600_000, HashAlgorithmName.SHA256, len);
}

// Key Wrapping — protect a DEK with a KEK using AES-KW (RFC 3394)
public static class KeyWrapHelper
{
    // Wrap: encrypt the DEK with the KEK (AES-256 key wrapping)
    public static byte[] WrapKey(byte[] dek, byte[] kek)
    {
        // AES key wrap adds 8-byte integrity check + pads to 8-byte boundary
        using var aes = Aes.Create();
        aes.Key = kek;
        return aes.EncryptEcb(dek, PaddingMode.None);
        // Real AES-KW: use BouncyCastle AesWrapEngine or AWS KMS GenerateDataKey API
    }

    // Full envelope encryption pattern
    public static (byte[] wrappedDek, byte[] ciphertext) EnvelopeEncrypt(
        string plaintext, byte[] kek)
    {
        var dek        = AesGcmHelper.GenerateKey();           // ephemeral DEK
        var ciphertext = AesGcmHelper.Encrypt(plaintext, dek); // encrypt data
        var wrappedDek = WrapKey(dek, kek);                    // protect DEK with KEK
        Array.Clear(dek, 0, dek.Length);                       // zero DEK from memory
        return (wrappedDek, ciphertext);
    }
}

using System.Security.Cryptography;
using System.Security.Cryptography.X509Certificates;

public static class CertHelper
{
    // ── Load certificate from PEM file ──
    public static X509Certificate2 LoadCert(string certPemPath)
        => X509Certificate2.CreateFromPemFile(certPemPath);

    // ── Load cert + private key (for signing or TLS server) ──
    public static X509Certificate2 LoadCertWithKey(string certPem, string keyPem)
        => X509Certificate2.CreateFromPemFile(certPem, keyPem);

    // ── Inspect certificate fields ──
    public static void Inspect(X509Certificate2 cert)
    {
        Console.WriteLine($"Subject:    {cert.Subject}");
        Console.WriteLine($"Issuer:     {cert.Issuer}");
        Console.WriteLine($"Valid from: {cert.NotBefore:u}");
        Console.WriteLine($"Valid to:   {cert.NotAfter:u}");
        Console.WriteLine($"Thumbprint: {cert.Thumbprint}");    // SHA-1 of cert bytes
        Console.WriteLine($"SerialNum:  {cert.SerialNumber}");
        Console.WriteLine($"Key alg:    {cert.PublicKey.Oid.FriendlyName}");
        Console.WriteLine($"Expired:    {cert.NotAfter < DateTime.UtcNow}");
    }

    // ── Validate a certificate chain ──
    public static bool ValidateChain(X509Certificate2 cert)
    {
        using var chain = new X509Chain();
        chain.ChainPolicy.RevocationMode    = X509RevocationMode.Online;  // check OCSP/CRL
        chain.ChainPolicy.VerificationFlags = X509VerificationFlags.NoFlag;
        bool valid = chain.Build(cert);
        if (!valid)
            foreach (var status in chain.ChainStatus)
                Console.WriteLine($"Chain error: {status.StatusInformation}");
        return valid;
    }

    // ── Sign data with the cert's private key (ECDSA or RSA) ──
    public static byte[] SignData(byte[] data, X509Certificate2 certWithKey)
    {
        if (certWithKey.GetECDsaPrivateKey() is { } ec)
            return ec.SignData(data, HashAlgorithmName.SHA256);
        if (certWithKey.GetRSAPrivateKey() is { } rsa)
            return rsa.SignData(data, HashAlgorithmName.SHA256, RSASignaturePadding.Pss);
        throw new InvalidOperationException("No supported private key found in certificate");
    }

    // ── Generate a self-signed certificate (for dev/testing) ──
    public static X509Certificate2 CreateSelfSigned(string commonName)
    {
        using var ec  = ECDsa.Create(ECCurve.NamedCurves.nistP256);
        var request   = new CertificateRequest(
            $"CN={commonName}", ec, HashAlgorithmName.SHA256);

        request.CertificateExtensions.Add(
            new X509SubjectAlternativeNameExtension(new SubjectAlternativeNameBuilder()
                .AddDnsName(commonName).Build()));

        return request.CreateSelfSigned(
            DateTimeOffset.UtcNow, DateTimeOffset.UtcNow.AddYears(1));
    }
}

ChaCha20-Poly1305 was designed by Daniel Bernstein as an alternative to AES when hardware acceleration (AES-NI) isn't available. On mobile CPUs without AES-NI, it's 3× faster than AES-GCM. Used in TLS 1.3, WireGuard, Signal Protocol, and SSH.

✅ Use ChaCha20 When

  • Building a messaging app (Signal-style)

  • You want a software-only implementation

✅ Use AES-GCM When

  • Server-side with modern x86/x64 CPUs (AES-NI is standard)
  • FIPS-140 compliance required (AES is FIPS certified)

ECDH — Elliptic Curve Diffie-Hellman Key Exchange

ECDH lets two parties who have never communicated before derive a shared secret over an insecure channel — without transmitting the secret. Both parties generate an ephemeral key pair, exchange public keys, and independently compute the same shared secret.

👤 Alice generates ECDH key pair
  • Private key: a (secret)
  • Public key: A = a×G (sends to Bob)
👤 Bob generates ECDH key pair
  • Private key: b (secret)
  • Public key: B = b×G (sends to Alice)
↓
Exchange public keys over insecure channel (anyone can see A and B)
🔑 Alice computes: S = a×B
  • = a × (b×G)
🔑 Bob computes: S = b×A
  • = b × (a×G) = same point!
↓
🔑 Both have identical shared secret S
  • Feed through HKDF → derive AES encryption key
  • Ephemeral keys discarded → forward secrecy
  • Even if long-term private key is compromised later, past sessions are safe
C# · ECDH Key Exchange + ChaCha20-Poly1305 (via NSec / Libsodium) 
using System.Security.Cryptography;

// ════ ECDH: agree a shared secret between two parties ════
public static class EcdhHelper
{
    // Alice side: generate ephemeral key pair
    public static (ECDiffieHellman ecdh, byte[] publicKeyBytes) GenerateKeyPair()
    {
        var ecdh      = ECDiffieHellman.Create(ECCurve.NamedCurves.nistP256);
        var pubBytes  = ecdh.PublicKey.ExportSubjectPublicKeyInfo();
        return (ecdh, pubBytes);
    }

    // Derive shared secret from own private key + other party's public key
    public static byte[] DeriveSharedSecret(
        ECDiffieHellman myKey, byte[] theirPublicKeyBytes)
    {
        using var theirKey = ECDiffieHellman.Create();
        theirKey.ImportSubjectPublicKeyInfo(theirPublicKeyBytes, out _);
        var sharedSecret = myKey.DeriveRawSecretAgreement(theirKey.PublicKey);
        // ALWAYS feed through HKDF — raw shared secret has bias
        return HKDF.DeriveKey(HashAlgorithmName.SHA256, sharedSecret,
                              outputLength: 32,
                              salt:         null,
                              info:         System.Text.Encoding.UTF8.GetBytes("session-key-v1"));
    }
}

// Full ECDH handshake + AES-GCM encryption of message
var (aliceKey, alicePub) = EcdhHelper.GenerateKeyPair();
var (bobKey,   bobPub)   = EcdhHelper.GenerateKeyPair();

// Both derive the same 32-byte session key independently
var aliceSessionKey = EcdhHelper.DeriveSharedSecret(aliceKey, bobPub);
var bobSessionKey   = EcdhHelper.DeriveSharedSecret(bobKey,   alicePub);
// aliceSessionKey == bobSessionKey ✓

// Now use the session key for AES-GCM encryption
var encrypted = AesGcmHelper.Encrypt("Hello Bob, this is Alice!", aliceSessionKey);
var decrypted = AesGcmHelper.Decrypt(encrypted, bobSessionKey);

✅ Forward Secrecy — Why Ephemeral Keys Matter

When ECDH uses ephemeral (one-time) key pairs that are discarded after each session, compromising the long-term private key doesn't expose past sessions — the attacker would need the ephemeral key which is gone. TLS 1.3 mandates ephemeral key exchange (ECDHE) precisely for this reason. This is why TLS 1.3 is fundamentally safer than TLS 1.2 with RSA key exchange.

📱 TOTP & HOTP — One-Time Passwords

TOTP (Time-based OTP) is the algorithm behind Google Authenticator, Authy, and Microsoft Authenticator. It generates a 6-digit code that changes every 30 seconds using a shared secret and the current time — providing strong second-factor authentication without SMS.

HOTP → TOTP — How They Differ

HOTP — Counter-based (RFC 4226)

  • OTP = HMAC-SHA1(secret, counter)
  • Counter increments on each use
  • Code valid until used — never expires
  • Requires counter sync between client and server
  • Use: hardware tokens, YubiKey (OTP mode)

TOTP — Time-based (RFC 6238)

  • Counter = floor(UnixTime / 30)
  • OTP = HMAC-SHA1(secret, timeStep)
  • Code changes every 30 seconds automatically
  • Both sides just need clock sync (NTP)
  • Use: Google Authenticator, Microsoft Authenticator

C# · TOTP from scratch · Generate + Verify + QR URI 
using System.Security.Cryptography;
using System.Text;

public static class TotpHelper
{
    private const int Step    = 30;   // seconds per time window
    private const int Digits  = 6;    // length of OTP code
    private const int Window  = 1;    // allow ±1 window for clock drift

    // Generate a random TOTP secret (store server-side, show to user as QR)
    public static byte[] GenerateSecret() => RandomNumberGenerator.GetBytes(20);

    // Base32 encode — for QR code / manual entry
    public static string ToBase32(byte[] data)
    {
        const string alphabet = "ABCDEFGHIJKLMNOPQRSTUVWXYZ234567";
        var sb = new StringBuilder();
        int buffer = data[0], next = 1, bitsLeft = 8;
        while (bitsLeft > 0 || next < data.Length) {
            if (bitsLeft < 5) {
                if (next < data.Length) { buffer <<= 8; buffer |= data[next++]; bitsLeft += 8; }
                else { int pad = 5 - bitsLeft; buffer <<= pad; bitsLeft += pad; }
            }
            int index = 0x1F & (buffer >> (bitsLeft - 5));
            bitsLeft -= 5;
            sb.Append(alphabet[index]);
        }
        return sb.ToString();
    }

    // Core HOTP: HMAC-SHA1(secret, counter) → 6-digit code
    private static int Hotp(byte[] secret, long counter)
    {
        var counterBytes = BitConverter.GetBytes(counter);
        if (BitConverter.IsLittleEndian) Array.Reverse(counterBytes); // must be big-endian
        using var hmac   = new HMACSHA1(secret);
        var hash         = hmac.ComputeHash(counterBytes);
        int offset       = hash[^1] & 0x0F;                         // dynamic truncation
        int code         = (hash[offset] & 0x7F) << 24
                         | (hash[offset+1] & 0xFF) << 16
                         | (hash[offset+2] & 0xFF) << 8
                         |  (hash[offset+3] & 0xFF);
        return code % (int)Math.Pow(10, Digits);
    }

    // Generate current TOTP code
    public static string Generate(byte[] secret)
    {
        long counter = DateTimeOffset.UtcNow.ToUnixTimeSeconds() / Step;
        return Hotp(secret, counter).ToString($"D{Digits}");
    }

    // Verify — check current window ± drift window (handles clock skew)
    public static bool Verify(byte[] secret, string code)
    {
        long counter = DateTimeOffset.UtcNow.ToUnixTimeSeconds() / Step;
        for (long i = -Window; i <= Window; i++)
        {
            var expected = Hotp(secret, counter + i).ToString($"D{Digits}");
            if (CryptographicOperations.FixedTimeEquals(
                    Encoding.ASCII.GetBytes(code),
                    Encoding.ASCII.GetBytes(expected)))
                return true;
        }
        return false;
    }

    // OTPAuth URI — scan this as QR code to add to Google Authenticator
    public static string GetOtpAuthUri(byte[] secret, string issuer, string account)
        => $"otpauth://totp/{Uri.EscapeDataString(issuer)}:{Uri.EscapeDataString(account)}"
         + $"?secret={ToBase32(secret)}&issuer={Uri.EscapeDataString(issuer)}&digits=6&period=30";
}

// Usage
var secret = TotpHelper.GenerateSecret();    // store in DB for this user
var uri    = TotpHelper.GetOtpAuthUri(secret, "MyApp", "user@example.com");
// Convert uri to QR code — user scans with Google Authenticator
var code   = TotpHelper.Generate(secret);     // "482910"
var valid  = TotpHelper.Verify(secret, "482910"); // true within 30s window

⚠️ TOTP Security Rules

Store the secret securely (encrypted at rest). Mark each code as used to prevent replay attacks within the same 30-second window. Offer backup codes in case the user loses their device. Log failed attempts — 3 failures in a row should lock out TOTP temporarily. Never transmit the secret over insecure channels — always QR code via HTTPS.

❌ System.Random is Predictable

System.Random is seeded from the clock — an attacker knowing your startup time can enumerate the seed space and predict every value produced. Always use RandomNumberGenerator for nonces, salts, tokens, session IDs, OTPs, and keys.

✅ Best Practices & What Never to Do

✅ Always Do These

  • Use AES-256-GCM for symmetric encryption — authenticated, fast, NIST approved
  • Generate a fresh random nonce for every single AES-GCM encryption — never reuse
  • Use RandomNumberGenerator — never System.Random — for all crypto values
  • Compare hashes/MACs with CryptographicOperations.FixedTimeEquals() — never ==
  • Use BCrypt, Argon2id, or PBKDF2 ≥600,000 iterations for password storage
  • Store the salt alongside the hash — it's not secret, just unique per user
  • Use HKDF to derive separate enc and auth subkeys — never reuse same key for two purposes
  • Store keys in AWS Secrets Manager / Azure Key Vault / HSM — never in code or config
  • Use X.509 certs with SAN (not just CN) — modern browsers require SAN for HTTPS
  • Enable certificate transparency + OCSP stapling on production HTTPS servers
  • Use ephemeral ECDH (ECDHE) for key exchange — provides forward secrecy
  • Use TLS 1.3 — disable TLS 1.0 and 1.1, prefer disabling TLS 1.2 where possible
  • Mark each TOTP code as used — prevent replay within same 30-second window
  • Use envelope encryption for large datasets — DEK encrypts data, KEK protects DEK
  • Include algorithm + parameters in stored hashes — enables upgrading work factor later

❌ Never Do These

  • Never roll your own encryption — use vetted, peer-reviewed implementations only
  • Never use MD5 or SHA-1 for security — both are broken (real collisions demonstrated)
  • Never use AES-ECB — same input block → same output block, patterns leak through
  • Never reuse an IV/nonce with the same key — in GCM this is catastrophic
  • Never use plain SHA-256 to store passwords — too fast, no work factor, no salt
  • Never compare security values with == or string equality — timing attacks
  • Never store encryption keys next to the data they protect
  • Never use RSA PKCS#1 v1.5 padding — use OAEP for enc, PSS for signing
  • Never use a self-signed cert in production — users get security warnings, train to ignore them
  • Never log decrypted data, plaintext passwords, or raw keys — ever
  • Never use SMS-based OTP as a primary 2FA — SIM-swapping attacks are common
  • Never trust a certificate without verifying the full chain up to a trusted root

📁 PGP / OpenPGP — Hybrid File Encryption

PGP (Pretty Good Privacy) is a hybrid encryption system that combines the best of symmetric and asymmetric cryptography — AES for fast bulk encryption and RSA/ECC for secure key exchange. It's the standard for secure file transfer, encrypted email, and cross-organisation data exchange.

💡 Why Hybrid? Why Not Just RSA or Just AES?

AES only: Fast ✅ but how do you safely share the secret key? ❌
RSA only: Safe key sharing ✅ but too slow for large files ❌
PGP hybrid: AES encrypts the file (fast), RSA encrypts the AES key (safe) ✅ Best of both worlds.

  • Contains: metadata + encrypted AES key + encrypted content + signature
  • Safe to send over email, FTP, S3, any channel

📬 Bob receives .pgp file
↓
🔑 Bob uses his PGP private key to decrypt the AES session key
  • Private key never leaves Bob's machine — only the AES key is unwrapped
↓
🔓 AES session key decrypts the file content
  • Original file recovered exactly
↓
✅ Optional: Verify Alice's signature with her public key
  • Confirms authenticity — file really came from Alice, unchanged

PGP Key Pair — Public vs Private

🔓 PGP Public Key — Share Freely

  • Distributed to anyone who wants to send you encrypted files
  • Published on key servers (keys.openpgp.org)
  • Used by senders to encrypt the AES session key
  • Used by verifiers to check your signatures
  • Safe to email, post on website, share in API responses

🔐 PGP Private Key — Keep Secret

  • Never shared — stays on your machine only
  • Always protected with a strong passphrase
  • Used to decrypt received files (unwrap AES key)
  • Used to sign files you send
  • Loss = cannot decrypt old files. Compromise = all past files exposed.

Why PGP Is Ideal for Enterprise File Exchange

✅ PGP Strengths

  • No shared secrets — each party keeps their own private key
  • Works across organisations — just exchange public keys
  • Scales to multiple recipients — encrypt the same AES key for each
  • Optional signature proves sender identity
  • Long history (1991) — widely supported, battle-tested
  • Used in healthcare, banking, pharmacy, legal sectors

⚠️ PGP Limitations

  • Key management can be complex at scale
  • No perfect forward secrecy (unlike TLS with ECDHE)
  • Web of Trust model is hard to manage in large orgs
  • Private key passphrase loss = permanent data loss
  • Not designed for real-time communication (use TLS for that)

    public static async Task EncryptAndSignAsync(
        string inputFilePath,
        string outputPgpPath,
        string recipientPublicKeyPath,
        string senderPrivateKeyPath,
        string senderPassphrase)
    {
        var encryptionKeys = new EncryptionKeys(
            new FileInfo(recipientPublicKeyPath),
            new FileInfo(senderPrivateKeyPath),
            senderPassphrase);

        using var pgp = new PGP(encryptionKeys);
        await pgp.EncryptFileAndSignAsync(
            new FileInfo(inputFilePath),
            new FileInfo(outputPgpPath));
    }

    // ── 5. Decrypt AND verify signature ──
    public static async Task<bool> DecryptAndVerifyAsync(
        string inputPgpPath,
        string outputFilePath,
        string recipientPrivateKeyPath,
        string recipientPassphrase,
        string senderPublicKeyPath)
    {
        var encryptionKeys = new EncryptionKeys(
            new FileInfo(senderPublicKeyPath),      // to verify signature
            new FileInfo(recipientPrivateKeyPath),  // to decrypt
            recipientPassphrase);

        using var pgp = new PGP(encryptionKeys);
        await pgp.DecryptFileAndVerifyAsync(
            new FileInfo(inputPgpPath),
            new FileInfo(outputFilePath));
        return true; // throws if signature invalid
    }

    // ── 6. Encrypt in-memory stream (no files — useful in APIs) ──
    public static async Task<Stream> EncryptStreamAsync(
        Stream inputStream, string recipientPublicKeyPath)
    {
        var outputStream = new MemoryStream();
        var encryptionKeys = new EncryptionKeys(
            new FileInfo(recipientPublicKeyPath));
        using var pgp = new PGP(encryptionKeys);
        await pgp.EncryptStreamAsync(inputStream, outputStream);
        outputStream.Position = 0;
        return outputStream;
    }
}

// ── Usage ──
// 1. Generate once per user/organisation
await PgpHelper.GenerateKeyPairAsync(
    "Bob <bob@example.com>", "str0ngPassphr@se",
    "bob_public.asc", "bob_private.asc");

    "bob_private.asc", "str0ngPassphr@se", "alice_public.asc");

❌ Common PGP Mistakes

❌ Using legacy algorithms (CAST5, IDEA, 3DES) — always configure AES-256
❌ RSA-1024 keys — minimum RSA-2048, prefer ECC P-256
❌ No passphrase on private key — if stolen, attacker gets everything
❌ Encrypting data directly with RSA — PGP uses hybrid, never raw RSA on file content
❌ Implementing PGP from scratch — use PgpCore or BouncyCastle
❌ Never rotating keys — rotate annually and when a team member leaves

  • Use AES-256 + RSA-2048+ or ECC P-256+ in your PGP configuration
  • Always protect private keys with a strong passphrase (20+ chars)
  • Use encrypt-and-sign together for full confidentiality + authenticity
  • Store private keys in secure locations — HSM, encrypted vault, or OS certificate store
  • Rotate PGP key pairs annually and immediately after any suspected compromise
  • Use a trusted library (PgpCore, BouncyCastle) — never hand-roll PGP

🪟 Windows DPAPI — Platform Secret Protection

DPAPI (Data Protection API) is a Windows built-in service that encrypts secrets without you managing encryption keys. Windows ties the encryption to the user account or machine identity — making it the safest way to store secrets in Windows-based applications.

💡 The Problem DPAPI Solves

You've learned AES encryption — but where do you store the AES key? If you hard-code it or put it in a config file, an attacker who gets the file gets the key too. DPAPI solves this by making Windows itself the key manager — the key is derived from your Windows credentials and never stored anywhere you can see it.

DPAPI Internals — What Windows Does for You

📝 Your Secret (connection string, API key, password)
  • Plain bytes that need protecting at rest
↓
🏛️ Windows Derives a Master Key
  • CurrentUser: derived from Windows user account credentials (NTLM hash)
  • LocalMachine: derived from machine-specific secrets stored in LSA
  • You never see or store this key — Windows manages it entirely
↓
🔒 AES-256 + SHA-512 HMAC Encryption
  • Windows uses AES internally for encryption
  • HMAC protects integrity — detects tampering
  • Optional entropy adds an extra layer (like a second secret)
↓
📦 Encrypted BLOB — Safe to Store
  • Store in config file, registry, database, or app settings
  • Meaningless without the same Windows user/machine context

Two Protection Scopes — CurrentUser vs LocalMachine

👤 CurrentUser Scope

  • Encryption tied to the specific Windows user account
  • Only that user — on any machine they log into — can decrypt
  • If you copy the encrypted blob to another machine under same user account with roaming profile, it still works
  • Best for: desktop apps, per-user secrets, developer tools
  • Example: VS Code stores Git credentials this way

🖥️ LocalMachine Scope

  • Encryption tied to the specific machine
  • Any process running on that machine can decrypt
  • Copy the blob to another machine → cannot decrypt
  • Best for: Windows services, IIS apps, on-premise servers
  • ⚠️ Lower isolation — any local admin can decrypt

Optional Entropy — Extra Security Layer

🧂 What is Optional Entropy?

Optional entropy is like a second secret password that DPAPI requires during both encrypt and decrypt. Without it, any process running as the same user can decrypt your data. With entropy, an attacker also needs to know the entropy value — adding a second layer of protection. Think of it as a salt that the caller must supply.

ProtectedData.Protect(secret, optionalEntropy, scope)
// optionalEntropy = extra bytes both sides must agree on
// Same entropy required on Unprotect — acts like a second factor

C# — Full DPAPI Examples

C# · Windows DPAPI · Protect · Unprotect · Scope · Entropy · Connection String Pattern 
using System.Security.Cryptography;
using System.Text;

public static class DpapiHelper
{
    // ── Encrypt a secret string with DPAPI ──
    public static string Protect(
        string secret,
        DataProtectionScope scope = DataProtectionScope.CurrentUser,
        string? entropy = null)    // optional extra secret
    {
        var secretBytes  = Encoding.UTF8.GetBytes(secret);
        var entropyBytes = entropy != null
            ? Encoding.UTF8.GetBytes(entropy)
            : null;

        var encrypted = ProtectedData.Protect(
            secretBytes,
            entropyBytes,   // null = no extra entropy
            scope);         // CurrentUser or LocalMachine

        return Convert.ToBase64String(encrypted);
        // Store this Base64 string in config / registry / DB — it's safe
    }

    // ── Decrypt a DPAPI-protected secret ──
    public static string Unprotect(
        string encryptedBase64,
        DataProtectionScope scope = DataProtectionScope.CurrentUser,
        string? entropy = null)    // must match what was used in Protect
    {
        var encryptedBytes = Convert.FromBase64String(encryptedBase64);
        var entropyBytes   = entropy != null
            ? Encoding.UTF8.GetBytes(entropy)
            : null;

        var decryptedBytes = ProtectedData.Unprotect(
            encryptedBytes,
            entropyBytes,
            scope);

        return Encoding.UTF8.GetString(decryptedBytes);
        // throws CryptographicException if wrong user/machine/entropy
    }

    // ── Protect raw bytes (for keys, tokens etc.) ──
    public static byte[] ProtectBytes(byte[] data, DataProtectionScope scope)
        => ProtectedData.Protect(data, null, scope);

    public static byte[] UnprotectBytes(byte[] encrypted, DataProtectionScope scope)
        => ProtectedData.Unprotect(encrypted, null, scope);
}

// Pattern 1: Protect a DB connection string (run once at setup)
var connStr    = "Server=db01;User=admin;Password=Secret123;";
var encrypted  = DpapiHelper.Protect(connStr, DataProtectionScope.LocalMachine);
// Store encrypted in appsettings.json → "ConnectionString": "AQAAANCMnd8..."

// Pattern 2: Read it back at runtime

var plainConn      = DpapiHelper.Unprotect(encFromConfig, DataProtectionScope.LocalMachine);
// Only works on the same machine — other machines cannot decrypt

// Pattern 3: Protect an API key with entropy (extra security)

var appEntropy     = "MyApp-v2-ApiKeyProtection"; // agreed by your app — not stored with data
var encApiKey      = DpapiHelper.Protect(apiKey, DataProtectionScope.CurrentUser, appEntropy);
var decApiKey      = DpapiHelper.Unprotect(encApiKey, DataProtectionScope.CurrentUser, appEntropy);

// Pattern 4: Safe memory handling — zero out after use
string? secret = null;
try
{
    secret = DpapiHelper.Unprotect(encFromConfig, DataProtectionScope.LocalMachine);
    // Use secret briefly here...
    UseSecret(secret);
}
finally
{
    // Clear from memory when done
    secret = null;
    GC.Collect();
}

What DPAPI Protects Against vs Not

✅ DPAPI Protects Against

  • Reading secrets from stolen config files or databases
  • Copying encrypted blob to another machine (LocalMachine)
  • Another user on the same machine (CurrentUser)
  • Basic malware scanning configuration files at rest
  • Programmatic extraction without Windows credentials

❌ DPAPI Does NOT Protect Against

  • Admin-level or SYSTEM-level compromise
  • Malware running as the same Windows user
  • Memory dumps while secret is in use (in-memory)
  • Cross-platform attacks (Linux, Docker, cloud)
  • Decryption on the same machine by any local admin (LocalMachine scope)

When to Use DPAPI

✅ Use DPAPI When

  • Windows-only application (desktop, WinForms, WPF, console)
  • On-premise Windows server (IIS, Windows Service)
  • Protecting secrets on a developer machine
  • Avoiding key management complexity
  • ASP.NET Core Data Protection (uses DPAPI internally on Windows)

⚠️ Do NOT Use DPAPI When

  • Running in Docker containers (no Windows user context)
  • Deploying to Linux (DPAPI is Windows-only)
  • Need to share encrypted data across multiple machines
  • Cloud-native apps (use AWS Secrets Manager or Azure Key Vault)
  • Need audit logs of secret access (DPAPI has no audit trail)
  • Use DataProtectionScope.CurrentUser for desktop apps — better isolation than LocalMachine
  • Use DataProtectionScope.LocalMachine for Windows Services and IIS apps
  • Add optional entropy to bind secrets to your specific application — prevents other apps on same machine decrypting
  • Encrypt once at setup/deployment — decrypt only at runtime when needed
  • Zero out the decrypted string from memory as soon as you're done with it
  • Combine with TLS (for transit) and file-system ACLs (restrict who can read the encrypted file)
  • For cloud/cross-platform — migrate to Azure Key Vault or AWS Secrets Manager instead

⚠️ ASP.NET Core Data Protection

ASP.NET Core has a built-in Data Protection API (IDataProtector) that uses DPAPI internally on Windows. It handles key rotation, purpose isolation, and time-limited tokens. For ASP.NET Core apps, prefer services.AddDataProtection() over calling DPAPI directly — it's higher-level and handles key management automatically.
