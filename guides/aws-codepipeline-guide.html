⚡ What is AWS CodePipeline?

AWS CodePipeline is a fully managed continuous delivery (CD) service that automates your release process. Every time you push code, CodePipeline automatically runs your build, tests, approval gates, and deployments in sequence. It orchestrates the entire flow — connecting CodeCommit, CodeBuild, CodeDeploy, CloudFormation, Lambda, ECS, and third-party tools into a single automated pipeline.

💡 The Core Mental Model

CodePipeline is the conductor. CodeCommit provides the source, CodeBuild compiles and tests, CodeDeploy delivers to servers, CloudFormation updates infrastructure. CodePipeline sequences and coordinates all of them — you define the flow once and it runs automatically on every push.

Key Concepts

⚡

Pipeline

The top-level entity. A series of stages that run in sequence automatically.

📌

Stage

A logical phase (Source, Build, Test, Deploy). Each stage has one or more actions.

▶️

Action

The actual work — invoke CodeBuild, deploy with CloudFormation, approve manually, etc.

📦

Artifact

Files passed between stages via S3. Each action can have input and output artifacts.

🔀

Parallel Actions

Multiple actions in a stage run in parallel — e.g., unit tests + linting simultaneously.

🛑

Manual Approval

Pause the pipeline and wait for a human to approve before proceeding to production.

🔔

Notification

EventBridge events on pipeline state changes — notify Slack, email, or trigger Lambda.

📊

Execution

One run of the pipeline. Has its own status, artifacts, and execution ID.

🌍

Multi-Region

Deploy the same artifact to multiple AWS regions in one pipeline execution.

📌 Stages & Actions — Building Blocks

Every pipeline is composed of Stages. Every stage contains one or more Actions. Actions in the same stage run in parallel by default; stages run sequentially.

Action Categories

📦 Source
Category: Source
Watches for changes and pulls the revision. Triggers the pipeline. Providers: CodeCommit, GitHub, S3, ECR.
🔨 Build
Category: Build
Compiles, packages, and runs unit tests. Providers: CodeBuild, Jenkins.
🧪 Test
Category: Test
Run integration or end-to-end tests. Providers: CodeBuild, Device Farm, BlazeMeter.
🚀 Deploy
Category: Deploy
Deploy artifacts. Providers: CodeDeploy, CloudFormation, ECS, S3, Elastic Beanstalk, AppConfig.
✅ Approval
Category: Approval
Pause pipeline and wait for human approval. Sends SNS notification with review URL and custom message.
⚡ Invoke
Category: Invoke
Call a Lambda function or Step Functions state machine as a pipeline action. Useful for custom logic.

📊 Variable Scopes — Where Variables Come From
Pipeline Variables
Defined at pipeline level, set at start of execution. Passed to any stage.#{variables.ENVIRONMENT} → prod
Source Namespace
Automatically populated by the Source action with commit metadata.#{SourceVariables.CommitId} → abc1234
Action Namespace
Output variables exported by an action (CodeBuild, Lambda, etc.) for downstream use.#{BuildVariables.IMAGE_TAG} → 1.2.3-abc1234
Execution ID
Built-in — the current pipeline execution ID. Always available.#{codepipeline.PipelineExecutionId} → exec-abc123

C# · Lambda Function as a CodePipeline Action using Amazon.CodePipeline; using Amazon.CodePipeline.Model; using Amazon.Lambda.Core; using System.Text.Json; public class SmokeTestFunction { private readonly IAmazonCodePipeline _cp; private readonly HttpClient _http; public SmokeTestFunction() { _cp = new AmazonCodePipelineClient(); _http = new HttpClient(); } public async Task HandleAsync( IDictionary<string, JsonElement> input, ILambdaContext context) { // CodePipeline passes job details to the Lambda var jobId = input["CodePipeline.job"] .GetProperty("id").GetString()!; var jobData = input["CodePipeline.job"] .GetProperty("data"); var userParams = jobData .GetProperty("actionConfiguration") .GetProperty("configuration") .GetProperty("UserParameters") .GetString()!; var config = JsonSerializer .Deserialize<SmokeTestConfig>(userParams)!; try { context.Logger.LogLine( $"Running smoke tests on {config.ApiUrl}"); // Run your validation var resp = await _http.GetAsync( $"{config.ApiUrl}/health"); resp.EnsureSuccessStatusCode(); context.Logger.LogLine("✅ Smoke tests PASSED"); // MUST call PutJobSuccessResult or PutJobFailureResult await _cp.PutJobSuccessResultAsync(new PutJobSuccessResultRequest { JobId = jobId, OutputVariables = new Dictionary<string, string> { ["HEALTH_CHECK_STATUS"] = "PASSED" } }); } catch (Exception ex) { context.Logger.LogLine( $"❌ Smoke tests FAILED: {ex.Message}"); await _cp.PutJobFailureResultAsync(new PutJobFailureResultRequest { JobId = jobId, FailureDetails = new FailureDetails { Type = FailureType.JobFailed, Message = ex.Message } }); } } } public record SmokeTestConfig(string Env, string ApiUrl);

Execution Modes (V2 Only)

🔁 QUEUED

New executions wait in a queue if the pipeline is already running. Each execution runs in order. Safe for sequential deploys — no commits skipped.

⚡ PARALLEL

Multiple executions can run simultaneously. Good for feature branch pipelines where each branch runs independently.

🔄 SUPERSEDED

If a newer execution starts, the older one is abandoned. Fastest — always deploys the latest commit. Good for rapid iteration pipelines.

Best Practices & Common Mistakes

  • Always use V2 pipelines for all new projects — variables, trigger filters, and execution modes are essential features
  • Use EventBridge detection (not polling) for CodeCommit sources — triggers in seconds, not minutes, and is cheaper
  • Use QUEUED execution mode for production pipelines — prevents concurrent deploys that can cause race conditions
  • Store all pipeline definitions in CloudFormation / SAM templates in Git — version-controlled infrastructure
  • Add a Manual Approval stage before any production deploy — even a 2-minute window to review reduces incidents dramatically
  • Use source namespaces (e.g., #{SourceVariables.CommitId}) in approval messages — reviewers see exactly what they're approving
  • Set up notification rules on pipeline failure and approval required — team should never have to poll the console
  • Scope the pipeline IAM role to minimum permissions — separate roles for CodeBuild, CloudFormation deploy, and CodeDeploy

⚠️ Top CodePipeline Mistakes

❌ Using polling instead of EventBridge for CodeCommit — 1 min delay + extra API call costs every minute
❌ No manual approval before prod — every commit auto-deploys; one bad push takes down production
❌ Overly permissive pipeline IAM role — pipeline role with Admin access can deploy anything, creating a privilege escalation risk
❌ Not storing pipeline in CloudFormation — manually created pipelines are hard to review, audit, and recreate after incidents
❌ Forgetting PutJobSuccessResult/PutJobFailureResult in Lambda invoke actions — pipeline hangs for the full timeout then fails
❌ Using V1 pipelines for new projects — missing variables, trigger filters, and parallel execution modes
❌ Single artifact bucket for multi-region pipeline — cross-region actions require artifact buckets in each target region
