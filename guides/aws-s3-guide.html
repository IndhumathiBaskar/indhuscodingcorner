🪣 What is AWS S3?

S3 = Simple Storage Service. Infinitely scalable object storage in the cloud. Store any file — images, videos, backups, logs, CSVs, HTML pages — and access them from anywhere.

💡 Simple Analogy

S3 is like Google Drive for your applications. A Bucket is a folder. An Object is a file. You can store unlimited files, each up to 5 TB. You pay only for what you use.

How S3 Works — Overview

👤 Your Application / User
  • Upload a file (PUT request)
  • Download a file (GET request)
  • Delete a file (DELETE request)
  • List files (LIST request)
↓
🪣 S3 Bucket
  • Globally unique name (e.g. mycompany-prod-uploads)
  • Tied to a specific AWS region
  • Access controlled by bucket policy + IAM
  • Stores unlimited objects (files)
↓
📄 S3 Object (File)
  • Key: the full "path" — e.g. uploads/2025/photo.jpg
  • Value: the actual file bytes
  • Metadata: content-type, custom tags
  • Max size: 5 TB per object
↓
🌍 Access via HTTPS
  • Direct URL: https://bucket.s3.region.amazonaws.com/key
  • Pre-signed URL: time-limited secure access
  • CloudFront CDN: fast global delivery

S3 Key Facts

📐 Limits

  • Max object size: 5 TB
  • Max single upload: 5 GB (use multipart for larger)
  • Unlimited objects per bucket
  • Unlimited buckets per account (default 100)
  • Bucket names: globally unique across ALL AWS

💰 Pricing Model

  • Pay per GB stored per month
  • Pay per request (GET, PUT, LIST)
  • Pay per GB data transferred out
  • No charge for inbound transfers
  • No charge for S3 → EC2 same region

🧩 Core Concepts — Buckets, Objects, Keys

S3 Structure Visualised

🪣 Bucket: mycompany-prod (ap-south-1)
  • Globally unique name
  • Region-specific (data stays in your region)
  • Holds all your objects (files)
  • One set of permissions, versioning, lifecycle rules
↓
📁 "Folder": uploads/
  • uploads/2025/jan/photo.jpg
  • uploads/2025/feb/doc.pdf
  • (not real folders — just key prefixes)
📁 "Folder": backups/
  • backups/daily/2025-01-01.zip
  • backups/weekly/2025-W01.zip
↓
📄 Object: uploads/2025/jan/photo.jpg
  • Key (full path): uploads/2025/jan/photo.jpg
  • Value: actual file bytes (2.4 MB JPEG)
  • ETag: MD5 hash for integrity check
  • Metadata: ContentType=image/jpeg, custom tags
  • Version ID: abc123 (if versioning enabled)

Key Concepts Explained

1

Bucket Name Rules

3–63 characters · lowercase letters, numbers, hyphens only · no underscores · must start with letter or number · globally unique across ALL of AWS · cannot be changed after creation

2

Object Key = Full Path

There are no real folders in S3. Everything is a flat key-value store. uploads/2025/photo.jpg is just a key with slashes. The console displays them as folders for convenience.

3

Eventual Consistency

S3 is strongly consistent for new objects (since 2020). After a PUT, subsequent GETs always return the latest version. After a DELETE, subsequent GETs will return 404.

4

Pre-Signed URLs

Generate a temporary URL that allows someone to download or upload a specific object — without giving them AWS credentials. Expires after a set time (e.g. 15 minutes). Essential for secure file sharing.

5

Multipart Upload

For files larger than 100 MB, split into parts and upload in parallel. Faster, more reliable — if one part fails, retry only that part. Required for files over 5 GB. AWS SDK handles this automatically.

📦 Storage Classes — Choose the Right Tier

S3 offers different storage classes with different cost/availability tradeoffs. Choosing the right one can save significant money.

How to Choose a Storage Class

❓ How often do you access this data?
  • Frequently? → Standard
  • Monthly? → Standard-IA or One Zone-IA
  • Rarely (once a year)? → Glacier
  • Almost never? → Glacier Deep Archive
  • Unpredictable? → Intelligent-Tiering
↓
🟠 Standard
  • Frequent access
  • Highest availability 99.99%
  • Most expensive storage
  • No retrieval fee
  • Use: app files, uploads
🟢 Standard-IA
  • Infrequent access
  • Availability 99.9%
  • 30% cheaper storage
  • Retrieval fee per GB
  • Use: backups, logs
🔵 Glacier
  • Archive — rare access
  • Retrieval: 1 min–12 hrs
  • 80% cheaper than Standard
  • Min 90-day storage
  • Use: compliance archives

💰 Cost Saving Rule of Thumb

Use Standard for anything your app actively reads. Use Standard-IA for backups older than 30 days. Use Glacier for anything older than 90 days you keep for compliance. Use Lifecycle Policies to move data automatically (next tab).

🔐 Security & Access Control

S3 has multiple layers of access control. Understanding all of them is critical — misconfigured S3 buckets are one of the most common causes of data breaches.

How S3 Access is Evaluated

👤 Request Arrives (GET /bucket/key)
  • Who is making the request? (IAM identity or anonymous)
  • What action? (GetObject, PutObject, DeleteObject)
↓
🔒 Check 1: Block Public Access Settings
  • Account-level AND bucket-level settings
  • If "Block all public access" = ON → no public access ever
  • This overrides everything else — the master switch
↓
✅ Check 2: IAM Policy (for authenticated users)
  • Does this IAM user/role have s3:GetObject permission?
  • Attached to the user, group, or role
  • Can grant or deny access to specific buckets/keys
↓
📋 Check 3: Bucket Policy (resource-based)
  • JSON policy attached to the bucket itself
  • Can grant access to other AWS accounts, services
  • Can grant public access (if Block Public Access is OFF)
  • Evaluated alongside IAM policy
↓
🔑 Check 4: Object ACL (legacy)
  • Per-object access control list
  • Mostly replaced by bucket policies — avoid using
  • Can make individual objects public (but bucket policy preferred)

Bucket Policy Examples

Allow specific IAM role to read all objects 
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Principal": {
      "AWS": "arn:aws:iam::123456789:role/MyAppRole"
    },
    "Action": ["s3:GetObject", "s3:ListBucket"],
    "Resource": [
      "arn:aws:s3:::my-bucket",
      "arn:aws:s3:::my-bucket/*"
    ]
  }]
}
Enforce HTTPS only (deny HTTP) 
{
  "Statement": [{
    "Effect": "Deny",
    "Principal": "*",
    "Action": "s3:*",
    "Resource": ["arn:aws:s3:::my-bucket/*"],
    "Condition": {
      "Bool": { "aws:SecureTransport": "false" }
    }
  }]
}

Encryption

🔒 Server-Side Encryption (SSE)

  • SSE-S3: AWS manages keys (free, default)
  • SSE-KMS: Your KMS key (audit trail, extra cost)
  • SSE-C: You provide the key per request
  • Enabled per bucket — encrypts all new objects

🔑 Client-Side Encryption

  • Encrypt before uploading to S3
  • S3 stores already-encrypted data
  • AWS never sees the plaintext
  • Use when compliance requires it

⚠️ Most common S3 security mistake

❌ Turning off "Block Public Access" without a specific reason
❌ Bucket policy with "Principal": "*" and no conditions
❌ Storing AWS credentials in S3 (use IAM roles instead!)
❌ Not enabling server-side encryption on sensitive data buckets

🔄 Versioning — Keep Every Version

When versioning is enabled, S3 keeps every version of every object. Overwriting or deleting a file doesn't lose the old one — it creates a new version.

How Versioning Works

📤 Upload: photo.jpg (Version 1)
  • Key: uploads/photo.jpg
  • Version ID: abc111 ← current version
  • S3 stores this as the latest
↓
📤 Upload same key again (Version 2)
  • Key: uploads/photo.jpg (same key!)
  • New Version ID: xyz222 ← now the current
  • Version abc111 still exists — not deleted!
↓
🗑️ Delete photo.jpg
  • S3 adds a "Delete Marker" (not a real delete!)
  • GET photo.jpg → 404 (latest is delete marker)
  • GET photo.jpg?versionId=xyz222 → still works! ✅
  • Delete the marker to restore the file
↓
🗂️ Version History in S3
  • Delete marker (current)
  • Version xyz222 (still exists)
  • Version abc111 (still exists)
  • Delete the marker → xyz222 becomes current again

💡 When to enable versioning

Enable versioning on any bucket where accidental deletion or overwrites would be a problem — user uploads, config files, important documents. Combine with lifecycle policies to auto-delete old versions after 30 days to control cost.

⚠️ Versioning and cost

Every version is stored and billed separately. A 10 MB file with 100 versions = 1 GB of storage. Always pair versioning with a lifecycle rule to expire old versions.

♻️ Lifecycle Policies — Automate Storage Management

Lifecycle rules automatically transition objects between storage classes or delete them after a set time. Set once, runs forever — saves money without manual work.

Typical Lifecycle Journey

📤 Day 0 — Object Created
  • Storage class: Standard
  • Actively used by the application
  • Cost: $$$$
↓ After 30 days
📦 Day 30 — Transition to Standard-IA
  • Still accessible instantly
  • 30% cheaper than Standard
  • Good for backups, old logs
↓ After 90 days
🧊 Day 90 — Transition to Glacier
  • Retrieval takes minutes to hours
  • 80% cheaper than Standard
  • Good for compliance archives
↓ After 365 days
🗑️ Day 365 — Permanently Deleted
  • Object and all versions deleted
  • No more storage cost

Common Lifecycle Rule Examples

📸 User Upload Photos

  • Day 0: Standard (active use)
  • Day 90: Standard-IA
  • Day 365: Glacier
  • Day 730: Delete

💾 Daily Database Backups

  • Day 0: Standard
  • Day 30: Standard-IA
  • Day 90: Glacier
  • Day 180: Delete old backups

📋 Application Logs

  • Day 0: Standard
  • Day 7: Standard-IA
  • Day 30: Delete
  • (Logs rarely needed after 30 days)

🔢 Versioned Objects

  • Keep current version: Standard
  • Previous versions after 30d: IA
  • Previous versions after 90d: Glacier
  • Previous versions after 365d: Delete

S3 Service — All Operations

S3Service.cs 
using Amazon.S3;
using Amazon.S3.Model;

public class S3Service
{
    private readonly IAmazonS3 _s3;
    private const string Bucket = "mycompany-prod-uploads";

    public S3Service(IAmazonS3 s3) => _s3 = s3;

    // ── UPLOAD a file ─────────────────────────────
    public async Task UploadAsync(
        string key, Stream fileStream, string contentType)
    {
        await _s3.PutObjectAsync(new PutObjectRequest
        {
            BucketName  = Bucket,
            Key         = key,          // e.g. "uploads/2025/photo.jpg"
            InputStream = fileStream,
            ContentType = contentType,  // e.g. "image/jpeg"
            ServerSideEncryptionMethod =
                ServerSideEncryptionMethod.AES256
        });
    }

    // ── DOWNLOAD a file ───────────────────────────
    public async Task<Stream> DownloadAsync(string key)
    {
        var response = await _s3.GetObjectAsync(new GetObjectRequest
        {
            BucketName = Bucket,
            Key        = key
        });
        return response.ResponseStream;
    }

    // ── DELETE a file ─────────────────────────────
    public async Task DeleteAsync(string key)
    {
        await _s3.DeleteObjectAsync(new DeleteObjectRequest
        {
            BucketName = Bucket,
            Key        = key
        });
    }

    // ── LIST files with a prefix ──────────────────
    public async Task<List<string>> ListFilesAsync(string prefix)
    {
        var response = await _s3.ListObjectsV2Async(
            new ListObjectsV2Request
        {
            BucketName = Bucket,
            Prefix     = prefix // e.g. "uploads/2025/"
        });
        return response.S3Objects
            .Select(o => o.Key)
            .ToList();
    }

    // ── PRE-SIGNED URL (time-limited access) ──────
    public string GeneratePresignedUrl(
        string key, int expiryMinutes = 15)
    {
        return _s3.GetPreSignedURL(new GetPreSignedUrlRequest
        {
            BucketName = Bucket,
            Key        = key,
            Expires    = DateTime.UtcNow
                .AddMinutes(expiryMinutes),
            Protocol   = Protocol.HTTPS
        });
    }

    // ── PRE-SIGNED URL for UPLOAD (browser upload) ─
    public string GenerateUploadUrl(
        string key, string contentType, int expiryMinutes = 10)
    {
        return _s3.GetPreSignedURL(new GetPreSignedUrlRequest
        {
            BucketName  = Bucket,
            Key         = key,
            Verb        = HttpVerb.PUT,
            ContentType = contentType,
            Expires     = DateTime.UtcNow.AddMinutes(expiryMinutes)
        });
    }
}

✅ Best Practices & Common Mistakes

Security

  • Always keep "Block Public Access" ON unless you have a specific reason
  • Use IAM roles for EC2/ECS — never hardcode access keys
  • Grant least privilege — only the actions the app actually needs
  • Enable default encryption (SSE-S3) on every bucket
  • Use pre-signed URLs to share files — not public bucket access
  • Enable S3 server access logging for audit trail
  • Use bucket policies with specific Principal ARN (not "*")
  • Enable MFA Delete on critical buckets to prevent accidental deletion

Performance & Cost

  • Use multipart upload for files larger than 100 MB
  • Use S3 Transfer Acceleration for global users (pays extra)
  • Use CloudFront CDN in front of S3 for frequently accessed static files
  • Use lifecycle rules to auto-transition to cheaper storage classes
  • Enable Intelligent-Tiering for objects with unknown access patterns
  • Use S3 Inventory to find and clean up old/unused objects
  • Prefix your keys randomly (or by date) for performance at scale

Application Design

  • Never store files on EC2 instance disk — use S3 for all file storage
  • Use pre-signed PUT URLs for browser-direct uploads (no server proxy needed)
  • Store the S3 key in your database, not the full URL (URLs can change)
  • Enable versioning on all buckets that hold user content
  • Add CORS rules only when browsers upload directly to S3
  • Use S3 Event Notifications to trigger Lambda on new uploads

Common Mistakes to Avoid

⚠️ Top S3 Mistakes

❌ Public bucket without reason — data breach waiting to happen
❌ Hardcoded AWS keys in app code — use IAM roles instead
❌ No lifecycle rules + versioning — storage costs grow forever
❌ Storing full S3 URL in DB — breaks if region or bucket changes
❌ Uploading all files with same key prefix — performance bottleneck
❌ No server access logging — blind during security incidents
❌ Forgetting to set ContentType — browser downloads instead of displays
