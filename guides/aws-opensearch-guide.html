🔍 What is AWS OpenSearch Service?

AWS OpenSearch Service is a fully managed search and analytics engine built on OpenSearch (a fork of Elasticsearch). It lets you ingest, store, search, and visualise large volumes of data — from log files and application events to e-commerce product catalogues and real-time dashboards.

💡 OpenSearch in One Sentence

AWS OpenSearch = a managed Elasticsearch-compatible service for full-text search, log analytics, and real-time dashboards — with AWS handling clusters, patching, backups, and scaling.

Brief History — Elasticsearch → OpenSearch

📌 Why OpenSearch exists

AWS offered Elasticsearch Service for years. In 2021, Elastic changed Elasticsearch's licence to non-open-source (SSPL). AWS forked Elasticsearch 7.10 and OpenSearch Distro, creating the OpenSearch project (Apache 2.0 licence). AWS OpenSearch Service now runs OpenSearch natively. It remains compatible with most Elasticsearch 7.x APIs.

🧱 Core Concepts — How OpenSearch Thinks

The Inverted Index — Why OpenSearch is Fast at Text Search

📖 Inverted Index — Word → Documents mapping

How Text is Analysed Before Indexing

🔬 Text Analysis Pipeline — "Running Quickly!" → tokens
① Character Filter
  • Remove HTML tags, replace special chars
  • "Running <b>Quickly</b>!" → "Running Quickly!"
↓
② Tokeniser
  • Split text into individual tokens (words)
  • "Running Quickly!" → ["Running", "Quickly"]
↓
③ Token Filters
  • Lowercase: ["running", "quickly"]
  • Stop words removed: "the", "a", "is" removed
  • Stemming: "running" → "run", "quickly" → "quick"
↓
④ Final Tokens → Stored in Inverted Index
  • ["run", "quick"] → indexed and searchable
  • Searching "runs" or "runner" also finds this doc!

Sharding and Replication

🔀 Shards — Horizontal Scale

  • An index is split into N primary shards
  • Each shard = a full Lucene index on one node
  • More shards = more parallelism = faster search at scale
  • Shard count is fixed at index creation — plan ahead!
  • Rule of thumb: shard size 10–50 GB, max 200 shards/node
  • Over-sharding is a common mistake — too many small shards = overhead

📋 Replicas — High Availability & Read Scale

  • Each primary shard has N replica shards on different nodes
  • Replicas serve read/search requests — increases search throughput
  • If a node fails, replicas are promoted to primaries automatically
  • Replica count can be changed after creation (unlike shards)
  • Minimum for production: 1 replica per shard (so 2 copies of data)

Dynamic vs Explicit Mappings

🤖 Dynamic Mapping (Default)

  • OpenSearch auto-detects field types from first document
  • Strings → text + keyword sub-field
  • Numbers → long or float
  • Dates (ISO format) → date
  • ⚠️ Can cause "mapping explosion" if keys are dynamic/random
  • Good for: prototyping, log schemas

📋 Explicit Mapping (Production)

  • You define every field and its type upfront
  • Control analyser, store, index, doc_values per field
  • Set "dynamic": "strict" to reject unknown fields
  • Prevents accidental indexing of sensitive fields
  • Good for: production indices, performance-optimised schemas

⚠️ Mapping Explosion

If you index documents where field names are dynamic (e.g. user-defined metadata keys), OpenSearch creates a new mapping entry for every unique key. With millions of keys, the cluster state grows enormous and causes instability. Always use "dynamic": "strict" or flattened field type for dynamic keys.

🔎 Search & Queries — Query DSL

OpenSearch uses a JSON-based Query DSL. All queries go to POST /<index>/_search. There are two kinds of clauses: queries (affect relevance score) and filters (yes/no, cached, faster).

Query vs Filter Context

🎯 Query Context — "How relevant?"

  • Calculates a relevance _score for each document
  • Use for: full-text search, ranking results
  • Examples: match, multi_match, match_phrase
  • Slower (scoring overhead)

⚡ Filter Context — "Yes or No?"

  • No score — binary include/exclude
  • Results are cached — much faster for repeated filters
  • Use for: status, date range, category, boolean flags
  • Examples: term, range, exists, terms

Common Query Types

json — match query (full-text search, scored) 
// Search for "wireless headphones" in the name field
{
  "query": {
    "match": {
      "name": {
        "query": "wireless headphones",
        "operator": "and"    // both words must be present
      }
    }
  }
}
json — bool query (combine multiple clauses) 
// Find: in-stock Electronics, name matches "headphones",
// price < 200, boost if tagged "wireless"
{
  "query": {
    "bool": {
      "must": [
        { "match": { "name": "headphones" } }
      ],
      "filter": [
        { "term":  { "category": "Electronics" } },
        { "term":  { "in_stock": true } },
        { "range": { "price": { "lte": 200 } } }
      ],
      "should": [
        { "term": { "tags": "wireless" } }
      ],
      "must_not": [
        { "term": { "tags": "refurbished" } }
      ]
    }
  }
}

💡 Bool Query Clauses

must = must match (affects score) · filter = must match (no score, cached) · should = nice to have (boosts score) · must_not = must NOT match (cached)

json — fuzzy search (typo tolerance) 
{
  "query": {
    "match": {
      "name": {
        "query": "hedfones",   // typo — still finds "headphones"
        "fuzziness": "AUTO"  // AUTO: 0 for 1-2 chars, 1 for 3-5, 2 for 6+
      }
    }
  }
}
json — aggregation query (group by + metrics) 
{
  "size": 0,  // return only aggregation results, not documents
  "aggs": {
    "by_category": {
      "terms": { "field": "category", "size": 10 },
      "aggs": {
        "avg_price": { "avg":  { "field": "price" } },
        "max_price": { "max":  { "field": "price" } },
        "count":     { "value_count": { "field": "product_id" } }
      }
    },
    "price_histogram": {
      "histogram": { "field": "price", "interval": 50 }
    }
  }
}

Sizing Guidelines

📐 Data Node Sizing

  • Target 10–50 GB per shard (sweet spot)
  • Total data (with replicas) ÷ nodes = data per node
  • Leave 25% headroom for merges and overhead
  • JVM heap = 50% of RAM, max 32 GB per node
  • r6g (memory-optimised) for search-heavy workloads
  • c6g (compute-optimised) for log analytics

⚙️ Master Node Sizing

  • Always use 3 dedicated masters (odd number, quorum)
  • m6g.large is usually enough — masters don't store data
  • Master node RAM scales with cluster state size (number of shards)
  • Rule: 1 GB RAM per 3,000 shards on master
  • Never use the same node as data + master in production

Multi-AZ Deployment

✅ Production: Always 3 Availability Zones

Deploy data nodes across 3 AZs with zone awareness enabled. OpenSearch will ensure primary and replica shards of the same shard are never on the same AZ — so an AZ failure never causes data loss. Use an even number of data nodes divisible by 3 (e.g. 3, 6, 9 nodes).

🔐 Security & IAM

Security Layers in OpenSearch

🔐 OpenSearch Security — Four Layers
🌐 Layer 1 — Network
  • VPC-only or public endpoint
  • Security Groups control which resources can reach the domain
  • Always use VPC endpoint for production
↓
🔑 Layer 2 — IAM Resource Policy
  • Domain-level access policy (like an S3 bucket policy)
  • Controls which AWS principals can call the OpenSearch API at all
  • Whitelist by IAM ARN, account, condition (source IP)
↓
👤 Layer 3 — Fine-Grained Access Control (FGAC)
  • Internal OpenSearch users and roles
  • Role-based access: index patterns, document-level, field-level
  • Map IAM roles or SAML identities → OpenSearch roles
↓
🔒 Layer 4 — Encryption
  • In-transit: TLS 1.2/1.3 enforced
  • At-rest: KMS managed or customer-managed CMK
  • Automated snapshots encrypted at rest

IAM-based Access — Resource Policy Example

json — domain access policy (allow specific IAM role) 
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::123456789012:role/MyAppRole"
      },
      "Action": [
        "es:ESHttpGet",
        "es:ESHttpPost",
        "es:ESHttpPut",
        "es:ESHttpDelete"
      ],
      "Resource": "arn:aws:es:eu-west-1:123456789012:domain/my-domain/*"
    }
  ]
}

Fine-Grained Access Control (FGAC)

🎭 Roles & Permissions

  • Create OpenSearch internal roles with specific permissions
  • Assign index patterns: logs-*, products
  • Grant: read, write, manage, create_index, etc.
  • Map AWS IAM role → OpenSearch role via role mapping
  • Also supports HTTP Basic auth (username/password)

📄 Document & Field Level

  • Document-level security: filter which docs a role can see
  • E.g. "finance team sees only region=EU documents"
  • Field-level security: hide specific fields from a role
  • E.g. "mask the salary field for non-HR roles"
  • Applied transparently — user's query returns filtered results

⚠️ Never use a public endpoint in production

Public endpoints expose the OpenSearch API to the internet. Even with an access policy, it's a risk. Always deploy within a VPC and access via private endpoint. Use VPN or AWS Systems Manager Session Manager for dashboard access.

var searchResponse = await client.SearchAsync<Product>(s => s
    .Index("products")
    .Size(10)
    .Query(q => q

5. Bool Query — Filters + Full-Text

c# — bool query with must, filter, should 
var result = await client.SearchAsync<Product>(s => s
    .Query(q => q
        .Bool(b => b
            .Must(m => m
                .Match(ma => ma.Field(f => f.Name).Query("headphones"))
            )
            .Filter(f => f
                .Term(t => t.Field(fi => fi.Category).Value("Electronics")) &&
                f.Term(t => t.Field(fi => fi.InStock).Value(true)) &&
                f.Range(r => r.Field(fi => fi.Price).LessThanOrEquals(200))
            )
            .Should(sh => sh
                .Term(t => t.Field(fi => fi.Tags).Value("wireless"))
            )
        )
    )
);

7. Aggregations in C#

c# — terms aggregation grouped by category 
var aggResult = await client.SearchAsync<Product>(s => s
    .Size(0)  // only return aggregations, no documents
    .Aggregations(a => a
        .Terms("by_category", t => t
            .Field(f => f.Category)
            .Size(10)
            .Aggregations(aa => aa
