🔑 What is AWS KMS?

KMS = Key Management Service. A fully managed AWS service that creates, stores, and controls cryptographic keys used to encrypt your data. You never see the raw key material — KMS keeps it locked away and only uses it when you explicitly request an operation.

💡 Real-life analogy

KMS is like a bank vault for your encryption keys. You don't carry the vault key with you — you go to the bank (KMS), prove who you are (IAM), and the bank performs the operation (encrypt/decrypt) for you. The key never leaves the vault.

Without KMS vs With KMS

❌ Without KMS
  • You manage encryption keys yourself
  • Keys stored alongside data — risky!
  • Key rotation is manual and error-prone
  • No audit trail — who used the key?
  • If key is lost → data unrecoverable
  • Compliance nightmare
→
✅ With KMS
  • AWS stores keys in HSMs (tamper-proof)
  • Keys never leave KMS unencrypted
  • Automatic annual key rotation
  • Every use logged in CloudTrail
  • IAM controls who can use which key
  • Meets FIPS 140-2 compliance

KMS Architecture Overview

👤 Your Application / AWS Service
  • Wants to encrypt or decrypt data
  • Calls KMS API with the key ARN
  • Must have IAM permission to use the key
↓
🔑 AWS KMS Service
  • Validates caller identity via IAM
  • Checks key policy allows this operation
  • Performs encrypt/decrypt inside HSM
  • Logs every API call to CloudTrail
  • Returns result — key never exposed
↓
🏛️ Hardware Security Module (HSM)
  • Physical tamper-proof hardware
  • Keys exist ONLY inside the HSM
  • Validated to FIPS 140-2 Level 3
  • Cannot be exported or accessed directly
↓
✅ Encrypted / Decrypted Data Returned
  • Ciphertext (encrypted) or plaintext (decrypted) returned
  • Audit record written to CloudTrail automatically

🗝️ KMS Key Types — Choosing the Right One

KMS offers different key types depending on who controls the key material and how it's managed.

Key Types at a Glance

🟠 AWS Managed Keys
  • Created and managed by AWS services
  • Free to use (no monthly charge)
  • Named: aws/s3, aws/rds, aws/ebs
  • Auto-rotated every year
  • Cannot customize key policy
  • Good for: quick start, basic needs
🔵 Customer Managed Keys (CMK)
  • You create and control them
  • $1/month per key + API charges
  • Full key policy control
  • Cross-account sharing possible
  • Custom rotation schedule
  • Good for: production, compliance
↓
🟢 AWS CloudHSM Keys (Your Own HSM)
  • You own a dedicated physical HSM in AWS
  • Full control over key material
  • Most expensive — dedicated hardware
  • Required for: government, highest compliance needs
  • Good for: FIPS 140-2 Level 3 strict requirements

CMK Key Material Origin

🏭 KMS Generated (Default)

  • AWS generates key material inside HSM
  • Key never leaves AWS infrastructure
  • Simplest option — recommended for most
  • Supports automatic rotation

📥 External Key Material (BYOK)

  • You generate key material yourself
  • Import it into KMS
  • BYOK = Bring Your Own Key
  • You can expire/delete it anytime
  • No automatic rotation supported

🔧 Asymmetric Keys

  • RSA or ECC key pairs
  • Public key can be downloaded
  • Private key never leaves KMS
  • Use for: digital signing, TLS

🔁 Symmetric Keys (AES-256)

  • Default — same key encrypts and decrypts
  • Used by all AWS service integrations
  • Never leaves KMS — most secure
  • Use for: data encryption at rest

🔒 Encrypt & Decrypt — How It Works

KMS uses symmetric AES-256 encryption by default. You never handle the key — you hand data to KMS and get back ciphertext (or plaintext).

Encryption Flow

📤 App wants to encrypt sensitive data
  • Plaintext: "CardNumber=4111111111111111"
  • Calls KMS Encrypt API with key ARN
  • Max direct encrypt size: 4 KB
↓
🔑 KMS validates and encrypts
  • Checks caller has kms:Encrypt permission
  • Retrieves CMK from HSM
  • Encrypts data inside HSM using AES-256-GCM
  • Logs operation to CloudTrail
↓
✅ Ciphertext returned to app
  • Base64-encoded ciphertext blob
  • Contains: encrypted data + key ARN metadata
  • Store this in database — safe even if stolen
  • Only KMS with the right key can decrypt it

Decryption Flow

📥 App needs to read sensitive data
  • Retrieves ciphertext blob from database
  • Calls KMS Decrypt API (no key ARN needed — it's embedded in blob)
↓
🔑 KMS validates and decrypts
  • Checks caller has kms:Decrypt permission
  • Extracts key ARN from ciphertext metadata
  • Retrieves CMK from HSM
  • Decrypts data inside HSM
  • Logs to CloudTrail
↓
✅ Plaintext returned to app
  • "CardNumber=4111111111111111" returned
  • Plaintext only lives in memory — never persisted

Encryption Context — Extra Security Layer

🔏 What is Encryption Context?

An optional key-value map you provide during encryption. KMS cryptographically binds this context to the ciphertext. Decryption MUST provide the exact same context — or it fails.

Example: { "userId": "usr_123", "purpose": "payment" }

Why use it: Prevents a ciphertext from user A being decrypted in the context of user B. Adds binding between encrypted data and its intended use. Also included in CloudTrail logs for richer audit trails.

💡 4 KB limit — what about larger data?

KMS direct encrypt/decrypt is limited to 4 KB. For larger data (files, documents, blobs), use Envelope Encryption — the next tab explains this pattern in detail.

📦 Envelope Encryption — For Large Data

Envelope encryption solves the 4 KB KMS limit. Instead of encrypting data with KMS directly, you generate a Data Encryption Key (DEK) from KMS, use it to encrypt your large data locally, then let KMS encrypt the DEK itself.

💡 Analogy

Put your letter (data) in a lockbox (DEK encryption). Then put the lockbox key inside a bank vault (KMS encrypts the DEK). To read your letter: open the vault (KMS decrypt), get the lockbox key (plaintext DEK), open the lockbox (decrypt data).

Envelope Encryption — Encrypt Flow

📤 App wants to encrypt a 50 MB file
  • File is too large for direct KMS encrypt (4 KB limit)
  • Step 1: Call KMS GenerateDataKey with your CMK ARN
↓
🔑 KMS generates a Data Encryption Key (DEK)
  • Returns TWO things:
  • ① Plaintext DEK — use this to encrypt your file
  • ② Encrypted DEK — store this alongside your file
  • Logs the GenerateDataKey call to CloudTrail
↓
⚡ App encrypts file locally (fast — no KMS call)
  • Use plaintext DEK to encrypt 50 MB file with AES-256
  • This happens in memory — no network call to KMS
  • Immediately discard plaintext DEK from memory
↓
✅ Store two things
  • Encrypted file → S3 or database
  • Encrypted DEK → alongside the encrypted file
  • Plaintext DEK → GONE from memory (never stored!)

Envelope Encryption — Decrypt Flow

📥 App needs to read the 50 MB file
  • Retrieves encrypted file + encrypted DEK from storage
  • Step 1: Call KMS Decrypt with the encrypted DEK
↓
🔑 KMS decrypts the DEK
  • Checks caller has kms:Decrypt permission
  • Returns plaintext DEK (just the key — tiny!)
  • Only ONE KMS API call regardless of file size
↓
⚡ App decrypts file locally
  • Use plaintext DEK to decrypt file in memory
  • Fast — no more KMS calls needed
  • Discard plaintext DEK after use
↓
✅ Plaintext file available in memory
  • 50 MB file decrypted and usable
  • Only 1 KMS API call regardless of file size
  • Massively more efficient than direct KMS encryption

Why Envelope Encryption is the Right Pattern

✅ Benefits

  • No size limit — encrypt files of any size
  • Fast — local encryption after getting DEK
  • Low KMS cost — one call per file operation
  • Key isolation — each file has its own DEK
  • CMK never touches your data directly

🔐 Security Properties

  • CMK compromise doesn't expose all data
  • Each file encrypted with unique DEK
  • Plaintext DEK only exists in memory briefly
  • Re-encrypting just means new DEK per file
  • AWS SDK Encryption Client uses this automatically

📋 Key Policies — Controlling Key Access

Every KMS key has a key policy — a JSON document that controls who can use and manage the key. Unlike IAM policies, a key policy is required — without one, nobody can use the key.

KMS Access Control — Two Layers

📨 Request: ECS task calls kms:Decrypt
  • Identity: arn:aws:iam::123:role/my-api-task-role
  • Action: kms:Decrypt on key arn:aws:kms:...
↓
🔒 Check 1: Key Policy
  • Does the key policy allow this identity?
  • Key policy MUST explicitly allow — no default inheritance
  • If key policy denies → BLOCKED immediately
↓
✅ Check 2: IAM Policy
  • Does the role's IAM policy allow kms:Decrypt?
  • Both key policy AND IAM must allow — not just one
  • If both allow → operation proceeds
↓
🔑 Decryption performed inside HSM
  • Logged to CloudTrail
  • Plaintext returned to caller

Key Policy Examples

Default key policy — enable IAM to manage access 
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "EnableRootAccountAccess",
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::123456789:root"
      },
      "Action": "kms:*",
      "Resource": "*"
    },
    {
      "Sid": "AllowAppRoleToUseKey",
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::123456789:role/my-api-task-role"
      },
      "Action": [
        "kms:Decrypt",
        "kms:GenerateDataKey"
      ],
      "Resource": "*"
    }
  ]
}

Key Policy Permissions — Who Gets What

🔧 Key Administrators

  • kms:Create*, kms:Describe*
  • kms:Enable*, kms:Disable*
  • kms:Get*, kms:Delete*, kms:List*
  • kms:Put*, kms:Update*, kms:Revoke*
  • kms:ScheduleKeyDeletion, kms:CancelKeyDeletion

⚡ Key Users (App Roles)

  • kms:Encrypt
  • kms:Decrypt
  • kms:ReEncrypt*
  • kms:GenerateDataKey*
  • kms:DescribeKey

⚠️ Key Policy vs IAM Policy — Both Required

Unlike S3 or other resources where IAM alone can grant access, KMS requires BOTH the key policy AND IAM policy to allow an operation. If either one denies — access is blocked. This two-factor authorization is a feature, not a bug.

🔄 Key Rotation — Staying Secure Over Time

Key rotation means periodically replacing the underlying cryptographic material of a key. Old data remains decryptable — KMS keeps old key versions internally.

How Automatic Rotation Works

📅 Year 1 — Key Created
  • CMK arn:aws:kms:...:key/abc-123 created
  • Key material version 1 (KM-v1) inside HSM
  • All new encryptions use KM-v1
↓ 365 days later
🔄 Year 2 — Automatic Rotation
  • AWS generates new key material: KM-v2
  • Key ARN stays the same — no app changes needed!
  • KM-v1 is retained inside HSM (not deleted)
  • All new encryptions use KM-v2
↓
✅ Old data still decryptable
  • Ciphertext encrypted with KM-v1 → decrypts using KM-v1
  • Ciphertext encrypted with KM-v2 → decrypts using KM-v2
  • KMS automatically uses the right version
  • Your app code needs zero changes

Rotation Options

🔁 Automatic Rotation (Recommended)

  • Enable once — runs every year forever
  • AWS manages everything — zero effort
  • Key ARN never changes
  • Only for AWS-generated symmetric CMKs
  • Old versions kept for decryption

✋ Manual Rotation

  • Create a brand new CMK
  • Update all references to use new ARN
  • Re-encrypt data with new key
  • Required for: asymmetric keys, BYOK
  • More control, more work
FeatureAutomatic RotationManual Rotation

✅ Best Practice

Always enable automatic rotation on all Customer Managed Keys. It costs nothing extra, requires no code changes, and reduces the risk of key compromise over time. It's a one-checkbox decision with no downside.

KMS Service — Encrypt, Decrypt, Data Keys

KmsService.cs 
using Amazon.KeyManagementService;
using Amazon.KeyManagementService.Model;

public class KmsService
{
    private readonly IAmazonKeyManagementService _kms;
    private const string KeyId = "alias/myapp-prod-encryption";

    public KmsService(IAmazonKeyManagementService kms)
        => _kms = kms;

    // ── Direct Encrypt (max 4 KB) ─────────────────
    public async Task<string> EncryptAsync(
        string plaintext,
        Dictionary<string, string>? context = null)
    {
        var plaintextBytes = Encoding.UTF8.GetBytes(plaintext);

        var response = await _kms.EncryptAsync(new EncryptRequest
        {
            KeyId             = KeyId,
            Plaintext         = new MemoryStream(plaintextBytes),
            EncryptionContext = context  // optional extra binding
        });

        return Convert.ToBase64String(
            response.CiphertextBlob.ToArray());
    }

    // ── Direct Decrypt ────────────────────────────
    public async Task<string> DecryptAsync(
        string ciphertextBase64,
        Dictionary<string, string>? context = null)
    {
        var ciphertextBytes = Convert.FromBase64String(ciphertextBase64);

        var response = await _kms.DecryptAsync(new DecryptRequest
        {
            CiphertextBlob    = new MemoryStream(ciphertextBytes),
            EncryptionContext = context
        });

        return Encoding.UTF8.GetString(
            response.Plaintext.ToArray());
    }

    // ── Generate Data Key (Envelope Encryption) ───
    public async Task<(byte[] PlaintextKey, byte[] EncryptedKey)>
        GenerateDataKeyAsync()
    {
        var response = await _kms.GenerateDataKeyAsync(
            new GenerateDataKeyRequest
        {
            KeyId   = KeyId,
            KeySpec = DataKeySpec.AES_256
        });

        return (
            response.Plaintext.ToArray(),     // use → then discard!
            response.CiphertextBlob.ToArray() // store with your data
        );
    }
}

Envelope Encryption — Full Example

EnvelopeEncryptionExample.cs 
using System.Security.Cryptography;

public class EnvelopeEncryptionService
{
    private readonly KmsService _kms;

    public EnvelopeEncryptionService(KmsService kms)
        => _kms = kms;

    // ENCRYPT a large payload
    public async Task<EncryptedPayload> EncryptLargeAsync(
        byte[] data)
    {
        // Step 1: Get data key from KMS
        var (plaintextDek, encryptedDek) =
            await _kms.GenerateDataKeyAsync();

        try
        {
            // Step 2: Encrypt data locally using DEK
            using var aes = Aes.Create();
            aes.Key = plaintextDek;
            aes.GenerateIV();

            using var ms = new MemoryStream();
            using (var cs = new CryptoStream(
                ms, aes.CreateEncryptor(), CryptoStreamMode.Write))
            {
                await cs.WriteAsync(data);
            }

            return new EncryptedPayload
            {
                EncryptedData = ms.ToArray(),
                EncryptedDek  = encryptedDek,  // store this!
                Iv            = aes.IV
            };
        }
        finally
        {
            // Step 3: Always clear plaintext DEK!
            Array.Clear(plaintextDek, 0, plaintextDek.Length);
        }
    }

    // DECRYPT a large payload
    public async Task<byte[]> DecryptLargeAsync(
        EncryptedPayload payload)
    {
        // Step 1: Ask KMS to decrypt the DEK
        var decryptResp = await _kms.DecryptAsync(
            Convert.ToBase64String(payload.EncryptedDek));
        var plaintextDek = Convert.FromBase64String(decryptResp);

        try
        {
            // Step 2: Decrypt data locally
            using var aes = Aes.Create();
            aes.Key = plaintextDek;
            aes.IV  = payload.Iv;

            using var ms = new MemoryStream(payload.EncryptedData);
            using var cs = new CryptoStream(
                ms, aes.CreateDecryptor(), CryptoStreamMode.Read);
            using var result = new MemoryStream();
            await cs.CopyToAsync(result);
            return result.ToArray();
        }
        finally
        {
            Array.Clear(plaintextDek, 0, plaintextDek.Length);
        }
    }
}

public record EncryptedPayload(
    byte[] EncryptedData,
    byte[] EncryptedDek,
    byte[] Iv);

✅ Best Practices & Common Mistakes

Key Management

  • Always enable automatic rotation on all Customer Managed Keys
  • Use one CMK per application and environment — don't share keys across apps
  • Always add a meaningful alias — never reference just the key ARN in code
  • Add descriptive tags: App, Environment, Team, Purpose
  • Set the deletion waiting period to 30 days — gives time to recover mistakes
  • Never delete a key unless you're certain no data is encrypted with it

Access Control

  • Use separate key administrators and key users — not the same role
  • Grant kms:Decrypt and kms:GenerateDataKey only — not kms:*
  • Always use encryption context for sensitive operations
  • Use resource-based key policies rather than IAM policies alone
  • Regularly review key policies — remove unused principals
  • Never grant kms:* to application roles — admin permission in app is wrong

Encryption Patterns

  • Use envelope encryption for data larger than 4 KB — always
  • Always immediately clear plaintext DEK from memory after use
  • Use encryption context to bind ciphertext to its intended use
  • Enable SSE-KMS on all S3 buckets with sensitive data
  • Enable KMS encryption on RDS, DynamoDB, EBS, ECR
  • Use Bucket Key on S3 to reduce KMS API call costs

Monitoring & Audit

  • Enable CloudTrail in all regions to capture KMS API calls
  • Set CloudWatch alarms on unusually high kms:Decrypt call rates
  • Alert on kms:ScheduleKeyDeletion events — someone is deleting a key!
  • Review CloudTrail logs for unexpected kms:Decrypt from unknown principals
  • Use AWS Config rules to detect CMKs without rotation enabled

Common Mistakes to Avoid

⚠️ Top KMS Mistakes

❌ Deleting a CMK with encrypted data — that data is gone forever
❌ Sharing one CMK across all apps — blast radius too large if compromised
❌ Not enabling key rotation — increases risk of long-term key compromise
❌ Using direct encrypt for large files — hits 4 KB limit, use envelope encryption
❌ Not clearing plaintext DEK from memory — key exposed in memory dump
❌ Granting kms:* to app roles — apps should use keys, not manage them
❌ No encryption context — ciphertext can be replayed in wrong context
❌ Hardcoding key ARN — use alias so rotation doesn't break your code
