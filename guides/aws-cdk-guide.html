🏗️ What is AWS CDK?

AWS CDK (Cloud Development Kit) lets you define your entire AWS infrastructure using real programming languages — C#, TypeScript, Python, Java, Go. You write code; CDK synthesises it into CloudFormation templates and deploys them. You get all the power of a real language: loops, classes, inheritance, unit tests, IDE autocomplete, and refactoring tools — applied to infrastructure.

💡 The Core Mental Model

CDK sits above CloudFormation. You write C# code → CDK synthesises it into a CloudFormation template (cdk synth) → CloudFormation deploys the real AWS resources. The CDK layer is purely a code-time tool — nothing extra runs in your AWS account.

Key Concepts at a Glance

🧱

Construct

The basic building block. Represents one or many AWS resources. Three levels: L1, L2, L3.

📦

Stack

A deployable unit — maps to one CloudFormation stack. Groups constructs.

🌐

App

The root of the construct tree — contains one or more stacks.

⚗️

Synth

Converts your C# code to CloudFormation JSON in cdk.out/.

🔄

Diff

Shows what will change before you deploy. Like a preview / change set.

🔒

Bootstrap

One-time setup per account/region — creates S3 bucket + ECR repo for CDK assets.

🎯

Token

A placeholder for a value not yet known at synth time (e.g., a bucket ARN).

🔍

Aspect

Visit every construct in a scope — apply policy, tagging, or validation cross-cutting.

🚀

CDK Pipelines

Self-mutating CI/CD pipeline — deploys itself and your app using CodePipeline.

🧱 Constructs — L1, L2, L3

Constructs are the fundamental building blocks of CDK. Every resource in your CDK app is a construct. They come in three abstraction levels.

⬇️ L1 — CloudFormation Resources (Cfn*)

1:1 mapping to CloudFormation resource types. Low-level, verbose, full control. Class names start with Cfn. Every CloudFormation property is available.


CfnBucket, CfnFunction, CfnTable

⬆️ L2 — Higher-Level Constructs

Opinionated wrappers with sensible defaults, IAM helpers, and event-driven connections. Most common choice — good defaults, still configurable. Classes named naturally.


Bucket, Function, Table, Queue

🚀 L3 — Patterns (aws-solutions-constructs)

Multi-resource patterns combining multiple L2 constructs into opinionated, battle-tested architectures. One line deploys a complete pattern.


LambdaToDynamoDB, SqsToLambda

Custom Constructs — Reusable Components

C# · Custom L2-style Construct for Team Watson 
/// <summary>
/// Reusable construct: Lambda + DynamoDB + CloudWatch Alarm
/// Use this instead of wiring all three manually every time.
/// </summary>
public class WatsonLambdaWithStorage : Construct
{
    public Function LambdaFunction { get; }
    public Table    StorageTable   { get; }

    public WatsonLambdaWithStorage(
        Construct scope, string id,
        WatsonLambdaWithStorageProps props) : base(scope, id)
    {
        // Table
        StorageTable = new Table(this, "Table", new TableProps
        {
            PartitionKey  = props.PartitionKey,
            BillingMode   = BillingMode.PAY_PER_REQUEST,
            RemovalPolicy = RemovalPolicy.RETAIN
        });

        // Lambda
        LambdaFunction = new Function(this, "Function", new FunctionProps
        {
            Runtime     = Runtime.PROVIDED_AL2023,
            Architecture= Architecture.ARM_64,
            Handler     = "bootstrap",
            Code        = Code.FromAsset(props.CodePath),
            MemorySize  = props.MemorySize ?? 512,
            Timeout     = Duration.Seconds(30),
            Tracing     = Tracing.ACTIVE,
            Environment = new Dictionary<string, string>
            {
                ["TABLE_NAME"] = StorageTable.TableName,
                ["ENVIRONMENT"] = props.Environment
            }
        });

        // Grant least-privilege access
        StorageTable.GrantReadWriteData(LambdaFunction);

        // CloudWatch Alarm — auto-wired
        LambdaFunction.MetricErrors().CreateAlarm(this, "ErrorAlarm",
            new CreateAlarmOptions
            {
                AlarmName          = $"{id}-errors",
                Threshold          = 5,
                EvaluationPeriods  = 1,
                TreatMissingData   = TreatMissingData.NOT_BREACHING
            });
    }
}

// Usage — one line instead of 30+
var orders = new WatsonLambdaWithStorage(this, "Orders",
    new WatsonLambdaWithStorageProps
    {
        CodePath     = "./src/WatsonOrders",
        PartitionKey = new Attribute { Name = "orderId", Type = AttributeType.STRING },
        Environment  = "prod"
    });

Aspects — Cross-Cutting Concerns

An Aspect visits every construct in a scope. Use them for policy enforcement, tagging, or compliance checks that should apply everywhere — without touching every construct individually.

🔍 Aspect — Visiting All Constructs in a Stack
WatsonApiStack (scope)
OrdersTable GetOrdersFunction WatsonApi ServiceRole LogGroup SecurityGroup
↓ Aspects.Of(stack).Add(new WatsonTaggingAspect())
🏷️ WatsonTaggingAspect — visits every construct above

Adds Team=Watson, CostCenter=Platform, ManagedBy=CDK to every resource. No need to set Tags on each construct individually.

C# · Aspects — Tagging + Compliance Enforcement // ── Tagging Aspect ──────────────────────────────────────────── public class WatsonTaggingAspect : IAspect { private readonly string _env; public WatsonTaggingAspect(string env) => _env = env; public void Visit(IConstruct node) { if (node is CfnResource cfn) { Tags.Of(node).Add("Team", "Watson"); Tags.Of(node).Add("CostCenter", "Platform"); Tags.Of(node).Add("Environment", _env); Tags.Of(node).Add("ManagedBy", "CDK"); } } } // ── Security Compliance Aspect ──────────────────────────────── public class BucketEncryptionAspect : IAspect { public void Visit(IConstruct node) { // Enforce encryption on all S3 buckets if (node is CfnBucket bucket) { bucket.BucketEncryption = new CfnBucket.BucketEncryptionProperty { ServerSideEncryptionConfiguration = new[] { new CfnBucket.ServerSideEncryptionRuleProperty { ServerSideEncryptionByDefault = new CfnBucket.ServerSideEncryptionByDefaultProperty { SseAlgorithm = "AES256" } } } }; } // Fail synth if public Lambda found without approval if (node is CfnFunction fn && fn.ReservedConcurrentExecutions is null) { Annotations.Of(node).AddWarning( "Lambda function missing ReservedConcurrentExecutions — set to prevent runaway cost"); } } } // Apply aspects to entire app in Program.cs Aspects.Of(app).Add(new WatsonTaggingAspect("prod")); Aspects.Of(app).Add(new BucketEncryptionAspect()); // Every construct in every stack gets visited — zero duplication

Tokens — Late-Bound Values

A Token is a placeholder for a value not known at synth time — like a bucket ARN, which only exists after deployment. CDK encodes these as special strings and resolves them in the CloudFormation template.

C# · Understanding Tokens var bucket = new Bucket(this, "Artifacts"); // bucket.BucketArn is a TOKEN at synth time — not the real ARN yet // It looks like: "${Token[TOKEN.12]}" // CloudFormation resolves it to the real ARN at deploy time Console.WriteLine(bucket.BucketArn); // → "${Token[AWS.Arn.2]}" ← NOT a real ARN yet // ✅ Safe: pass token to another construct — CDK resolves it fn.AddEnvironment("BUCKET_ARN", bucket.BucketArn); // ❌ Unsafe: don't try to use token value at synth time if (bucket.BucketArn.Contains("my-bucket")) // WRONG — token is a placeholder! { // This will never work as expected } // ✅ Use CfnOutput to expose values after deployment new CfnOutput(this, "BucketArn", new CfnOutputProps { Value = bucket.BucketArn }); // Check if a value is a token Token.IsUnresolved(bucket.BucketArn); // → true (before synth)

💡 Always Run cdk diff Before cdk deploy in Prod

The diff output shows IAM changes, resource additions, modifications (~), and deletions (-). Deletions of stateful resources (DynamoDB, RDS, S3) are highlighted — always check before deploying. Use RemovalPolicy.RETAIN on critical resources as a safety net.

🧪 Testing CDK Infrastructure

CDK's killer advantage over YAML-based tools — you can write unit tests for your infrastructure using familiar test frameworks. Tests run in milliseconds without touching AWS.

Three Types of CDK Tests

✅ Assertion Tests

Assert that the synthesised CloudFormation template contains expected resources, properties, and counts. Most common type.

📸 Snapshot Tests

Capture the full synthesised template as a snapshot. Fail if it changes unexpectedly — great for detecting accidental diffs.

✔️ Validation Tests

Test that invalid inputs throw exceptions during construction. Verifies guardrails and error messages in custom constructs.

CDK Assertions — xUnit Tests in C#

🚀 CDK Pipelines — Self-Mutating CI/CD

CDK Pipelines is a high-level construct that creates a self-mutating CodePipeline. "Self-mutating" means the pipeline updates itself when you change the pipeline definition — you never manually update the pipeline in the console.

Best Practices & Common Mistakes

  • Set RemovalPolicy.RETAIN on all stateful resources (DynamoDB, RDS, S3) in prod — CDK's default for some is DESTROY
  • Use Amazon.CDK.Lib (the unified package) — not the old per-service packages like Amazon.CDK.AWS.Lambda
  • Write unit tests for every stack — tests run in milliseconds and catch breaking changes before deployment
  • Never use Stack.of(this).Account or Region as a concrete string at synth time — they may return tokens; use environment-specific configuration instead
  • Run cdk diff before every production deploy — especially watch for resource replacements or deletions
  • Keep stacks under ~200 resources — split large stacks into nested or sibling stacks to avoid CloudFormation limits
  • Use cdk watch with --hotswap during Lambda development — avoids a full CloudFormation deploy for code-only changes
  • Store cdk.context.json in Git — it caches VPC lookups and other AWS API calls for reproducible synth
  • Use Aspects for cross-cutting concerns (tags, encryption enforcement) — don't repeat config on every construct

⚠️ Top CDK Mistakes to Avoid

❌ Using RemovalPolicy.DESTROY on prod databases — a cdk destroy wipes your data permanently
❌ Renaming construct IDs after deploy — CDK uses the ID in the CloudFormation logical ID; renaming = delete + recreate the resource
❌ Not bootstrapping before first deploy — cdk deploy fails without the CDK bootstrap stack in that account/region
❌ Treating Token strings as real values — don't parse bucket.BucketName at synth time; it's a placeholder
❌ Deploying without cdk diff in prod — you might miss a resource replacement that causes downtime
❌ Using --hotswap in production — hotswap bypasses CloudFormation change sets; for dev only
❌ Putting secrets as plain strings in code — use Secrets Manager or SSM Parameter Store and reference by ARN
