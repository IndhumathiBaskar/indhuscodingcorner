🌊 What is Amazon Kinesis?

Amazon Kinesis is a platform for real-time streaming data on AWS. Instead of collecting data in batches and processing it hours later, Kinesis lets you ingest, process, and act on data in milliseconds to seconds — making it ideal for dashboards, fraud detection, IoT telemetry, log analytics, and event-driven architectures.

💡 Kinesis in One Sentence

Kinesis = a managed, durable, ordered, real-time pipe that can ingest millions of events per second and fan them out to multiple consumers — without losing a single record.

When to Use Kinesis Data Streams

✅ Use Kinesis When You Need…

  • Sub-second (real-time) latency
  • Multiple independent consumers on same stream
  • Ordered records within a shard (partition key ordering)
  • Replay — re-read records up to 7 days back
  • High-volume ingestion (GB/s across multiple shards)
  • Event-driven fan-out (Lambda + analytics + archival simultaneously)

⚠️ Consider SQS Instead When…

  • You need simple task queuing — one consumer per message
  • You need automatic message deletion after processing
  • Ordering is not critical (standard SQS)
  • Volume is low and latency of seconds is acceptable
  • You want AWS to handle visibility timeouts & dead-letter queues
  • Fan-out not needed — message consumed once and gone

🏗️ Kinesis Architecture — End-to-End Data Flow

A Kinesis Data Stream sits at the centre: producers push records in, shards store them durably and in order, and consumers pull or receive them in near real-time. Multiple consumers can read the same stream independently without interfering with each other.

🔀 Shards, Throughput & Limits

A shard is the unit of capacity in Kinesis. Every stream has one or more shards. Your throughput is a direct function of how many shards you have. Understanding shard math is critical for sizing and cost optimization.

💡 Hot Shard Warning

If many records share the same partition key (e.g. "default" or a single user ID), they all land on one shard — one shard getting throttled while others sit idle. Use a high-cardinality partition key like userId, deviceId, or add a random suffix.

Shard Math — Sizing Your Stream

📐 How Many Shards Do I Need?

Use the higher of the write limit or record rate limit:

Shards needed = MAX(
  CEIL(incoming_MB_per_sec / 1),      // write data rate
  CEIL(incoming_records_per_sec / 1000) // write record rate
)

// Example: 3.5 MB/s, 2,200 records/s
// CEIL(3.5/1) = 4 shards
// CEIL(2200/1000) = 3 shards
// Answer: 4 shards (data rate is the bottleneck)

Add 20–30% headroom for burst. Enable on-demand mode if traffic is highly variable.

Provisioned vs On-Demand Mode

🔧 Provisioned Mode

  • You specify the exact number of shards
  • Shard splits and merges are manual (or via UpdateShardCount API)
  • Predictable cost — pay per shard-hour
  • Best for steady, predictable throughput
  • Maximum control over partition key distribution

⚡ On-Demand Mode

  • AWS auto-scales shards based on traffic — no capacity planning
  • Instant scale-up, gradual scale-down
  • Pricing: per GB ingested + per GB retrieved (no shard-hour)
  • Best for unpredictable, spiky workloads
  • ~2x cost vs provisioned at sustained high throughput

⚠️ Shard Split / Merge Rules

You can only double (split) or halve (merge) shards at a time. A split takes the hot shard and divides its hash range across 2 new shards. The old shard enters CLOSED state — records are still readable until the retention window expires. Merging is the reverse. Resharding takes 30–60 seconds per split.

👂 Consumers — Reading from Kinesis

Kinesis supports multiple consumer patterns. The right pattern depends on your latency needs, the number of consumers, and whether you manage checkpointing yourself.

Consumer Patterns Overview

🌊 Kinesis Stream
  • Records retained for 24h–365 days
  • Each consumer tracks its own position (iterator)
↓
⚡ Lambda ESM
  • AWS manages polling & batching
  • Serverless — no infrastructure
  • Configurable: batch size, parallelism
  • Best for: event-driven processing
🖥️ KCL App
  • Library handles shard leases + checkpoints
  • DynamoDB table tracks state
  • Re-processes from last checkpoint on restart
  • Best for: complex stateful logic
↓
📡 GetRecords (SDK polling)
  • Direct API — full control, no library
  • You manage iterators and checkpoints
  • 5 calls/s limit per shard
  • Best for: simple custom consumers
🔔 EFO (SubscribeToShard)
  • HTTP/2 push — no polling overhead
  • 2 MB/s dedicated per consumer per shard
  • Register consumer first (RegisterStreamConsumer)
  • Best for: latency-sensitive, many consumers

Lambda Event Source Mapping — Key Settings

⚙️ Batch Settings

  • Batch size: 1–10,000 records per invocation
  • Batch window: 0–300s — buffer time before invoking
  • Starting position: TRIM_HORIZON or LATEST
  • Bisect on error: Split failing batch to isolate poison pill
  • Max retry attempts: Before sending to DLQ

⚡ Parallelization Factor

  • Default: 1 concurrent Lambda per shard
  • Parallelization factor 1–10: multiple concurrent Lambdas per shard
  • 10 shards × factor 5 = 50 concurrent Lambdas
  • Order preserved within a factor's sub-shard partition
  • Use for CPU-bound processing to speed up consumption

💡 KCL Checkpointing — How Restart Recovery Works

KCL stores the last successfully processed sequence number in a DynamoDB table (one row per shard). On restart, it reads the checkpoint and resumes from AFTER_SEQUENCE_NUMBER — never re-processing committed records, never skipping uncommitted ones. This makes KCL exactly-once safe at the checkpoint level (your processing code must be idempotent for true exactly-once semantics).

🔥 Kinesis Data Firehose

Firehose is the zero-code path from a data stream (or direct producers) to a storage/analytics destination. You configure a delivery stream, set buffer size & interval, optionally transform records with Lambda, and Firehose handles compression, format conversion, and batched delivery automatically.

Encryption

🔒 Server-Side Encryption (SSE)

  • Enabled per-stream — encrypts at rest in AWS infrastructure
  • Uses AWS-managed key (free) or customer-managed KMS CMK
  • Transparent — producer/consumer code unchanged
  • Enable at stream creation or update after creation
  • KMS CMK: ~$1/month per key + API call costs

🔗 In-Transit Encryption

  • All Kinesis API calls use HTTPS (TLS 1.2+) — always on
  • No configuration needed — encrypted by default
  • VPC endpoints (PrivateLink) keep traffic off public internet
  • Useful in financial/healthcare regulated environments

VPC Endpoints for Kinesis

1

Create Interface Endpoint

In your VPC, create an Interface VPC Endpoint for com.amazonaws.{region}.kinesis-streams. This gives Kinesis a private IP inside your VPC.

2

Update Security Groups

Allow outbound HTTPS (443) from your producer/consumer EC2 instances or Lambda functions to the endpoint's private IPs.

3

Private DNS

Enable private DNS on the endpoint. The standard Kinesis endpoint URL (kinesis.{region}.amazonaws.com) now resolves to the private IP — no code changes in your application.

Resource-Based Policy (Stream Policy)

📋 When to Use Stream Policies

Stream policies let you grant cross-account access — e.g., a producer in Account A writing to a stream in Account B — without sharing credentials. Both the stream policy (resource-based) and the producer's IAM role (identity-based) must allow the action.

💡 Least-Privilege Checklist

✅ Separate IAM roles for producers and consumers
✅ Restrict by kinesis:stream/StreamName ARN — not wildcard *
✅ Enable SSE with CMK for PII or regulated data
✅ Use VPC endpoint to avoid public internet traversal
✅ Enable CloudTrail for PutRecord and GetRecords auditing
✅ Rotate KMS keys annually

⌨️ C# SDK Examples

Pattern 1 — Producer: PutRecords (Batch Write)

C# · Producer · Batch PutRecords 
using Amazon.Kinesis;
using Amazon.Kinesis.Model;
using System.Text;
using System.Text.Json;

public class KinesisProducer
{
    private readonly IAmazonKinesis _kinesis;
    private const string StreamName = "my-app-events";

    public KinesisProducer(IAmazonKinesis kinesis) => _kinesis = kinesis;

    // Sends up to 500 records in one API call
    public async Task<PutRecordsResponse> SendBatchAsync(
        IEnumerable<AppEvent> events)
    {
        var records = events.Select(e => new PutRecordsRequestEntry
        {
            // PartitionKey determines shard routing (high cardinality = even spread)
            PartitionKey = e.UserId ?? Guid.NewGuid().ToString(),
            Data         = new MemoryStream(
                              Encoding.UTF8.GetBytes(
                                  JsonSerializer.Serialize(e)))
        }).ToList();

        var request = new PutRecordsRequest
        {
            StreamName = StreamName,
            Records    = records   // max 500 per call, max 5 MB total
        };

        var response = await _kinesis.PutRecordsAsync(request);

        // Partial failures — FailedRecordCount > 0 means some records were throttled
        if (response.FailedRecordCount > 0)
        {
            var failed = response.Records
                .Select((r, i) => (r, i))
                .Where(x => x.r.ErrorCode != null)
                .ToList();

            // Retry failed records with exponential backoff
            await RetryFailedAsync(records, failed);
        }

        return response;
    }

    private async Task RetryFailedAsync(
        List<PutRecordsRequestEntry> all,
        List<(PutRecordsResultEntry r, int i)> failed)
    {
        var toRetry = failed.Select(f => all[f.i]).ToList();
        await Task.Delay(TimeSpan.FromMilliseconds(100));  // back-off
        await _kinesis.PutRecordsAsync(new PutRecordsRequest
        {
            StreamName = StreamName,
            Records    = toRetry
        });
    }
}

public record AppEvent(string UserId, string EventType, DateTimeOffset Timestamp);

Pattern 2 — Consumer: GetRecords (Standard Polling)

C# · Consumer · GetRecords polling loop 
public class KinesisConsumer
{
    private readonly IAmazonKinesis _kinesis;
    private const string StreamName = "my-app-events";

    public KinesisConsumer(IAmazonKinesis kinesis) => _kinesis = kinesis;

    public async Task ConsumeAsync(CancellationToken ct)
    {
        // 1. List all shards in the stream
        var shardList = await _kinesis.ListShardsAsync(new ListShardsRequest
        {
            StreamName = StreamName
        });

        var tasks = shardList.Shards.Select(shard =>
            ConsumeShard(shard.ShardId, ct));

        await Task.WhenAll(tasks);
    }

    private async Task ConsumeShard(string shardId, CancellationToken ct)
    {
        // 2. Get an iterator — TRIM_HORIZON = start from oldest record
        var iterResp = await _kinesis.GetShardIteratorAsync(new GetShardIteratorRequest
        {
            StreamName        = StreamName,
            ShardId           = shardId,
            ShardIteratorType = ShardIteratorType.TRIM_HORIZON
        });

        string? iterator = iterResp.ShardIterator;

        while (!ct.IsCancellationRequested && iterator != null)
        {
            var resp = await _kinesis.GetRecordsAsync(new GetRecordsRequest
            {
                ShardIterator = iterator,
                Limit         = 1000   // max per call
            });

            foreach (var record in resp.Records)
            {
                var json = Encoding.UTF8.GetString(record.Data.ToArray());
                var evt  = JsonSerializer.Deserialize<AppEvent>(json);
                await ProcessEventAsync(evt!, record.SequenceNumber);
            }

            // Null iterator = shard is CLOSED (after split/merge)
            iterator = resp.NextShardIterator;

            // Wait if no records — avoid tight loop and respect 5 calls/s limit
            if (resp.Records.Count == 0)
                await Task.Delay(200, ct);
        }
    }

    private async Task ProcessEventAsync(AppEvent evt, string seqNum)
    {
        // Your business logic here
        Console.WriteLine($"Processing {evt.EventType} for user {evt.UserId} @ {seqNum}");
        await Task.CompletedTask;
    }
}

Pattern 3 — Describe Stream & Monitor Shard Health

C# · Admin · Describe stream + shard stats 
public class KinesisAdmin
{
    private readonly IAmazonKinesis _kinesis;
    private readonly IAmazonCloudWatch _cw;

    public KinesisAdmin(IAmazonKinesis k, IAmazonCloudWatch cw)
        => (_kinesis, _cw) = (k, cw);

    public async Task<StreamDescription> DescribeStreamAsync(string name)
    {
        var resp = await _kinesis.DescribeStreamSummaryAsync(
            new DescribeStreamSummaryRequest { StreamName = name });

        var s = resp.StreamDescriptionSummary;
        Console.WriteLine($"Stream: {s.StreamName}");
        Console.WriteLine($"Status: {s.StreamStatus}");
        Console.WriteLine($"Shards: {s.OpenShardCount}");
        Console.WriteLine($"Retention: {s.RetentionPeriodHours}h");
        Console.WriteLine($"Consumers: {s.ConsumerCount}");
        Console.WriteLine($"Encryption: {s.EncryptionType}");

        return resp.StreamDescriptionSummary;
    }

    // GetShardLevelMetrics — IteratorAge tells you if consumers are keeping up
    public async Task<double> GetMaxIteratorAgeAsync(string streamName)
    {
        var now = DateTime.UtcNow;
        var resp = await _cw.GetMetricStatisticsAsync(new()
        {
            Namespace  = "AWS/Kinesis",
            MetricName = "GetRecords.IteratorAgeMilliseconds",
            Dimensions = [new { Name = "StreamName", Value = streamName }],
            StartTime  = now.AddMinutes(-10),
            EndTime    = now,
            Period     = 60,
            Statistics = ["Maximum"]
        });

        var maxAge = resp.Datapoints
            .Select(d => d.Maximum)
            .DefaultIfEmpty(0)
            .Max();

        // > 60,000ms (1 minute) = consumer is falling behind → investigate
        if (maxAge > 60_000)
            Console.WriteLine($"⚠️ Consumer lag: {maxAge / 1000:F0}s — consider adding shards");

        return maxAge;
    }
}

✅ Best Practices & Common Mistakes

Producer Best Practices

  • Use PutRecords (batch) over PutRecord (single) — reduces API calls and cost by up to 500x
  • Choose a high-cardinality partition key (userId, deviceId) — distributes load evenly across shards
  • Retry on ProvisionedThroughputExceededException with exponential backoff + jitter — never drop records
  • Use KPL for very high throughput — it aggregates micro-records before sending, reducing cost
  • Partial failure: always check FailedRecordCount in PutRecords response — it can be >0 with HTTP 200
  • Keep records ≤ 1 MB — compress large payloads before putting (gzip reduces JSON ~60–80%)

Consumer Best Practices

  • Use Enhanced Fan-Out (EFO) when you have ≥3 consumers — dedicated 2 MB/s eliminates read throttling
  • Use KCL for stateful, long-running consumers — it handles shard leases, checkpoints, and resharding automatically
  • Make processing idempotent — Kinesis can deliver the same record more than once (at-least-once)
  • Checkpoint after successful processing — never before — or you risk losing records on restart
  • Monitor GetRecords.IteratorAgeMilliseconds — rising iterator age means consumers are falling behind
  • Handle ExpiredIteratorException — iterators expire after 5 minutes; re-acquire if your loop stalls

Scaling & Cost

  • Start with On-Demand mode for new streams — switch to Provisioned only after observing traffic patterns
  • Set CloudWatch alarms on WriteProvisionedThroughputExceeded — automate shard splits via Lambda trigger
  • Enable Extended Retention (7 days) only if you need replay capability — adds ~$0.023/shard-hour
  • Use Firehose for S3 archival — it's cheaper than running an EC2 consumer just to write files
  • Delete idle streams — a 2-shard stream with no data still costs ~$21/month in shard-hour fees
  • Compress records before putting — reduces PUT payload units (pricing is per 25 KB chunk)

Common Mistakes to Avoid

⚠️ Top Kinesis Mistakes

❌ Using a low-cardinality partition key (e.g. "all" or a single string) — all records land on one shard; others sit idle while one gets throttled
❌ Ignoring FailedRecordCount in PutRecords — a 200 OK response can still have failed records; not checking = silent data loss
❌ Not handling ExpiredIteratorException — if your poll loop pauses >5 minutes, the iterator expires; you need to re-acquire it
❌ Checkpointing before processing — if your app crashes after checkpoint but before successful processing, those records are lost forever
❌ 5 consumers on one standard shard — each GetRecords call competes for the same 2 MB/s; use EFO for dedicated throughput
❌ Forgetting partial batch failures — KPL/PutRecords success ≠ all records stored; always check per-record result entries
❌ Using Kinesis for task queuing — Kinesis does not delete records after read; consumers must track their own position. SQS is the right tool for task queues
❌ Not alarming on IteratorAge — you can have a consumer that is silently hours behind with no errors; add a CloudWatch alarm on P99 IteratorAgeMilliseconds
