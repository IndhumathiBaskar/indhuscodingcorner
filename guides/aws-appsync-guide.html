AWS AppSync Managed GraphQL

What is AWS AppSync?

AWS AppSync is a fully managed GraphQL API service. It sits between your client apps and your backend data sources — handling schema parsing, resolver execution, authentication, real-time subscriptions, and caching. You define your GraphQL schema; AppSync takes care of the infrastructure.

💡 Simple mental model Think of AppSync as API Gateway for GraphQL. Instead of REST endpoints, clients send structured queries. AppSync routes each field in that query to the right backend data source through a resolver.

Why use AppSync?

🚫
No server to manage

No Apollo Server, Express, or container to run. AppSync scales automatically.

⚡
Real-time built-in

GraphQL subscriptions over WebSockets. Supports millions of connected clients. No custom pub/sub needed.

🔗
Multi-source in one query

Aggregate data from DynamoDB, Lambda, RDS, OpenSearch, and HTTP endpoints in a single GraphQL request.

🔐
Multiple auth modes

Cognito, IAM, OpenID Connect, and API Keys — mix and match on the same API.

💾
Server-side caching

High-speed in-memory caches reduce direct data source hits and lower latency.

📡
Offline support

With AWS Amplify: offline data access and automatic conflict resolution on reconnect.

Core Concepts

GraphQL Operations

AppSync exposes three types of GraphQL operations:

🔍
Query

Read data. Like a GET request — fetches fields you ask for, nothing more.

✏️
Mutation

Write / update / delete data. Like POST/PUT/DELETE. Triggers subscriptions.

📡
Subscription

Real-time push. Client subscribes over WebSocket; receives data when a linked mutation fires.

Schema

The schema is the contract of your API. It defines types, which queries exist, which mutations exist, and which subscriptions exist.

type Post {
  id:      ID!
  title:   String!
  content: String
  author:  String!
  createdAt: AWSDateTime
}

type Query {
  getPost(id: ID!): Post
  listPosts: [Post]
}

type Mutation {
  createPost(title: String!, content: String, author: String!): Post
  updatePost(id: ID!, title: String): Post
  deletePost(id: ID!): Post
}

type Subscription {
  onCreatePost: Post @aws_subscribe(mutations: ["createPost"])
  onUpdatePost: Post @aws_subscribe(mutations: ["updatePost"])
}
💡 AWS Scalar types AppSync adds special AWS scalars: AWSDateTime, AWSDate, AWSTime, AWSEmail, AWSJSON, AWSPhone, AWSURL, AWSIPAddress.

Data Source

A data source is a registered backend that your resolvers talk to. You register it once with a name (e.g., BlogPostsTable) and reference it in resolvers.

Resolver

A resolver connects a specific schema field to a data source. It has two parts:

Request Handler
Transforms GraphQL → Data Source format
→
Data Source
→
Response Handler
Transforms result → GraphQL response

Pipeline Resolver

Chains multiple functions in sequence. Useful for auth checks + data fetch + transformation in one request.

→
Function 1
Validate
→
Function 2
Fetch Data
→
Function 3
Transform
→
After Step

Merged API

AppSync supports composing multiple GraphQL APIs (source APIs) into a single Merged API endpoint. Each team owns their sub-schema independently; the merged API unifies them for clients.

AppSync Events (Pub/Sub)

A newer AppSync feature separate from GraphQL — a lightweight HTTP publish / WebSocket subscribe event system using channels and namespaces. No GraphQL schema required.

⚠️ Two distinct AppSync products AppSync GraphQL API — traditional schema-driven queries/mutations/subscriptions.
AppSync Events API — schema-free pub/sub over HTTP + WebSocket channels.

Resolvers In Depth

Resolvers are written in JavaScript (recommended) or the older VTL (Velocity Template Language). Each resolver has a request function and a response function.

Resolver types

TypeWhen to useNotes
Unit Resolver One field → one data source call Simplest; most common
Pipeline Resolver Complex multi-step logic Chain of functions; before/after hooks
NONE Resolver Computed/static fields; subscription triggers No external call; useful for local data

JavaScript resolver — DynamoDB GetItem

// Resolver for: Query.getPost
import { util } from '@aws-appsync/utils'
import * as ddb from '@aws-appsync/utils/dynamodb'

export function request(ctx) {
  return ddb.get({ key: { id: ctx.args.id } })
}

export function response(ctx) {
  const { error, result } = ctx
  if (error) util.error(error.message, error.type)
  return result
}

JavaScript resolver — DynamoDB PutItem (Mutation)

// Resolver for: Mutation.createPost
import { util } from '@aws-appsync/utils'
import * as ddb from '@aws-appsync/utils/dynamodb'

export function request(ctx) {
  const item = {
    ...ctx.args,
    id: util.autoId(),
    createdAt: util.time.nowISO8601()
  }
  return ddb.put({ key: { id: item.id }, item })
}

export function response(ctx) {
  const { error, result } = ctx
  if (error) util.error(error.message, error.type)
  return result
}

JavaScript resolver — Lambda data source

// Resolver for any field backed by a Lambda function
export function request(ctx) {
  return {
    operation: 'Invoke',
    payload: { field: ctx.info.fieldName, arguments: ctx.args }
  }
}

export function response(ctx) {
  if (ctx.error) {
    util.error(ctx.error.message, ctx.error.type)
  }
  return ctx.result
}

Pipeline resolver — chained functions

// Function 1 — authorize the caller
export function request(ctx) {
  if (!ctx.identity?.sub) util.unauthorized()
  return {}   // NONE data source
}
export function response(ctx) { return ctx.prev.result }

// Function 2 — fetch from DynamoDB
import * as ddb from '@aws-appsync/utils/dynamodb'
export function request(ctx) {
  return ddb.get({ key: { id: ctx.args.id } })
}
export function response(ctx) { return ctx.result }

Real-Time with Subscriptions

AppSync subscriptions let clients receive data pushed in real-time when mutations occur. AppSync manages all WebSocket infrastructure, connection lifecycle, and fanout to connected clients.

💡 How subscriptions work A subscription is always triggered by a mutation. When a mutation fires, AppSync broadcasts the result to all clients currently subscribed to that subscription field.

Schema setup

type Subscription {
  # Fires when ANY createPost mutation runs
  onCreatePost: Post
    @aws_subscribe(mutations: ["createPost"])

  # Fires only for a specific author (argument filter)
  onPostByAuthor(author: String!): Post
    @aws_subscribe(mutations: ["createPost"])
}

How subscription filtering works

🧑‍💻
Basic Filtering (client-defined)

Client passes subscription arguments. AppSync filters results client-side — only sends matching data.

🔒
Enhanced Filtering (backend-defined)

Resolver defines filters with extra operators (gt, lt, contains, beginsWith…). Enforced centrally on the server.

Client subscription example (JavaScript)

// Using AWS Amplify client
import { generateClient } from 'aws-amplify/api'

const client = generateClient()

const subscription = client
  .graphql({
    query: `subscription OnCreatePost {
      onCreatePost {
        id title author createdAt
      }
    }`
  })
  .subscribe({
    next: ({ data }) => console.log('New post:', data.onCreatePost),
    error: (err) =>  console.error(err)
  })

// Unsubscribe when done
subscription.unsubscribe()

Subscription with enhanced filtering — resolver

// Subscription resolver attached to onSpecialTicketCreated
// Uses NONE data source — no external call needed
import { extensions } from '@aws-appsync/utils'

export function request(ctx) { return {} }

export function response(ctx) {
  // Only deliver if severity >= 7
  extensions.setSubscriptionFilter(
    extensions.filter.ge('severity', 7)
  )
  return {}
}

Authentication & Authorization

AppSync supports four auth modes. You set a default auth mode when creating the API and can add additional modes. Field-level auth directives let you mix modes on the same schema.

Auth ModeBest ForDirective
API Key Public APIs, demos, development @aws_api_key
Amazon Cognito User Pools End-user apps with login/signup @aws_cognito_user_pools
AWS IAM Lambda, EC2, backend services @aws_iam
OpenID Connect Third-party identity providers (Auth0, etc.) @aws_oidc
Lambda Authorizer Custom auth logic @aws_lambda

Field-level auth in schema

type Post {
  id:      ID!
  title:   String!
  # Any authenticated user (Cognito or API Key) can read
  content: String @aws_api_key @aws_cognito_user_pools
  # Admin-only field — IAM only
  adminNotes: String @aws_iam
}

type Mutation {
  # Only callable by backend services (IAM)
  createPost(title: String!): Post @aws_iam
  # Callable by logged-in users via Cognito
  updatePost(id: ID!, title: String): Post @aws_cognito_user_pools
}

Auth in resolvers — using identity context

// Resolver: only return posts the caller owns
import * as ddb from '@aws-appsync/utils/dynamodb'
import { util } from '@aws-appsync/utils'

export function request(ctx) {
  const callerId = ctx.identity?.sub    // Cognito user ID

  if (!callerId) util.unauthorized()

  return ddb.query({
    index: 'authorIndex',
    query: {
      expression: 'author = :userId',
      expressionValues: { ':userId': callerId }
    }
  })
}

export function response(ctx) {
  if (ctx.error) util.error(ctx.error.message)
  return ctx.result.items
}
⚠️ IAM as primary auth mode for EventBridge If you're using AppSync as an EventBridge target, IAM must be the primary authorization mode — not an additional mode. This is a commonly missed configuration.

Using AppSync from C# / .NET

Your C# backend can interact with AppSync in two main ways: as a Lambda resolver (handling inbound GraphQL requests) or as a client calling AppSync mutations (to trigger subscriptions).

1. Lambda resolver — handling AppSync requests in C#

When AppSync resolves a field via a Lambda data source, it sends a structured JSON payload to your Lambda. Your C# function receives and responds to it.

// NuGet: Amazon.Lambda.Core, Amazon.Lambda.Serialization.SystemTextJson
using Amazon.Lambda.Core;
using System.Text.Json.Serialization;

[assembly: LambdaSerializer(typeof(Amazon.Lambda.Serialization.SystemTextJson.DefaultLambdaJsonSerializer))]

public class AppSyncEvent
{
    [JsonPropertyName("arguments")]
    public Dictionary<string, object> Arguments { get; set; }

    [JsonPropertyName("identity")]
    public AppSyncIdentity Identity { get; set; }

    [JsonPropertyName("info")]
    public AppSyncInfo Info { get; set; }
}

public class AppSyncIdentity
{
    [JsonPropertyName("sub")]
    public string Sub { get; set; }

    [JsonPropertyName("username")]
    public string Username { get; set; }
}

public class AppSyncInfo
{
    [JsonPropertyName("fieldName")]
    public string FieldName { get; set; }
}

public class Post
{
    public string Id      { get; set; }
    public string Title   { get; set; }
    public string Author  { get; set; }
    public string CreatedAt { get; set; }
}

public class PostResolver
{
    public async Task<Post> Handler(AppSyncEvent appsyncEvent, ILambdaContext context)
    {
        context.Logger.LogLine($"Field: {appsyncEvent.Info.FieldName}");
        context.Logger.LogLine($"Called by: {appsyncEvent.Identity?.Username}");

        var title = appsyncEvent.Arguments["title"]?.ToString();

        // … your business logic + DynamoDB call …
        return new Post
        {
            Id        = Guid.NewGuid().ToString(),
            Title     = title,
            Author    = appsyncEvent.Identity?.Username ?? "unknown",
            CreatedAt = DateTime.UtcNow.ToString("o")
        };
    }
}

2. Calling AppSync mutations from C# (IAM-signed)

A backend Lambda or service can call AppSync mutations directly using AWS Signature V4 (IAM auth). This triggers subscriptions for connected clients.

// NuGet: AWSSDK.Core, System.Net.Http.Json
using Amazon.Runtime;
using Amazon.Runtime.Internal.Auth;
using System.Net.Http.Headers;
using System.Text;
using System.Text.Json;

public class AppSyncClient
{
    private readonly string _endpoint = "https://xxxxxx.appsync-api.us-east-1.amazonaws.com/graphql";
    private readonly HttpClient _http = new();

    public async Task<string> MutateAsync(string title, string author)
    {
        var mutation = new
        {
            query = @"mutation CreatePost($title: String!, $author: String!) {
                        createPost(title: $title, author: $author) { id title author }
                      }",
            variables = new { title, author }
        };

        var body = JsonSerializer.Serialize(mutation);
        var request = new HttpRequestMessage(HttpMethod.Post, _endpoint)
        {
            Content = new StringContent(body, Encoding.UTF8, "application/json")
        };

        // Add AWS SigV4 headers using AWS credentials from environment
        var credentials = await FallbackCredentialsFactory.GetCredentials()
                                                                .GetCredentialsAsync();

        // Sign the request (see AWS SDK SigV4Signer for full implementation)
        // This example shows the pattern; use AWS4Signer in production
        request.Headers.Add("x-amz-security-token", credentials.Token);

        var response = await _http.SendAsync(request);
        return await response.Content.ReadAsStringAsync();
    }
}

3. Reading AppSync config from appsettings.json

// appsettings.json
{
  "AppSync": {
    "Endpoint": "https://xxxxxx.appsync-api.us-east-1.amazonaws.com/graphql",
    "ApiKey":  "da2-xxxxxxxxxxxxxxxxxxxx",
    "Region":  "us-east-1"
  }
}

// Using API Key auth (dev/testing only)
public class AppSyncApiKeyClient
{
    private readonly HttpClient _http;
    private readonly string _endpoint;

    public AppSyncApiKeyClient(IConfiguration config, HttpClient http)
    {
        _http     = http;
        _endpoint = config["AppSync:Endpoint"];
        _http.DefaultRequestHeaders.Add("x-api-key", config["AppSync:ApiKey"]);
    }

    public async Task<string> QueryAsync(string graphqlQuery)
    {
        var payload = JsonSerializer.Serialize(new { query = graphqlQuery });
        var response = await _http.PostAsync(
            _endpoint,
            new StringContent(payload, Encoding.UTF8, "application/json")
        );
        return await response.Content.ReadAsStringAsync();
    }
}
✅ Best practices for C# + AppSync Use Lambda as data source when you need complex C# business logic — DynamoDB resolvers (JS) are better for simple CRUD. Register HttpClient via IHttpClientFactory in Lambda to reuse connections across warm invocations.

AWS Console Setup — Step by Step

This tab walks through creating a complete AppSync API from the AWS Console.

Step 1 — Create the API

1
Open AppSync

Search "AppSync" in the AWS Console → click Create API.

2
Choose API type

Select GraphQL APIs → Design from scratch. (Choose Event APIs for schema-free pub/sub.)

3
Fill in API details
API Name
Give a meaningful name. e.g. BlogAPI. Used to identify the API in console and CloudWatch.
Default Auth Mode
Choose how clients authenticate by default. API Key = simplest for dev. Cognito = user-facing apps. IAM = backend services. You can add more auth modes after creation.
Additional Auth Modes
Optional. Add a second auth mode for field-level control. E.g. primary = Cognito, additional = IAM for backend calls.
API Key Expiration
Only if API Key auth selected. Keys expire in 1–365 days. Rotate before expiry.

Step 2 — Define your schema

1
Go to Schema tab

In the left nav, click Schema. You'll see an editor.

2
Paste your schema

Write or paste your GraphQL schema with types, queries, mutations, and subscriptions.

3
Save schema

Click Save Schema. AppSync validates the SDL immediately.

Step 3 — Create a data source

Data Source Name
Logical name referenced in resolvers. e.g. BlogPostsTable. No spaces allowed.
Data Source Type
Pick the backend: Amazon DynamoDB table, AWS Lambda function, Amazon OpenSearch domain, HTTP endpoint, or None.
Region
AWS region of the data source. Usually same region as your AppSync API.
Table Name (DynamoDB)
The DynamoDB table to connect. The table must already exist.
IAM Role
IAM role AppSync will assume to access the data source. Use New Role to let AWS auto-create a scoped role, or pick an existing one.

Step 4 — Attach resolvers

1
Go to Schema → find a field

In the Schema editor, each resolver-attachable field has an Attach button in the right panel.

2
Choose resolver type

Unit resolver (one data source) or Pipeline resolver (chain of functions).

3
Select data source

Pick the data source you created in Step 3.

4
Write resolver code

Choose JavaScript (recommended). Write the request and response functions. The console provides helpers and templates for common operations.

5
Test locally

Use the Test tab in the resolver editor to run with mock data — without touching real data sources.

6
Save resolver

Click Save Resolver. Repeat for every field that needs resolving.

Step 5 — Test your API

1
Open Queries tab

Left nav → Queries. A built-in GraphiQL-style editor.

2
Run a mutation

e.g. mutation { createPost(title: "Hello", author: "Indhu") { id title } }

3
Run a query

e.g. query { listPosts { id title author createdAt } }

4
Test a subscription

Start a subscription in the Queries tab, then fire a mutation in a separate browser tab to see real-time data pushed.

Step 6 — Enable caching (optional)

Cache Instance Type
Size of the managed cache cluster. Start with SMALL for dev, scale up for production.
TTL
Time-to-live in seconds for cached responses. 5–300s typical for dynamic data; longer for static data.
Encryption
Toggle encryption at rest and in transit for the cache. Recommended for production.
Per-resolver caching
After enabling API-level caching, go to each resolver and enable caching individually + set per-resolver TTL.

Step 7 — Enable CloudWatch logging

CloudWatch Logs Role ARN
IAM role that allows AppSync to write to CloudWatch. Use New Role for auto-creation.
Field Log Level
NONE — no field-level logs. ERROR — only errors. ALL — every resolver request/response (verbose, useful for debugging; disable in production to save cost).
Exclude verbose content
When log level is ALL, toggle this to suppress request/response body from logs — reduces log volume.

Pagination pattern

type PostConnection {
  items:     [Post]
  nextToken: String
}

type Query {
  listPosts(limit: Int, nextToken: String): PostConnection
}

// Resolver (DynamoDB scan with pagination)
import * as ddb from '@aws-appsync/utils/dynamodb'

export function request(ctx) {
  return ddb.scan({
    limit:     ctx.args.limit ?? 20,
    nextToken: ctx.args.nextToken
  })
}
export function response(ctx) { return ctx.result }

Cost considerations

💡 AppSync pricing model You pay per query/mutation operation, per real-time update (subscription message delivered), per connection-minute (WebSocket open duration), and for caching instance hours. Real-time features with many connected clients can increase costs — monitor with CloudWatch.
