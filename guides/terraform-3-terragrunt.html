🤔 Why Does Terragrunt Exist?

You've followed the Part 1 folder structure — separate folders for dev, staging, and prod. Each environment has its own backend.tf, providers.tf, and terraform.tfvars. Your prod and staging backend configs are identical except for the S3 key name. Your provider blocks are copy-pasted three times. When you need to change the AWS region, you edit three files. When you add a fourth environment, you copy-paste everything again.
🤔 Can't I just use a shared variables file or module for that repeated config?

The problem is that backend blocks and provider blocks in Terraform cannot use variables or module outputs — they're evaluated before any variable resolution happens. So you literally cannot DRY them out in plain Terraform. That's the core gap Terragrunt fills.

💡 What Terragrunt Actually Is

Terragrunt is a thin wrapper around Terraform. It doesn't replace Terraform — it runs Terraform under the hood. What it adds: a terragrunt.hcl config file that CAN use variables, functions, and inheritance. Terragrunt reads this file, generates the backend and provider config from it dynamically, then calls terraform init / plan / apply on your behalf. Your Terraform modules are completely unchanged.

The Problems Terragrunt Solves

😩 Without Terragrunt

  • Backend config copy-pasted in every environment folder
  • Provider blocks duplicated everywhere
  • To deploy all environments: cd into each, run apply separately
  • No enforced ordering — forgetting to apply networking before compute causes errors
  • Common inputs (account ID, region) repeated in every tfvars

😊 With Terragrunt

  • Backend config defined ONCE in a root terragrunt.hcl, inherited everywhere
  • Provider config generated from a single shared file
  • terragrunt run-all apply deploys everything in dependency order
  • Dependencies declared explicitly — networking always applies before compute
  • Common inputs defined once at root level, available everywhere

Terragrunt vs Plain Terraform — When to Use Which

✅ Use plain Terraform when...

  • Single environment or small team
  • One or two AWS accounts
  • Learning Terraform (don't add Terragrunt complexity yet)
  • Simple repo with 2-3 root modules

✅ Add Terragrunt when...

  • Multiple environments (dev/staging/prod) with lots of repeated config
  • Multiple AWS accounts or regions
  • Split state across components that depend on each other
  • Need to deploy ALL components in the right order with one command

💡 Terragrunt Doesn't Change Your Terraform Modules

Your existing modules/ folder, resource definitions, and HCL code stay exactly as they are. Terragrunt just wraps the invocation — generating backend config, passing inputs, and orchestrating order. If you decided to remove Terragrunt later, your Terraform modules work exactly the same.

📁 Folder Structure with Terragrunt

Terragrunt introduces a layered folder structure where each level has a terragrunt.hcl file. The root file sets account-wide defaults. Environment files inherit and extend. Component files declare which module to call and what inputs to pass. Each level adds only what's different from the level above.

The Standard Terragrunt Structure

infrastructure/ ├── terragrunt.hcl # ROOT — backend config, common inputs, provider ├── common.hcl # shared variables (account ID, region, etc.) │ ├── modules/ # plain Terraform modules — no terragrunt.hcl needed │ ├── networking/ │ │ ├── main.tf │ │ ├── variables.tf │ │ └── outputs.tf │ ├── ecs-service/ │ └── rds/ │ └── environments/ ├── dev/ │ ├── env.hcl # env-level vars: environment="dev", etc. │ ├── networking/ │ │ └── terragrunt.hcl # calls modules/networking, passes dev vars │ ├── database/ │ │ └── terragrunt.hcl # depends on networking │ └── ecs-service/ │ └── terragrunt.hcl # depends on networking + database ├── staging/ │ ├── env.hcl │ ├── networking/ │ │ └── terragrunt.hcl # identical structure — different env.hcl values │ ├── database/ │ └── ecs-service/ └── prod/ ├── env.hcl ├── networking/ ├── database/ └── ecs-service/
🤔 Each environment has the same subfolder structure — isn't that still repetition?

The STRUCTURE is repeated (same folder names), but the terragrunt.hcl files inside are tiny — they just declare which module to call and what env-specific values to pass. All the backend config, provider config, and common settings come from the root and env.hcl via include blocks. The difference between a dev and prod ECS service terragrunt.hcl might be just 5 lines — the rest is inherited.

What Each File Does

📄 Root terragrunt.hcl

Defines the S3 backend config using dynamic values — no copy-pasting. Also sets provider config and account-wide inputs. Every child terragrunt.hcl can inherit this with include "root".

📄 common.hcl

A plain HCL file (not a terragrunt.hcl) holding shared locals — AWS account ID, region, project name. Read by other files using read_terragrunt_config().

📄 env.hcl

Environment-specific locals — environment = "dev", CIDR blocks, instance sizes for this env. Read by component terragrunt.hcl files within the same environment folder.

📄 Component terragrunt.hcl

The leaf file — inherits root config, reads env.hcl, declares the Terraform module source, and passes inputs. Usually 15-30 lines.

⚡ Your First terragrunt.hcl

Let's build from scratch. We'll create the root config that generates backend config dynamically, then a component config that calls a Terraform module and inherits from root.

Root terragrunt.hcl

HCL — infrastructure/terragrunt.hcl (root)
# Read shared variables from common.hcl
locals {
  common = read_terragrunt_config(find_in_parent_folders("common.hcl"))

  aws_region  = local.common.locals.aws_region
  account_id  = local.common.locals.account_id
  project     = local.common.locals.project
}

# S3 backend — generated dynamically from path and locals
# path_relative_to_include() gives the folder path relative to this root file
# e.g. "environments/dev/networking"
remote_state {
  backend = "s3"
  generate = {
    path      = "backend.tf"
    if_exists = "overwrite_terragrunt"
  }
  config = {
    bucket         = "${local.project}-${local.account_id}-terraform-state"
    key            = "${path_relative_to_include()}/terraform.tfstate"
    region         = local.aws_region
    dynamodb_table = "${local.project}-terraform-state-lock"
    encrypt        = true
  }
}

# Generate a providers.tf file in every component — no copy-pasting
generate "provider" {
  path      = "provider.tf"
  if_exists = "overwrite_terragrunt"
  contents  = <
}

# Inputs available to every component that includes this root
inputs = {
  aws_region = local.aws_region
  account_id = local.account_id
}
HCL — infrastructure/common.hcl
locals {
  project    = "myapp"
  aws_region = "eu-west-1"
  account_id = "123456789012"
}
HCL — environments/dev/env.hcl
locals {
  environment    = "dev"
  vpc_cidr       = "10.1.0.0/16"
  instance_type  = "t3.micro"
  min_capacity   = 1
  max_capacity   = 3
  multi_az_db    = false
}
HCL — environments/dev/networking/terragrunt.hcl (a leaf component)
# Inherit everything from the root terragrunt.hcl
# find_in_parent_folders() walks up until it finds the file
include "root" {
  path   = find_in_parent_folders()
  expose = true
}

# Read environment-specific values
locals {
  env = read_terragrunt_config(find_in_parent_folders("env.hcl"))
}

# Point to the reusable Terraform module
terraform {
  source = "../../../modules/networking"
}

# Inputs passed to the Terraform module as variables
inputs = {
  environment  = local.env.locals.environment
  vpc_cidr     = local.env.locals.vpc_cidr
  aws_region   = include.root.locals.aws_region
}

💡 path_relative_to_include() Is the Key to Unique State Keys

In the root backend config, key = "${path_relative_to_include()}/terraform.tfstate" automatically generates a unique S3 key for each component — environments/dev/networking/terraform.tfstate, environments/dev/database/terraform.tfstate, etc. No manual key management, no collisions.

🔁 DRY Configs & Includes

The core Terragrunt superpower is the include block — it lets a child config inherit everything from a parent, override just what's different, and merge inputs. This is how you avoid writing the same backend config 20 times across your environments.

include — Inheriting Parent Config

HCL — include patterns
# Basic include — inherit everything from root
include "root" {
  path = find_in_parent_folders()
}

# expose = true — makes parent's locals available via include.root.locals
include "root" {
  path   = find_in_parent_folders()
  expose = true
}
# Then use: include.root.locals.aws_region, include.root.inputs.account_id etc.

# Multiple includes — inherit from both root and a shared environment config
include "root" {
  path   = find_in_parent_folders()
  expose = true
}

include "env" {
  path   = find_in_parent_folders("env.hcl")
  expose = true
}
# Then: include.env.locals.environment, include.env.locals.instance_type

generate — Creating Files Automatically

HCL — generate blocks for providers and versions
# In root terragrunt.hcl — generate a versions.tf in every component
generate "versions" {
  path      = "versions.tf"
  if_exists = "overwrite_terragrunt"
  contents  = <
}

# generate for a provider with assume_role (cross-account deployments)
generate "provider" {
  path      = "provider.tf"
  if_exists = "overwrite_terragrunt"
  contents  = <
}

Merging Inputs at Multiple Levels

HCL — how inputs merge from root → env → component
# Root terragrunt.hcl — account-wide inputs
inputs = {
  aws_region = "eu-west-1"
  account_id = "123456789012"
}

# environments/dev/env.hcl — environment inputs
# (if this file is included, its inputs also merge in)
inputs = {
  environment  = "dev"
  instance_type = "t3.micro"
}

# environments/dev/ecs-service/terragrunt.hcl — component inputs
# All three levels merge together. Component values override parent values
# if the same key appears at multiple levels (deepest wins)
inputs = {
  service_name  = "api"
  desired_count = 1
  # The Terraform module receives ALL of:
  # aws_region, account_id, environment, instance_type, service_name, desired_count
}

# Explicit merge using merge() function
inputs = merge(
  include.root.inputs,
  {
    service_name  = "api"
    desired_count = 1
  }
)

read_terragrunt_config — Reading Sibling Files

HCL — reading account and env config
locals {
  # Read the common.hcl at the root of the repo
  common = read_terragrunt_config(find_in_parent_folders("common.hcl"))

  # Read the env.hcl in the environment folder above this component
  env    = read_terragrunt_config(find_in_parent_folders("env.hcl"))

  # Now use the values
  project     = local.common.locals.project
  environment = local.env.locals.environment
  vpc_cidr    = local.env.locals.vpc_cidr
}

🔗 Dependencies

Your ECS service needs the VPC ID and private subnet IDs from the networking component. In plain Terraform with separate state files, you'd use terraform_remote_state data sources. Terragrunt has a cleaner way: declare a dependency block, and use the outputs of one component directly as inputs in another.
🤔 What's the difference between dependency and terraform_remote_state?

Both read another component's outputs. The difference: terraform_remote_state is Terraform code inside your module — it tightly couples your module to the state location. A Terragrunt dependency block is outside the module — your module stays clean, and Terragrunt passes the outputs as inputs. Much better separation of concerns.

Declaring and Using Dependencies

HCL — environments/dev/ecs-service/terragrunt.hcl
include "root" {
  path   = find_in_parent_folders()
  expose = true
}

locals {
  env = read_terragrunt_config(find_in_parent_folders("env.hcl"))
}

# Declare dependency on the networking component
# Terragrunt reads its state to get the outputs
dependency "networking" {
  config_path = "../networking"

  # Mock outputs — used when running plan BEFORE networking is applied
  # Without this, plan would fail because networking state doesn't exist yet
  mock_outputs = {
    vpc_id             = "vpc-00000000"
    private_subnet_ids = ["subnet-00000000", "subnet-11111111"]
    ecs_cluster_id     = "arn:aws:ecs:eu-west-1:123:cluster/mock"
  }
  mock_outputs_allowed_terraform_commands = ["plan", "validate"]
}

dependency "database" {
  config_path = "../database"
  mock_outputs = {
    db_endpoint = "mock.rds.endpoint:5432"
  }
  mock_outputs_allowed_terraform_commands = ["plan", "validate"]
}

terraform {
  source = "../../../modules/ecs-service"
}

inputs = {
  environment       = local.env.locals.environment
  desired_count     = local.env.locals.min_capacity
  image_tag         = "latest"

  # Values from dependencies — pulled directly from networking's outputs
  vpc_id            = dependency.networking.outputs.vpc_id
  private_subnet_ids = dependency.networking.outputs.private_subnet_ids
  cluster_id        = dependency.networking.outputs.ecs_cluster_id

  # From database dependency
  db_endpoint       = dependency.database.outputs.db_endpoint
}

💡 Always Add mock_outputs for plan

Without mock outputs, terragrunt plan on the ECS service would fail if the networking state doesn't exist yet — it tries to read the state and finds nothing. Mock outputs let CI/CD pipelines validate and plan ALL components even before any are applied, which is essential for PR checks.

🚀 run-all & Stacks

The single biggest time-saver Terragrunt offers: instead of cd-ing into networking, applying, then cd-ing into database, applying, then ecs-service — you run one command from the environment folder and Terragrunt figures out the order from dependency declarations and applies everything.

run-all Commands

Shell — run-all applies all components in dependency order
# From the environment folder — apply ALL components
cd environments/dev
terragrunt run-all apply

# Terragrunt builds the dependency graph, then:
# 1. Applies networking (no dependencies)
# 2. Applies database (depends on networking — waits for step 1)
# 3. Applies ecs-service (depends on both — waits for steps 1 and 2)
# Independent components apply in PARALLEL

# Plan all components
terragrunt run-all plan

# Destroy in REVERSE dependency order
terragrunt run-all destroy

# Validate all
terragrunt run-all validate

# From the root — apply ALL environments (careful in prod!)
cd infrastructure
terragrunt run-all apply

# Apply only specific environments by targeting folders
terragrunt run-all apply --terragrunt-include-dir "environments/dev/*"

# Exclude a specific component
terragrunt run-all apply --terragrunt-exclude-dir "environments/prod/database"
🤔 What if one component fails mid-run-all? Does everything roll back?

No — Terragrunt doesn't do atomic rollbacks. If networking applies successfully but database fails, networking stays applied. Terragrunt will show which components succeeded and which failed. You fix the failing component and run run-all apply again — already-applied components will show "no changes" and continue.

Stacks (Terragrunt v0.67+)

Stacks are Terragrunt's newer, more explicit way to define a group of units that should be deployed together. Instead of Terragrunt discovering components by folder traversal, you declare them explicitly in a terragrunt.stack.hcl file.
HCL — environments/dev/terragrunt.stack.hcl
# Explicitly declare the units in this stack
unit "networking" {
  source = "./networking"
}

unit "database" {
  source    = "./database"
  # Dependencies still respected from the unit's own terragrunt.hcl
}

unit "ecs_service" {
  source = "./ecs-service"
}

# Run all units in the stack
# terragrunt stack run apply
# terragrunt stack run plan
# terragrunt stack run destroy

💡 run-all vs Stacks

run-all discovers components by recursively finding all terragrunt.hcl files under the current folder. Stacks are explicit — you list exactly which units belong. For large repos, stacks are more predictable and easier to reason about. run-all is simpler and works great for smaller setups.

🪝 Hooks & Inputs

Hooks let you run arbitrary commands before or after Terraform runs. This is how you do things like validate a Docker image exists before deploying ECS, run database migrations after applying, or generate a config file that Terraform needs to read.

before_hook and after_hook

HCL — hooks in a component terragrunt.hcl
terraform {
  source = "../../../modules/ecs-service"

  # Run BEFORE terraform init
  before_hook "validate_image" {
    commands = ["apply", "plan"]
    execute  = ["bash", "-c", "aws ecr describe-images --repository-name myapp --image-ids imageTag=${var.image_tag}"]
    run_on_error = false  # abort if this fails
  }

  # Run AFTER terraform apply succeeds
  after_hook "run_migrations" {
    commands     = ["apply"]
    execute      = ["bash", "-c", "./scripts/run_migrations.sh"]
    run_on_error = false
  }

  # Run on errors — e.g., upload logs to S3
  error_hook "notify_slack" {
    commands     = ["apply"]
    execute      = ["bash", "-c", "./scripts/notify_slack.sh 'Terragrunt apply failed in ${local.env}'"]
    run_on_error = true
  }
}

Advanced Inputs — Dynamic and Conditional

HCL — dynamic inputs from locals and conditionals
locals {
  env     = read_terragrunt_config(find_in_parent_folders("env.hcl"))
  is_prod = local.env.locals.environment == "prod"

  # Get the current git commit SHA for image tagging
  image_tag = run_cmd("--terragrunt-quiet", "git", "rev-parse", "--short", "HEAD")

  # Get AWS account ID dynamically from STS
  account_id = run_cmd("--terragrunt-quiet", "aws", "sts", "get-caller-identity", "--query", "Account", "--output", "text")
}

inputs = {
  environment    = local.env.locals.environment
  image_tag      = local.image_tag
  account_id     = local.account_id

  # Conditional values based on environment
  desired_count  = local.is_prod ? 3 : 1
  multi_az       = local.is_prod
  deletion_protection = local.is_prod

  # From get_env() — read from OS environment variables
  slack_webhook  = get_env("SLACK_WEBHOOK_URL", "")
}

Useful Built-in Functions

HCL — terragrunt built-in functions
locals {
  # find_in_parent_folders() — walk up until the file is found
  root_path = find_in_parent_folders()

  # get_terragrunt_dir() — absolute path of current terragrunt.hcl
  current_dir = get_terragrunt_dir()

  # get_parent_terragrunt_dir() — parent's directory
  parent_dir = get_parent_terragrunt_dir()

  # path_relative_to_include() — relative path from root to current file
  # "environments/dev/networking" — used for S3 state keys
  rel_path = path_relative_to_include()

  # get_env() — read OS environment variable with optional default
  image_tag = get_env("IMAGE_TAG", "latest")

  # run_cmd() — run a shell command and capture its output
  git_sha = run_cmd("--terragrunt-quiet", "git", "rev-parse", "--short", "HEAD")

  # get_aws_account_id() — built-in, no need for run_cmd
  account_id = get_aws_account_id()

  # get_aws_region() — current region
  region = get_aws_region()
}

Pattern 2 — Using Public Registry Modules

HCL — calling a registry module from Terragrunt
terraform {
  # Use a public registry module — double slash separates module path from submodule
  source = "tfr:///terraform-aws-modules/vpc/aws?version=5.1.0"
  # or from git:
  source = "git::https://github.com/terraform-aws-modules/terraform-aws-vpc.git///?ref=v5.1.0"
}

inputs = {
  name = "${local.name_prefix}-vpc"
  cidr = local.env.locals.vpc_cidr
  azs  = ["eu-west-1a", "eu-west-1b", "eu-west-1c"]
  private_subnets    = ["10.0.1.0/24", "10.0.2.0/24", "10.0.3.0/24"]
  public_subnets     = ["10.0.101.0/24", "10.0.102.0/24", "10.0.103.0/24"]
  enable_nat_gateway = true
  single_nat_gateway = !local.is_prod
}

Pattern 3 — CI/CD Pipeline Integration

YAML — GitHub Actions workflow
name: Terragrunt Deploy
on:
  push:
    branches: [main]

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Setup Terraform
        uses: hashicorp/setup-terraform@v3
        with:
          terraform_version: "1.7.0"

      - name: Install Terragrunt
        run: |
          wget -q https://github.com/gruntwork-io/terragrunt/releases/download/v0.57.0/terragrunt_linux_amd64
          chmod +x terragrunt_linux_amd64
          sudo mv terragrunt_linux_amd64 /usr/local/bin/terragrunt

      - name: Configure AWS Credentials
        uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: arn:aws:iam::123456789012:role/GitHubActionsRole
          aws-region: eu-west-1

      - name: Plan Dev
        run: |
          cd environments/dev
          terragrunt run-all plan --terragrunt-non-interactive
        env:
          IMAGE_TAG: ${{ github.sha }}

      - name: Apply Dev
        run: |
          cd environments/dev
          terragrunt run-all apply --terragrunt-non-interactive
        env:
          IMAGE_TAG: ${{ github.sha }}

Pattern 4 — Terragrunt + Atlantis

YAML — atlantis.yaml for Terragrunt
version: 3

  mode: enabled
projects:
  - name: dev-networking
    dir: environments/dev/networking
    workflow: terragrunt
  - name: dev-ecs-service
    dir: environments/dev/ecs-service
    workflow: terragrunt
workflows:
  terragrunt:
    plan:
      steps:
        - run: terragrunt plan -out $PLANFILE
    apply:
      steps:
        - run: terragrunt apply $PLANFILE

💡 --terragrunt-non-interactive in CI/CD

Always pass --terragrunt-non-interactive in CI/CD pipelines — it tells Terragrunt not to prompt for confirmation on destructive operations (like run-all destroy). Combine with -auto-approve on the Terraform side when you want fully automated applies.
