🔩 What Are Structural Patterns?

👶 What Do Structural Patterns Do?

Structural patterns deal with how classes and objects are composed together to form larger structures. They help you build flexible, clean relationships between objects — making incompatible interfaces work together, adding new behaviour without changing existing code, simplifying complex subsystems, and treating groups of objects as a single object.

The 6 Structural Patterns — At a Glance

🔌
Adapter
Make incompatible interfaces work together. Wrap an existing class to match an expected interface.
"I need X interface but only have Y class"
🎁
Decorator
Add behaviour to objects dynamically without changing their class. Wrap objects in other objects.
"Add features to an object without subclassing"
🏠
Facade
Provide a simple interface to a complex subsystem. Hide complexity behind one easy-to-use class.
"This system is too complex — give me one simple entry point"
🛡️
Proxy
Provide a substitute that controls access to another object — for caching, security, or lazy loading.
"I need to control access to this object"
🌳
Composite
Treat individual objects and groups of objects the same way. Build tree structures.
"I need to treat single items and collections identically"
🌉
Bridge
Separate abstraction from implementation so both can vary independently.
"Abstraction and implementation should evolve separately"

🔌 Adapter Pattern

👶 In Plain English

The Adapter pattern is exactly like a power plug adapter. Your phone charger has a USB-C plug, but the wall socket accepts three-prong plugs. The adapter sits in between and makes them compatible. In code, an Adapter wraps a class with an incompatible interface and makes it look like the interface you expect.

💡 Real World Analogy

You hire a French-speaking developer. Your team speaks English. The translator (Adapter) sits between them — your team talks to the translator in English, the translator talks to the developer in French. Your team doesn't change how they speak; the developer doesn't change how they work. The adapter makes them compatible.

The Problem — Incompatible Interfaces

PROBLEM C# — Third-party class has the wrong interface
// You have an application that uses ILogger for all logging
public interface ILogger
{
    void Log(string message);
    void LogError(string message);
}

// But the third-party logging library you want to use has a DIFFERENT interface
public class ThirdPartyLogger  // you can't modify this — it's external
{
    public void WriteInfo(string msg)  => Console.WriteLine($"[INFO] {msg}");
    public void WriteError(string msg) => Console.WriteLine($"[ERROR] {msg}");
}

// ❌ Can't use ThirdPartyLogger where ILogger is expected — different methods
public class OrderService
{
    private readonly ILogger _logger;
    public OrderService(ILogger logger) => _logger = logger;
    public void PlaceOrder() => _logger.Log("Order placed");
}

// ❌ This won't compile — ThirdPartyLogger doesn't implement ILogger
var service = new OrderService(new ThirdPartyLogger());

Adapter Solution

C# — Adapter wraps ThirdPartyLogger to match ILogger
// ✅ ADAPTER — wraps ThirdPartyLogger and translates calls to ILogger
public class ThirdPartyLoggerAdapter : ILogger
{
    private readonly ThirdPartyLogger _adaptee;

    public ThirdPartyLoggerAdapter(ThirdPartyLogger adaptee)
        => _adaptee = adaptee;

    // Translate ILogger.Log → ThirdPartyLogger.WriteInfo
    public void Log(string message)
        => _adaptee.WriteInfo(message);

    // Translate ILogger.LogError → ThirdPartyLogger.WriteError
    public void LogError(string message)
        => _adaptee.WriteError(message);
}

// ✅ Now ThirdPartyLogger works everywhere ILogger is expected
ILogger logger  = new ThirdPartyLoggerAdapter(new ThirdPartyLogger());
var     service = new OrderService(logger);
service.PlaceOrder();
// [INFO] Order placed ✅

// OrderService didn't change. ThirdPartyLogger didn't change.
// Adapter bridges the gap.

Real-World Example — Payment Gateway Adapter

C# — Adapting old PayPal SDK to new IPaymentGateway interface
// Your application's payment interface
public interface IPaymentGateway
{
    Task<bool> ProcessPaymentAsync(decimal amount, string currency);
    Task<bool> RefundAsync(string transactionId, decimal amount);
}

// Old PayPal SDK — different method names and signatures (can't modify)
public class LegacyPayPalSdk
{
    public async Task<string> MakePaymentAsync(double usdAmount)
    {
        await Task.Delay(100);
        return "TXN_12345";  // returns transaction ID
    }

    public async Task<bool> CancelPaymentAsync(string txnId, double amount)
    {
        await Task.Delay(100);
        return true;
    }
}

// Adapter bridges old SDK to new interface
public class PayPalAdapter : IPaymentGateway
{
    private readonly LegacyPayPalSdk _paypal;

    public PayPalAdapter(LegacyPayPalSdk paypal) => _paypal = paypal;

    public async Task<bool> ProcessPaymentAsync(decimal amount, string currency)
    {
        // Translate: convert decimal + currency to double USD
        var usdAmount = (double)ConvertToUsd(amount, currency);
        var txnId     = await _paypal.MakePaymentAsync(usdAmount);
        return !string.IsNullOrEmpty(txnId);
    }

    public async Task<bool> RefundAsync(string transactionId, decimal amount)
        => await _paypal.CancelPaymentAsync(transactionId, (double)amount);

    private decimal ConvertToUsd(decimal amount, string currency)
        => currency == "GBP" ? amount * 1.27m : amount;
}

// CheckoutService only knows IPaymentGateway — works with ANY adapted gateway
public class CheckoutService
{
    private readonly IPaymentGateway _gateway;
    public CheckoutService(IPaymentGateway gateway) => _gateway = gateway;

    public async Task CheckoutAsync(decimal total)
    {
        var success = await _gateway.ProcessPaymentAsync(total, "GBP");
        Console.WriteLine(success ? "Payment successful!" : "Payment failed!");
    }
}

// Wire it up:
var checkout = new CheckoutService(
    new PayPalAdapter(new LegacyPayPalSdk()));
await checkout.CheckoutAsync(99.99m);

When to Use Adapter

  • Integrating a third-party library that has a different interface to what your code expects
  • Using legacy code in a new system without modifying the legacy code
  • When you want to reuse existing classes that are not compatible with the rest of your code
  • Making different implementations of the same concept work through a common interface
SOLID: O — Add new adapters without changing existing code D — Client depends on interface, not legacy class S — Adapter only handles translation, nothing else

🎁 Decorator Pattern

👶 In Plain English

The Decorator pattern lets you add new behaviour to an object by wrapping it in another object that implements the same interface. Each wrapper adds one piece of new behaviour, then delegates to the wrapped object. You can stack multiple decorators like layers, and the original object never changes.

💡 Real World Analogy

Think of coffee at a café. You start with a plain espresso (the base object). Add milk — now it's a latte (Decorator 1). Add caramel — now it's a caramel latte (Decorator 2). Add whipped cream (Decorator 3). Each addition wraps the previous drink, adding flavour and cost. The espresso itself never changed. You can add or remove decorators in any combination.

Decorator vs Inheritance

❌ Inheritance — Explosion of Subclasses
  • LoggingOrderService
  • CachingOrderService
  • LoggingCachingOrderService
  • ValidationOrderService
  • LoggingValidationOrderService
  • LoggingValidationCachingOrderService
  • 2^N subclasses for N features!
✅ Decorator — Mix and Match at Runtime
  • OrderService (base)
  • LoggingDecorator wraps any IOrderService
  • CachingDecorator wraps any IOrderService
  • ValidationDecorator wraps any IOrderService
  • Combine freely: Log(Cache(Validate(base)))
  • N decorators for N features — not 2^N classes

Decorator Implementation

C# — Decorating a data service with logging and caching
// ── Base interface ────────────────────────────────────────────────
public interface IDataService
{
    Task<string> GetDataAsync(string key);
}

// ── Concrete implementation ────────────────────────────────────────
public class DataService : IDataService
{
    public async Task<string> GetDataAsync(string key)
    {
        await Task.Delay(200);  // simulate slow DB call
        return $"Data for {key}";
    }
}

// ── Decorator 1: Logging ───────────────────────────────────────────
public class LoggingDataService : IDataService
{
    private readonly IDataService _inner;   // the wrapped service

    public LoggingDataService(IDataService inner) => _inner = inner;

    public async Task<string> GetDataAsync(string key)
    {
        Console.WriteLine($"[LOG] Calling GetDataAsync({key})");
        var stopwatch = Stopwatch.StartNew();

        var result = await _inner.GetDataAsync(key);   // delegate to wrapped object

        stopwatch.Stop();
        Console.WriteLine($"[LOG] Completed in {stopwatch.ElapsedMilliseconds}ms");
        return result;
    }
}

// ── Decorator 2: Caching ───────────────────────────────────────────
public class CachingDataService : IDataService
{
    private readonly IDataService               _inner;
    private readonly Dictionary<string, string> _cache = new();

    public CachingDataService(IDataService inner) => _inner = inner;

    public async Task<string> GetDataAsync(string key)
    {
        if (_cache.TryGetValue(key, out var cached))
        {
            Console.WriteLine($"[CACHE] Hit for {key}");
            return cached;
        }

        var result  = await _inner.GetDataAsync(key);  // delegate to wrapped
        _cache[key] = result;
        Console.WriteLine($"[CACHE] Stored {key}");
        return result;
    }
}

// ── Stack decorators — order matters! ─────────────────────────────
IDataService service =
    new LoggingDataService(       // outer: logs every call
        new CachingDataService(    // middle: caches results
            new DataService()));    // inner: actually fetches data

await service.GetDataAsync("user:1");
// [LOG] Calling GetDataAsync(user:1)
// [CACHE] Miss — fetching...
// [LOG] Completed in 203ms

await service.GetDataAsync("user:1");
// [LOG] Calling GetDataAsync(user:1)
// [CACHE] Hit for user:1
// [LOG] Completed in 0ms ← cache works!

Real-World Example — HTTP Request Pipeline

C# — Decorator chain for HTTP middleware-style pipeline
public interface IRequestHandler
{
    Task<string> HandleAsync(string request);
}

// Base handler
public class HttpRequestHandler : IRequestHandler
{
    public async Task<string> HandleAsync(string request)
    {
        await Task.Delay(100);
        return $"Response for: {request}";
    }
}

// Auth decorator
public class AuthDecorator : IRequestHandler
{
    private readonly IRequestHandler _inner;
    public AuthDecorator(IRequestHandler inner) => _inner = inner;

    public async Task<string> HandleAsync(string request)
    {
        if (!request.Contains("token"))
            throw new UnauthorizedAccessException("No token provided");
        return await _inner.HandleAsync(request);
    }
}

// Rate limiting decorator
public class RateLimitDecorator : IRequestHandler
{
    private readonly IRequestHandler _inner;
    private int _requestCount;
    public RateLimitDecorator(IRequestHandler inner) => _inner = inner;

    public async Task<string> HandleAsync(string request)
    {
        if (++_requestCount > 100)
            throw new Exception("Rate limit exceeded");
        return await _inner.HandleAsync(request);
    }
}

// Stack: RateLimit → Auth → HttpHandler
IRequestHandler pipeline =
    new RateLimitDecorator(
        new AuthDecorator(
            new HttpRequestHandler()));

var response = await pipeline.HandleAsync("GET /api/users?token=abc");

Decorator in .NET — Real Examples You Use Every Day

🔍 Decorator is Everywhere in .NET

  • StreamReader(new BufferedStream(new FileStream(...))) — streams are Decorators stacked on each other
  • ASP.NET Core Middleware pipeline — each middleware decorates the next

🏠 Facade Pattern

👶 In Plain English

The Facade pattern provides a simple interface to a complex system. Instead of making callers interact with many subsystems, a Facade puts one friendly front-facing class in between. The complexity doesn't go away — it's just hidden behind a cleaner, simpler API.

💡 Real World Analogy

To watch a movie at home you need to: turn on the TV, switch to HDMI, turn on the Blu-ray player, turn on the sound system, set surround sound mode, dim the lights, close the blinds. A smart home controller (Facade) does all of this with one button: "Watch Movie". You don't care about the complexity — you just press one button.

The Problem — Dealing with Subsystem Complexity

PROBLEM C# — Caller must know every subsystem
// ❌ Without Facade — caller must orchestrate all subsystems manually
var inventory  = new InventoryService();
var payment    = new PaymentService();
var shipping   = new ShippingService();
var email      = new EmailService();
var loyalty    = new LoyaltyPointsService();

// Every place that places an order must repeat this orchestration!
if (!inventory.CheckStock(order.Items))
    throw new Exception("Out of stock");

inventory.ReserveItems(order.Items);
var txn = payment.Charge(order.Total, order.PaymentDetails);
var trackingNum = shipping.CreateShipment(order);
email.SendConfirmation(order.CustomerEmail, trackingNum);
loyalty.AwardPoints(order.CustomerId, order.Total);
// ❌ Caller knows too much. Change any subsystem → update every caller.

Facade Solution

C# — OrderFacade hides all subsystem complexity
// The individual subsystems stay unchanged
public class InventoryService
{
    public bool CheckStock(List<OrderItem> items)  => true;
    public void ReserveItems(List<OrderItem> items) => Console.WriteLine("Items reserved");
}
public class PaymentService
{
    public string Charge(decimal amount, string card) { Console.WriteLine($"Charged £{amount}"); return "TXN001"; }
}
public class ShippingService
{
    public string CreateShipment(Order order) { Console.WriteLine("Shipment created"); return "TRACK123"; }
}
public class EmailService
{
    public void SendConfirmation(string email, string tracking) => Console.WriteLine($"Email sent to {email}");
}
public class LoyaltyPointsService
{
    public void AwardPoints(int customerId, decimal total) => Console.WriteLine($"Awarded {total * 10} points");
}

// ✅ FACADE — one simple entry point that orchestrates everything
public class OrderFacade
{
    private readonly InventoryService    _inventory;
    private readonly PaymentService      _payment;
    private readonly ShippingService      _shipping;
    private readonly EmailService         _email;
    private readonly LoyaltyPointsService _loyalty;

    public OrderFacade()
    {
        _inventory = new InventoryService();
        _payment   = new PaymentService();
        _shipping  = new ShippingService();
        _email     = new EmailService();
        _loyalty   = new LoyaltyPointsService();
    }

    // One method replaces all the complexity above
    public void PlaceOrder(Order order)
    {
        if (!_inventory.CheckStock(order.Items))
            throw new InvalidOperationException("Items out of stock");

        _inventory.ReserveItems(order.Items);
        var txnId    = _payment.Charge(order.Total, order.CardDetails);
        var tracking = _shipping.CreateShipment(order);
        _email.SendConfirmation(order.CustomerEmail, tracking);
        _loyalty.AwardPoints(order.CustomerId, order.Total);
    }
}

// ✅ Caller only needs to know one simple method
var facade = new OrderFacade();
facade.PlaceOrder(order);
// That's it! One line. All subsystems handled internally.

Facade vs Adapter — What's the Difference?

🔌 Adapter

  • Makes ONE incompatible interface compatible with another
  • Does NOT simplify — just translates
  • Usually wraps a single class
  • Required when interfaces don't match

🏠 Facade

  • Simplifies access to MULTIPLE classes/subsystems
  • Does NOT change interfaces — just hides complexity
  • Wraps an entire subsystem of classes
  • Optional — subsystems still usable directly if needed
SOLID: S — Facade orchestrates; each subsystem has its own responsibility D — Callers depend on the Facade, not on each subsystem directly

🛡️ Proxy Pattern

👶 In Plain English

A Proxy is a stand-in that controls access to another object. The proxy implements the same interface as the real object so the caller doesn't know or care whether they're talking to the real thing or the proxy. The proxy can do things before/after forwarding to the real object — like caching results, checking permissions, or delaying expensive creation.

💡 Real World Analogy

A credit card is a proxy for your bank account. When you pay at a shop, you hand over your card — the shop doesn't directly access your bank. The card (proxy) mediates access, adds security (PIN), keeps a record (statement), and handles the transaction on your behalf. Your money is in the bank, but the proxy controls access to it.

Three Types of Proxy

💾

Caching Proxy

Cache results from the real service. Return cached data for repeated calls. Avoid hitting the real object unnecessarily.

🔒

Protection Proxy

Check permissions before forwarding. Block unauthorised access to the real object.

😴

Virtual Proxy

Delay creating the expensive real object until it's actually needed (lazy loading).

Caching Proxy Example

C# — Caching Proxy for expensive database calls
public interface IProductService
{
    Task<List<Product>> GetAllProductsAsync();
    Task<Product?>      GetByIdAsync(int id);
}

// Real service — hits the database
public class ProductService : IProductService
{
    public async Task<List<Product>> GetAllProductsAsync()
    {
        await Task.Delay(500);  // slow DB query
        Console.WriteLine("[DB] Fetching all products...");
        return [ new Product(1, "Laptop"), new Product(2, "Mouse") ];
    }

    public async Task<Product?> GetByIdAsync(int id)
    {
        await Task.Delay(100);
        Console.WriteLine($"[DB] Fetching product {id}...");
        return new Product(id, $"Product {id}");
    }
}

// ✅ Caching Proxy — same interface, adds caching transparently
public class CachedProductService : IProductService
{
    private readonly IProductService                   _real;
    private readonly Dictionary<int, Product>          _byIdCache   = new();
    private          List<Product>?                      _allCache;
    private          DateTime                             _cacheExpiry = DateTime.MinValue;

    public CachedProductService(IProductService real) => _real = real;

    public async Task<List<Product>> GetAllProductsAsync()
    {
        if (_allCache != null && DateTime.UtcNow < _cacheExpiry)
        {
            Console.WriteLine("[CACHE] Returning cached products");
            return _allCache;
        }

        _allCache    = await _real.GetAllProductsAsync();
        _cacheExpiry = DateTime.UtcNow.AddMinutes(5);
        return _allCache;
    }

    public async Task<Product?> GetByIdAsync(int id)
    {
        if (_byIdCache.TryGetValue(id, out var cached))
        {
            Console.WriteLine($"[CACHE] Hit for product {id}");
            return cached;
        }

        var product   = await _real.GetByIdAsync(id);
        if (product != null) _byIdCache[id] = product;
        return product;
    }
}

// Registration — swap real service for cached proxy transparently
IProductService service = new CachedProductService(new ProductService());

await service.GetAllProductsAsync();  // [DB] Fetching all products...
await service.GetAllProductsAsync();  // [CACHE] Returning cached products ✅

Protection Proxy Example

C# — Protection Proxy for authorisation
public class SecureAdminService : IAdminService { private readonly IAdminService _real; private readonly ICurrentUser _user; public SecureAdminService(IAdminService real, ICurrentUser user) { _real = real; _user = user; } public void DeleteUser(int userId) { // Check permission BEFORE delegating to real service if (!_user.HasRole("Admin")) throw new UnauthorizedAccessException("Admin role required"); _real.DeleteUser(userId); // delegate to real service } public void UpdateSettings(AppSettings settings) { if (!_user.HasRole("SuperAdmin")) throw new UnauthorizedAccessException("SuperAdmin role required"); _real.UpdateSettings(settings); } }

Proxy vs Decorator — What's the Difference?

🎁 Decorator

  • Adds new behaviour/features to the object
  • The client knows it's using a decorator chain
  • Decorators are often stacked by the client
  • Focus: enhancement

🛡️ Proxy

  • Controls access to the real object
  • Client believes it talks to the real thing
  • Usually manages the real object's lifecycle
  • Focus: access control (security, caching, lazy load)
SOLID: O — Add caching/security via proxy without modifying real class S — Proxy handles access control; real service handles business logic D — Callers depend on interface; proxy or real injected transparently

🌳 Composite Pattern

👶 In Plain English

The Composite pattern lets you treat single objects and groups of objects the same way. It builds tree structures where individual items (leaves) and containers of items (composites) both implement the same interface. This means you can call the same method on a single file or an entire folder — and it works correctly in both cases.

💡 Real World Analogy

A company org chart. An individual employee has a salary. A department has a salary — which is the sum of all its employees' and sub-departments' salaries. You can ask "what is the salary cost?" to a single person OR to the entire company and the question makes sense in both cases. Individual employees and containers of employees are treated identically.

File System Example

C# — File system tree with Composite
// ── Component — common interface for both files and folders ───────
public abstract class FileSystemItem
{
    public string Name { get; }
    protected FileSystemItem(string name) => Name = name;

    public abstract long   GetSize();
    public abstract void   Display(int indent = 0);
}

// ── Leaf — a single file (no children) ────────────────────────────
public class File : FileSystemItem
{
    private readonly long _sizeBytes;

    public File(string name, long sizeBytes) : base(name)
        => _sizeBytes = sizeBytes;

    public override long GetSize() => _sizeBytes;

    public override void Display(int indent = 0)
        => Console.WriteLine($"{new string(' ', indent)}📄 {Name} ({_sizeBytes:N0} bytes)");
}

// ── Composite — a folder that contains other items ─────────────────
public class Folder : FileSystemItem
{
    private readonly List<FileSystemItem> _children = new();

    public Folder(string name) : base(name) { }

    public void Add(FileSystemItem item) => _children.Add(item);
    public void Remove(FileSystemItem item) => _children.Remove(item);

    // Composite: size = sum of all children's sizes (recursively)
    public override long GetSize()
        => _children.Sum(c => c.GetSize());

    public override void Display(int indent = 0)
    {
        Console.WriteLine($"{new string(' ', indent)}📁 {Name}/ ({GetSize():N0} bytes)");
        foreach (var child in _children)
            child.Display(indent + 4);  // recursion — works for any depth
    }
}

// ── Build the tree ────────────────────────────────────────────────
var root = new Folder("Documents");

var projects = new Folder("Projects");
projects.Add(new File("app.cs",   45_000));
projects.Add(new File("tests.cs", 12_000));

var images = new Folder("Images");
images.Add(new File("logo.png",    500_000));
images.Add(new File("banner.jpg",  1_200_000));

root.Add(projects);
root.Add(images);
root.Add(new File("readme.md", 3_000));

// ✅ Same method works on file, folder, or entire tree
root.Display();
// 📁 Documents/ (1,760,000 bytes)
//     📁 Projects/ (57,000 bytes)
//         📄 app.cs (45,000 bytes)
//         📄 tests.cs (12,000 bytes)
//     📁 Images/ (1,700,000 bytes)
//         📄 logo.png (500,000 bytes)
//         📄 banner.jpg (1,200,000 bytes)
//     📄 readme.md (3,000 bytes)

Console.WriteLine($"Total: {root.GetSize():N0} bytes");
// Works the same on root, projects folder, or a single file

Real-World Example — UI Component Tree

C# — UI rendering with Composite
public abstract class UIComponent
{
    public abstract void Render(int depth = 0);
}

public class Button : UIComponent
{
    private readonly string _label;
    public Button(string label) => _label = label;
    public override void Render(int d = 0) => Console.WriteLine($"{new string(' ', d * 2)}[Button: {_label}]");
}

public class TextBox : UIComponent
{
    private readonly string _placeholder;
    public TextBox(string ph) => _placeholder = ph;
    public override void Render(int d = 0) => Console.WriteLine($"{new string(' ', d * 2)}[TextBox: {_placeholder}]");
}

public class Panel : UIComponent
{
    private readonly string             _name;
    private readonly List<UIComponent> _children = new();

    public Panel(string name) => _name = name;
    public void Add(UIComponent c) => _children.Add(c);

    public override void Render(int d = 0)
    {
        Console.WriteLine($"{new string(' ', d * 2)}[Panel: {_name}]");
        _children.ForEach(c => c.Render(d + 1));
    }
}

var form = new Panel("LoginForm");
form.Add(new TextBox("Username"));
form.Add(new TextBox("Password"));

var buttonRow = new Panel("ButtonRow");
buttonRow.Add(new Button("Login"));
buttonRow.Add(new Button("Cancel"));
form.Add(buttonRow);

form.Render();
// [Panel: LoginForm]
//   [TextBox: Username]
//   [TextBox: Password]
//   [Panel: ButtonRow]
//     [Button: Login]
//     [Button: Cancel]
SOLID: O — Add new leaf or composite types without changing existing ones L — Leaf and Composite are interchangeable wherever component is expected

🌉 Bridge Pattern

👶 In Plain English

The Bridge pattern separates an abstraction from its implementation so that both can change independently. Instead of having one giant inheritance hierarchy that mixes two dimensions of variation (e.g., Shape × Colour), you split them into two separate hierarchies connected by a "bridge" (a reference/interface). Change the shape without affecting the colour, or change the rendering without affecting the shape logic.

💡 Real World Analogy

A TV remote control (abstraction) and a TV (implementation) are connected by a standard interface (the IR signal). You can buy a different remote (universal remote) without getting a new TV. You can replace the TV with a newer model without getting a new remote. Both can vary independently because they're connected by a standard bridge, not welded together.

The Problem — Inheritance Explosion

PROBLEM C# — Two dimensions = class explosion
// ❌ Without Bridge: every combination needs its own class
// Shapes: Circle, Square
// Renderers: Vector, Raster
// Result: 2 × 2 = 4 classes — and grows as N × M

public class VectorCircle  { public void Draw() => Console.WriteLine("Vector Circle"); }
public class RasterCircle  { public void Draw() => Console.WriteLine("Raster Circle"); }
public class VectorSquare  { public void Draw() => Console.WriteLine("Vector Square"); }

Bridge Solution — Separate the Two Dimensions

C# — Bridge separates Shape and Renderer
// ── Implementation side: Renderer hierarchy ───────────────────────
public interface IRenderer
{
    void RenderCircle(float radius);
    void RenderSquare(float side);
}

public class VectorRenderer : IRenderer
{
    public void RenderCircle(float r) => Console.WriteLine($"Vector: drawing circle r={r}");
    public void RenderSquare(float s) => Console.WriteLine($"Vector: drawing square s={s}");
}

public class RasterRenderer : IRenderer
{
    public void RenderCircle(float r) => Console.WriteLine($"Raster: pixels for circle r={r}");
    public void RenderSquare(float s) => Console.WriteLine($"Raster: pixels for square s={s}");
}

// ── Abstraction side: Shape hierarchy (holds bridge to renderer) ───
public abstract class Shape
{
    protected readonly IRenderer _renderer;  // THE BRIDGE

    protected Shape(IRenderer renderer) => _renderer = renderer;

    public abstract void Draw();
    public abstract void Resize(float factor);
}

public class Circle : Shape
{
    private float _radius;
    public Circle(IRenderer renderer, float radius)
        : base(renderer) => _radius = radius;

    public override void Draw()                    => _renderer.RenderCircle(_radius);
    public override void Resize(float factor)     => _radius *= factor;
}

public class Square : Shape
{
    private float _side;
    public Square(IRenderer renderer, float side)
        : base(renderer) => _side = side;

    public override void Draw()                    => _renderer.RenderSquare(_side);
    public override void Resize(float factor)     => _side *= factor;
}

// ✅ Any shape with any renderer — N+M classes instead of N×M
IRenderer vector = new VectorRenderer();
IRenderer raster = new RasterRenderer();

var shapes = new Shape[]
{
    new Circle(5f, vector),   new Circle(3f, raster),
    new Square(4f, vector),   new Square(2f, raster),
};

foreach (var shape in shapes) shape.Draw();
// Vector: drawing circle r=5
// Raster: pixels for circle r=3
// Vector: drawing square s=4
// Raster: pixels for square s=2

// ✅ Add Triangle? One new class — works with all existing renderers.
// ✅ Add PDF renderer? One new class — works with all existing shapes.

Real-World Example — Notification Bridge

C# — Notification type × delivery channel bridge
// Implementation: delivery channels
public interface IDeliveryChannel
{
    void Send(string recipient, string subject, string body);
}

public class EmailChannel : IDeliveryChannel
{ public void Send(string r, string s, string b) => Console.WriteLine($"📧 Email → {r}: {s}"); }

public class SmsChannel : IDeliveryChannel
{ public void Send(string r, string s, string b) => Console.WriteLine($"📱 SMS → {r}: {b}"); }

// Abstraction: notification types
public abstract class Notification
{
    protected readonly IDeliveryChannel _channel;
    protected Notification(IDeliveryChannel channel) => _channel = channel;
    public abstract void Notify(string recipient);
}

public class OrderNotification : Notification
{
    public OrderNotification(IDeliveryChannel c) : base(c) {}
    public override void Notify(string r) =>
        _channel.Send(r, "Order Confirmed", "Your order has been placed.");
}

public class ShippingNotification : Notification
{
    public ShippingNotification(IDeliveryChannel c) : base(c) {}
    public override void Notify(string r) =>
        _channel.Send(r, "Your order shipped!", "Tracking: TRACK123");
}

// Mix and match freely:
new OrderNotification(new EmailChannel()).Notify("alice@example.com");
new OrderNotification(new SmsChannel()).Notify("+447700000001");
new ShippingNotification(new EmailChannel()).Notify("alice@example.com");

When to Use Bridge

  • When you see a class explosion from combining two independent hierarchies
  • When abstraction and implementation should vary independently
  • When you want to switch implementations at runtime
  • Device drivers — device type (abstract) × operating system (implementation)
  • Reporting — report type × export format (PDF, Excel, HTML)
SOLID: O — Add new shapes or renderers independently S — Shape handles shape logic; renderer handles rendering D — Shape depends on IRenderer interface, not concrete class
