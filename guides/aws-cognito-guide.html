👤 What is AWS Cognito?

Cognito is AWS's fully managed authentication, authorization, and user management service. Instead of building login systems from scratch, Cognito gives you sign-up, sign-in, MFA, social login, and token management — all managed by AWS.

💡 Real-life analogy

Cognito is like a stadium's entry system. Every attendee (user) enters through one of several gates — email/password gate, Google gate, company SSO gate. At each gate, their identity is verified. They receive a wristband (JWT token) proving who they are and which areas they can access. Security at each internal door (your APIs) just checks the wristband — they never re-verify identity themselves.

🏊 User Pools vs Identity Pools

Cognito has two completely different products that work together. Many developers confuse them. Understanding the difference is critical.

Side-by-Side: What Each Pool Does

👥 User Pool
  • A directory of your users
  • Handles: sign-up, sign-in, MFA, forgot password
  • Stores: email, name, phone, custom attributes
  • Issues: JWT tokens (ID, Access, Refresh)
  • Think: "Who are you?" → verify → give token
  • Use: when you need a user database
🔑 Identity Pool
  • Exchanges tokens for AWS credentials
  • Handles: federated identities from ANY source
  • Grants: temporary IAM credentials (STS)
  • Issues: AWS AccessKey + SecretKey + SessionToken
  • Think: "Here's your token → here's AWS access"
  • Use: when app needs direct AWS service calls

How They Work Together

📱 Mobile / Web App
  • User taps "Sign In" button
↓
👥 User Pool — Step 1: Who are you?
  • User enters email + password
  • Cognito verifies credentials
  • Returns JWT tokens: IdToken, AccessToken, RefreshToken
↓ IdToken passed to Identity Pool
🔑 Identity Pool — Step 2: What can you do?
  • Receives User Pool IdToken
  • Calls AWS STS → AssumeRoleWithWebIdentity
  • Returns temp AWS credentials (AccessKey, SecretKey, Token)
  • Maps to IAM role based on user's groups
↓
✅ App calls AWS services directly
  • Upload to S3 with temp credentials
  • Query DynamoDB directly from mobile app
  • Invoke Lambda function
  • Read from IoT — without going through your backend!

When to use which

ScenarioUser PoolIdentity Pool
Web app login → call your API✅ User Pool onlyNot needed
Mobile app → upload to S3 directly✅ User Pool (auth)✅ Identity Pool (AWS creds)
Guest access to public S3 bucket❌ Not needed✅ Unauthenticated role

✅ Most web/mobile apps only need a User Pool

If your app authenticates users and calls your own API backend, you only need a User Pool. Identity Pool is for apps that need to call AWS services directly from the client — like a mobile app uploading images to S3 without going through your backend server.

🎟️ JWT Tokens — The Wristbands Explained

After a user signs in, Cognito returns three tokens. Understanding what each one is, what's inside them, and how long they last is essential for building secure apps.

Three Tokens — Purpose & Lifetime

ID Token
1 hour
Who the user IS — name, email, groups, custom claims. Send to your API to identify the user.
Access Token
1 hour
What the user CAN DO — scopes and permissions. Used to call Cognito APIs (e.g. change password).
Refresh Token
30 days
Gets new ID + Access tokens without re-login. Store securely — it's the long-lived secret. Revocable.

How it works
Cognito signs header+payload with its RSA private key. Your API fetches Cognito's public key from the JWKS endpoint and verifies the signature. If anyone tampers with the payload — signature check fails → reject!
JWKS URLhttps://cognito-idp.{region}.amazonaws.com/{poolId}/.well-known/jwks.json

⚠️ JWT payload is NOT encrypted — it's just Base64!

Anyone can decode a JWT and read its payload (try jwt.io). Never put sensitive data (passwords, credit cards, full PII) in JWT claims. The signature ensures it can't be tampered with — but the content is readable. Only put what your API needs for authorization decisions.

Cognito supports multiple auth flows for different scenarios. The most common is USER_PASSWORD_AUTH for web/mobile apps, but understanding all flows helps you choose the right one.

🌐 Social Login & Federation

Cognito lets users sign in with their existing Google, Facebook, Apple, or corporate SSO accounts. No separate password to remember. All identity providers funnel into your User Pool.

Google Social Login Flow

📱 User clicks "Sign in with Google"
  • App redirects to Cognito Hosted UI
  • URL: https://auth.myapp.com/oauth2/authorize
  • Parameters: client_id, redirect_uri, scope, response_type=code
↓
🟣 Cognito Hosted UI — shows Google button
  • Cognito shows its pre-built login page
  • User clicks Google → Cognito redirects to Google
  • Cognito acts as OIDC Relying Party to Google
↓
🔵 Google Authentication
  • User signs in with Google account (on Google's page)
  • User consents to share: email, profile, openid
  • Google issues authorization code → back to Cognito
↓
🟣 Cognito exchanges code + creates/links user
  • Exchanges Google auth code for Google tokens
  • Extracts: email, name, Google user ID
  • Creates User Pool user if first time (email as username)
  • Links to existing user if email matches
  • Issues Cognito JWT tokens (not Google tokens!)
↓
✅ App receives Cognito tokens via redirect
  • Redirect to: https://myapp.com/callback?code=xyz
  • Exchange code for Cognito JWT tokens
  • App works with Cognito tokens — not Google tokens
  • JWT has: email, sub (Cognito ID), identities array

SAML / Enterprise SSO Flow

🏢 Employee clicks "Sign in with Company SSO"
  • App redirects to Cognito Hosted UI
  • Cognito detects SAML IdP → redirects to corporate IdP
↓
🏢 Corporate Identity Provider (Okta / AD FS)
  • Employee enters corporate username + password
  • IdP validates against Active Directory
  • IdP generates SAML assertion (XML) with user attributes
  • Assertion contains: email, department, groups, employee ID
↓
🟣 Cognito receives SAML assertion
  • Validates SAML signature with IdP's certificate
  • Maps SAML attributes to Cognito user attributes
  • Creates/updates User Pool user
  • Maps SAML groups to Cognito groups
  • Issues Cognito JWT tokens
↓
✅ App receives standard Cognito JWT
  • Your API validates the same JWT — no SAML code needed
  • User's corporate groups reflected in JWT claims
  • Single sign-on: user stays logged in across apps

💡 Supported identity providers

Social (OIDC): Google, Facebook, Apple, Amazon, any OIDC provider
Enterprise (SAML 2.0): Okta, Azure AD, Ping Identity, OneLogin, AD FS, any SAML 2.0 IdP
Developer authenticated: Your own backend validates identity and tells Cognito to issue tokens

🛡️ MFA & Advanced Security

Cognito supports multiple MFA methods and an Advanced Security feature that detects compromised credentials and unusual sign-in patterns automatically.

MFA Challenge Flow

👤 User enters email + password ✓
  • First factor validated — credentials correct
  • MFA required → challenge returned
↓
📱 TOTP (Authenticator App)
  • ChallengeName: SOFTWARE_TOKEN_MFA
  • User opens Google Auth / Authy
  • Enters 6-digit rotating code
  • Valid for 30 seconds
  • Most secure option
📲 SMS OTP
  • ChallengeName: SMS_MFA
  • Cognito sends 6-digit code via SNS
  • User enters code
  • Valid for 3 minutes
  • Requires SNS + phone number
↓
✅ MFA passed → tokens issued
  • Both factors verified → full JWT tokens returned
  • amr claim in JWT: ["mfa"] — your API can enforce MFA

MFA Options Comparison

📱 TOTP (Time-based OTP) — Recommended

  • Apps: Google Authenticator, Authy, Microsoft Auth
  • Works offline — no SMS needed
  • Phishing-resistant (code only valid 30s)
  • No cost — no SNS charges
  • Best security option
  • User sets up once by scanning QR code

📲 SMS MFA

  • Code sent to verified phone number
  • Requires AWS SNS — costs per SMS
  • Vulnerable to SIM swapping
  • Easiest for users — no app needed
  • Acceptable for most business apps
  • Avoid for high-security scenarios

Advanced Security Features

🛡️ Cognito Advanced Security (extra cost)
  • Compromised credential check: compares password against HaveIBeenPwned database — auto-block or force reset
  • Adaptive auth: risk scores each login. New device + new location + new IP = high risk → require MFA even if not normally required
  • Risk signals: IP reputation, device fingerprint, velocity checks, impossible travel detection
  • Actions on risk: Allow, Optional MFA, Require MFA, Block

Lambda Triggers — Customize Any Auth Step

⚡ Pre-Authentication Trigger

Runs before Cognito validates credentials. Use to: block banned users, check subscription status, log login attempts, enforce IP allowlists.

⚡ Post-Authentication Trigger

Runs after successful auth. Use to: log successful logins, sync user data to your DB, update last-seen timestamp, send login notifications.

⚡ Pre-Token Generation Trigger

Modify JWT claims before issuance. Use to: add custom claims from your DB (subscription tier, tenant ID), suppress unwanted claims, add dynamic roles.

⚡ Custom Message Trigger

Customize verification and reset emails. Use to: branded email templates, custom OTP messages, localized content based on user's language preference.

Cognito Auth Service — Sign Up, Sign In, Refresh

CognitoAuthService.cs 
using Amazon.CognitoIdentityProvider;
using Amazon.CognitoIdentityProvider.Model;

public class CognitoAuthService
{
    private readonly IAmazonCognitoIdentityProvider _cognito;
    private const string ClientId  = "your-app-client-id";
    private const string UserPoolId = "ap-south-1_XXXXXXXX";

    public CognitoAuthService(IAmazonCognitoIdentityProvider c)
        => _cognito = c;

    // ── Sign Up ────────────────────────────────────
    public async Task<string> SignUpAsync(
        string email, string password, string name)
    {
        var response = await _cognito.SignUpAsync(
            new SignUpRequest
        {
            ClientId = ClientId,
            Username = email,
            Password = password,
            UserAttributes = [
                new AttributeType { Name="email", Value=email },
                new AttributeType { Name="name",  Value=name  }
            ]
        });
        return response.UserSub; // stable user ID
    }

    // ── Confirm Email (code from verification email) ─
    public async Task ConfirmSignUpAsync(
        string email, string confirmationCode)
    {
        await _cognito.ConfirmSignUpAsync(
            new ConfirmSignUpRequest
        {
            ClientId         = ClientId,
            Username         = email,
            ConfirmationCode = confirmationCode
        });
    }

    // ── Sign In → returns JWT tokens ───────────────
    public async Task<AuthTokens> SignInAsync(
        string email, string password)
    {
        var response = await _cognito.InitiateAuthAsync(
            new InitiateAuthRequest
        {
            AuthFlow = AuthFlowType.USER_PASSWORD_AUTH,
            ClientId = ClientId,
            AuthParameters = new() {
                ["USERNAME"] = email,
                ["PASSWORD"] = password
            }
        });

        if (response.ChallengeName != null)
            throw new MfaRequiredException(
                response.ChallengeName,
                response.Session);

        var result = response.AuthenticationResult;
        return new AuthTokens(
            result.IdToken,
            result.AccessToken,
            result.RefreshToken,
            result.ExpiresIn);
    }

    // ── MFA Response ───────────────────────────────
    public async Task<AuthTokens> RespondToMfaChallengeAsync(
        string session, string username, string code)
    {
        var response = await _cognito.RespondToAuthChallengeAsync(
            new RespondToAuthChallengeRequest
        {
            ChallengeName = ChallengeNameType.SOFTWARE_TOKEN_MFA,
            ClientId      = ClientId,
            Session       = session,
            ChallengeResponses = new() {
                ["USERNAME"]                = username,
                ["SOFTWARE_TOKEN_MFA_CODE"] = code
            }
        });
        var r = response.AuthenticationResult;
        return new AuthTokens(r.IdToken, r.AccessToken, r.RefreshToken, r.ExpiresIn);
    }

    // ── Refresh Tokens ─────────────────────────────
    public async Task<AuthTokens> RefreshTokensAsync(
        string refreshToken)
    {
        var response = await _cognito.InitiateAuthAsync(
            new InitiateAuthRequest
        {
            AuthFlow = AuthFlowType.REFRESH_TOKEN_AUTH,
            ClientId = ClientId,
            AuthParameters = new() {
                ["REFRESH_TOKEN"] = refreshToken
            }
        });
        var r = response.AuthenticationResult;
        return new AuthTokens(r.IdToken, r.AccessToken, refreshToken, r.ExpiresIn);
    }

    // ── Sign Out (revoke refresh token) ────────────
    public async Task SignOutAsync(string accessToken)
    {
        await _cognito.GlobalSignOutAsync(
            new GlobalSignOutRequest
            { AccessToken = accessToken });
    }

    // ── Admin: get user info ───────────────────────
    public async Task<AdminGetUserResponse> GetUserAsync(
        string email)
    {
        return await _cognito.AdminGetUserAsync(
            new AdminGetUserRequest
        {
            UserPoolId = UserPoolId,
            Username   = email
        });
    }
}

public record AuthTokens(
    string IdToken,
    string AccessToken,
    string RefreshToken,
    int    ExpiresIn);

JWT Validation in ASP.NET Core

Program.cs — validate Cognito JWT in your API 
var region  = "ap-south-1";
var poolId  = "ap-south-1_XXXXXXXX";
var clientId = "your-app-client-id";

builder.Services.AddAuthentication(JwtBearerDefaults.AuthenticationScheme)
    .AddJwtBearer(options =>
    {
        // Cognito's public key endpoint
        options.Authority = $"https://cognito-idp.{region}.amazonaws.com/{poolId}";

        options.TokenValidationParameters = new TokenValidationParameters
        {
            ValidateIssuer           = true,
            ValidIssuer              = $"https://cognito-idp.{region}.amazonaws.com/{poolId}",
            ValidateAudience         = true,
            ValidAudience            = clientId,
            ValidateLifetime         = true, // checks exp
            ValidateIssuerSigningKey = true  // verifies signature
        };
    });

builder.Services.AddAuthorization(options =>
{
    // Require "admin" Cognito group
    options.AddPolicy("AdminOnly", policy =>
        policy.RequireClaim("cognito:groups", "admin"));

    // Require MFA was used
    options.AddPolicy("MFARequired", policy =>
        policy.RequireClaim("amr", "mfa"));
});

Extract Claims in Controller

[ApiController]
[Authorize]
public class OrdersController : ControllerBase
{
    [HttpGet]
    public IActionResult GetOrders()
    {
        // Extract Cognito claims
        var userId = User.FindFirst("sub")?.Value;
        var email  = User.FindFirst("email")?.Value;
        var groups = User.Claims
            .Where(c => c.Type == "cognito:groups")
            .Select(c => c.Value)
            .ToList();
        var plan = User.FindFirst("custom:plan")?.Value;

        return Ok(new { userId, email, groups, plan });
    }

    [HttpDelete("{id}")]
    [Authorize(Policy = "AdminOnly")] // requires admin group
    public IActionResult DeleteOrder(int id) => Ok();
}

✅ Best Practices & Common Mistakes

User Pool Configuration

  • Choose sign-in attributes carefully — they cannot be changed after pool creation
  • Use SES for email delivery in production — Cognito's built-in limit is 50/day
  • Enable TOTP MFA for admin accounts at minimum — SMS MFA is vulnerable to SIM swap
  • Set token expiry based on security needs: access token 1hr, refresh token 7–30 days
  • Create separate app clients per application type (web, mobile, backend server)
  • Never generate client secret for browser or mobile apps — it can't be kept secret
  • Use custom attributes (custom:tenantId, custom:plan) to store app-specific user data

Token Handling

  • Store tokens in memory (not localStorage) — immune to XSS attacks
  • For mobile apps: use Secure Enclave / Keychain for token storage
  • Never put sensitive data in JWT claims — payload is Base64, not encrypted
  • Always validate JWT signature server-side — never trust client-provided user data
  • Check exp, iss, and aud claims in every token validation
  • Use sub (not email) as your user primary key — email can change, sub never does
  • Implement silent refresh: detect tokens expiring → auto-call refresh before expiry

Security

  • Enable Advanced Security Mode for protection against compromised credentials
  • Use Pre-Authentication Lambda to block banned or flagged users
  • Implement account lockout with Lambda triggers for too many failed attempts
  • Enable email verification on sign-up — prevents fake accounts
  • Use Cognito Groups for role-based access control — check cognito:groups claim in API
  • Rotate app client secrets periodically if used (server-side clients only)

Common Mistakes to Avoid

⚠️ Top Cognito Mistakes

❌ Using email as user primary key in your DB — use sub (stable UUID), email can change
❌ Storing tokens in localStorage — XSS vulnerable, use memory or HttpOnly cookies
❌ Trusting JWT without signature verification — anyone can forge an unsigned JWT
❌ Not checking exp claim — expired token = still trusted without the check
❌ Generating client secret for browser app — secrets in JS bundles are public
❌ Using Cognito built-in email in production — 50/day limit breaks sign-ups at scale
❌ Confusing User Pool token and Identity Pool credentials — they're for different things
❌ Not revoking refresh tokens on sign-out — GlobalSignOut revokes all sessions
