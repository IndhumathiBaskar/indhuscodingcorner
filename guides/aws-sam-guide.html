🚀 What is AWS SAM?

AWS SAM (Serverless Application Model) is an open-source framework built on top of CloudFormation that makes it dramatically faster to build, test, and deploy serverless applications. It provides shorthand syntax for Lambda, API Gateway, DynamoDB, SQS, SNS and more — what would be 50 lines of raw CloudFormation becomes 5 lines of SAM.

💡 SAM = CloudFormation + Serverless Superpowers

SAM is not a separate service — it's a transform that runs before CloudFormation. You write concise SAM YAML; AWS expands it into full CloudFormation and deploys it. The Transform: AWS::Serverless-2016-10-31 line at the top of your template is what activates the SAM macro.

🧩 SAM Resource Types — Serverless Shorthand

SAM introduces 8 new resource types under the AWS::Serverless:: namespace. Each one expands into multiple CloudFormation resources automatically.

AWS::Serverless::Function
Lambda function + IAM role + permissions
→ Lambda::Function + IAM::Role + Lambda::Permission
AWS::Serverless::Api
REST API Gateway (v1) with stages, auth, CORS
→ ApiGateway::RestApi + Deployment + Stage
AWS::Serverless::HttpApi
HTTP API Gateway (v2) — lower latency, cheaper
→ ApiGatewayV2::Api + Integration + Stage
AWS::Serverless::SimpleTable
DynamoDB table with simple key schema
→ DynamoDB::Table (billing: PAY_PER_REQUEST)
AWS::Serverless::Application
Embed a nested SAM app or SAR application
→ CloudFormation::Stack (nested)
AWS::Serverless::LayerVersion
Lambda Layer — shared code/deps across functions
→ Lambda::LayerVersion
AWS::Serverless::StateMachine
Step Functions state machine from ASL file
→ StepFunctions::StateMachine + IAM::Role
AWS::Serverless::Connector
Auto-generate IAM permissions between resources
→ IAM::Policy attached to source resource role

Canary Deployment — Zero-Downtime Traffic Shifting

With AutoPublishAlias and DeploymentPreference, SAM orchestrates Lambda traffic shifting with automatic rollback on CloudWatch alarms.

🐦 Canary10Percent5Minutes — Traffic Shift Timeline
t=0 (deploy starts)
v1 (90%)
v2
t+0 min
Canary period
v1 monitoring…
v2
t+1–5 min
Alarm OK → shift all
v2 (100%)
t+5 min
YAML · DeploymentPreference Options 
DeploymentPreference:
  Type:    Canary10Percent5Minutes   # 10% → wait 5min → 100%
  # Other options:
  # Linear10PercentEvery1Minute  — +10% every minute
  # Linear10PercentEvery10Minutes
  # Canary10Percent10Minutes
  # Canary10Percent30Minutes
  # AllAtOnce                     — instant (no canary)
  Alarms:
    - !Ref GetOrdersErrorAlarm     # Auto-rollback if this fires
    - !Ref GetOrdersLatencyAlarm
  Hooks:
    PreTraffic:  !Ref PreTrafficHookFunction   # Run before shifting traffic
    PostTraffic: !Ref PostTrafficHookFunction  # Run after full shift
  TriggerConfigurations:
    - TriggerEvents: [DeploymentRollback]
        TriggerTargetArn: !Ref AlertSNSTopic

⚠️ Lambda in VPC — Cold Start Impact

Adding VPC config to Lambda significantly increases cold start time (500ms–2s extra) because AWS must provision an Elastic Network Interface. Only use VPC when your function must reach private resources (RDS, ElastiCache). For DynamoDB and S3, access via public endpoints — no VPC needed.

HTTP API Handler (.NET 8 Native AOT)

C# · Function.cs — API Gateway HTTP API Handler 
using Amazon.Lambda.APIGatewayEvents;
using Amazon.Lambda.Core;
using Amazon.Lambda.RuntimeSupport;
using Amazon.Lambda.Serialization.SystemTextJson;
using AWS.Lambda.Powertools.Logging;
using AWS.Lambda.Powertools.Tracing;

// Native AOT entry point
var handler = FunctionHandler.Handle;
await LambdaBootstrapBuilder
    .Create(handler, new DefaultLambdaJsonSerializer())
    .Build()
    .RunAsync();

public static class FunctionHandler
{
    private static readonly IOrderService _orders =
        new DynamoOrderService(Environment.GetEnvironmentVariable("ORDERS_TABLE")!);

    [Logging(LogEvent = true)]
    [Tracing]
    public static async Task<APIGatewayHttpApiV2ProxyResponse> Handle(
        APIGatewayHttpApiV2ProxyRequest request,
        ILambdaContext context)
    {
        var status  = request.QueryStringParameters?.GetValueOrDefault("status");
        var limit   = int.TryParse(
            request.QueryStringParameters?.GetValueOrDefault("limit"), out var l) ? l : 20;

        var userId  = request.RequestContext.Authorizer?.Jwt?.Claims?["sub"]
                      ?? throw new UnauthorizedException("Missing user claim");

        Logger.LogInformation("Fetching orders",
            new { UserId = userId, Status = status, Limit = limit });

        var orders = await _orders.GetOrdersAsync(userId, status, limit);

        return new APIGatewayHttpApiV2ProxyResponse
        {
            StatusCode = 200,
            Body       = JsonSerializer.Serialize(orders),
            Headers    = new Dictionary<string, string>
            {
                ["Content-Type"] = "application/json",
                ["X-Request-Id"] = context.AwsRequestId
            }
        };
    }
}

SQS Handler — Batch Processing with Partial Failures

C# · SQS Lambda Handler (.NET) 
using Amazon.Lambda.SQSEvents;

public static class SqsHandler
{
    [Tracing]
    public static async Task<SQSBatchResponse> Handle(
        SQSEvent sqsEvent, ILambdaContext context)
    {
        var failures = new List<SQSBatchResponse.BatchItemFailure>();

        foreach (var record in sqsEvent.Records)
        {
            try
            {
                var order = JsonSerializer.Deserialize<OrderMessage>(record.Body)!;
                await ProcessOrderAsync(order);
                Logger.LogInformation($"Processed order {order.OrderId}");
            }
            catch (Exception ex)
            {
                Logger.LogError(ex, $"Failed: {record.MessageId}");

                // Only this message re-queued, not the whole batch
                failures.Add(new SQSBatchResponse.BatchItemFailure
                    { ItemIdentifier = record.MessageId });
            }
        }

        return new SQSBatchResponse { BatchItemFailures = failures };
    }

    private static async Task ProcessOrderAsync(OrderMessage order)
    {
        // Business logic here — save to DynamoDB, call payment API, etc.
        await Task.Delay(50); // Simulate async work
    }
}
// In SAM: FunctionResponseTypes: [ReportBatchItemFailures]
// Enables partial batch failure — other messages still succeed

Best Practices — .NET Lambda Performance

  • Use Native AOT (provided.al2023 + arm64) for the fastest cold starts — typically under 100ms vs 800ms+ for managed runtimes
  • Mark Lambda handler classes and types with [JsonSerializable] for Native AOT — the compiler needs all types to be known at build time
  • Initialise SDK clients (DynamoDB, S3) outside the handler method — they are reused across invocations in the same container
  • Use AWS Lambda Powertools for .NET — structured logging, X-Ray tracing, and custom metrics with no boilerplate
  • Set MemorySize based on profiling — Lambda allocates CPU proportional to memory; 512MB+ often completes faster and costs the same
  • Use ReportBatchItemFailures on SQS event sources — partial failure support prevents the entire batch from re-queuing on one bad message
