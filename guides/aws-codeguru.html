🏗️ What is AWS CodeGuru?

A suite of ML-powered developer tools that automatically review code and profile running applications.

One-liner CodeGuru = an automated senior engineer reviewing every pull request for bugs, security issues, and bad practices — plus a profiler watching your live app to find the exact lines of code burning CPU and money.

CodeGuru in Your SDLC

🔁 Pre-Merge (Reviewer)

Runs automatically when a pull request is opened/updated. Posts inline comments directly on the PR — just like a human reviewer. Catches issues before they're merged.

🏃 Post-Deploy (Profiler)

Lightweight agent runs alongside your application in production. Continuously samples stack traces with minimal overhead (<1% CPU). Surfaces cost/performance issues that only appear under real traffic.

💡 Why ML-based over static linters? Traditional linters (ESLint, SonarQube) catch syntax-level and style issues using fixed rules. CodeGuru is trained on patterns from millions of real Amazon code reviews — it catches semantic issues like resource leaks, race conditions, and inefficient AWS SDK usage that rule-based linters typically miss.

Two Review Types

🔄 Pull Request Reviews

Triggered automatically on every new PR and subsequent commit. Inline comments appear directly on the diff in GitHub/CodeCommit/Bitbucket. Fast feedback loop, incremental.

📦 Full Repository Analysis

One-time deep scan of an entire repository/branch. Good for onboarding an existing codebase, periodic audits, or before a major refactor. Triggered manually via console/API.

What Reviewer Detects

🐛 Code Quality

Resource leaks (unclosed streams/connections), null pointer risks, inefficient loops, duplicate code blocks, unused variables.

🔀 Concurrency

Race conditions, deadlock risks, improper synchronization, thread-safety violations in shared state access.

☁️ AWS Best Practices

Inefficient AWS SDK usage (e.g. creating a new client per request instead of reusing), missing pagination, hardcoded regions/credentials.

Example Finding

Java CodeGuru Reviewer flags a resource leak 
// ❌ CodeGuru finding: "Resource leak detected"
public String readFile(String path) throws IOException {
    FileInputStream stream = new FileInputStream(path);
    return new String(stream.readAllBytes());
    // stream is never closed — leak on every call
}

// ✅ CodeGuru suggested fix — try-with-resources
public String readFile(String path) throws IOException {
    try (FileInputStream stream = new FileInputStream(path)) {
        return new String(stream.readAllBytes());
    } // stream automatically closed, even on exception
}
💡 Finding Recommendations Every finding includes: a description of the issue, why it matters, a link to AWS/industry documentation, and (for many findings) a suggested code fix you can apply directly from the PR comment.

⚡ CodeGuru Profiler

Continuous, low-overhead production profiling — find exactly which code is costing you CPU and money.

🔥 Flame Graphs

Visual representation of where CPU time is spent. Wider bars = more time. Click to drill into nested function calls. Compare graphs across two time periods to spot regressions.

💰 Cost Visualisation

Profiler estimates the dollar cost of specific code paths based on compute time consumed. Helps prioritise optimisation work by actual cost impact, not just CPU %.

🐢 Anomaly Detection

Automatically detects when a function's CPU/latency profile deviates significantly from its historical baseline — flags potential performance regressions after a deploy.

🪶 Low Overhead

Sampling-based profiling agent — typically <1% CPU overhead. Safe to run continuously in production, unlike traditional profilers that require dedicated profiling sessions.

🛡️ Security Detector

Security-focused findings within CodeGuru Reviewer — mapped to industry standards.

Standards Coverage Security Detector findings map to OWASP Top 10, AWS security best practices, CWE (Common Weakness Enumeration), and PCI-DSS-relevant patterns where applicable.

Example Security Findings

Hardcoded AWS Credentials Critical
Detects AWS access keys, secret keys, or other credentials hardcoded in source code. Recommends using IAM roles, Secrets Manager, or environment variables instead.
SQL Injection Risk Critical
Flags string concatenation used to build SQL queries from user input. Recommends parameterized queries / prepared statements.
Weak Cryptography High
Detects use of deprecated/weak algorithms (MD5, SHA-1 for security purposes, DES). Recommends modern alternatives (SHA-256+, AES-256).
Insecure Deserialization High
Flags deserializing untrusted data without validation — a common vector for remote code execution attacks.
Missing Input Validation Medium
Identifies user-facing endpoints that process input without validation, potentially allowing injection or unexpected behaviour.

Example: SQL Injection Detection

Java Security Detector flags string-concatenated SQL 
// ❌ CodeGuru finding: "SQL injection risk"
String query = "SELECT * FROM users WHERE username = '" + userInput + "'";
Statement stmt = connection.createStatement();
ResultSet rs = stmt.executeQuery(query);

// ✅ Suggested fix — parameterized query
String query = "SELECT * FROM users WHERE username = ?";
PreparedStatement stmt = connection.prepareStatement(query);
stmt.setString(1, userInput);
ResultSet rs = stmt.executeQuery();

🔧 CI/CD Integration

Wire CodeGuru into your pull request workflow — GitHub Actions, CodeCommit, or Bitbucket.

  • 1
    Associate RepositoryIn the CodeGuru console, connect your CodeCommit/GitHub/Bitbucket repo. Grants CodeGuru read access via OAuth or IAM role.
  • 2
    Configure TriggerChoose "Pull Request" trigger (recommended) for inline PR comments, and/or "Full Repository Scan" for periodic deep analysis.
  • 3
    Open a Pull RequestCodeGuru automatically analyses the diff. Findings appear as inline review comments within minutes.
  • 4
    Address or Dismiss FindingsApply suggested fixes directly, or dismiss with a reason (false positive, won't fix, etc.) — improves future relevance.
  • 5
    Track MetricsView aggregated metrics: findings by severity, code quality trend over time, recommendation acceptance rate.

GitHub Actions Workflow

YAML .github/workflows/codeguru-review.yml 
name: CodeGuru Reviewer
on: [pull_request]

jobs:
  codeguru-review:
    runs-on: ubuntu-latest
    permissions:
      id-token: write
      contents: read
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0

      - name: Configure AWS Credentials
        uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: arn:aws:iam::123456789012:role/CodeGuruReviewRole
          aws-region: us-east-1

      - name: AWS CodeGuru Reviewer
        uses: aws-actions/codeguru-reviewer@v1.1
        with:
          s3_bucket: codeguru-reviewer-artifacts-myorg
          build_path: ./target/classes  # compiled output for Java

      - name: Upload Results
        uses: github/codeql-action/upload-sarif@v3
        with:
          sarif_file: codeguru-results.sarif.json

🧩 C# / .NET — Workarounds & Alternatives

CodeGuru natively supports Java and Python only. Here's how .NET teams get equivalent value.

⚠️ No Native .NET Support As of this writing, neither CodeGuru Reviewer nor Profiler officially support C#/.NET. If your stack is .NET (like Indhu's), use the alternatives below for equivalent code review and profiling capability.
🔍 Code Review Alternatives

Amazon Q Developer — AWS's newer AI coding assistant DOES support C#/.NET for code review and security scanning. Best AWS-native alternative.

SonarQube / SonarCloud — strong C# rule sets, security hotspots, code smells.

GitHub Copilot / CodeQL — AI review + semantic security analysis for C#.

⚡ Profiling Alternatives

AWS X-Ray .NET SDK — distributed tracing, not full CPU profiling, but identifies slow service calls.

Application Insights (Azure) — if hybrid cloud, strong .NET profiling.

dotnet-trace / PerfView — Microsoft's native .NET profiling tools, can export flame graphs.

JetBrains dotTrace — commercial, excellent .NET flame graphs.

C# AWS X-Ray instrumentation for .NET (closest AWS-native equivalent) 
using Amazon.XRay.Recorder.Core;
using Amazon.XRay.Recorder.Handlers.AwsSdk;

// Register X-Ray tracing for all AWS SDK calls
AWSSDKHandler.RegisterXRayForAllServices();

public async Task ProcessOrderAsync(Order order)
{
    await AWSXRayRecorder.Instance.TraceMethodAsync("ProcessOrder", async () =>
    {
        // Annotate with custom metadata — visible in X-Ray console
        AWSXRayRecorder.Instance.AddAnnotation("orderId", order.Id);

        await ValidateOrderAsync(order);     // sub-segment auto-traced
        await SaveToDynamoDbAsync(order);    // AWS SDK call auto-traced
        await PublishEventAsync(order);      // AWS SDK call auto-traced
    });
}

📊 Finding Types & Severity

Understanding how CodeGuru categorises and prioritises what it finds.

Finding Categories

🐛 AWS Best Practices

S3 client recreation per request, missing retry/backoff logic, synchronous calls in async contexts, missing pagination on list operations.

🔀 Concurrency

Unsynchronized access to shared mutable state, potential deadlocks, improper use of concurrent collections, thread pool exhaustion risks.

🛡️ Security

Injection vulnerabilities, hardcoded secrets, weak cryptographic algorithms, insecure deserialization, missing authentication checks.

📦 Resource Leaks

Unclosed file handles, database connections, HTTP clients, streams — anything implementing Closeable/AutoCloseable not properly released.

⚡ Code Inefficiency

Inefficient string concatenation in loops, redundant computation, suboptimal data structure choices, unnecessary object creation.

📐 Code Maintainability

Duplicate code blocks, overly complex methods (high cyclomatic complexity), inconsistent naming, missing documentation on public APIs.

💡 Suppression & Feedback Loop Mark findings as "Useful" or "Not Useful" directly in the PR comment or console. CodeGuru uses this feedback to improve future recommendations for your specific codebase patterns over time.

💡 Best Practices

Get the most value from CodeGuru across your development workflow.

✅ Enable on Every Repository

Associate every Java/Python repository with CodeGuru Reviewer — there's no extra cost barrier to entry for catching bugs early. The earlier issues are caught, the cheaper they are to fix.

✅ Treat Critical/High as Merge Blockers

Configure your branch protection rules to require addressing Critical and High severity findings before merge. Medium/Low can be tracked as follow-up tech debt.

✅ Run Full Repository Scan on Onboarding

When adopting CodeGuru on an existing codebase, run a full repository analysis first to get a baseline. Don't try to fix everything at once — triage by severity and tackle incrementally.

✅ Profile in Production, Not Just Staging

Real cost/performance hotspots often only appear under genuine production traffic patterns. Run the Profiler agent continuously in production with its near-zero overhead.

✅ Compare Flame Graphs Before/After Deploy

Use Profiler's comparison feature to diff CPU profiles between two time windows — instantly spot performance regressions introduced by a recent deployment.

✅ Give Feedback on Findings

Mark findings as useful/not useful consistently. This trains CodeGuru's relevance for your team's specific codebase and reduces noise over time.

✅ For .NET Teams — Build the Equivalent Stack

Combine Amazon Q Developer (review) + AWS X-Ray (.NET tracing) + dotnet-trace (profiling) to approximate CodeGuru's value until/if native .NET support arrives.

✅ Don't Ignore Security Findings

Security Detector findings (hardcoded credentials, injection risks) should always be treated as blocking — these are the exact patterns that lead to real breaches.
