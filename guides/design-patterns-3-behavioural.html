💬 What Are Behavioural Patterns?

👶 What Do Behavioural Patterns Do?

Behavioural patterns deal with how objects talk to each other and how responsibilities are distributed. They define clear communication between objects, so your code is loosely coupled and each class only knows what it needs to know. They answer the question: "Who does what, and how do they tell each other about it?"

The 8 Behavioural Patterns

🔀
Strategy
Define a family of algorithms, encapsulate each one, and make them interchangeable at runtime.
👁️
Observer
When one object changes, automatically notify all objects that depend on it.
📦
Command
Encapsulate a request as an object — queue it, undo it, log it.
📋
Template Method
Define the skeleton of an algorithm, letting subclasses override specific steps.
🔁
Iterator
Traverse a collection without exposing its internal structure.
🔗
Chain of Responsibility
Pass a request along a chain of handlers until one handles it.
📡
Mediator
Objects communicate through a central mediator instead of directly with each other.
🚦
State
Allow an object to change its behaviour when its internal state changes.

🔀 Strategy Pattern

👶 In Plain English

Strategy lets you swap algorithms at runtime without changing the code that uses them. You define a family of algorithms (strategies), put each in its own class, and pass the one you need to the object that uses it. The object doesn't care which algorithm it uses — it just calls the interface.

💡 Real World Analogy

You need to travel from London to Paris. You have strategies: fly (fast, expensive), take Eurostar (medium, comfortable), drive (slow, flexible). You pick a strategy based on your situation. The destination doesn't change — only the how. Tomorrow you might pick a different strategy. Your travel planner just says "execute the travel strategy" without caring which one it is.

The Problem — If/Else Algorithm Switching

PROBLEM C# — Hardcoded algorithm selection
// ❌ Adding a new sort algorithm means editing this class
public class DataProcessor
{
    public void Sort(List<int> data, string algorithm)
    {
        if      (algorithm == "bubble") BubbleSort(data);
        else if (algorithm == "quick")  QuickSort(data);
        else if (algorithm == "merge")  MergeSort(data);
        // ❌ New algorithm = edit this method + risk breaking others
    }
    private void BubbleSort(List<int> d) { /* ... */ }
    private void QuickSort(List<int> d)  { /* ... */ }
    private void MergeSort(List<int> d)  { /* ... */ }
}
C# — Strategy pattern: each algorithm in its own class
// ── Strategy interface ────────────────────────────────────────────
public interface ISortStrategy
{
    void Sort(List<int> data);
}

// ── Concrete strategies ───────────────────────────────────────────
public class BubbleSortStrategy : ISortStrategy
{
    public void Sort(List<int> data)
    {
        Console.WriteLine("Bubble sorting...");
        // bubble sort implementation
        for (int i = 0; i < data.Count - 1; i++)
            for (int j = 0; j < data.Count - i - 1; j++)
                if (data[j] > data[j + 1]) (data[j], data[j + 1]) = (data[j + 1], data[j]);
    }
}

public class LinqSortStrategy : ISortStrategy
{
    public void Sort(List<int> data)
    {
        Console.WriteLine("LINQ sorting...");
        var sorted = data.OrderBy(x => x).ToList();
        data.Clear(); data.AddRange(sorted);
    }
}

// ── Context — uses whichever strategy is injected ─────────────────
public class DataProcessor
{
    private ISortStrategy _strategy;

    public DataProcessor(ISortStrategy strategy) => _strategy = strategy;

    // ✅ Change strategy at runtime
    public void SetStrategy(ISortStrategy strategy) => _strategy = strategy;

    public void ProcessData(List<int> data)
    {
        _strategy.Sort(data);   // delegates — doesn't care which algorithm
        Console.WriteLine($"Result: {string.Join(", ", data)}");
    }
}

// Usage — swap strategies freely
var data      = new List<int> { 5, 2, 8, 1, 9, 3 };
var processor = new DataProcessor(new BubbleSortStrategy());
processor.ProcessData(data);

processor.SetStrategy(new LinqSortStrategy());  // swap at runtime
processor.ProcessData(data);

Real-World Example — Discount Strategy

C# — Discount calculation strategies
public interface IDiscountStrategy
{
    decimal Calculate(decimal originalPrice);
    string  Description { get; }
}

public class NoDiscount         : IDiscountStrategy
{ public string  Description => "No discount";
  public decimal Calculate(decimal p) => p; }

public class PercentageDiscount  : IDiscountStrategy
{
    private readonly decimal _pct;
    public PercentageDiscount(decimal pct) => _pct = pct;
    public string  Description => $"{_pct}% off";
    public decimal Calculate(decimal p) => p * (1 - _pct / 100);
}

public class BuyOneGetOneFree   : IDiscountStrategy
{ public string  Description => "BOGOF";
  public decimal Calculate(decimal p) => p / 2; }

public class FlatAmountDiscount : IDiscountStrategy
{
    private readonly decimal _amount;
    public FlatAmountDiscount(decimal amount) => _amount = amount;
    public string  Description => $"£{_amount} off";
    public decimal Calculate(decimal p) => Math.Max(0, p - _amount);
}

public class ShoppingCart
{
    private IDiscountStrategy _discount = new NoDiscount();

    public void ApplyDiscount(IDiscountStrategy strategy) => _discount = strategy;

    public decimal Checkout(decimal subtotal)
    {
        var final = _discount.Calculate(subtotal);
        Console.WriteLine($"{_discount.Description}: £{subtotal} → £{final:F2}");
        return final;
    }
}

// Usage — strategy set based on customer type or promo code
var cart = new ShoppingCart();
cart.ApplyDiscount(new PercentageDiscount(20));
cart.Checkout(100m);  // 20% off: £100 → £80.00

cart.ApplyDiscount(new BuyOneGetOneFree());
cart.Checkout(100m);  // BOGOF: £100 → £50.00
SOLID: O — Add new strategies without touching existing code S — Each strategy class has one algorithm D — Context depends on IDiscountStrategy, not concrete class

👁️ Observer Pattern

👶 In Plain English

Observer is a subscription mechanism. When an object (the Subject/Publisher) changes state, all subscribed objects (Observers/Subscribers) are notified automatically. The subject doesn't need to know who is listening — it just announces changes. Observers sign up to receive those announcements.

💡 Real World Analogy

A newspaper subscription. The newspaper (subject) publishes new editions. Thousands of subscribers (observers) receive the paper. The newspaper doesn't know who specifically reads it — it just sends to all subscribers. You subscribe and unsubscribe freely. When a new edition is published, every subscriber gets it automatically without the newspaper knowing anything about them.

Observer Implementation

C# — Stock price observer
// ── Observer interface — anything that can receive notifications ───
public interface IStockObserver
{
    void OnPriceChanged(string symbol, decimal newPrice);
}

// ── Subject — maintains observers, notifies on change ─────────────
public class StockMarket
{
    private readonly List<IStockObserver>          _observers = new();
    private readonly Dictionary<string, decimal>    _prices    = new();

    public void Subscribe(IStockObserver observer)   => _observers.Add(observer);
    public void Unsubscribe(IStockObserver observer) => _observers.Remove(observer);

    public void UpdatePrice(string symbol, decimal newPrice)
    {
        _prices[symbol] = newPrice;
        NotifyAll(symbol, newPrice);   // tell everyone
    }

    private void NotifyAll(string symbol, decimal price)
    {
        foreach (var observer in _observers)
            observer.OnPriceChanged(symbol, price);
    }
}

// ── Concrete observers ────────────────────────────────────────────
public class TradingBot : IStockObserver
{
    private readonly decimal _buyThreshold;
    public TradingBot(decimal buyAt) => _buyThreshold = buyAt;

    public void OnPriceChanged(string symbol, decimal price)
    {
        if (price <= _buyThreshold)
            Console.WriteLine($"🤖 Bot: BUY {symbol} at £{price}");
    }
}

public class PriceAlertService : IStockObserver
{
    public void OnPriceChanged(string symbol, decimal price)
        => Console.WriteLine($"🔔 Alert: {symbol} is now £{price}");
}

public class PortfolioDashboard : IStockObserver
{
    public void OnPriceChanged(string symbol, decimal price)
        => Console.WriteLine($"📊 Dashboard updated: {symbol} = £{price}");
}

// Usage
var market    = new StockMarket();
var bot       = new TradingBot(150m);
var alerts    = new PriceAlertService();
var dashboard = new PortfolioDashboard();

market.Subscribe(bot);
market.Subscribe(alerts);
market.Subscribe(dashboard);

market.UpdatePrice("AAPL", 148m);
// 🤖 Bot: BUY AAPL at £148
// 🔔 Alert: AAPL is now £148
// 📊 Dashboard updated: AAPL = £148

// Unsubscribe bot — still notifies others
market.Unsubscribe(bot);
market.UpdatePrice("AAPL", 155m);
// 🔔 Alert: AAPL is now £155
// 📊 Dashboard updated: AAPL = £155

Observer with C# Events (Built-in Pattern)

C# — Using built-in events and delegates
// C# has Observer built-in via events — this is the idiomatic approach
public class OrderService
{
    // Declare the event — any subscriber can listen
    public event EventHandler<OrderPlacedEventArgs>? OrderPlaced;

    public void PlaceOrder(Order order)
    {
        // Business logic...
        Console.WriteLine($"Order {order.Id} placed");

        // Notify all subscribers
        OrderPlaced?.Invoke(this, new OrderPlacedEventArgs(order));
    }
}

public class OrderPlacedEventArgs : EventArgs
{
    public Order Order { get; }
    public OrderPlacedEventArgs(Order order) => Order = order;
}

// Subscribers — attach with +=, detach with -=
var orders = new OrderService();

orders.OrderPlaced += (sender, e) =>
    Console.WriteLine($"📧 Email sent for order {e.Order.Id}");

orders.OrderPlaced += (sender, e) =>
    Console.WriteLine($"📦 Inventory reserved for order {e.Order.Id}");

orders.OrderPlaced += (sender, e) =>
    Console.WriteLine($"🏷️ Loyalty points awarded for order {e.Order.Id}");

orders.PlaceOrder(new Order { Id = 42 });
// All three handlers fire automatically
SOLID: O — Add new observers without touching the subject S — Subject manages subscriptions; each observer handles its own response D — Subject depends on IObserver interface, not concrete observers

📦 Command Pattern

👶 In Plain English

Command turns a request into a standalone object. Instead of calling a method directly, you wrap the call in a Command object and pass it around. This lets you queue commands, log them, undo them, or replay them. The sender doesn't know how the command is executed — it just calls Execute().

💡 Real World Analogy

A restaurant order slip. The waiter doesn't cook — they write your order on a slip (Command object) and give it to the kitchen. The kitchen (Receiver) executes it. The slip can be queued, cancelled, or reordered. The waiter doesn't know how to cook anything — they just hand the slip and the kitchen does the work.

Command with Undo/Redo

C# — Text editor with undo/redo using Command
// ── Command interface ─────────────────────────────────────────────
public interface ICommand
{
    void Execute();
    void Undo();
}

// ── Receiver — the text document ──────────────────────────────────
public class TextDocument
{
    private string _content = "";

    public void InsertText(int pos, string text)
    {
        _content = _content.Insert(pos, text);
        Console.WriteLine($"Doc: \"{_content}\"");
    }

    public void DeleteText(int pos, int length)
    {
        _content = _content.Remove(pos, length);
        Console.WriteLine($"Doc: \"{_content}\"");
    }

    public string Content => _content;
}

// ── Concrete commands ─────────────────────────────────────────────
public class InsertCommand : ICommand
{
    private readonly TextDocument _doc;
    private readonly int          _position;
    private readonly string       _text;

    public InsertCommand(TextDocument doc, int pos, string text)
    { _doc = doc; _position = pos; _text = text; }

    public void Execute() => _doc.InsertText(_position, _text);
    public void Undo()    => _doc.DeleteText(_position, _text.Length);
}

public class DeleteCommand : ICommand
{
    private readonly TextDocument _doc;
    private readonly int          _position;
    private readonly int          _length;
    private          string       _deletedText = "";  // saved for undo

    public DeleteCommand(TextDocument doc, int pos, int len)
    { _doc = doc; _position = pos; _length = len; }

    public void Execute()
    {
        _deletedText = _doc.Content.Substring(_position, _length); // save for undo
        _doc.DeleteText(_position, _length);
    }
    public void Undo() => _doc.InsertText(_position, _deletedText);
}

// ── Invoker — manages command history ─────────────────────────────
public class TextEditor
{
    private readonly Stack<ICommand> _history = new();
    private readonly Stack<ICommand> _redoStack = new();

    public void Execute(ICommand cmd)
    {
        cmd.Execute();
        _history.Push(cmd);
        _redoStack.Clear();    // new action clears redo history
    }

    public void Undo()
    {
        if (_history.Count == 0) { Console.WriteLine("Nothing to undo"); return; }
        var cmd = _history.Pop();
        cmd.Undo();
        _redoStack.Push(cmd);
    }

    public void Redo()
    {
        if (_redoStack.Count == 0) { Console.WriteLine("Nothing to redo"); return; }
        var cmd = _redoStack.Pop();
        cmd.Execute();
        _history.Push(cmd);
    }
}

// Usage
var doc    = new TextDocument();
var editor = new TextEditor();

editor.Execute(new InsertCommand(doc, 0, "Hello"));   // Doc: "Hello"
editor.Execute(new InsertCommand(doc, 5, " World"));  // Doc: "Hello World"
editor.Undo();                                          // Doc: "Hello"
editor.Redo();                                          // Doc: "Hello World"
SOLID: O — Add new commands without changing the editor/invoker S — Editor manages history; each command manages one action

📋 Template Method Pattern

👶 In Plain English

Template Method defines the skeleton of an algorithm in a base class, and lets subclasses override specific steps without changing the overall structure. The base class says "here is the sequence of steps" — subclasses fill in the details of individual steps.

💡 Real World Analogy

Making tea and making coffee follow the same template: boil water → brew → pour into cup → add extras. The template (algorithm skeleton) is the same. The brewing step is different (tea bag vs coffee grounds), and the extras are different (milk/sugar vs cream/sugar). Template Method says: define the sequence once, let subclasses fill in their specific steps.

C# — Data import pipeline with Template Method
// Base class defines the TEMPLATE — the fixed sequence of steps
public abstract class DataImporter
{
    // Template Method — the algorithm skeleton (sealed = can't override)
    public sealed void Import(string source)
    {
        // Fixed sequence — always the same order
        var rawData  = ReadData(source);      // Step 1: read
        var parsed   = ParseData(rawData);    // Step 2: parse
        var valid    = ValidateData(parsed);  // Step 3: validate
        SaveData(valid);                      // Step 4: save
        SendReport(valid.Count);             // Step 5: report (optional hook)
    }

    // Abstract steps — subclasses MUST implement these
    protected abstract string       ReadData(string source);
    protected abstract List<object> ParseData(string raw);
    protected abstract List<object> ValidateData(List<object> data);
    protected abstract void         SaveData(List<object> data);

    // Hook — optional override (has default behaviour)
    protected virtual void SendReport(int count)
        => Console.WriteLine($"Imported {count} records");
}

// ── CSV Importer — implements CSV-specific steps ───────────────────
public class CsvImporter : DataImporter
{
    protected override string ReadData(string path)
    {
        Console.WriteLine($"Reading CSV: {path}");
        return File.ReadAllText(path);
    }
    protected override List<object> ParseData(string raw)
    {
        Console.WriteLine("Parsing CSV rows...");
        return raw.Split('\n').Select(r => (object)r).ToList();
    }
    protected override List<object> ValidateData(List<object> data)
    {
        Console.WriteLine("Validating CSV data...");
        return data.Where(r => !string.IsNullOrWhiteSpace(r.ToString())).ToList();
    }
    protected override void SaveData(List<object> data)
        => Console.WriteLine($"Saving {data.Count} CSV rows to DB");
}

// ── JSON Importer — same template, different steps ────────────────
public class JsonImporter : DataImporter
{
    protected override string ReadData(string url)
    {
        Console.WriteLine($"Fetching JSON from: {url}");
        return "[{\"id\":1},{\"id\":2}]";  // simplified
    }
    protected override List<object> ParseData(string raw)
    {
        Console.WriteLine("Parsing JSON array...");
        return [ "record1", "record2" ]; // simplified
    }
    protected override List<object> ValidateData(List<object> data)
    {
        Console.WriteLine("Validating JSON schema..."); return data;
    }
    protected override void SaveData(List<object> data)
        => Console.WriteLine($"Saving {data.Count} JSON records");

    // Override the hook — send email report instead of console
    protected override void SendReport(int count)
        => Console.WriteLine($"📧 Email: JSON import complete — {count} records");
}

// Usage — same Import() call, different behaviour
DataImporter csv  = new CsvImporter();
DataImporter json = new JsonImporter();

csv.Import("data.csv");
json.Import("https://api.example.com/data");
SOLID: O — Add new importers without touching the template S — Base class defines structure; each subclass handles its format L — Any DataImporter subtype works wherever DataImporter is expected

🔁 Iterator Pattern

👶 In Plain English

Iterator provides a way to access elements of a collection one by one without knowing how the collection is stored internally. Whether the data is in an array, a linked list, a tree, or a database — the iterator gives you a consistent way to go through it: has next? → get next.

💡 Real World Analogy

A TV remote's channel button is an iterator. You don't care how the TV stores its channel list (array? linked list? sorted?). You just press Next and get the next channel. Press it again, get the one after. The remote gives you a consistent way to traverse the channels without knowing the internal storage.

C# — Custom iterator with IEnumerable
// ── Custom collection with iteration ──────────────────────────────
public class PagedResult<T> : IEnumerable<T>
{
    private readonly List<T> _items;
    private readonly int    _pageSize;

    public int TotalCount  => _items.Count;
    public int TotalPages  => (int)Math.Ceiling(_items.Count / (double)_pageSize);

    public PagedResult(List<T> items, int pageSize)
    { _items = items; _pageSize = pageSize; }

    // Implementing IEnumerable makes this work with foreach
    public IEnumerator<T> GetEnumerator()
        => _items.GetEnumerator();

    System.Collections.IEnumerator System.Collections.IEnumerable.GetEnumerator()
        => GetEnumerator();

    // Get items for a specific page
    public IEnumerable<T> GetPage(int pageNumber)
        => _items.Skip((pageNumber - 1) * _pageSize).Take(_pageSize);
}

// ── Iterator using yield return (C# generator) ────────────────────
public class TreeNode<T>
{
    public T               Value    { get; }
    public List<TreeNode<T>> Children { get; } = new();
    public TreeNode(T value) => Value = value;

    // Depth-first traversal using yield return
    public IEnumerable<T> Traverse()
    {
        yield return Value;                          // yield this node
        foreach (var child in Children)
            foreach (var value in child.Traverse())  // recurse into children
                yield return value;
    }
}

// ── Async Iterator with yield return (C# 8+) ─────────────────────
public async IAsyncEnumerable<Product> GetProductsPagedAsync(int pageSize)
{
    int page = 1;
    while (true)
    {
        var products = await _db.GetProductsAsync(page, pageSize);
        if (!products.Any()) yield break;

        foreach (var p in products)
            yield return p;         // process as they arrive

        page++;
    }
}

// Usage — foreach works on ALL of the above
var root = new TreeNode<string>("root");
root.Children.Add(new TreeNode<string>("child1"));
root.Children.Add(new TreeNode<string>("child2"));

foreach (var node in root.Traverse())
    Console.WriteLine(node);  // root, child1, child2

💡 Iterator is Built Into C#

IEnumerable<T> and IEnumerator<T> are the Iterator pattern built into .NET. Every foreach loop uses an iterator. The yield return keyword makes writing custom iterators trivial — the compiler generates the state machine for you.

SOLID: S — Collection manages data; iterator manages traversal I — IEnumerable is a small focused interface

🔗 Chain of Responsibility Pattern

👶 In Plain English

Chain of Responsibility passes a request along a chain of handlers. Each handler decides: can I handle this? If yes, handle it. If no, pass it to the next handler. The sender doesn't know which handler will process the request — it just sends it to the first one in the chain.

💡 Real World Analogy

A customer support escalation. Your complaint goes to Level 1 support. They can't solve it → escalates to Level 2. They can't solve it → escalates to a Manager. The Manager resolves it. You didn't choose who handles it — it flows up the chain until someone can deal with it. Each level has authority to handle certain types of problems.

C# — HTTP request validation pipeline
// ── Handler interface ─────────────────────────────────────────────
public abstract class RequestHandler
{
    private RequestHandler? _next;

    // Fluent: handler.SetNext(auth).SetNext(rateLimit).SetNext(logger)
    public RequestHandler SetNext(RequestHandler next)
    {
        _next = next;
        return next;  // return next for fluent chaining
    }

    // Each handler calls this to pass to the next handler
    protected void PassToNext(HttpRequest request)
        => _next?.Handle(request) ?? Console.WriteLine("✅ Request processed!");

    public abstract void Handle(HttpRequest request);
}

// ── Concrete handlers ─────────────────────────────────────────────
public class AuthenticationHandler : RequestHandler
{
    public override void Handle(HttpRequest request)
    {
        Console.WriteLine("🔐 Auth: checking token...");
        if (string.IsNullOrEmpty(request.Token))
        {
            Console.WriteLine("❌ Auth FAILED: No token");
            return;  // stop the chain
        }
        Console.WriteLine("✅ Auth passed");
        PassToNext(request);  // continue to next handler
    }
}

public class RateLimitHandler : RequestHandler
{
    private int _requestCount;

    public override void Handle(HttpRequest request)
    {
        Console.WriteLine($"⏱️ Rate limit: request #{++_requestCount}");
        if (_requestCount > 100)
        {
            Console.WriteLine("❌ Rate limit exceeded");
            return;
        }
        PassToNext(request);
    }
}

public class ValidationHandler : RequestHandler
{
    public override void Handle(HttpRequest request)
    {
        Console.WriteLine("📋 Validating request body...");
        if (request.Body?.Length > 10_000)
        {
            Console.WriteLine("❌ Body too large");
            return;
        }
        PassToNext(request);
    }
}

public class LoggingHandler : RequestHandler
{
    public override void Handle(HttpRequest request)
    {
        Console.WriteLine($"📝 Logging: {request.Path}");
        PassToNext(request);  // always continues
    }
}

// ── Build the chain ───────────────────────────────────────────────
var auth      = new AuthenticationHandler();
var rateLimit = new RateLimitHandler();
var validate  = new ValidationHandler();
var logging   = new LoggingHandler();

// Chain: auth → rateLimit → validate → logging → execute
auth.SetNext(rateLimit).SetNext(validate).SetNext(logging);

var req = new HttpRequest { Token = "abc123", Path = "/api/users", Body = "data" };
auth.Handle(req);
// 🔐 Auth: checking token... ✅ Auth passed
// ⏱️ Rate limit: request #1
// 📋 Validating request body...
// 📝 Logging: /api/users
// ✅ Request processed!
SOLID: O — Add new handlers without modifying existing ones S — Each handler has one responsibility D — Handlers depend on abstract RequestHandler, not concrete next

📡 Mediator Pattern

👶 In Plain English

Mediator prevents objects from referring to each other directly. Instead, all communication goes through a central mediator object. Objects only know about the mediator — not about each other. This reduces the spaghetti of direct connections between many objects.

💡 Real World Analogy

An air traffic controller. Planes don't communicate directly with each other — they'd crash! Every plane talks only to the control tower (Mediator). The tower knows all the planes and coordinates their paths. Planes only need to know how to talk to the tower — not to every other plane in the sky. Remove the tower and you have chaos.

C# — Chat room mediator
// ── Mediator interface ────────────────────────────────────────────
public interface IChatRoom
{
    void Register(ChatUser user);
    void SendMessage(string from, string? to, string message);
}

// ── Concrete Mediator ─────────────────────────────────────────────
public class ChatRoom : IChatRoom
{
    private readonly Dictionary<string, ChatUser> _users = new();

    public void Register(ChatUser user)
    {
        _users[user.Name] = user;
        Console.WriteLine($"🟢 {user.Name} joined the room");
    }

    public void SendMessage(string from, string? to, string message)
    {
        if (to != null)
        {
            // Direct message
            if (_users.TryGetValue(to, out var recipient))
                recipient.Receive(from, message);
            else
                Console.WriteLine($"User {to} not found");
        }
        else
        {
            // Broadcast to everyone except sender
            foreach (var user in _users.Values.Where(u => u.Name != from))
                user.Receive(from, message);
        }
    }
}

// ── Colleague — knows only the mediator, not other users ──────────
public class ChatUser
{
    private readonly IChatRoom _room;
    public string Name { get; }

    public ChatUser(string name, IChatRoom room)
    { Name = name; _room = room; room.Register(this); }

    public void Send(string message, string? to = null)
    {
        Console.WriteLine($"{Name} → {(to ?? "all")}: \"{message}\"");
        _room.SendMessage(Name, to, message);
    }

    public void Receive(string from, string message)
        => Console.WriteLine($"  [{Name} receives from {from}]: \"{message}\"");
}

// Usage — users talk through the mediator, never directly
IChatRoom room  = new ChatRoom();
var        alice = new ChatUser("Alice", room);
var        bob   = new ChatUser("Bob",   room);
var        carol = new ChatUser("Carol", room);

alice.Send("Hello everyone!");         // broadcast
bob.Send("Hey Alice!", to: "Alice");  // direct message
carol.Send("Welcome Bob!");             // broadcast

Mediator in Real .NET — MediatR

📦 MediatR Library — Mediator in ASP.NET Core

MediatR is the most popular .NET Mediator implementation. Commands and queries are sent through a central mediator — handlers don't know about controllers, and controllers don't know about handlers.

  • Install: dotnet add package MediatR
  • Controller sends: await _mediator.Send(new GetUserQuery(id))
  • Handler implements: IRequestHandler<GetUserQuery, User>
  • Perfectly decoupled — controller never references handler directly
  • Used heavily in CQRS architecture patterns

🚦 State Pattern

👶 In Plain English

The State pattern allows an object to change its behaviour when its internal state changes. Instead of giant if/else or switch statements checking the state everywhere, each state is its own class with its own behaviour. The object delegates to its current state object, which handles actions appropriately for that state.

💡 Real World Analogy

A traffic light. When it is Red, cars stop. When Green, cars go. When Amber, cars prepare to stop. The same physical light behaves completely differently depending on its current state. You don't check "if red then stop, else if green then go" in your car — you react to whatever state the light currently shows. The light manages its own state transitions.

The Problem — If/Else State Checking

PROBLEM C# — State scattered in if/else throughout the class
// ❌ State scattered everywhere — hard to add new states
public class Order
{
    public string State { get; private set; } = "Pending";

    public void Pay()
    {
        if (State == "Pending")     State = "Paid";
        else if (State == "Paid")   Console.WriteLine("Already paid");
        else                        Console.WriteLine("Cannot pay");
    }
    public void Ship()
    {
        if (State == "Paid")        State = "Shipped";
        else if (State == "Pending") Console.WriteLine("Not paid yet");
        else                          Console.WriteLine("Cannot ship");
    }
    // ❌ Every method checks state. Adding "Refunded" state = edit EVERY method!
}

State Pattern Solution

C# — Order lifecycle with State pattern
// ── State interface — defines what an order can do in ANY state ───
public interface IOrderState
{
    void Pay(OrderContext order);
    void Ship(OrderContext order);
    void Cancel(OrderContext order);
    void Refund(OrderContext order);
    string StateName { get; }
}

// ── Context — holds current state and delegates to it ─────────────
public class OrderContext
{
    private IOrderState _state;

    public OrderContext()  => _state = new PendingState();
    public void SetState(IOrderState state)
    {
        Console.WriteLine($"  → Transitioning to {state.StateName}");
        _state = state;
    }
    public string CurrentState => _state.StateName;

    // Delegate ALL actions to the current state
    public void Pay()    => _state.Pay(this);
    public void Ship()   => _state.Ship(this);
    public void Cancel() => _state.Cancel(this);
    public void Refund() => _state.Refund(this);
}

// ── Concrete States — each handles actions for its specific state ──
public class PendingState : IOrderState
{
    public string StateName => "Pending";
    public void Pay(OrderContext o)    { Console.WriteLine("💳 Payment received");      o.SetState(new PaidState()); }
    public void Ship(OrderContext o)   => Console.WriteLine("❌ Pay first before shipping");
    public void Cancel(OrderContext o) { Console.WriteLine("🚫 Order cancelled");          o.SetState(new CancelledState()); }
    public void Refund(OrderContext o) => Console.WriteLine("❌ Cannot refund — not paid");
}

public class PaidState : IOrderState
{
    public string StateName => "Paid";
    public void Pay(OrderContext o)    => Console.WriteLine("❌ Already paid");
    public void Ship(OrderContext o)   { Console.WriteLine("📦 Order shipped!");         o.SetState(new ShippedState()); }
    public void Cancel(OrderContext o) { Console.WriteLine("🚫 Order cancelled + refund"); o.SetState(new CancelledState()); }
    public void Refund(OrderContext o) { Console.WriteLine("💰 Refund issued");          o.SetState(new RefundedState()); }
}

public class ShippedState : IOrderState
{
    public string StateName => "Shipped";
    public void Pay(OrderContext o)    => Console.WriteLine("❌ Already shipped and paid");
    public void Ship(OrderContext o)   => Console.WriteLine("❌ Already shipped");
    public void Cancel(OrderContext o) => Console.WriteLine("❌ Cannot cancel shipped order");
    public void Refund(OrderContext o) { Console.WriteLine("💰 Return + refund initiated"); o.SetState(new RefundedState()); }
}

public class CancelledState : IOrderState
{
    public string StateName => "Cancelled";
    public void Pay(OrderContext o)    => Console.WriteLine("❌ Order is cancelled");
    public void Ship(OrderContext o)   => Console.WriteLine("❌ Order is cancelled");
    public void Cancel(OrderContext o) => Console.WriteLine("❌ Already cancelled");
    public void Refund(OrderContext o) => Console.WriteLine("❌ Nothing to refund");
}

public class RefundedState : IOrderState
{
    public string StateName => "Refunded";
    public void Pay(OrderContext o)    => Console.WriteLine("❌ Order was refunded");
    public void Ship(OrderContext o)   => Console.WriteLine("❌ Order was refunded");
    public void Cancel(OrderContext o) => Console.WriteLine("❌ Order was refunded");
    public void Refund(OrderContext o) => Console.WriteLine("❌ Already refunded");
}

// Usage — clean state lifecycle
var order = new OrderContext();
Console.WriteLine($"State: {order.CurrentState}");   // Pending
order.Ship();                                           // ❌ Pay first
order.Pay();                                            // 💳 Payment received → Paid
order.Pay();                                            // ❌ Already paid
order.Ship();                                           // 📦 Shipped → Shipped
order.Cancel();                                        // ❌ Cannot cancel shipped
order.Refund();                                        // 💰 Return + refund → Refunded
Console.WriteLine($"Final state: {order.CurrentState}"); // Refunded

// ✅ Add "OnHold" state? Add ONE new class — zero changes to existing states!
SOLID: O — Add new states by adding new classes, not editing existing ones S — Each state class handles behaviour for exactly one state L — Any IOrderState can substitute for another
