AWS AppConfig

Deploy, validate, and manage application configuration safely — with feature flags, gradual rollouts, and instant rollback.

🏗️ What is AWS AppConfig?

A feature of AWS Systems Manager for safe, dynamic application configuration management.

One-liner AppConfig = deploy config changes to running apps safely, with validation, gradual rollout, bake time, and auto-rollback — no redeploy needed.
📦 Application

Top-level logical grouping. e.g. MyOrderService. Contains environments and config profiles.

🌍 Environment

A deployment target: Production, Staging, Dev. Each can have its own monitors and alarms.

📋 Config Profile

Defines what config to deploy — JSON, YAML, feature flags, or free-form text. Linked to a source (SSM / S3 / inline).

🚀 Deployment Strategy

Controls rollout speed — linear, exponential, or all-at-once. Defines bake time between steps.

🛡️ Validators

JSON Schema or Lambda validators that reject bad config before it reaches production.

🔔 Monitors

CloudWatch alarms attached to an environment. If alarm fires during rollout → auto-rollback.

Why AppConfig over plain SSM Parameter Store / S3? AppConfig adds validation, gradual rollout, bake time, automatic rollback on alarms, and a retrieval API with caching — turning raw config into a safe deployment pipeline.

Config Sources

🗂️ Hosted (Inline)

Store config directly in AppConfig (Hosted Configuration). Best for feature flags and simple JSON/YAML. Version-controlled by AppConfig.

🔑 SSM Parameter Store

Reference an SSM SecureString or StringList. Validated by AppConfig before deployment. Good for existing SSM workflows.

🪣 S3 Bucket

Reference a JSON/YAML file in S3. Supports large configs. AppConfig fetches, validates, and deploys the S3 object.

🚩 Feature Flags

AppConfig has first-class support for feature flags — toggle features without redeploy.

Feature Flag Profile Use config type AWS.AppConfig.FeatureFlags for a structured schema with built-in flag management UI in the AWS Console.

Feature Flag JSON Structure

JSON AppConfig feature flags format 
{
  "flags": {
    "enableNewCheckout": {
      "name": "Enable New Checkout Flow",
      "description": "Rolls out the redesigned checkout UX"
    },
    "darkModeSupport": {
      "name": "Dark Mode Support"
    },
    "maxItemsPerCart": {
      "name": "Max Items Per Cart",
      "attributes": {
        "limit": { "constraints": { "type": "number", "minimum": 1, "maximum": 100 } }
      }
    }
  },
  "values": {
    "enableNewCheckout": { "enabled": true },
    "darkModeSupport":   { "enabled": false },
    "maxItemsPerCart":   { "enabled": true, "limit": 50 }
  },
  "version": "1"
}
✅ Boolean Flags

Simple on/off flags — "enabled": true/false. Use for feature toggles, kill switches, and A/B flags.

🔢 Flags with Attributes

Flags with extra typed values (numbers, strings). E.g. toggle + set a numeric limit — all deployed atomically.

🎯 Use Cases

New feature rollout, dark launches, kill switches for buggy features, A/B testing config, gradual % rollout.

⚡ Zero Redeploy

App polls AppConfig endpoint — flag changes take effect within your configured poll interval (e.g. 30s) without any restart.

🔄 Deployment Strategies

Control how config rolls out — from instant to very gradual with bake time in between.

AllAtOnce
100% instantly · 0 min bake
Deploy to all targets immediately. Fast but risky — no gradual exposure. Use only for non-critical config or Dev environments.
Linear50PercentEvery30Seconds
50% → 100% · 30s bake
50% of traffic gets new config, waits 30s, then remainder. Good for staging. Balances speed vs safety.
Canary10Percent20Minutes
10% → 100% · 20 min bake
10% of traffic first, monitors for 20 minutes, then full rollout. Safe for production — alarms can trigger rollback during the bake window.
Custom Strategy
You define % · bake time
Define your own growth type (Linear / Exponential), growth factor (%), deployment duration, and bake time to suit your SLOs.

Deployment Lifecycle

  • 1
    Start DeploymentSpecify app, environment, config profile, config version, deployment strategy, and optional KMS key.
  • 2
    ValidateJSON Schema or Lambda validator runs against the new config. Deployment is blocked if validation fails.
  • 3
    Gradual RolloutConfig is deployed incrementally per the strategy. Clients polling the API receive new config in waves.
  • 4
    Bake TimeAppConfig waits between steps, monitoring CloudWatch alarms. If any alarm triggers → automatic rollback.
  • 5
    Complete / RollbackDeployment completes when 100% is reached and bake time passes. If alarm fires at any step, previous config is restored instantly.
⏱️ Bake Time The bake time is a monitoring window between deployment steps. During this window, AppConfig watches your CloudWatch alarms. If an alarm breaches, rollback is immediate — even if you're already at 90% rollout.

✅ Validators

Prevent bad config from reaching production — validate before deployment starts.

📐 JSON Schema Validator

Attach a JSON Schema document to a config profile. AppConfig validates the new config content against the schema before deploying.

Good for: type checks, required fields, min/max values, enums.

Built-in No Lambda needed.

λ Lambda Validator

Call a Lambda function with the config content. Lambda returns pass/fail + error message. Can do complex business logic validation.

Good for: cross-field validation, external dependency checks, custom rules.

Custom Full flexibility.

Lambda Validator (C#)

C# Lambda validator function for AppConfig 
using Amazon.Lambda.Core;
using System.Text.Json;

public class AppConfigValidator
{
    public void FunctionHandler(AppConfigValidationEvent evt, ILambdaContext ctx)
    {
        try
        {
            var config = JsonSerializer.Deserialize<AppSettings>(evt.Content)
                ?? throw new Exception("Config is null");

            // Custom business rule: timeout must be less than maxRetries * 30
            if (config.TimeoutSeconds >= config.MaxRetries * 30)
                throw new Exception(
                    $"TimeoutSeconds ({config.TimeoutSeconds}) must be less than MaxRetries*30 ({config.MaxRetries * 30})");

            // If we reach here, validation passes (no exception = success)
        }
        catch (Exception ex)
        {
            // AppConfig treats any exception as a validation failure
            ctx.Logger.LogError($"Validation failed: {ex.Message}");
            throw;
        }
    }
}

public record AppSettings(int MaxRetries, int TimeoutSeconds, bool FeatureEnabled);
public record AppConfigValidationEvent(string Content, string Uri);
⚠️ Validation Timing Validators run when you start a deployment, not when you save a config version. If validation fails, the deployment is blocked and no traffic receives the bad config.

🧩 C# / .NET SDK Examples

Using AWSSDK.AppConfigData (retrieval) and AWSSDK.AppConfig (management) NuGet packages.

Two SDKs — Different Purposes AWSSDK.AppConfig = manage apps, environments, profiles, deployments. AWSSDK.AppConfigData = retrieve config from a running app (the hot path).
C# Retrieve config with AppConfigData (with session caching) 
using Amazon.AppConfigData;
using Amazon.AppConfigData.Model;
using System.Text.Json;

public class AppConfigService
{
    private readonly IAmazonAppConfigData _client;
    private string? _sessionToken;
    private AppSettings? _cachedConfig;

    public AppConfigService(IAmazonAppConfigData client) => _client = client;

    public async Task<AppSettings> GetConfigAsync()
    {
        // Step 1: Start a session (once per app lifecycle)
        if (_sessionToken is null)
        {
            var session = await _client.StartConfigurationSessionAsync(
                new StartConfigurationSessionRequest
                {
                    ApplicationIdentifier         = "OrderService",
                    EnvironmentIdentifier         = "Production",
                    ConfigurationProfileIdentifier = "AppSettings",
                    RequiredMinimumPollIntervalInSeconds = 30
                });
            _sessionToken = session.InitialConfigurationToken;
        }

        // Step 2: Poll for latest config
        var response = await _client.GetLatestConfigurationAsync(
            new GetLatestConfigurationRequest
                { ConfigurationToken = _sessionToken });

        // Update token for next poll
        _sessionToken = response.NextPollConfigurationToken;

        // Step 3: Only deserialise if config changed (non-empty response)
        if (response.Configuration.Length > 0)
        {
            using var reader = new StreamReader(response.Configuration);
            var json = await reader.ReadToEndAsync();
            _cachedConfig = JsonSerializer.Deserialize<AppSettings>(json);
        }

        return _cachedConfig ?? throw new InvalidOperationException("No config loaded");
    }
}

public record AppSettings(int MaxRetries, int TimeoutSeconds, bool FeatureEnabled);
C# Create AppConfig resources programmatically 
using Amazon.AppConfig;
using Amazon.AppConfig.Model;

var mgmt = new AmazonAppConfigClient();

// 1. Create Application
var app = await mgmt.CreateApplicationAsync(new CreateApplicationRequest
    { Name = "OrderService", Description = "Order processing service config" });

// 2. Create Environment
var env = await mgmt.CreateEnvironmentAsync(new CreateEnvironmentRequest
{
    ApplicationId = app.Id,
    Name          = "Production",
    Monitors      =
    [
        new Monitor
        {
            AlarmArn     = "arn:aws:cloudwatch:us-east-1:123:alarm:HighErrorRate",
            AlarmRoleArn = "arn:aws:iam::123:role/AppConfigMonitorRole"
        }
    ]
});

// 3. Create Config Profile (hosted)
var profile = await mgmt.CreateConfigurationProfileAsync(new CreateConfigurationProfileRequest
{
    ApplicationId  = app.Id,
    Name           = "AppSettings",
    LocationUri    = "hosted",
    Type           = "AWS.Freeform",
    Validators     =
    [
        new Validator { Type = ValidatorType.JSON_SCHEMA, Content = schemaJson }
    ]
});

// 4. Create a Hosted Config Version
var version = await mgmt.CreateHostedConfigurationVersionAsync(new CreateHostedConfigurationVersionRequest
{
    ApplicationId          = app.Id,
    ConfigurationProfileId = profile.Id,
    Content                = new MemoryStream(
        System.Text.Encoding.UTF8.GetBytes("""{"maxRetries":3,"timeoutSeconds":30,"featureEnabled":true}""")),
    ContentType            = "application/json"
});

// 5. Start Deployment
await mgmt.StartDeploymentAsync(new StartDeploymentRequest
{
    ApplicationId          = app.Id,
    EnvironmentId          = env.Id,
    ConfigurationProfileId = profile.Id,
    ConfigurationVersion   = version.VersionNumber.ToString(),
    DeploymentStrategyId   = "AppConfig.Canary10Percent20Minutes"
});

🔗 Integrations

AppConfig plugs into the wider AWS ecosystem for config retrieval, secrets, and observability.

🔑 AWS Secrets Manager

Reference Secrets Manager ARNs inside AppConfig JSON. Secrets are resolved at retrieval time. Keep credentials out of config content.

🔐 AWS KMS

Encrypt hosted config versions at rest using a customer-managed KMS key. Specify the KMS key when starting a deployment.

📊 CloudWatch

Attach CloudWatch alarms to environments as monitors. AppConfig auto-rolls back if an alarm fires during the bake window.

🗂️ SSM Parameter Store

Use SSM parameters as config source. AppConfig validates and deploys the parameter value, acting as a safe wrapper around SSM.

🪣 S3

Store large configs in S3. AppConfig fetches the object, validates it, and deploys. Supports versioned S3 objects.

🏗️ CloudFormation / CDK

Define AppConfig resources as IaC — Application, Environment, ConfigProfile, HostedConfigVersion, and Deployment all have CFN resources.

Lambda Extension (No SDK polling needed)

AppConfig Lambda Extension Add the AppConfig agent as a Lambda layer. It runs as a local HTTP proxy on port 2772 — your Lambda calls localhost:2772 instead of the AppConfig API directly. The extension handles caching, session management, and polling automatically.
C# Retrieve config via Lambda Extension (localhost proxy) 
using System.Net.Http;

// Lambda Extension exposes AppConfig on localhost:2772
var url = "http://localhost:2772/applications/OrderService"
         + "/environments/Production"
         + "/configurations/AppSettings";

using var http = new HttpClient();
var json = await http.GetStringAsync(url);
var settings = JsonSerializer.Deserialize<AppSettings>(json)!;

// Extension caches for 300s by default — no extra session management needed
Console.WriteLine($"MaxRetries: {settings.MaxRetries}");

💡 Best Practices

Production-grade AppConfig setup for safety, performance, and cost efficiency.

✅ Always Validate

Attach a JSON Schema validator to every config profile. For complex configs, add a Lambda validator on top. Never deploy unvalidated config.

✅ Use Canary in Production

Always use Canary10Percent20Minutes or a custom slow strategy in production. Never use AllAtOnce in prod — you lose rollback protection.

✅ Attach CloudWatch Monitors

Link your error rate / latency alarms to the production environment. AppConfig's auto-rollback only works if monitors are configured.

✅ Cache Config in App

Don't call AppConfig on every request. Use RequiredMinimumPollIntervalInSeconds (min 15s). Cache in memory and refresh in background.

✅ Use Lambda Extension

For Lambda functions, use the AppConfig agent extension. It handles caching, retries, and background refresh — no SDK boilerplate needed.

✅ Separate Environments

Always create separate AppConfig environments for Dev, Staging, and Production — even if they share the same config profile. Gives per-environment alarm config.

Polling Strategy Comparison

ApproachMechanismProsCons
SDK (AppConfigData) Manual session + poll in app code Full control, works anywhere More boilerplate, manage token yourself
Lambda Extension localhost:2772 HTTP proxy Zero SDK code, built-in caching Lambda only
ECS Agent / EC2 Agent AppConfig agent sidecar / daemon Same localhost pattern, works on ECS/EC2 Needs agent setup
💰 Cost Note AppConfig charges per config retrieval call — not per deployment. Caching is important both for performance and cost. Each GetLatestConfiguration call is billed, but if config hasn't changed, the response body is empty (reducing data transfer cost too).

🏁 Summary

Everything you need to remember about AWS AppConfig.

📦 Object Model

Application → Environment → Config Profile → Hosted Version → Deployment. Each layer serves a distinct purpose.

🚩 Feature Flags

Use AWS.AppConfig.FeatureFlags profile type for structured flag management with typed attributes and console UI support.

🔄 Safe Deployments

Linear, exponential, or canary rollout strategies. Bake time + CloudWatch monitors = automatic rollback on bad deployments.

✅ Validation

JSON Schema for type/structure, Lambda for business logic. Both run before deployment starts — bad config is blocked at the gate.

🧩 Retrieval

SDK session + polling for full control. Lambda Extension (localhost:2772) for Lambda — zero boilerplate, built-in caching.

🔗 Sources

Hosted (inline), SSM Parameter Store, or S3. All sources go through the same validation + deployment pipeline.
