SignalR — Beginner to Advanced

How browsers and servers actually push data to each other in real-time, and how to build it in ASP.NET Core

WebSockets Hubs Groups Connection Lifecycle Scaling Out Streaming

⚡ Why Does SignalR Even Exist?

Think about how a normal web request works: your browser asks for something, the server answers, done. The server never speaks unless spoken to. Now imagine you're building a live sports score app — when the score changes, you want every single person watching to see it update INSTANTLY, without them refreshing the page. The server needs to be the one to start the conversation this time. Nothing about normal HTTP lets it do that.
🤔 Couldn't the browser just keep asking "any updates?" every second?

Yes, and that's actually the first thing everyone tries — it's called polling. Your browser sends a request every second asking "anything new?" Most of the time the answer is "nope," so you're burning requests and server resources on mostly-empty responses. And there's an unavoidable delay — if the score changes right after you asked, you won't know for up to a second. For a sports score, mildly annoying. For a stock trading app or a multiplayer game, that delay is a dealbreaker.

🤔 What if the browser just... holds the request open and waits for something to happen?

That's the next idea people land on, called long polling. The browser sends a request, and instead of the server replying immediately, it just holds the connection open until there's actually something to say — then responds, and the browser immediately opens a new one. It's better than naive polling, but you're still paying the overhead of opening a fresh HTTP connection over and over, and there's still a small gap between one request ending and the next one starting.

💡 What If the Connection Just... Stayed Open?

This is the idea behind a WebSocket — instead of the request/response back-and-forth, the browser and server open ONE connection and just keep it alive. Either side can send a message down that pipe at any moment, with no "asking permission" first. The server can push the score update the INSTANT it happens, because the pipe to your browser is just sitting there, open, ready.

So Why Not Just Use WebSockets Directly?

🤔 If WebSockets solve the problem, why does SignalR need to exist at all?

Because raw WebSockets only solve the "keep a pipe open" part — they leave you to handle a long list of real-world problems yourself: what if the browser is an old one that doesn't support WebSockets? What if a corporate firewall blocks them (this genuinely happens)? What happens when the connection drops — does your code reconnect automatically? How do you call a specific C# method on the server from JavaScript, with proper method names and arguments, instead of just shoving raw text down a pipe? None of that comes for free with a bare WebSocket.

💡 SignalR's Actual Job

SignalR is a library that gives you the real-time "server can push to client" capability, while automatically picking the best transport available (WebSockets if possible, falling back to Server-Sent Events, then long polling, without you writing that fallback logic), automatically reconnecting when a connection drops, and letting you write code that LOOKS like you're just calling a normal method — hub.SendMessage("hello") — even though under the hood it's serializing that call and sending it across whatever transport got negotiated.

The Three Pieces You'll Keep Hearing About

  • Hub — a class on your server that's the meeting point for real-time communication; clients call methods on it, and it can call methods back on connected clients
  • Connection — one client's link to the hub, identified by a unique connection ID, which exists for as long as that browser tab stays connected
  • Transport — the actual mechanism moving the data (WebSocket, Server-Sent Events, or long polling) — SignalR picks the best one automatically and you usually don't need to think about it

💡 You Don't Need to Pick a Transport Yourself

SignalR negotiates this automatically when a client connects: try WebSockets first, and only fall back to something else if the browser or network doesn't support them. Your C# and JavaScript code stays exactly the same regardless of which transport actually got used underneath — that's the whole point of the abstraction.

🛠️ Your First Hub

Let's actually build the simplest possible real-time thing: a counter that goes up for EVERYONE watching, the instant anyone clicks a button — no refresh, no polling. This is small enough to hold in your head completely, but it contains every core SignalR concept you'll use in much bigger projects.

Step 1 — The Hub Itself

🤔 What actually IS a Hub, mechanically?

It's just a regular C# class that inherits from SignalR's Hub base class. Any PUBLIC method you write on it becomes something JavaScript can call directly from the browser — SignalR handles the wiring. Think of it as the server-side "phone number" that clients dial into.

C# — The simplest possible Hub
public class CounterHub : Hub
{
    private static int _count = 0;

    // Called by JavaScript when the button is clicked.
    // "public" + inside a Hub = automatically callable from the client.
    public async Task Increment()
    {
        _count++;

        // Push the new value to EVERY connected client — not just whoever clicked.
        // "Clients.All" means "everyone currently connected to this hub".
        // "ReceiveCount" is just a NAME — the client decides what to do with it.
        await Clients.All.SendAsync("ReceiveCount", _count);
    }
}
🤔 "ReceiveCount" isn't a method I defined anywhere — where does it come from?

Good catch, and this trips people up the first time. It's not a real C# method — it's just a STRING that acts as a label. On the JavaScript side, you'll register a handler that listens for messages labeled "ReceiveCount" and the names just have to match between server and client. There's no compiler checking this for you in the basic setup (we'll see a stricter, strongly-typed version later in Advanced Patterns).

Step 2 — Wiring It Into Your App

C# — Program.cs registration
var builder = WebApplication.CreateBuilder(args);

builder.Services.AddSignalR();   // 1. Register SignalR services

var app = builder.Build();

app.MapHub<CounterHub>("/counterhub");  // 2. Expose the hub at this URL

app.Run();

Step 3 — The Browser Side

🤔 How does JavaScript even talk to a C# class?

Through the SignalR JavaScript client library, which gives you a HubConnection object. You point it at the URL you mapped the hub to, start the connection, and from then on you can call server methods by NAME (as a string) and listen for server-pushed messages by name too. It feels a bit like calling a remote API, except the connection stays open the whole time.

JavaScript — Connecting and using the hub
const connection = new signalR.HubConnectionBuilder()
    .withUrl("/counterhub")
    .build();

// Listen for the server calling "ReceiveCount" — name must match the SendAsync call in C#
connection.on("ReceiveCount", (newCount) => {
    document.getElementById("counterDisplay").innerText = newCount;
});

await connection.start();   // Opens the actual connection

document.getElementById("incrementBtn").addEventListener("click", () => {
    // Calls the C# "Increment" method by name — matches the public method on CounterHub
    connection.invoke("Increment");
});

Seeing the Whole Loop

🛠️ One Click, Everyone Updates

💡 The Part That Should Feel Surprising

Browser B never asked for anything. It just sat there with an open connection, and the server pushed data to it the instant Browser A clicked. That's the entire point of everything we just built — and it's the thing plain HTTP could never do.

⚠️ That Static Counter Won't Survive Multiple Servers

Notice _count is a static field living in server memory. This works fine on ONE server. The moment you run two server instances behind a load balancer, each has its OWN separate _count, and clients connected to different servers will see different numbers. We'll hit this exact problem properly in the Scaling Out tab — for now, just notice that it's lurking.

👥 Groups — Sending to Some People, Not Everyone

The counter example pushed updates to absolutely everyone connected — fine for a public counter, but imagine a chat app with 10,000 active conversations on the same server. If Alice sends a message in her conversation with Bob, you definitely don't want that pushed to all 10,000 conversations' worth of connected users. You need a way to say "just THESE people."
🤔 Couldn't I just keep my own list of "which connection IDs are in which conversation" and loop through it myself?

You could build that yourself, and it's not a crazy idea — but you'd be reinventing something SignalR already gives you for free, with extra bookkeeping (what happens when someone disconnects? do you remember to remove them from your list?). SignalR's built-in Groups feature is exactly this: a named bucket of connections, managed for you, that you can add a connection to, remove it from, and send messages to as a unit.

Adding Connections to a Group

C# — Joining and messaging a group
public class ChatHub : Hub
{
    // Called by a client when they open a specific conversation
    public async Task JoinConversation(string conversationId)
    {
        // Context.ConnectionId = THIS specific connection's unique ID
        // Groups.AddToGroupAsync just files this connection under that group name
        await Groups.AddToGroupAsync(Context.ConnectionId, conversationId);
    }

    public async Task SendMessage(string conversationId, string message)
    {
        // Only connections that joined THIS conversation's group receive it —
        // everyone in other conversations is completely unaffected
        await Clients.Group(conversationId).SendAsync("ReceiveMessage", message);
    }

    public async Task LeaveConversation(string conversationId)
    {
        await Groups.RemoveFromGroupAsync(Context.ConnectionId, conversationId);
    }
}
🤔 What if I want to message everyone EXCEPT the person who triggered it?

That's an extremely common need — you usually don't want to echo a message back to the person who just sent it (their own UI already shows it locally). SignalR has a few of these targeted options built in, which save you from manually filtering lists yourself.

📍 Common Targeting Options

  • Clients.All — every connected client
  • Clients.Caller — only the connection that called this method
  • Clients.Others — everyone EXCEPT the caller
  • Clients.Group("name") — everyone in that group
  • Clients.OthersInGroup("name") — group members except the caller
  • Clients.Client(connectionId) — one specific connection

🤔 Why So Many Options?

Each one maps to a real, common scenario: Others for "don't echo back to sender," Group for "this room/conversation only," Client for "send a private notification to one specific connection." Having these built in means you're rarely writing manual filtering logic.

Seeing Group Isolation in Action

👥 Two Conversations, Completely Isolated

⚠️ Group Membership Doesn't Automatically Survive a Reconnect

If a client's connection drops and SignalR reconnects them, they get a brand NEW connection ID — and group membership was tied to the OLD connection ID. This means you typically need to re-join groups after reconnecting (often inside an OnConnectedAsync override or a reconnect handler), which we'll cover properly in the next tab.

🔄 Connection Lifecycle & State

Someone opens your chat app. You want to show "Bob is online" to his friends. Later, Bob closes his laptop without clicking any "logout" button — his wifi just dies. How does your server even find out Bob is gone, if nobody told it?
🤔 If nobody explicitly says "I'm leaving," how would the server ever know?

This is exactly what the Hub lifecycle methods are for. SignalR automatically detects when a connection drops — whether that's a clean disconnect or wifi dying — and calls a method on your Hub when it happens. You don't poll for this or guess; SignalR's underlying transport notices the connection is gone and tells you.

The Two Methods You'll Override Constantly

C# — Tracking who's online
public class ChatHub : Hub
{
    private static readonly ConcurrentDictionary<string, string> _onlineUsers = new();

    // Called automatically the moment a client's connection is established
    public override async Task OnConnectedAsync()
    {
        var userId = Context.UserIdentifier ?? Context.ConnectionId;
        _onlineUsers[Context.ConnectionId] = userId;

        // Tell everyone else this person just came online
        await Clients.Others.SendAsync("UserOnline", userId);
        await base.OnConnectedAsync();
    }

    // Called automatically when the connection ends, for ANY reason —
    // clean logout, wifi dying, browser tab closed, server restart
    public override async Task OnDisconnectedAsync(Exception? exception)
    {
        if (_onlineUsers.TryRemove(Context.ConnectionId, out var userId))
        {
            await Clients.Others.SendAsync("UserOffline", userId);
        }
        await base.OnDisconnectedAsync(exception);
    }
}
🤔 Why does OnDisconnectedAsync take an Exception parameter?

Because it gets called for BOTH a graceful disconnect (parameter is null) and an abrupt one — network failure, crash, wifi dropping (parameter holds details about what went wrong). You usually don't need to inspect it for normal logic like "mark them offline," but it's there if you want to log or react differently to ungraceful disconnects specifically.

Reconnection — What Happens When Wifi Blips for 3 Seconds

📡 The Scenario

Someone's on a train, wifi cuts out for a few seconds, then comes back. Did they get logged out? Did they lose their group memberships? Does the app show a scary error?

🤔 Does SignalR automatically try to reconnect, or do I have to write that myself?

It can do it automatically, but you have to explicitly opt in by calling .withAutomaticReconnect() when building the connection on the client side — it's not the default. Once enabled, if the connection drops the client will keep attempting to reconnect using a backoff pattern, and your code hooks into events for what's currently happening: reconnecting, successfully reconnected, or given up entirely.

JavaScript — Handling reconnection explicitly
const connection = new signalR.HubConnectionBuilder()
    .withUrl("/chathub")
    .withAutomaticReconnect()   // opt in — not automatic by default
    .build();

connection.onreconnecting((error) => {
    // Connection just dropped — SignalR is retrying in the background
    showBanner("Reconnecting...");
});

connection.onreconnected((connectionId) => {
    // Success! But note: this is a BRAND NEW connection ID.
    // Group membership tied to the old ID is gone — re-establish it here.
    showBanner("Back online");
    connection.invoke("JoinConversation", currentConversationId);
});

connection.onclose((error) => {
    // Gave up entirely — show a real error, offer a "Retry" button
    showBanner("Connection lost. Please refresh.");
});
💡 The Detail That Catches People Off Guard

A reconnect gives you a NEW connection ID, not the old one back. Anything keyed by the old connection ID — group membership, any per-connection state you were tracking — is gone and needs to be rebuilt. This is exactly why we flagged it as a warning back in the Groups tab.

Storing Per-Connection State Safely

⚠️ Don't Store State as Instance Fields on the Hub

A new instance of your Hub class is created for EVERY method call — Hubs are not long-lived objects sitting around between requests. If you write private string _username; as a field and set it in one method, it'll be gone by the next method call, because that's a completely different instance. Use Context.Items for per-connection data that needs to persist across calls within the same connection lifetime.

C# — Context.Items for per-connection scratch space
public override async Task OnConnectedAsync()
{
    // Survives across multiple method calls FROM THIS SAME CONNECTION
    // (gone the instant this connection disconnects — that's the point)
    Context.Items["JoinedAt"] = DateTime.UtcNow;
    await base.OnConnectedAsync();
}

public TimeSpan GetSessionDuration()
{
    var joinedAt = (DateTime)Context.Items["JoinedAt"]!;
    return DateTime.UtcNow - joinedAt;
}

📈 Scaling Out — The Backplane Problem

Your chat app is a hit. One server isn't enough anymore, so you spin up a second instance behind a load balancer. Alice connects — routed to Server 1. Bob connects — routed to Server 2. Alice sends Bob a message. Nothing happens on Bob's screen. What went wrong?
🤔 Wasn't the whole point of Groups and Clients.All that SignalR handles routing for us?

It does — but only among connections that THAT SAME SERVER PROCESS knows about. Server 1's in-memory list of "who's connected" only contains Alice — it has never heard of Bob, because Bob's connection lives entirely in Server 2's memory. When Alice's message triggers Clients.Group(...).SendAsync(...) on Server 1, it can only reach connections Server 1 knows about. Bob is invisible to it.

💡 Multiple Servers Need a Shared Communication Channel

SignalR's answer is called a backplane — a shared message bus (typically Redis pub/sub) that every server both publishes to AND subscribes to. When Server 1 wants to send a message, it publishes to Redis. Every OTHER server (including Server 2, which holds Bob's connection) is subscribed and receives it — then Server 2 forwards it on to Bob through his open WebSocket. Your Hub code doesn't change at all.

📈 Without a Backplane vs With One

Adding the Redis Backplane — One Line of Config

C# — Program.cs
var builder = WebApplication.CreateBuilder(args);

builder.Services
    .AddSignalR()
    .AddStackExchangeRedis("localhost:6379", options =>
    {
        options.Configuration.ChannelPrefix = RedisChannel.Literal("MyApp");
    });

// Your Hub code stays EXACTLY the same.
// Clients.All, Clients.Group(...), Clients.Others — all now correctly
// reach connections on every server instance, not just the current one.

// NuGet package needed:
// Microsoft.AspNetCore.SignalR.StackExchangeRedis

💡 Your Hub Code Never Needs to Know Scaling Exists

Write Hub logic once, assuming a single server. Adding horizontal scale later is purely a configuration change — not a rewrite of your real-time logic. This is the whole point of the backplane abstraction.

⚠️ The Backplane Fixes Message Routing — Not Shared State

Remember that private static int _count from the counter in Tab 2? A Redis backplane does NOT fix that — each server still has its own separate memory for it. If you need actual shared application state across servers, you store it in Redis (or a database) yourself, separately from the backplane. These are two different concerns.

⚠️ Other Things to Know

  • Redis becomes a dependency — if it goes down, cross-server messaging breaks even if each server individually is fine
  • Sticky sessions matter less with a backplane, since any server can now route to any connection — but long polling fallback still benefits from routing the same session to the same server

🔄 The Trade-off

  • The backplane adds a small hop through Redis for every cross-server message. For most apps imperceptible, but not literally free — you're trading a tiny bit of latency for the ability to run multiple servers at all

🎯 Advanced Patterns

Everything so far used string-based method names — "ReceiveMessage", "UserOnline" — which work, but typo a name and you get a silent failure with no compiler warning. As your hub grows, that gets risky. Let's tighten this up, then look at streaming and authentication, which come up as soon as you're past the basics.

Strongly-Typed Hubs

🤔 Can the compiler actually catch a typo'd method name?

Yes — if you define an interface describing what methods exist on the client, and have your Hub inherit from Hub<TInterface> instead of plain Hub. Now Clients.All isn't a generic "send anything by string name" object — it's typed to that interface. Calling a method that doesn't exist, or passing wrong argument types, is a compile error instead of a silent runtime miss.

C# — Strongly-typed hub
// Define exactly what the client can be told to do
public interface IChatClient
{
    Task ReceiveMessage(string user, string message);
    Task UserOnline(string userId);
    Task UserOffline(string userId);
}

public class ChatHub : Hub<IChatClient>
{
    public async Task SendMessage(string conversationId, string message)
    {
        // Autocompletes, and a typo here is a COMPILE ERROR, not a silent miss
        await Clients.Group(conversationId)
            .ReceiveMessage(Context.UserIdentifier!, message);
    }
}

💡 Worth Adopting Once a Hub Grows Past a Few Methods

For the tiny counter example, plain SendAsync is fine. Once you have a Hub with a dozen different message types flowing both directions, strongly-typed pays for itself the first time it catches a renamed method before it ships.

Calling Hub Methods From Outside a Hub (IHubContext)

🤔 What if I want to push a notification from a background job or a controller — not from inside a Hub method?

This comes up all the time — a background worker finishes processing an order and needs to push "your order is ready!" to the right user. You can't just call Hub methods directly from outside a Hub, but ASP.NET Core's DI system gives you IHubContext<THub>, which lets any class in your app send messages to connected clients without being a Hub itself.

C# — Pushing from a background service
public class OrderProcessingService
{
    private readonly IHubContext<ChatHub, IChatClient> _hubContext;

    public OrderProcessingService(IHubContext<ChatHub, IChatClient> hubContext)
    {
        _hubContext = hubContext;
    }

    public async Task OnOrderCompletedAsync(string userId, string orderId)
    {
        // Pushes directly to all connections belonging to this userId —
        // no need to be inside a Hub method to do this
        await _hubContext.Clients
            .User(userId)
            .ReceiveMessage("system", $"Order {orderId} is ready!");
    }
}

Streaming — When the Server Has a Lot to Send

📊 The Scenario

A server-side report is generating thousands of rows. Sending them all as one giant response means the client waits for everything before seeing anything, and holds the whole result in memory at once.

C# — Streaming results to the client
public class ReportHub : Hub
{
    // Returns IAsyncEnumerable — SignalR streams each yielded value
    // to the client as it becomes available, not all at the end
    public async IAsyncEnumerable<string> StreamReportRows(
        string reportId,
        [EnumeratorCancellation] CancellationToken token)
    {
        await foreach (var row in _reportService.GetRowsAsync(reportId, token))
        {
            yield return row;  // sent to client immediately — no buffering
        }
    }
}
JavaScript — Consuming the stream
const stream = connection.stream("StreamReportRows", reportId);

stream.subscribe({
    next: (row) => appendRow(row),        // called for EACH row as it arrives
    complete: () => showMessage("Done"), // all rows received
    error: (err) => showError(err)
});

Authentication on Hubs

🤔 Does standard ASP.NET Core [Authorize] work on a Hub?

Yes, the same attribute works the same way — apply it to the Hub class or individual methods and unauthenticated connections get rejected. The one wrinkle: for WebSocket connections specifically, browsers can't attach custom Authorization headers the way a normal HTTP request can, so the typical pattern is passing the JWT token via query string instead, and telling the JWT middleware to also look there.

C# — Securing a Hub + accepting token from query string
[Authorize]
public class ChatHub : Hub<IChatClient>
{
    public async Task SendMessage(string message)
    {
        // Context.User populated from auth token, same as in a controller
        var username = Context.User?.Identity?.Name ?? "unknown";
        await Clients.All.ReceiveMessage(username, message);
    }
}

// In Program.cs — also accept the JWT from ?access_token= query string
// (needed because browsers can't set headers on WebSocket handshake)
.AddJwtBearer(options =>
{
    options.Events = new JwtBearerEvents
    {
        OnMessageReceived = ctx =>
        {
            var token = ctx.Request.Query["access_token"];
            if (!string.IsNullOrEmpty(token))
                ctx.Token = token;
            return Task.CompletedTask;
        }
    };
});
