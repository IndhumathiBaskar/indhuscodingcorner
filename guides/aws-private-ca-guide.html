🏛️ What is AWS Private CA?

AWS Private CA (formerly ACM Private CA) is a managed private certificate authority service. It lets you create your own internal PKI (Public Key Infrastructure) to issue X.509 certificates for internal services, devices, users, and applications — without managing your own CA software, HSMs, or CRL infrastructure.

💡 Private CA in One Sentence

AWS Private CA = a fully managed internal certificate authority that issues trusted X.509 certificates for your own organisation's resources (servers, services, IoT devices, users) — trusted inside your organisation but not by browsers or public internet.

Public CA vs Private CA — The Key Difference

🌍 Public CA (Let's Encrypt, DigiCert)

  • Trusted by all browsers and OS trust stores globally
  • Must prove domain ownership (DNS/HTTP validation)
  • Certificates for public-facing websites and APIs
  • Cannot issue certs for internal IPs or internal hostnames
  • Free (Let's Encrypt) or paid — but not controlled by you
  • Subject to public CA/Browser Forum rules

🏛️ Private CA (AWS Private CA)

  • Trusted only within your organisation (you control the trust store)
  • Issue certs for internal IPs, hostnames, services, devices, users
  • Full control: validity period, extensions, subject fields
  • Used for: mTLS, internal APIs, IoT devices, VPN, code signing
  • ~$400/month per CA + $0.75 per cert issued
  • No domain validation — you control who gets what cert

🔗 CA Hierarchy — Root, Subordinate & Trust

A well-designed CA hierarchy separates the high-value Root CA (kept offline or rarely used) from Subordinate CAs that do the day-to-day certificate issuance. This limits blast radius — if a Subordinate CA is compromised, you revoke it without touching the Root.

🔗 Hybrid PKI — AWS Subordinate under On-Prem Root

  • Create AWS Private CA as a Subordinate CA
  • Generate CSR from AWS Private CA → send to your on-prem Root CA to sign
  • Import the signed certificate back into AWS Private CA → activate it
  • Now AWS Subordinate CA is trusted by your existing on-prem PKI trust stores
  • Use case: enterprises with existing PKI who want cloud-native certificate issuance
  • Best of both worlds — existing trust chain + AWS managed infrastructure

🔄 Certificate Lifecycle — Issue, Renew, Revoke

Full Certificate Lifecycle

📝 Step 1 — Generate Key Pair + CSR
  • Application generates RSA/EC key pair locally (private key NEVER leaves)
  • Creates CSR with: Subject (CN, O, OU), SANs, public key
  • OR: Use ACM to manage key generation and CSR automatically
↓
Submit CSR to AWS Private CA via SDK / ACM / console
🏛️ Step 2 — AWS Private CA Signs the Certificate
  • Validates CSR structure and policy constraints
  • Applies certificate template (TLS, code signing, mTLS client, etc.)
  • Signs with Subordinate CA private key (in HSM)
  • Returns signed certificate + certificate chain (Sub CA + Root CA)
↓
Install cert + chain on server / device / service
✅ Step 3 — Certificate in Use
  • Server presents cert during TLS handshake
  • Client verifies: chain → Root CA in trust store, not expired, not revoked
  • Certificate valid until expiry (or until revoked)
↓
⏰ Step 4a — Renewal (near expiry)
  • ACM auto-renews if managed via ACM
  • Manual: generate new key pair + CSR → issue new cert
  • Old cert remains valid until expiry or revocation
❌ Step 4b — Revocation (emergency)
  • Private key compromised or no longer needed
  • Call RevokeCertificate API → reason code
  • Published to CRL + OCSP within minutes

💡 PathLen Constraint

PathLen:0 means this Subordinate CA can only issue end-entity certs — it cannot sign other CA certificates. PathLen:1 means it can sign one more level of sub CAs. Use PathLen:0 on all leaf-issuing Subordinate CAs for maximum security.

Common Certificate Types Explained

🖥️ TLS Server Certificate

  • Used by servers to prove their identity during TLS handshake
  • Subject: CN=api.internal.myorg.com
  • SAN: DNS:api.internal, DNS:*.svc.cluster.local, IP:10.0.1.5
  • EKU: TLS Web Server Authentication (1.3.6.1.5.5.7.3.1)
  • Key Usage: Digital Signature, Key Encipherment
  • Presented by server — verified by client against trust store

👤 mTLS Client Certificate

  • Used by clients (services, users, devices) to authenticate TO a server
  • Subject: CN=order-service or CN=user@myorg.com
  • EKU: TLS Web Client Authentication (1.3.6.1.5.5.7.3.2)
  • Key Usage: Digital Signature only (no Key Encipherment)
  • Presented by client — server verifies it against its trusted CA list
  • Proves who the client is — no passwords or API keys needed

📡 IoT Device Certificate

  • Unique identity certificate per IoT device
  • Subject: CN=device-serial-001
  • Private key generated on device — never transmitted
  • Device presents cert to AWS IoT Core for authentication
  • EKU: TLS Client Auth (device authenticates to cloud)
  • Longer validity (1–5 years) — devices hard to update

📦 Code Signing Certificate

  • Signs software artifacts: Lambda packages, containers, executables
  • EKU: Code Signing (1.3.6.1.5.5.7.3.3)
  • Key Usage: Digital Signature
  • Signed code verifiable even after cert expires (with timestamp)
  • Works with AWS Signer for Lambda function signing
  • Prevents deployment of unsigned/tampered code

🚫 Revocation — CRL & OCSP

Revocation is how you invalidate a certificate before its expiry — typically when a private key is compromised, an employee leaves, or a service is decommissioned. AWS Private CA supports two revocation mechanisms: CRL (batch file) and OCSP (real-time).

💡 Short-Lived Certs Eliminate Revocation

If you issue certs valid for 24 hours or less, revocation is irrelevant — the cert expires before an attacker can exploit it. For high-security microservice environments, use the Short-lived cert mode CA ($50/month) with validity <7 days. Much simpler operationally.

CRL Flow — How It Works

🔓 Private key compromised — must revoke
↓
Call RevokeCertificate API with reason code
🏛️ AWS Private CA marks cert as revoked
  • Reason codes: KEY_COMPROMISE · CA_COMPROMISE · AFFILIATION_CHANGED · SUPERSEDED · CESSATION_OF_OPERATION · PRIVILEGE_WITHDRAWN
  • New CRL generated immediately with this cert's serial number
  • CRL published to configured S3 bucket within minutes
↓
📄 CRL file on S3 (public read)
  • Clients download CRL from URL embedded in cert (CDP extension)
  • CRL is signed by CA — tamper-evident
  • Contains: list of revoked serial numbers + revocation timestamps + reason codes
  • CRL TTL: clients cache until nextUpdate field (typically 7 days)
↓
✅ Client checks CRL during TLS handshake
  • If serial number found in CRL → reject connection
  • If not found → accept (cert is valid)
  • If CRL download fails → depends on soft/hard fail policy

Revocation Reason Codes

Reason CodeWhen to Use
KEY_COMPROMISEPrivate key was exposed, stolen, or suspected stolen
CA_COMPROMISEThe CA that issued this cert was compromised
AFFILIATION_CHANGEDSubject changed role/department — cert subject no longer accurate
SUPERSEDEDCertificate replaced by a new one (normal renewal with old key)
CESSATION_OF_OPERATIONService or device decommissioned — cert no longer needed
PRIVILEGE_WITHDRAWNSubject no longer authorised to hold this cert (employee offboarding)
UNSPECIFIEDUse only when no other reason applies — avoid if possible

🔗 mTLS & Real-World Use Cases

Mutual TLS (mTLS) is the most powerful use case for AWS Private CA. Unlike standard TLS where only the server presents a certificate, in mTLS both sides present certificates — enabling cryptographic authentication without passwords or API keys.

Standard TLS vs mTLS

🔐 Standard TLS vs Mutual TLS (mTLS) — Side by Side

mTLS with ALB — Authenticate at the Load Balancer

1

Issue client cert from Private CA

Order-service gets a cert with CN=order-service, EKU: TLS Client Auth, issued by your Subordinate CA.

2

Configure ALB with trust store

Create an ALB trust store containing your Root CA cert. ALB will only allow requests from clients presenting certs signed by this CA. Optionally configure "Verify" mode (reject without cert) vs "Passthrough" mode.

3

ALB passes client identity headers downstream

After mTLS, ALB injects headers: X-Amzn-Mtls-Clientcert-Subject, X-Amzn-Mtls-Clientcert-Fingerprint — your backend reads these without any auth logic of its own.

4

Backend authorises based on CN/SAN

Read X-Amzn-Mtls-Clientcert-Subject → extract CN=order-service → check authorisation. Rejected certs never reach your backend at all.

Other Real-World Use Cases

🌐 Internal HTTPS (VPC/Kubernetes)

  • Issue certs for internal services: api.svc.cluster.local
  • Kubernetes cert-manager integration — auto-issue and renew certs
  • Eliminate self-signed cert warnings in internal dashboards
  • Enable HTTPS on internal ALBs with trusted certs
  • Works with internal IPs and RFC 1918 addresses (public CA can't do this)

📡 IoT Device Identity

  • Each device gets a unique cert during manufacturing
  • Device authenticates to AWS IoT Core with its cert
  • No shared secrets — compromise of one device doesn't affect others
  • Revoke compromised devices individually
  • Scale to millions of devices — Private CA handles volume

🛡️ SPIFFE/SVID — Service Mesh

  • Issue SPIFFE-compatible SVIDs for Istio/Linkerd
  • URI SAN: spiffe://myorg.com/ns/default/sa/order-service
  • Short-lived (1hr) certs — auto-rotated by the mesh
  • Private CA acts as the SPIRE server upstream CA
  • Zero-trust networking inside your cluster

👤 Employee / VPN Client Certs

  • Issue personal certs to employees for VPN auth
  • Client VPN (AWS) uses Private CA for client authentication
  • Employee leaves → revoke their cert → VPN access gone instantly
  • No password resets — revocation is the offboarding step
  • Works with smart cards and hardware tokens (YubiKey)

💡 RAM vs Resource Policy

Use AWS RAM (Resource Access Manager) to share a Private CA with other AWS accounts or AWS Organizations. RAM sharing is simpler than resource policies for org-wide sharing. Both RAM and resource-based policies work — RAM is preferred for org-level, resource policy for specific accounts.

CloudTrail Audit — Every CA Operation is Logged

📋 What CloudTrail Captures for Private CA

  • IssueCertificate — who issued, which CA, template, subject, validity
  • RevokeCertificate — who revoked, serial number, reason code, timestamp
  • CreateCertificateAuthority — CA creation details
  • DeleteCertificateAuthority — with 7-day safety window before permanent deletion
  • All API calls: IAM principal, source IP, AWS account, request parameters
  • Use CloudTrail Insights to detect unusual issuance volumes (bulk cert issuance attack)

Security Best Practices

  • Use two-tier hierarchy — Root CA issues only to Subordinate CAs, never to end entities directly
  • Restrict IssueCertificate to specific template ARNs via IAM conditions — prevent CA cert issuance
  • Enable CRL and OCSP on all CAs — clients need a way to check revocation
  • Store Root CA cert in AWS Secrets Manager or SSM Parameter Store for distribution to trust stores
  • Use short-lived certs (<24hr) for service mesh / SPIFFE to eliminate revocation overhead
  • Enable CloudTrail for all Private CA API calls — alert on RevokeCertificate and unusual IssueCertificate volume
  • Use separate Subordinate CAs per environment (dev/staging/prod) — compromise in dev doesn't affect prod
  • Tag all CAs and certs for cost allocation and audit — Environment=prod, Team=platform
  • Configure CloudWatch alarm on CertificateCount approaching CA limits
  • Use RAM to share CA across accounts — avoids each account needing its own $400/month CA

Private CA Deletion Safety

⚠️ Deleting a CA — Restoration Window

When you call DeleteCertificateAuthority, the CA enters a 7-30 day restoration window (you choose). During this time it's disabled but can be restored. After the window, deletion is permanent and the CA private key is destroyed. All certs issued by this CA will immediately fail validation since the CA cert is gone. Plan CA deletion carefully — revoke all issued certs first.

⌨️ C# SDK — AWS Private CA

Install: dotnet add package AWSSDK.ACMPCA

Pattern 1 — Issue a Certificate from a CSR

C# · ACMPCA SDK · Issue certificate from CSR + download 
using Amazon.ACMPCA;
using Amazon.ACMPCA.Model;
using System.Security.Cryptography;
using System.Security.Cryptography.X509Certificates;
using System.Text;

public class PrivateCaService
{
    private readonly IAmazonACMPCA _pca;
    private const string CaArn = "arn:aws:acm-pca:us-east-1:123456789:certificate-authority/CA-ID";

    public PrivateCaService(IAmazonACMPCA pca) => _pca = pca;

    // ── Step 1: Generate key pair + CSR locally ──
    public static (string csrPem, RSA privateKey) GenerateCsrAndKey(
        string commonName, string[] sanDnsNames)
    {
        var rsa     = RSA.Create(2048);
        var request = new CertificateRequest(
            $"CN={commonName},O=MyOrg,C=US",
            rsa,
            HashAlgorithmName.SHA256,
            RSASignaturePadding.Pkcs1);

        // Add Subject Alternative Names
        var sanBuilder = new SubjectAlternativeNameBuilder();
        foreach (var dns in sanDnsNames) sanBuilder.AddDnsName(dns);
        request.CertificateExtensions.Add(sanBuilder.Build());

        var csrBytes = request.CreateSigningRequest();
        var csrPem   = "-----BEGIN CERTIFICATE REQUEST-----\n"
                     + Convert.ToBase64String(csrBytes, Base64FormattingOptions.InsertLineBreaks)
                     + "\n-----END CERTIFICATE REQUEST-----";

        return (csrPem, rsa);  // Store privateKey securely — NEVER transmit it
    }

    // ── Step 2: Submit CSR to AWS Private CA ──
    public async Task<string> IssueCertificateAsync(
        string csrPem,
        int validityDays = 365)
    {
        var response = await _pca.IssueCertificateAsync(new IssueCertificateRequest
        {
            CertificateAuthorityArn = CaArn,
            Csr = new MemoryStream(Encoding.UTF8.GetBytes(csrPem)),
            // Template determines Key Usage and EKU extensions
            TemplateArn = "arn:aws:acm-pca:::template/EndEntityCertificate/V1",
            SigningAlgorithm = SigningAlgorithm.SHA256WITHRSA,
            Validity = new Validity
            {
                Type  = ValidityPeriodType.DAYS,
                Value = validityDays
            },
            // IdempotencyToken ensures same request = same cert if retried
            IdempotencyToken = Guid.NewGuid().ToString("N")
        });

        return response.CertificateArn;  // ARN to retrieve cert once issued
    }

    // ── Step 3: Wait and download the signed certificate ──
    public async Task<(string certPem, string chainPem)> GetCertificateAsync(string certArn)
    {
        while (true)
        {
            try
            {
                var resp = await _pca.GetCertificateAsync(new GetCertificateRequest
                {
                    CertificateAuthorityArn = CaArn,
                    CertificateArn          = certArn
                });
                return (resp.Certificate, resp.CertificateChain);
                // Certificate = end-entity cert PEM
                // CertificateChain = Subordinate CA + Root CA PEM (install alongside cert)
            }
            catch (RequestInProgressException)
            {
                await Task.Delay(TimeSpan.FromSeconds(2));  // issuance takes 1-5 seconds
            }
        }
    }

    // ── All-in-one: Generate key + CSR → Issue → Download ──
    public async Task<(string certPem, string chainPem, RSA privateKey)>
        ProvisionCertificateAsync(string commonName, string[] dnsNames, int validityDays = 365)
    {
        var (csrPem, privateKey) = GenerateCsrAndKey(commonName, dnsNames);
        var certArn              = await IssueCertificateAsync(csrPem, validityDays);
        var (cert, chain)        = await GetCertificateAsync(certArn);
        return (cert, chain, privateKey);
    }
}

public async Task RevokeCertificateAsync(string certArn, string reason = "KEY_COMPROMISE")
{
    // Extract serial number from cert ARN (last part after /certificate/)
    var serialNumber = certArn.Split('/').Last();

    await _pca.RevokeCertificateAsync(new RevokeCertificateRequest
    {
        CertificateAuthorityArn = CaArn,
        CertificateSerial       = serialNumber,
        RevocationReason        = reason  // KEY_COMPROMISE · CESSATION_OF_OPERATION · etc
    });

    Console.WriteLine($"✅ Revoked: {certArn} — reason: {reason}");
    // CRL updated automatically. OCSP reflects revocation within minutes.
}

Pattern 3 — List & Monitor CAs

C# · List all CAs · Describe CA · Monitor expiry 
public async Task MonitorCAsAsync()
{
    var resp = await _pca.ListCertificateAuthoritiesAsync(new ListCertificateAuthoritiesRequest());

    foreach (var ca in resp.CertificateAuthorities)
    {
        var detail = await _pca.DescribeCertificateAuthorityAsync(
            new DescribeCertificateAuthorityRequest { CertificateAuthorityArn = ca.Arn });

        var d = detail.CertificateAuthority;
        Console.WriteLine($"CA: {d.CertificateAuthorityConfiguration.Subject.CommonName}");
        Console.WriteLine($"Status: {d.Status}");          // ACTIVE · DISABLED · EXPIRED
        Console.WriteLine($"Type: {d.Type}");              // ROOT · SUBORDINATE
        Console.WriteLine($"Created: {d.CreatedAt:u}");
        Console.WriteLine($"Expiry: {d.NotAfter:u}");

        // Warn if CA expires within 90 days
        if (d.NotAfter.HasValue && d.NotAfter.Value < DateTime.UtcNow.AddDays(90))
            Console.WriteLine($"⚠️ WARNING: CA expires soon — {d.NotAfter:yyyy-MM-dd}");
    }
}
