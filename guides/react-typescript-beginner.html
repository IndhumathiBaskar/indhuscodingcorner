🏗️ What is React?

A JavaScript library for building user interfaces out of small, reusable pieces called components.

One-liner React lets you describe what the UI should look like for any given state, and React figures out how to update the actual webpage efficiently when that state changes.
🧩 Component-Based

UI is broken into small, independent, reusable pieces. A button, a card, a whole page — all components. Build complex UIs by combining simple ones.

📐 Declarative

You describe the end result ("show this list of items"), not the step-by-step DOM manipulation. React handles updating the browser for you.

🌳 Virtual DOM

React keeps a lightweight in-memory copy of the UI. When state changes, it compares old vs new and updates only what actually changed in the real browser.

Why TypeScript with React?

TypeScript = JavaScript + Types TypeScript catches mistakes while you type — wrong prop names, missing fields, calling a function with the wrong argument type. For React specifically, it makes props, state, and event handlers self-documenting: you (and your editor) always know exactly what shape of data a component expects.
Without TypeScriptWith TypeScript
Typo in a prop name → runtime crash, found by a userTypo in a prop name → red squiggle in your editor, instantly

🧩 Components & JSX

A component is just a function that returns JSX — HTML-like syntax that describes what should appear on screen.

TSX Your first component — components/Greeting/Greeting.tsx 
// A component is just a function. By convention, starts with a Capital letter.
function Greeting() {
  return (
    <div>
      <h1>Hello, Indhu! 👋</h1>
      <p>Welcome to React + TypeScript.</p>
    </div>
  );
}

export default Greeting;
TSX Using the component — App.tsx 
import Greeting from './components/Greeting/Greeting';

function App() {
  return (
    <div>
      <Greeting />   {/* Use a component like an HTML tag */}
    </div>
  );
}

export default App;

JSX Rules to Remember

1️⃣ One Root Element

A component must return a single parent element. Wrap multiple elements in a <div> or a <>...</> Fragment (no extra DOM node).

2️⃣ camelCase Attributes

HTML's class becomes className. onclick becomes onClick. Most attributes follow JS camelCase convention, not HTML's lowercase.

3️⃣ Curly Braces = JavaScript

Anything inside { } in JSX is evaluated as JavaScript. {user.name}, {2 + 2}, {isLoggedIn ? 'Hi' : 'Login'} all work.

4️⃣ Self-Closing Tags

Elements with no children must self-close: <img />, <input />, <Greeting /> — the trailing slash is required.

TSX JSX expressions and conditional rendering 
function UserCard() {
  const name = 'Indhu';
  const isOnline = true;
  const unreadCount = 3;

  return (
    <div>
      <h2>{name}</h2>                          {/* embed a variable */}

      {isOnline ? <span>🟢 Online</span> : <span>⚫ Offline</span>}  {/* ternary */}

      {unreadCount > 0 && <span>{unreadCount} new messages</span>}     {/* && shortcut */}
    </div>
  );
}

📦 Props — Passing Data Down

Props let a parent component pass data into a child component — like function arguments, but for components.

This is where TypeScript shines Define a type or interface describing exactly what props a component accepts. Your editor will autocomplete them and flag an error immediately if you forget a required one or pass the wrong type.
TSX components/UserCard/UserCard.tsx — typed props 
// 1. Define the shape of the props this component accepts
type UserCardProps = {
  name: string;
  age: number;
  isOnline: boolean;
  avatarUrl?: string;        // the ? makes this prop optional
};

// 2. Destructure props directly in the function signature
function UserCard({ name, age, isOnline, avatarUrl }: UserCardProps) {
  return (
    <div className="user-card">
      <img src={avatarUrl ?? '/default-avatar.png'} alt={name} />
      <h3>{name}, {age}</h3>
      <span>{isOnline ? '🟢 Online' : '⚫ Offline'}</span>
    </div>
  );
}

export default UserCard;
TSX Passing props from a parent component 
import UserCard from './components/UserCard/UserCard';

function App() {
  return (
    <div>
      <UserCard name="Indhu" age={28} isOnline={true} />
      <UserCard name="Alex" age={34} isOnline={false} />

      {/* ❌ TypeScript error: missing required prop "isOnline" */}
      {/*  */}
    </div>
  );
}

Props with Children

TSX A component that wraps other content — using React.ReactNode 
type PanelProps = {
  title: string;
  children: React.ReactNode;   // accepts any valid JSX as children
};

function Panel({ title, children }: PanelProps) {
  return (
    <div className="panel">
      <h3>{title}</h3>
      <div className="panel-body">{children}</div>
    </div>
  );
}

// Usage — anything between the tags becomes "children"
<Panel title="Account Settings">
  <p>Manage your preferences here.</p>
</Panel>
💡 Props flow one way — down Data only flows from parent → child via props. A child can never directly change its parent's data. To let a child trigger a change, the parent passes down a function as a prop (covered in the Events tab).

🪝 useState — Component Memory

State is data that a component "remembers" and can change over time — and when it changes, React re-renders the UI.

The Golden Rule of State Never modify state directly (count = count + 1 ❌). Always use the setter function React gives you (setCount(count + 1) ✅). Direct mutation won't trigger a re-render.
TSX components/Counter/Counter.tsx — basic useState 
import { useState } from 'react';

function Counter() {
  // useState — TypeScript infers the type from the initial value (0)
  const [count, setCount] = useState<number>(0);

  return (
    <div>
      <p>Count: {count}</p>
      <button onClick={() => setCount(count + 1)}>
        Increment
      </button>
      <button onClick={() => setCount(0)}>
        Reset
      </button>
    </div>
  );
}

useState with Objects & Arrays

TSX Typed object state — a form's data 
type FormState = {
  name: string;
  email: string;
};

function SignupForm() {
  const [form, setForm] = useState<FormState>({ name: '', email: '' });

  const updateName = (newName: string) => {
    // ✅ Spread the old object, override just the changed field
    setForm({ ...form, name: newName });
    // ❌ form.name = newName — never mutate state directly!
  };

  return (
    <input
      value={form.name}
      onChange={(e) => updateName(e.target.value)}
    />
  );
}
TSX Typed array state — a todo list 
type Todo = { id: number; text: string; done: boolean };

function TodoList() {
  const [todos, setTodos] = useState<Todo[]>([]);

  const addTodo = (text: string) => {
    const newTodo: Todo = { id: Date.now(), text, done: false };
    // ✅ Spread the old array, add the new item — never push() directly
    setTodos([...todos, newTodo]);
  };

  const toggleTodo = (id: number) => {
    setTodos(todos.map(t => t.id === id ? { ...t, done: !t.done } : t));
  };

  return (
    <ul>
      {todos.map(todo => (
        <li key={todo.id} onClick={() => toggleTodo(todo.id)}>
          {todo.done ? '✅' : '⬜'} {todo.text}
        </li>
      ))}
    </ul>
  );
}
⚠️ The "key" prop When rendering a list with .map(), every item needs a unique key prop. React uses it to track which items changed, were added, or removed — without it you'll get a console warning and potential rendering bugs.

🔄 useEffect — Side Effects

Run code in response to a component rendering, or when specific values change — fetching data, subscriptions, timers.

What counts as a "side effect"? Anything that reaches outside of React's rendering — fetching data from an API, setting up a timer, subscribing to an event, manually changing the page title. useEffect is where this code belongs.
TSX The three forms of useEffect — dependency array controls when it runs 
import { useEffect, useState } from 'react';

function Example({ userId }: { userId: number }) {
  const [count, setCount] = useState(0);

  // 1️⃣ No dependency array — runs after EVERY render (rarely what you want)
  useEffect(() => {
    console.log('Component rendered');
  });

  // 2️⃣ Empty array [] — runs ONCE, when the component first mounts
  useEffect(() => {
    console.log('Component mounted — runs once');
  }, []);

  // 3️⃣ With dependencies [userId] — runs once, AND whenever userId changes
  useEffect(() => {
    console.log(`Fetching data for user ${userId}`);
  }, [userId]);

  return <div>{count}</div>;
}

Real Example: Fetching Data

TSX pages/UserProfile/UserProfile.tsx — fetch on mount with cleanup 
import { useEffect, useState } from 'react';

type User = { id: number; name: string; email: string };

function UserProfile({ userId }: { userId: number }) {
  const [user, setUser] = useState<User | null>(null);
  const [loading, setLoading] = useState(true);

  useEffect(() => {
    let isCancelled = false;   // prevents setting state after unmount

    async function fetchUser() {
      setLoading(true);
      const response = await fetch(`/api/users/${userId}`);
      const data: User = await response.json();

      if (!isCancelled) {
        setUser(data);
        setLoading(false);
      }
    }

    fetchUser();

    // 🧹 Cleanup function — runs before the next effect, or on unmount
    return () => { isCancelled = true; };
  }, [userId]);  // re-run whenever userId changes

  if (loading) return <p>Loading...</p>;
  if (!user) return <p>User not found</p>;

  return <h2>{user.name}</h2>;
}
Dependency ArrayWhen it RunsCommon Use
(none)After every single renderRarely needed — easy to cause infinite loops
[]Once, on mount onlyInitial data fetch, subscribe to an event
[userId]On mount + whenever userId changesRe-fetch data when a prop/id changes

🎯 Events & Forms

Handling clicks, input changes, and form submissions — with properly typed event objects.

🖱️ Click Events

Use React.MouseEvent when you need the event object. Most of the time you don't — just use an arrow function.

⌨️ Input Events

Use React.ChangeEvent<HTMLInputElement> for typed access to e.target.value.

TSX Typed event handlers 
import { useState } from 'react';

function LoginForm() {
  const [email, setEmail] = useState('');
  const [password, setPassword] = useState('');

  // Typed change handler — knows e.target.value is a string
  const handleEmailChange = (e: React.ChangeEvent<HTMLInputElement>) => {
    setEmail(e.target.value);
  };

  // Typed click handler
  const handleButtonClick = (e: React.MouseEvent<HTMLButtonElement>) => {
    console.log('Button clicked at', e.clientX, e.clientY);
  };

  // Typed form submit handler — prevents default page reload
  const handleSubmit = (e: React.FormEvent<HTMLFormElement>) => {
    e.preventDefault();
    console.log('Submitting:', { email, password });
  };

  return (
    <form onSubmit={handleSubmit}>
      <input
        type="email"
        value={email}
        onChange={handleEmailChange}
        placeholder="Email"
      />
      <input
        type="password"
        value={password}
        onChange={(e) => setPassword(e.target.value)}
        placeholder="Password"
      />
      <button type="submit" onClick={handleButtonClick}>
        Log In
      </button>
    </form>
  );
}

Common Event Types Reference

EventTypeScript TypeUsed On
onClickReact.MouseEvent<HTMLButtonElement>button, div, any clickable element
onChange (input)React.ChangeEvent<HTMLInputElement>input, checkbox, radio
onChange (select)React.ChangeEvent<HTMLSelectElement>select dropdown

💡 Controlled vs Uncontrolled Inputs The examples above are "controlled" — the input's value comes from React state, and every keystroke updates that state via onChange. This is the recommended pattern: React state is always the single source of truth for what's on screen.

🏷️ TypeScript Basics for React

The core TypeScript concepts you'll use constantly when writing React components.

📝
type vs interface
Both define object shapes. interface can be extended; type is more flexible (unions, primitives). Either works for props — pick one convention and stick to it.
❓
Optional properties
name?: string — the ? means this field may be omitted entirely. Without it, TypeScript requires the field every time.
🔀
Union types
status: 'idle' | 'loading' | 'error' — restricts a value to one of a specific set of options. Far safer than a generic string.
🧬
Generics
useState<number>(0) — the <number> tells TypeScript exactly what type this piece of state holds.
🚫
null vs undefined
User | null is common for "data not loaded yet". Always check before accessing properties: if (!user) return ....
🔍
Type inference
TypeScript often figures out types automatically: useState(0) already knows it's a number — no need to always annotate explicitly.
TypeScript types/user.ts — a shared type used across multiple components 
// Define once, import everywhere — single source of truth for this shape
export type User = {
  id: number;
  name: string;
  email: string;
  role: 'admin' | 'editor' | 'viewer';   // union type — only these 3 values allowed
  avatarUrl?: string;                          // optional
};

// Usage in another file:
// import type { User } from '../types/user';
TypeScript interface vs type — both are valid for props 
// Option A: interface (can be extended later with "extends")
interface ButtonProps {
  label: string;
  onClick: () => void;
}

// Option B: type (more flexible — supports unions directly)
type ButtonProps2 = {
  label: string;
  onClick: () => void;
};

// Both work identically for component props — this guide uses "type" for consistency
💡 Don't fight the compiler — listen to it When TypeScript shows a red squiggle, it has almost always found a real bug before you ran the code. Resist the urge to use any to silence errors — that throws away the safety net TypeScript gives you.
