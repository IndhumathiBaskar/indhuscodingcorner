📦 What is Amazon ECR?

ECR = Elastic Container Registry. A fully managed Docker-compatible container image registry hosted by AWS. ECR stores, versions, and distributes your Docker and OCI container images. It integrates natively with ECS, EKS, Lambda, and CodePipeline — pulling images without extra credentials.

💡 Real-life analogy — The Library

ECR is like a secure private library for your container images. Each repository is a shelf (e.g. "my-api"). Each image tag is a specific edition of a book (e.g. v1.2.0 or latest). The image digest is the ISBN — unique and immutable regardless of what tag you give the book. AWS is the librarian — securing access, automatically archiving old editions (lifecycle policies), and inspecting books for known defects (image scanning). Your CI/CD pipeline is the publishing team pushing new editions to the shelf.

What AWS Manages — So You Don't Have To

😓 Self-managed Registry (Docker Hub / self-hosted)

  • Set up and maintain the registry server
  • Manage TLS certificates yourself
  • Handle authentication and access control manually
  • Build cleanup scripts to remove old image versions
  • Integrate vulnerability scanning tools separately
  • Handle replication to multiple regions yourself
  • Pay Docker Hub egress / pull rate limits
  • Manage storage capacity and scaling

✅ ECR — AWS handles all of this

  • Fully managed — no server to provision or patch
  • TLS encryption in transit, managed automatically
  • IAM-based auth — no separate credential system
  • Lifecycle policies auto-delete old/untagged images
  • Native vulnerability scanning (Basic + Inspector)
  • Cross-region replication with one config rule
  • Free data transfer to ECS/EKS in same region
  • Scales to any image size and pull concurrency

Repository URI Structure

🔒 Private Repository URI
  • 123456789012.dkr.ecr.ap-south-1.amazonaws.com/my-api:v1.2.0
  • Format: {account-id}.dkr.ecr.{region}.amazonaws.com/{repo-name}:{tag}
  • Account ID is baked into the registry URL — unique per account
  • Tag is optional — defaults to latest if omitted (avoid this!)
↓ vs
🌍 Public Repository URI
  • public.ecr.aws/myorg/my-api:v1.2.0
  • Format: public.ecr.aws/{alias}/{repo-name}:{tag}
  • Alias is a human-readable name you claim on ECR Public
  • Pulls work without docker login from anywhere

Tag Immutability — Why You Should Enable It

❌ Tag Immutability OFF (default)

  • Any push can overwrite an existing tag
  • docker push my-api:latest silently replaces the old image
  • Your ECS task revision 42 might pull a totally different image than revision 41
  • Rollbacks become unreliable — the old tag is gone
  • Fine for development; dangerous in production

✅ Tag Immutability ON (recommended for prod)

  • Overwriting an existing tag returns an error
  • Forces unique tags per push (git SHA, build number, semver)
  • Every deployment is traceable and reproducible
  • Rollback = just deploy the previous unique tag
  • Audit trail: which exact bytes ran at what time

⚠️ Never rely on :latest in production

The latest tag is just a convention — not a guarantee of the newest image. It can be overwritten by any push. Use semantic versioning (v1.2.3), git commit SHA (git-abc1234), or build numbers (build-482) as your image tags. Enable tag immutability to enforce this at the registry level.

🚀 Push & Pull — The Image Workflow

Every image goes through a build → tag → authenticate → push cycle to reach ECR. Pulling reverses it: authenticate → pull → run. Understanding each step prevents the most common CI/CD failures.

Push Workflow — Developer & CI/CD

🚢 Full push flow from source code to ECR
📝 Source Code

Dockerfile + application code
in your repo or CI runner

↓ docker build
🏗️ Build Image

Layers assembled by Docker daemon from Dockerfile

docker build -t my-api .
🏷️ Tag Image

Rename to include full ECR registry URI

docker tag my-api {uri}:v1.2
🔑 Authenticate

Get 12-hour token from AWS, pipe to docker login

aws ecr get-login-password | docker login
📤 Push

Layers uploaded — only changed layers are sent

docker push {uri}:v1.2

Step-by-Step Push Commands

1

Authenticate Docker to ECR

Get a 12-hour temporary password from AWS and pipe it to docker login. Run this once per CI job or before your push window expires.

bash — authenticate to private ECR
# Replace ACCOUNT_ID and REGION with your values
aws ecr get-login-password --region ap-south-1 \
  | docker login \
      --username AWS \
      --password-stdin \
      123456789012.dkr.ecr.ap-south-1.amazonaws.com
2

Build & Tag the Image

Build locally with a short name, then tag it with the full ECR URI. The tag should be a unique identifier — git SHA is a great choice.

bash — build and tag
# Build the image
docker build -t my-api .

# Tag with full ECR URI + git SHA
GIT_SHA=$(git rev-parse --short HEAD)
docker tag my-api \
  123456789012.dkr.ecr.ap-south-1.amazonaws.com/my-api:${GIT_SHA}
3

Push to ECR

Docker uploads only the layers that changed. First push of a new image sends all layers. Subsequent pushes with only app-layer changes are fast — base OS layers are cached.

bash — push to ecr
docker push \
  123456789012.dkr.ecr.ap-south-1.amazonaws.com/my-api:${GIT_SHA}

# Output shows each layer: "Layer already exists" = cached
# Only changed layers are actually uploaded

Pull Workflow — ECS / EKS / Lambda

🖥️ ECS Task / EKS Pod starts
  • Task definition specifies ECR image URI + tag or digest
  • ECS agent / kubelet needs to pull the image
↓ get auth token
🔑 IAM Role provides credentials
  • ECS Task Role / EKS Node Role calls ecr:GetAuthorizationToken
  • 12-hour token returned automatically — no manual login
  • Token scoped to the account's ECR registry
↓ pull layers
📦 ECR serves image layers
  • Only missing layers pulled — Docker cache re-used
  • Data transfer is free within same AWS region
  • Digest validates integrity — tampered image = pull failure
↓ container starts
✅ Container running
  • Image layers mounted as read-only overlay filesystem
  • Writable container layer on top for runtime changes

💡 Pull by digest, not tag, for guaranteed immutability

A tag can be moved. A digest cannot. In ECS task definitions and Kubernetes manifests, pin to the digest: 123456789012.dkr.ecr.ap-south-1.amazonaws.com/my-api@sha256:abc123.... This guarantees the exact same bytes run every time, making deployments truly reproducible and rollbacks safe.

Required IAM Permissions — Push vs Pull

📤 Push permissions (CI/CD role)

  • ecr:GetAuthorizationToken — get login token (account-level)
  • ecr:BatchCheckLayerAvailability — check which layers exist
  • ecr:InitiateLayerUpload — start a layer upload
  • ecr:UploadLayerPart — send layer bytes
  • ecr:CompleteLayerUpload — finalise the layer
  • ecr:PutImage — write the image manifest + tag

📥 Pull permissions (ECS/EKS task role)

  • ecr:GetAuthorizationToken — get login token (account-level)
  • ecr:BatchGetImage — fetch the image manifest
  • ecr:GetDownloadUrlForLayer — get presigned layer URLs
  • ecr:BatchCheckLayerAvailability — verify cached layers

Cross-Account Access — Repository Policy

To let another AWS account pull from your private ECR, attach a resource-based policy directly to the repository. This is separate from IAM identity policies.

JSON — ECR repository policy (cross-account pull)
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowCrossAccountPull",
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::987654321098:root"
      },
      "Action": [
        "ecr:BatchGetImage",
        "ecr:GetDownloadUrlForLayer",
        "ecr:BatchCheckLayerAvailability"
      ]
    }
  ]
}

⚠️ GetAuthorizationToken is account-level — not repository-level

The first IAM permission needed, ecr:GetAuthorizationToken, must be granted at the IAM policy level — it cannot be granted via a repository resource policy. The token it returns works for all repositories in that AWS account. Repository policies then control access per-repo for push/pull operations.

VPC Endpoint — Pull Without Internet

❌ Without VPC Endpoint
  • ECS/EKS must route ECR pulls through NAT Gateway or IGW
  • NAT Gateway costs: $0.045/GB data processed
  • Traffic leaves your VPC — less secure
  • NAT Gateway is a potential single point of failure
↓ solution
✅ With VPC Interface Endpoint (PrivateLink)
  • Create endpoints: com.amazonaws.{region}.ecr.api + ecr.dkr + s3
  • ECR pulls stay entirely inside the AWS network
  • No NAT Gateway needed — saves cost at scale
  • Required for ECS/EKS in private subnets with no NAT

♻️ Lifecycle Policies — Auto-Clean Old Images

Without lifecycle policies, ECR repositories grow forever. Every CI/CD build pushes a new image, and old ones accumulate — burning storage costs and cluttering the repository. Lifecycle policies define rules to automatically expire images by count, age, or tag status.

Lifecycle Policy JSON — Common Patterns

JSON — expire untagged images after 1 day + keep last 10 tagged
{
  "rules": [
    {
      "rulePriority": 1,
      "description": "Expire untagged images after 1 day",
      "selection": {
        "tagStatus": "untagged",
        "countType": "sinceImagePushed",
        "countUnit": "days",
        "countNumber": 1
      },
      "action": { "type": "expire" }
    },
    {
      "rulePriority": 2,
      "description": "Keep last 10 tagged images",
      "selection": {
        "tagStatus": "tagged",
        "tagPrefixList": ["v"],
        "countType": "imageCountMoreThan",
        "countNumber": 10
      },
      "action": { "type": "expire" }
    }
  ]
}

Common Lifecycle Rule Patterns

🔁 Count-based — keep last N

  • Keep the most recent 10 production images (v*)
  • Keep the most recent 5 release candidates (rc-*)
  • Keep the most recent 3 dev builds (dev-*)
  • Good when you don't care about age, just count
  • Rule: imageCountMoreThan + tag prefix filter

📅 Age-based — expire after N days

  • Expire all untagged images older than 1 day
  • Expire dev-* builds after 7 days
  • Expire rc-* images after 30 days
  • Good for time-bounded environments (feature branches)
  • Rule: sinceImagePushed + countUnit: days

💡 Use "Lifecycle Policy Preview" before saving

In the ECR console, click Dry run (Lifecycle Policy Preview) before saving a new rule. ECR shows you exactly which images would be deleted today. This prevents accidental deletion of images still referenced by running ECS services or EKS deployments.

✅ Automate scan result checks in CI/CD

After pushing an image, use aws ecr describe-image-scan-findings and fail the pipeline if findingSeverityCounts.CRITICAL > 0. Pair with EventBridge rules to send scan findings to your security team's Slack channel or ticketing system in real time.

💻 C# SDK — Working with ECR Programmatically

The AWS SDK for .NET lets you manage ECR repositories, retrieve auth tokens, query images, apply lifecycle policies, and read scan findings — all from C#. Install the NuGet package: AWSSDK.ECR.

Setup — ECR Client & Authentication Token

C# — setup and get ECR auth token
using Amazon.ECR;
using Amazon.ECR.Model;
using System.Text;

public class EcrService
{
    private readonly IAmazonECR _ecr;
    private readonly string _accountId;
    private readonly string _region;

    public EcrService(IAmazonECR ecr, string accountId, string region)
    {
        _ecr = ecr;
        _accountId = accountId;
        _region = region;
    }

    // ── Get 12-hour docker login token ────────────────────────
    public async Task<string> GetDockerLoginPasswordAsync()
    {
        var resp = await _ecr.GetAuthorizationTokenAsync(
            new GetAuthorizationTokenRequest());

        var token = resp.AuthorizationData[0].AuthorizationToken;
        var decoded = Encoding.UTF8.GetString(
            Convert.FromBase64String(token));

        // decoded = "AWS:{password}" — split and return password part
        return decoded.Split(':')[1];
    }

    // ── Get full ECR registry URI ──────────────────────────────
    public string GetRegistryUri()
        => $"{_accountId}.dkr.ecr.{_region}.amazonaws.com";
}

List Images & Read Scan Findings

C# — list images and check vulnerability scan results
    // ── List all images in a repository ───────────────────────
    public async Task<List<ImageDetail>> ListImagesAsync(string repoName)
    {
        var resp = await _ecr.DescribeImagesAsync(
            new DescribeImagesRequest
        {
            RepositoryName = repoName,
            Filter = new DescribeImagesFilter
            {
                TagStatus = TagStatus.TAGGED
            }
        });
        return resp.ImageDetails;
    }

    // ── Get scan findings for a specific image ─────────────────
    public async Task<ImageScanFindings> GetScanFindingsAsync(
        string repoName, string imageTag)
    {
        var resp = await _ecr.DescribeImageScanFindingsAsync(
            new DescribeImageScanFindingsRequest
        {
            RepositoryName = repoName,
            ImageId = new ImageIdentifier { ImageTag = imageTag }
        });

        return resp.ImageScanFindings;
    }

    // ── Block deployment if CRITICAL CVEs found ────────────────
    public async Task<bool> IsSafeToDeployAsync(
        string repoName, string imageTag)
    {
        var findings = await GetScanFindingsAsync(repoName, imageTag);

        findings.FindingSeverityCounts.TryGetValue(
            "CRITICAL", out var criticalCount);

        return criticalCount == 0;
    }

    // ── Delete an image by tag (cleanup) ──────────────────────
    public async Task DeleteImageAsync(string repoName, string imageTag)
    {
        await _ecr.BatchDeleteImageAsync(
            new BatchDeleteImageRequest
        {
            RepositoryName = repoName,
            ImageIds = [new ImageIdentifier { ImageTag = imageTag }]
        });
    }
}

✅ Best Practices & Common Mistakes

Repository & Tagging

  • Enable Tag Immutability on all production repositories — prevents silent overwrites
  • Never use :latest in ECS task definitions or Kubernetes manifests — pin to a versioned or git-SHA tag
  • Namespace repositories with slashes for large teams: team-payments/checkout-api
  • Use digest pinning in production deployments for true immutability: image@sha256:abc...
  • Create separate repositories per service — do not push different apps into the same repo
  • Tag every production image with both a version tag and the git SHA for traceability

Security & Access

  • Use IAM roles — never embed AWS credentials in Dockerfiles or CI environment variables
  • Separate CI/CD push role from ECS/EKS pull role — principle of least privilege
  • Enable scan-on-push for all repositories — it's free with Basic scanning
  • Set up EventBridge rule to alert on CRITICAL scan findings before they reach production
  • Use VPC Endpoints for ECR in private subnets — avoid NAT Gateway costs and keep traffic off the public internet
  • Use customer-managed KMS keys if you need audit control over who can decrypt image layers

Cost & Lifecycle

  • Always create lifecycle policies — even a simple "expire untagged after 1 day" saves significant storage over time
  • Run "Lifecycle Policy Dry Run" before saving any new policy to avoid deleting images in use
  • ECR data transfer to ECS/EKS in the same region is free — no need to cache images externally for same-region workloads
  • For multi-region workloads, use ECR Replication rather than pushing to multiple registries from CI
  • Monitor repository sizes with CloudWatch — set billing alarms if storage grows unexpectedly

Common Mistakes to Avoid

⚠️ Top ECR Mistakes

❌ Using :latest in production — every new push silently changes what runs next deployment
❌ No lifecycle policy — repository grows to thousands of images, storage bill grows monthly with no benefit
❌ Storing AWS credentials in CI environment — use IAM roles with OIDC (GitHub Actions) or instance profiles
❌ Missing ecr:GetAuthorizationToken on the IAM role — causes "no basic auth credentials" error even with correct repo permissions
❌ Not creating a VPC endpoint — private ECS tasks silently route ECR pulls through NAT Gateway, incurring data transfer costs
❌ Deleting a repository with images referenced by running tasks — ECS cannot pull images if the repository is deleted, causing failed deployments on new task starts
❌ Tag immutability off in prod — a failed CI push can overwrite a known-good image tag, corrupting your production artifacts
❌ Ignoring scan findings — a CRITICAL CVE in your base image running in production is a breach waiting to happen
