🔐 What is AWS IAM?

IAM = Identity and Access Management. The AWS service that controls WHO can do WHAT in your AWS account. Every API call in AWS goes through IAM first.

💡 Real-life analogy

IAM is like a company's access card system. Each employee (User) gets a card with specific permissions. Contractors (Roles) get temporary badges. The security rules (Policies) define which doors each card can open. The front desk (AWS) checks every card before letting anyone through.

Without IAM vs With IAM

❌ Without IAM (root only)
  • One account with full power
  • If credentials leaked — everything gone
  • Can't track who did what
  • Everyone shares the same access
  • No audit trail
→
✅ With IAM
  • Each person has their own identity
  • Least privilege — only what they need
  • Full audit trail (CloudTrail)
  • Temporary credentials for apps
  • MFA for critical operations

IAM Overview — How It All Connects

👤 Identity (Who are you?)
  • IAM User — a person with username + password
  • IAM Role — assumed by services, apps, or users temporarily
  • IAM Group — collection of users with shared policies
  • Federated Identity — SSO / external identity provider
↓
📋 Policy (What can you do?)
  • JSON document defining Allow / Deny rules
  • Specifies: Actions, Resources, Conditions
  • Attached to Users, Groups, or Roles
  • AWS evaluates all attached policies together
↓
🔒 Authorization Decision
  • Explicit Deny anywhere? → DENIED immediately
  • Explicit Allow found? → ALLOWED
  • No match? → DENIED (default deny)
↓
✅ AWS Service Action Executed
  • s3:GetObject, ec2:StartInstances, rds:CreateDBInstance...
  • Logged in CloudTrail for full audit trail

🧩 Core Components — Users, Groups, Roles

How Components Relate

👤 IAM User
  • A real person in your team
  • Has username + password
  • Optional: access keys for CLI/SDK
  • Gets policies directly or via group
👥 IAM Group
  • Collection of IAM Users
  • Attach policies to group → all users inherit
  • e.g. Developers, DBAdmins, ReadOnly
  • A user can be in multiple groups
↓
🎭 IAM Role
  • Like a user — but for services/apps
  • No password — temporary credentials
  • Assumed by EC2, ECS, Lambda, etc.
  • Also for cross-account access
📋 IAM Policy
  • JSON rules: Allow/Deny actions
  • Attached to users, groups, or roles
  • AWS managed or customer managed
  • Can be reused across identities

IAM User — Deep Dive

1

Console Access

Username + password for AWS Management Console login.
URL: https://[account-id].signin.aws.amazon.com/console
Enforce password policies — min length, complexity, rotation.

2

Programmatic Access (Access Keys)

Access Key ID + Secret Access Key for CLI/SDK use.
Best practice: Use roles instead of access keys wherever possible.
Never commit access keys to Git — rotate immediately if exposed.

3

IAM Group — Smart Team Management

Create groups like: Developers, DBAdmins, ReadOnly, DevOps
Attach policies to groups — not individual users.
New team member joins → add to group → they instantly get correct access.

⚠️ Root Account Rule

Never use the root account for day-to-day work. Create an IAM admin user immediately after creating your AWS account. Enable MFA on root. Lock away the root credentials — treat them like a nuclear launch key.

📋 IAM Policies — Who Can Do What

Policies are JSON documents that define what actions are allowed or denied on which AWS resources. They are the heart of IAM.

Policy Structure

Policy JSON structure 
{
  "Version": "2012-10-17",      // always this date string
  "Statement": [
    {
      "Sid": "AllowS3ReadOnly", // optional label
      "Effect": "Allow",         // Allow or Deny
      "Action": [             // what actions?
        "s3:GetObject",
        "s3:ListBucket"
      ],
      "Resource": [           // which resources?
        "arn:aws:s3:::my-bucket",
        "arn:aws:s3:::my-bucket/*"
      ],
      "Condition": {          // optional extra conditions
        "StringEquals": {
          "s3:prefix": "uploads/"
        }
      }
    }
  ]
}

How Policy Evaluation Works

📨 API Request Arrives
  • Identity: arn:aws:iam::123:user/indhu
  • Action: s3:DeleteObject
  • Resource: arn:aws:s3:::prod-bucket/file.pdf
↓
🔴 Check 1: Any explicit DENY?
  • Scan ALL attached policies for Deny statements
  • If ANY policy denies this action → STOP → DENIED
  • Explicit Deny ALWAYS wins — overrides everything
↓
🟠 Check 2: Any explicit ALLOW?
  • Scan all policies for matching Allow
  • Must match: Action + Resource + Conditions all together
  • If found → ALLOWED ✅
↓
🔴 Default: DENIED
  • No explicit Allow found = DENIED
  • AWS is deny-by-default — never guess-allows

Policy Types

🏢 AWS Managed Policies

  • Pre-built by AWS — ready to use
  • Examples: AmazonS3ReadOnlyAccess, AdministratorAccess
  • AWS updates them when new services launch
  • Good starting point — but often too broad

✍️ Customer Managed Policies

  • You create and manage them
  • Precise — grant exactly what's needed
  • Reusable across users, groups, roles
  • Version controlled — rollback if needed

📌 Inline Policies

  • Embedded directly in a user, group, or role
  • Deleted when the identity is deleted
  • Not reusable — use sparingly
  • Good for one-off specific permissions

🏢 Service Control Policies (SCP)

  • For AWS Organizations — account-level guardrails
  • Even admins can't exceed SCP limits
  • e.g. "No one can create resources outside us-east-1"
  • Don't grant permissions — they restrict them

Real Policy Examples

Allow ECS task to read from S3 and write to DynamoDB 
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "ReadFromS3",
      "Effect": "Allow",
      "Action": ["s3:GetObject", "s3:ListBucket"],
      "Resource": [
        "arn:aws:s3:::mycompany-prod-uploads",
        "arn:aws:s3:::mycompany-prod-uploads/*"
      ]
    },
    {
      "Sid": "WriteToDynamo",
      "Effect": "Allow",
      "Action": [
        "dynamodb:PutItem",
        "dynamodb:UpdateItem",
        "dynamodb:GetItem"
      ],
      "Resource": "arn:aws:dynamodb:ap-south-1:123:table/Orders"
    }
  ]
}

🎭 IAM Roles — The Right Way for Apps

Roles are the most important IAM concept for applications. Instead of hardcoding credentials, your app assumes a role and gets temporary credentials automatically.

How Role Assumption Works

🖥️ EC2 Instance / ECS Task starts
  • Has an IAM Role attached (e.g. my-app-role)
  • AWS automatically provides credentials via metadata service
  • URL: http://169.254.169.254/latest/meta-data/iam/security-credentials/
↓
🔑 AWS STS (Security Token Service)
  • Issues temporary credentials for the role
  • AccessKeyId + SecretAccessKey + SessionToken
  • Expires in 1–12 hours (auto-rotated!)
  • Your app SDK picks these up automatically
↓
✅ App Makes AWS API Call
  • SDK uses the temp credentials automatically
  • No hardcoded keys — ever!
  • If credentials expire, SDK auto-refreshes them
  • Credentials never stored anywhere insecure

Trust Policy — Who Can Assume This Role

Every role has two policies: a Trust Policy (who can assume it) and a Permission Policy (what they can do once they assume it).

Trust Policy — Allow EC2 to assume this role 
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Principal": {
      "Service": "ec2.amazonaws.com"
    },
    "Action": "sts:AssumeRole"
  }]
}
Trust Policy — Allow ECS tasks to assume this role 
{
  "Statement": [{
    "Effect": "Allow",
    "Principal": {
      "Service": "ecs-tasks.amazonaws.com"
    },
    "Action": "sts:AssumeRole"
  }]
}

Common Role Types

🖥️ EC2 Instance Role

  • Attached to EC2 instance profile
  • App code uses SDK — credentials auto-provided
  • Rotate automatically every few hours
  • No access keys needed — ever

🐳 ECS Task Role

  • Per-task permissions (not per host)
  • Different tasks can have different roles
  • Defined in task definition
  • Best practice for microservices

λ Lambda Execution Role

  • Lambda needs this to run at all
  • Must include CloudWatch Logs permission
  • Add permissions for what Lambda accesses
  • One role per Lambda function

🔁 Cross-Account Role

  • Account A assumes role in Account B
  • Trust policy allows Account A's ARN
  • Add External ID for 3rd-party security
  • Used for auditors, partners, CI/CD

🔒 How IAM Authentication Works End to End

Human User — Console Login Flow

👤 Developer opens AWS Console
  • Goes to: https://[account].signin.aws.amazon.com/console
  • Enters: IAM username + password
  • If MFA enabled: enters 6-digit code from app
↓
🔐 AWS IAM Validates Credentials
  • Checks username + password hash
  • Verifies MFA token (if enabled)
  • Creates a session token (valid 12 hours)
↓
✅ Console Access Granted
  • Developer sees only services their policies allow
  • Every action they take is checked against IAM policies
  • Every action is logged in CloudTrail

Application / Service — SDK Auth Flow

🖥️ ECS Task starts with IAM Role attached
  • Role: my-api-task-role
  • Permissions: read S3, write DynamoDB
↓
🔑 AWS SDK requests credentials from metadata
  • SDK auto-calls: http://169.254.169.254/... (ECS credential endpoint)
  • Gets: AccessKeyId + SecretKey + SessionToken + Expiry
  • Caches them in memory until near expiry
↓
📤 App calls AWS API (e.g. S3 GetObject)
  • SDK signs the request with temp credentials (SigV4 signing)
  • Sends signed HTTP request to AWS
↓
✅ AWS validates + executes
  • Verifies signature using credentials
  • Checks IAM policies allow this action
  • Executes S3 GetObject → returns file
  • Logs to CloudTrail

Credential Provider Chain (SDK Order)

The AWS SDK checks for credentials in this order — first match wins:

1

Environment Variables

AWS_ACCESS_KEY_ID, AWS_SECRET_ACCESS_KEY, AWS_SESSION_TOKEN

2

AWS Credentials File

~/.aws/credentials — set by aws configure command on local machine

3

AWS Config File

~/.aws/config — profile-based configuration

4

EC2/ECS Instance Metadata (Roles) ← Production

Automatically provided when an IAM Role is attached. No configuration needed. This is the correct approach in production.

🛡️ MFA & Account Security

Multi-Factor Authentication (MFA) adds a second layer of security beyond username and password. Even if your password is stolen, without the MFA device the attacker cannot log in.

MFA Login Flow

👤 User enters username + password
  • First factor — something you KNOW
  • Password submitted to AWS
↓
📱 AWS prompts for MFA code
  • User opens Authenticator app (Google Auth, Authy)
  • Gets a 6-digit time-based code (changes every 30s)
  • Second factor — something you HAVE
↓
✅ Access Granted
  • Both factors validated successfully
  • Even if password leaked → still protected
  • Attacker would need your physical device too

MFA Device Types

📱 Virtual MFA (Most Common)

  • App on your phone: Google Authenticator, Authy, Microsoft Authenticator
  • Generates 6-digit TOTP code every 30 seconds
  • Free — no hardware needed
  • Recommended for all IAM users

🔑 Hardware MFA (FIDO/U2F)

  • Physical USB key: YubiKey
  • Tap the key to authenticate
  • Phishing-resistant — best security
  • Recommended for root and admin accounts

Enforce MFA with Policy Condition

Deny all actions if MFA not used 
{
  "Version": "2012-10-17",
  "Statement": [{
    "Sid": "DenyWithoutMFA",
    "Effect": "Deny",
    "Action": "*",
    "Resource": "*",
    "Condition": {
      "BoolIfExists": {
        "aws:MultiFactorAuthPresent": "false"
      }
    }
  }]
}

Access Key Security

✅ Access Key Best Practices

  • Never commit keys to Git — ever
  • Rotate keys every 90 days
  • Use IAM roles instead wherever possible
  • Delete unused keys immediately
  • Use separate keys per environment
  • Enable GuardDuty to detect key misuse

🚨 If Keys Are Exposed

  • Deactivate the key IMMEDIATELY
  • Check CloudTrail for what was accessed
  • Rotate all affected credentials
  • Review resources for unauthorized changes
  • Delete the compromised key
  • Create new key from scratch

IAM Service — Create Users, Roles, Policies

IamService.cs 
using Amazon.IdentityManagement;
using Amazon.IdentityManagement.Model;

public class IamService
{
    private readonly IAmazonIdentityManagementService _iam;

    public IamService(IAmazonIdentityManagementService iam)
        => _iam = iam;

    // ── Create IAM User ───────────────────────────
    public async Task<string> CreateUserAsync(string userName)
    {
        var response = await _iam.CreateUserAsync(
            new CreateUserRequest { UserName = userName });
        return response.User.Arn;
    }

    // ── Attach Policy to User ─────────────────────
    public async Task AttachPolicyToUserAsync(
        string userName, string policyArn)
    {
        await _iam.AttachUserPolicyAsync(
            new AttachUserPolicyRequest
        {
            UserName  = userName,
            PolicyArn = policyArn
            // e.g. "arn:aws:iam::aws:policy/AmazonS3ReadOnlyAccess"
        });
    }

    // ── Create IAM Role ───────────────────────────
    public async Task<string> CreateRoleAsync(
        string roleName, string trustPolicyJson)
    {
        var response = await _iam.CreateRoleAsync(
            new CreateRoleRequest
        {
            RoleName                 = roleName,
            AssumeRolePolicyDocument = trustPolicyJson
        });
        return response.Role.Arn;
    }

    // ── Create Custom Policy ──────────────────────
    public async Task<string> CreatePolicyAsync(
        string policyName, string policyJson)
    {
        var response = await _iam.CreatePolicyAsync(
            new CreatePolicyRequest
        {
            PolicyName     = policyName,
            PolicyDocument = policyJson,
            Description    = "Custom policy for my application"
        });
        return response.Policy.Arn;
    }

    // ── Add User to Group ─────────────────────────
    public async Task AddUserToGroupAsync(
        string userName, string groupName)
    {
        await _iam.AddUserToGroupAsync(
            new AddUserToGroupRequest
        {
            UserName  = userName,
            GroupName = groupName
        });
    }

    // ── List all Users ────────────────────────────
    public async Task<List<string>> ListUsersAsync()
    {
        var response = await _iam.ListUsersAsync(
            new ListUsersRequest());
        return response.Users
            .Select(u => u.UserName)
            .ToList();
    }
}

Assume a Role (STS — Cross-Account)

StsService.cs 
using Amazon.SecurityToken;
using Amazon.SecurityToken.Model;

public class StsService
{
    private readonly IAmazonSecurityTokenService _sts;

    public StsService(IAmazonSecurityTokenService sts)
        => _sts = sts;

    // Assume a role and get temp credentials
    public async Task<Credentials> AssumeRoleAsync(
        string roleArn, string sessionName)
    {
        var response = await _sts.AssumeRoleAsync(
            new AssumeRoleRequest
        {
            RoleArn         = roleArn,
            RoleSessionName = sessionName,
            DurationSeconds = 3600 // 1 hour
        });
        return response.Credentials;
        // Returns: AccessKeyId, SecretAccessKey,
        //          SessionToken, Expiration
    }
}

✅ Best Practices & Common Mistakes

Users & Access

  • Never use the root account for daily work — create IAM admin user immediately
  • Enable MFA on root account AND all IAM users with console access
  • One IAM user per person — never share credentials
  • Use IAM Groups to manage permissions — not individual user policies
  • Delete or deactivate unused IAM users promptly
  • Review IAM users and access keys quarterly

Roles & Applications

  • Always use IAM Roles for EC2, ECS, Lambda — never access keys in app code
  • Use separate roles per service — don't share one role for all apps
  • Use ECS Task Role (not Execution Role) for app permissions to AWS services
  • Set the shortest practical session duration on roles
  • Use resource-based policies alongside IAM for extra security

Policies — Least Privilege

  • Start with minimal permissions — add more only when actually needed
  • Specify exact resource ARNs — avoid using wildcard "*" for resources
  • Never use AdministratorAccess for application roles
  • Use AWS managed policies as starting point — refine to custom
  • Use IAM Access Analyzer to find over-permissive policies
  • Test policies with IAM Policy Simulator before deploying

Access Keys

  • Never hardcode access keys in code — use roles or environment variables
  • Never commit access keys to Git — use git-secrets or pre-commit hooks
  • Rotate access keys every 90 days
  • Delete unused access keys immediately — not just deactivate
  • Use AWS Secrets Manager for any credentials your app needs

Common Mistakes to Avoid

⚠️ Top IAM Mistakes

❌ Using root account for daily work — one breach = everything lost
❌ Hardcoding access keys in application code — use IAM roles
❌ Committing AWS credentials to Git — GitHub scans for this, attackers do too
❌ Wildcard resources in policies — "Resource": "*" is too broad
❌ Sharing IAM users between people — no individual accountability
❌ Not enabling MFA — password leaks are common
❌ Not rotating access keys — old keys increase breach window
❌ One role for all microservices — blast radius too large if compromised
