🏗️ What Is System Design?

👶 Start Here — What Is System Design?

System design is the process of planning how a large software system will work at scale — before you write code. It answers questions like: How will this handle 1 million users? What happens when a server crashes? How do we keep data consistent across multiple databases? Where do we store files? System design interviews test whether you can reason about these tradeoffs — there's rarely one "correct" answer, only better and worse tradeoffs for a given situation.

Latency vs Throughput — Two Key Measurements

⏱️ Latency

The time it takes for ONE request to complete — from sending the request to receiving the response. Measured in milliseconds (ms).

  • "This API responds in 50ms"
  • Lower is better
  • Affected by: network distance, processing time, queueing

🚀 Throughput

How many requests a system can handle in a given time period. Measured in requests per second (RPS) or transactions per second (TPS).

  • "This server handles 10,000 RPS"
  • Higher is better
  • Affected by: parallelism, resource limits, bottlenecks

💡 They're Not the Same Thing

A system can have low latency but low throughput (fast for one user, but can't handle many at once) or high throughput but high latency (handles lots of requests, but each one is slow). Good system design optimises both, but sometimes you trade one for the other — e.g., batching requests increases throughput but adds latency per individual request.

📈 Scalability — Vertical vs Horizontal

👶 In Plain English

Scalability is your system's ability to handle more load — more users, more data, more requests — by adding resources. There are two fundamental ways to scale: Vertical Scaling (make one server bigger/more powerful) and Horizontal Scaling (add more servers). Most large systems use horizontal scaling because it has no upper limit and provides redundancy.

💡 Real World Analogy

A restaurant kitchen is overwhelmed with orders. Vertical scaling = hire a faster chef with better equipment (one super-chef does everything quicker). Horizontal scaling = hire 5 more chefs and split the orders between them. The super-chef approach has a limit — there's only so fast one person can work, and if they get sick, the kitchen stops. With 5 chefs, if one gets sick, the other 4 keep going.

Stateless vs Stateful — Why It Matters for Scaling

🔄 Stateless Services

Each request contains everything needed to process it. The server doesn't remember anything between requests.

  • Any server can handle any request
  • Easy to horizontally scale — just add more identical servers
  • Example: REST APIs using JWT tokens (no server-side session)

📌 Stateful Services

The server remembers information about the client between requests (session data stored in server memory).

  • Same client must hit the same server ("sticky sessions")
  • Harder to scale — server failure loses session data
  • Solution: move state to external store (Redis) → becomes stateless

💡 The Golden Rule

Design your application servers to be stateless. Store session data, user state, and shopping carts in an external store like Redis instead of server memory. This means any server can handle any request, making horizontal scaling trivial — just add more identical servers behind a load balancer.

⚖️ Load Balancing

👶 In Plain English

A Load Balancer sits in front of multiple servers and distributes incoming requests across them. Clients only know the load balancer's address — they don't know or care which actual server handles their request. This enables horizontal scaling, provides redundancy (if one server dies, traffic goes to the others), and can do health checks to avoid sending traffic to unhealthy servers.

💡 Real World Analogy

A supermarket queue manager. Instead of customers picking a checkout lane themselves (some queues get long while others are empty), a staff member directs each customer to the next available till. If a till closes (breaks down), the manager stops sending customers there. Customers don't need to know which till is "best" — they just follow directions.

Load Balancing Algorithms

🔁 Round Robin

Requests distributed sequentially: Server 1, Server 2, Server 3, Server 1, Server 2... Simple and fair if all servers are equal capacity.

⚖️ Weighted Round Robin

Like round robin but servers with more capacity get proportionally more requests. Server A (weight 3) gets 3 requests for every 1 to Server B (weight 1).

📊 Least Connections

Send the next request to whichever server currently has the fewest active connections. Good when requests take varying amounts of time.

🔑 IP Hash / Consistent Hashing

Hash the client's IP to consistently route the same client to the same server. Useful for sticky sessions without external session storage.

Load Balancer in the Request Flow

⚖️ Load Balancer with Health Checks

Health Checks

🏥 How Health Checks Work

The load balancer periodically sends a request (e.g., GET /health) to each server. If a server responds successfully within a timeout, it's marked healthy and receives traffic. If it fails repeatedly, it's marked unhealthy and removed from rotation — until it recovers.

  • Health endpoint — typically checks DB connectivity, disk space, dependencies
  • Interval — how often to check (e.g., every 10 seconds)
  • Threshold — how many failures before marking unhealthy (avoids false positives from one slow response)

⚡ Caching

👶 In Plain English

Caching means storing a copy of frequently accessed data in a fast storage layer (usually in-memory, like Redis) so future requests for that data don't need to repeat expensive work — like querying a database or calling an external API. Cached data is faster but might be slightly outdated (stale).

💡 Real World Analogy

You keep frequently used spices on your kitchen counter (cache) instead of walking to the pantry (database) every time you cook. It's much faster to grab salt from the counter. But if someone restocks the pantry with a new spice brand, your counter copy might be "stale" until you refill it. Most-used items stay on the counter; rarely-used items stay in the pantry.

Where Caching Happens

🌍

CDN / Edge

Static assets cached close to users — images, CSS, JS, videos.

🖥️

Browser Cache

Browser stores responses locally based on Cache-Control headers.

⚖️

API / App Cache

Redis/Memcached stores computed results, session data, API responses.

🗄️

Database Cache

Query result caching, buffer pools — built into most databases.

Cache-Aside Pattern — The Most Common Strategy

⚡ Cache-Aside — Application Manages the Cache

Cache Eviction Policies

⏰ TTL (Time To Live)

Each cached item expires after a set time (e.g., 5 minutes). Simple and predictable — guarantees data isn't stale for longer than the TTL.

📉 LRU (Least Recently Used)

When the cache is full, remove the item that hasn't been accessed for the longest time. Most common eviction policy — keeps "hot" data in cache.

📊 LFU (Least Frequently Used)

Remove the item that has been accessed the fewest times. Good when access patterns are stable over time.

🔄 Write-Through / Write-Behind

Write-through: write to cache AND database simultaneously. Write-behind: write to cache immediately, database updated asynchronously later.

Cache Invalidation — "The Hardest Problem"

⚠️ There Are Only Two Hard Problems in Computer Science

Cache invalidation, naming things, and off-by-one errors. When underlying data changes, the cache becomes stale. Strategies: (1) short TTLs accept some staleness, (2) explicitly delete/update cache keys when data changes, (3) versioned cache keys (e.g., product:123:v2) so old versions naturally expire. Choose based on how critical freshness is for that data.

🗄️ Databases — SQL vs NoSQL, Replication, Sharding

👶 In Plain English

Databases store your application's data. The big choices are: SQL vs NoSQL (structured tables with relationships vs flexible documents/key-value), Replication (copies of the database for read scaling and redundancy), and Sharding (splitting data across multiple databases for write scaling). Each choice has real tradeoffs — there's no universally "best" database.

SQL vs NoSQL

AspectSQL (Relational)NoSQL
StructureTables with fixed schema, rows and columnsDocuments, key-value, graphs, wide-column — flexible schema
RelationshipsForeign keys, JOINs across tablesUsually denormalised — related data embedded together
ConsistencyACID transactions — strong consistencyOften eventual consistency (tunable in some)

Query languageSQL — powerful ad-hoc queries, JOINsVaries — often simpler key-based lookups
Best forFinancial data, complex relationships, transactionsLarge scale, flexible schema, high write throughput
ExamplesPostgreSQL, MySQL, SQL ServerMongoDB (document), Redis (key-value), Cassandra (wide-column), Neo4j (graph)

💡 It's Not Either/Or

Most real systems use multiple databases for different purposes — "polyglot persistence". E.g., PostgreSQL for orders (need ACID transactions), Redis for sessions/cache (need speed), Elasticsearch for search (need full-text search), MongoDB for product catalogues (flexible schema per category).

Database Replication

🗄️ Primary-Replica Replication — Scale Reads

Sharding — Splitting Data Across Databases

🔀 What Is Sharding?

Replication copies the WHOLE database to multiple servers (helps with reads). Sharding splits the database HORIZONTALLY — each shard holds a different subset of rows, e.g., users A-M on Shard 1, users N-Z on Shard 2. This scales WRITES, since each shard handles only a portion of the data.

🔑 Sharding Strategies

  • Range-based: Shard by ID ranges (1-1000 → Shard A, 1001-2000 → Shard B). Simple but can cause hotspots.
  • Hash-based: Hash(user_id) % num_shards. Even distribution but hard to add shards later.
  • Geographic: Shard by region (EU users → EU database, US users → US database). Helps with latency and compliance (GDPR).
  • Directory-based: A lookup table maps each key to its shard. Flexible but adds a lookup hop.

⚠️ Sharding Challenges

  • Cross-shard JOINs are slow or impossible — denormalise data instead
  • Re-sharding (adding shards later) requires moving lots of data
  • Hotspots — uneven distribution if one shard gets disproportionate traffic (e.g., a celebrity user)
  • Transactions across shards are complex — avoid if possible

💡 Replication vs Sharding — Use Both

They solve different problems and are often combined: shard the database to scale writes across multiple primaries, then replicate each shard for read scaling and redundancy. This is how massive systems like Twitter and Instagram handle billions of rows.

⚖️ CAP Theorem

👶 In Plain English

CAP Theorem says that in a distributed system (data spread across multiple machines), when a network partition happens (machines can't talk to each other), you must choose between Consistency (every read gets the latest write) and Availability (every request gets a response, even if not the latest data). You can't have perfect both during a partition — Partition tolerance is essentially mandatory for distributed systems, so the real choice is C vs A.

The Three Properties

✅

Consistency (C)

Every read receives the most recent write or an error. All nodes see the same data at the same time.

🟢

Availability (A)

Every request receives a response (success or failure), without guarantee it's the latest data.

🔌

Partition Tolerance (P)

The system continues operating despite network failures between nodes.

💡 Real World Analogy

Two bank branches (Node A and Node B) normally sync balances instantly. The phone line between them goes down (network partition). Someone withdraws £100 from Branch A. Branch B doesn't know yet. If Branch B refuses all transactions until the line is restored = Consistent but not Available. If Branch B allows transactions using its old balance = Available but possibly inConsistent (someone could overdraw).

CP vs AP — The Real-World Choice

⚖️ During a Network Partition — CP vs AP

When to Choose CP vs AP

ScenarioChoiceWhy
Bank account balancesCPShowing wrong balance could cause overdrafts — better to show error than wrong data
Social media like countsAPShowing 999 likes instead of 1000 for a moment doesn't matter — must stay available
Inventory stock levelsDependsOverselling is bad (CP-leaning) but showing "unavailable" loses sales (AP-leaning) — often hybrid
Shopping cartAPUsers must always be able to add to cart — resolve conflicts later
Configuration / feature flagsAPSlightly stale config is fine; system must stay running

💡 PACELC — A More Practical Extension

CAP only describes behaviour DURING a partition. PACELC adds: even when there's No partition, you still trade off Latency vs Consistency (waiting for all replicas to confirm a write = more consistent but slower). Most real systems spend most of their time NOT partitioned, so this day-to-day latency/consistency tradeoff often matters more in practice.

📨 Message Queues

👶 In Plain English

A Message Queue lets one part of your system (Producer) send a message and move on, while another part (Consumer) processes it whenever it's ready. The queue sits in between, storing messages until they're consumed. This decouples services — the producer doesn't wait for the consumer, and if the consumer is temporarily down, messages just wait in the queue.

💡 Real World Analogy

A restaurant order ticket rail. The waiter (Producer) clips an order ticket onto the rail and immediately goes to serve other tables — they don't stand and wait for the food to be cooked. The chef (Consumer) picks up tickets from the rail when ready and cooks them in order. If the chef is briefly overwhelmed, tickets just wait on the rail — nothing is lost, and the waiter keeps working.

Why Use a Message Queue?

❌ Without a Queue (Synchronous)
  • API waits for email service to finish before responding
  • If email service is slow, the whole API request is slow
  • If email service is down, the API request fails entirely
  • Traffic spikes overwhelm downstream services directly
✅ With a Queue (Asynchronous)
  • API publishes a message and responds immediately
  • Email service processes messages at its own pace
  • If email service is down, messages wait safely in the queue
  • Queue absorbs traffic spikes — consumers catch up over time

Queue vs Topic (Pub/Sub)

AspectQueue (Point-to-Point)Topic (Pub/Sub)
ConsumersOne message → ONE consumer processes itOne message → ALL subscribers receive a copy

Message Queue Concepts

🔄

At-Least-Once Delivery

Message may be delivered more than once (e.g., consumer crashes after processing but before acknowledging). Consumers must be idempotent.

☠️

Dead Letter Queue

Messages that repeatedly fail processing are moved to a separate queue for investigation, instead of blocking the main queue forever.

👁️

Visibility Timeout

When a consumer picks up a message, it becomes invisible to other consumers for a period. If not acknowledged in time, it becomes visible again for retry.

📶

Backpressure

When consumers can't keep up, the queue grows. Monitoring queue depth helps you scale consumers before it becomes a problem.

🔢

Ordering

Most queues don't guarantee global order. FIFO queues (e.g., SQS FIFO) preserve order but typically have lower throughput.

🔁

Idempotency

Processing the same message twice should have the same effect as processing it once — critical for at-least-once delivery.

🌍 CDN — Content Delivery Network

👶 In Plain English

A CDN is a network of servers spread around the world that cache and serve static content (images, videos, CSS, JS) from a location physically close to the user. Instead of every user fetching a video from one server in London, users in Tokyo get it from a server in Tokyo — dramatically reducing latency and load on your origin server.

💡 Real World Analogy

A global pizza chain with local franchises vs one central kitchen shipping pizzas worldwide. If everyone ordered from one kitchen in Italy, delivery to Australia would take days and the kitchen would be overwhelmed. Instead, local franchises (CDN edge servers) bake using the same recipe (cached content) — Australians get pizza from Australia, fast. The central kitchen (origin server) just needs to occasionally update the recipe.

💡 Cache Headers Control CDN Behaviour

The Cache-Control HTTP header tells the CDN (and browsers) how long to cache content: Cache-Control: max-age=86400 caches for 1 day. For content that changes, use cache-busting (e.g., app.v2.js or content-hashed filenames like app.a3f9c1.js) so updates are immediately visible without waiting for cache expiry.

🚦 Rate Limiting

👶 In Plain English

Rate limiting restricts how many requests a client can make in a given time window — e.g., "100 requests per minute per user". This protects your system from being overwhelmed (accidentally by buggy clients, or deliberately by attacks) and ensures fair usage across all clients. When the limit is exceeded, the server responds with 429 Too Many Requests.

💡 Real World Analogy

A theme park ride with a "fast pass" system. Each person can use the fast pass a limited number of times per day. If you try to use it too many times too quickly, the gate simply won't let you through — "come back later". This prevents one person from monopolising the ride and ensures everyone gets a fair chance, while the ride itself doesn't get overwhelmed beyond its capacity.

Where to Rate Limit

🚪

API Gateway

Centralised — applies to all services, simplest to manage globally.

⚖️

Load Balancer

Early rejection before traffic even reaches application servers.

🖥️

Application Level

Per-endpoint limits — e.g., login endpoint stricter than read endpoints.

What to Key Limits By

  • Per user / API key — most common for authenticated APIs, fair usage per account
  • Per IP address — protects against anonymous abuse, but shared IPs (NAT, corporate networks) can over-restrict legitimate users
  • Per endpoint — expensive endpoints (search, reports) get stricter limits than cheap ones (health checks)
  • Global — overall system-wide limit as a last-resort circuit breaker

💡 Always Communicate Limits to Clients

Return headers like X-RateLimit-Limit, X-RateLimit-Remaining, and X-RateLimit-Reset so well-behaved clients can self-throttle before hitting 429s. This improves the developer experience for anyone building on top of your API.
