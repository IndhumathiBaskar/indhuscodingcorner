🔒 What is AWS Secrets Manager?

Secrets Manager is an AWS service that stores, retrieves, and automatically rotates sensitive credentials — database passwords, API keys, OAuth tokens, TLS certificates — so your code never needs to hardcode them.

💡 Real-life analogy

Think of Secrets Manager as a bank vault for your application's passwords. Instead of writing the vault combination on a sticky note (hardcoding), you call the bank (Secrets Manager), prove your identity (IAM), and the bank tells you the combination. If the combination needs changing, the bank handles it automatically — and tells everyone who needs it.

The Problem It Solves

😱 Without Secrets Manager

  • DB password in appsettings.json
  • API key committed to Git
  • Same password in dev + prod
  • Password never rotated (too risky)
  • Anyone with code access has secrets
  • Breach = unlimited access forever
  • Audit: who used the secret? No idea.

✅ With Secrets Manager

  • Code only stores the secret NAME
  • Nothing sensitive in Git — ever
  • Different secrets per environment
  • Auto-rotation every 30/60/90 days
  • IAM controls who can read which secret
  • Breach = credentials expire soon anyway
  • Full CloudTrail audit: who accessed what, when

⚙️ How Secrets Manager Works

Storing & Retrieving a Secret

👤 You / DevOps (one-time setup)
  • Create a secret in Secrets Manager console or CLI
  • Name: myapp/prod/database
  • Value: JSON with host, username, password
  • This is done ONCE — never touches code again
↓
🔒 Secrets Manager stores it encrypted
  • Encrypts with KMS key (your CMK or aws/secretsmanager)
  • Stored in AWS-managed, redundant storage
  • Versioned — every change creates a new version
  • Resource policy controls who can access
↓ When your app starts…
🚀 App needs the database password
  • Calls: GetSecretValue("myapp/prod/database")
  • IAM role attached to ECS task must allow secretsmanager:GetSecretValue
  • No password in config file — just the secret name
↓
🔒 Secrets Manager validates + decrypts
  • Checks caller's IAM identity and role
  • Checks resource policy on the secret
  • Decrypts using KMS
  • Logs the access to CloudTrail
  • Returns the secret value
↓
✅ App receives the secret — in memory only
  • JSON parsed: { host, username, password }
  • Used to open DB connection
  • Never written to disk or logs
  • SDK caches it in memory to reduce API calls

How Secrets Are Protected Inside

🔑 KMS Encryption
  • Every secret encrypted with AES-256
  • Key stored in KMS (never alongside data)
  • Default: aws/secretsmanager key (free)
  • Custom: your own CMK ($1/month)
  • Envelope encryption under the hood
📋 Versioning
  • Every secret has versions
  • AWSCURRENT = active version
  • AWSPREVIOUS = last version
  • AWSPENDING = during rotation
  • Custom labels possible

Use Secrets Manager for anything sensitive that needs rotation: DB passwords, API keys, tokens. Use Parameter Store for non-secret configuration: environment names, feature flags, URLs, non-sensitive settings.

Naming Convention — Best Practice

📂 Hierarchical Naming

Use / to create logical groupings (like folders). Makes it easy to grant IAM access to a whole namespace.

myapp/dev/database — dev DB credentials
myapp/prod/database — prod DB credentials
myapp/prod/stripe — prod Stripe keys
myapp/prod/sendgrid — prod email keys
shared/prod/datadog-api-key — shared across apps

IAM condition: "secretsmanager:SecretId": "myapp/prod/*" — grants access to all prod secrets for myapp only.

🔄 Automatic Secret Rotation

Auto rotation is the killer feature of Secrets Manager. It changes your secrets on a schedule — automatically updating both the stored secret AND the system it authenticates with (like RDS). Your app never notices the change.

Rotation Lifecycle — What Happens During Rotation

Step 1 — createSecret
Lambda creates new credentials
Rotation Lambda generates a new random password. Creates a new version in Secrets Manager with label AWSPENDING. Old password (AWSCURRENT) still works.
Step 2 — setSecret
Lambda updates the actual system
Lambda calls the RDS API (or calls your DB) to set the new password on the database user. Now both old and new passwords work in the DB simultaneously.
Step 3 — testSecret
Lambda verifies the new secret works
Lambda attempts to connect to the database using the AWSPENDING version. If it fails — rotation aborts, old secret stays AWSCURRENT. Safety net!
Step 4 — finishSecret
Secrets Manager promotes the new version
AWSPENDING → AWSCURRENT. Old AWSCURRENT → AWSPREVIOUS. Your app fetches the new secret on next retrieval. Zero downtime — the old password is still valid briefly.

What Your App Experiences During Rotation

🚀 App has cached the old secret
  • Connected to DB using version v5 (AWSCURRENT)
  • Working fine — no issues
↓ Rotation completes at 2am
🔄 Secret rotated — v6 is now AWSCURRENT
  • v5 is now AWSPREVIOUS (still valid in DB briefly)
  • App's cached v5 still works — DB allows both temporarily
  • AWS SDK cache TTL: default 5 minutes
↓ App refreshes cache
✅ App seamlessly uses v6
  • SDK fetches AWSCURRENT → gets v6 automatically
  • New DB connection uses v6 password
  • Zero downtime — no restart needed
  • Zero code changes — rotation is fully transparent

Built-in vs Custom Rotation

🔧 Built-in Rotation (AWS managed Lambda)

  • RDS MySQL, PostgreSQL, Oracle, SQLServer
  • Amazon Redshift
  • Amazon DocumentDB
  • AWS provides the Lambda function
  • Configure and forget — zero code needed

⚙️ Custom Rotation (your Lambda)

  • Any 3rd party API key (Stripe, Twilio)
  • Any database not on the AWS list
  • Custom token refresh logic
  • You write a Lambda with 4 functions
  • Template available in AWS Lambda console

💡 Rotation schedule options

Set rotation every: 30 days (high security), 60 days (standard), 90 days (compliance minimum), or a custom cron expression like cron(0 3 1 * ? *) — 3am on the 1st of every month. Stagger rotation times across services to avoid simultaneous rotations.

🔐 Access Control — Who Can Read What

Secrets Manager uses two layers of access control: IAM policies (identity-based) and resource policies (secret-based). Both must allow the operation.

Access Evaluation Flow

📨 ECS Task calls GetSecretValue
  • Identity: arn:aws:iam::123:role/my-api-task-role
  • Secret: myapp/prod/database
↓
🔍 Check 1: IAM Policy on the Role
  • Does the role policy allow secretsmanager:GetSecretValue?
  • On this specific secret ARN or name prefix?
  • If NO → DENIED immediately
↓
🔍 Check 2: Resource Policy on the Secret
  • Does the secret's resource policy allow this role?
  • If no resource policy exists → IAM alone decides
  • If resource policy exists → both must allow
  • Cross-account access requires resource policy
↓
🔍 Check 3: KMS Key Policy
  • Does the KMS key policy allow this role to decrypt?
  • Needed if using custom CMK (not aws/secretsmanager)
  • Role needs: kms:Decrypt on the key
↓
✅ All checks passed — secret returned
  • Decrypted secret value returned to caller
  • Access logged to CloudTrail

IAM Policy Examples

Allow app role to read specific secrets (least privilege) 
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "ReadProdSecrets",
      "Effect": "Allow",
      "Action": [
        "secretsmanager:GetSecretValue",
        "secretsmanager:DescribeSecret"
      ],
      "Resource": [
        "arn:aws:secretsmanager:ap-south-1:123:secret:myapp/prod/*"
      ]
    },
    {
      "Sid": "AllowKmsDecrypt",
      "Effect": "Allow",
      "Action": "kms:Decrypt",
      "Resource": "arn:aws:kms:ap-south-1:123:key/your-key-id"
    }
  ]
}
Resource policy — allow cross-account access 
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Principal": {
      "AWS": "arn:aws:iam::444455556666:role/partner-app-role"
    },
    "Action": "secretsmanager:GetSecretValue",
    "Resource": "*"
  }]
}

Tag-based Access Control

🏷️ Control access using tags

Tag secrets with Environment=prod. Add IAM condition:

"Condition": { "StringEquals": { "secretsmanager:ResourceTag/Environment": "prod" } }

Now only roles tagged with Environment=prod can access production secrets. Developers with dev-only access can't read prod secrets — even if they know the secret name.

Pattern 1 — ECS Microservice + Database Secret

  • Task definition: no env vars for passwords
  • IAM Task Role: allows secretsmanager:GetSecretValue
  • App code: reads secret name from env var SM_SECRET_NAME
↓ App startup

  • GetSecretValue("myapp/prod/database")
  • Parses JSON → host, username, password
  • Caches in memory — not fetched again until TTL expires
↓
✅ App connects to RDS with credentials
  • Connection pool established
  • When secret rotates → SDK cache expires → fresh fetch
  • Connection re-established with new password
  • Zero downtime, zero developer involvement

Pattern 2 — CI/CD Pipeline (GitHub Actions)

🤖 GitHub Actions workflow runs
  • Uses OIDC to assume AWS role (no stored keys!)
  • Role has: secretsmanager:GetSecretValue on deploy/* secrets
↓
🔒 Pipeline fetches deployment secrets
  • Docker registry password for pushing images
  • Slack webhook for deployment notifications
  • Test DB password for integration tests
↓
✅ Deployment proceeds with real credentials
  • Secrets used in memory during build
  • Never stored in GitHub Secrets or workflow logs
  • STS credentials expire after pipeline finishes

Pattern 3 — Multi-Environment Secret Strategy

🟡 Dev Environment
  • Secret: myapp/dev/database
  • Dev RDS credentials
  • No rotation (saves cost)
  • Dev team has read access
🟠 Staging
  • Secret: myapp/staging/database
  • Staging RDS credentials
  • 60-day rotation
  • CI/CD role access only
🟢 Production
  • Secret: myapp/prod/database
  • Prod RDS — encrypted CMK
  • 30-day rotation
  • ECS task role only

Secrets Manager Service — Full Implementation

SecretsManagerService.cs 
using Amazon.SecretsManager;
using Amazon.SecretsManager.Model;
using Microsoft.Extensions.Caching.Memory;
using System.Text.Json;

public class SecretsManagerService
{
    private readonly IAmazonSecretsManager _sm;
    private readonly IMemoryCache _cache;

    public SecretsManagerService(
        IAmazonSecretsManager sm, IMemoryCache cache)
    {
        _sm    = sm;
        _cache = cache;
    }

    // ── Get secret as raw string ──────────────────
    public async Task<string> GetSecretAsync(
        string secretName,
        TimeSpan? cacheDuration = null)
    {
        var cacheKey = $"secret:{secretName}";
        if (_cache.TryGetValue(cacheKey, out string cached))
            return cached!;

        var response = await _sm.GetSecretValueAsync(
            new GetSecretValueRequest
        {
            SecretId     = secretName,
            VersionStage = "AWSCURRENT"
        });

        var secret = response.SecretString
            ?? Convert.ToBase64String(
                response.SecretBinary.ToArray());

        // Cache for 5 min by default
        _cache.Set(cacheKey, secret,
            cacheDuration ?? TimeSpan.FromMinutes(5));

        return secret;
    }

    // ── Get secret deserialized to typed object ───
    public async Task<T> GetSecretAsync<T>(
        string secretName) where T : class
    {
        var json = await GetSecretAsync(secretName);
        return JsonSerializer.Deserialize<T>(json)
            ?? throw new InvalidOperationException(
                $"Could not deserialize secret: {secretName}");
    }

    // ── Get single field from JSON secret ─────────
    public async Task<string> GetSecretFieldAsync(
        string secretName, string fieldName)
    {
        var json = await GetSecretAsync(secretName);
        var doc  = JsonDocument.Parse(json);
        return doc.RootElement
            .GetProperty(fieldName)
            .GetString()!;
    }

    // ── Create a new secret ───────────────────────
    public async Task<string> CreateSecretAsync(
        string name, object secretValue,
        string? kmsKeyId = null)
    {
        var response = await _sm.CreateSecretAsync(
            new CreateSecretRequest
        {
            Name         = name,
            SecretString = JsonSerializer.Serialize(secretValue),
            KmsKeyId     = kmsKeyId,
            Description  = $"Created by app at {DateTime.UtcNow:O}",
            Tags         = [
                new Tag { Key="ManagedBy", Value="application" }
            ]
        });
        return response.ARN;
    }

    // ── Update an existing secret ─────────────────
    public async Task UpdateSecretAsync(
        string secretName, object newValue)
    {
        await _sm.UpdateSecretAsync(new UpdateSecretRequest
        {
            SecretId     = secretName,
            SecretString = JsonSerializer.Serialize(newValue)
        });
        // Invalidate cache
        _cache.Remove($"secret:{secretName}");
    }

    // ── Rotate secret immediately ─────────────────
    public async Task RotateNowAsync(string secretName)
    {
        await _sm.RotateSecretAsync(new RotateSecretRequest
        {
            SecretId            = secretName,
            RotateImmediately   = true
        });
    }
}

✅ Best Practices & Common Mistakes

Storing & Naming

  • Use hierarchical names: app/environment/service (e.g. myapp/prod/database)
  • Store ALL related fields in ONE secret as JSON — not separate secrets per field
  • Tag every secret: App, Environment, Team, OwnerEmail, RotationEnabled
  • Never store the secret name in the secret itself — circular and useless
  • Use descriptive descriptions — "DB creds" is not good, "PostgreSQL app_user creds for orders DB" is good
  • Use separate secrets per environment — never share dev/prod credentials

Accessing Secrets in Code

  • Cache secrets in memory — don't call GetSecretValue on every request
  • Set cache TTL to 5 minutes — gives time to auto-refresh after rotation
  • Handle SecretNotFoundException gracefully — log and fail fast at startup
  • Never log or print secret values — even partial values in logs are dangerous
  • Only pass specific fields to functions — not the whole secret object
  • Use strongly-typed models — never access raw JSON strings in business logic

Rotation

  • Always enable rotation on any secret that supports it (RDS, Redshift, etc.)
  • Test rotation manually right after configuring it — don't assume it works
  • Stagger rotation schedules — don't rotate DB and Redis at the same time
  • Monitor rotation Lambda CloudWatch Logs for failures
  • Alert on rotation failure — set SNS notification on Lambda error metric
  • For custom secrets: implement all 4 rotation steps (create, set, test, finish)

Security

  • Use IAM roles — never access keys — for all app-to-Secrets Manager calls
  • Grant only secretsmanager:GetSecretValue — not full secretsmanager:*
  • Scope IAM resource to specific ARN prefix — not * (wildcard everything)
  • Enable CloudTrail — know who accessed which secret and when
  • Alert on DeleteSecret events — someone deleting secrets is a red flag
  • Use KMS CMK for secrets shared cross-account — better key control

Common Mistakes to Avoid

⚠️ Top Secrets Manager Mistakes

❌ Still hardcoding secrets in appsettings.json — defeats the whole point
❌ Not caching secrets — calling GetSecretValue on every request drives up cost
❌ Setting up rotation but never testing it — broken rotation = locked out of DB
❌ One secret for all environments — dev team can read prod passwords
❌ Logging secret values — even in "debug" logs that go to CloudWatch
❌ Using wildcard * in IAM resource — app can read ALL your secrets
❌ Deleting secrets immediately — use recovery window (7–30 days) for safety
❌ Not alerting on rotation failures — silent failure = expired credentials at 3am
