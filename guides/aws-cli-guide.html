⌨️ What is the AWS CLI?

The AWS Command Line Interface (CLI) is an open-source tool that lets you interact with AWS services directly from your terminal. Instead of clicking through the AWS Console, you type commands — making it fast, scriptable, and automatable.

💡 AWS CLI in One Sentence

AWS CLI = a unified terminal tool to manage every AWS service using commands — the same API calls the Console makes, but scriptable, repeatable, and CI/CD-friendly.

CLI v2 vs v1

📌 Always use AWS CLI v2

CLI v2 is the current major version. It ships as a standalone binary (no Python dependency), has better SSO support, improved auto-complete, --no-cli-pager flag, and streaming binary output. CLI v1 is in maintenance mode. This guide covers CLI v2.

How the CLI works under the hood

🔄 AWS CLI — Request Flow
⌨️ You type a command
  • aws s3 ls s3://my-bucket
  • CLI parses service, operation, parameters
↓
🔑 CLI resolves credentials
  • Checks environment vars → ~/.aws/credentials → IAM role → SSO
  • Signs the request with AWS Signature V4 (SigV4)
↓
🌐 HTTPS request to AWS API
  • e.g. GET https://s3.eu-west-1.amazonaws.com/my-bucket
  • With Authorization header (SigV4 signature)
↓
📄 Response returned as JSON
  • CLI formats JSON output to your terminal
  • You can filter with --query, pipe to jq, or redirect to file

Enable Auto-complete (Highly Recommended)

bash — enable aws cli tab completion 
# Add to ~/.bashrc or ~/.bash_profile
complete -C '/usr/local/bin/aws_completer' aws
source ~/.bashrc

# Now type: aws s3 [TAB] and see all subcommands
powershell — enable completion on windows 
# Add to $PROFILE
Register-ArgumentCompleter -Native -CommandName aws -ScriptBlock {
    param($commandName, $wordToComplete, $cursorPosition)
    $env:COMP_LINE = $wordToComplete
    $env:COMP_POINT = $cursorPosition
    aws_completer | ForEach-Object { [System.Management.Automation.CompletionResult]::new($_, $_, 'ParameterValue', $_) }
    Remove-Item Env:\COMP_LINE
    Remove-Item Env:\COMP_POINT
}

⚠️ Never use long-term access keys if avoidable

Access keys stored in ~/.aws/credentials are a security risk if your machine is compromised. Prefer AWS SSO (IAM Identity Center), IAM roles on EC2/ECS/Lambda, or short-term credentials from aws sts assume-role.

Credential Resolution Order

🔑 How CLI resolves credentials — priority order
1️⃣ Command-line options --profile
↓ if not set
2️⃣ Environment variables AWS_ACCESS_KEY_ID etc.
↓ if not set
3️⃣ AWS CLI credentials file ~/.aws/credentials
↓ if not set
4️⃣ AWS CLI config file ~/.aws/config
↓ if not set
5️⃣ Container credentials (ECS task role)
↓ if not set
6️⃣ Instance profile (EC2 / Lambda role) — IMDS

📋 AWS CLI Command Structure

Full Command Anatomy with Flags

anatomy — every part labelled 
aws            # base command — always first
  s3api        # service (ec2, s3, iam, lambda, sts...)
  get-object   # operation (list-buckets, describe-instances...)
  --bucket     # parameter name (prefixed with --)
  my-bucket    # parameter value
  --key "data/report.csv"         # another param
  --output-file "/tmp/report.csv"  # local destination
  --region eu-west-1               # global flag (override region)
  --profile prod                   # global flag (use named profile)
  --query 'Body'                    # JMESPath filter on JSON response
  --output json                    # response format
  --no-cli-pager                   # don't use a pager (useful in scripts)

s3 vs s3api — What's the difference?

🪣 aws s3 — High-level commands

  • Human-friendly: cp, mv, sync, ls, rm
  • Handles multipart uploads automatically
  • Recursive operations with --recursive
  • Best for: day-to-day file operations
  • Example: aws s3 sync ./dist s3://my-site/

🔧 aws s3api — Low-level API

  • Maps directly to S3 REST API operations
  • Full control: ACLs, metadata, versioning, lifecycle
  • Returns full JSON response
  • Best for: scripts needing fine-grained control
  • Example: aws s3api put-bucket-versioning

--query — JMESPath Filtering

--query uses JMESPath syntax to extract specific parts of the JSON response. This is the most important CLI skill to master.

--query examples — jmespath patterns 
# Get a top-level field
aws s3api list-buckets --query 'Owner.DisplayName'

# Get an array element field
aws s3api list-buckets --query 'Buckets[].Name'

# Filter array by condition
aws ec2 describe-instances \
  --query 'Reservations[].Instances[?State.Name==`running`].InstanceId'

# Multiple fields as array (inner brackets)
aws ec2 describe-instances \
  --query 'Reservations[].Instances[].[InstanceId,InstanceType,State.Name]'

# Sort by a field
aws ec2 describe-instances \
  --query 'sort_by(Reservations[].Instances[], &LaunchTime)[].InstanceId'

# Get first item only
aws s3api list-buckets --query 'Buckets[0].Name'

# Alias fields with different names
aws ec2 describe-instances \
  --query 'Reservations[].Instances[].{ID:InstanceId,Type:InstanceType,Status:State.Name}' \
  --output table

Practical --query Patterns

common patterns you'll use daily 
# Get just instance IDs of running instances
aws ec2 describe-instances \
  --filters "Name=instance-state-name,Values=running" \
  --query 'Reservations[].Instances[].InstanceId' \
  --output text

# Get S3 bucket names as plain text (useful in loops)
aws s3api list-buckets \
  --query 'Buckets[].Name' \
  --output text

# Get all Lambda function names
aws lambda list-functions \
  --query 'Functions[].FunctionName' \
  --output text

# Check if a stack exists and get its status
aws cloudformation describe-stacks \
  --stack-name my-stack \
  --query 'Stacks[0].StackStatus' \
  --output text

Piping to jq — More Powerful Filtering

jq — json processor (install separately) 
# Install jq (Windows via winget)
winget install jqlang.jq

# Pretty-print the full response
aws s3api list-buckets | jq '.'

# Extract just bucket names
aws s3api list-buckets | jq '.Buckets[].Name'

# Filter running instances and get IDs + IPs
aws ec2 describe-instances | jq \
  '.Reservations[].Instances[] | select(.State.Name == "running") | {id: .InstanceId, ip: .PrivateIpAddress}'

# Extract and export a secret value

  --secret-id prod/db-password | jq -r '.SecretString')

🤖 Scripting & Automation

Bash — Practical Automation Scripts

bash — stop all running ec2 instances in a region 
#!/bin/bash
REGION="eu-west-1"

# Get all running instance IDs
INSTANCE_IDS=$(aws ec2 describe-instances \
  --region $REGION \
  --filters "Name=instance-state-name,Values=running" \
  --query 'Reservations[].Instances[].InstanceId' \
  --output text)

if [ -z "$INSTANCE_IDS" ]; then
  echo "No running instances found."
  exit 0
fi

echo "Stopping: $INSTANCE_IDS"
aws ec2 stop-instances \
  --instance-ids $INSTANCE_IDS \
  --region $REGION

echo "Done."
bash — wait for cloudformation stack to complete 
#!/bin/bash
STACK_NAME="my-app-stack"

# Deploy the stack
aws cloudformation deploy \
  --stack-name $STACK_NAME \
  --template-file template.yaml \
  --capabilities CAPABILITY_IAM

# Wait until deploy is complete (built-in waiter)
aws cloudformation wait stack-create-complete \
  --stack-name $STACK_NAME

# Get the output URL
URL=$(aws cloudformation describe-stacks \
  --stack-name $STACK_NAME \
  --query 'Stacks[0].Outputs[?OutputKey==`ApiUrl`].OutputValue' \
  --output text)

echo "Deployed! API URL: $URL"
bash — sync s3 website and invalidate cloudfront 
#!/bin/bash
BUCKET="my-site-bucket"
DISTRIBUTION_ID="EXXXXXXXXXXXXX"
BUILD_DIR="./dist"

echo "📦 Syncing to S3..."
aws s3 sync $BUILD_DIR s3://$BUCKET \
  --delete \
  --cache-control "max-age=86400"

echo "🔄 Invalidating CloudFront cache..."
INVALIDATION_ID=$(aws cloudfront create-invalidation \
  --distribution-id $DISTRIBUTION_ID \
  --paths "/*" \
  --query 'Invalidation.Id' \
  --output text)

echo "✅ Done! Invalidation: $INVALIDATION_ID"

Essential Tips

  • Add --output table for any list command — much easier to read than JSON
  • Use --query to extract exactly what you need — avoid parsing JSON with grep/sed
  • Use --no-cli-pager in scripts — prevents hanging on interactive prompts
  • Use aws logs tail --follow instead of CloudWatch console for live debugging
  • aws sts get-caller-identity first — always verify you're in the right account
  • Use --dry-run on EC2 commands to test permissions before running for real
  • Store secrets in environment variables for CI/CD, never in scripts or config files
  • Use named profiles (--profile) — never mix dev/prod in the same session

Common Mistakes

  • Running destructive commands without --profile — could affect wrong account
  • Using long-term access keys in ~/.aws/credentials for production
  • Forgetting --cli-binary-format raw-in-base64-out when passing Lambda payloads
  • Using --output text in scripts — whitespace handling is fragile, use JSON
  • Not using aws ... wait in scripts — race conditions on resource creation
  • Storing credentials in shell scripts or committing ~/.aws/credentials to Git
