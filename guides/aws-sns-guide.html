📡 What is Amazon SNS?

SNS = Simple Notification Service. A fully managed pub/sub messaging service. One publisher sends a single message to a Topic — SNS instantly delivers it to all subscribers simultaneously, across multiple protocols and endpoints.

💡 Real-life analogy

SNS is like a radio broadcast tower. When a radio station (publisher) transmits a signal (message), every radio tuned to that frequency (subscribers) receives it instantly — whether it's a car radio, a home stereo, or a phone app. The station doesn't know or care how many listeners there are. It just broadcasts. Each listener receives the same signal at the same time.

SNS vs SQS — The Fundamental Difference

📡 SNS — Push (Broadcast)

  • Pub/Sub model — one → many
  • Message delivered IMMEDIATELY to all subscribers
  • Subscribers don't poll — SNS pushes to them
  • Message not stored — if delivery fails → retry or DLQ
  • Use when: multiple systems need same event
  • Think: broadcast, notification, fan-out

📬 SQS — Pull (Queue)

  • Point-to-point — one → one consumer
  • Message sits in queue until consumer polls it
  • Consumer pulls at its own pace
  • Message stored up to 14 days
  • Use when: one system processes each message
  • Think: task queue, work queue, job queue

Without Fan-out vs With Fan-out

❌ Without SNS — tight coupling
  • Order Service calls Inventory API directly
  • Order Service calls Email Service directly
  • Order Service calls Analytics API directly
  • Each new consumer: add more code to Order Service
  • Inventory Service down → Order Service fails
  • Order Service knows about ALL downstream systems

  • Order Service publishes to SNS Topic — done
  • All subscribers receive automatically
  • New consumer: add subscription — zero code change
  • Inventory Service down → SNS retries independently
  • Order Service knows NOTHING about consumers
  • Add or remove subscribers at any time

📨 Messages, Attributes & Filtering

Every SNS message has a body and optional message attributes. Subscribers can set filter policies so they only receive messages that match — avoiding unnecessary processing.

Anatomy of an SNS Message

📨 SNS Message — full structure
MessageId "a1b2c3d4-e5f6-7890-abcd-ef1234567890" AWS-assigned unique ID
TopicArn "arn:aws:sns:ap-south-1:123456789:OrderEvents" Source topic
Subject "OrderPlaced" Optional — for email subscriptions
Message {"orderId":"ORD-123","userId":"usr_456","amount":2499.00,"currency":"INR"} Your payload — up to 256 KB
Timestamp "2025-04-07T10:30:00.000Z" When published
MessageAttributes {"eventType":{"Type":"String","Value":"order.placed"}, "region":{"Type":"String","Value":"india"}, "amount":{"Type":"Number","Value":"2499"}} Used for filtering ← key feature

Message Filtering — Route Messages to Right Subscribers

Without filtering: every subscriber receives every message — wasted processing. With filtering: each subscriber only receives messages matching its filter policy.

🔍 Topic: OrderEvents — Message attributes: eventType=order.placed, region=india, amount=2499
📬 Inventory Queue {"eventType": ["order.placed", "order.cancelled"]} ✅ DELIVER
λ Email Lambda {"eventType": ["order.placed"], "region": ["india"]} ✅ DELIVER
📬 EU Analytics {"region": ["europe", "uk", "germany"]} ⛔ SKIP
📬 HighValue Queue {"amount": [{"numeric": [">=", 1000]}]} ✅ DELIVER
📬 Refund Queue {"eventType": ["order.refunded"]} ⛔ SKIP
🌐 Partner Hook (no filter policy — receives ALL messages) ✅ DELIVER

Filter Policy Operators

🔤 String Operators

  • ["order.placed"] — exact match
  • ["order.placed", "order.cancelled"] — OR match
  • [{"prefix": "order."}] — starts with
  • [{"suffix": ".failed"}] — ends with
  • [{"anything-but": "test"}] — NOT match

🔢 Numeric Operators

  • [{"numeric": ["=", 100]}] — equals
  • [{"numeric": [">", 1000]}] — greater than
  • [{"numeric": [">=", 500, "<=", 5000]}] — range
  • [{"exists": true}] — attribute present
  • [{"exists": false}] — attribute absent

💡 Filter policy — subscription vs message body

Filters work on Message Attributes (metadata) — NOT on the message body (JSON payload). This is intentional: SNS can filter without parsing/deserializing the full message body, making filtering extremely fast and cheap. Always include routing-relevant data as message attributes, keep detailed payload in the message body.

📋 Topic Types — Standard vs FIFO

SNS offers two topic types with fundamentally different delivery guarantees. Choosing the right one depends on whether message order and deduplication matter.

Standard Topic vs FIFO Topic

🔶 Standard Topic (Default)

  • Throughput: nearly unlimited (30M+ msg/sec)
  • Delivery order: best-effort (may be out of order)
  • Delivery: at-least-once (may deliver duplicate)
  • Subscribers: SQS, Lambda, HTTP, Email, SMS, Push
  • Cost: lower
  • Use for: most event notifications, alerts, async workflows

🔷 FIFO Topic

  • Throughput: 300 messages/sec (3000 with batching)
  • Delivery order: strict FIFO per Message Group ID
  • Delivery: exactly-once (deduplication window: 5 min)
  • Subscribers: FIFO SQS queues ONLY
  • Cost: higher (~10× standard)
  • Use for: financial transactions, inventory updates, ordered workflows

When FIFO Order Matters — Stock Price Updates

❌ Standard — wrong order possible
  • Published: AAPL=$180 → AAPL=$175 → AAPL=$182
  • Received: AAPL=$175 → AAPL=$182 → AAPL=$180 ❌
  • UI shows $180 as latest — wrong!
  • Financial decisions based on stale data
✅ FIFO — guaranteed order
  • Published: AAPL=$180 → AAPL=$175 → AAPL=$182
  • Received: AAPL=$180 → AAPL=$175 → AAPL=$182 ✓
  • UI always shows correct latest price
  • MessageGroupId: "AAPL" — all grouped together

FIFO — Key Concepts

🗂️ Message Group ID

Groups related messages together. Within a group, FIFO order is guaranteed. Different groups can process in parallel. Example: OrderID as group → all events for one order are ordered. Different orders process simultaneously.

🔁 Deduplication ID

5-minute deduplication window. If two messages with same ID are published within 5 minutes — second is dropped silently. Use for: idempotent republishing after network error. Option: content-based deduplication (SHA-256 of body).

💡 Large message payload workaround

SNS max is 256 KB. For larger payloads: publish the data to S3, then publish an SNS message containing the S3 URL. Subscribers fetch the full payload from S3. The SNS Extended Client Library for Java automates this — no official .NET version, but easy to implement manually in C#.

🔄 Delivery, Retry & Dead-Letter Queues

SNS guarantees delivery attempts — not instant delivery. When a subscriber endpoint is unavailable, SNS retries with exponential backoff. If all retries fail, the message goes to a Dead-Letter Queue (DLQ).

Delivery Reliability by Protocol

SQS
~100% Most reliable — messages stored in SQS if SNS delivers successfully
Lambda
~100% Highly reliable — AWS manages invocation
HTTP/S
~85% Depends on your endpoint availability — use DLQ
SMS
~90% Network carrier dependent — varies by country
Email
~75% Spam filters + invalid emails reduce delivery

SNS Retry Policy — What Happens When Delivery Fails

📡 SNS attempts delivery to HTTP endpoint
  • Endpoint returns 5xx or times out
  • Delivery attempt 1 fails
↓
⏳ Phase 1 — Immediate Retries
  • 3 immediate retries with no delay
  • Fast-fail scenario: endpoint may have just restarted
↓
⏳ Phase 2 — Pre-Backoff Retries
  • Short delays between retries (1–2 seconds)
  • Gives endpoint time to recover from brief outage
↓
⏳ Phase 3 — Exponential Backoff
  • Delays grow exponentially: 1s → 2s → 4s → 8s...
  • Maximum delay: 20 seconds between retries
  • Total retry window: up to 23 days
  • Configurable: min/max delay, backoff function
↓ After all retries exhausted
💀 No DLQ configured
  • Message silently dropped
  • No recovery possible
  • Lost forever
✅ DLQ configured
  • Message sent to SQS Dead-Letter Queue
  • Inspect failed messages
  • Fix endpoint → redrive messages
  • Zero message loss

Dead-Letter Queue (DLQ) Setup

💡 How DLQ works with SNS

Create an SQS queue as the DLQ. Attach it to the SNS subscription (not the topic). Each subscription can have its own DLQ — giving fine-grained failure handling per endpoint.

When to inspect DLQ: set a CloudWatch alarm on ApproximateNumberOfMessagesVisible > 0 → SNS alert → engineer investigates.

After fixing the underlying issue: use SQS Dead-Letter Queue Redrive to replay all failed messages back through SNS automatically.

SNS Service — Publish, Subscribe, Manage

SnsService.cs 
using Amazon.SimpleNotificationService;
using Amazon.SimpleNotificationService.Model;
using System.Text.Json;

public class SnsService
{
    private readonly IAmazonSimpleNotificationService _sns;

    public SnsService(IAmazonSimpleNotificationService sns)
        => _sns = sns;

    // ── Publish a message with attributes (for filtering) ─
    public async Task<string> PublishAsync<T>(
        string topicArn,
        T      payload,
        Dictionary<string, string>? attributes = null)
    {
        var request = new PublishRequest
        {
            TopicArn = topicArn,
            Message  = JsonSerializer.Serialize(payload)
        };

        // Add message attributes for filter policies
        if (attributes != null)
        {
            foreach (var (key, value) in attributes)
            {
                request.MessageAttributes[key] =
                    new MessageAttributeValue
                    {
                        DataType    = "String",
                        StringValue = value
                    };
            }
        }

        var response = await _sns.PublishAsync(request);
        return response.MessageId;
    }

    // ── Publish with numeric attribute (for numeric filters) ─
    public async Task<string> PublishOrderAsync(
        string topicArn, OrderPlacedEvent order)
    {
        var response = await _sns.PublishAsync(new PublishRequest
        {
            TopicArn = topicArn,
            Message  = JsonSerializer.Serialize(order),
            MessageAttributes = new()
            {
                ["eventType"] = new MessageAttributeValue
                {
                    DataType = "String",
                    StringValue = "order.placed"
                },
                ["region"] = new MessageAttributeValue
                {
                    DataType = "String",
                    StringValue = order.Region
                },
                ["amount"] = new MessageAttributeValue
                {
                    DataType    = "Number", // for numeric filters
                    StringValue = order.Amount.ToString("F2")
                }
            }
        });
        return response.MessageId;
    }

    // ── Batch publish (up to 10 messages) ─────────────────
    public async Task<PublishBatchResponse> PublishBatchAsync<T>(
        string topicArn,
        IEnumerable<T> payloads)
    {
        var entries = payloads
            .Select((p, i) => new PublishBatchRequestEntry
            {
                Id      = i.ToString(), // unique within batch
                Message = JsonSerializer.Serialize(p)
            })
            .ToList();

        return await _sns.PublishBatchAsync(
            new PublishBatchRequest
        {
            TopicArn                 = topicArn,
            PublishBatchRequestEntries = entries
        });
    }

    // ── Subscribe SQS queue to topic ──────────────────────
    public async Task<string> SubscribeSqsAsync(
        string topicArn,
        string sqsArn,
        string? filterPolicyJson = null)
    {
        var response = await _sns.SubscribeAsync(
            new SubscribeRequest
        {
            TopicArn = topicArn,
            Protocol = "sqs",
            Endpoint = sqsArn,
            Attributes = filterPolicyJson != null
                ? new()
                {
                    ["FilterPolicy"]           = filterPolicyJson,
                    ["RawMessageDelivery"]      = "true"
                }
                : new() { ["RawMessageDelivery"] = "true" }
        });
        return response.SubscriptionArn;
    }

    // ── List all subscriptions for a topic ────────────────
    public async Task<List<Subscription>>
        ListSubscriptionsAsync(string topicArn)
    {
        var response = await _sns
            .ListSubscriptionsByTopicAsync(topicArn);
        return response.Subscriptions;
    }
}

public record OrderPlacedEvent(
    string  OrderId,
    string  UserId,
    decimal Amount,
    string  Currency,
    string  Region);

✅ Best Practices & Common Mistakes

Topic Design

  • Create one topic per event domain — OrderEvents, UserEvents, PaymentEvents (not one giant "Events" topic)
  • Name topics descriptively: service-entity-action pattern (OrderService-OrderEvents)
  • Enable server-side encryption (KMS) on all production topics
  • Choose FIFO only when ordering truly matters — it has lower throughput limits and higher cost
  • Use message attributes for routing metadata — keep business payload in the message body
  • Enable delivery status logging for Lambda and SQS subscribers from day one

Subscriptions & Filtering

  • Always use filter policies — avoid subscribers receiving irrelevant messages
  • Enable Raw Message Delivery for SQS subscribers — cleaner payload, no SNS envelope unwrapping
  • Configure DLQ on every subscription — never lose messages silently
  • Set CloudWatch alarm on DLQ depth > 0 — get alerted immediately on delivery failures
  • Set SQS queue access policy to allow SNS — common mistake: subscription created but delivery denied
  • Test filter policies by publishing with different attributes — verify routing before production

Publishing

  • Use batch publish (up to 10 messages) to reduce API calls and cost
  • Include a timestamp and correlationId in every message for distributed tracing
  • For messages > 256 KB: store payload in S3, publish S3 URL in SNS message
  • Use idempotent message designs — subscribers should handle duplicate delivery gracefully
  • Set a MessageDeduplicationId on FIFO publishes to prevent duplicates on retry
  • Never put secrets or PII in message attributes — they appear in delivery logs

Common Mistakes to Avoid

⚠️ Top SNS Mistakes

❌ No DLQ on subscriptions — failed messages disappear silently after retries
❌ SQS queue missing SNS publish permission — subscription shows confirmed but messages never arrive
❌ Not using filter policies — every subscriber processes every message (expensive + slow)
❌ Using SNS for request/response — SNS is fire-and-forget, not for synchronous responses
❌ Putting payload in message attributes — max 256 bytes per attribute, use message body
❌ One giant topic for all events — hard to manage, filter, and secure per event type
❌ Not enabling raw delivery for SQS — consumers receive wrapped SNS envelope, not payload
❌ Using email subscriptions in production flows — email is for humans, not reliable machine-to-machine
