🌲 What is Depth-First Search?

Let's start from zero. No jargon. Just plain understanding.

💡 DFS in One Sentence

DFS is a way to explore or search through a structure (like a tree or graph) by going as deep as possible into one path before coming back and trying the next path.

🏠 The House Exploration Analogy

🏠Imagine you're exploring a big old house

You enter the house. You see 3 doors. You pick door 1 and walk in. Inside, there are 2 more doors. You pick the first one. It leads to a room with no doors — dead end. You come back. Try the second door. Another room. Come back. Now you've finished door 1 completely. Go back to the hallway. Try door 2. Repeat.

That is DFS — you go as deep as possible into one branch, and only come back when you hit a dead end.

🆚DFS vs BFS — The maze analogy

DFS (Depth-First): You pick one corridor of a maze and follow it all the way until you hit a wall. Then you backtrack and try the next corridor. You go deep first.

BFS (Breadth-First): You check every corridor that's one step away, then every corridor two steps away, then three. You go wide first.

DFS = "Go deep, explore fully, come back."   BFS = "Check all neighbours first, then go further."

What "structures" does DFS work on?

🌳 Trees

  • Hierarchical structure — parent → children
  • No cycles — every node has exactly one parent (except root)
  • Examples: file system, HTML DOM, org chart, family tree
  • DFS on trees: go left child all the way down, then right

🕸️ Graphs

  • Network of nodes connected by edges
  • Can have cycles — need to track "visited" nodes
  • Examples: road map, social network, internet links, dependencies
  • DFS on graphs: visit node, mark visited, explore all unvisited neighbours

🧱 Core Concepts — Build Your Foundation

Before DFS makes full sense, you need to understand the structures it works on. Let's build them from scratch.

Part 1: What is a Tree?

🌳Family Tree — the easiest mental model

Think of your family tree. Your grandparent is the root. They have children (your parent, uncle/aunt). Each of them have their own children (you, your cousins). Nobody connects back upward — it's a one-way, top-down structure. That's a tree.

🌳 A Simple Binary Tree — visualised
■ Root (A) — top level ■ Internal nodes (B, C) — have children ■ Leaves (D,E,F,G) — no children

Part 2: What is a Graph?

🕸️Graph = Tree + extra connections (and cycles!)

A graph is like a map of cities and roads. Cities = nodes. Roads = edges. Unlike trees, in a graph you can have roads looping back — City A → City B → City C → back to City A. That's a cycle. Graphs can be directed (one-way roads) or undirected (two-way roads).

🕸️ Graph with Cycles — Chennai Metro Example

Unlike trees, graphs can have edges going in multiple directions — creating cycles. DFS needs a visited set to avoid infinite loops.

Part 3: The Stack — DFS's secret weapon

DFS uses a stack to remember where it came from. A stack is like a pile of plates — you add to the top, and remove from the top (Last In, First Out = LIFO).

📚 Stack Operations

  • Push — put item on top (enter a room)
  • Pop — remove item from top (leave a room)
  • Peek — look at top item without removing
  • DFS uses the stack (or recursion's call stack) to track path

📞 Recursion = Implicit Stack

  • When a function calls itself, it uses the call stack
  • Each recursive call is like "pushing" to the stack
  • When it returns, it "pops" back to the previous call
  • DFS naturally maps to recursion — that's why they pair perfectly

Part 4: Recursion — the most important concept

🪞Recursion = Russian nesting dolls

A Matryoshka doll contains a smaller version of itself inside. Open it → another doll. Open that → another. Until you reach the smallest one (base case). That's recursion. A function calls a smaller version of itself until it hits a stopping point.

Base Case = the smallest doll (stop condition). Recursive Case = keep opening dolls. Without a base case → infinite loop = stack overflow!

🧠 Mental Model Summary

Tree/Graph = the structure you're exploring. Node = each item in it. Edge = connection between nodes. Stack = your memory of where you've been. Recursion = the elegant way to implement DFS. Visited set = don't go in circles.

🤔 Why DFS? When Should You Use It?

💡 Why DFS exists

Many real-world problems involve hierarchical or connected data. You need a systematic way to visit every node exactly once, discover paths, detect cycles, or check reachability. DFS is one of the most efficient and elegant algorithms for all of these.

When DFS is the RIGHT choice ✅

🔍

Does a path exist from A to B? DFS finds it fast.

🌀

Detect cycles

Is there a circular dependency? DFS spots it.

🗺️

🔗

Connected components

Which nodes are connected to each other?

🌳

Tree traversal

📁

⚠️ Stack Overflow Warning

Recursive DFS can hit a stack overflow on very deep trees (10,000+ levels). For such cases, use iterative DFS with an explicit stack — same algorithm, no recursion limit.

🌍 Real-World Examples — Where You See DFS Daily

DFS isn't just a textbook concept — it's running behind the scenes in systems you use every day.

📁File System Search — Windows Explorer / FinderDFS in disguise

When you do Ctrl+F to search for a file in a folder, the OS uses DFS. It opens the first subfolder, goes into its subfolders, all the way down — then comes back up and tries the next subfolder. Your entire C:\Users\Indhu\Documents is a tree. DFS visits every folder depth-first.

In code (Team Watson): Searching for all .csproj files in a solution folder — DFS through the directory tree.

🌐Web Crawlers (Google Bot)Indexing the web

Google's crawler starts at a webpage, finds all links on it, follows the first link deeply into its sub-pages, sub-sub-pages, until it hits dead ends. Then backtracks and follows the next link. The entire web is a massive graph — DFS is one of the traversal strategies used to index it.

Visited set = required! Without it, the crawler would loop forever between mutually linked pages.

🏗️Build Systems (MSBuild, npm, NuGet)Dependency resolution

Your C# project depends on Package A → Package B → Package C. Before building, the system must resolve all dependencies in the right order. It uses DFS to traverse the dependency graph. Topological sort (a DFS variant) ensures Package C is built before B, B before A.

Cycle detection via DFS: If Package A depends on Package B which depends back on Package A — that's a circular dependency. DFS catches it and throws an error.

🧩Sudoku / Puzzle SolversBacktracking = DFS

Solving Sudoku programmatically uses DFS + backtracking. Try placing 1 in cell (0,0). Go to next empty cell. Try 1. Conflict? Try 2. Still conflict? Backtrack — go back to (0,0) and try 2. Continue. This "try → fail → backtrack → try next" pattern is pure DFS on a decision tree.

Chess AI, word search games, N-Queens problem — all use this same DFS backtracking approach.

🗺️GPS Navigation — Route DiscoveryReachability check

Before calculating the shortest path, apps like Google Maps first check: "Is there ANY route from A to B?" DFS answers this quickly. Start at location A, explore roads, go deep — if you reach B, a path exists. Also used to find all cities reachable from a given city (flood-fill style).

🖧Network Topology AnalysisAWS VPC / Team Watson

In AWS, checking if a resource in VPC-A can reach a resource in VPC-B through peering connections — that's a graph reachability problem. DFS traverses the network graph to determine connectivity. Also used to find strongly connected components — groups of services that can all communicate with each other.

🌳HTML DOM ParsingEvery browser does this

Your browser parses HTML into a tree (the DOM). <html> → <body> → <div> → <p>. DFS is used to traverse this tree for operations like getElementById, CSS selector matching, rendering, and JavaScript DOM manipulation. Even React's virtual DOM diffing uses tree traversal algorithms rooted in DFS.

🎮Game Development — Fog of War / PathfindingMap exploration

In strategy games, "fog of war" reveals areas as your unit explores. The map is a grid graph. DFS determines all connected visible cells from your current position. Also used in procedural dungeon generation — the game generates one corridor deeply before branching.

⚙️ How DFS Works — Step by Step

Let's trace through DFS manually, step by step, with no guessing.

DFS Algorithm — Plain English

📋 DFS Algorithm Flow
🟢 Start at a node
  • Mark it as visited
  • Add it to your path / result
↓
🔍 Look at this node's neighbours
  • Get the list of connected nodes (children / adjacent)
  • For each unvisited neighbour...
↓
🔁 Recursively visit the neighbour
  • Call DFS on that neighbour
  • This goes deep into that branch completely
↓
⬅️ Backtrack when done
  • All neighbours explored? Return to the previous node
  • Continue with next unvisited neighbour there
↓
✅ Done!
  • When all nodes reachable from start are visited, DFS is complete

Pseudocode — The Blueprint

pseudocode — dfs algorithm 
DFS(node, visited):
    // Base case: if null or already visited, stop
    if node is null OR node in visited:
        return

    // Mark as visited
    visited.add(node)

    // Process this node (print, store, check, etc.)
    process(node)

    // Recurse into each unvisited neighbour
    for each neighbour in node.neighbours:
        DFS(neighbour, visited)

    // (Implicit backtrack when this call returns)

✅ Key Insight

The magic of recursion: when DFS(neighbour) is called, it will go all the way deep into that subtree before returning. When it returns, we automatically "backtrack" to the current node. No explicit backtracking code needed — the call stack does it!

🌳 DFS Tree Traversal — Pre, In, Post Order

When DFS is applied to a tree, there are 3 classic ordering strategies. The only difference is when you process the current node — before, between, or after visiting children.

🌳 Example Tree for Traversal

① Pre-order Traversal — Root → Left → Right

📋 Rule: Process node FIRST, then explore children

Order: 4 → 2 → 1 → 3 → 6 → 5 → 7

Use case: Copy a tree, serialize a tree, prefix expressions, print directory structure (parent folder before its contents).

📋 Pre-order: visit node → go left → go right
① Visit 4 (root)
② Left subtree
  • Visit 2, then 1, then 3
③ Right subtree
  • Visit 6, then 5, then 7
Result: [4, 2, 1, 3, 6, 5, 7]

② In-order Traversal — Left → Root → Right

📋 Rule: Explore left first, THEN process node, THEN right

Order: 1 → 2 → 3 → 4 → 5 → 6 → 7

Use case: Binary Search Tree (BST) — gives sorted order! Extremely useful for BST validation and sorted output.

💡 BST Magic

For a Binary Search Tree, in-order traversal always gives nodes in ascending sorted order. This is the single most common DFS interview use case.

③ Post-order Traversal — Left → Right → Root

📋 Rule: Explore all children FIRST, process node LAST

Order: 1 → 3 → 2 → 5 → 7 → 6 → 4

Use case: Delete a tree, calculate directory size, evaluate arithmetic expressions, dependency resolution.

Quick Comparison

TraversalOrderResult on example treeClassic use case
Pre-orderNode → Left → Right4, 2, 1, 3, 6, 5, 7Copy tree, print hierarchy
In-orderLeft → Node → Right1, 2, 3, 4, 5, 6, 7 ✨BST sorted output
Post-orderLeft → Right → Node1, 3, 2, 5, 7, 6, 4Delete tree, folder size

🧠 Memory Trick

PRE = node comes PRE (before) children. IN = node is IN the middle. POST = node comes POST (after) children. The "root" position relative to left/right tells you everything.

📝 Pseudocode — Every DFS Pattern

Pseudocode bridges concept and real code. Master these patterns and any DFS variant becomes recognisable instantly.

📌 How to read the pseudocode

Pseudocode uses natural English + common programming constructs — not tied to any language. ← means assign. Words in CAPS are operations. Indentation shows nesting. for each = loop. return = stop and give back a value.

① Core DFS — The Universal Recursive Template

pseudocode — recursive dfs (works on trees and graphs) 
DFS(node, visited):
    // Base case: stop if null or already seen
    if node is null OR node in visited:
        return

    // Mark as visited (prevents revisiting in graphs)
    ADD(visited, node)

    // ── place PROCESS(node) here for Pre-order ──

    for each neighbour in NEIGHBOURS(node):
        DFS(neighbour, visited)      // recurse deeper

    // ── place PROCESS(node) here for Post-order ──
    // (implicit backtrack happens when this call returns)

🧠 The Three-Position Rule

Every DFS recursive template has three positions where you can "process" the current node — before the recursive calls (Pre-order), between them (In-order, binary trees only), or after them (Post-order). Same skeleton, different placement = completely different output.

② Pre-order DFS — Root → Left → Right

pseudocode — pre-order traversal 
DFS_PREORDER(node):
    if node is null: return

    PROCESS(node)                 // ① node FIRST
    DFS_PREORDER(node.left)       // ② then left subtree
    DFS_PREORDER(node.right)      // ③ then right subtree

// Tree: 4 → 2 → 1 → 3 → 6 → 5 → 7
// Output: [4, 2, 1, 3, 6, 5, 7]
// Use case: copy a tree, print folder structure (parent before contents)

③ In-order DFS — Left → Root → Right

pseudocode — in-order traversal (sorted output for BST) 
DFS_INORDER(node):
    if node is null: return

    DFS_INORDER(node.left)        // ① left subtree first
    PROCESS(node)                 // ② node IN the middle
    DFS_INORDER(node.right)       // ③ right subtree last

// Tree: 4 → 2 → 1 → 3 → 6 → 5 → 7
// Output: [1, 2, 3, 4, 5, 6, 7]  ← sorted! BST magic ✨
// Use case: BST sorted output, BST validation

④ Post-order DFS — Left → Right → Root

pseudocode — post-order traversal 
DFS_POSTORDER(node):
    if node is null: return

    DFS_POSTORDER(node.left)      // ① left subtree
    DFS_POSTORDER(node.right)     // ② right subtree
    PROCESS(node)                 // ③ node LAST

// Tree: 4 → 2 → 1 → 3 → 6 → 5 → 7
// Output: [1, 3, 2, 5, 7, 6, 4]
// Use case: delete tree, calculate folder size, evaluate expressions

⑤ Iterative DFS — Using an Explicit Stack

pseudocode — iterative dfs (no recursion, no stack overflow risk) 
DFS_ITERATIVE(root):
    if root is null: return

    stack ← new Stack
    PUSH(stack, root)

    while stack is not empty:
        node ← POP(stack)           // take from TOP (LIFO)
        PROCESS(node)

        // Push RIGHT first so LEFT is processed first
        if node.right is not null: PUSH(stack, node.right)
        if node.left  is not null: PUSH(stack, node.left)

// Same pre-order result as recursive: [4, 2, 1, 3, 6, 5, 7]
// Use when: tree depth > 10,000 nodes (stack overflow risk with recursion)

⚠️ Why push RIGHT before LEFT?

Because a stack is LIFO — the last item pushed is the first popped. To process the left child first, it must be on top — so push right first, then left. This mirrors the recursive call order.

⑥ DFS on a Graph — with Visited Set

pseudocode — graph dfs (handles cycles with visited set) 
DFS_GRAPH(graph, start):
    visited ← new Set
    DFS_VISIT(graph, start, visited)

DFS_VISIT(graph, node, visited):
    if node in visited: return   // already seen — stop

    ADD(visited, node)
    PROCESS(node)

    for each neighbour in NEIGHBOURS(graph, node):
        DFS_VISIT(graph, neighbour, visited)

// Without visited set → infinite loop on cycles!
// Example graph: A→B→D, A→C→F
// Output: A, B, D, C, F  (goes deep into one branch first)

⑦ DFS — Check if Path Exists (Reachability)

pseudocode — dfs path existence check 
DFS_HAS_PATH(graph, current, target, visited):
    if current == target: return true    // found it!
    if current in visited: return false  // already tried, dead end

    ADD(visited, current)

    for each neighbour in NEIGHBOURS(graph, current):
        if DFS_HAS_PATH(graph, neighbour, target, visited):
            return true                  // path found via this branch

    return false                         // no path from current

// Usage: DFS_HAS_PATH(graph, "EntryAPI", "Database", new Set)

⑧ DFS — Find All Paths (Backtracking)

pseudocode — dfs all paths with backtracking 
DFS_ALL_PATHS(graph, current, target, path, allPaths):
    ADD(path, current)

    if current == target:
        ADD(allPaths, COPY(path))          // save a copy of this path
    else:
        for each neighbour in NEIGHBOURS(graph, current):
            if neighbour not in path:       // avoid revisiting on THIS path
                DFS_ALL_PATHS(graph, neighbour, target, path, allPaths)

    REMOVE_LAST(path)                      // ← backtrack! undo the choice

// Backtracking = add to path, recurse, REMOVE from path on return
// This "undo" is what makes DFS explore all possibilities

⑨ Cycle Detection in a Directed Graph

pseudocode — cycle detection using dfs recursion stack 
DFS_HAS_CYCLE(graph, node, visited, inStack):
    ADD(visited, node)
    ADD(inStack, node)           // this node is on the current DFS path

    for each neighbour in NEIGHBOURS(graph, node):
        if neighbour not in visited:
            if DFS_HAS_CYCLE(graph, neighbour, visited, inStack):
                return true      // cycle found deeper
        else if neighbour in inStack:
            return true          // back edge = cycle!

    REMOVE(inStack, node)        // backtrack — no longer on this path
    return false

// visited  = nodes ever explored (don't re-enter)
// inStack  = nodes on the CURRENT recursive path
// If we reach a node that's already inStack → we looped back = cycle

⑩ Topological Sort — DFS Post-order

pseudocode — topological sort using dfs (post-order + result stack) 
TOPO_SORT(graph, nodes):
    visited    ← new Set
    resultStack ← new Stack      // collect in post-order

    for each node in nodes:
        if node not in visited:
            DFS_TOPO(graph, node, visited, resultStack)

    return REVERSE(resultStack)  // or pop all — gives topo order

DFS_TOPO(graph, node, visited, resultStack):
    ADD(visited, node)

    for each neighbour in NEIGHBOURS(graph, node):
        if neighbour not in visited:
            DFS_TOPO(graph, neighbour, visited, resultStack)

    PUSH(resultStack, node)      // push AFTER all dependencies done

// Post-order = node added after all its dependencies
// Use case: C# build order, NuGet dependency resolution

c# — recursive pre-order traversal 
// Pre-order: visit node FIRST, then explore children
// Use case: printing a folder structure (parent before contents)
public static void PreOrder(TreeNode? node)
{

    Console.Write(node.Val + " ");  // ① process node

3. In-order DFS (Left → Root → Right)

c# — recursive in-order traversal (gives sorted output for BST) 
public static void InOrder(TreeNode? node)
{
    if (node == null) return;

    InOrder(node.Left);             // ① go left first
    Console.Write(node.Val + " ");  // ② THEN process node
    InOrder(node.Right);            // ③ then right
}

// Output: 1 2 3 4 5 6 7  ← sorted! BST magic ✨

4. Post-order DFS (Left → Right → Root)

c# — recursive post-order traversal 
// Post-order: children fully explored before processing parent
// Use case: calculating folder sizes (need child sizes first)
public static void PostOrder(TreeNode? node)
{
    if (node == null) return;

    PostOrder(node.Left);           // ① go left
    PostOrder(node.Right);          // ② go right

}

5. Iterative DFS (using Stack — no recursion)

c# — iterative dfs using explicit stack 
// Use this when recursion depth is too large (stack overflow risk)
public static List<int> IterativeDFS(TreeNode? root)
{
    var result = new List<int>();
    if (root == null) return result;

    var stack = new Stack<TreeNode>();
    stack.Push(root);  // start with root on stack

    while (stack.Count > 0)
    {
        var node = stack.Pop();  // take top of stack
        result.Add(node.Val);    // process it

        // Push RIGHT first so LEFT is processed first (LIFO)
        if (node.Right != null) stack.Push(node.Right);
        if (node.Left != null)  stack.Push(node.Left);
    }

    return result;
    // Returns: [4, 2, 1, 3, 6, 5, 7]  (pre-order)
}

    }

7. Detect Cycle in Directed Graph

c# — cycle detection using dfs (for dependency checking) 
// Team Watson: detect circular NuGet dependencies
// Uses "recursion stack" — tracks current DFS path
public class CycleDetector
{
    private Dictionary<string, List<string>> _deps;

    public bool HasCycle(string node,
                          HashSet<string> visited,
                          HashSet<string> inStack)
    {
        visited.Add(node);
        inStack.Add(node);  // add to current path

        foreach (var dep in _deps.GetValueOrDefault(node, new()))
        {
            if (!visited.Contains(dep))
            {
                if (HasCycle(dep, visited, inStack))
                    return true;  // cycle found deeper
            }
            else if (inStack.Contains(dep))
            {
                return true;  // back edge = cycle!
            }
        }

        inStack.Remove(node);  // remove from path on backtrack
        return false;
    }
}

8. Find File in Directory (Real-world DFS)

c# — file system dfs using directoryinfo 
// Real Team Watson use case: find all .csproj files in a solution
public static IEnumerable<string> FindFiles(string rootPath,
                                              string extension)
{
    var dir = new DirectoryInfo(rootPath);

    // DFS: process files in this dir first
    foreach (var file in dir.GetFiles($"*{extension}"))
        yield return file.FullName;

    // Then recurse into each subdirectory (going deeper)
    foreach (var subDir in dir.GetDirectories())
    {
        foreach (var found in FindFiles(subDir.FullName, extension))
            yield return found;
    }
}

    Console.WriteLine(p);

// ======= ITERATIVE DFS TEMPLATE =======
void IterativeDFS(TreeNode? root)
{
    if (root == null) return;
    var stack = new Stack<TreeNode>();
    stack.Push(root);
    while (stack.Count > 0)
    {
        var node = stack.Pop();
        // process node here
        if (node.Right != null) stack.Push(node.Right);
        if (node.Left != null)  stack.Push(node.Left);
    }
}

Complexity Summary

⏱️ Time Complexity

  • Tree DFS: O(n) — visit every node once
  • Graph DFS: O(V + E) — vertices + edges
  • Backtracking: O(b^d) — branching factor, depth

💾 Space Complexity

  • Balanced tree: O(log n) — call stack height
  • Skewed tree: O(n) — all nodes in stack
  • Graph: O(V) — visited set + stack

Common Mistakes to Avoid

  • Forgetting the null check base case in recursive DFS → NullReferenceException
  • No visited set in graph DFS → infinite loop on cycles
  • Pushing LEFT before RIGHT in iterative DFS → wrong order (push RIGHT first!)
  • Using DFS for shortest path in graphs → wrong answer (use BFS)
  • Recursion on very deep trees without stack overflow protection
  • Mutating visited set without backtracking in "all paths" problems

Signs You Need DFS

  • Problem involves a tree or graph structure
  • You need to explore "all possibilities" or "all paths"
  • Problem mentions "reachability" or "connected"
  • You need to detect cycles or dependencies
  • You're solving a puzzle where you try options and backtrack
  • You need to visit every node exactly once
  • Problem involves hierarchical/recursive structure
