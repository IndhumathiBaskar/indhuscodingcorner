🛫 What is Amazon EventBridge?

EventBridge is AWS's fully managed serverless event bus. It receives events from AWS services, your own applications, and SaaS partners — then routes each event to the right target based on rules you define. No polling, no coupling between producers and consumers, no infrastructure to manage.

💡 Real-life analogy — Air Traffic Control

EventBridge is like an air traffic control (ATC) system. Planes (events) arrive from different airports (AWS services, your apps, SaaS tools). The ATC tower (EventBridge bus) doesn't carry passengers — it receives flight plans and routes each flight to the correct runway (target). If a flight matches "international AND heavy aircraft" → runway 28L (Lambda). If it matches "domestic small" → runway 09R (SQS). The pilot (event producer) doesn't know which runway they'll land on — ATC decides everything.

EventBridge vs SNS vs SQS — The Right Tool

🛫 Choose EventBridge when

  • Content-based routing — route by event data
  • Events from AWS services or SaaS partners
  • Complex event filtering and transformation
  • Scheduled tasks (cron replacement)
  • Schema discovery and validation
  • 200+ target types in one rule
  • Audit log of all events (Archive + Replay)

📡 Keep SNS / SQS when

  • Simple fan-out to many subscribers (SNS)
  • Durable message queue with retry (SQS)
  • High-throughput message delivery (SNS)
  • SMS / Email / Mobile push (SNS)
  • Messages with 14-day retention needed (SQS)
  • FIFO ordering guarantees (SQS FIFO)

📨 Events & The Event Bus

Everything in EventBridge revolves around events — JSON objects describing something that happened. The event bus is the central receiver that accepts events and evaluates rules against them.

Anatomy of an EventBridge Event

📨 OrderPlaced event — every field explained

The Three Event Buses

🔵 Default Event Bus

  • Exists in every AWS account automatically
  • Receives ALL AWS service events (EC2, S3, RDS, etc.)
  • Also accepts custom events from your app
  • Cannot be deleted
  • Good for: reacting to AWS infrastructure events
  • Example: EC2 instance stopped → notify ops team

🟣 Custom Event Bus

  • You create for your own events
  • Isolate events by domain or microservice
  • Cross-account access via resource policy
  • Up to 100 custom buses per account
  • Good for: microservices event-driven architecture
  • Example: orders-bus, payments-bus, users-bus

🔌 SaaS Partner Event Bus

  • Provided by SaaS vendors (Shopify, Stripe, etc.)
  • Partner publishes events directly to your AWS account
  • No webhook infrastructure needed on your side
  • Configure in AWS Marketplace
  • Good for: e-commerce, payments, CRM integrations
  • Example: Shopify order.created → your Lambda

📦 Event Archive

  • Archive all or filtered events on a bus
  • Replay archived events at any time
  • Retention: days to indefinite
  • Use for: debugging, re-processing after bug fix
  • Replay at controlled rate to avoid overwhelming targets
  • Like a flight recorder — every event captured

📋 Rules & Event Patterns

Rules are the heart of EventBridge. Each rule contains an event pattern (what to match) and one or more targets (where to route). Patterns are JSON objects — if the event structure matches the pattern, the rule fires.

Event Pattern Matching — How It Works

Pattern matching compares pattern fields against event fields. A field in the pattern must exist in the event and the value must match. Fields NOT in the pattern are ignored.

🎯 Pattern: {"source":["myapp.orders"],"detail":{"status":["placed"]}} — does this event match?
✅ Full match source="myapp.orders", detail.status="placed", detail.amount=2499 ✅ MATCHES
✅ Extra fields OK source="myapp.orders", detail.status="placed", detail.currency="INR", detail.region="india" ✅ MATCHES
❌ Wrong status source="myapp.orders", detail.status="shipped" (not "placed") ⛔ NO MATCH
❌ Wrong source source="myapp.payments", detail.status="placed" ⛔ NO MATCH
❌ Missing source detail.status="placed" (no source field in event) ⛔ NO MATCH

All Pattern Operators

🔤 Value Matching

  • ["placed"] — exact match (string)
  • ["placed","shipped"] — OR: matches either
  • [{"prefix":"order."}] — starts with
  • [{"suffix":".failed"}] — ends with
  • [{"anything-but":"test"}] — not this value
  • [{"anything-but":["dev","staging"]}] — not any of these

🔢 Numeric & Existence

  • [{"numeric":["=",100]}] — equals
  • [{"numeric":[">",1000]}] — greater than
  • [{"numeric":[">=",500,"<=",5000]}] — range
  • [{"exists":true}] — field must exist
  • [{"exists":false}] — field must NOT exist
  • [{"cidr":"10.0.0.0/24"}] — IP range match

Input Transformation — Reshape Before Delivery

🔄 Transform the event before it reaches the target

Sometimes your target (Lambda, SQS, HTTP endpoint) needs a different event shape than what EventBridge delivers. Input Transformer lets you extract fields and build a new payload:

Input paths: extract values → {"orderId": "$.detail.orderId", "amount": "$.detail.amount"}
Template: build new JSON → {"order": "<orderId>", "total": <amount>, "source": "eventbridge"}

Result: your Lambda receives {"order":"ord_201","total":2499,"source":"eventbridge"} — clean, no AWS envelope wrapping.

🔗 EventBridge Pipes & Scheduler

Pipes and Scheduler are two powerful EventBridge features that go beyond simple event routing — connecting sources directly to targets with enrichment, and triggering events on a time schedule.

EventBridge Pipes — Point-to-Point with Enrichment

💡 What are Pipes?

Pipes create point-to-point integrations between event sources and targets — with optional filtering, enrichment (add data), and transformation along the way. Unlike the event bus (which is many-to-many via rules), a Pipe is a single, fully-managed integration channel.

📥 Pipe Source
  • SQS queue (polls automatically)
  • Kinesis stream
  • DynamoDB stream
  • Kafka topic (MSK or self-managed)
  • API Gateway, MQ, RabbitMQ
↓ optional
🔍 Filter (optional)
  • Same pattern syntax as EventBridge rules
  • Drop events that don't match before enrichment
  • Save cost — don't invoke enrichment for irrelevant events
↓ optional
✨ Enrichment (optional)
  • Call Lambda / Step Functions / API GW / EventBridge bus
  • Add data to the event before delivering to target
  • Example: SQS message has userId → Lambda looks up user details → enriched event goes to target
  • Keeps target code clean — no lookup logic needed
↓
🎯 Pipe Target
  • Lambda, SQS, SNS, Step Functions, EventBridge bus
  • HTTP endpoint (API Destination)
  • Receives the enriched, filtered, transformed event

EventBridge Scheduler — Next-Gen Cron

Scheduler replaces CloudWatch Events cron rules. It creates one-time or recurring schedules to invoke any AWS target — at any scale, with flexible time zone support.

Schedule Expressions

rate rate(5 minutes) Every 5 minutes — simplest recurring schedule
rate rate(1 day) Daily — good for batch jobs, cleanup tasks, reports
cron cron(0 2 * * ? *) Every day at 2:00 AM UTC — maintenance window
cron cron(0 9 ? * MON-FRI *) 9 AM on weekdays only — business-hours jobs
cron cron(0 0 1 * ? *) Midnight on 1st of every month — monthly reports
one-time at(2025-12-31T23:59:00) Exactly once at specified datetime — fire-and-forget

EventBridge Service — Publish & Manage

EventBridgeService.cs 
using Amazon.EventBridge;
using Amazon.EventBridge.Model;
using System.Text.Json;

public class EventBridgeService
{
    private readonly IAmazonEventBridge _eb;
    private const string BusName = "myapp-orders-bus";

    public EventBridgeService(IAmazonEventBridge eb) => _eb = eb;

    // ── Publish single event ───────────────────────
    public async Task<string> PublishAsync<T>(
        string source,
        string detailType,
        T      detail)
    {
        var response = await _eb.PutEventsAsync(
            new PutEventsRequest
        {
            Entries = [new PutEventsRequestEntry
            {
                EventBusName = BusName,
                Source       = source,     // "myapp.orders"
                DetailType   = detailType, // "Order Placed"
                Detail       = JsonSerializer.Serialize(detail),
                Time         = DateTime.UtcNow
            }]
        });

        if (response.FailedEntryCount > 0)
        {
            var err = response.Entries[0];
            throw new Exception(
                $"EventBridge publish failed: {err.ErrorCode} - {err.ErrorMessage}");
        }

        return response.Entries[0].EventId;
    }

    // ── Publish batch (up to 10 events) ───────────
    public async Task PublishBatchAsync(
        IEnumerable<EventRecord> events)
    {
        var entries = events.Select(e =>
            new PutEventsRequestEntry
        {
            EventBusName = BusName,
            Source       = e.Source,
            DetailType   = e.DetailType,
            Detail       = JsonSerializer.Serialize(e.Detail)
        }).ToList();

        // EventBridge allows up to 10 per PutEvents call
        foreach (var batch in entries.Chunk(10))
        {
            var response = await _eb.PutEventsAsync(
                new PutEventsRequest { Entries = batch.ToList() });

            if (response.FailedEntryCount > 0)
                throw new Exception(
                    $"{response.FailedEntryCount} events failed");
        }
    }

    // ── Create a rule programmatically ────────────
    public async Task<string> CreateRuleAsync(
        string name, string patternJson)
    {
        var response = await _eb.PutRuleAsync(new PutRuleRequest
        {
            Name          = name,
            EventBusName  = BusName,
            EventPattern  = patternJson,
            State         = RuleState.ENABLED,
            Description   = $"Created by app at {DateTime.UtcNow:O}"
        });
        return response.RuleArn;
    }

    // ── Add Lambda target to rule ─────────────────
    public async Task AddLambdaTargetAsync(
        string ruleName, string lambdaArn)
    {
        await _eb.PutTargetsAsync(new PutTargetsRequest
        {
            Rule         = ruleName,
            EventBusName = BusName,
            Targets      = [new Target
            {
                Id  = "lambda-target-1",
                Arn = lambdaArn,
                // Optional: transform input
                InputTransformer = new InputTransformer
                {
                    InputPathsMap = new()
                    {
                        ["orderId"] = "$.detail.orderId",
                        ["amount"]  = "$.detail.amount"
                    },
                    InputTemplate =
                        "{\"order\":\"<orderId>\",\"total\":<amount>}"
                },
                // Optional: DLQ for failed deliveries
                DeadLetterConfig = new DeadLetterConfig
                    { Arn = "arn:aws:sqs:...:OrderEvents-DLQ" }
            }]
        });
    }
}

public record OrderDetail(

{

            "Event: {Type} from {Source} at {Time}",
            evt.DetailType, evt.Source, evt.Time);

        var order = evt.Detail;
        ctx.Logger.LogInformation(
            "Processing order {Id} for ${Amount}",
            order.OrderId, order.Amount);

        // Your business logic here...
        await Task.CompletedTask;
    }
}

// ── Scheduler Target Handler ─────────────────────────
// Scheduler injects its own payload
public class NightlyCleanupFunction
{
    public async Task HandleAsync(
        JsonElement input, ILambdaContext ctx)
    {
        // Get scheduled time from Scheduler context

            .Custom?.GetValueOrDefault("scheduled-time");

        ctx.Logger.LogInformation(
            "Nightly cleanup at {Time}", scheduledTime);

        await Task.CompletedTask;
    }

✅ Best Practices & Common Mistakes

Event Design

  • Always include source and detail-type — they are the primary routing fields for rules
  • Use consistent naming: source = "myapp.domain" (e.g. myapp.orders), detail-type = "Noun Verb" (e.g. Order Placed)
  • Include a correlationId in every event detail for distributed tracing across services
  • Include a version field in detail — allows schema evolution without breaking consumers
  • Keep events as facts about what happened — not commands telling consumers what to do
  • Use ISO 8601 timestamps — EventBridge adds its own "time" but include your own event time in detail

Rules & Routing

  • Create separate custom buses per domain — don't put all events on the default bus
  • Enable event archive on every production bus — enables replay after bugs
  • Always configure DLQ on every target — never lose events silently on failure
  • Test patterns before deploying — use the EventBridge console Pattern Tester
  • Use Input Transformer — deliver only what the target needs, not the full AWS envelope
  • Use Scheduler (not CloudWatch Events rules) for all new scheduled work

Operations & Monitoring

  • Enable CloudWatch metrics: FailedInvocations, ThrottledRules, MatchedEvents
  • Alert on FailedInvocations > 0 — event delivery is failing
  • Alert on DLQ depth > 0 — events being dropped after retries
  • Use X-Ray tracing to follow events across Lambda invocations
  • Use Schema Registry schema discovery to generate typed C# classes
  • Tag all buses and rules — enables cost allocation per team or domain

Common Mistakes to Avoid

⚠️ Top EventBridge Mistakes

❌ Using default bus for all custom events — mixes infrastructure alerts with app events, hard to manage
❌ No DLQ on targets — failed events silently dropped after retries exhaust
❌ No event archive — cannot replay events after fixing a bug
❌ Events as commands ("ProcessOrder") — use facts ("OrderPlaced") — consumers decide what to do
❌ Not using Input Transformer — Lambda receives AWS envelope JSON, not just your payload
❌ Forgetting to grant EventBridge permission to invoke Lambda — rule created but Lambda never fires
❌ Using CloudWatch Events cron when Scheduler exists — Scheduler has timezone support, one-time schedules, millions of schedules
❌ No correlationId in events — impossible to trace a request across multiple EventBridge-triggered services
