AWS CloudWatch

Monitor metrics, collect logs, set alarms, and gain full observability across your AWS resources — all in one place.

🏗️ What is CloudWatch?

A native AWS monitoring and observability service that collects metrics, logs, and events in real time.

One-liner: CloudWatch = Metrics + Logs + Alarms + Dashboards + Events all under one roof for your AWS workloads.
📈 Metrics

Time-series numeric data points (CPU, memory, requests/sec) published by AWS services or custom apps.

📋 Logs

Centralised log storage via Log Groups & Log Streams. Query with CloudWatch Logs Insights.

🔔 Alarms

Trigger SNS notifications, Auto Scaling actions, or EC2 actions when metrics cross thresholds.

📊 Dashboards

Custom visualisations — line charts, number widgets, alarm status boards — shareable across teams.

⚡ Events / Rules

Schedule tasks or react to AWS resource state changes via EventBridge (formerly CW Events).

🎯 Synthetics

Canary scripts that monitor APIs and UI endpoints on a schedule — catch issues before users do.

⚙️ Core Architecture

How AWS services and your apps push data into CloudWatch.

Data Flow Summary

Sources → EC2, Lambda, RDS, ECS, EKS, API Gateway, custom apps (via SDK/agent)

Collection → CloudWatch Agent / PutMetricData API / Log Groups

Storage → Metrics retained 15 months (1-min granularity 15 days, 5-min 63 days, 1-hr 15 months)

Action → Alarms → SNS / Auto Scaling / SSM / Lambda; Dashboards → Visualise

📈 Metrics Deep Dive

Every data point has a namespace, metric name, dimensions, and a timestamp.

Key terms: Namespace (e.g. AWS/EC2) > Metric Name (e.g. CPUUtilization) > Dimension (e.g. InstanceId=i-1234)

ALGORITHM PublishCustomMetric
  INPUT: namespace, metricName, value, unit, dimensions[]
  STEP 1: Build MetricDatum { MetricName, Value, Unit, Dimensions, Timestamp }
  STEP 2: Call PutMetricData(Namespace, [MetricDatum])
  STEP 3: CloudWatch stores → aggregates at 1-min resolution

  OUTPUT: Metric visible in Console / GetMetricData API

📋 CloudWatch Logs

Centralised log management — collect, store, query, and route logs from any AWS service or app.

🗂️ Log Group

Logical container for a set of log streams, e.g. /aws/lambda/myFunction. Set retention (1 day – 10 years) here.

📄 Log Stream

Sequence of log events from a single source (one Lambda instance, one EC2 pod). Auto-created per resource.

📝 Log Event

A single record: timestamp + message. Max 256 KB per event.

Logs Insights Query Syntax

CloudWatch Logs Insights Find Lambda errors in last 1hr 
-- Filter errors from Lambda logs
fields @timestamp, @message, @logStream
| filter @message like /ERROR/
| sort @timestamp desc
| limit 50

-- Calculate error rate per 5 minutes
fields @timestamp, @message
| stats count(*) as totalRequests,
        sum(if(@message like /ERROR/, 1, 0)) as errors
    by bin(5m)
| sort @timestamp desc

-- Parse JSON structured logs
fields @timestamp, @message
| parse @message '{"level":"*","msg":"*","duration":*}' as level, msg, dur
| filter level = "ERROR"
| stats avg(dur) as avgDuration by msg

🔔 CloudWatch Alarms

Watch a metric over a period, compare to a threshold, and take automated actions.

Alarm States

✅ OKMetric within threshold
⚠️ INSUFFICIENT_DATANot enough data points yet
🚨 ALARMThreshold breached
OK⇄INSUFFICIENT_DATA⇄ALARM
🎛️ Alarm Configuration

Metric – which metric to watch (namespace + name + dimensions)

Period – evaluation window (60s, 300s…)

Statistic – Average / Sum / Max / p99…

Threshold – the value to compare against

Datapoints to alarm – M out of N evaluation periods

⚡ Actions on ALARM

📢 SNS notification → email / SMS / Lambda

📈 Auto Scaling → scale out / in EC2 or ECS

🔌 EC2 actions → stop / terminate / reboot / recover

🤖 Systems Manager → run automation runbook

Composite Alarms

Composite Alarm – combine multiple alarms with AND/OR logic. Reduce alert noise: only fire when BOTH CPU > 80% AND Latency > 500ms are in ALARM state.
COMPOSITE ALARM LOGIC:
  HighCPU = CPUUtilization > 80% for 3/3 periods
  HighLatency = Latency p99 > 500ms for 2/3 periods

  RULE: CriticalServiceAlarm = HighCPU AND HighLatency
  ACTION: Trigger PagerDuty via SNS only when BOTH fire

🧩 C# / .NET SDK Examples

Using AWSSDK.CloudWatch and AWSSDK.CloudWatchLogs NuGet packages.

C# Publish a custom metric 
using Amazon.CloudWatch;
using Amazon.CloudWatch.Model;

var client = new AmazonCloudWatchClient();

await client.PutMetricDataAsync(new PutMetricDataRequest
{
    Namespace = "MyApp/Orders",
    MetricData =
    [
        new MetricDatum
        {
            MetricName = "OrdersPlaced",
            Value      = 1,
            Unit       = StandardUnit.Count,
            Timestamp  = DateTime.UtcNow,
            Dimensions =
            [
                new Dimension { Name = "Environment", Value = "Prod" }
            ]
        }
    ]
});
C# Create a CloudWatch Alarm 
await client.PutMetricAlarmAsync(new PutMetricAlarmRequest
{
    AlarmName          = "HighCPUAlarm",
    AlarmDescription   = "Fires when EC2 CPU exceeds 80%",
    Namespace          = "AWS/EC2",
    MetricName         = "CPUUtilization",
    Dimensions         =
    [
        new Dimension { Name = "InstanceId", Value = "i-0abc1234" }
    ],
    Statistic          = Statistic.Average,
    Period             = 300,               // 5 minutes
    EvaluationPeriods  = 3,
    DatapointsToAlarm  = 2,                // 2 of 3 periods
    Threshold          = 80.0,
    ComparisonOperator = ComparisonOperator.GreaterThanThreshold,
    TreatMissingData   = "notBreaching",
    AlarmActions       =
    [
        "arn:aws:sns:us-east-1:123456789:MyAlarmTopic"
    ]
});
C# Write logs to CloudWatch Logs 
using Amazon.CloudWatchLogs;
using Amazon.CloudWatchLogs.Model;

var logClient = new AmazonCloudWatchLogsClient();

// Ensure log group + stream exist
await logClient.CreateLogGroupAsync(new CreateLogGroupRequest
    { LogGroupName = "/myapp/orders" });
await logClient.CreateLogStreamAsync(new CreateLogStreamRequest
    { LogGroupName = "/myapp/orders", LogStreamName = "stream-1" });

// Put log events
await logClient.PutLogEventsAsync(new PutLogEventsRequest
{
    LogGroupName  = "/myapp/orders",
    LogStreamName = "stream-1",
    LogEvents     =
    [
        new InputLogEvent
        {
            Timestamp = DateTimeOffset.UtcNow.ToUnixTimeMilliseconds(),
            Message   = """{"level":"INFO","msg":"Order placed","orderId":"ORD-001"}"""
        }
    ]
});
C# Query metrics with GetMetricData 
var response = await client.GetMetricDataAsync(new GetMetricDataRequest
{
    StartTime = DateTime.UtcNow.AddHours(-1),
    EndTime   = DateTime.UtcNow,
    MetricDataQueries =
    [
        new MetricDataQuery
        {
            Id    = "m1",
            Label = "Lambda Errors",
            MetricStat = new MetricStat
            {
                Metric = new Metric
                {
                    Namespace  = "AWS/Lambda",
                    MetricName = "Errors",
                    Dimensions =
                    [
                        new Dimension { Name = "FunctionName", Value = "myFunction" }
                    ]
                },
                Period = 300,
                Stat   = "Sum"
            }
        }
    ]
});

foreach (var result in response.MetricDataResults)
    Console.WriteLine($"[{result.Id}] Values: {string.Join(", ", result.Values)}");

Collects metrics & logs from EKS, ECS, and Kubernetes clusters.

Automated problem detection for .NET, SQL Server, IIS, Java apps running on EC2 / ECS.

Detects anomalies, surfaces correlated metrics + logs, and creates alarms automatically.

Identifies internet issues affecting specific cities, ISPs, or geographies.

💡 Best Practices

Production-grade CloudWatch setup for reliability and cost efficiency.

✅ Metrics Best Practices

Use dimensions wisely — high-cardinality dims (like userId) explode metric count and cost.

Prefer EMF (Embedded Metric Format) in Lambda for structured metrics without extra API calls.

Enable detailed monitoring on EC2 (1-min) for critical workloads; default is 5-min.

✅ Logs Best Practices

Always set log retention on log groups — default is "never expire" which costs money.

Use structured JSON logs (not free-text) so Logs Insights can parse fields efficiently.

Route high-volume logs to S3 via subscription filter for cheap long-term storage.

✅ Alarms Best Practices

Use M of N datapoints (e.g. 3 of 5) to avoid transient spike alerts.

Set TreatMissingData = notBreaching for low-traffic metrics to avoid false ALARM.

Use Composite Alarms to reduce noise — only page on-call when multiple signals align.

✅ Cost Optimisation

Custom metrics cost $0.30/metric/month — batch PutMetricData (max 1000 per call).

Logs Insights charges per GB scanned — filter time window precisely.

Use metric math to derive new metrics from existing ones instead of publishing new ones.

🏁 Summary

Quick recap of CloudWatch's core pillars.

📈 Metrics

Time-series data from AWS services + custom PutMetricData. Namespace → MetricName → Dimensions. Retained up to 15 months.

📋 Logs

Log Group → Log Stream → Log Event hierarchy. Metric filters extract metrics from logs. Insights for ad-hoc querying.

🔔 Alarms

3 states: OK / INSUFFICIENT_DATA / ALARM. M-of-N evaluation. Actions: SNS, Auto Scaling, EC2, SSM. Composite alarms for noise reduction.

📊 Dashboards

Cross-account, cross-region. Automatic dashboards per service. JSON-based widget definitions. Shareable via IAM policies.

🔍 Insights

Container Insights (EKS/ECS), Application Insights (.NET/Java), Internet Monitor, Evidently (feature flags), Synthetics (canaries).

💰 Cost Tips

Set log retention. Batch PutMetricData. Use metric math. Filter Insights queries by time window. Avoid high-cardinality dimensions.
