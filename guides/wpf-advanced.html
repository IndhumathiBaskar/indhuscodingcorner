🏛️ MVVM Pattern

👶 What is MVVM?

MVVM stands for Model–View–ViewModel. It is a pattern for organising your WPF code so the UI (View) is completely separated from the logic (ViewModel). The View knows nothing about how data is fetched or saved. The ViewModel knows nothing about buttons or text boxes. They communicate only through data binding.

💡 Real World Analogy

A restaurant menu is the View — it displays food options. The kitchen is the Model — it holds the actual ingredients and recipes. The waiter is the ViewModel — takes your order from the menu and communicates it to the kitchen, then brings the result back to display. The menu never goes into the kitchen. The kitchen never writes on the menu.

The Three Layers

🏛️ MVVM Architecture — Data Flow

ViewModelBase — Your Foundation Class

Every ViewModel inherits from a base class that implements INotifyPropertyChanged. Write this once and reuse everywhere.

C# — ViewModelBase.cs 
using System.ComponentModel;
using System.Runtime.CompilerServices;

namespace MyApp.ViewModels;

/// Base class for all ViewModels.
/// Provides INotifyPropertyChanged so the UI auto-updates.
public abstract class ViewModelBase : INotifyPropertyChanged
{
    public event PropertyChangedEventHandler? PropertyChanged;

    // Call this in any property setter to notify WPF of changes
    protected void OnPropertyChanged([CallerMemberName] string? name = null)
        => PropertyChanged?.Invoke(this, new PropertyChangedEventArgs(name));

    // Set a backing field, notify if changed, return true if changed
    protected bool SetProperty<T>(ref T field, T value,
                                   [CallerMemberName] string? name = null)
    {
        if (EqualityComparer<T>.Default.Equals(field, value)) return false;
        field = value;
        OnPropertyChanged(name);
        return true;
    }
}

A Complete ViewModel

C# — ContactsViewModel.cs (full example) 
using System.Collections.ObjectModel;
using System.Windows.Input;

namespace MyApp.ViewModels;

public class ContactsViewModel : ViewModelBase
{
    // ── Backing fields ──────────────────────────────────────────
    private string   _searchText   = "";
    private Contact? _selectedContact;
    private bool     _isLoading;

    // ── Properties (bound to UI) ────────────────────────────────
    public ObservableCollection<Contact> Contacts { get; } = new();

    public string SearchText
    {
        get => _searchText;
        set
        {
            if (SetProperty(ref _searchText, value))
                FilterContacts();   // re-filter when search changes
        }
    }

    public Contact? SelectedContact
    {
        get => _selectedContact;
        set => SetProperty(ref _selectedContact, value);
    }

    public bool IsLoading
    {
        get => _isLoading;
        set => SetProperty(ref _isLoading, value);
    }

    // Computed property — no backing field needed
    public bool HasContacts => Contacts.Count > 0;
    public string Title    => $"Contacts ({Contacts.Count})";

    // ── Commands (bound to Buttons) ─────────────────────────────
    public ICommand AddCommand    { get; }
    public ICommand DeleteCommand { get; }
    public ICommand RefreshCommand{ get; }

    public ContactsViewModel()
    {
        AddCommand     = new RelayCommand(_ => AddContact());
        DeleteCommand  = new RelayCommand(_ => DeleteContact(),
                                            _ => SelectedContact != null);
        RefreshCommand = new AsyncRelayCommand(LoadContactsAsync);

        // Load initial data
        _ = LoadContactsAsync();
    }

    private async Task LoadContactsAsync()
    {
        IsLoading = true;
        Contacts.Clear();
        var data = await _contactService.GetAllAsync();
        foreach (var c in data) Contacts.Add(c);
        IsLoading = false;
        OnPropertyChanged(nameof(HasContacts));
        OnPropertyChanged(nameof(Title));
    }

    private void AddContact()    { /* open add dialog */ }
    private void DeleteContact() { if (SelectedContact != null) Contacts.Remove(SelectedContact); }
    private void FilterContacts(){ /* filter logic */ }

    private readonly IContactService _contactService = new ContactService();
}

MVVM Rules — What Goes Where

🖼️

View (XAML)

XAML only. Bindings to ViewModel. No business logic. Code-behind can have UI-only code (animations, focus).

⚙️

ViewModel (C#)

Properties with INPC. ICommand for buttons. Calls services. No Button, TextBox, MessageBox references.

📦

Model (C#)

Plain C# classes. Data shapes. No WPF, no ViewModel references. Can be shared with non-WPF projects.

🔧

Service (C#)

Database access, API calls, file I/O. Injected into ViewModel. Returns Model objects.

🔄

Repository

Abstracts data access. ViewModel depends on interface, not concrete class — easy to mock in tests.

📬

Messenger/EventAgg.

ViewModels talk to each other without direct references. Publish/subscribe pattern for loose coupling.

⚡ Commands Deep Dive

👶 Why Commands Instead of Click Events?

With events, your button logic lives in code-behind, tightly coupled to the UI. With Commands, the logic lives in your ViewModel — the button just binds to it. This means you can unit test your logic without ever touching the UI, and reuse commands on multiple buttons, keyboard shortcuts, and menu items at once.

RelayCommand — Synchronous

C# — RelayCommand (full production version) 
public class RelayCommand : ICommand
{
    private readonly Action<object?>     _execute;
    private readonly Func<object?, bool>? _canExecute;

    public RelayCommand(Action<object?>     execute,
                        Func<object?, bool>? canExecute = null)
    {
        _execute    = execute    ?? throw new ArgumentNullException(nameof(execute));
        _canExecute = canExecute;
    }

    // WPF asks this before each user interaction — enables/disables button
    public bool CanExecute(object? parameter)
        => _canExecute?.Invoke(parameter) ?? true;

    public void Execute(object? parameter)
        => _execute(parameter);

    // Hook into WPF's CommandManager — auto re-evaluates CanExecute
    public event EventHandler? CanExecuteChanged
    {
        add    => CommandManager.RequerySuggested += value;
        remove => CommandManager.RequerySuggested -= value;
    }

    // Call this manually if CanExecute depends on something WPF can't detect
    public void RaiseCanExecuteChanged()
        => CommandManager.InvalidateRequerySuggested();
}

AsyncRelayCommand — For async Operations

C# — AsyncRelayCommand (handles async properly) 
public class AsyncRelayCommand : ICommand
{
    private readonly Func<Task>         _execute;
    private readonly Func<bool>?        _canExecute;
    private bool _isExecuting;

    public AsyncRelayCommand(Func<Task> execute, Func<bool>? canExecute = null)
    {
        _execute    = execute;
        _canExecute = canExecute;
    }

    public bool CanExecute(object? parameter)
        => !_isExecuting && (_canExecute?.Invoke() ?? true);

    public async void Execute(object? parameter)
    {
        _isExecuting = true;
        CommandManager.InvalidateRequerySuggested(); // disable button
        try
        {
            await _execute();
        }
        finally
        {
            _isExecuting = false;
            CommandManager.InvalidateRequerySuggested(); // re-enable button
        }
    }

    public event EventHandler? CanExecuteChanged
    {
        add    => CommandManager.RequerySuggested += value;
        remove => CommandManager.RequerySuggested -= value;
    }
}

// Usage in ViewModel:
SaveCommand = new AsyncRelayCommand(
    execute:    SaveAsync,             // button auto-disables while saving
    canExecute: () => IsFormValid
);

private async Task SaveAsync()
{
    await _service.SaveAsync(CurrentContact);
    MessageBox.Show("Saved!");
}

🔧 Dependency Properties

👶 What is a Dependency Property?

A Dependency Property is WPF's special type of property that supports data binding, styles, animations, and inheritance. Every WPF property you've bound to — Width, Background, Text — is a Dependency Property. When you create custom controls, you define your own Dependency Properties so they can be bound to, styled, and animated just like built-in ones.

💡 Analogy

A normal C# property is like writing a value on a sticky note attached to an object. A Dependency Property is like having a smart property that can ask several sources for its value — "What did the Style say? What did the animation set? What did the binding provide?" — and uses the highest-priority answer.

Value Priority — How WPF Decides the Final Value

① Highest — Local Value (set directly on the element)
  • button.Background = Brushes.Red; or Background="Red" in XAML
↓ if not set, use
② Animation value (while animating)
  • Storyboard animations temporarily override everything below
↓ if not set, use
③ Style Setter value
  • Value set via a Style's Setter
↓ if not set, use
④ Inherited from parent element
  • FontSize, FontFamily, DataContext inherit down the visual tree
↓ if not set, use
⑤ Lowest — Default value (set when property is registered)
  • e.g. Button.Background default is transparent

Creating a Dependency Property

C# — Custom control with a Dependency Property 
public class RatingControl : Control
{
    // ── Step 1: Register the dependency property ──────────────────
    // This is the "real" property that WPF knows about
    public static readonly DependencyProperty ValueProperty =
        DependencyProperty.Register(
            name:             "Value",          // must match CLR property name
            propertyType:     typeof(int),
            ownerType:        typeof(RatingControl),
            typeMetadata:     new PropertyMetadata(
                defaultValue:         0,                // default = 0 stars
                propertyChangedCallback: OnValueChanged // callback on change
            ),
            validateValueCallback: ValidateValue      // validation
        );

    // ── Step 2: Create CLR wrapper property ──────────────────────
    // This is the C# property developers use in XAML and code
    public int Value
    {
        get => (int)GetValue(ValueProperty);
        set => SetValue(ValueProperty, value);
    }

    // ── Step 3: Callback when value changes ──────────────────────
    private static void OnValueChanged(
        DependencyObject d, DependencyPropertyChangedEventArgs e)
    {
        var control = (RatingControl)d;
        control.UpdateStars((int)e.NewValue);
    }

    // ── Step 4: Validation callback ──────────────────────────────
    private static bool ValidateValue(object value)
        => (int)value is >= 0 and <= 5;  // only 0-5 stars allowed

    private void UpdateStars(int value) { /* redraw stars */ }

    // ── Attached Property (a property that belongs to another element) ──
    // Example: Grid.Row, Grid.Column are attached properties on Grid
    public static readonly DependencyProperty IsHighlightedProperty =
        DependencyProperty.RegisterAttached(
            "IsHighlighted",
            typeof(bool),
            typeof(RatingControl),
            new PropertyMetadata(false));

    // Attached properties need static get/set accessors
    public static bool GetIsHighlighted(DependencyObject obj)
        => (bool)obj.GetValue(IsHighlightedProperty);

    public static void SetIsHighlighted(DependencyObject obj, bool value)
        => obj.SetValue(IsHighlightedProperty, value);
}

💡 Never Put Logic in the CLR Wrapper

The CLR wrapper (get / set) calls GetValue and SetValue — nothing else. WPF sometimes calls SetValue directly, bypassing your wrapper. Any logic goes in the PropertyChangedCallback, not the wrapper setter.

🎛️ Custom Controls & User Controls

👶 User Control vs Custom Control

A User Control is like a reusable form section — you compose existing controls together (a label + textbox + button as one unit). A Custom Control is a fully new control with its own look-and-feel defined in a ControlTemplate — like building a star rating widget or a custom progress ring from scratch.

✨ Animations

👶 How WPF Animations Work

WPF animations change a property's value over time. You say "change Opacity from 0 to 1 over 0.3 seconds" and WPF does it smoothly. Animations are defined in XAML using Storyboard and triggered by events or triggers. You can also start animations from C# code.

Storyboard — Container for Animations

XAML — Fade in animation on load 
<Border x:Name="myPanel"
        Background="#1E1A30"
        Opacity="0">

    <Border.Triggers>
        <!-- Run this animation when the element loads -->
        <EventTrigger RoutedEvent="Loaded">
            <BeginStoryboard>
                <Storyboard>
                    <!-- Fade in: Opacity 0 → 1 over 0.5 seconds -->
                    <DoubleAnimation
                        Storyboard.TargetProperty="Opacity"
                        From="0" To="1"
                        Duration="0:0:0.5"
                        EasingFunction="{StaticResource EaseOut}"/>
                    <!-- Slide up: move from 20px below to original position -->
                    <DoubleAnimation
                        Storyboard.TargetProperty="(UIElement.RenderTransform).(TranslateTransform.Y)"
                        From="20" To="0"
                        Duration="0:0:0.5"/>
                </Storyboard>
            </BeginStoryboard>
        </EventTrigger>
    </Border.Triggers>

    <Border.RenderTransform>
        <TranslateTransform/>
    </Border.RenderTransform>

    <TextBlock Text="I fade in and slide up!"/>
</Border>

Animations from C# Code

C# — Programmatic animations 
// Fade an element in
public static void FadeIn(UIElement element, double durationSeconds = 0.3)
{
    var anim = new DoubleAnimation
    {
        From             = 0,
        To               = 1,
        Duration         = TimeSpan.FromSeconds(durationSeconds),
        EasingFunction   = new CubicEase { EasingMode = EasingMode.EaseOut }
    };
    element.BeginAnimation(UIElement.OpacityProperty, anim);
}

// Shake animation for validation errors
public static void Shake(FrameworkElement element)
{
    var transform = new TranslateTransform();
    element.RenderTransform = transform;

    var anim = new DoubleAnimationUsingKeyFrames
    {
        Duration = new Duration(TimeSpan.FromMilliseconds(400))
    };
    anim.KeyFrames.Add(new EasingDoubleKeyFrame(-10, KeyTime.FromPercent(0.1)));
    anim.KeyFrames.Add(new EasingDoubleKeyFrame( 10, KeyTime.FromPercent(0.3)));
    anim.KeyFrames.Add(new EasingDoubleKeyFrame( -6, KeyTime.FromPercent(0.5)));
    anim.KeyFrames.Add(new EasingDoubleKeyFrame(  6, KeyTime.FromPercent(0.7)));
    anim.KeyFrames.Add(new EasingDoubleKeyFrame(  0, KeyTime.FromPercent(1.0)));

    transform.BeginAnimation(TranslateTransform.XProperty, anim);
}

🧵 Threading in WPF

👶 The Golden Rule of WPF Threading

WPF has one rule that you must never break: only the UI thread can touch UI elements. If you run a database query or API call on a background thread and then try to update a TextBlock directly from that thread, WPF will throw an exception. You must marshal the UI update back to the UI thread using the Dispatcher.

❌ This Will Crash Your App

Never update a UI element (TextBlock, ListBox, Button, etc.) from a background thread. This includes updating an ObservableCollection that is bound to a UI control — modifications must happen on the UI thread.

The Dispatcher — Marshalling to UI Thread

C# — Dispatcher patterns 
// ❌ WRONG — updating UI from background thread
Task.Run(() =>
{
    var data = LoadFromDatabase();
    txtResult.Text = data;  // CRASH: not on UI thread
});

// ✅ CORRECT — marshal back to UI thread with Dispatcher
Task.Run(() =>
{
    var data = LoadFromDatabase();

    // Dispatcher.Invoke: synchronous (waits for UI to process)
    Application.Current.Dispatcher.Invoke(() =>
    {
        txtResult.Text = data;  // safe on UI thread
    });
});

// ✅ BETTER — async/await handles threading for you
private async void LoadButton_Click(object sender, RoutedEventArgs e)
{
    btnLoad.IsEnabled = false;
    progressBar.Visibility = Visibility.Visible;

    // Task.Run moves work to background thread
    var data = await Task.Run(() => LoadFromDatabase());

    // After await, we're back on the UI thread automatically
    txtResult.Text = data;                 // safe
    progressBar.Visibility = Visibility.Collapsed;
    btnLoad.IsEnabled = true;
}

Async/Await — The Modern Way

C# — Proper async patterns in WPF ViewModels 
public class DataViewModel : ViewModelBase
{
    private bool   _isBusy;
    private string _status = "";

    public bool   IsBusy  { get => _isBusy;  set => SetProperty(ref _isBusy,  value); }
    public string Status  { get => _status; set => SetProperty(ref _status, value); }

    public ObservableCollection<string> Items { get; } = new();
    public ICommand LoadCommand { get; }

    public DataViewModel()
        => LoadCommand = new AsyncRelayCommand(LoadDataAsync);

    private async Task LoadDataAsync()
    {
        IsBusy = true;
        Status = "Loading...";
        Items.Clear();

        try
        {
            // Run heavy work on background thread
            var results = await Task.Run(() =>
            {
                // This runs on a ThreadPool thread
                return FetchDataFromDatabase();
            });

            // After await = back on UI thread — safe to update UI
            foreach (var item in results)
                Items.Add(item);

            Status = $"Loaded {results.Count} items";
        }
        catch (Exception ex)
        {
            Status = $"Error: {ex.Message}";
        }
        finally
        {
            IsBusy = false;
        }
    }

    private List<string> FetchDataFromDatabase()
    {
        Thread.Sleep(2000);  // simulate slow query
        return ["Item A", "Item B", "Item C"];
    }
}

BackgroundWorker — Legacy but Still Seen

C# — BackgroundWorker (older apps) 
private readonly BackgroundWorker _worker = new();

public MainWindow()
{
    InitializeComponent();
    _worker.WorkerReportsProgress = true;
    _worker.DoWork             += Worker_DoWork;
    _worker.ProgressChanged    += Worker_ProgressChanged;  // safe on UI thread
    _worker.RunWorkerCompleted += Worker_Completed;         // safe on UI thread
}

private void Worker_DoWork(object? sender, DoWorkEventArgs e)
{
    // Runs on background thread
    for (int i = 0; i <= 100; i++)
    {
        Thread.Sleep(50);
        _worker.ReportProgress(i);   // triggers ProgressChanged on UI thread
    }
    e.Result = "All done!";
}

private void Worker_ProgressChanged(object? sender, ProgressChangedEventArgs e)
    => progressBar.Value = e.ProgressPercentage;  // UI thread — safe

private void Worker_Completed(object? sender, RunWorkerCompletedEventArgs e)
    => txtStatus.Text = e.Result?.ToString();     // UI thread — safe

💡 async/await vs BackgroundWorker

For all new code, use async/await with Task.Run() for background work. It is simpler, more readable, handles exceptions better, and is the modern .NET way. BackgroundWorker is only relevant when working on legacy WPF applications.

🏭 Real-World Patterns

Messenger / Event Aggregator

👶 Why Messenger?

ViewModels should not hold references to each other. But sometimes ContactsViewModel needs to tell DetailsViewModel that a contact was selected. A Messenger lets them communicate through a central hub without knowing about each other — like a radio broadcast system.

C# — Simple Messenger (lightweight pub/sub) public static class Messenger { private static readonly Dictionary<Type, List<WeakReference<Action<object>>>> _handlers = new(); // Subscribe to messages of type T public static void Subscribe<T>(Action<T> handler) { var type = typeof(T); if (!_handlers.ContainsKey(type)) _handlers[type] = new(); _handlers[type].Add(new WeakReference<Action<object>>( m => handler((T)m))); } // Publish a message — all subscribers receive it public static void Publish<T>(T message) { if (!_handlers.TryGetValue(typeof(T), out var handlers)) return; foreach (var weakRef in handlers.ToList()) if (weakRef.TryGetTarget(out var handler)) handler(message!); } } // Message types (plain C# records) public record ContactSelectedMessage(Contact Contact); public record ContactDeletedMessage(int ContactId); public record ThemeChangedMessage(string Theme); // In ContactsViewModel — publish when selection changes public Contact? SelectedContact { get => _selected; set { SetProperty(ref _selected, value); if (value != null) Messenger.Publish(new ContactSelectedMessage(value)); } } // In DetailsViewModel — subscribe to receive the message public DetailsViewModel() { Messenger.Subscribe<ContactSelectedMessage>(msg => { CurrentContact = msg.Contact; }); }

PropertyChanged; protected void OnPropertyChanged([CallerMemberName] string? n = null) => PropertyChanged?.Invoke(this, new PropertyChangedEventArgs(n)); }

The ViewModel

C# — ViewModels/TaskListViewModel.cs 
public class TaskListViewModel : ViewModelBase
{
    private readonly ITaskService _service;
    private string    _filterText = "";
    private bool      _showCompleted = true;
    private bool      _isBusy;
    private AppTask?  _selected;

    public ObservableCollection<AppTask> Tasks    { get; } = new();
    public ICollectionView               TasksView { get; }

    public string   FilterText
    {
        get => _filterText;
        set { SetProperty(ref _filterText, value); TasksView.Refresh(); }
    }
    public bool     ShowCompleted
    {
        get => _showCompleted;
        set { SetProperty(ref _showCompleted, value); TasksView.Refresh(); }
    }
    public bool     IsBusy   { get => _isBusy;   set => SetProperty(ref _isBusy,   value); }
    public AppTask? Selected { get => _selected; set => SetProperty(ref _selected, value); }

    public string Summary => $"{Tasks.Count(t => !t.IsCompleted)} pending · {Tasks.Count(t => t.IsCompleted)} done";

    public ICommand LoadCommand       { get; }
    public ICommand AddCommand        { get; }
    public ICommand DeleteCommand     { get; }
    public ICommand CompleteCommand   { get; }
    public ICommand ClearDoneCommand  { get; }

    public TaskListViewModel(ITaskService service)
    {
        _service = service;

        // ICollectionView wraps ObservableCollection — adds sorting and filtering
        TasksView = CollectionViewSource.GetDefaultView(Tasks);
        TasksView.Filter = FilterTask;
        TasksView.SortDescriptions.Add(new SortDescription(nameof(AppTask.Priority),
            ListSortDirection.Descending));

        LoadCommand      = new AsyncRelayCommand(LoadAsync);
        AddCommand       = new RelayCommand(_ => AddTask());
        DeleteCommand    = new RelayCommand(_ => Delete(), _ => Selected != null);
        CompleteCommand  = new RelayCommand(_ => ToggleComplete(), _ => Selected != null);
        ClearDoneCommand = new RelayCommand(_ => ClearCompleted());

        _ = LoadAsync();
    }

    private bool FilterTask(object obj)
    {
        if (obj is not AppTask task) return false;
        if (!ShowCompleted && task.IsCompleted) return false;
        if (!string.IsNullOrEmpty(FilterText) &&
            !task.Title.Contains(FilterText, StringComparison.OrdinalIgnoreCase))
            return false;
        return true;
    }

    private async Task LoadAsync()
    {
        IsBusy = true;
        Tasks.Clear();
        var data = await _service.GetAllAsync();
        foreach (var t in data) Tasks.Add(t);
        IsBusy = false;
        OnPropertyChanged(nameof(Summary));
    }

    private void AddTask()
    {
        var dlg = new AddTaskWindow();
        if (dlg.ShowDialog() == true && dlg.NewTask != null)
        {
            Tasks.Add(dlg.NewTask);
            OnPropertyChanged(nameof(Summary));
        }
    }
    private void Delete()
    {
        if (Selected == null) return;
        Tasks.Remove(Selected);
        OnPropertyChanged(nameof(Summary));
    }
    private void ToggleComplete()
    {
        if (Selected == null) return;
        Selected.IsCompleted = !Selected.IsCompleted;
        TasksView.Refresh();
        OnPropertyChanged(nameof(Summary));
    }
    private void ClearCompleted()
    {
        var done = Tasks.Where(t => t.IsCompleted).ToList();
        foreach (var t in done) Tasks.Remove(t);
        OnPropertyChanged(nameof(Summary));
    }
}
