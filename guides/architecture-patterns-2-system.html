🌐 System Architecture Patterns

👶 Part 1 vs Part 2 — What's the Difference?

Part 1 covered how you structure code inside a single application — MVC, MVVM, Clean Architecture. Part 2 covers how systems talk to each other and how infrastructure is organised across multiple services, databases, and communication channels. The 6 System Architecture Patterns

🔬

📡

⚡ Event-Driven Architecture

👶 In Plain English

In Event-Driven Architecture, services communicate by publishing and reacting to events. When something happens ("OrderPlaced", "PaymentProcessed", "UserRegistered"), a service publishes that event. Other services that care about it subscribe and react — independently, asynchronously. The publisher doesn't know who reacts. This creates extremely loose coupling between services.

💡 Real World Analogy

A radio station. The station (producer) broadcasts a show (event) without knowing who is listening. Every radio tuned to that frequency (consumer) receives and reacts to it. Add a new listener? The station doesn't need to change. Remove a listener? The station doesn't notice. Listeners are completely independent of each other and of the station's internal workings.

Event-Driven Flow — One Event, Many Independent Reactions

⚡ Event-Driven — Publisher Doesn't Know the Subscribers

Synchronous vs Event-Driven Communication

❌ Synchronous (Tight Coupling)
  • OrderService directly calls EmailService, InventoryService, LoyaltyService
  • OrderService must know about all dependent services
  • If EmailService is down, the order fails
  • Adding a new reaction (SMS) means editing OrderService
  • OrderService waits for all downstream calls to complete
✅ Event-Driven (Loose Coupling)
  • OrderService publishes "OrderPlaced" event and moves on
  • EmailService, InventoryService, LoyaltyService subscribe independently
  • If EmailService is down, it retries later — order is still placed
  • Add SMS notifications? New subscriber — zero changes to OrderService
  • OrderService doesn't wait — fast and non-blocking

// ── Step 2: Publish from domain entity / service ───────────────────
public class OrderService
{
    private readonly IOrderRepository _repo;
    private readonly IMediator         _mediator;

    public OrderService(IOrderRepository repo, IMediator mediator)
    { _repo = repo; _mediator = mediator; }

    public async Task<Guid> PlaceOrderAsync(PlaceOrderRequest req)
    {
        var order = Order.Create(req.CustomerId, req.Lines);
        await _repo.SaveAsync(order);

        // ✅ Publish event — OrderService doesn't know or care who handles it
        await _mediator.Publish(new OrderPlacedEvent(
            order.Id, req.CustomerId, order.Total, DateTime.UtcNow));

        return order.Id;
    }
}

🔬 Microservices vs Monolith

👶 In Plain English

A Monolith is one big application where everything is together — all code, all features, all in one deployable unit. A Microservices architecture splits the application into many small, independent services — each runs its own process, owns its own database, deploys independently, and communicates over HTTP/messaging. Both have real tradeoffs — neither is always right.

💡 Real World Analogy

A Swiss Army knife (Monolith) has everything in one tool — convenient, always together, easy to carry. But if one blade breaks, you can't replace just that blade. A toolbox (Microservices) has separate tools — you can upgrade, replace, or add individual tools without affecting others. But you need a toolbox to carry them, and coordination between tools is more effort.

💡 Start with a Monolith — Evolve to Microservices

Most successful microservice architectures started as monoliths. Build a well-structured monolith first (Clean Architecture internally). When you hit real scaling or team boundary pain points, extract services. "Microservices first" is one of the most common architecture mistakes — the coordination overhead crushes small teams.

public interface IPaymentServiceClient
{
    Task<PaymentResult> ProcessPaymentAsync(PaymentRequest request);
}

public class PaymentServiceClient : IPaymentServiceClient
{
    private readonly HttpClient _http;
    public PaymentServiceClient(HttpClient http) => _http = http;

    public async Task<PaymentResult> ProcessPaymentAsync(PaymentRequest request)
    {
        var response = await _http.PostAsJsonAsync("/api/payments", request);
        response.EnsureSuccessStatusCode();
        return await response.Content.ReadFromJsonAsync<PaymentResult>()
               ?? throw new InvalidOperationException("Empty response");
    }
}

Key Microservices Patterns

🔄

Circuit Breaker

Stop calling a failing service. Let it recover. Prevents cascade failures across services.

🏥

Health Checks

Each service exposes /health endpoint. Kubernetes and load balancers route away from unhealthy instances.

🔍

Distributed Tracing

Trace a request across multiple services. OpenTelemetry + Jaeger/Zipkin show the full journey.

📋

Saga Pattern

Distributed transactions across services. Each step publishes events; compensating actions undo on failure.

📦

Service Registry

Services register themselves. Clients discover addresses. No hardcoded URLs in code.

🔐

JWT Auth

Each service validates JWT tokens independently. No central session store needed.

📡 Publish/Subscribe Pattern

👶 In Plain English

Pub/Sub is a messaging pattern where Publishers send messages to a topic/channel without knowing who receives them, and Subscribers listen to topics they care about without knowing who sends them. A message broker (Azure Service Bus, RabbitMQ, AWS SQS) sits in the middle. This is how loosely coupled, scalable, asynchronous communication works between services.

💡 Real World Analogy

A YouTube channel. The creator (Publisher) uploads a video to their channel (Topic). All subscribers receive a notification. The creator doesn't know or care who is subscribed. Subscribers don't know or care who else is subscribed. New subscriber? The creator doesn't need to do anything. Unsubscribe? Same. The platform (Broker) handles all the routing.

Pub/Sub Flow — Decoupled via the Broker

📡 Pub/Sub — Publisher and Subscribers Never Meet Directly

Pub/Sub vs Direct HTTP

Pub/Sub with Azure Service Bus

C# — Publisher sending messages to Azure Service Bus
// Install: dotnet add package Azure.Messaging.ServiceBus

public interface IMessagePublisher
{
    Task PublishAsync<T>(T message, string topicOrQueue,
                          CancellationToken ct = default);
}

public class ServiceBusPublisher : IMessagePublisher
{
    private readonly ServiceBusClient _client;

    public ServiceBusPublisher(ServiceBusClient client) => _client = client;

    public async Task PublishAsync<T>(T message, string topicOrQueue,
        CancellationToken ct = default)
    {
        await using var sender = _client.CreateSender(topicOrQueue);

        var json = JsonSerializer.Serialize(message);
        var msg  = new ServiceBusMessage(Encoding.UTF8.GetBytes(json))
        {
            MessageId    = Guid.NewGuid().ToString(),
            ContentType  = "application/json",
            Subject      = typeof(T).Name    // message type for routing
        };

        await sender.SendMessageAsync(msg, ct);
        Console.WriteLine($"Published {typeof(T).Name} to {topicOrQueue}");
    }
}

// Publisher usage — OrderService publishes to "orders" topic
await _publisher.PublishAsync(
    new OrderPlacedIntegrationEvent(order.Id, customerId, total),
    topicOrQueue: "orders");
C# — Subscriber consuming messages (Background Worker)
// NotificationService subscribes to "orders" topic
public class OrderEventConsumer : BackgroundService
{
    private readonly ServiceBusClient    _client;
    private readonly IEmailService       _email;
    private readonly ILogger<OrderEventConsumer> _logger;

    public OrderEventConsumer(
        ServiceBusClient client,
        IEmailService email,
        ILogger<OrderEventConsumer> logger)
    { _client = client; _email = email; _logger = logger; }

    protected override async Task ExecuteAsync(CancellationToken stoppingToken)
    {
        // Subscribe to "orders" topic, "notification-service" subscription
        await using var processor = _client.CreateProcessor(
            topicName:         "orders",
            subscriptionName:  "notification-service",
            options:           new ServiceBusProcessorOptions
            {
                MaxConcurrentCalls  = 5,
                AutoCompleteMessages = false   // we manually complete/abandon
            });

        processor.ProcessMessageAsync += HandleMessageAsync;
        processor.ProcessErrorAsync   += HandleErrorAsync;

        await processor.StartProcessingAsync(stoppingToken);
        await Task.Delay(Timeout.Infinite, stoppingToken);
        await processor.StopProcessingAsync();
    }

    private async Task HandleMessageAsync(ProcessMessageEventArgs args)
    {
        try
        {
            var json  = args.Message.Body.ToString();
            var evt   = JsonSerializer.Deserialize<OrderPlacedIntegrationEvent>(json)!;

            _logger.LogInformation("Processing order {OrderId}", evt.OrderId);

            await _email.SendOrderConfirmationAsync(evt.CustomerId, evt.OrderId);

            // ✅ Manually complete — removes from queue
            await args.CompleteMessageAsync(args.Message);
        }
        catch (Exception ex)
        {
            _logger.LogError(ex, "Failed processing message");
            // ❌ Abandon — message returns to queue for retry
            await args.AbandonMessageAsync(args.Message);
        }
    }

    private Task HandleErrorAsync(ProcessErrorEventArgs args)
    {
        _logger.LogError(args.Exception, "Service Bus error");
        return Task.CompletedTask;
    }
}

// Register in Program.cs:
builder.Services.AddSingleton(new ServiceBusClient(connectionString));
builder.Services.AddHostedService<OrderEventConsumer>();

🗄️ Repository & Unit of Work Pattern

👶 In Plain English

The Repository pattern wraps your database access behind an interface — your business logic talks to IOrderRepository, not directly to EF Core or SQL. The Unit of Work groups multiple repository operations into a single transaction — either all succeed together or all roll back together. Together they make your data access testable and transactional.

💡 Real World Analogy

A librarian (Repository) is your interface to the library's books. You ask the librarian for a book by title — you don't walk the stacks yourself. The librarian knows where everything is and handles the physical storage. A shopping basket (Unit of Work) — you add multiple items to your basket, and only when you check out do all items get purchased together as one transaction. Put one back = none are purchased.

Repository Pattern — Full Implementation

C# — Generic Repository with EF Core
// ── Generic Repository interface ───────────────────────────────────
public interface IRepository<T> where T : class
{
    Task<T?>         GetByIdAsync(int id, CancellationToken ct = default);
    Task<List<T>>    GetAllAsync(CancellationToken ct = default);
    Task<List<T>>    FindAsync(Expression<Func<T, bool>> predicate, CancellationToken ct = default);
    Task              AddAsync(T entity, CancellationToken ct = default);
    void              Update(T entity);
    void              Remove(T entity);
    Task<bool>        ExistsAsync(Expression<Func<T, bool>> predicate, CancellationToken ct = default);
}

// ── Generic EF Core implementation ────────────────────────────────
public class EfRepository<T> : IRepository<T> where T : class
{
    protected readonly AppDbContext  _ctx;
    protected readonly DbSet<T>     _set;

    public EfRepository(AppDbContext ctx) { _ctx = ctx; _set = ctx.Set<T>(); }

    public async Task<T?>       GetByIdAsync(int id, CancellationToken ct = default)
        => await _set.FindAsync(new object[] { id }, ct);

    public async Task<List<T>> GetAllAsync(CancellationToken ct = default)
        => await _set.AsNoTracking().ToListAsync(ct);

    public async Task<List<T>> FindAsync(Expression<Func<T, bool>> pred,
        CancellationToken ct = default)
        => await _set.Where(pred).ToListAsync(ct);

    public async Task       AddAsync(T entity, CancellationToken ct = default)
        => await _set.AddAsync(entity, ct);

    public void             Update(T entity) => _set.Update(entity);
    public void             Remove(T entity) => _set.Remove(entity);

    public async Task<bool> ExistsAsync(Expression<Func<T, bool>> pred,
        CancellationToken ct = default)
        => await _set.AnyAsync(pred, ct);
}

// ── Specific repository with custom queries ────────────────────────
public interface IOrderRepository : IRepository<Order>
{
    // Custom queries beyond the generic ones
    Task<List<Order>> GetByCustomerIdAsync(int customerId, CancellationToken ct = default);
    Task<List<Order>> GetPendingOrdersAsync(CancellationToken ct = default);
    Task<decimal>      GetTotalRevenueAsync(DateTime from, DateTime to);
}

public class OrderRepository : EfRepository<Order>, IOrderRepository
{
    public OrderRepository(AppDbContext ctx) : base(ctx) { }

    public async Task<List<Order>> GetByCustomerIdAsync(int customerId, CancellationToken ct = default)
        => await _set
            .Where(o => o.CustomerId == customerId)
            .Include(o => o.Lines)
            .OrderByDescending(o => o.CreatedAt)
            .ToListAsync(ct);

    public async Task<List<Order>> GetPendingOrdersAsync(CancellationToken ct = default)
        => await _set
            .Where(o => o.Status == OrderStatus.Pending)
            .ToListAsync(ct);

    public async Task<decimal> GetTotalRevenueAsync(DateTime from, DateTime to)
        => await _set
            .Where(o => o.CreatedAt >= from && o.CreatedAt <= to && o.Status == OrderStatus.Confirmed)
            .SumAsync(o => o.Total);
}

Unit of Work Pattern

C# — Unit of Work groups operations into one transaction
// ── Unit of Work interface ─────────────────────────────────────────
public interface IUnitOfWork : IAsyncDisposable
{
    IOrderRepository    Orders    { get; }
    IRepository<Customer> Customers { get; }
    IRepository<Product>  Products  { get; }

    Task<int> SaveChangesAsync(CancellationToken ct = default);
}

// ── EF Core implementation ─────────────────────────────────────────
public class EfUnitOfWork : IUnitOfWork
{
    private readonly AppDbContext _ctx;

    // Repositories share the same DbContext — same transaction
    public IOrderRepository     Orders    { get; }
    public IRepository<Customer> Customers { get; }
    public IRepository<Product>  Products  { get; }

    public EfUnitOfWork(AppDbContext ctx)
    {
        _ctx      = ctx;
        Orders    = new OrderRepository(ctx);
        Customers = new EfRepository<Customer>(ctx);
        Products  = new EfRepository<Product>(ctx);
    }

    // ONE call saves ALL changes from ALL repositories
    public Task<int> SaveChangesAsync(CancellationToken ct = default)
        => _ctx.SaveChangesAsync(ct);

    public async ValueTask DisposeAsync() => await _ctx.DisposeAsync();
}

// ── Service using Unit of Work ─────────────────────────────────────
public class OrderService
{
    private readonly IUnitOfWork _uow;
    public OrderService(IUnitOfWork uow) => _uow = uow;

    public async Task<Guid> PlaceOrderAsync(int customerId, List<OrderLine> lines)
    {
        // All operations on the same DbContext = same transaction
        var customer = await _uow.Customers.GetByIdAsync(customerId)
            ?? throw new NotFoundException($"Customer {customerId} not found");

        // Check and reserve products
        foreach (var line in lines)
        {
            var product = await _uow.Products.GetByIdAsync(line.ProductId)
                ?? throw new NotFoundException($"Product {line.ProductId} not found");

            if (product.Stock < line.Quantity)
                throw new InsufficientStockException(product.Name);

            product.Stock -= line.Quantity;   // deduct stock
            _uow.Products.Update(product);
        }

        var order = Order.Create(customerId, lines);
        await _uow.Orders.AddAsync(order);

        // ✅ ONE SaveChanges = order saved + stock deducted in ONE transaction
        // If ANYTHING above throws, NOTHING is saved — atomic operation
        await _uow.SaveChangesAsync();

        return order.Id;
    }
}
SOLID: D — Services depend on IRepository/IUnitOfWork interfaces S — Repository handles data access; service handles business logic O — Swap EF Core for Dapper by replacing EfRepository implementation

💉 Dependency Injection Architecture

👶 In Plain English

Dependency Injection (DI) means: instead of a class creating its own dependencies with new, they are provided from outside (injected). A DI Container is a framework that automatically creates and injects all dependencies based on registrations you define at startup. In .NET, this is built-in via IServiceCollection.

💡 Real World Analogy

A new employee (class) doesn't build their own desk, computer, and tools — the company (DI container) provides them. The employee just says "I need a computer and a desk" (constructor parameters), and facilities (the container) provides them. The employee doesn't know or care who made the computer or where the desk came from. This makes it easy to give a different computer (mock in tests) without changing the employee.

How the DI Container Resolves Dependencies

💉 Dependency Injection — Container Wires Everything Together

The Three Lifetimes

🔁

Transient

New instance every time it's requested. Lightweight, stateless services. Default for most services.

🌐

Scoped

One instance per HTTP request (web) or per scope. DbContext, repositories. Shared within a request.

1️⃣

Singleton

One instance for the entire app lifetime. Caches, configuration, loggers, HttpClient factories.

// ── Extension methods for clean registration (recommended) ────────
public static class ServiceCollectionExtensions
{
    public static IServiceCollection AddApplicationServices(this IServiceCollection services)
    {
        services.AddScoped<IOrderService,    OrderService>();
        services.AddScoped<IPaymentService,  PaymentService>();
        services.AddTransient<IEmailService, SmtpEmailService>();
        return services;
    }

    public static IServiceCollection AddInfrastructure(this IServiceCollection services,
        IConfiguration config)
    {
        services.AddDbContext<AppDbContext>(o =>
            o.UseSqlServer(config.GetConnectionString("Default")));
        services.AddScoped<IUnitOfWork,       EfUnitOfWork>();
        services.AddScoped<IOrderRepository,  OrderRepository>();
        return services;
    }
}

// Program.cs becomes clean:
builder.Services.AddApplicationServices();
builder.Services.AddInfrastructure(builder.Configuration);

⚠️ Captive Dependency — The Most Common DI Mistake

Never inject a Scoped service into a Singleton. The singleton lives forever but the scoped service was only meant to live for one request — now it's stuck ("captive") inside the singleton forever, causing stale data and memory leaks. Rule: a longer-lived service should never depend on a shorter-lived service.

SOLID: D — Classes depend on interfaces injected from outside S — Composition Root (Program.cs) handles wiring; classes handle logic O — Swap implementations by changing one registration line

🚪 API Gateway Pattern

👶 In Plain English

An API Gateway is a single entry point that sits in front of all your services. Every client request goes through it first. The gateway handles cross-cutting concerns — authentication, rate limiting, SSL termination, request routing, logging, response caching — so your individual services don't have to. Without a gateway, every client would need to know the address of every service.

💡 Real World Analogy

A hotel reception desk. Every guest (client) goes to reception first — they don't wander directly to the kitchen, housekeeping, or the spa. Reception (Gateway) authenticates you (checks your booking), routes you ("Room 302 is on the third floor"), applies rules ("spa closes at 10pm"), and coordinates services. The individual departments don't deal with guest authentication — reception handles it.

API Gateway — Single Entry Point, Many Services

🚪 API Gateway — Routing, Auth, and Rate Limiting in One Place

What a Gateway Does

🔐

Authentication

Validate JWT tokens once at the gateway. Services trust the gateway and don't re-validate.

🗺️

Request Routing

Route /api/orders → OrderService, /api/users → UserService. Clients use one URL.

⏱️

Rate Limiting

Limit requests per client/IP. Prevent abuse without every service implementing it.

📋

Logging & Tracing

Log every request centrally. Attach correlation IDs for distributed tracing.

💾

Response Caching

Cache frequent responses. Reduce load on downstream services for popular endpoints.

🔄

Load Balancing

Distribute requests across multiple instances of the same service.

API Gateway with YARP in ASP.NET Core

C# — YARP (Yet Another Reverse Proxy) Gateway
// Install: dotnet add package Yarp.ReverseProxy

// ── Program.cs — Gateway setup ─────────────────────────────────────
var builder = WebApplication.CreateBuilder(args);

builder.Services
    .AddReverseProxy()
    .LoadFromConfig(builder.Configuration.GetSection("ReverseProxy"));

builder.Services.AddAuthentication().AddJwtBearer();
builder.Services.AddRateLimiter(opts =>
{
    opts.AddFixedWindowLimiter("api", o =>
    {
        o.Window             = TimeSpan.FromMinutes(1);
        o.PermitLimit        = 100;
        o.QueueProcessingOrder = QueueProcessingOrder.OldestFirst;
        o.QueueLimit         = 10;
    });
});

var app = builder.Build();
app.UseAuthentication();
app.UseAuthorization();
app.UseRateLimiter();

// Middleware: attach correlation ID to every proxied request
app.Use(async (context, next) =>
{
    var correlationId = context.Request.Headers["X-Correlation-ID"]
        .FirstOrDefault() ?? Guid.NewGuid().ToString();

    context.Response.Headers["X-Correlation-ID"] = correlationId;
    await next();
});

app.MapReverseProxy();
app.Run();

Custom Middleware in the Gateway

C# — Gateway middleware for auth, logging, transformation
// Add custom headers to proxied requests (e.g. authenticated user info)
app.MapReverseProxy(proxyPipeline =>
{
    proxyPipeline.Use(async (context, next) =>
    {
        // Forward the authenticated user's ID to downstream services
        if (context.User.Identity?.IsAuthenticated == true)
        {
            var userId = context.User.FindFirst("sub")?.Value;
            context.Request.Headers["X-User-ID"] = userId;
        }

        var sw = Stopwatch.StartNew();
        await next();
        sw.Stop();

        // Log request duration
        app.Logger.LogInformation(
            "Gateway: {Method} {Path} → {StatusCode} in {Ms}ms",
            context.Request.Method,
            context.Request.Path,
            context.Response.StatusCode,
            sw.ElapsedMilliseconds);
    });
});

When to Use API Gateway

  • Microservices architecture — clients need a single address, not one per service
  • Cross-cutting concerns (auth, logging, rate limiting) need to apply to all services
  • Different clients need different API shapes (mobile vs web vs third-party)
  • SSL termination — only the gateway handles HTTPS, services use HTTP internally
  • You need load balancing and failover across multiple service instances
  • Monolithic apps — unnecessary overhead for a single deployable
  • Very simple internal services with no external clients
  • When the gateway becomes a bottleneck — must be highly available
SOLID: S — Gateway handles routing/auth; services handle business logic O — Add new routes/services without changing gateway middleware D — Services don't depend on the gateway; it sits in front
