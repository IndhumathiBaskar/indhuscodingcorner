🐚 What is AWS CloudShell?

AWS CloudShell is a browser-based, pre-authenticated shell environment built directly into the AWS Management Console. Open it and you instantly have a Linux terminal with the AWS CLI, common tools, and your IAM permissions — no setup, no credentials to configure, no local installation needed.

💡 CloudShell in One Sentence

AWS CloudShell = a free, browser-based Linux terminal inside the AWS Console, with AWS CLI pre-installed, pre-authenticated using your current Console IAM session, and 1 GB of persistent home directory storage.

Why does CloudShell exist?

😩 The Problem it Solves

  • You're on a new machine, no AWS CLI installed
  • You're on a shared/corporate machine, can't install software
  • You need a quick one-off command without setting up profiles
  • You're on your phone or tablet accessing the Console
  • You don't want to manage access keys on local machine

✅ What CloudShell Gives You

  • Instant AWS CLI access — no install, no configure
  • Pre-authenticated — your Console IAM permissions automatically
  • 1 GB persistent storage in $HOME — survives session restarts
  • Common tools pre-installed (Python, Node, git, jq, and more)
  • Free — no extra charge for CloudShell itself

⚙️ How CloudShell Works Under the Hood

The Architecture

🏗️ CloudShell — Architecture & Request Flow

Credential Flow — No Keys Needed

🔑 How Authentication Works in CloudShell
👤 You log in to AWS Console
  • Sign in as IAM user, federated role (SSO), or root (don't!)
  • Your IAM identity has policies attached
↓
🐚 You open CloudShell
  • AWS provisions a fresh container for you
  • Injects short-lived, auto-rotated credentials into the environment
  • These map exactly to your Console IAM session
↓
⌨️ You run AWS CLI commands
  • CLI reads credentials from environment (auto-injected)
  • You can verify: aws sts get-caller-identity
  • Your IAM policies determine what you can and cannot do
↓
🔄 Credentials auto-rotate
  • CloudShell automatically refreshes short-lived credentials
  • No manual token refresh needed — it just works
  • Session ends after 20 minutes of inactivity

Session Lifecycle

EventWhat happens
Open CloudShellFresh container provisioned, takes ~5–10 seconds to start
Active useSession stays alive as long as you're typing or running commands
20 min idleSession is terminated — running processes stop
Session endContainer destroyed — $HOME (1 GB) persists, everything else lost
Reopen CloudShellNew container, but same $HOME directory contents restored
Close browser tabSame as idle — session eventually terminates
Region switchDifferent CloudShell environment per region — separate $HOME

⚠️ Running processes are killed on idle timeout

If you start a long-running script and leave it idle (no terminal output), the session will time out after 20 minutes and kill your process. For long jobs, use nohup, keep interacting, or better yet run them as Lambda/ECS tasks instead.

Opening CloudShell

1

Click the CloudShell icon in the Console toolbar

Look for the terminal icon (🐚) in the top navigation bar of the AWS Console, right next to the search bar. Or press the keyboard shortcut Alt+F1.

2

Wait ~5–10 seconds for provisioning

The first launch takes a moment. Subsequent launches in the same session are faster. You'll see "Creating your environment…".

3

Terminal appears at the bottom of the Console

You can resize it, pop it out into a full tab, or open multiple tabs. You're now in Amazon Linux 2023 with full bash access.

Installing Additional Tools

bash — install extra packages (persist in $HOME) 
# Install Python packages (persist in $HOME/.local)
pip3 install boto3 --user
pip3 install awswrangler --user

# Install Node packages globally in $HOME
npm install -g serverless
npm install -g aws-cdk

# Install a binary tool to $HOME/bin (persists!)
mkdir -p ~/bin
curl -sL https://github.com/mikefarah/yq/releases/latest/download/yq_linux_amd64 \
  -o ~/bin/yq
chmod +x ~/bin/yq
export PATH=$HOME/bin:$PATH

# Add to ~/.bashrc so it's available next session
echo 'export PATH=$HOME/bin:$PATH' >> ~/.bashrc

📌 What persists vs what doesn't

Persists (in $HOME): your scripts, configs (~/.bashrc, ~/.gitconfig), pip --user installs, Node global packages under ~/, binaries in ~/bin. Does NOT persist: system packages installed with sudo yum install, running processes, anything outside /home/cloudshell-user.

Customise Your Shell — ~/.bashrc

bash — useful ~/.bashrc customisations 
# Add to ~/.bashrc to customise every CloudShell session

# Useful aliases
alias ll='ls -la'
alias awswho='aws sts get-caller-identity'
alias myregion='aws configure get region'

# Show AWS account + region in prompt
get_aws_info() {
  ACCT=$(aws sts get-caller-identity --query Account --output text 2>/dev/null)
  REGION=$(aws configure get region)
  echo "[$ACCT / $REGION]"
}
export PS1='\u@cloudshell $(get_aws_info) \w \$ '

# Always use eu-west-1 in this shell
export AWS_DEFAULT_REGION=eu-west-1

# PATH for user-installed binaries
export PATH=$HOME/bin:$HOME/.local/bin:$PATH

💡 AWS CodeCommit works seamlessly

Since CloudShell already has your IAM credentials, AWS CodeCommit repos clone without any separate authentication. Just use the HTTPS (GRC) clone URL: codecommit::region://repo-name. No tokens, no SSH keys needed.

🌐 Networking in CloudShell

Default Networking — How it Works

✅ What CloudShell CAN Access

  • All AWS service API endpoints (via the internet)
  • Public internet — pip, npm, curl, wget, git clone
  • Public S3 buckets and objects
  • Any public HTTPS endpoint
  • AWS PrivateLink endpoints (via VPC-enabled CloudShell)

❌ What Default CloudShell CANNOT Access

  • Resources inside a private VPC (RDS, private EC2, private ALB)
  • On-premises networks via VPN/Direct Connect (unless VPC mode)
  • Inbound connections — CloudShell has no public IP for inbound
  • No inbound ports accessible from outside

VPC-Enabled CloudShell

📌 Connect CloudShell to your VPC

From the CloudShell Actions menu, you can configure a VPC connection — choose your VPC, subnet, and security group. This puts the CloudShell environment inside your VPC, giving it access to private resources like RDS databases, private API endpoints, and internal services. This is perfect for production database queries or private API testing.

🌐 CloudShell Networking — Default vs VPC Mode
🌐 Default Mode
  • CloudShell in AWS-managed network
  • Outbound internet access ✅
  • Access public AWS API endpoints ✅
  • Access private VPC resources ❌
  • Best for: CLI commands, public API calls
🏠 VPC Mode
  • CloudShell attached to your subnet + SG
  • Access private VPC resources ✅
  • Access RDS, private ALB, internal APIs ✅
  • Outbound internet depends on VPC routing
  • Best for: database queries, private services

Connecting to a Private RDS via VPC CloudShell

bash — connect to private rds from vpc cloudshell 
# 1. Configure CloudShell VPC settings first:
#    Actions → VPC settings → Select VPC + private subnet + security group
#    (Security group must allow outbound to RDS port 5432/3306)

# 2. Install a DB client (persists in $HOME via pip or ~/bin)
pip3 install psycopg2-binary --user   # PostgreSQL via Python

# 3. Connect to RDS PostgreSQL
psql -h my-rds.xxxx.eu-west-1.rds.amazonaws.com \
     -U admin \
     -d mydb \
     -p 5432

# 4. Or get the password from Secrets Manager first
DB_PASS=$(aws secretsmanager get-secret-value \
  --secret-id prod/rds/password \
  --query SecretString \
  --output text | jq -r '.password')

PGPASSWORD=$DB_PASS psql \
  -h my-rds.xxxx.eu-west-1.rds.amazonaws.com \
  -U admin -d mydb

🛠️ Real Use Cases — What to Actually Do in CloudShell

1. Quick S3 Operations

bash — s3 tasks without local cli setup 
# List all buckets and their regions
aws s3api list-buckets \
  --query 'Buckets[].[Name,CreationDate]' \
  --output table

# Check a specific bucket's size
aws s3 ls s3://my-bucket --recursive --human-readable --summarize

# Copy a file between S3 buckets across accounts
aws s3 cp s3://source-bucket/data.csv s3://dest-bucket/data.csv

# Make a quick script, run it, delete it — all in CloudShell
cat > ~/cleanup.sh << 'EOF'
#!/bin/bash
BUCKET="my-logs-bucket"
DAYS_OLD=90
CUTOFF=$(date -d "$DAYS_OLD days ago" +%Y-%m-%d)
aws s3 ls s3://$BUCKET/ | awk '{print $4}' | while read KEY; do
  DATE=$(aws s3api head-object --bucket $BUCKET --key $KEY \
    --query LastModified --output text | cut -d'T' -f1)
  if [[ "$DATE" < "$CUTOFF" ]]; then
    echo "Deleting: $KEY"
    aws s3 rm s3://$BUCKET/$KEY
  fi
done
EOF
chmod +x ~/cleanup.sh

4. Python & boto3 Scripts

python3 — run boto3 scripts directly in cloudshell 
# boto3 is pre-installed and pre-authenticated!
python3 << 'EOF'
import boto3

s3 = boto3.client('s3')

# List all buckets
response = s3.list_buckets()
for bucket in response['Buckets']:
    print(f"  {bucket['Name']}  —  created: {bucket['CreationDate']:%Y-%m-%d}")

# Get Lambda function count per runtime
lam = boto3.client('lambda', region_name='eu-west-1')
fns = lam.list_functions()['Functions']
runtimes = {}
for fn in fns:
    rt = fn.get('Runtime', 'N/A')
    runtimes[rt] = runtimes.get(rt, 0) + 1

print("\nLambda runtimes:")
for rt, count in sorted(runtimes.items()):
    print(f"  {rt}: {count}")
EOF

⚖️ CloudShell vs Local AWS CLI — When to Use Which

💡 They're complementary, not competing

CloudShell and local AWS CLI both call the same AWS APIs. The choice is about convenience, context, and use case — not capability.

Detailed Comparison

Feature🐚 CloudShell⌨️ Local AWS CLI
Setup timeZero — open browser and goInstall + configure profiles/credentials
AuthenticationAuto — inherits Console IAM sessionManual — access keys, SSO, or role
Works on any machineYes — just a browser neededOnly on machines where CLI is installed
Storage1 GB $HOME (persistent)Full local disk
Long-running scripts⚠️ Killed after 20 min idleRuns as long as needed
CI/CD pipelinesNot suitableYes — GitHub Actions, CodeBuild, Jenkins
Private VPC accessVPC mode onlyVia SSM or VPN/Direct Connect
Multiple AWS accountsOne at a time (current Console account)Named profiles for unlimited accounts
Custom tool versionsLimited — system packages ephemeralFull control — install any version
Team sharingIndividual — tied to your IAM sessionShareable scripts/configs in git
CostFree (CloudShell itself)Free (CLI itself)
Internet accessOutbound onlyFull bidirectional

Decision Guide — Which to Use

🤔 CloudShell vs Local CLI — Quick Decision
✅ Use CloudShell when...
  • You're on a machine without AWS CLI installed
  • You need a quick one-off command or investigation
  • You want zero credential setup hassle
  • You're doing a live incident investigation
  • You want to run a Python/boto3 script quickly
  • You're checking logs, stack status, IAM policies
  • You're demoing something in the Console
✅ Use Local CLI when...
  • You're building CI/CD automation (GitHub Actions, etc.)
  • Scripts need to run for more than 20 minutes
  • You need to work across multiple AWS accounts simultaneously
  • You're generating large files (>1 GB)
  • You need full Docker daemon (build images)
  • You need a specific tool version not in CloudShell
  • You're writing scripts others need to run locally

💡 Tips & Tricks — Power User Moves

Keep Sessions Alive During Long Commands

bash — prevent idle timeout during long operations 
# Use nohup to keep a process running after terminal disconnect
nohup aws s3 sync s3://huge-bucket/ ~/backup/ > ~/sync.log 2>&1 &
echo "PID: $!"

# Check progress later
tail -f ~/sync.log

# Or use screen (if available) for persistent sessions
screen -S mysession
# ... run your long command ...
# Ctrl+A then D to detach
screen -r mysession   # reattach later

Use CloudShell for Safe Credential Testing

bash — test assume-role without touching local creds 
# Test what a role can do — assume it and try commands
CREDS=$(aws sts assume-role \
  --role-arn arn:aws:iam::999999999:role/ReadOnlyRole \
  --role-session-name test-session \
  --output json)

export AWS_ACCESS_KEY_ID=$(echo $CREDS | jq -r '.Credentials.AccessKeyId')
export AWS_SECRET_ACCESS_KEY=$(echo $CREDS | jq -r '.Credentials.SecretAccessKey')
export AWS_SESSION_TOKEN=$(echo $CREDS | jq -r '.Credentials.SessionToken')

aws sts get-caller-identity     # confirm role assumed
aws s3 ls                       # test what it can do

# Unset to go back to your original credentials
unset AWS_ACCESS_KEY_ID AWS_SECRET_ACCESS_KEY AWS_SESSION_TOKEN

#!/bin/bash
for BUCKET in $(aws s3api list-buckets --query 'Buckets[].Name' --output text); do

done

🚨 Security Reminder

CloudShell runs with your full IAM permissions. If you have admin access, CloudShell has admin access. Never share your CloudShell session URL, never leave it open on shared screens, and be careful running scripts from unknown sources — they run with your IAM identity.

Limitations to Remember

  • Sessions time out after 20 minutes of idle — running processes are killed
  • Only 1 GB persistent storage in $HOME — use S3 for large files
  • System-level packages (sudo yum install) are ephemeral — gone on restart
  • Cannot run Docker daemon — no image builds (CLI is available, daemon is not)
  • Tied to one AWS account/region at a time — no multi-account in same session
  • Not available in every AWS region — check region availability if icon is missing
  • No inbound connectivity — CloudShell cannot serve as a web server or receive connections

When CloudShell Shines

  • Quick commands when you don't have AWS CLI locally set up
  • Live incident investigation — tail logs while invoking functions
  • Running one-off Python/boto3 scripts without local environment setup
  • Testing IAM permissions by assuming roles safely
  • CloudFormation deployments straight from a cloned git repo
  • Auditing resources (security groups, IAM, S3 buckets) quickly
  • Connecting to private RDS in VPC mode without a bastion host
  • Learning and experimenting with AWS CLI safely
