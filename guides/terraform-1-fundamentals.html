🌍 What Is Terraform?

You've just been told to set up a new AWS environment — a VPC, subnets, EC2 instances, an RDS database, S3 buckets, security groups, IAM roles. You could click through the AWS console for an hour, screenshot everything, and hope you remember what you did when you need to recreate it next month for staging. Or you could write it in code, run one command, and have Terraform build the whole thing — reproducibly, every time.
🤔 Isn't that what AWS CloudFormation does? Why use Terraform?

CloudFormation is AWS-only — it manages AWS resources in AWS-flavoured YAML/JSON. Terraform uses HCL and has providers for AWS, Azure, GCP, Cloudflare, Datadog, GitHub, and hundreds more — so you can manage your ENTIRE infrastructure stack in one language across multiple clouds and services. Teams also find Terraform's plan output and error messages significantly clearer.

The Core Workflow — Just Three Commands

⚡ The Terraform Workflow
🤔 How does Terraform know WHAT to change?

Terraform keeps a state file recording exactly what it last created. When you run terraform plan, it compares: your code (what you WANT) vs the state file (what Terraform thinks EXISTS). The difference becomes the plan — a list of creates, updates, and deletes. Nothing actually changes until you run terraform apply.

Key Concepts

🔧 Provider

A plugin that knows how to talk to a specific API (AWS, Azure, GCP, GitHub...). Downloaded automatically on terraform init.

📦 Resource

A single piece of infrastructure — an S3 bucket, EC2 instance, DNS record. The basic unit you declare.

📖 Data Source

Read-only lookup — fetch info about something already existing (existing VPC, latest AMI ID) without managing it.

🗃️ State

A JSON file Terraform maintains recording what it created. The single source of truth for "what currently exists."

📐 Module

A reusable, packaged group of resources — like a function in programming. Can be local or from a registry.

🔌 Backend

Where the state file is stored — locally by default, in S3/Azure Blob in production for team sharing and locking.

What HCL Looks Like

HCL — The basics at a glance
# Tell Terraform which providers to use
terraform {
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"
    }
  }
}

provider "aws" {
  region = "eu-west-1"
}

# A resource block: create an S3 bucket
resource "aws_s3_bucket" "my_bucket" {
  bucket = "my-unique-bucket-name-12345"
  tags   = { Environment = "dev", ManagedBy = "Terraform" }
}

# A data source: look up an existing VPC — do NOT create it
data "aws_vpc" "existing" {
  filter {
    name   = "tag:Name"
    values = ["my-existing-vpc"]
  }
}

# An output: expose a value after apply
output "bucket_arn" {
  value = aws_s3_bucket.my_bucket.arn
}

# Resource address pattern: TYPE.NAME.ATTRIBUTE
# aws_s3_bucket.my_bucket.arn
# aws_vpc.main.id  etc.

💡 Implicit Dependencies

When you write vpc_id = aws_vpc.main.id, Terraform detects that this resource depends on aws_vpc.main and creates the VPC first automatically — no manual ordering needed.

📁 Folder Structure

Terraform doesn't enforce any folder structure — you could put everything in one file. But teams that skip structure quickly end up with an enormous, unfindable mess, prod and dev mixed together, and modules duplicated everywhere. Here are the patterns that actually work at different scales.

Level 1 — Single Environment (Getting Started)

my-project/ ├── main.tf # primary resources ├── variables.tf # input variable declarations ├── outputs.tf # output value declarations ├── providers.tf # provider + terraform block ├── terraform.tfvars # actual variable values (don't commit secrets!) └── backend.tf # where to store state
🤔 Why split across multiple .tf files? Does it matter?

Functionally Terraform treats ALL .tf files in a folder as one combined config — splitting is purely for readability. The convention of main/variables/outputs/providers is not mandatory, but it's so universally adopted that any Terraform developer landing in your repo immediately knows where to look.

Level 2 — Multiple Environments (The Standard Pattern)

infrastructure/ ├── environments/ │ ├── dev/ │ │ ├── main.tf # calls modules, dev-specific config │ │ ├── variables.tf │ │ ├── outputs.tf │ │ ├── providers.tf │ │ ├── backend.tf # points to dev state bucket/key │ │ └── terraform.tfvars # dev-specific values │ ├── staging/ # same structure │ └── prod/ # same structure └── modules/ # reusable, environment-agnostic ├── networking/ │ ├── main.tf │ ├── variables.tf │ └── outputs.tf ├── ecs-service/ └── rds/

💡 Each Environment Folder Is a Separate Terraform Root

Run terraform init / plan / apply from INSIDE the environment folder. Dev and prod have completely separate state files and separate apply operations. A mistake in dev never touches prod's state.

Level 3 — Enterprise / Monorepo Structure

infrastructure/ ├── global/ # org-level: Route53, IAM org policies ├── shared/ │ ├── networking/ # shared VPC, Transit Gateway │ └── security/ # shared KMS keys, IAM roles ├── environments/ │ ├── dev/ │ │ ├── networking/ # ← separate state per component │ │ ├── compute/ │ │ ├── database/ │ │ └── services/ │ └── prod/ # mirrors dev └── modules/ ├── aws/ │ ├── vpc/ │ ├── ecs-service/ │ ├── rds-postgres/ │ └── lambda-function/ └── app/ └── web-service/ # VPC + ECS + ALB + RDS bundled
🤔 Why split networking/compute/database into separate folders in Level 3?

Each folder = separate state = separate lock. If everything is one folder, applying a Lambda code change locks the ENTIRE environment — blocking everyone. Splitting by component means smaller blast radius, faster plan times, and parallel work across teams. This is exactly the coordination problem Terragrunt (Part 3) automates.

Files Terraform Creates Automatically

📁 .terraform/

Created by init — downloaded providers and modules. Never commit. Add to .gitignore.

🔒 .terraform.lock.hcl

Provider version lock file — SHOULD be committed. Like package-lock.json — ensures everyone uses identical provider versions.

📊 terraform.tfstate

The state file. Never commit when using local backend. Use remote backend (S3) for teams.

🧱 Core Building Blocks

Every Terraform config is built from a small set of block types. Once you know what each one does, you can read any Terraform code regardless of which cloud or what it creates.

1. terraform block

HCL — terraform block
terraform {
  required_version = ">= 1.6.0"
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"   # allow 5.x, block 6.x
    }
  }
  backend "s3" {
    bucket         = "my-tf-state"
    key            = "dev/main.tfstate"
    region         = "eu-west-1"
    dynamodb_table = "terraform-state-lock"
    encrypt        = true
  }
}

2. provider block

HCL — provider with default_tags and aliases
provider "aws" {
  region = "eu-west-1"
  default_tags {
    tags = {
      ManagedBy   = "Terraform"
      Environment = var.environment
    }
  }
}

# Alias for multi-region (e.g., ACM certs must be us-east-1 for CloudFront)
provider "aws" {
  alias  = "us_east"
  region = "us-east-1"
}

resource "aws_acm_certificate" "cert" {
  provider          = aws.us_east
  domain_name       = "example.com"
  validation_method = "DNS"
}

3. resource block

HCL — resource with lifecycle rules
resource "aws_security_group" "web" {
  name   = "web-sg"
  vpc_id = aws_vpc.main.id   # implicit dependency

  ingress {
    from_port   = 443
    to_port     = 443
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }

  lifecycle {
    create_before_destroy = true   # new SG before old deleted
    prevent_destroy       = false  # true for critical DBs
    ignore_changes        = [tags] # ignore manual tag changes
  }
}

4. data block — Reading Existing Resources

HCL — data sources
# Always look up the latest AL2023 AMI — never hardcode!
data "aws_ami" "amazon_linux" {
  most_recent = true
  owners      = ["amazon"]
  filter {
    name   = "name"
    values = ["al2023-ami-*-x86_64"]
  }
}

data "aws_secretsmanager_secret_version" "db_pass" {
  secret_id = "prod/db/password"
}

resource "aws_instance" "web" {
  ami           = data.aws_ami.amazon_linux.id   # always current
  instance_type = "t3.micro"
}

5. locals block

HCL — locals
locals {
  name_prefix  = "${var.project}-${var.environment}"
  is_prod      = var.environment == "prod"
  common_tags  = {
    Project     = var.project
    Environment = var.environment
    ManagedBy   = "Terraform"
  }
}

resource "aws_s3_bucket" "data" {
  bucket = "${local.name_prefix}-data"
  tags   = local.common_tags
}
🤔 When do I need explicit depends_on?

Only when there's a dependency Terraform CAN'T detect from attribute references — usually a side-effect dependency, like an ECS service that needs an IAM policy to be attached before it starts, but there's no attribute linking them directly. Use it sparingly — if you find yourself using it often, that often means you're missing an attribute reference.

HCL — explicit depends_on (use sparingly)
resource "aws_ecs_service" "api" {
  name    = "api-service"
  cluster = aws_ecs_cluster.main.id

  # ECS needs this IAM attachment to exist first, but no attribute links them
  depends_on = [aws_iam_role_policy_attachment.ecs_exec]
}

📥 Variables, Outputs & Locals

Imagine writing Terraform for dev, then needing staging identical except for instance size, domain name, and deletion protection. Without variables, you copy-paste everything and manually hunt through changing values. Variables make the same code work across every environment.

Input Variables

HCL — variables.tf
variable "environment" {
  description = "Deployment environment"
  type        = string
  default     = "dev"
  validation {
    condition     = contains(["dev", "staging", "prod"], var.environment)
    error_message = "Must be dev, staging, or prod."
  }
}

variable "instance_sizes" {
  type = map(string)
  default = {
    dev     = "t3.micro"
    staging = "t3.small"
    prod    = "t3.large"
  }
}

variable "database_config" {
  type = object({
    instance_class    = string
    allocated_storage = number
    multi_az          = bool
  })
  default = {
    instance_class    = "db.t3.micro"
    allocated_storage = 20
    multi_az          = false
  }
}

variable "db_password" {
  type      = string
  sensitive = true   # never shown in plan/apply output
}

Setting Variable Values

📄 terraform.tfvars (most common)

environment = "dev"
instance_sizes = {
  dev  = "t3.micro"
  prod = "t3.large"
}

🔑 Environment variables (CI/CD)

export TF_VAR_environment="prod"
export TF_VAR_db_password="secret"

⌨️ CLI flags

terraform apply \
  -var="environment=staging" \
  -var-file="staging.tfvars"

📊 Precedence (highest wins)

  • CLI -var flags
  • *.auto.tfvars files
  • terraform.tfvars
  • TF_VAR_ env vars
  • Variable default

💡 Outputs Are the API Between Terraform Configs

When splitting infra into separate state files (networking vs compute), the compute config reads networking outputs via terraform_remote_state or by passing the VPC ID as a variable. Outputs are how Terraform modules and root configs talk to each other.

Locals — DRY Computed Values

HCL — real-world locals usage
locals {
  name_prefix   = "${var.project}-${var.environment}"
  instance_type = var.environment == "prod" ? "t3.large" : "t3.micro"

  tags = merge(
    { Project = var.project, ManagedBy = "Terraform" },
    var.extra_tags
  )

  # Transform list to map — AZ => subnet ID
  subnet_by_az = {
    for s in aws_subnet.private : s.availability_zone => s.id
  }
}
# Use: local.name_prefix, local.tags, local.instance_type
🤔 What is the difference between a local and a variable?

Variables are INPUTS — set from outside (tfvars, CLI, environment). Locals are COMPUTED from variables or other values — calculated inside your config. A local can reference variables and resource attributes, but can't be set from outside. Think of variables as function parameters and locals as internal const values.

🗃️ State & Backends

Terraform's state file is its memory. It's how Terraform knows what already exists in the real world, what it created last time, and what needs to change next time you run plan. Get state management wrong and you'll end up with duplicated resources, or worse — orphaned resources that Terraform doesn't know to clean up.
🤔 What actually goes IN the state file?

The state file is a JSON document containing every resource Terraform manages: the resource type, name, all its attributes, its real cloud ID (e.g., the actual S3 bucket ARN), and metadata about dependencies. When you run terraform plan, Terraform diffs your code against this state to figure out what to create, update, or delete.

Local vs Remote Backend

💻 Local (default)

  • State saved as terraform.tfstate on your disk
  • Fine for solo learning projects
  • Never commit to git — it may contain secrets
  • No locking — two people running apply simultaneously = corruption

☁️ Remote (S3 + DynamoDB)

  • State stored in S3, encrypted at rest
  • DynamoDB table provides state locking — only one apply at a time
  • Shared across your team
  • Versioned — roll back to previous state if needed

Setting Up an S3 Backend (the AWS Standard)

HCL — backend.tf
terraform {
  backend "s3" {
    bucket         = "my-company-terraform-state"
    key            = "environments/dev/main.tfstate"
    region         = "eu-west-1"
    dynamodb_table = "terraform-state-lock"
    encrypt        = true
    kms_key_id     = "arn:aws:kms:eu-west-1:123:key/abc"   # optional CMK
  }
}
HCL — creating the state bucket and lock table
# This is the bootstrap — create ONCE manually or with a separate Terraform config
resource "aws_s3_bucket" "tf_state" {
  bucket = "my-company-terraform-state"
}

resource "aws_s3_bucket_versioning" "tf_state" {
  bucket = aws_s3_bucket.tf_state.id
  versioning_configuration { status = "Enabled" }
}

resource "aws_s3_bucket_server_side_encryption_configuration" "tf_state" {
  bucket = aws_s3_bucket.tf_state.id
  rule {
    apply_server_side_encryption_by_default {
      sse_algorithm = "AES256"
    }
  }
}

resource "aws_dynamodb_table" "tf_lock" {
  name         = "terraform-state-lock"
  billing_mode = "PAY_PER_REQUEST"
  hash_key     = "LockID"
  attribute {
    name = "LockID"
    type = "S"
  }
}

Essential State Commands

Shell — state management commands
# See what Terraform is tracking
terraform state list
terraform state show aws_s3_bucket.my_bucket

# Import an existing resource into state (so Terraform manages it)
terraform import aws_s3_bucket.my_bucket my-existing-bucket-name

# Move a resource in state (when you rename it in code)
terraform state mv aws_s3_bucket.old_name aws_s3_bucket.new_name

# Remove a resource from state WITHOUT deleting it from AWS
# (use when you want Terraform to stop managing something)
terraform state rm aws_s3_bucket.my_bucket

# See the raw plan in JSON (useful for automation)
terraform plan -out=tfplan
terraform show -json tfplan

# Refresh state to match real-world without applying changes
terraform refresh   # (deprecated — use terraform apply -refresh-only)
terraform apply -refresh-only

Workspaces — Poor Man's Multi-Environment

🤔 What are Terraform workspaces and when should I use them?

Workspaces let you maintain multiple separate state files within the SAME backend configuration — e.g., a "dev" and "prod" workspace both storing state in the same S3 bucket, but under different keys. This sounds convenient, but it has a big downside: both workspaces share the same code, so your prod config IS your dev code. For real environments with different variable values, the separate-folder approach (Tab 2, Level 2) is far safer. Workspaces are genuinely useful for short-lived feature branches and ephemeral test environments.

Shell — workspace commands
terraform workspace list          # list all workspaces
terraform workspace new feature-x # create new workspace
terraform workspace select dev    # switch to dev workspace
terraform workspace show          # current workspace name

# In HCL — use workspace name in resource names to avoid clashes
resource "aws_s3_bucket" "app" {
  bucket = "myapp-${terraform.workspace}-data"
  # → "myapp-dev-data", "myapp-prod-data"
}

⚠️ Never Manually Edit the State File

The state file is JSON and technically editable, but manually modifying it almost always leads to corruption. Use the terraform state subcommands instead — they do the same thing safely with proper validation.

📐 Modules

You've defined an ECS service in dev. Now you need the same ECS service in staging and prod. Copy-pasting 200 lines of Terraform three times means any bug fix or improvement needs to be made in three places. Modules are the solution — write the ECS service config once, use it everywhere, just pass different variables each time.
🤔 What IS a module structurally? Is it something special?

A module is just a folder of .tf files — the same structure you've already seen. What makes something a "module" is that another config calls it using a module block. Every root config you write is technically a module too (the "root module"). The difference is just whether something is called by something else.

Creating a Module

modules/ecs-service/ ├── main.tf # ECS task definition, service, IAM roles ├── variables.tf # what the caller must/can provide └── outputs.tf # what the caller gets back
HCL — modules/ecs-service/variables.tf
variable "service_name" { type = string }
variable "cluster_id"   { type = string }
variable "image_uri"    { type = string }
variable "cpu"          { type = number; default = 256 }
variable "memory"       { type = number; default = 512 }
variable "desired_count"{ type = number; default = 1 }
variable "environment_vars" {
  type    = map(string)
  default = {}
}
HCL — modules/ecs-service/main.tf (simplified)
resource "aws_ecs_task_definition" "this" {
  family                   = var.service_name
  cpu                      = var.cpu
  memory                   = var.memory
  network_mode             = "awsvpc"
  requires_compatibilities = ["FARGATE"]

  container_definitions = jsonencode([{
    name  = var.service_name
    image = var.image_uri
    environment = [
      for k, v in var.environment_vars : { name = k, value = v }
    ]
  }])
}

resource "aws_ecs_service" "this" {
  name            = var.service_name
  cluster         = var.cluster_id
  task_definition = aws_ecs_task_definition.this.arn
  desired_count   = var.desired_count
  launch_type     = "FARGATE"
}
HCL — modules/ecs-service/outputs.tf
output "service_name" { value = aws_ecs_service.this.name }
output "task_definition_arn" { value = aws_ecs_task_definition.this.arn }

Calling a Module

HCL — environments/dev/main.tf — calling a local module
module "api_service" {
  source = "../../modules/ecs-service"   # relative path to module folder

  service_name  = "api"
  cluster_id    = aws_ecs_cluster.main.id
  image_uri     = "123456789.dkr.ecr.eu-west-1.amazonaws.com/api:latest"
  cpu           = 256
  memory        = 512
  desired_count = 2
  environment_vars = {
    ENVIRONMENT = "dev"
    LOG_LEVEL   = "debug"
  }
}

# Access module outputs via module.NAME.OUTPUT
output "api_task_def" {
  value = module.api_service.task_definition_arn
}

Using Public Registry Modules

HCL — calling a module from the Terraform Registry
# Use a verified community module — saves building common things from scratch
module "vpc" {
  source  = "terraform-aws-modules/vpc/aws"   # registry.terraform.io
  version = "~> 5.0"

  name = "my-vpc"
  cidr = "10.0.0.0/16"

  azs             = ["eu-west-1a", "eu-west-1b", "eu-west-1c"]
  private_subnets = ["10.0.1.0/24", "10.0.2.0/24", "10.0.3.0/24"]
  public_subnets  = ["10.0.101.0/24", "10.0.102.0/24", "10.0.103.0/24"]

  enable_nat_gateway = true
  single_nat_gateway = true   # cheaper for dev, false for prod HA
}

# Use outputs:
# module.vpc.vpc_id
# module.vpc.private_subnets  (list of subnet IDs)
# module.vpc.public_subnets

💡 Always Pin Module Versions

Whether using local modules or registry modules, always specify a version constraint. Without pinning, terraform init -upgrade could pull in a breaking change and your previously-working apply will suddenly fail in a way that's hard to debug.

🔁 Loops & Conditionals

You need to create S3 buckets for 5 different teams, or create IAM roles for a list of services, or only create a CloudWatch alarm in prod and not dev. Without loops and conditionals you'd have 5 almost-identical resource blocks — and if the bucket config changes, 5 edits to make. Terraform gives you three main tools for this.

count — Simple Repetition

HCL — count
# Create 3 identical EC2 instances
resource "aws_instance" "web" {
  count         = 3
  ami           = data.aws_ami.amazon_linux.id
  instance_type = "t3.micro"
  tags          = { Name = "web-${count.index}" }  # 0, 1, 2
}

# Conditional — only create in prod
resource "aws_cloudwatch_metric_alarm" "high_cpu" {
  count           = var.environment == "prod" ? 1 : 0
  alarm_name      = "high-cpu-alarm"
  metric_name     = "CPUUtilization"
  threshold       = 80
  comparison_operator = "GreaterThanThreshold"
  evaluation_periods  = 2
  period              = 120
  statistic           = "Average"
  namespace           = "AWS/EC2"
}

# Reference counted resources — it becomes a list
# aws_instance.web[0].id, aws_instance.web[1].id, etc.
# aws_instance.web[*].id  → all IDs as a list

⚠️ count Has a Major Gotcha

Resources created with count are indexed by position (0, 1, 2...). If you remove the middle item from a list driving count, Terraform will try to DESTROY and recreate everything at index ≥ the removed item. Use for_each instead whenever you're iterating over a list that might change.

for_each — The Better Choice for Collections

HCL — for_each with maps and sets
# Create S3 buckets for each team — keyed by name, not position
variable "teams" {
  type    = set(string)
  default = ["platform", "data", "security"]
}

resource "aws_s3_bucket" "team" {
  for_each = var.teams          # each.key = "platform", "data", etc.
  bucket   = "mycompany-${each.key}-data"
  tags     = { Team = each.key }
}

# With a map — gives you both key and value
variable "service_ports" {
  type = map(number)
  default = { api = 8080, admin = 9000, metrics = 9100 }
}

resource "aws_security_group_rule" "services" {
  for_each          = var.service_ports
  type              = "ingress"
  from_port         = each.value   # the port number
  to_port           = each.value
  protocol          = "tcp"
  cidr_blocks       = ["10.0.0.0/8"]
  security_group_id = aws_security_group.internal.id
  description       = "Allow ${each.key} traffic"
}

# Reference for_each resources:
# aws_s3_bucket.team["platform"].id
# aws_s3_bucket.team["data"].arn
# { for k, v in aws_s3_bucket.team : k => v.arn }  ← all ARNs as map

for Expressions — Transforming Collections

HCL — for expressions (like list/dict comprehensions)
locals {
  # List comprehension — uppercase all team names
  team_names_upper = [for t in var.teams : upper(t)]

  # Map comprehension — subnet ID keyed by AZ
  subnet_by_az = {
    for s in aws_subnet.private : s.availability_zone => s.id
  }

  # Filter — only subnets in eu-west-1a
  primary_subnets = [
    for s in aws_subnet.private : s.id
    if s.availability_zone == "eu-west-1a"
  ]

  # Convert list to map (for_each needs map/set, not list)
  services_map = { for s in var.services : s.name => s }
}

dynamic blocks — Repeated Nested Blocks

HCL — dynamic blocks
# Without dynamic — hardcoded ingress rules (not flexible)
# resource "aws_security_group" "web" {
#   ingress { from_port = 80  ... }
#   ingress { from_port = 443 ... }
# }

# WITH dynamic — driven by a variable
variable "ingress_rules" {
  type = list(object({
    port        = number
    description = string
  }))
  default = [
    { port = 80,  description = "HTTP"  },
    { port = 443, description = "HTTPS" }
  ]
}

resource "aws_security_group" "web" {
  name   = "web-sg"
  vpc_id = aws_vpc.main.id

  dynamic "ingress" {
    for_each = var.ingress_rules   # iterate over the list
    content {
      from_port   = ingress.value.port
      to_port     = ingress.value.port
      protocol    = "tcp"
      cidr_blocks = ["0.0.0.0/0"]
      description = ingress.value.description
    }
  }
}

🚀 Advanced Patterns

moved block — Renaming Without Destroying

You created a resource as aws_s3_bucket.data three months ago. Now you want to rename it to aws_s3_bucket.app_data. If you just rename it in the code, Terraform will try to DESTROY the old one and CREATE a new one — which deletes your actual S3 bucket. The moved block tells Terraform "this is the same resource, just renamed."
HCL — moved block
# In main.tf — add this when renaming a resource
moved {
  from = aws_s3_bucket.data
  to   = aws_s3_bucket.app_data
}

# Same for resources inside modules
moved {
  from = module.old_name
  to   = module.new_name
}

# After applying, you can remove the moved block
# (or keep it as documentation — it has no effect once applied)

import block — Bringing Existing Resources Under Terraform

HCL — import block (Terraform 1.5+)
# Add the resource to your .tf file first, THEN import it
# The id is the AWS resource identifier (check provider docs for format)
import {
  to = aws_s3_bucket.legacy
  id = "my-already-existing-bucket"
}

resource "aws_s3_bucket" "legacy" {
  bucket = "my-already-existing-bucket"
}

# Then run: terraform plan  → shows import
#           terraform apply → imports into state, no actual changes to bucket

# Terraform 1.5+ can also GENERATE the resource block for you:
# terraform plan -generate-config-out=generated.tf

terraform_remote_state — Reading Outputs Across State Files

HCL — reading networking outputs from compute state
# In environments/dev/compute/main.tf
# Read outputs from the networking state without duplicating config
data "terraform_remote_state" "networking" {
  backend = "s3"
  config  = {
    bucket = "my-company-terraform-state"
    key    = "environments/dev/networking/main.tfstate"
    region = "eu-west-1"
  }
}

resource "aws_ecs_service" "api" {
  name    = "api"
  cluster = data.terraform_remote_state.networking.outputs.ecs_cluster_id
}

# Alternative: pass the VPC ID as a variable instead of using remote state
# Simpler, more decoupled, preferred by many teams

precondition & postcondition — Runtime Validation

HCL — lifecycle precondition/postcondition
resource "aws_db_instance" "main" {
  instance_class    = var.database_config.instance_class
  allocated_storage = var.database_config.allocated_storage
  multi_az          = var.database_config.multi_az

  lifecycle {
    # Check BEFORE creating — validated at plan time
    precondition {
      condition     = var.environment == "prod" ? var.database_config.multi_az : true
      error_message = "Production database MUST have multi_az = true"
    }
    # Check AFTER creating — validated after apply
    postcondition {
      condition     = self.status == "available"
      error_message = "Database did not reach available status"
    }
  }
}

templatefile() — Generating Config from Templates

HCL — templatefile for EC2 user_data scripts
# templates/user_data.sh.tpl
#!/bin/bash
yum update -y
echo "ENVIRONMENT=${environment}" >> /etc/environment
aws s3 cp s3://${config_bucket}/app.conf /etc/app/app.conf

# In main.tf
resource "aws_instance" "app" {
  ami           = data.aws_ami.amazon_linux.id
  instance_type = "t3.micro"

  user_data = templatefile("${path.module}/templates/user_data.sh.tpl", {
    environment   = var.environment
    config_bucket = aws_s3_bucket.config.id
  })
}

null_resource & terraform_data — Running Scripts

HCL — running a script after a resource is created
resource "terraform_data" "db_migration" {
  # Re-run whenever the Lambda image changes
  triggers_replace = [aws_lambda_function.migrator.image_uri]

  provisioner "local-exec" {
    command = "aws lambda invoke --function-name ${aws_lambda_function.migrator.function_name} /dev/null"
  }

  depends_on = [aws_db_instance.main]
}

⚠️ Avoid Provisioners When Possible

Provisioners (local-exec, remote-exec) are a last resort. They run outside of Terraform's normal plan/apply cycle — errors can leave state out of sync, and they're hard to make idempotent. Prefer Lambda, SSM Documents, EC2 user_data, or purpose-built resources over provisioners.
