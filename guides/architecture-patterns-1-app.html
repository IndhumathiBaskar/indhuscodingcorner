🏛️ What Are Architecture Patterns?

👶 Start Here — What Is Application Architecture?

If Design Patterns answer "how do I structure this class?", Architecture Patterns answer "how do I structure my entire application?" They define the big-picture organisation — which folders exist, what responsibilities each layer has, how data flows between them, and how different parts of your codebase talk to each other. Getting architecture right means your app is easy to test, easy to change, and easy to grow.

The 6 Application Architecture Patterns

🌐
MVC
Model-View-Controller. Classic web pattern. Controller handles requests, Model holds data, View renders HTML.
ASP.NET Core MVC · Rails · Django
🖥️
MVVM
Model-View-ViewModel. Two-way data binding. View binds to ViewModel, ViewModel has no UI knowledge.
WPF · MAUI · Angular · Vue
📱
MVP
Model-View-Presenter. Presenter drives the View through an interface. Fully testable UI logic.
WinForms · Android · Legacy desktop
🏢
Layered / N-Tier
Split application into horizontal layers — Presentation, Business Logic, Data Access. Each layer talks only to the one below it.
Enterprise apps · Classic 3-tier
⭕

Concentric circles. Domain at the centre. Dependencies only point inward. Infrastructure at the outer edge.
Modern .NET · Domain-driven apps
⚡
CQRS
Command Query Responsibility Segregation. Separate the read model from the write model. Optimise each independently.
High-scale APIs · Event sourcing

Architecture vs Design Patterns

🔩 Design Patterns (Parts 1-3)

  • Solve problems within a single class or small group of classes
  • Examples: Singleton, Factory, Observer, Strategy
  • Scope: a few lines to a few classes
  • Language-specific (C# patterns)

🏛️ Architecture Patterns (This Guide)

  • Define the entire high-level structure of the application
  • Examples: MVC, Clean Architecture, CQRS
  • Scope: folders, projects, entire solutions
  • Language-agnostic (apply to any tech stack)

How They Relate to SOLID

🔗 Architecture Patterns Are SOLID at the Application Level

  • SRP — Each layer/component has one responsibility (Controller routes, Service processes, Repository saves)
  • OCP — Add new features in new classes/layers without changing existing infrastructure
  • DIP — All patterns depend on abstractions (interfaces) not concrete implementations
  • Clean Architecture is the most explicit expression of all 5 SOLID principles at the architectural level

🌐 MVC — Model View Controller

👶 In Plain English

MVC splits your application into three parts. The Model holds your data and business logic. The View is what the user sees (HTML page, JSON response). The Controller sits in the middle — it receives the request, asks the Model for data, and tells the View what to display. They each have one job and don't need to know each other's internals.

💡 Real World Analogy

A restaurant. The menu (View) is what the customer sees. The waiter (Controller) takes the order and communicates between customer and kitchen. The kitchen (Model) holds the recipes and prepares the food. The customer never goes into the kitchen. The kitchen never designs the menu. The waiter is the coordinator.

// Model — represents data and business rules
public class Product
{
    public int     Id       { get; set; }
    public string  Name     { get; set; } = "";
    public decimal Price    { get; set; }
    public int     Stock    { get; set; }
    public bool    InStock  => Stock > 0;
}

// Service (part of the Model layer — business logic)
public interface IProductService
{
    Task<List<Product>> GetAllAsync();
    Task<Product?>      GetByIdAsync(int id);
    Task                 CreateAsync(Product product);
    Task                 UpdateAsync(Product product);
    Task                 DeleteAsync(int id);
}

public class ProductService : IProductService
{
    private readonly IProductRepository _repo;
    public ProductService(IProductRepository repo) => _repo = repo;

    public Task<List<Product>> GetAllAsync()        => _repo.GetAllAsync();
    public Task<Product?>      GetByIdAsync(int id) => _repo.GetByIdAsync(id);
    public Task                 CreateAsync(Product p) => _repo.AddAsync(p);
    public Task                 UpdateAsync(Product p) => _repo.UpdateAsync(p);
    public Task                 DeleteAsync(int id)    => _repo.DeleteAsync(id);
}
C# — Controller (handles HTTP requests)
// Controller — receives requests, calls Model, passes data to View
public class ProductsController : Controller
{
    private readonly IProductService _service;

    public ProductsController(IProductService service) => _service = service;

    // GET /Products — List all products
    public async Task<IActionResult> Index()
    {
        var products = await _service.GetAllAsync();
        return View(products);   // passes model to Index.cshtml
    }

    // GET /Products/Details/5
    public async Task<IActionResult> Details(int id)
    {
        var product = await _service.GetByIdAsync(id);
        if (product == null) return NotFound();
        return View(product);
    }

    // POST /Products/Create
    [HttpPost]
    [ValidateAntiForgeryToken]
    public async Task<IActionResult> Create(Product product)
    {
        if (!ModelState.IsValid) return View(product);  // re-show form with errors
        await _service.CreateAsync(product);
        return RedirectToAction(nameof(Index));
    }

    // DELETE /Products/Delete/5
    [HttpPost, ActionName("Delete")]
    public async Task<IActionResult> DeleteConfirmed(int id)
    {
        await _service.DeleteAsync(id);
        return RedirectToAction(nameof(Index));
    }
}

[ApiController]
[Route("api/[controller]")]
public class ProductsController : ControllerBase
{
    private readonly IProductService _service;
    public ProductsController(IProductService service) => _service = service;

    [HttpGet]
    public async Task<ActionResult<List<Product>>> GetAll()
        => Ok(await _service.GetAllAsync());

    [HttpGet("{id}")]
    public async Task<ActionResult<Product>> GetById(int id)
    {
        var product = await _service.GetByIdAsync(id);
        return product == null ? NotFound() : Ok(product);
    }

    [HttpPost]
    public async Task<ActionResult<Product>> Create(Product product)
    {
        await _service.CreateAsync(product);
        return CreatedAtAction(nameof(GetById), new { id = product.Id }, product);
    }

    [HttpDelete("{id}")]
    public async Task<IActionResult> Delete(int id)
    {
        await _service.DeleteAsync(id);
        return NoContent();
    }
}

🖥️ MVVM — Model View ViewModel

👶 In Plain English

MVVM is the go-to pattern for data-bound desktop and mobile apps. The View (XAML) binds directly to properties on the ViewModel (C# class). When the ViewModel's data changes, the View updates automatically — no code needed. The ViewModel knows nothing about buttons, labels, or textboxes — it just exposes properties and commands.

💡 Real World Analogy

A live scoreboard at a stadium. The scoreboard (View) is wired to a data feed (ViewModel). When the score changes in the system, the board updates immediately — you don't send a worker up to change numbers manually. The data feed doesn't know what the board looks like — it just updates the numbers. The board doesn't know how the score is calculated — it just shows what it's given.

🖥️ MVVM — Two-Way Data Binding

Complete MVVM Example — WPF Contact Form

C# — ViewModelBase + ContactViewModel
// ── Base class: INotifyPropertyChanged in one place ───────────────
public abstract class ViewModelBase : INotifyPropertyChanged
{
    public event PropertyChangedEventHandler? PropertyChanged;

    protected bool SetProperty<T>(ref T field, T value,
        [CallerMemberName] string? name = null)
    {
        if (EqualityComparer<T>.Default.Equals(field, value)) return false;
        field = value;
        PropertyChanged?.Invoke(this, new PropertyChangedEventArgs(name));
        return true;
    }
}

// ── ContactViewModel — zero WPF references ────────────────────────
public class ContactViewModel : ViewModelBase
{
    private readonly IContactService _service;

    private string _firstName = "";
    private string _lastName  = "";
    private string _email     = "";
    private string _status    = "";
    private bool   _isBusy;

    public string FirstName
    {
        get => _firstName;
        set { SetProperty(ref _firstName, value); OnPropertyChanged(nameof(FullName)); }
    }
    public string LastName
    {
        get => _lastName;
        set { SetProperty(ref _lastName, value); OnPropertyChanged(nameof(FullName)); }
    }
    public string Email   { get => _email;   set => SetProperty(ref _email,   value); }
    public string Status  { get => _status;  set => SetProperty(ref _status,  value); }
    public bool   IsBusy  { get => _isBusy;  set => SetProperty(ref _isBusy,  value); }

    // Computed property
    public string FullName => $"{FirstName} {LastName}".Trim();
    public bool   CanSave  => !string.IsNullOrWhiteSpace(FirstName) &&
                               !string.IsNullOrWhiteSpace(Email);

    // Commands — what buttons bind to
    public ICommand SaveCommand  { get; }
    public ICommand ClearCommand { get; }

    public ContactViewModel(IContactService service)
    {
        _service     = service;
        SaveCommand  = new AsyncRelayCommand(SaveAsync, () => CanSave);
        ClearCommand = new RelayCommand(_ => Clear());
    }

    private async Task SaveAsync()
    {
        IsBusy = true;
        Status = "Saving...";
        try
        {
            await _service.SaveAsync(new Contact
                { FirstName = FirstName, LastName = LastName, Email = Email });
            Status = "✅ Contact saved!";
        }
        catch (Exception ex) { Status = $"❌ Error: {ex.Message}"; }
        finally { IsBusy = false; }
    }

    private void Clear()
    { FirstName = ""; LastName = ""; Email = ""; Status = ""; }

    private void OnPropertyChanged(string name)
        => PropertyChanged?.Invoke(this, new PropertyChangedEventArgs(name));
}
XAML — View binds to ViewModel (zero C# code-behind)

📱 MVP — Model View Presenter

👶 In Plain English

MVP is similar to MVC but the Presenter drives the View directly through an interface. The View is completely passive — it just displays what the Presenter tells it to, and passes every user action to the Presenter. The Presenter contains all the logic. Because the View is behind an interface, you can test the Presenter without touching any real UI.

💡 Real World Analogy

A puppet show. The puppet (View) has no mind of its own — it does exactly what the puppeteer (Presenter) makes it do. The audience sees the puppet moving, but all the intelligence is in the puppeteer's hands. The puppeteer can work with any puppet that has the right control points (interface) — they don't care whether it's a wooden puppet or a cloth one.

C# — MVP full example (WinForms style)
// ── Step 1: IView interface — what the Presenter can tell the View ─
public interface ILoginView
{
    // Properties: Presenter reads/writes these
    string Username    { get; }
    string Password    { get; }
    bool   IsLoading   { set; }
    string ErrorMessage{ set; }

    // Events: View fires these when user acts
    event EventHandler LoginClicked;
    event EventHandler CancelClicked;

    // Methods: Presenter calls these
    void NavigateToDashboard();
    void ShowError(string message);
}

// ── Step 2: Presenter — all the logic, no UI knowledge ────────────
public class LoginPresenter
{
    private readonly ILoginView    _view;
    private readonly IAuthService  _auth;

    public LoginPresenter(ILoginView view, IAuthService auth)
    {
        _view = view;
        _auth = auth;
        // Subscribe to View events
        _view.LoginClicked  += OnLoginClicked;
        _view.CancelClicked += OnCancelClicked;
    }

    private async void OnLoginClicked(object? sender, EventArgs e)
    {
        // Validation
        if (string.IsNullOrWhiteSpace(_view.Username))
        {
            _view.ShowError("Username is required");
            return;
        }

        // Update View state
        _view.IsLoading    = true;
        _view.ErrorMessage = "";

        try
        {
            var success = await _auth.LoginAsync(_view.Username, _view.Password);
            if (success)
                _view.NavigateToDashboard();
            else
                _view.ShowError("Invalid username or password");
        }
        catch (Exception ex)
        {
            _view.ShowError($"Login failed: {ex.Message}");
        }
        finally { _view.IsLoading = false; }
    }

    private void OnCancelClicked(object? sender, EventArgs e)
        => Environment.Exit(0);
}

// ── Step 3: Concrete View — WinForms Form ─────────────────────────
public class LoginForm : Form, ILoginView
{
    private readonly TextBox _txtUsername = new();
    private readonly TextBox _txtPassword = new();
    private readonly Button  _btnLogin    = new();
    private readonly Label   _lblError    = new();

    public string Username     => _txtUsername.Text;
    public string Password     => _txtPassword.Text;
    public bool   IsLoading    { set => _btnLogin.Enabled = !value; }
    public string ErrorMessage { set => _lblError.Text = value; }

    public event EventHandler? LoginClicked;
    public event EventHandler? CancelClicked;

    public LoginForm()
    {
        _btnLogin.Click += (s, e) => LoginClicked?.Invoke(s, e);
        // layout setup...
    }

    public void NavigateToDashboard() { new DashboardForm().Show(); Close(); }
    public void ShowError(string msg)  => _lblError.Text = msg;
}

// ── Testable! Fake view in unit tests ─────────────────────────────
public class FakeLoginView : ILoginView
{
    public string Username     { get; set; } = "alice";
    public string Password     { get; set; } = "pass";
    public bool   IsLoading    { get; set; }
    public string ErrorMessage { get; set; } = "";
    public bool   NavigatedToDashboard { get; private set; }

    public event EventHandler? LoginClicked;
    public event EventHandler? CancelClicked;
    public void NavigateToDashboard() => NavigatedToDashboard = true;
    public void ShowError(string msg)  => ErrorMessage = msg;

}

🏢 Layered / N-Tier Architecture

👶 In Plain English

Layered architecture divides your application into horizontal layers, each with a specific responsibility. The most common is 3-tier: Presentation (UI/API), Business Logic (services), Data Access (database). Each layer can only talk to the layer directly below it. This creates clear boundaries and makes each layer independently testable and replaceable.

💡 Real World Analogy

A restaurant kitchen brigade. The head chef (Presentation) takes orders and plates food. The sous chef (Business Logic) cooks and applies recipes. The prep cook (Data Access) fetches ingredients from the walk-in (Database). Each level only communicates with the one immediately below — the head chef doesn't go to the walk-in themselves.

How Layers Talk to Each Other

Layered Architecture in C#

C# — Typical 3-tier project structure
// ── Folder / Project Structure ────────────────────────────────────
// MyApp.sln
// ├── MyApp.Web           (Presentation Layer — ASP.NET Core project)
// │   ├── Controllers/
// │   ├── Views/
// │   └── Program.cs
// ├── MyApp.Application   (Business Logic Layer — class library)
// │   ├── Services/
// │   ├── Interfaces/
// │   └── DTOs/
// ├── MyApp.Infrastructure(Data Access Layer — class library)
// │   ├── Repositories/
// │   ├── DbContext/
// │   └── ExternalServices/
// └── MyApp.Domain        (Shared models — class library)
//     └── Models/

// ── Data Access Layer — Repository ────────────────────────────────
// MyApp.Infrastructure/Repositories/OrderRepository.cs
public class OrderRepository : IOrderRepository
{
    private readonly AppDbContext _ctx;
    public OrderRepository(AppDbContext ctx) => _ctx = ctx;

    public async Task<List<Order>> GetByCustomerIdAsync(int customerId)
        => await _ctx.Orders
            .Where(o => o.CustomerId == customerId)
            .Include(o => o.Items)
            .ToListAsync();

    public async Task SaveAsync(Order order)
    {
        _ctx.Orders.Add(order);
        await _ctx.SaveChangesAsync();
    }
}

// ── Business Logic Layer — Service ────────────────────────────────
// MyApp.Application/Services/OrderService.cs
public class OrderService : IOrderService
{
    private readonly IOrderRepository _repo;
    private readonly IEmailService    _email;
    private readonly IInventoryService _inventory;

    public OrderService(
        IOrderRepository repo,
        IEmailService    email,
        IInventoryService inventory)
    { _repo = repo; _email = email; _inventory = inventory; }

    public async Task<OrderResult> PlaceOrderAsync(PlaceOrderRequest request)
    {
        // Business rule: check stock
        if (!await _inventory.CheckStockAsync(request.Items))
            return OrderResult.Fail("Insufficient stock");

        // Business rule: calculate total
        var order = new Order
        {
            CustomerId = request.CustomerId,
            Items      = request.Items,
            Total      = request.Items.Sum(i => i.Price * i.Quantity),
            OrderDate  = DateTime.UtcNow
        };

        // Save to DB via repository
        await _repo.SaveAsync(order);

        // Send confirmation email
        await _email.SendOrderConfirmationAsync(order);

        return OrderResult.Success(order.Id);
    }
}

// ── Presentation Layer — Controller ───────────────────────────────
// MyApp.Web/Controllers/OrdersController.cs
[ApiController][Route("api/[controller]")]
public class OrdersController : ControllerBase
{
    private readonly IOrderService _service;
    public OrdersController(IOrderService service) => _service = service;

    [HttpPost]
    public async Task<IActionResult> PlaceOrder(PlaceOrderRequest request)
    {
        var result = await _service.PlaceOrderAsync(request);
        return result.Success ? Ok(result) : BadRequest(result.Error);
    }
}
SOLID: S — Each layer has one responsibility D — Each layer depends on interfaces of the layer below O — Swap infrastructure (SQL → NoSQL) without touching business logic

⭕ Clean Architecture

👶 In Plain English

Clean Architecture (by Robert C. Martin) is like onion rings — concentric circles where the most important code is at the centre. The Domain (your business rules) sits at the very centre and has zero dependencies on anything else. The database, UI, and external services are at the outer edge. Dependencies only point inward — inner circles never know about outer ones. This means you can swap your entire database without touching a single line of business logic.

💡 Real World Analogy

Your brain doesn't know or care what operating system it's running on, which keyboard you're using, or what database stores your memories. The brain (Domain) is independent of all I/O devices. This is Clean Architecture — your core business logic is completely independent of frameworks, databases, UI, and external services. You could swap the entire database and your brain logic wouldn't change.

The Concentric Layers

⭕ Clean Architecture — Dependency Rule: Always Inward

Clean Architecture in C# — Project Structure

C# — Clean Architecture solution layout
// Solution structure:
// MyApp.sln
// ├── MyApp.Domain              ← innermost circle, NO dependencies
// │   ├── Entities/
// │   │   └── Order.cs          ← pure business entity
// │   ├── Interfaces/
// │   │   ├── IOrderRepository.cs ← interface defined HERE, implemented elsewhere
// │   │   └── IEmailService.cs
// │   └── ValueObjects/
// │       └── Money.cs
// ├── MyApp.Application         ← depends on Domain only
// │   ├── UseCases/
// │   │   └── PlaceOrderUseCase.cs
// │   └── DTOs/
// │       └── PlaceOrderRequest.cs
// ├── MyApp.Infrastructure      ← depends on Application + Domain
// │   ├── Persistence/
// │   │   └── OrderRepository.cs  ← implements IOrderRepository
// │   └── Email/
// │       └── SmtpEmailService.cs ← implements IEmailService
// └── MyApp.Api                 ← depends on Application only
//     └── Controllers/
//         └── OrdersController.cs

// ── Domain Entity — zero dependencies ────────────────────────────
namespace MyApp.Domain.Entities;

public class Order
{
    public Guid              Id         { get; private set; } = Guid.NewGuid();
    public int               CustomerId { get; private set; }
    public List<OrderLine>  Lines      { get; private set; } = new();
    public OrderStatus       Status     { get; private set; } = OrderStatus.Pending;
    public DateTime          CreatedAt  { get; } = DateTime.UtcNow;

    // Domain logic lives in the entity itself
    public decimal CalculateTotal() => Lines.Sum(l => l.Quantity * l.UnitPrice);

    public void Confirm()
    {
        if (Status != OrderStatus.Pending)
            throw new InvalidOperationException("Only pending orders can be confirmed");
        Status = OrderStatus.Confirmed;
    }

    public static Order Create(int customerId, List<OrderLine> lines)
    {
        if (!lines.Any()) throw new ArgumentException("Order must have at least one line");
        return new Order { CustomerId = customerId, Lines = lines };
    }
}

// ── Domain Interface — defined in Domain, implemented in Infrastructure
namespace MyApp.Domain.Interfaces;

public interface IOrderRepository
{
    Task<Order?>        GetByIdAsync(Guid id);
    Task<List<Order>>  GetByCustomerAsync(int customerId);
    Task                SaveAsync(Order order);
}

// ── Application Layer — Use Case ──────────────────────────────────
namespace MyApp.Application.UseCases;

public class PlaceOrderUseCase
{
    private readonly IOrderRepository _repo;   // Domain interface
    private readonly IEmailService    _email;  // Domain interface

    public PlaceOrderUseCase(IOrderRepository repo, IEmailService email)
    { _repo = repo; _email = email; }

    public async Task<Guid> ExecuteAsync(PlaceOrderRequest request)
    {
        var order = Order.Create(request.CustomerId, request.Lines);
        order.Confirm();
        await _repo.SaveAsync(order);
        await _email.SendConfirmationAsync(order);
        return order.Id;
    }
}

// ── Infrastructure — implements Domain interfaces ─────────────────
namespace MyApp.Infrastructure.Persistence;

public class EfOrderRepository : IOrderRepository   // implements Domain interface
{
    private readonly AppDbContext _ctx;
    public EfOrderRepository(AppDbContext ctx) => _ctx = ctx;

    public async Task<Order?> GetByIdAsync(Guid id)
        => await _ctx.Orders.FindAsync(id);
    public async Task SaveAsync(Order order)
    { await _ctx.Orders.AddAsync(order); await _ctx.SaveChangesAsync(); }
    public async Task<List<Order>> GetByCustomerAsync(int cId)
        => await _ctx.Orders.Where(o => o.CustomerId == cId).ToListAsync();
}

The Dependency Rule — The Most Important Rule

💡 The Golden Rule of Clean Architecture

Source code dependencies can only point inward. Nothing in an inner circle can know about anything in an outer circle. The Domain doesn't know about EF Core. The Application doesn't know about SQL Server. The API doesn't know about the database. This is enforced by having infrastructure implement interfaces defined in the Domain.

SOLID: D — All layers depend on abstractions (interfaces in Domain) S — Each circle has one reason to change O — Swap DB/UI by adding new implementations, not editing Domain L — Any IOrderRepository implementation works anywhere

⚡ CQRS — Command Query Responsibility Segregation

👶 In Plain English

CQRS is simple: separate reading data (Queries) from writing data (Commands). You have one model/path for writes (creating, updating, deleting) and a completely separate model/path for reads (fetching, displaying). This lets you optimise each side independently — your read queries can return flat, denormalised data perfect for your screen, while your writes enforce all business rules on rich domain objects.

💡 Real World Analogy

A bank. The teller (Command) who processes deposits and withdrawals follows strict rules and updates the account. The bank statement generator (Query) reads a pre-prepared view of your account history in a flat, readable format. They're completely separate systems — the statement generator doesn't use the same complex transactional model that the teller uses. Both are optimised for their specific job.

Traditional vs CQRS Approach

❌ Traditional — One Model for Everything
  • Same complex Order entity used for reads AND writes
  • Read queries drag along all navigation properties and business rules
  • Fetching a list needs full domain model — slow and heavy
  • Write model needs validation, state machines, events — complex
  • Scaling reads means scaling writes too
✅ CQRS — Separate Read and Write Models
  • Write model: rich domain entities, business rules, events
  • Read model: flat DTOs optimised for what the screen needs
  • Read queries are fast — select exactly what's needed
  • Scale reads and writes independently
  • Each side optimised for its specific job

CQRS with MediatR

C# — CQRS Commands and Queries with MediatR
// Install: dotnet add package MediatR

// ── COMMANDS — change state ────────────────────────────────────────
// Command: represents an intent to change something
public record PlaceOrderCommand(
    int              CustomerId,
    List<OrderLine> Lines
) : IRequest<Guid>;   // IRequest = MediatR command with return type

// Command Handler: processes the command
public class PlaceOrderCommandHandler
    : IRequestHandler<PlaceOrderCommand, Guid>
{
    private readonly IOrderRepository _repo;
    private readonly IEmailService    _email;

    public PlaceOrderCommandHandler(IOrderRepository repo, IEmailService email)
    { _repo = repo; _email = email; }

    public async Task<Guid> Handle(PlaceOrderCommand cmd, CancellationToken ct)
    {
        var order = Order.Create(cmd.CustomerId, cmd.Lines);
        order.Confirm();
        await _repo.SaveAsync(order, ct);
        await _email.SendConfirmationAsync(order, ct);
        return order.Id;
    }
}

// ── QUERIES — read state ───────────────────────────────────────────
// Read DTO — flat, optimised for what the UI needs (not domain model)
public record OrderSummaryDto(
    Guid     Id,
    string   CustomerName,
    decimal  Total,
    string   Status,
    DateTime OrderDate,
    int      ItemCount
);

// Query: represents a request for data
public record GetOrdersQuery(int CustomerId) : IRequest<List<OrderSummaryDto>>;

// Query Handler: reads optimised data — can use Dapper, raw SQL, etc.
public class GetOrdersQueryHandler
    : IRequestHandler<GetOrdersQuery, List<OrderSummaryDto>>
{
    private readonly IDbConnection _db;   // raw Dapper — fast reads
    public GetOrdersQueryHandler(IDbConnection db) => _db = db;

    public async Task<List<OrderSummaryDto>> Handle(
        GetOrdersQuery query, CancellationToken ct)
    {
        // Flat, optimised SQL — no complex domain objects loaded
        var sql = """
            SELECT o.Id, c.Name AS CustomerName,
                   SUM(ol.Quantity * ol.UnitPrice) AS Total,
                   o.Status, o.CreatedAt AS OrderDate,
                   COUNT(ol.Id) AS ItemCount
            FROM Orders o
            JOIN Customers c ON c.Id = o.CustomerId
            JOIN OrderLines ol ON ol.OrderId = o.Id
            WHERE o.CustomerId = @CustomerId
            GROUP BY o.Id, c.Name, o.Status, o.CreatedAt
            ORDER BY o.CreatedAt DESC
            """;

        var results = await _db.QueryAsync<OrderSummaryDto>(sql,
            new { query.CustomerId });
        return results.ToList();
    }
}

// ── Controller — sends Commands and Queries through MediatR ───────
[ApiController][Route("api/[controller]")]
public class OrdersController : ControllerBase
{
    private readonly IMediator _mediator;
    public OrdersController(IMediator mediator) => _mediator = mediator;

    // QUERY — thin read path
    [HttpGet("customer/{customerId}")]
    public async Task<IActionResult> GetOrders(int customerId)
        => Ok(await _mediator.Send(new GetOrdersQuery(customerId)));

    // COMMAND — rich write path
    [HttpPost]
    public async Task<IActionResult> PlaceOrder(PlaceOrderCommand cmd)
    {
        var orderId = await _mediator.Send(cmd);
        return CreatedAtAction(nameof(GetOrders), new { orderId }, orderId);
    }
}

When to Use CQRS

  • Read and write workloads have very different performance requirements
  • Complex domain with many business rules on writes but simple flat reads
  • Working with Event Sourcing — commands generate events, queries read projections
  • Teams can work independently on read and write paths
  • You need to scale reads independently from writes
  • Simple CRUD apps — CQRS adds unnecessary complexity
  • Small teams where the overhead isn't justified
  • When read and write models are identical
